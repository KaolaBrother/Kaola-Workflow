# D-513-01. Planner authoring rubric for speculative-open-eligible topologies

Date: 2026-06-16
Status: **Accepted** — shipped in the #513 build run (bundle-513-514).
Issue: #513 (follow-up to #500 L3)
Related: #439 (speculative-read kernel / `speculative_open_policy`), #500 (`D-500-02` — L3 prose wire),
         `docs/plan-run-cards/speculative-open.md` (operator card + new § Authoring),
         D-419-01 (scheduler-default posture — the adjacent rubric), `agents/workflow-planner.md`.

## Context

#500 (L3) made the speculative-open mechanism *reachable*: the plan-run skeleton, the operator card,
and the route-reachability T9 pin all describe `open-ready --speculative-consent`. But the *realizable*
makespan win needs a plan SHAPED to use it — a read-only node whose sole blocker is an in-progress gate
that is very likely to pass. No live plan authored that shape, and the planner had no rubric teaching it
to recognize one. #500's own decision record (D-500-01, L3) explicitly deferred this proactive-authoring
win to a separate follow-up so it would not hold up the wiring. This is that follow-up.

## Decision

Add an **authoring rubric** to the `workflow-planner` profile — prose only, **zero new grammar and zero
new mechanism**. The rubric teaches the planner to recognize a speculative-open-eligible topology and to
set `## Meta: speculative_open_policy: consent` (default `off`).

- **Eligibility (ALL of):** (a) the candidate node is read-only (`declared_write_set: —`) — never a write
  node; (b) its only unsatisfied predecessor is a single in-progress gate; (c) the gate is
  high-probability-pass. (d) verdict:fail rollback is acceptable (bounded rework on a gate fail).
- **Authoring control is the `## Meta` key ONLY.** Eligibility stays runtime/validator-derived; the
  planner never hand-adds a per-node `speculative: true` annotation (the same INV-17 discipline that
  forbids hand-adding `parallel_safe`).
- **Placement:** the fuller rubric sits adjacent to the D-419-01 scheduler-default posture in
  `agents/workflow-planner.md`; a dense forge-neutral paragraph mirrors it byte-identically across the
  three `workflow-planner.toml` twins; a worked in-grammar example (a read-only
  `adversarial-verifier`/`code-explorer` depending only on a `code-reviewer` gate) appears in the profile
  and in `docs/plan-run-cards/speculative-open.md` (§ Authoring).

## Why this shape (not the alternatives)

- **Not a new grammar / not a new mechanism.** The runtime (`open-ready --speculative-consent`,
  `resolveSpeculativePolicy`, the `next-action` `speculativePending` emission) already exists since
  #439/#500. The only gap was planner *judgment*, which is prose by nature.
- **A worked example is mandatory, not optional.** The #463 lesson (a capability that ships as inert
  scaffolding reads as "done" but never runs) applies to rubrics too: a recognition rule with no in-grammar
  worked topology tends to ship unused. The example is the cheap hedge against that relapse.
- **Machine-enforced propagation.** A `test-agent-profile-parity.js` needle
  (`unsatisfied predecessor is a high-probability-pass gate`) pins the rubric across all three twins
  (parity assertions 15→18), so it cannot silently drift out of an edition.

## Consequence / non-claim

This change ships an authoring rubric, a parity needle, and a worked example — **no live makespan probe**.
The proactive makespan win is realized only when a future real plan authors the speculative-open shape and
the executor overlaps the node ahead of its gate. Accordingly the deliverable is the *rubric* (Refs-#513
discipline), not a demonstrated overlap; a later run that authors the shape on a live issue is the artifact
that would prove the win. `decision: ask` audit metadata on the build run is advisory only — no approval
gate was added (planner-first, #44/#287).
