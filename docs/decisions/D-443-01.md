## Context

D-420-01 and D-420-02 defined the autopilot loop concept (Part 1 and Part 3) and the
consent-halt + triage payload (Part 2), but left the following questions open for the P1
implementation:

- **D1** — Receipt-reader boundary: how strictly is the lean-orchestrator boundary (#44)
  enforced in the driver script vs. the agent layer?
- **D2** — `next` statefulness: does the subcommand carry any in-process mutable state, or is it
  a pure function of on-disk artifacts?
- **D3** — Crash-resume strategy: separate state file or derived from the digest?
- **D4** — Bundle chaining: does one invocation of the driver chain through multiple bundles,
  or does it stop after one?
- **D5** — Stop classification: how are stop reasons derived (prose-grep vs. ground-truth field)?

Issue #443 implements the driver and settles all five questions. This record captures the binding
decisions and their consequences.

## Decision

### D1 — Pure receipt-reader, zero forge tokens, zero agent dispatch

`kaola-workflow-autopilot.js` is a **pure receipt-reader**. It holds zero forge-CLI tokens (`gh`,
`glab`, `tea`, GitHub, GitLab, Gitea identifiers) anywhere in its body. It never dispatches an
agent, never invokes `sink-merge.js`, `run-chains.js`, `adaptive-node.js`, or any forge binary.
Its only outputs are:

1. A stage descriptor or stop payload on stdout (when `--json` is passed).
2. One append-only JSONL line in the digest file (only via the `digest` subcommand or the
   `repair_applied` path inside `next`).

**Why.** Lean-orchestrator boundary (#44): agent owns selection, consent, and dispatch; script
owns stage atomicity and the digest receipt. Forge-neutrality: the aggregator composes over
receipt SHAPES, not forge APIs, so it runs byte-identical across all four editions.

**Constraint for downstream.** Any code path inside the driver that calls a forge CLI, invokes
`sink-merge.js`, or dispatches an `Agent()` is OUT OF SPEC and must be removed.

### D2 — `next` is a stateless function

The `next` subcommand carries no in-process mutable state. It is a deterministic function of:

- `--goal` (required) and `--project` (required once active)
- The last non-empty line of `kaola-workflow/<project>/.cache/autopilot-digest.jsonl`
- The relevant stage receipt on disk
- `--scout-result <path>` (required to advance out of the scout stage)
- `KAOLA_AUTOPILOT_REPAIR` env (repair consent mode)
- `KAOLA_GOAL` env (advisory, fed through to goal_check)

Calling `next` twice with the same on-disk state produces the same output. The only mutation is
the `repair_applied` digest append on the mechanical-repair path (logged immediately before
emitting the descriptor so that crash-during-repair yields a conservative replay on restart).

### D3 — Crash-resume via digest replay (append-only; no separate state file)

The digest file (`kaola-workflow/<project>/.cache/autopilot-digest.jsonl`) is the **only**
durable autopilot-state artifact. It is:

- **Append-only** — never rewritten or truncated.
- **JSONL** — one JSON object per line:
  `{"ts":"<ISO-Z-no-ms>","stage":"<scout|claim|plan|run|finalize>","result":"<advanced|stop:<reason>|repair_applied|goal_progress>","receipt_path":"<path|null>"}`.
  A `repair` field is also written when `result === 'repair_applied'`.
- **Crash-resume** — on restart, `next` reads the last non-empty, parseable line (corrupt or
  absent → cold start) and uses it plus the current stage receipt to compute the next descriptor.
  No separate state file, no rollback, no seek.

**Timestamp format:** `new Date().toISOString().replace(/\.\d{3}Z$/, 'Z')` — milliseconds
stripped, matching the claim.js convention.

**Write contract:** `mkdirSync(dir, {recursive:true})` then `appendFileSync(path, line+'\n')`.

### D4 — One bundle per invocation (OQ-5)

A single `next` invocation advances **one bundle** through its lifecycle. After a successful
finalize:

- `goal_check === 'satisfied'` → `stop:'goal_satisfied'` — the loop terminates.
- `goal_check === 'absent'` (v1 advisory path) → `result:'goal_progress'` with the scout's next
  recommendation; the operator re-invokes the driver for the next bundle.

No in-process chaining between bundles. This keeps the driver stateless per-invocation and
avoids re-entering the scout/claim cycle within one script call.

### D5 — Structural stops (ground-truth field, never prose-grep)

Each stop reason maps to a specific field or marker in a receipt artifact:

| Stop | Verified source and field |
|------|---------------------------|
| `goal_satisfied` | cmdFinalize stdout parsed as JSON → `closure_receipt.goal_check === 'satisfied'` (persisted to `.cache/finalize-stdout.json`; #441 `computeGoalCheck`) |
| `backlog_empty` | scout JSON (via `--scout-result`) → `backlog_empty === true && recommended_bundle === null` |
| `consent_halt` | `workflow-state.md` `^escalated_to_full: consent$` OR `workflow-plan.md` `^consent_halt: pending$` (#440) |
| `security_halt` | `workflow-state.md` `^escalated_to_full: security$` WITHOUT a concurrent consent marker and WITHOUT `consent_halt: pending` in the ledger |
| `typed_refusal` | `.cache/barrier-failed.json` with `result === 'barrier_failed'` (carries #440 triage); OR a claim/handoff/validator `{result:'refuse', reason}` envelope |
| `repair_limit` | Two `repair_applied` entries for the same `node_id` in the digest + a new `barrier_failed` for that node |

**Disambiguation — consent vs. security.** A consent halt writes BOTH `escalated_to_full: consent`
and `escalated_to_full: security` into `workflow-state.md` AND sets `consent_halt: pending` in
`workflow-plan.md`. Resolution: if `consent_halt: pending` is present → `consent_halt`; if only
`security` is present without a consent marker → `security_halt`.

## Verified Merged-Contract Field Names

The following field names and shapes are verified against the merged implementation in this
worktree; downstream consumers must use these EXACT names:

**`closure_receipt.goal_check`** — enum `'satisfied' | 'absent'`; `'unsatisfied'` is reserved and
never returned in v1. `goal_satisfied` stop fires on `=== 'satisfied'` only
(`computeGoalCheck`, `kaola-workflow-claim.js`).

**`sink-receipt.json` `steps.push_main`** — string `'done'` when the terminal sink step completes.
The full step-key sequence is defined in `SINK_STEPS` in `kaola-workflow-sink-merge.js`; the
autopilot gates ONLY on `steps.push_main`. Full-completion envelope: `{result:'ok', status:'sinked', receipt}`.

**`barrier-failed.json` `triage`** — shape
`{class:'lockfile_write'|'mirror_write'|'count_bump'|'write_set_overflow'|'unclassified', testDelta?:'<red>/<green>', proposed_repair?:{kind, node, paths[]}}`.
`proposed_repair.kind` valid values (verified against `adaptive-node.js`): `write_set_swap`,
`add_to_write_set`, `revert_overflow`. There is NO `repair_node` kind. Mechanical-class =
exactly `{add_to_write_set, write_set_swap}`; `revert_overflow` and absent/`unclassified` always
halt even under `auto`.

**`chain-receipt.json` `chains`** — array of `{name, exitCode, command, duration_ms, accepted_red?, accepted_red_issue?}`.
Tests-green = every element satisfies `exitCode === 0 || accepted_red === true`.

**`backlog_empty` scout envelope** — `{backlog_empty: true, recommended_bundle: null}` (top-level).
Standard envelope is `{recommended_bundle:{…}}`. Both shapes are now documented in
`agents/issue-scout.md` and the three TOML twins.

## Mechanical-Class Repair Boundary

Under `KAOLA_AUTOPILOT_REPAIR=auto`, the driver auto-applies ONLY two `proposed_repair.kind`
values: `add_to_write_set` and `write_set_swap`. These are "mechanical" because the orchestrator
can apply them without judgment (adding a declared path or swapping a write-set entry).

All other cases halt even under `auto`:

- `revert_overflow` — requires a `revert-overflow` script invocation; never auto-applied.
- Absent `proposed_repair` — no structured repair available.
- `class: 'unclassified'` — unknown failure; requires operator judgment.

The bounded retry rule: one auto-repair per node. If the digest already contains a
`repair_applied` entry for the same `node_id`, a second `barrier_failed` for that node emits
`stop:'repair_limit'` instead of attempting another repair.

The driver emits the repair descriptor in the `repair` field of the stage descriptor; the
orchestrator applies the write. The driver itself NEVER edits `workflow-plan.md`.

## `readPlanAllDone` Ledger-Last Fix

The section-boundary regex that locates `## Node Ledger` uses `$(?![\s\S])` (a zero-width
assertion that matches end-of-string in JS) rather than `\Z` (a Perl-ism not supported in the
JS `RegExp` engine). Without this fix, when `## Node Ledger` is the FINAL section in
`workflow-plan.md` (i.e. not followed by another `##` heading), the lazy `([\s\S]*?)` terminator
would collapse to zero characters and no ledger rows would be parsed, causing `allDone` to
return `false` even when every row is `done`. The fix is at
`kaola-workflow-autopilot.js:189`: `/^## Node Ledger\s*\n([\s\S]*?)(?=\n## |$(?![\s\S]))/m`.

## Forge-Neutrality and Edition Strategy

`kaola-workflow-autopilot.js` carries a forge-neutral header comment block (mirroring
`run-chains.js` L40–42). The script body contains zero forge tokens. The four editions are:

- `scripts/kaola-workflow-autopilot.js` — canonical (claude edition)
- `plugins/kaola-workflow/scripts/kaola-workflow-autopilot.js` — byte-identical codex twin
- `plugins/kaola-workflow-gitlab/scripts/kaola-gitlab-workflow-autopilot.js` — prefix-rename, body-identical
- `plugins/kaola-workflow-gitea/scripts/kaola-gitea-workflow-autopilot.js` — prefix-rename, body-identical

Registration surface:

- `COMMON_SCRIPTS` + `RENAME_NORMALIZED_FAMILIES` in `validate-script-sync.js`
- `SUPPORT_SCRIPTS` in `kaola-workflow-install-manifest.js` (both byte-paired copies)
- `AUTO_COMMAND='/kaola-workflow-auto'` + `AUTO_SKILL='kaola-workflow-auto'` in
  `kaola-workflow-adaptive-schema.js` (×4 byte-identical)
- Route-reachability pin ×6 in `test-route-reachability.js` + 4 `validate-*-contracts.js`

All four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains must be green, run
sequentially, before finalization (#307). A green claude chain alone is insufficient (#400 ×6
command/SKILL propagation contract).

## Consequences

**A single `next` call** is the orchestrator's complete decision oracle for the autopilot loop:
it reads the digest, inspects on-disk receipts, and emits either the next action or a typed
terminal stop — with no forge calls and no agent dispatch in the critical path.

**The digest is the recovery primitive.** A driver process killed at any point restarts from the
last committed digest line. Digest corruption (non-JSON last line) is safe: `next` cold-starts
from scout.

**What is now closed from D-420-01/D-420-02:** D1 (pure receipt-reader enforced by zero forge
tokens), D2 (stateless `next`), D3 (digest-only crash-resume), D4 (one-bundle-per-invocation),
D5 (structural stop derivation from verified field names). The repair consent boundary
(mechanical-class = `{add_to_write_set, write_set_swap}`) is the binding implementation of the
D-420-02 taxonomy.

**Follow-up noted (out of this ADR's scope):** `KAOLA_AUTOPILOT_REPAIR` is not yet documented in
`.env.example` — tracked as a follow-up; the present ADR fixes the implementation contract only.
