# D-580-01. Pin shared engine `workflow-state.md` fields to `SHARED_STATE_FIELDS` and enforce behavioral parity across all active-folders ports

Date: 2026-06-30
Status: Accepted
Issue: #580
Related: D-579-01 (prior fix that introduced the last three shared fields — `main_root`,
             `session_marker`, `claim_ts` — by authoring-time audit, exposing the class of
             silent forge-port miss that this decision closes off)

## Context

#579 added three new shared engine fields (`main_root`, `session_marker`, `claim_ts`) to
`workflow-state.md` and confirmed all four `active-folders` ports parse them. That alignment
was established by an authoring-time audit: a developer inspected each port and verified the
fields were present. No machine contract prevented a future shared field from silently
missing a forge port.

The structural gap is a class problem, not a one-off:

1. **Chain coverage stops at structure, not field round-trip.** The four-chain
   cross-edition obligation (#307) certifies edition registries, forbidden tokens, and
   static file content. It does NOT verify that a field written into `workflow-state.md` is
   parsed and surfaced by every edition's `active-folders` port — a new shared field that a
   port silently ignores produces no chain failure.

2. **Walkthrough behavioral coverage is too coarse.** The existing `cmdStatus` integration
   scenarios in the walkthrough suites assert the claim-record shape (project name, status,
   branch). They do not drive every shared field through the parse-and-surface pipeline with
   a distinguishable sentinel, so a missing field is invisible.

3. **No single source lists which fields are shared-engine.** A developer adding a new
   field has no machine signal of which ports must be updated, and no gate that fires when
   a port is missed.

## Decision

### Move 1 — Single source of truth: `SHARED_STATE_FIELDS`

Add an exported `SHARED_STATE_FIELDS` constant to `scripts/kaola-workflow-adaptive-schema.js`
— the byte-identical cross-edition drift anchor already enforced by `validate-script-sync.js`.
The constant is `Object.freeze([...])` containing exactly 13 shared engine field names:

```
issue_number, phase, issue_numbers, status, bundle_id, closure_policy,
next_command, branch, worktree_path, sink, main_root, session_marker, claim_ts
```

Because `kaola-workflow-adaptive-schema.js` is byte-identical across all four editions, the
declaration lives in one place and is visible to all four editions without manual porting or
per-edition divergence.

The list is the **strict shared intersection** — fields that every edition's `active-folders`
port MUST parse and surface. Per-edition fields (gitlab `mr_*` / `project_id`; gitea
`full_name` / `pr_*`) are deliberately NOT in `SHARED_STATE_FIELDS` and are not pinned by
this mechanism.

### Move 2 — Behavioral parity gate: `test-active-folders-field-parity.js`

Add `scripts/test-active-folders-field-parity.js` (root-only; not edition-ported). For each
of the four editions, the gate:

1. Requires that edition's `active-folders` module.
2. Writes a temp `workflow-state.md` into a `$TMPDIR` scratch root, populating every
   `SHARED_STATE_FIELDS` key with a distinct type-appropriate sentinel value. Sentinels are
   chosen to be distinguishable from parser defaults: for example, `sink` defaults to
   `merge` in the parser, so the sentinel is a non-default value; `status` defaults to
   `unknown`, so the sentinel is a concrete status string. This makes the gate non-vacuous —
   a field that the parser omits and falls through to its default produces a mismatch, not a
   silent pass.
3. Calls `readActiveFolders(tmpRoot, { excludeClosedIssues: false })`.
4. Asserts that every `SHARED_STATE_FIELDS` key appears on the returned lane item with the
   expected sentinel value.

The gate drives future additions forward: adding a new key to `SHARED_STATE_FIELDS`
immediately reds any edition that does not yet surface it. The gate does NOT automatically
re-run after removing a field from a port's parser — it asserts presence, not absence, so
removal detection comes from the test assertion failing for that key.

### Move 3 — Wire into all four chains

`scripts/test-active-folders-field-parity.js` is added to the
`test:kaola-workflow:{claude,codex,gitlab,gitea}` chains in `package.json`. A missing or
mis-parsing field reds the cross-edition gate in every chain that runs it, satisfying the
#307 cross-edition greenness obligation.

### No port edits required — prior audit confirmed full alignment

An authoring-time audit at the time of #580 confirmed all 13 shared fields are already
parsed in every edition's `active-folders` port. #579 aligned the last three (`main_root`,
`session_marker`, `claim_ts`). The gate is GREEN at HEAD from day one; its value is forward
enforcement, not retroactive repair.

## Consequences

- `SHARED_STATE_FIELDS` in `kaola-workflow-adaptive-schema.js` (byte-identical ×4) is the
  single authoritative list of shared engine `workflow-state.md` fields. Adding a new shared
  field means adding it to this one constant; the parity gate then surfaces any edition not
  yet surfacing it.
- The development protocol for a new shared field is: add to `SHARED_STATE_FIELDS` in
  `kaola-workflow-adaptive-schema.js`, run `validate-script-sync.js` to propagate
  byte-identical copies, update every edition's `active-folders` parser, and verify with
  `test-active-folders-field-parity.js`. The gate provides the signal; the developer still
  performs the port edit.
- Per-edition fields (`mr_*`, `project_id`, `full_name`, `pr_*`) remain outside
  `SHARED_STATE_FIELDS` and are tested only by each edition's own scenarios, not this gate.
- The gate participates in the #307 four-chain greenness obligation; a red gate blocks
  Finalization.
- The `active-folders` ports are byte-unchanged by this decision; no backward-compatibility
  concern arises.

## Alternatives considered

- **Authoring-time audit only (pre-#580 posture).** Each PR author manually verifies all
  four ports when adding a shared field. Rejected: the #579 experience shows this is
  error-prone even for careful authors; the failure mode is silent under all four chains.

- **Static needle assertions in the per-edition contract validators.** Add text-search
  needles to each `validate-kaola-workflow-{edition}-contracts.js` for every
  `SHARED_STATE_FIELDS` member. Rejected: contract validators check static text (field-name
  string presence), not end-to-end parsing behavior — a field name present in a comment or
  dead branch but not returned by `readActiveFolders` would pass the needle check silently.
  The behavioral gate is the correct test.

- **Shared walkthrough scenario.** Add a `cmdStatus` scenario to each edition's
  `simulate-*-walkthrough.js` that inspects output fields. Rejected as too coarse: `cmdStatus`
  exercises the whole pipeline including display formatting and JSON serialization; a
  sentinel-based `readActiveFolders` unit call is cheaper, more precise, and easier to extend
  when a new field is added.
