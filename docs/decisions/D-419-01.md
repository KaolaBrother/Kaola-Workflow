# D-419-01. One coordination kernel (serial = running-set max 1) + scheduler-default posture

Date: 2026-06-12
Status: Accepted

This record covers **Part 1** (the coordination kernel unification) and **Part 3** (the
scheduler-default posture) together, because Part 3's "scheduler is the default, serial is the
degraded mode" framing presupposes the Part 1 kernel model in which serial IS the scheduler with
`max_concurrent = 1`. Parts 2 and 4 are recorded in D-419-02.

## Context

The adaptive path carries three coordination surfaces that all live in
`scripts/kaola-workflow-adaptive-node.js` and `scripts/kaola-workflow-parallel-batch.js`:

1. **The serial loop** — `open-next` / `close-and-open-next`. The original and still-dominant
   path: exactly one `in_progress` ledger row at a time, baseline recorded by `commit-node --start`,
   fused-advance close. Under the serial fallback (no `running-set.json`, no active batch, ≤1
   `in_progress` row) every guard-prologue layer is vacuously-pass and the path is byte-identical to
   pre-#383 behavior.
2. **The running-set scheduler** — `open-ready` / `close-node` / `reconcile-running-set` (#377). A
   per-node event-driven fan-out that tracks individual concurrent nodes in `.cache/running-set.json`
   with a two-phase crash-safe write (`state:'opening'` → baselines+flips → `state:'open'`). Reads
   fan out up to `KAOLA_FANOUT_CAP_READONLY` (default 8); write nodes today open ALONE
   (`write_node_exclusive`).
3. **The batch scheduler** — `parallel-batch.js`, which treats a whole frontier as a unit
   (`opening → open → sealed → joined`). Read-only fan-out only since ADR 0008 excised the write-role
   member-worktree path; write-role frontiers serial-degrade unconditionally.

The `readCoordinationState` helper (`adaptive-node.js` L1947/L1965) ALREADY treats "serial" as a
derived, degenerate predicate:

```
serialLive = inProgressIds.length === 1 && !runningSetLive && !batchLive
             && !runningSetOpening && !batchOpening
```

So the kernel concept — "serial is just the scheduler with concurrency 1" — is already half-built in
the runtime, but it is not NAMED, not documented, and not exploited. The planner, the six plan-run
surfaces, and the prose all describe the executor as "dispatching one frontier unit at a time," which
under-sells the scheduler that already exists and biases plan authors toward long serial chains.

Two pressures make naming the kernel worthwhile now:

- **Part 2 / Part 4 need a coherent "max > 1" arm to layer on.** Disjoint-write parallelism and
  speculative gate overlap both require the running-set scheduler to be the active surface and both
  need to reason about "the scheduler with N > 1." Without a settled kernel model, each would have to
  re-derive the serial-vs-scheduler relationship independently.
- **The "one frontier unit at a time" prose is a posture regression.** The runtime supports
  background dispatch + rolling top-up + critical-path-first ordering today, but the documented
  default posture is serial. Plan authors serialize independent work for ordering simplicity because
  nothing tells them that every serialized independent node adds its full duration to the makespan.

The risk is over-reaching: a naive "collapse `open-next` into `open-ready --max 1`" rewrite would
threaten the byte-identical serial fallback, which is the hard invariant of the whole adaptive path.
The serial path deliberately has a DIFFERENT guard matrix (`open-next`: `excl:['scheduler','batch']`,
no integrity layer because `orient` already ran `--resume-check`; `open-ready`: `integrity:true,
excl:['serial','batch']`) and a DIFFERENT on-disk footprint (it writes NO `running-set.json`).

## Decision

### Part 1 — Unify the model by SUBSUMPTION, not deletion

**Decision (P1):** Unify the serial loop and the running-set scheduler conceptually onto ONE kernel
where **serial = running-set with `max_concurrent = 1`** — but do so by SUBSUMPTION, not deletion.
`open-next` / `close-and-open-next` are RETAINED as the `max=1` fast-path aliases; they are NOT
deprecated and NOT rewritten into thin wrappers in this issue. The unification is a SEMANTIC contract
(one invariant defining "serial") plus a thin shared-derivation refactor — never a behavioral merge
that would risk the byte-identical fallback. We unify the MODEL and leave the two code paths as the
kernel's `max=1` and `max=N` entry points.

**The `max_concurrent` field.** Add an OPTIONAL `max_concurrent` integer to `running-set.json`:
`{ state, max_concurrent?, nodes:[...], updatedAt }`. It is set at OPEN time, NOT freeze time —
concurrency is a runtime resource limit (the same reasoning that keeps `KAOLA_FANOUT_CAP_READONLY` a
runtime cap, not a validity cap), so it must NOT enter the frozen plan body or `plan_hash`. Absence is
read as `max_concurrent = 1` (the legacy `open-next` default), never as "unbounded" — fail-closed.
`open-ready` already resolves the effective cap from `--max` and `KAOLA_FANOUT_CAP_READONLY` at open
time; `max_concurrent` simply PERSISTS that resolved value so a `reconcile-running-set` roll-forward
re-uses it.

**Part 1 invariants:**

- **[INV-1]** Serial mode ⟺ `max_concurrent = 1`, enforced by the kernel. The kernel's "serial"
  predicate is the EXISTING `serialLive` derivation re-stated as a cap statement: a run is in serial
  mode at instant T iff at most one node is `in_progress` AND that single node was opened under a
  `max=1` schedule. The on-disk witness: either there is no `running-set.json` (the legacy `open-next`
  footprint) OR the running-set carries `max_concurrent: 1`. No `running-set.json` IS the `max=1`
  witness for the legacy path; the kernel does NOT require the file to exist to be "in serial mode."
- **[INV-2] (byte-identity, the HARD invariant):** With `KAOLA_LANE_CONTAINMENT` off + no
  running-set + no active-batch + ≤1 `in_progress` row, every guard layer is vacuously-pass and the
  serial path is byte-identical to pre-#383 behavior. Any kernel refactor that makes `open-next`
  start writing a `running-set.json` VIOLATES [INV-2] and is rejected. `open-next` MUST NOT begin
  writing a `running-set.json`.
- **[INV-3]** `max_concurrent` is advisory metadata for crash-resume continuity; the ENFORCED cap at
  each `open-ready` call remains `min(KAOLA_FANOUT_CAP_READONLY default 8, --max)`. The two must agree
  (open-ready re-resolves and overwrites on each open). The persisted `max_concurrent` is the WITNESS,
  not the source of truth — a tampered manifest cannot raise concurrency beyond the env-resolved cap.
- **[INV-4]** The four ledger states are invariant: `pending`, `in_progress`, `complete`, `n/a`. No
  new ledger state is introduced by Part 1 (or any other part — see D-419-02 Part 4).
- **[INV-5]** The running-set `opening` crash state is invariant. The two-phase write (Phase 1
  `state:'opening'` with the full intended node set → Phase 2 baselines + ledger flips → Phase 3
  promote to `state:'open'`) and its `reconcile-running-set` roll-forward/back must survive
  byte-for-byte. Adding `max_concurrent` must NOT alter the three-phase ordering or the reconcile
  semantics.
- **[INV-6]** The serial fallback for write-role nodes is invariant (a write node opens ALONE). Part
  1 does not change this; Part 2 (D-419-02) is the ONLY part that conditionally lifts it.
- **[INV-7]** The `serialLive ⟺ crossCheckStatus single_in_progress` cross-consistency invariant (the
  #293 invariant, tested explicitly) survives unchanged. Part 1 must not perturb the `serialLive`
  derivation in `readCoordinationState`.

**Fate of `open-next` / `close-and-open-next`.** RETAINED as the `max=1` serial aliases — the
kernel's degenerate entry points. What changes is documentation/model unification only (the doc
records that serial = kernel `max=1`, and that `open-next` / `open-ready --max 1` are the two surfaces
of one kernel) plus an optional, low-risk shared longest-path-to-sink derivation.

**Guard prologue — keep three distinct refusal arms.** DO NOT merge `serial_node_live`,
`scheduler_active`, and `batch_active` into fewer arms. The emit-envelope contract (#406) requires
callers to classify failures STRUCTURALLY by a stable `reason` code, never by string-match.
`serial_node_live` and `scheduler_active` carry DIFFERENT repair pointers (close the live serial node
via `close-and-open-next` vs. `close-node` / `reconcile-running-set`); collapsing them would degrade
the repair guidance and break any downstream `reason`-matching test. The `serialLive` PREDICATE
remains a derived view in the kernel model, but the REFUSAL taxonomy stays three-armed.

**Part 1 open questions (recorded, NOT resolved here):**

- **OQ-P1-a:** Should `open-next` adopt longest-path-to-sink pick order to match `open-ready --max 1`?
  Today `open-next` picks the first pending node in document order; `open-ready` sorts `readyPending`
  by `longestPathToSink`. Aligning them is a BEHAVIOR change (a different node could open first) that
  adversarial fixtures may pin, so it is DEFERRED and flagged as a known divergence between the two
  `max=1` surfaces.
- **OQ-P1-b:** The full `open-next` → `open-ready --max 1` wrapper collapse is OUT OF SCOPE for v3. It
  is recorded as a future consolidation contingent on re-proving [INV-2] under the four-chain gate.

### Part 3 — Scheduler-default posture

**Decision (P3):** Make the running-set scheduler (background dispatch + rolling top-up) the
DOCUMENTED default posture across the six plan-run surfaces. Serial (`max=1`) remains the documented
DEGRADED mode. This is a PROSE + planner-rubric change — the runtime already supports both posture; we
are documenting the capability the runtime already has, not adding behavior.

**Six-surface frontmatter / body prose change.** The current frontmatter ("...via a running-set
scheduler; each frontier unit dispatched when its dependencies complete. Resume-safe.") becomes prose
that names concurrent dispatch up to the fan-out cap, critical-path-first ordering, and serial as the
degraded fallback for write nodes / lane-containment-off. The body's "dispatching one frontier unit at
a time and checkpointing between calls" becomes "dispatching the ready frontier concurrently up to the
fan-out cap, topping up as nodes close, and checkpointing between calls; serial when a write node is
live or lane containment is off."

**Planner rubric line (rewards overlap; never instructs authoring `parallel_safe`):**

> "Prefer a WIDE ready frontier over a long serial chain when nodes are independent: author parallel
> read-only analysis/review nodes (they fan out to the read cap) and parallel write nodes with
> DISJOINT declared write sets (they co-schedule under lane containment). The scheduler opens highest
> `longest-path-to-sink` nodes first (critical-path-first list scheduling), so place the longest
> dependency chain on the critical path and let short independent branches overlap it. Do NOT
> serialize independent work behind a single chain merely for ordering simplicity — every serialized
> independent node adds its full duration to the makespan; every overlapped node hides behind the
> critical path for free."

Concrete heuristic for the rubric: if two write nodes touch DISJOINT files and neither depends on the
other, do not add a dep edge between them — leave them as an antichain so the validator stamps them
`parallel_safe` (D-419-02 Part 2) and the scheduler overlaps them. Only serialize (add a dep) when
their write sets overlap.

**Part 3 invariants:**

- **[INV-15]** Six-surface propagation is mandatory. Per CLAUDE.md Validation Policy and #400 the
  prose must reach the 3 Claude commands + the 3 Codex SKILL packs (including the two forge-codex
  packs — the historic dead zone). The route-reachability contract (`scripts/test-route-reachability.js`
  + all four `validate-*-contracts.js`) machine-enforces it. A change reaching only 4 of 6 (the
  CHANGELOG "×4" wording is the symptom) is a propagation gap.
- **[INV-16]** Cross-edition: all four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}`
  chains must be green, run SEQUENTIALLY, before finalization. A green claude chain alone is
  insufficient evidence (`npm test` short-circuits on the first `&&` failure).
- **[INV-17]** The planner rubric REWARDS overlap but never INSTRUCTS the planner to author
  `parallel_safe` (that annotation is validator-derived — see D-419-02 [INV-8]). The planner authors
  TOPOLOGY (dep edges) and DISJOINT write sets; the validator derives concurrency-safety.
- **[INV-18]** "Scheduler-default" is a DOCUMENTATION/posture change; it does NOT change the OFF
  default of `KAOLA_LANE_CONTAINMENT`. Read fan-out is default-on (the scheduler already fans reads to
  cap 8); WRITE parallelism remains gated behind the flag. "Default scheduler posture" means "the
  scheduler is the documented primary executor and reads fan out by default," NOT "write parallelism
  is on by default." This distinction is stated explicitly so no reader assumes v3 turns on write
  concurrency by default.

**Serial remains the documented degraded mode in ALL of these cases (any one ⟹ serial):**

1. A write node is live AND lane containment is off (`KAOLA_LANE_CONTAINMENT != true`) — the
   permanent default; the [INV-6] write-alone fallback.
2. A write node is live whose write set OVERLAPS a candidate, even with containment ON (the
   `parallel_safe: false` case — overlapping writes never co-schedule).
3. `KAOLA_LANE_CONTAINMENT` off generally — no write parallelism at all; reads still fan out.
4. A `main-session-gate` node — excluded from fan-out; the main session cannot run concurrently with
   itself.
5. A frontier with only one ready node — trivially `max=1` for that step (degenerate, not a mode
   switch).
6. Integrity / halt / cross-surface refusal — any guard-prologue trip forces a STOP, not a silent
   serial degrade (a refusal is distinct from degraded-but-running).

**Part 3 open questions (recorded, NOT resolved here):**

- **OQ-P3-a:** Should the frontmatter name `KAOLA_LANE_CONTAINMENT` explicitly, or keep it abstract
  ("lane containment")? Recommendation: abstract in frontmatter (length), name it in the body + the
  env-var docs section.
- **OQ-P3-b:** Confirm the exact six surface file paths so the route-reachability contract stays green
  (3 commands + 3 SKILL packs).

## Consequences

**What ships in v3 (this record's scope):**

- The kernel MODEL is documented and named: serial = running-set `max_concurrent = 1`. The
  `max_concurrent` field is added to `running-set.json` as optional runtime metadata (absence ⟹ 1).
- The six plan-run surfaces and the planner rubric document the scheduler as the default posture and
  serial as the degraded mode, rewarding wide independent frontiers without changing any default flag.
- Read fan-out remains default-on; write parallelism remains OFF (gated behind P2's
  `KAOLA_LANE_CONTAINMENT`, see D-419-02).

**What is explicitly DEFERRED / OUT OF SCOPE:**

- The `open-next` → `open-ready --max 1` wrapper collapse (OQ-P1-b) — future, contingent on
  re-proving [INV-2].
- Aligning `open-next`'s single-pick order to `open-ready`'s critical-path order (OQ-P1-a) — a
  behavior change deferred to avoid perturbing adversarial fixtures.

**For day-to-day development:**

- A change to the six plan-run surfaces must reach ALL SIX or the route-reachability contract reds
  the chain ([INV-15]). The CHANGELOG "×4" wording is the canonical symptom of a propagation gap.
- Any cross-edition diff requires all four chains green, run sequentially, before finalization
  ([INV-16]).
- Any kernel refactor that makes the serial `open-next` path write a `running-set.json` reds the
  byte-identity tests ([INV-2]) and is rejected.

**For downstream issues:** the invariants that bind every Part-1/Part-3 implementation are [INV-1]
through [INV-7] (kernel model + crash-resume + write-alone fallback) and [INV-15] through [INV-18]
(six-surface propagation + four-chain green + validator-derived stamp + flag default unchanged).
