# D-639-01. Harden the metric-optimizer OPT freeze rules: require `metric_command`, refuse unresolvable `metric_paths`, and refuse duplicate/decoy `optimize(<id>)` blocks

Date: 2026-07-08
Status: Accepted
Issue: #639 (primary — three non-blocking freeze-rule gaps surfaced by #634's own n5-review/
       n6-adversary follow-up)
Related: #634/D-634-01 (the original `metric-optimizer` role and its OPT-1..6 freeze rules this
         decision hardens); #640 (residual — additional `metric_paths` shape completeness: bare
         existing-directory, backslash-separated, and absolute-path entries — filed as non-blocking
         defense-in-depth, not fixed here)

## Context

#634 introduced `parseOptimizeContracts` and the OPT-1..6 freeze rules governing the
`optimize(<node-id>)` metric-optimizer contract. A review/adversary pass on that work surfaced three
gaps, all judged non-shipping-blocking at the time (the OPT-5 change-gate reproduction and the D6
evidence-shape backstop bound the practical impact to one wasted dispatch) and filed as #639:

- **R1** — `metric_command` was parsed into the contract but no OPT rule ever read it. A block with
  no `metric_command` froze in-grammar and only failed later, at dispatch, when the role had nothing
  to run.
- **R2 / R5** — OPT-2's evaluation-isolation check compared `metric_paths` to `declared_write_set` by
  exact string equality only. A directory-shaped entry (`bench/`), a glob (`bench/*.js`), or a
  `../`-aliasing entry (`bench/../src/hot.js`) can cover or alias a write-set file while comparing
  string-distinct — defeating evaluation isolation without ever tripping OPT-2.
- **R3 / R7** — `parseOptimizeContracts`' `Map.set` is last-wins on a duplicate `optimize(<id>):`
  header for the same node, including a decoy header fenced inside `## Meta` (`classifier.sectionBody`
  returns fenced content verbatim, so the fence hides nothing from the parser). A second block
  silently clobbers the first — a tampered field (e.g. a raised `budget_iterations`) could freeze
  a plan with the tampered value in effect, while the pre-#639 OPT-1 check (which only asserted the
  contract Map had *an* entry for the node) saw nothing wrong.
- **R6** — OPT-3/OPT-4's numeric fields (`budget_iterations`, `budget_wallclock_minutes`,
  `metric_repeats`, `min_delta`) parse via a plain `Number()` call, which accepts hex (`0x14`) and
  exponent (`2e1`) notation. Investigated whether an unusual numeric notation needed its own freeze
  rule.

## Decision

1. **R3/R7 → OPT-1.** Add `optimizeHeaderCounts(content)`: counts raw `optimize(<id>):` headers per
   node id from the same fence-inclusive `## Meta` body `parseOptimizeContracts` reads (identical
   column-0-anchored header regex). OPT-1 now refuses when a `metric-optimizer` node has zero
   (unchanged behavior) **or more than one** header, naming the count in the refusal. Counting raw
   headers — rather than inspecting the already-parsed (and possibly already-clobbered) contract Map
   — is what lets OPT-1 catch the duplicate/decoy *before* the last-wins clobber can hide a tampered
   field. A duplicate/decoy header is a structural block-cardinality defect, the same family OPT-1
   already governs (exactly one block per node), so no new marker was needed.
2. **R1/R2/R5 → OPT-2.** OPT-2 is reframed from a pure "evaluation isolation" check to a broader
   "metric harness definition" check: `metric_command` must be non-empty (closes R1), and every
   `metric_paths` entry must be **exactly-resolvable to one concrete file** before the
   (now second-priority) exact-string write-set-intersection check runs. Exact-resolvability reuses
   `hasUnresolvableEntry` — the same directory-shape (`endsWith('/')`) and glob-metacharacter detector
   the write-set freeze-wall already applies for co-open disjointness — plus a `../`-path-segment
   split check. This closes R2 (directory/glob) and R5 (`../`-alias) under the existing OPT-2 marker:
   all three (R1/R2/R5) are facets of the same underlying invariant — "is the declared metric harness
   well-formed and provably isolated from the mutable scope?" — so folding them into OPT-2 keeps one
   marker per invariant rather than fragmenting it across several.
3. **R6 → documentation-only, no new rule.** `Number('0x10') === 16` and `Number('1e1') === 10`: OPT-3
   (`budget_iterations`, `budget_wallclock_minutes`) and OPT-4 (`metric_repeats`, `min_delta`) already
   validate the **converted** numeric value's range, not its source notation. Whatever form a numeric
   field is written in, an out-of-range or malformed value is still caught by the existing bound
   checks — there is no unbounded escape via hex/exponent/other `Number()`-parseable notation. Adding
   a notation-restricting rule would add validator surface without closing any actual defect, so none
   was added; this decision documents the investigation and its (negative) conclusion instead.

## Consequences

- A metric-optimizer contract missing `metric_command`, naming a directory-shaped/glob/`../`-aliasing
  `metric_paths` entry, or declaring a duplicate or fenced-decoy `optimize(<id>)` block now refuses at
  freeze with a typed `OPT-1`/`OPT-2` marker — instead of silently freezing in-grammar and failing
  later (R1: at dispatch, when the role has no command to run) or never (R2/R5/R3/R7: the
  ambiguous/tampered value would simply have taken effect).
- All of OPT-1..6 remain freeze-time-only (no `--resume-check` change) and remain `plan_hash`-covered
  (the `optimize(<id>)` block lives inside `## Meta`, which `computePlanHash` already normalizes in
  full).
- Regenerated across the codex byte mirror and both forge rename ports via `npm run sync:editions`
  (byte-/rename-identical to the canonical edit — no hand-porting).
- Tests: `simulate-workflow-walkthrough.js`'s `testMetricOptimizerContract` gained RED-first accept
  and typed-refuse fixtures for R1, R2, R5, R3, and R7, plus a direct `optimizeHeaderCounts` unit
  assertion (counting a planted duplicate header as 2). A pre-existing fully-valid optimize plan, and
  the existing OPT-1..OPT-6 accept/refuse fixtures, remain green (no regression). All four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains verified green.

## Non-goals

- Does not add a new `OPT-N` marker — every hardening folds into the existing `OPT-1` or `OPT-2`
  marker, since each closes a gap in an invariant that marker already governs.
- Does not extend `metric_paths` shape-checking beyond directory-shape / glob / `../`-alias. A bare
  existing directory with no trailing slash, a backslash-separated path, and an absolute path all
  remain unrefused today — filed as the non-blocking `#640` residual (defense-in-depth only: the
  write-set freeze-wall already refuses an absolute entry in `declared_write_set` itself, and any
  physical aliasing that slips past is still backstopped by the OPT-5 change-gate metric
  reproduction).
- Does not change `parseOptimizeContracts`' own last-wins parsing behavior — the clobber on a
  duplicate header still happens at parse time; `optimizeHeaderCounts` instead refuses the plan at
  freeze *before* the clobbered value can be trusted for anything.

## Alternatives considered

- **Teach `parseOptimizeContracts` to reject a duplicate header at parse time**, instead of adding a
  separate counting function consulted by OPT-1. Rejected to keep the parser itself permissive/
  first-class-data and push all freeze-time judgment into the OPT-N rule layer — consistent with
  every other OPT rule, which reads an already-parsed contract and layers a judgment on top, rather
  than teaching the parser itself to refuse.
