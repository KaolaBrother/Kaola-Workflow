# D-584-01. Codex install config readiness is agent-guided, not silently mutated

**Date:** 2026-07-01
**Status:** Accepted
**Issue:** #584
**Related:** #581, #582

## Context

Kaola-Workflow's Codex install path writes role profiles, hooks, manifests, and a managed
`[agents.*]` block. It must also prove that Codex is configured for the workflow's intended
subagent dispatch behavior.

The unsafe shortcut would be to make normal install scripts rewrite the user's global
`~/.codex/config.toml` feature settings automatically. That would conflate a profile install with
user-owned machine policy and could churn unrelated Codex configuration.

## Decision

- Normal profile install remains focused on Kaola-owned files and managed blocks.
- Codex config readiness is handled by an agent-guided audit before claiming dispatch readiness.
- The audit recognizes V2 when the active `[features]` config clearly enables it through boolean,
  inline-object, or table-form syntax.
- Warning suppression is independent from feature availability.
- Missing required feature or agent-limit settings are reported with a minimal config delta.
- The delta is applied only when the user has asked the agent to configure the machine or explicitly
  approves the change.
- Runtime effort reliability is still proved separately by inspecting child-session metadata; config
  text is not a substitute for the effort proof.

## Consequences

- Existing correct Codex configs are accepted without churn.
- Stale text matching no longer misreports object/table V2 config as V1.
- Install guidance preserves user agency over global Codex config while still failing closed when
  effort-safe dispatch cannot be proved.
