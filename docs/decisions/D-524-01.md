# D-524-01. Issue-scout ranks by roadmap priority first — cohesion and actionability are within-tier tiebreaks only

Date: 2026-06-18
Status: Accepted
Issue: #524
Related: `agents/issue-scout.md` (the profile this decision governs),
         `plugins/kaola-workflow/agents/issue-scout.toml` ×3 forge editions,
         `scripts/kaola-workflow-roadmap.js` (emits `next_step:` per-issue ordering + `### Project rules`),
         #44 (Agent Owns Reasoning — the scout's pick is still an agent decision; priority is the ordering
              input the roadmap makes available, not a hidden script selection),
         D-512-01 / D-522-01 (recent ADR format reference).

## Context

### The cohesion-and-actionability-only scout

The `issue-scout` agent profile was designed to help an agent select the best next GitHub issue to
work on. Prior to this decision, its ranking model scored candidates on two axes:

1. **Scope cohesion** — how tightly the candidate clusters with already-completed or in-progress
   work, so that context stays warm and the edit surface stays narrow.
2. **Actionability** — whether the issue has clear, implementable acceptance criteria, no hard
   blockers, and a verifiable done-state.

Neither axis consulted the **roadmap priority or drive-order**: the per-issue `next_step:` ordering
emitted by `kaola-workflow-roadmap.js`, the master-epic `Next Step` column, and the `### Project
rules` guardrails embedded in the roadmap mirror (`kaola-workflow/ROADMAP.md`) were not inputs to
the scout's ranking.

### The live mis-ranking (vrpai-cli)

On the `vrpai-cli` repository the scout recommended `primary_issue: 82` (an adjacent environment /
dependency issue) and then `652` (an SDK configuration issue) instead of the `#488 / #502 / #561`
epic frontier that the roadmap explicitly drove as the next required work. The scout's reasoning in
both cases was: these adjacent issues are "the closest actionable proxy" because the frontier issues
had unresolved dependencies or uncertain scope.

This is the structural failure the prior model produces under any project where:

- the highest-priority cluster is a large / partially-blocked epic, AND
- smaller, adjacent, immediately-actionable issues exist nearby in cohesion-space.

The scout's cohesion + actionability score will always favor the adjacent actionable item. The
roadmap's drive-order is never consulted, so the mis-pick is silent: there is no "I am ignoring the
frontier" basis statement in the output.

## Decision

### Priority/drive-order is the primary (hard) ranking tier

The `issue-scout` profiles (`agents/issue-scout.md` + the three `.toml` byte-identical forge ports)
now impose a **three-tier strict ranking**:

1. **Tier 1 — roadmap priority / drive-order (hard tier):** The scout extracts two priority signals
   from the roadmap mirror:
   - The `### Project rules` guardrails (e.g. "X must not preempt the correctness frontier Y")
     treated as **hard constraints**, not suggestions. A candidate that violates a guardrail is
     ineligible regardless of its cohesion or actionability score.
   - The per-issue `next_step:` ordering emitted by `kaola-workflow-roadmap.js` and the master-epic
     `Next Step` column — the explicit drive-order the roadmap owner has declared.
   Only within the top-priority frontier cluster does the scout proceed to Tier 2.

2. **Tier 2 — scope cohesion (within-tier tiebreak):** Among candidates at the same priority
   level, cohesion still guides the pick toward the cluster with the warmest context.

3. **Tier 3 — actionability (within-tier tiebreak):** Among candidates with similar priority and
   cohesion, actionability (clear ACs, no hard blockers, verifiable done-state) breaks the final
   tie.

Actionability NEVER promotes a lower-priority cluster over a higher-priority one. A lower-priority
but immediately-actionable issue loses to a higher-priority but partially-blocked frontier issue.

### Explicit-frontier-blocked over silent-proxy substitution

When the top-priority frontier issue is genuinely blocked or unverifiable, the scout now:

1. States so EXPLICITLY — "frontier #N is blocked because `<reason>`" — as a basis statement in
   its output.
2. Only THEN falls to the next-priority actionable item, naming it as the next-best pick given the
   frontier's state.

The prior silent substitution ("the closest actionable proxy") is retired. Every pick that is not
the frontier must name the frontier and the reason it was not chosen.

### New required output field: `priority_basis`

A new required field `priority_basis` is added to the scout's output schema, consistent across the
`.md` and `.toml` surfaces:

```
priority_basis:
  frontier:            # the highest-priority issue(s) the roadmap drives
  pick_vs_frontier:    # "is_frontier" | "frontier_blocked_reason: <...>" | "next_after_blocked_frontier: <...>"
  guardrails_honored:  # list of Project-rules guardrails checked; "none" if none apply
```

This field makes the priority reconciliation auditable: a reviewer can confirm the pick is the
frontier or see exactly why the scout fell to a lower-priority candidate.

### Locus: scout profile only — auto surfaces untouched

The fix is applied exclusively to the `issue-scout` agent profile prose. The automated routing
surfaces (`commands/`, `plugins/*/skills/`) do not describe scout ranking and are therefore not
in scope for this change. The `kaola-workflow-roadmap.js` generator is unchanged — it already
emits the `next_step:` ordering and `### Project rules` that the scout now reads.

This is a pure agent-instruction-prose change; no code path is altered.

## Alternatives considered

### Orchestrator-applied priority filter (post-scout)

An alternative locus: keep the scout's cohesion+actionability ranking unchanged and apply a
priority filter in the orchestrator *after* the scout returns its pick — refusing a mis-ranked
pick and re-dispatching until a priority-consistent pick is returned.

Rejected: the orchestrator filter is a retry loop that wastes tokens and inference time re-running
the scout on the same candidate set. The root cause is the scout's objective function, not the
orchestrator's acceptance logic. Fixing the scout's ranking is the cheapest sufficient fix that
eliminates the retry entirely. (Design precedence #3: cheapest sufficient mechanism.)

### Add priority as a fourth scored dimension (weighted sum)

An alternative ranking model: add a `priority_score` to the existing cohesion+actionability sum,
with a high weight but not a hard constraint.

Rejected: a weighted sum can still produce a priority violation — a sufficiently high
cohesion+actionability score for a low-priority candidate can overcome a moderate priority penalty.
The vrpai-cli failure is precisely a case where the cohesion+actionability advantage of the
adjacent issues was large enough to dominate. A hard-tier model is the only form that provides the
non-negotiable accuracy guarantee (design precedence #1).

### Modify `kaola-workflow-roadmap.js` to surface explicit "do not pick" signals

A third alternative: extend the roadmap generator to emit an explicit ranked exclusion list ("do
not pick X until frontier Y is resolved"), which the scout would honor as a blocklist.

Deferred rather than rejected: the `### Project rules` mechanism already encodes this information
in human-readable prose guardrails, and teaching the scout to extract and honor those guardrails
achieves the same effect without a generator schema change. If the guardrails prove ambiguous in
practice, a machine-readable exclusion list is the natural follow-up, but it is premature here.

## Consequences

- A scout-driven pick will now always be the roadmap frontier issue (or, when blocked, will name
  the frontier and the reason explicitly). The vrpai-cli class of mis-ranking is structurally
  eliminated.
- Every pick output now contains a `priority_basis` field, making the roadmap-reconciliation
  auditable by any reviewer or follow-up agent.
- `### Project rules` guardrails in the roadmap mirror are now machine-honored by the scout as
  hard constraints — documenting a guardrail is sufficient to have it enforced; no separate
  scout configuration is required.
- The cohesion and actionability axes are retained as within-tier tiebreaks; their explanatory
  value (why this issue over its priority-peer neighbors) is preserved.
- No code path changes. The only blast radius is the four scout profile surfaces (`.md` + ×3
  `.toml`). All four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains apply as
  the cross-edition parity gate (#307).
