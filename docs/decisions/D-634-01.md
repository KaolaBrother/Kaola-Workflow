# D-634-01. Add `metric-optimizer` as the one closed-library extension for optimize-shaped work

Date: 2026-07-08
Status: Accepted
Issue: #634 (primary)
Related: #486 (question-shaped work — the sibling "shape" this design's write-side counterpart
         serves; not yet shipped); #472/#463/#439 (the decomposition/parallelism precedence this
         design defers to — unaffected, since the ratchet loop runs inside one dispatch, not a new
         scheduler mechanism); #639 (OPT freeze-rule hardening follow-ups surfaced building this)

## Context

Every node the adaptive planner could author was task-shaped: an acceptance criterion known at
freeze, one attempt (plus bounded repair), judged "did you do the thing?". A second shape has no
honest AC at freeze: work with a *direction* but no *destination* — "make the walkthrough faster,"
"shrink the bundle," "cut P99 latency," "reduce flakiness" — where nobody knows what's achievable
until the ratchet has actually run. Before this decision the planner had two dishonest options:
invent a fake threshold (too easy or impossible, either way laundering a guess into the frozen DAG),
or decline to serve the issue on the adaptive path at all. This is the write-side complement of
#486's still-unshipped question-shaped design ("target unknown" vs. #486's "answer unknown").

The design draws on the safe unattended-improvement-loop pattern demonstrated by
[karpathy/autoresearch](https://github.com/karpathy/autoresearch) (propose → fixed-budget run →
single frozen metric → commit-if-better / `git reset`-if-not → repeat), adapted to this repo's
existing invariants rather than adopted wholesale — see "Alternatives considered" and "Non-goals"
below for what was deliberately not carried over.

## Decision

**Add `metric-optimizer` as the one closed-library role extension for optimize-shaped work,
composing with the existing grammar rather than adding a new shape or scheduler mechanism.**

- **Role membership, not a new shape.** `metric-optimizer` joins `CANONICAL_ROLES`, `WRITE_ROLES`,
  and `IMPLEMENT_ROLES` in `kaola-workflow-plan-validator.js`, at the `sonnet` model tier
  (`resolve-agent-model.js`). `IMPLEMENT_ROLES` membership alone makes an optimize node
  `producesCode()` true ⇒ **G1 (`code-reviewer`) and G3 (`main-session-gate`, when present)
  post-dominance are inherited for free** — zero gate-plumbing change. The node itself is an
  ordinary `sequence` member; no new node shape was added.
- **The contract lives in `## Meta`, keyed by node id — not new node columns.** An
  `optimize(<node-id>):` block (`parseOptimizeContracts`) carries `metric_command`, `metric_paths`,
  `direction` (`min`/`max`), `budget_iterations` (1..`OPTIMIZE_ITER_CAP`=50), optional
  `budget_wallclock_minutes` (≤`OPTIMIZE_WALLCLOCK_CAP`=120, overriding the dispatch card's
  tier-derived wait budget), optional `regression_gate` (falling back to the Meta
  `validation_command`), `metric_repeats` (default 1, median-of-K), `min_delta` (default 0), and
  `patience`. `computePlanHash` already normalizes the whole `## Meta` body, so the contract is
  `plan_hash`-covered with no hash-computation change. Node columns are parsed twice in parity
  today (the validator's `parseNodes` and the hand-ported, per-edition `classifier.readPlanNodes`);
  a Meta block keeps this entire grammar change inside the validator, a GENERATED aggregator
  regenerated across editions, never touching the hand-ported data-layer script.
- **Six fail-closed freeze rules (OPT-1..6).** OPT-1: exactly one `optimize()` block per
  `metric-optimizer` node, and every block keys a real one. OPT-2 (evaluation isolation):
  `metric_paths` non-empty and disjoint from the node's `declared_write_set` — the measurement can
  never sit inside the mutable scope (a runtime write to it already trips the existing barrier).
  OPT-3: the budget caps above. OPT-4: `direction`/`metric_repeats`/`min_delta` well-formed. **OPT-5:
  a change-gate `adversarial-verifier` must post-dominate every optimize node**, reusing
  `gateUncovered` exactly as G1/G2 do — the measured metric claim is the node's entire deliverable,
  so a plan can never finalize with it unverified. OPT-6: a `regression_gate` must resolve from
  somewhere (explicit or inherited); a metric-only ratchet with no regression gate is refused as
  Goodhart bait.
- **The evidence contract (D6) closes a hollow-stub gap this build's own adversary found.**
  `ROLE_TOKEN_REGISTRY['metric-optimizer']` = `evidence-binding`, `metric_baseline`, `metric_final`,
  `iterations_used`, `regression-green`. As first implemented, `checkEvidenceShape` had no
  `metric-optimizer` branch and fell through to the generic "file present and non-empty" check — a
  metric-optimizer node could close COMPLETE on a fully hollow seeded stub (every token key present,
  every value empty, zero ratchet log). This run's own change-gate adversary reproduced it live and
  refused; the fix — a dedicated branch requiring each of the four non-binding tokens carry a
  non-empty column-0 value — landed in the same run, RED-proven and re-reviewed.
- **The ratchet protocol is role-profile prose, not script logic** (`agents/metric-optimizer.md` +
  the 3 forge `.toml` twins, forge-neutral, provenance-free): per iteration, propose one change
  inside the declared write set → apply → run the regression gate (red ⇒ automatic reject) →
  measure the metric (median of `metric_repeats`) → accept (commit
  `kw-opt(<node-id>) iter <k>: <old> -> <new>`, new baseline) or reject (a **scoped**
  `git restore --source=HEAD -- <write-set>`; `git reset --hard` is **forbidden** — unscoped to the
  write set, it could destroy uncommitted workflow artifacts in a shared worktree). Stop at
  `budget_iterations`, `patience` consecutive rejects, or the wallclock budget, whichever fires
  first; log every iteration, accepted and rejected. Never ask a judgment question inside the loop —
  anything that needs one stops the loop and routes through the existing `write-halt` valve; no new
  escalation mechanism.

## Consequences

- A direction-not-destination issue can now be served on the adaptive path honestly — no invented
  threshold, no declining the issue — while every existing correctness invariant (G1/G3
  post-dominance, `plan_hash` freeze coverage, the per-node barrier, the change-gate
  `--verdict-check`) applies to it unchanged.
- The barrier and lane-group machinery need zero changes: the per-node barrier is
  commit-count-indifferent (snapshot-to-snapshot), so N intermediate ratchet commits/reverts inside
  one node produce the same net diff a single commit would, and an optimize node co-opens as an
  ordinary lane-group member on a disjoint write set exactly like any other writer.
- Three OPT freeze-rule hardening opportunities surfaced by this run's own `n5-review` gate, plus two
  more this run's own adversary found (a directory-shaped `metric_paths` disjointness escape, and a
  fenced-decoy `optimize()` block silently parsing as real), are deferred as non-blocking — each has
  a downstream backstop (the change-gate verifier reproduces the metric, `plan_hash` covers both
  blocks of a duplicate, the evidence-shape gate still requires a non-empty final metric) — and are
  filed as **#639**.
- Cross-edition (#307): this diff touches `CANONICAL_ROLES`/`WRITE_ROLES`/`IMPLEMENT_ROLES`,
  `ROLE_TOKEN_REGISTRY`, the OPT-1..6 freeze rules, the dispatch-card threading, the new agent
  profile + its registration surface, and the plan-run routing prose; all four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains were run.

## Non-goals / accepted residual

- **Multi-metric / Pareto contracts, an agent-proposed metric with human confirmation, cross-run
  playbook accumulation, `patience` auto-tuning, and per-iteration script-side counters** are all
  deferred — the issue's own design lists them explicitly; none is needed for the single-metric,
  single-loop case this decision ships.
- **Optimize-role fan-out grammar** is not built: a lane-group co-open of two optimize nodes on
  disjoint scopes already works today with zero grammar change, so a dedicated fan-out shape would
  add machinery for a case the existing grammar already serves.
- **R1/R2/R3/R5/R7 OPT freeze-rule hardening** (see "Consequences" above) — filed as #639 rather
  than folded into this decision, since none can ship a wrong result (each has a downstream
  backstop) and the OPT-1..6 rules ship exactly as the issue's own design enumerated them.

## Alternatives considered

- **Reuse the `implementer` role.** Rejected: the evidence contract is role-keyed
  (`ROLE_TOKEN_REGISTRY`) and the two contracts are irreconcilable — `implementer` proves
  "change-type check passed once"; this role proves "baseline honest, N iterations budget-bounded,
  final metric reproducible." Forcing one registry row to mean both would weaken the contract for
  every existing `implementer` node.
- **Re-dispatch via `loop(<cap>)`.** Rejected: `LOOP_CAP` (5) is an order of magnitude too small for
  the ~20–40 cheap iterations this shape's working range needs, and per-iteration subagent
  re-dispatch pays full spin-up cost per attempt for no accuracy gain — the loop belongs inside one
  dispatch, not the plan's re-dispatch grammar.
- **Adopt autoresearch's "never ask, never stop" as a global rule.** Rejected: kept only *inside*
  the bounded loop. Autoresearch affords it because its blast radius is one file with instant
  rollback; this repo's blast radius is a real shared worktree, so the node-boundary consent-halt
  valve stays authoritative outside the loop.
- **Let the metric be agent-invented at runtime.** Rejected: an agent grading its own invented ruler
  is the textbook Goodhart failure mode. The metric is declared by the planner in the frozen
  contract; OPT-5's change-gate verifier reproduces it, never re-derives it.
