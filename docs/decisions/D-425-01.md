# D-425-01. ledger_header_invalid is a freeze wall, not an auto-repair

Date: 2026-06-12
Status: Accepted
Issue: #425
Related: `scripts/kaola-workflow-plan-validator.js` (`validatePlan`, `reconcileLedger`),
         `scripts/kaola-workflow-adaptive-node.js` (`spliceLedgerNode`, `readLedgerStatuses`),
         ADR 0005 (`docs/decisions/0005-plan-run-owns-node-lifecycle.md`),
         D-423-01 (the related test-fixture fix for the contractor guard)

## Context

The `## Node Ledger` section of `workflow-plan.md` carries a mandatory header row whose first column
must be named `id` (case-insensitive). The header identifies which column holds node identifiers
and which holds the ledger status. Before issue #425, `validatePlan()` accepted any `## Node Ledger`
header without inspecting its column names; a plan whose ledger opened with `| node | status |` or
`| node_id | status |` would pass `--freeze` and only fail at runtime — inside `open-next` — when
`spliceLedgerNode` or `readLedgerStatuses` attempted to look up a node by `id` and found no
matching column.

This produced a late, non-actionable error: the plan was already frozen, `plan_hash` was already
stamped, and the operator would encounter the failure mid-run (at `open-next`) rather than at
authoring time. The repair path for an in-flight frozen plan is complex (plan-repair via `--freeze`
re-stamping) and forces an interruption of a run that should never have started.

Two remediation postures were considered:

1. **Auto-repair at freeze:** detect a non-canonical header (`node`, `node_id`, `node-id`, etc.)
   during `validatePlan()` and silently rename it to `id` before stamping `plan_hash`.
2. **Freeze wall:** refuse `--freeze` (and the default `--json` validate) when a present
   `## Node Ledger` has a non-canonical header, with a typed error naming the columns found.
   Provide a separate `--repair` normalization path that the author can invoke explicitly before
   re-freezing.

## Decision

**`ledger_header_invalid` is a freeze wall, not an auto-repair.**

`validatePlan()` now refuses (typed error `ledger_header_invalid`) when it detects a `## Node
Ledger` section whose header row does not carry `id` as its first data column (case-insensitive
match; `| id | status |` is canonical). The refusal names the columns that WERE found, so the
author receives a diagnosis, not just a refusal.

**Why not auto-repair:**

**1. In-flight plans must not be bricked retroactively.**

`--resume-check` does NOT apply the freeze-wall rules — it re-validates only closed-library
membership, structural grammar, and `plan_hash` integrity, not the full gate rubric
(`docs/api.md` § Adaptive Plan Validation, `--resume-check` mode). This is the deliberate
contract: a plan that was legally frozen by a pre-#425 validator resumes without interruption even
though it would fail the new freeze wall. An auto-repair in `validatePlan()` would collide with
this invariant: if the validator auto-repaired the header and then computed `plan_hash`, the hash
would be computed over the REPAIRED header — but an already-running plan stamped its hash over the
ORIGINAL (non-canonical) header. Any `--resume-check` call after an accidental re-run of `--freeze`
on the live plan would then produce a `plan_hash_mismatch`, bricking an in-flight run.

**2. The freeze wall catches newly authored plans before they cause open-next failures.**

A plan submitted to `--freeze` for the first time is not yet in-flight. The author has full control
over the header text and can fix it in seconds. The freeze wall is maximally actionable at exactly
this moment. The alternative — freezing a bad header and failing at `open-next` — is a much worse
user experience: the error is deferred, context has switched to the execution phase, and the repair
requires understanding the plan-repair-via-`--freeze` pattern.

**3. The `--repair` path normalizes without re-stamping the hash of `## Nodes`.**

`reconcileLedger()` (the `--repair` mode helper, also used for adding missing ledger rows) handles
the header normalization: `node`, `node_id`, and `node-id` are recognized as non-canonical aliases
and renamed to `id`. Because `plan_hash` covers only `## Meta` + `## Nodes` (the mutable `## Node
Ledger` is excluded from the hash), normalizing the ledger header does not invalidate any
previously stamped hash. After `--repair`, the author re-runs `--freeze` to stamp the hash over the
now-canonical plan. The `--repair` output carries `header_normalized: true` so the caller knows the
normalization ran.

The `--repair` path is the ONLY path that normalizes; `--freeze` (without `--repair`) refuses on a
non-canonical header. This separation ensures the freeze is always the author's explicit act: the
validator cannot silently change the plan's structure on a bare `--freeze`.

**`node_not_in_ledger` diagnostic field (related, adaptive-node.js):**

When `open-next` (or `readLedgerStatuses`) cannot find an `id` column in the ledger, it now
surfaces a `diagnostic` field in the `node_not_in_ledger` refusal payload. The diagnostic includes
the columns actually found plus the requirement (`id` column required). This makes the mid-run
failure self-diagnosing — the operator is told exactly what is wrong and can apply `--repair` before
re-opening — even for a plan that was frozen by a pre-#425 validator (which was silent about a
non-canonical header). The freeze wall prevents NEW plans from reaching this state; the diagnostic
handles the legacy tail.

## Consequences

**For plan authors:**

- A `## Node Ledger` with `| node | status |` or `| node_id | status |` is now a typed freeze-time
  refusal. The error names the columns found. Fix by renaming the first column to `id`.
- Run `plan-validator.js <plan> --freeze --repair` to auto-normalize a non-canonical header, then
  re-run `--freeze` (without `--repair`) to stamp the hash.
- Plans already frozen by a pre-#425 validator are NOT affected — `--resume-check` does not apply
  the new freeze wall.

**For the runtime (adaptive-node.js):**

- `node_not_in_ledger` refusals now carry a `diagnostic` field when the ledger section is present
  but has a non-canonical header. The operator receives a self-diagnosing error instead of a bare
  refusal.

**For the test suite:**

- Walkthrough scenarios added for `ledger_header_invalid` (freeze refuses `| node |` and
  `| node_id |` headers) and the `--repair` normalization path (header normalized to `| id |`,
  `header_normalized: true` in output).

**For downstream freeze-wall additions:**

- The pattern established here — refuse at freeze, provide `--repair` for normalization, keep
  `--resume-check` unaffected — is the template for future freeze-wall additions that involve the
  `## Node Ledger` section.
