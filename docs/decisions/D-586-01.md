# D-586-01. Retire the standalone `parallel-batch` aggregator

Date: 2026-07-02
Status: Accepted
Issue: #586
Related: ADR-0008 (`docs/decisions/0008-excise-write-role-batch-isolation.md`, the prior
             excision precedent — retiring dead-but-costly machinery rather than fixing it);
             D-542-01 (default-on disjoint write co-open, the running-set scheduler capability
             `parallel-batch.js` never had); #364/#377 (the running-set scheduler that
             superseded the whole-frontier batch machine)

## Context

`kaola-workflow-parallel-batch.js` (canonical + the Codex byte-mirror + the GitLab/Gitea
forge ports, plus its `test-parallel-batch.js` suite) was a fourth adaptive-path aggregator
implementing a whole-frontier `active-batch.json` manifest lifecycle
(`opening → open → sealed → joined`) via `open-batch` / `top-up` / `seal-member` / `seal` /
`join` / `reconcile` / `status` subcommands.

**It was provably off the live executor path.** Nothing in the runtime composition chain
ever shelled it: `kaola-workflow-adaptive-node.js` — the script that owns the per-node
lifecycle for `/kaola-workflow-plan-run` — composes only `next-action.js`, `commit-node.js`,
and `plan-validator.js`; its remaining mentions of "parallel-batch" are design-lineage prose
comments (e.g. "Mirrors parallel-batch runReconcile"), never a `shell()`/`require()` call. The
plan-run skeleton's own Loop Skeleton (step 2, all six routing surfaces) already instructed
the operator to run `open-ready` — the running-set scheduler's own open transaction — on
`enterBatch: true`, not `open-batch`. The retired `<!-- CARD: frontier-batch -->` footnote
that pointed at the `parallel-batch.js` CLI as "batch MECHANICS" was reference material the
skeleton's own operative instructions never routed through — dead prose describing dead code.

**The running-set scheduler already owned the frontier path in full, including
default-on parallel writes.** `kaola-workflow-adaptive-node.js`'s `open-ready` / `close-node`
/ `reconcile-running-set` subcommands (issue #377, `.cache/running-set.json`) handle both
read-only fan-out and write-role co-open. Since D-542-01, a planner-proven-disjoint write
frontier co-opens as a lane group in isolated per-leg worktrees BY DEFAULT — a capability
`parallel-batch.js` never had. Its own write-role member-worktree isolation had already been
excised in ADR-0008 (#364): since that excision, `open-batch`/`top-up` unconditionally
serial-degraded any write-role frontier (`{degraded:true, reason:'cwd_unenforceable'}`), so at
retirement time the aggregator could do nothing for writes the single-node `open-next` path
could not already do, and nothing at all for the running-set scheduler's default-on parallel
writes. Its read-only fan-out capability was likewise fully duplicated by `open-ready`.

**It carried four real crash-safety and accuracy defects**, verified against the deleted
source (`git show`) before this retirement:

1. **No `'sealing'` transitional marker.** `runSeal` writes the per-member ledger/compliance
   rows to `workflow-plan.md` FIRST, then writes the manifest transition to `state: 'sealed'`
   in a SEPARATE, later write — with no intermediate crash-safe marker recording "the ledger
   write landed, the manifest write has not yet." The running-set scheduler's own `close-node`
   makes exactly this window safe via `reconcile-running-set`'s close-direction detection
   (`closedDropped`); `parallel-batch.js` had no equivalent.
2. **Double-applied compliance row on retry.** `sealOne` spliced a compliance row
   unconditionally, with no `complianceRowExists`-style idempotency guard (unlike
   `runCloseNode`, which explicitly skips a duplicate row on re-close). A crash between the
   ledger write and the manifest write in defect (1), followed by a `seal` retry, would
   re-process the same member and append a second compliance row for it.
3. **Non-atomic `reconcile --abort`.** The rollback path wrote the ledger rows back to
   `pending`, then shelled a SEPARATE `--drop-base` per member, then wrote or deleted the
   manifest — three independent, non-transactional writes. A crash between any two of them
   left the ledger, the per-node baselines, and the manifest mutually inconsistent, with no
   two-phase recovery (unlike `runReconcileRunningSet`'s single coherent repair pass).
4. **The operator card documented a CLI that never existed.** `docs/plan-run-cards/frontier-batch.md`
   (pre-retirement) instructed operators to call `open-batch --batch-id {batchId} --nodes
   n4,n5,n6 --plan ... --ledger ...` and `seal-member --evidence-file ...` — none of
   `--batch-id`, `--nodes`, `--evidence-file`, or `--ledger` were ever real flags on the actual
   `--project` / `--node-id` / `--max` / `--json` / `--abort` CLI. The card was aspirational
   from authorship, not merely stale.

Given these defects plus zero runtime consumers, fixing `parallel-batch.js` would have been
pure maintenance burden for a capability the running-set scheduler already provided more
safely and more completely. The issue author chose retirement, mirroring the ADR-0008
precedent of excising dead-but-costly machinery rather than repairing it in place.

## Decision

**Retire `kaola-workflow-parallel-batch.js` in full.** Specifically:

1. Delete the standalone aggregator — canonical (`scripts/kaola-workflow-parallel-batch.js`),
   the Codex byte-mirror (`plugins/kaola-workflow/scripts/kaola-workflow-parallel-batch.js`),
   and both forge ports (`plugins/kaola-workflow-{gitlab,gitea}/scripts/kaola-{gitlab,gitea}
   -workflow-parallel-batch.js`) — plus its dedicated test suite
   (`scripts/test-parallel-batch.js`).
2. Remove every registry/manifest reference: `edition-sync.js` `GENERATED_AGGREGATORS`,
   `validate-script-sync.js` `COMMON_SCRIPTS`, `kaola-workflow-install-manifest.js`
   `SUPPORT_SCRIPTS` (canonical + Codex twin), the existence/needle assertions in
   `validate-workflow-contracts.js` (canonical + Codex twin), `validate-kaola-workflow-contracts.js`,
   and both forge contract validators, and the `test-parallel-batch.js` invocation in
   `package.json`'s `test:kaola-workflow:claude` chain.
3. Remove the two `require('./kaola-workflow-parallel-batch')` import blocks in
   `test-adaptive-node.js` that exercised the deleted module's own `crossCheckStatus` (the
   independent `batch_active` guard test, which constructs `active-batch.json` directly with
   no import of the deleted module, is unaffected and stays).
4. **Rewrite, do not delete, `docs/plan-run-cards/frontier-batch.md`.** The "frontier unit"
   concept is a live concept — the running-set scheduler IS a frontier fan-out engine — and
   `<!-- PIN: frontier unit -->` + the `frontier unit` literal are machine-pinned on all six
   plan-run routing surfaces (`scripts/test-route-reachability.js` T5, plus three contract
   validators). Deleting the card would force retiring those pins and would delete the only
   operator reference for the live running-set scheduler. Instead, the card is rewritten to
   document ONLY the real `open-ready` / `close-node` / `reconcile-running-set` CLI, with every
   example traceable to the actual source (anti-fabrication — the defect this retirement is
   partly fixing).
5. Update the six plan-run routing surfaces' `<!-- CARD: frontier-batch -->` block prose (the
   parenthetical describing what the card covers) to name the real subcommands; the `<!-- CARD
   -->` marker, the `<!-- PIN: frontier unit -->` marker, and the card pointer itself are
   unchanged (`docs/plan-run-cards/frontier-batch.md` still exists, rewritten per item 4).
6. Scrub every other doc mention of the retired subsystem (`CLAUDE.md`, `README.md`,
   `docs/api.md`, `docs/architecture.md`, `docs/conventions.md`,
   `docs/workflow-state-contract.md`, `docs/plan-run-cards/README.md`,
   `docs/plan-run-cards/repair-routing.md`) — either removed or repointed at the running-set
   scheduler, per file.

## Scope boundary — deliberately untouched

The successor engine (`kaola-workflow-adaptive-node.js`, `kaola-workflow-adaptive-schema.js`,
`kaola-workflow-plan-validator.js`, and their edition ports) is **not** touched by this
retirement:

- **Design-lineage comments** that mention "parallel-batch" (e.g. "Mirrors parallel-batch
  runReconcile", "the SAME verdict the parallel-batch crossCheckStatus gate gives") are
  historical annotations on the live path, not dangling references to a live dependency — they
  stay.
- **The defensive `batch_active` / `active-batch.json` guard** in `adaptive-node.js`'s mutual-
  exclusion guard prologue stays. It is backward-compat crash detection: a residual
  `active-batch.json` on disk (a leftover from a pre-retirement checkout, or a hand-crafted
  test fixture) still refuses a live-coordination conflict rather than silently proceeding.
  Nothing writes this file anymore now that `parallel-batch.js` is gone, so the guard is
  vestigial-but-safe — removing it is a separate, deliberate live-path change with its own
  RED-first treatment, out of scope here.
- **Historical decision records** (`docs/decisions/0008-excise-write-role-batch-isolation.md`,
  `0010-runtime-neutral-per-leg-worktree-isolation.md`, and other existing ADRs referencing
  `parallel-batch.js`) are untouched — they record past decisions and are not rewritten.

## Consequences

- No runtime capability is lost: the running-set scheduler (`open-ready` / `close-node` /
  `reconcile-running-set`) already owned the entire frontier path, including default-on
  disjoint write co-open, before this retirement.
- Four aggregators become three: `OPERATOR_HINT_REGISTRY`-hosting scripts are now
  `adaptive-node.js`, `commit-node.js`, and `plan-validator.js`. The forge aggregator port
  count generated by `sync:editions` drops from five to four
  (`adaptive-node`/`next-action`/`commit-node`/`adaptive-handoff`).
- `docs/plan-run-cards/frontier-batch.md` now documents a CLI every example is traceable to
  the live source — the class of defect (item 4 above) this retirement closes cannot recur for
  this card without a future edit independently drifting from the source again.
- The `batch_active` / `active-batch.json` guard path remains reachable only via a stale or
  hand-crafted manifest; it can no longer be exercised by any first-party producer. A future
  decision may remove it once its diagnostic value is judged to no longer outweigh its
  maintenance cost.
- All four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains, plus
  `scripts/test-route-reachability.js`, must stay green after this retirement — the removal
  touches registries and validators across all four edition trees (#307 cross-edition
  obligation).

## Alternatives considered

- **Fix the four defects in place.** Rejected: the aggregator had zero runtime consumers, so
  fixing crash-safety in dead code is pure maintenance burden with no correctness or
  capability payoff — the running-set scheduler already covers everything it did, more safely.
- **Delete `frontier-batch.md` along with the aggregator.** Rejected: the "frontier unit"
  concept it documents is live (the running-set scheduler), and the card's PIN/CARD markers
  are machine-enforced on all six routing surfaces; deleting the card would force retiring
  those pins and remove the only operator reference for a live mechanism. Rewriting against
  the real CLI preserves the enforcement contract and the operator-facing value.
- **Leave the `batch_active` guard and design-lineage comments for a follow-up cleanup pass.**
  Accepted as the chosen scope boundary (see above) rather than an alternative — bundling a
  live-path behavior change (removing a defensive guard from `adaptive-node.js`) into a pure
  dead-code-removal change would mix two different risk profiles into one review.
