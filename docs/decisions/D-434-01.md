## Context

During an adaptive plan run, two failure modes require re-dispatching the writer of a node:

1. **Write-set overflow.** The writer agent touches files outside its declared write set. The barrier
   detects this at `close-and-open-next` and emits `outOfAllow` paths. The node cannot advance until
   the overflow is reverted. Prior to this decision, the only sanctioned recovery was `reopen-node`,
   which resets the node to `pending` and records a **fresh baseline** (new `commit-node --start`
   snapshot). Re-snapshotting after the overflow means the new baseline INCLUDES the overflow writes;
   the barrier's window now starts AFTER those writes, so it can no longer detect them. This is the
   re-snapshot anti-pattern: laundering overflow writes into the clean baseline.

2. **Reviewer finding requiring re-dispatch.** A code-reviewer or adversarial-verifier node finds
   issues that require the writer of a prior node to make changes. The repairs must happen in a window
   where the barrier can see them. If the reviewer's node is still open and the writer makes repairs
   during that window, those writes overflow the reviewer's empty write set — the #296 barrier-
   ownership trap. The only documented recovery was ledger-reset (reset reviewer to pending, reset
   writer to pending, delete reviewer's gate baseline) — a multi-step manual procedure with no script
   primitive.

A third, subtler case arises on crash-resume:

3. **Evidence-absent in_progress node.** After a crash, an `in_progress` node may have no evidence
   file (the writer was dispatched but never wrote). The orchestrator's crash-resume path distinguished
   between "node is in_progress, evidence present → re-run the commit bracket only" and "node is
   in_progress, evidence absent → re-dispatch the writer." This distinction existed only in prose; the
   `orient` subcommand did not signal it.

All three cases share a root cause: the barrier window can only be trusted if the baseline was
recorded BEFORE the writes it is comparing against. Any recovery action that moves the baseline
forward past existing writes undermines enforcement. The fix requires precise primitives with
well-defined invariants, not ad-hoc reopen procedures.

## Decision

### 1. `revert-overflow` subcommand

`adaptive-node.js revert-overflow --node-id <id> [--dry-run]` reverses a detected write-set
overflow for a node that is currently `in_progress`.

Steps:

1. Run `commit-node.js --barrier-check --node-id <id> --json` to get the current barrier report.
2. Extract `outOfAllow`: the list of paths the writer touched outside the declared write set.
3. If `outOfAllow` is empty, refuse with `no_overflow` — there is nothing to revert.
4. For each path in `outOfAllow`, run `git checkout <baselineSha> -- <path>` (where `baselineSha` is
   read from `barrier-base-{nodeId}`) to restore that file to its baseline state. Only those exact
   paths are touched; no other files are modified.
5. Re-run `commit-node.js --barrier-check` to confirm the barrier now reports clean.
6. Return `{ reverted: [<paths>], barrierNowClean: true }` on success.

`--dry-run` executes steps 1–3 and reports what would be reverted without making any git changes.

**Invariant.** `revert-overflow` does NOT move the baseline. The original `barrier-base-{nodeId}`
is left intact. After revert, the barrier's comparison base is still the pre-write snapshot, and the
writer's remaining in-set writes are still visible to the barrier. The barrier's enforcement window
is preserved in its entirety.

### 2. `repair-node` subcommand — the anti-laundering primitive

`adaptive-node.js repair-node --node-id <id>` transitions a `complete` node back to `in_progress`
for re-dispatch of the writer, WITHOUT recording a fresh baseline.

Steps:

1. Assert the node is `complete` (if `in_progress`, refuse with `already_open`; if `pending`, refuse
   with `not_yet_run` — the operator should use `open-next`).
2. Reset the node's ledger state from `complete` to `in_progress`.
3. Leave `barrier-base-{nodeId}` UNCHANGED. This is the defining invariant of `repair-node`.
4. Delete ONLY the gate baselines for downstream nodes (`barrier-base-{downstreamId}` for each
   successor of `nodeId`). Downstream gate baselines are stale because the node's commit will change;
   they must be regenerated by the downstream nodes' `open-next`. The upstream node's own baseline
   is NOT deleted.
5. Return `{ state: 'in_progress', baselineReused: true, deletedDownstreamBaselines: [...] }`.

The `baselineReused: true` flag is explicit in the return value so the orchestrator can log it. It is
audit evidence that the anti-laundering invariant was upheld: the baseline was NOT re-snapshotted.

**Contrast with `reopen-node`.** `reopen-node` (the existing primitive) resets the node to `pending`
and records a NEW baseline via `commit-node --start`. It is the correct choice when the writer needs
a genuinely clean slate (e.g., the prior attempt's writes are all garbage and should be discarded,
not repaired). `repair-node` is the correct choice when the prior attempt's writes are mostly correct
and the barrier must continue to see them. These two primitives have distinct contracts and must not
be substituted for each other. Using `reopen-node` when `repair-node` is correct launders the
writer's accumulated work; using `repair-node` when `reopen-node` is correct leaves stale writes
in the window that the new dispatch may not intend.

### 3. `requires_redispatch` signal in `orient`

`orient` (the read-only diagnostic subcommand) gains a `requires_redispatch` field in its output for
`in_progress` nodes:

```json
{
  "nodeId": "n2-impl",
  "state": "in_progress",
  "requires_redispatch": true,
  "redispatch_reason": "evidence_absent"
}
```

`requires_redispatch: true` is emitted when ALL of the following hold:

- The node is `in_progress`.
- Its evidence file (`.cache/{nodeId}.md`) is absent OR does not contain the `evidence-binding:`
  header matching the current nonce.
- The node has NOT yet emitted a close event in the provenance log (i.e., it is not in the
  commit-bracket phase).

When `requires_redispatch: false` and the node is `in_progress`, the evidence is present and the
node is in its normal run. When `requires_redispatch: true`, the orchestrator knows to re-dispatch
the writer without first running `repair-node` or `reopen-node` (the node is already open; the
evidence is simply missing or stale from a crash).

`redispatch_reason` is a typed string: `evidence_absent` (no file), `binding_mismatch` (file exists
but nonce is wrong), or `evidence_incomplete` (file exists but required tokens are absent per
`ROLE_TOKEN_REGISTRY`).

This closes the crash-resume gap: the distinction between "in_progress + evidence present → re-run
commit bracket" and "in_progress + evidence absent → re-dispatch writer" is now machine-signalled
rather than prose-described.

### 4. Anti-laundering invariant (normative statement)

The following invariant is normative for the adaptive node lifecycle:

> **The baseline recorded at `open-next` for a node MUST NOT be moved forward past writes that the
> barrier is responsible for detecting. Any recovery primitive that re-snapshots the baseline after
> writes have occurred in that window launders those writes and undermines barrier enforcement.**

Practically:

- `repair-node` MUST keep the original baseline (`baselineReused: true`).
- `revert-overflow` MUST NOT move the baseline (it undoes writes; baseline stays fixed).
- `reopen-node` (fresh baseline) is correct ONLY when starting from a genuinely clean slate. Its
  use for post-write recovery is the explicit anti-pattern.

The `baselineReused` flag on `repair-node` and the absence of any baseline-move in `revert-overflow`
are the structural enforcement of this invariant at the script level.

## Consequences

**For overflow recovery:**

- `revert-overflow` is the first-line response to a `barrier_fail` with non-empty `outOfAllow`. The
  operator calls it (optionally with `--dry-run` first), the overflow is surgically undone, and the
  writer re-attempts the omitted writes within the declared write set. No manual `git checkout`
  archaeology required.

**For reviewer-finding recovery:**

- When a reviewer closes with findings requiring writer re-dispatch, the orchestrator calls
  `repair-node --node-id <writer-node-id>` on the writer node (not the reviewer node). The writer's
  barrier window resumes from the original baseline. The reviewer's findings land in the reviewer's
  own window (which is correct). The #296 barrier-ownership trap — where repairs overflow the
  reviewer's empty write set — is avoidable by sequencing: reviewer closes first, then
  `repair-node` on the writer, then re-dispatch.

**For crash-resume:**

- `orient`'s `requires_redispatch` signal removes ambiguity: the orchestrator reads `orient` output
  and branches on the flag rather than inspecting the `.cache/` directory itself.

**For `docs/api.md`:**

- The `revert-overflow` subcommand, `repair-node` subcommand, and `requires_redispatch` signal in
  `orient` are documented under Adaptive Node Lifecycle. The anti-laundering invariant is stated
  normatively. The `baselineReused` flag and `deletedDownstreamBaselines` array are part of the
  `repair-node` response schema.

**For the test suite:**

- Scenarios cover: `revert-overflow` happy path (overflow reverted, barrier clean); `revert-overflow
  --dry-run` (no mutations); `revert-overflow` on clean node refuses `no_overflow`; `repair-node` on
  complete node transitions to in_progress with `baselineReused: true` and deletes downstream
  baselines; `repair-node` on in_progress refuses `already_open`; `orient` signals
  `requires_redispatch: true` on evidence-absent in_progress node; `orient` signals
  `requires_redispatch: false` on evidence-present in_progress node.

**For cross-edition parity:**

- All three new subcommand / signal additions land in the `adaptive-node.js` aggregator, which ships
  in four editions. This is a cross-edition diff; all four `npm run test:kaola-workflow:*` chains
  must be green before finalize, verified by a `run-chains.js` chain receipt (D-432-01).
