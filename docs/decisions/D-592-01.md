# D-592-01. Widen the sink-merge closure gate to cover no-primary bundle sinks and record `closed_issues` on the success path

Date: 2026-07-02
Status: Accepted
Issue: #592

## Context

`sink-merge --sink`'s `closure` step is gated by a single condition before it
runs the close loop. Pre-fix, that gate was `!OFFLINE && args.issue != null`
— it required a primary issue to be present at all. A bundle sink can be
invoked two ways: with a primary (`--issue N --issue-numbers N,M,...`) or
without one (`--issue-numbers A,B,...` only, no primary — every member is a
peer, none is "the" issue). The no-primary shape is a real, supported
invocation (observed live on bundle-587-589, 2026-07-02): the gate tripped
false, so the entire close loop — primary close, bundle-member loop, and the
`receipt.closed_issues` recording that lived inside it — was skipped
entirely. Execution then fell straight through to the unconditional
`stepDone('closure')` below the gate, so:

- The sink-receipt (`.cache/sink-receipt.json`) recorded `steps.closure:
  "done"` with no `closed_issues` field.
- The JSON output reported `status: "sinked"` — a clean success — having
  closed zero issues.
- `resume` treats a `"done"` step as already satisfied and never re-enters
  it, so the miss was **permanent**: no amount of re-running `--sink`
  recovered the skipped closes. Recovery required a manual `gh issue close`
  per member.

The bug was silent because it produced no error and no non-zero exit — the
transaction genuinely completed every OTHER step (merge, push, archive), so
`status: "sinked"` was not false on its face, only incomplete.

## Decision

**(a) Widen the closure gate to run the close loop for `--issue-numbers`-only
invocations, and record `closed_issues` on the receipt on both the success
and failure paths so a resume can verify-then-retry.**

1. **Gate.** The closure step's entry gate becomes
   `!OFFLINE && (args.issue != null || (Array.isArray(args.issueNumbers) &&
   args.issueNumbers.length > 0))` — the loop runs whenever a primary issue
   OR at least one bundle member is present. `OFFLINE` still short-circuits
   the whole step (unchanged).

2. **Primary close stays conditional.** The primary-issue close call
   (`closeOne(args.issue, ...)` + the `workflow:in-progress` label removal)
   now runs only `if (args.issue != null)` — it is no longer assumed to
   always fire just because the gate passed.

3. **Bundle-member loop bound widened.** The bundle-member loop's guard
   moves from `args.issueNumbers.length > 1` (which implicitly assumed a
   primary occupies one array slot) to `args.issueNumbers.length >
   (args.issue != null ? 1 : 0)` — with a primary, at least 2 members must
   exist for the loop to have any peer to close (unchanged threshold); with
   no primary, any non-empty `issueNumbers` array enters the loop and closes
   every member (none is skipped as "the primary").

4. **`receipt.closed_issues` recorded on success too.** Previously
   `receipt.closed_issues` was written into the sink-receipt only on the
   `partial`-failure branch (see the pre-existing `#497` comment: "only the
   FAILURE path records into the receipt + refuses — SUCCESS stays
   byte-equivalent"). It is now assigned (`closed.slice().sort(...)`,
   ascending) whenever `closed.length > 0`, regardless of whether `failed`
   is also non-empty, so `stepDone('closure')` on the success path persists
   the actually-closed set to disk. A resumed `--sink` reading a `"done"`
   closure step can now inspect `closed_issues` to confirm what already
   closed instead of having no record at all.

Applied identically across all four sink-merge editions: canonical
`scripts/kaola-workflow-sink-merge.js`, the byte-identical codex twin, and
both `plugins/kaola-workflow-{gitlab,gitea}/scripts/` forge ports. Behavior
for single-issue sinks and bundle-with-primary sinks is unchanged — the gate
was already true for both, and the loop bound's `> 1` branch is preserved
verbatim for the with-primary case. The existing `sink_incomplete` /
`step: "closure"` fail-closed refuse (issue #497) is unchanged: a genuine
close failure still leaves the step NOT done and still refuses rather than
reporting `status: "sinked"`.

## Alternatives considered

- **Refuse `--issue-numbers` without `--issue` at the CLI argument-parsing
  level** (require a primary on every bundle sink invocation). Rejected:
  this would reject a currently-legal, currently-used invocation shape
  (multiple prior bundle closures used bare `--issue-numbers`) rather than
  fixing its behavior — it trades a silent-miss defect for a hard refusal of
  legitimate calls, pushing the burden onto every caller to always supply a
  redundant primary even when the bundle members are genuine peers. It also
  does nothing for a receipt already stranded in the no-`closed_issues`
  `"done"` state from before the fix ships (option (a)'s gate widening at
  least makes the loop itself correct going forward; the receipt gap is
  closed by move 4 regardless of which repair the loop takes).

- **Leave the loop-skip behavior but make `stepDone('closure')` conditional
  on the gate having actually run** (i.e., don't mark `closure` done at all
  when there's no primary). Rejected: this converts a silent miss into a
  perpetual `not-done` state for the (legal) no-primary shape — every
  no-primary bundle sink would refuse to ever report `closure: "done"`,
  which is a worse regression than the bug (no invocation of this
  legitimate shape could ever complete). The real fix is to make the loop
  run over the members that exist, not to gate `stepDone` on an
  ad-hoc primary check.

- **Only fix `receipt.closed_issues` recording (move 4) without widening the
  gate (move 1).** Rejected as insufficient: it makes the receipt
  vocabulary honest about "what closed" but does not fix the actual defect
  — a no-primary bundle sink would still skip closing every member. The gate
  widening is the load-bearing fix; the receipt change is what makes the fix
  observable and resumable.

## Consequences

- A bundle sink invoked as `--sink --issue-numbers A,B[,...]` (no `--issue`)
  now closes every member on the forge, matching the behavior a caller
  reasonably expects from `status: "sinked"`.
- `receipt.closed_issues` is now a reliable signal of what closed on ANY
  successful or partially-successful closure step, not just a failure
  artifact — a future resume/verify tool can read it without special-casing
  the failure path.
- Single-issue and bundle-with-primary invocations are byte-behaviorally
  unchanged; no caller of the with-primary shape needs to change anything.
- The fix is purely inside the `closure` step's own gate and bookkeeping; no
  schema field was removed or renamed, so no backward-compatibility concern
  arises for existing `.cache/sink-receipt.json` files written before this
  change (a pre-fix `"done"` closure step with no `closed_issues` remains
  parseable — the field is simply absent, as it always was for that legacy
  case).
