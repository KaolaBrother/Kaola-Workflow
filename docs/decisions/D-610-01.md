# D-610-01. Rename the plan model-column vocabulary to neutral tier tokens with legacy aliases

Date: 2026-07-03
Status: Accepted
Issue: #610
Related: #609 (the codex/opencode B2 model-noun purge this decision's legacy aliases must keep
         passing), #537 (the opencode-edition root-cause note this decision supersedes), #382
         (the original `{opus, sonnet}` model-column grammar this decision renames)
Supersedes: the "`opus`/`sonnet` stay the portable cross-edition vocabulary" ruling recorded in
            `docs/opencode-edition.md` (the "Model effort — two tiers as reasoning-effort variants"
            section) and in the #537 CHANGELOG entry's "Canonical-vocabulary preserved" note.

## Context

The adaptive plan grammar's optional `model` column (#382) used the closed vocabulary `{opus,
sonnet}` for every edition. That choice was deliberate at the time: Claude consumes the tokens as a
literal `Agent(model=…)` parameter, so naming the column after Claude's own models was the path of
least resistance. Two later rulings entrenched this as a permanent design constraint rather than an
implementation detail — the opencode-edition doc's "Model effort" section states "`opus`/`sonnet`
stay the plan's portable per-node vocabulary, `NODE_MODEL_TIERS`", and the #537 CHANGELOG entry
explicitly preserves `NODE_MODEL_TIERS = Object.freeze(['opus','sonnet'])` as an "architectural
constraint" while only translating the *displayed* opencode surface.

That constraint no longer matches what any edition actually does with the tokens. Codex maps them
to a per-spawn `reasoning_effort` (`opus`→`xhigh`, `sonnet`→`high`, #405/#451/#582); opencode maps
them to a provider-relative effort-variant rank via `mapTier` (top/second, #537). Only the Claude
editions still consume the raw token as a literal model name, and even there it is really "the
tier this node should run at," not "the model this node happens to use." The `{opus, sonnet}`
column was always a rank in fact; it was a Claude noun in name. That mismatch surfaced concretely
as the #609 B2 class of leaks — Codex/opencode prompt surfaces had to special-case-exempt the
column tokens from a "no Claude model nouns" scan, because the vocabulary itself was borrowed
Claude prose.

## Decision

**Rename the vocabulary to neutral tier tokens; keep the old tokens working forever as aliases.**

1. `NODE_MODEL_TIERS` becomes `['reasoning', 'standard']` (schema-level rename, landed by n4 of this
   run). A `normalizeTier(token)` helper accepts the neutral tokens as pass-through and the legacy
   `opus`→`reasoning` / `sonnet`→`standard` as aliases; every tier consumer (`TIER_RANK`,
   `dispatchEffort`, `mapTier`, `dispatchEffortOpencode`, `dispatchModelClaude`, the synthesizer
   reasoning-floor check) routes through the normalizer, so a legacy token resolves identically to
   its neutral counterpart everywhere — no per-consumer special-casing.
2. **Frozen plans keep their bytes.** An archived plan's `opus`/`sonnet` cell is never rewritten;
   `plan_hash` is unaffected; `--resume-check` and dispatch behave byte-for-byte as before parsing
   through the normalizer. New plans author the neutral tokens; the legacy tokens remain a
   permanently-accepted superset, not a deprecated-and-removed one.
3. **Prose stays neutral-token-first, alias-mentioned-once.** Every authoring/dispatch surface (the
   adapt/plan-run command + SKILL pairs, `agents/workflow-planner.md`, the three
   `workflow-planner.toml` twins) leads with `{reasoning, standard}` and states, once per surface,
   that the legacy `opus`/`sonnet` aliases remain accepted and resolve to the same dispatch. This
   keeps the #609 B2 model-noun scan meaningful — the scan's B1 exemption list gains exactly one
   new shape (the literal `` `opus`/`sonnet` `` alias-pair mention), so a *new*, unexempted
   capitalized "Opus"/"Sonnet" mention is still caught, while the intentional lowercase alias
   explanation is not a false positive.
4. **The Claude-runtime dispatch mapping uses lowercase `Agent(model=…)` literals, not capitalized
   prose nouns.** `agents/workflow-planner.md` states "`reasoning` dispatches via
   `Agent(model="opus")`, `standard` via `Agent(model="sonnet")`" rather than prose like "the Opus
   model" — this is the more precise statement (the Agent tool's `model=` parameter is literally
   lowercase) and, because this file is also the canonical source for the generated opencode
   mirror, it avoids reintroducing a #609-class capitalized-noun leak that `sync-opencode-edition.js
   rewriteClaudeModelNouns()` has no rule for yet (that function targets specific known noun-phrase
   shapes, not a blanket `/Opus|Sonnet/` strip).

## Consequences

- A codex/opencode adaptive dialogue for a *newly authored* plan contains zero `opus`/`sonnet`
  tokens in the plan's own `model` column — only the neutral tier names, plus the one-time alias
  disclosure in the authoring/dispatch prose.
- An archived plan authored under the old vocabulary keeps working forever: same `plan_hash`, same
  `--resume-check` result, same dispatch effort. Nothing in the resume path re-validates against a
  vocabulary the plan predates.
- The docs/opencode-edition.md "Model effort" ruling is updated in this same change: `opus`/`sonnet`
  are now documented as the legacy aliases of `reasoning`/`standard`, not the primary vocabulary.
- Cross-edition (#307): the schema rename (n4) touches the ×4 byte-identical
  `kaola-workflow-adaptive-schema.js` plus `plan-validator`/`next-action`/`resolve-agent-model`
  across all four editions; this decision's prose-and-enforcement half touches the six
  adapt/plan-run authoring surfaces, the three `workflow-planner.toml` twins, and the three
  `validate-*-contracts.js` / `test-route-reachability.js` enforcement pins. All four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains, plus `test-opencode-edition.js`
  (additive, D-530-02), must stay green.

## Non-goals

- **Not a deprecation.** The legacy `opus`/`sonnet` tokens are not scheduled for removal; they are
  a permanent accepted alias pair, exactly as durable a contract as the neutral tokens themselves.
- **No third tier.** This decision does not add a `haiku`-equivalent rank or any tier beyond the
  existing two; it only renames the two that exist.
- **No change to WHICH model/effort a tier resolves to on any runtime.** `reasoning` dispatches
  exactly where `opus` used to (Claude `opus`, Codex `xhigh`, opencode top-variant) and `standard`
  exactly where `sonnet` used to — this is a vocabulary rename with a proven-identical alias
  resolution, not a re-tuning of any dispatch table.

## Alternatives considered

- **Leave `{opus, sonnet}` as the permanent vocabulary and keep exempting it from the B2 model-noun
  scan on Codex/opencode surfaces.** Rejected: the #609 exemption already existed as a workaround
  for exactly this mismatch; a rank vocabulary borrowed from one runtime's model names is confusing
  on the other three, and every future B2-adjacent guard would need to keep re-deriving the same
  "these two Claude nouns are actually fine here" carve-out.
- **Rename and hard-break old plans (no alias, resume refuses on a legacy cell).** Rejected: this
  would force every archived/in-flight frozen plan to re-freeze or fail `--resume-check` for a
  purely cosmetic rename, which is a disproportionate blast radius for a vocabulary change that
  carries zero behavior difference.
- **Capitalized "Opus"/"Sonnet" prose in the Claude dispatch-mapping sentence (matching the literal
  request wording), extending `rewriteClaudeModelNouns()` with a new pattern for it.** Considered
  and set aside in favor of the lowercase `Agent(model=…)` phrasing (Decision point 4): the
  lowercase form is more precise (it is literally the tool parameter value) and needs no new
  purge-pattern maintenance in the opencode generator, whereas a new capitalized noun shape would
  need its own bespoke `rewriteClaudeModelNouns()` rule to keep the opencode mirror clean.
