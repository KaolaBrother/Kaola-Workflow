# D-518-01. Bind sink-receipt cycle identity via `branch_head` to prevent stale-receipt false-resume across cycles

**Date:** 2026-06-17
**Status:** Accepted
**Issue:** #518
**Related:** #484 (freshness guard — the `stale_sink_receipt` ancestry backstop; preserved by this fix),
            `scripts/kaola-workflow-sink-merge.js` (`loadOrInitReceipt`),
            `docs/workflow-state-contract.md` § Sink-receipt schema (new `branch_head` field),
            `docs/api.md` § Closure Contract (receipt field inventory).

## Context

`loadOrInitReceipt` reads the on-disk sink receipt and either resumes it (if partially
complete from a prior crash) or initializes a fresh one. Before #518, the receipt carried no
identity binding to the branch cycle that produced it. A completed all-`done` receipt from a
prior cycle (a prior slice of a multi-slice run, or a reused `project` + `branch` pair) was
indistinguishable from a fresh receipt at init time — so `loadOrInitReceipt` returned the
completed receipt verbatim, the step loop skipped every already-`done` step, and
`sink-merge` reported `{result:"ok", status:"sinked"}` without performing any merge or push.

This is the stale-receipt false-resume defect (#484 targeted the ancestry backstop as one
layer of defense; #518 addresses the root cause at init time).

The ancestry backstop (#484) checks, after the step loop, that the feature branch tip is an
ancestor of the default branch — a merged branch satisfies this, so a genuinely completed
prior-cycle receipt does not trip the backstop when the prior merge actually landed. This
means the backstop alone does not catch the case where a prior cycle's work was genuinely
merged and the receipt is correctly all-`done` for that cycle — but a NEW cycle on the same
branch should not resume from the prior cycle's receipt. Cycle identity binding at init time
is the correct fix.

## Decision

**Stamp `branch_head: <branch-tip-sha>` into the receipt at init time and use it to detect a
stale receipt at load time.**

`loadOrInitReceipt` is extended in two ways:

1. **At init:** when creating a fresh receipt, stamp `branch_head` with the current branch-tip
   SHA. This binds the receipt to the exact commit state of the branch at the start of the
   cycle.

2. **At load:** when an on-disk receipt exists, compare its `branch_head` against the current
   branch-tip SHA. A receipt is considered stale (belonging to a prior cycle) when:
   - `branch_head` is absent from the receipt (pre-#518 receipt, conservatively treated as
     unbound and potentially stale), OR
   - `branch_head` does not equal the current branch tip AND is not an ancestor of the
     current branch tip (the branch has advanced beyond the receipt's reference point, meaning
     new commits have been added since the receipt was written).

   On a stale detection, the receipt is **reinitalized** (not resumed): a fresh all-unset
   receipt is written with the current `branch_head`. The prior-cycle receipt is discarded.
   This emits `stale_sink_receipt` to signal that a fresh cycle was detected. Callers see
   `stale_sink_receipt` in the receipt and know the prior cycle's state was not carried
   forward.

3. **Genuine mid-cycle resume:** if the `branch_head` in the receipt IS an ancestor of the
   current branch tip (or equal to it), the receipt belongs to the current cycle and is
   resumed normally. The #484 ancestry backstop remains in place as a defense-in-depth
   check after the step loop.

**Why `branch_head` and not a timestamp or a nonce:** The branch-tip SHA is the natural cycle
identity token — it is the commit the current sink run was invoked to merge, is computable
from the local git state without a forge call, and is stable across legitimate mid-cycle
resumes (a crash + resume does not advance the branch tip). A timestamp or nonce would
require a separate storage or coordination mechanism; the SHA is already available and
semantically correct.

**Why ancestor-or-equal, not strict equality:** A sink-merge may update the feature branch
(e.g. to record a receipt commit) between steps. The strict-equality check would false-refuse
a legitimate mid-cycle resume if the branch advanced as part of a previous step in the same
cycle. Allowing ancestors covers mid-cycle branch advances while still catching a stale
receipt from a prior cycle whose tip is no longer in the ancestry chain of the current
branch.

## Implementation

`scripts/kaola-workflow-sink-merge.js` (canonical + the Codex byte-twin + the two forge ports):
`loadOrInitReceipt` is updated to stamp `branch_head` at init and to compare it at load.
The `stale_sink_receipt` emit path is updated to surface the detection reason when triggered
by the new `branch_head` guard (as distinct from the #484 ancestry backstop, which remains
in the post-step-loop position and continues to catch any missed cases). The `branch_head`
field is added to the receipt schema in `docs/workflow-state-contract.md` and `docs/api.md`.

## Consequences

- A completed prior-cycle receipt is now correctly detected as stale at load time and
  reinitalized, preventing a false `status:sinked` exit without a real merge.
- The #484 ancestry backstop is preserved as a defense-in-depth layer for any path that
  reaches the post-step-loop check with a stale receipt (e.g. a receipt missing `branch_head`
  from a pre-#518 run).
- Legitimate mid-cycle resumes (crash → restart on the same branch, same commit) are
  unaffected: `branch_head` equals or is an ancestor of the current tip → resume proceeds.
- The `branch_head` field is present in all new receipts; receipts written before #518 lack
  the field and are conservatively treated as unbound at load time.
