# D-590-01. Baseline-first `open-next` ordering

Date: 2026-07-02
Status: Accepted
Issue: #590
Related: #585 (the companion scheduler mutual-exclusion lock, `docs/decisions/D-585-01.md`,
             found by the same 2026-07-02 parallel-execution audit)

## Context

The serial `open-next` path inverted the crash-safe ordering the running-set path
deliberately uses. `runOpenNext` wrote the plan (ledger row → `in_progress`) FIRST and only
THEN shelled `commit-node --start` to record the node's barrier baseline. A crash in that
window left a `## Node Ledger` row `in_progress` with NO baseline on disk. `reconcile-
running-set` does not cover this window — there is no running set on the serial path to
reconcile — so a later `close-node`/`close-and-open-next` on that row dead-ended with the
plan-validator's `no_barrier_base` refusal, and (before this fix) `adaptive-node.js` had no
`OPERATOR_HINT_REGISTRY` entry for it, so the operator saw only the generic fallback hint
with no pointer to the actual repair.

`runOpenReady` (the running-set `max=N` entry point) already does this in the safe
direction: its Phase 2 shells `commit-node --start` (baseline on disk) BEFORE the Phase 3
single plan write that flips the ledger rows. `runOpenNext` predates the running-set open
protocol and was never retrofitted to the same ordering — a straightforward oversight, not
a design disagreement.

The window is recoverable today (re-invoking `open-next` is idempotent — the
`alreadyAtTarget` branch re-records the baseline), so severity is LOW; but it is an
avoidable asymmetric hazard between two openers that are otherwise meant to be crash-
equivalent.

## Decision

Reorder `runOpenNext` to record the baseline BEFORE the ledger flip, mirroring
`runOpenReady` Phase 2 → Phase 3:

1. Shell `commit-node --node-id <id> --start --json` (record the baseline) — refuse
   `baseline_failed` on failure, with the ledger row still `pending`.
2. Only on success: `spliceLedgerNode` the row `pending` → `in_progress` and write the plan.

This inverts the hazard direction. If the flip write now fails AFTER a successful baseline,
the orphaned baseline is harmless — `recordBase` is idempotent/overwriting on the node's
next open, so nothing needs cleanup. If the baseline itself fails, the ledger is UNTOUCHED
(still `pending`) — a clean, idempotent re-open, never a row stranded `in_progress` with no
baseline.

A new `no_barrier_base` entry was added to `adaptive-node.js`'s `OPERATOR_HINT_REGISTRY`
(mirroring the pre-existing `plan-validator.js` hint of the same name), naming the
idempotent `open-next` re-invoke as the repair for the close-time dead-end this reorder is
meant to make unreachable going forward on the serial path, and defense-in-depth for any
plan carrying a pre-fix orphan row.

## Consequences

- No state where the `## Node Ledger` says `in_progress` but no barrier baseline exists on
  disk can arise from a fresh `open-next` after this fix.
- Serial-path behavior is otherwise byte-identical — the change is purely a reordering of
  two existing writes (baseline shell, ledger flip); no new state, schema, or subcommand.
- RED-first: direct-call tests over `runOpenNext`'s injected seams asserted baseline-before-
  flip on both the failure and success paths and failed against the pre-change write-first
  code (a simulated baseline failure left the row `in_progress` with no baseline); now green.
- The reorder is orthogonal to and unaffected by the #585 scheduler lock (`open-next` is one
  of the `SPLIT_GUARDED_SUBCOMMANDS` the lock wraps, but the lock serializes ACCESS to the
  subcommand body, not the internal ordering of writes within one invocation).

## Alternatives considered

- **Leave the ordering and only add the operator hint.** The issue's proposed fix offered
  this as an "alternatively (or additionally)" option. Rejected as insufficient on its own:
  a hint only tells the operator how to recover AFTER the hazard has already occurred; the
  reorder removes the hazard's root cause (predates the running-set protocol, never
  retrofitted) so the dead-end becomes unreachable on a fresh open rather than merely easier
  to recover from. Both were implemented together — belt-and-suspenders for any pre-fix
  orphan row still on disk from before this change.
