# D-697-01 — Extend D-547-01 validation_command for schema 2 and execute it with a deterministic local runner

- **Status:** accepted
- **Date:** 2026-07-15
- **Context:** issue #697
- **Extends:** D-547-01 for newly authored schema-2 plans; legacy D-547-01 behavior is preserved

## Context

D-547-01 established one hash-covered `validation_command` recorded at plan freeze. For existing
plans it is reader-only metadata used by record-once/freshness behavior; it does not mandate a
runner, timeout, repeated execution, or environment semantics. Reviewer contract 2 needs comparable
current-candidate validation vectors and therefore needs bounded execution policy and deterministic
command identity.

Adding a second namespaced command key would create two authorities that could disagree. Reusing the
old key without an explicit version boundary would retroactively change frozen-plan semantics and
bytes.

## Decision

Extend the existing `validation_command` namespace for plan schema 2. Do not add a second command
field. Add hash-covered sibling fields:

- `validation_cwd` — normalized repository-relative path, default `.`;
- `validation_repetitions` — integer `1..5`, default `1`;
- `validation_pass_rule` — exactly `all`;
- `validation_timeout_minutes` — integer `1..120`; and
- `validation_env_allowlist` — sorted, duplicate-free shell identifiers, default empty.

A schema-2 code-producing plan requires both `validation_command` and
`validation_timeout_minutes`. Unknown/duplicate policy fields, path escape, non-all pass rules,
invalid bounds, or noncanonical environment keys refuse freeze.

Preserve D-547-01 exactly for a verified contract-1 plan. `parseValidationPolicy` maps it to:

```text
command = existing validation_command or null
cwd = .
repetitions = 1
pass_rule = all
timeout_minutes = null
env_allowlist = []
runner_required = false
source = legacy-d547
```

This mapping does not rewrite legacy plan or journal bytes and imposes no new execution/timeout
semantics. `validation_test_consumes` retains its existing freshness-band role.

Implement a forge-neutral, four-edition byte-identical
`kaola-workflow-validation-runner.js`. It starts child execution from a scrubbed environment,
injecting deterministic/platform-minimum keys plus the frozen allowlist. `command_id` binds exact
command bytes, normalized policy, secret-safe effective-value digests, Node/shell/simple-command
executable realpath-mode-version identities, and relevant lock/toolchain identities. The validation
vector separately binds candidate identity and every repetition's pre/post candidate digest. Raw
allowlisted values and raw child output are not persisted.

Every repetition binds pre/post candidate digests, exit/signal/timeout, output digests, and normalized
failure signature. Reduction is:

- `pass` only when every comparable run exits zero and the candidate stays unchanged;
- `fail` only when every comparable run exits nonzero with one stable failure signature and no
  mutation/signal/timeout; and
- `inconclusive` for mixed results/signatures, timeout, signal, candidate mutation, unresolved or
  incomparable identity, or any other case.

`vector_id` excludes audit timestamps and addresses deterministic semantic fields;
`receipt_sha256` binds the complete durable receipt including audit time. Inherited obligations are
`{command_id, required_pass_vector_id}` and may not be silently dropped or changed.

The opt-in local Claude/Codex qualification command records contract/profile/context identities and
invariant-class outcomes through mockable process adapters. It does not compare natural-language
finding equality and is not an external completion authority.

## Consequences

- There remains one frozen validation command authority across plan versions.
- Contract-1 behavior and bytes remain stable; contract 2 gains deterministic comparable vectors.
- Secrets influence command identity through digests without appearing in receipts.
- Timeout, signal, flake, mutation, and toolchain ambiguity fail closed as `inconclusive`.
- Validation is owned and local; no hosted pipeline is required as a plan or Finalization gate.

## Alternatives considered

- **Add `review_validation_command` or another namespace.** Rejected because two command fields can
  disagree and weaken freshness/comparability.
- **Retrofit runner semantics onto D-547-01 plans.** Rejected because it changes a frozen public
  contract and could brick resumable work.
- **Use an external service as the validator.** Rejected because completion must not depend on an
  authority the workflow does not own.
