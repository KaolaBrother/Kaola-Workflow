# D-427-01. probe-before-close makes cmdFinalize and sink-merge independently idempotent

Date: 2026-06-12
Status: Accepted
Issue: #427
Related: `scripts/kaola-workflow-claim.js` (`cmdFinalize`, `closeIssueIdempotent`),
         `scripts/kaola-workflow-sink-merge.js` (issue close step),
         `docs/api.md` § Closure Contract (closure receipt schema, `remote_issue_closed` enum),
         D-426-01 (copy-then-verify-then-delete, the related #426 fix)

## Context

Before issue #427, `cmdFinalize` never closed GitHub issues directly. The conventional ordering was:
`cmdFinalize` (archive + roadmap cleanup) → `sink-merge` (merge + issue close). Because `cmdFinalize`
did not close issues, the sequence was fragile:

**The ordering dependency was not enforced.** A contractor that ran `sink-merge` BEFORE `cmdFinalize`
would close the issue first, then `cmdFinalize` would run without issue-close logic. The receipt's
`remote_issue_closed` field would be `close_pending` forever — never `closed` — even though the
issue was actually closed.

**`sink-merge` was not idempotent on already-closed issues.** When `cmdFinalize` was later given
issue-close logic, running both `cmdFinalize` (closes) then `sink-merge` (tries to close again)
would cause `sink-merge`'s issue-close step to fail on an already-closed issue. The two paths had
no coordination: each assumed it owned the close, so their combined execution was not idempotent.

**No offline-safe probe path.** `cmdFinalize` had no probe-then-close pattern; adding one naively
would have introduced a hard dependency on forge availability even when `KAOLA_WORKFLOW_OFFLINE=1`.

Two remediation postures were considered:

1. **Exclusive ownership:** declare that either `cmdFinalize` XOR `sink-merge` owns the close, never
   both. Document the invariant, enforce it at runtime with a flag. This requires callers to know
   which path they are on and is fragile to ordering mistakes.
2. **Probe-before-close idempotency in both:** add `closeIssueIdempotent(n, opts)` to `cmdFinalize`
   that probes the issue state before attempting to close, and add the same probe to `sink-merge`'s
   close step. Both callers become independently idempotent; their ordering no longer matters.

## Decision

**Both `cmdFinalize` and `sink-merge` use probe-before-close, making them independently idempotent.**

`closeIssueIdempotent(n, opts)` in `cmdFinalize`:
- Probes the issue state via `probeIssueState(n, opts)` before any close attempt.
- If already closed → records `already_closed`, no close call made.
- If offline (`KAOLA_WORKFLOW_OFFLINE=1`) → records `skipped_offline`, no close call made.
- If probe unavailable (network error) → records `failed` with a warning.
- If open → attempts close; records `closed` on success or `failed` on error.
- Gated on `!keepIssueOpen && !OFFLINE && !args.keepWorktree`. Keep-open, offline, and worktree-keep
  runs are excluded: the issue should remain open, so no probe/close fires.

`sink-merge` issue close step:
- Probes before close, same probe-then-skip-if-already-closed pattern.
- When `cmdFinalize` already closed the issue, `sink-merge` sees `state: 'closed'` from the probe
  and records `already_closed` in the receipt rather than failing the close call.

**New `closure` receipt object:**

The closure receipt gains a `closure` object with per-issue-close audit fields:

```json
{
  "closure": {
    "attempted": [42, 47, 53],
    "closed": [42, 47],
    "failed": [53],
    "skipped_offline": [],
    "kept_open": []
  }
}
```

- `attempted` — issue numbers for which a close was attempted (probe found open or skipped).
- `closed` — issue numbers successfully closed by this caller.
- `failed` — issue numbers whose close call failed.
- `skipped_offline` — issue numbers skipped because `KAOLA_WORKFLOW_OFFLINE=1`.
- `kept_open` — issue numbers skipped because `keepIssueOpen` was requested.

**Why probe-before-close beats exclusive ownership:**

Exclusive ownership requires every caller to know in advance which path owns the close — this
knowledge must propagate through the contractor choreography, the plan nodes, and any manual
recovery flows. Probe-before-close removes that requirement: either caller can be run first, or
both can be run in sequence, or either can be re-run after a crash. The receipt captures what each
caller actually DID (closed vs already_closed), so the audit trail is accurate regardless of order.

**Consistency with the existing `probeIssueState` pattern:**

`probeIssueState` was already used by `sink-merge`, `watch-pr`, and the classifier for open/closed
state. Extending the probe-before-close pattern to `cmdFinalize` is the natural continuation of
the existing approach, not a new mechanism.

## Consequences

**For operators:**

- `cmdFinalize` now closes issues on the normal merge lane when not offline and not keep-open.
  Previous runs that relied on `sink-merge` being the sole closer continue to work: the probe
  sees `already_closed` (if `cmdFinalize` ran first) and no double-close is attempted.
- `sink-merge` no longer fails when it finds an already-closed issue. The receipt records
  `already_closed` instead of an error.
- Running `cmdFinalize` + `sink-merge` in EITHER order produces a correctly-closed issue.

**For the closure receipt schema:**

- `closure` object added (see above). Present on bundle and single-issue receipts.
- `remote_issue_closed: close_pending` → `cmdFinalize` previously emitted this on the merge lane
  because it did not close. Now it emits `closed` (online success) or `already_closed`
  (already closed before cmdFinalize ran). `close_pending` is still emitted under the pre-sink
  qualifier path (`close_disposition: close_pending`) documented in the D2 section of the
  Closure Contract.

**For the test suite:**

- Tests that expected `remote_issue_closed: close_pending` from `cmdFinalize` on the default merge
  lane need updating to expect `closed` or `already_closed`.
- Walkthrough scenarios added for the idempotent path: `cmdFinalize` → `sink-merge` both succeed
  with `cmdFinalize` recording `closed` and `sink-merge` recording `already_closed`.
