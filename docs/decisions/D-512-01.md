# D-512-01. Parameterize run-chains spawnSync timeout via KAOLA_RUN_CHAINS_TIMEOUT_MS

Date: 2026-06-16
Status: Accepted
Issue: #512
Related: #500 (observed false-kill at finalize; `--accept-known-red` waiver used as workaround),
         D-500-01 (existing) (makespan-lever decision that surfaced the ~574s chain measurement),
         `scripts/kaola-workflow-run-chains.js` (`resolveTimeoutMs` helper, `:272` spawnSync usage).

## Context

The claude edition test chain was measured at ~574s standalone. The prior hardcoded `spawnSync`
kill ceiling was 600000ms (10 min), leaving a margin of only ~26s. In practice the chain was
being false-killed at finalize, producing a spurious `chains_red` refusal and requiring an
`--accept-known-red claude:<issue>` waiver as a workaround (observed in the #500 run — see
`project_500_run_lessons.md` memory entry).

The root cause of the ~574s runtime (suite growth vs. environment slowdown) is not yet diagnosed.
Speeding up the chain without knowing the root cause would be a larger, uncertain effort.

## Decision

**Chosen fix:** Parameterize the per-chain `spawnSync` kill ceiling via the environment variable
`KAOLA_RUN_CHAINS_TIMEOUT_MS`. The default is raised from the prior hardcoded 600000ms (10 min)
to 900000ms (15 min). A non-numeric, zero, or negative value falls back to the 900000ms default
(mirrors the `KAOLA_GH_REMOTE_TIMEOUT_MS` parse-or-default pattern). **No upper clamp** is applied:
a long-running local test suite is not a remote-hang risk, in contrast to `KAOLA_GH_REMOTE_TIMEOUT_MS`
where the #185 hang-protection clamp guards against excessively large values silently disabling
API-call protection.

**Deferred:** The "speed up the chain" option. The ~574s slowness is not yet root-caused —
whether it is suite growth or an environment effect is unknown. Speeding up an undiagnosed suite
is a larger, uncertain effort relative to raising the ceiling. The owner can reopen the speed-up
track if they disagree; this is a deferral on the cheapest-sufficient directive, not a value
escalation.

## Implementation

`resolveTimeoutMs(env)` was added to `scripts/kaola-workflow-run-chains.js`:

```js
function resolveTimeoutMs(env) {
  const v = parseInt((env && env.KAOLA_RUN_CHAINS_TIMEOUT_MS) || '', 10);
  return (Number.isFinite(v) && v > 0) ? v : 900000;
}
```

The function is exported as `{ resolveTimeoutMs }` and consumed at the `spawnSync` call site
(`:272`). The `.cache/chain-receipt.json` receipt schema is unchanged — this knob governs only
the kill ceiling.

Cross-edition lockstep is enforced by `validate-script-sync.js`, which asserts byte identity of
`kaola-workflow-run-chains.js` across all four editions (GitHub, Codex, GitLab, Gitea).

## Consequences

- False `chains_red` kills at finalize are eliminated for chains running up to 15 minutes.
- Operators running unusually long suite variants may set `KAOLA_RUN_CHAINS_TIMEOUT_MS` to a
  higher value without rebuilding.
- The receipt schema and `--finalize-check` gate contract are unchanged.
- The root-cause investigation of the ~574s claude chain runtime is a deferred follow-up.
