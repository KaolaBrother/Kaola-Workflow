This record covers **Part 1** (the autopilot loop that crosses the stage boundaries
scout→claim→plan→run→finalize) and **Part 3** (the optional `goal:` line in `## Meta`)
together, because they form one coherent "driver + termination" cluster: the goal line
(Part 3) supplies the autopilot's (Part 1) primary stop condition, and the autopilot is the
only consumer that needs goal-satisfied to beat backlog-empty. Parts 2 (enriched halt
payload) and 4 (release aggregator) are recorded in D-420-02.

## Context

Today every stage of the Kaola-Workflow pipeline is human-driven through user-facing
commands; nothing chains the stages automatically. The pipeline's concrete stages and the
scripts that own them are enumerated in the grounding investigation §1.1
(`docs/investigations/2026-06-12-goal-driven-automation-design.md`):

1. **Survey (read-only).** `commands/workflow-next.md` (the router) may dispatch the
   read-only `issue-scout` agent (`agents/issue-scout.md`) to recommend ONE bundle before
   any claim, mapping `scout.recommended_bundle.primary_issue` into `KAOLA_TARGET_ISSUE`
   (`commands/workflow-next.md:143-147`, the #380 env-wiring block).
2. **Claim.** `scripts/kaola-workflow-claim.js` `cmdStartup` (line 1193) →
   `claimExplicitTarget` (line 805) → `claimProject` (line 665).
3. **Plan.** `workflow-planner` authors `workflow-plan.md`, then `adaptive-handoff.js`
   freezes it (SPAWN 1 `--freeze-checked`, line 268; SPAWN 2 `--freeze --governance-ack`,
   line 328), returning `handoff_status: 'ready_to_run'` (line 464).
4. **Run.** `commands/kaola-workflow-plan-run.md` drives `adaptive-node.js` per node.
5. **Finalize + sink.** `commands/kaola-workflow-finalize.md` + the contractor archive +
   `scripts/kaola-workflow-sink-merge.js` (`main`, line 609) merge, close the issue
   (`gh issue close`, the only issue-close call, `sink-merge.js:442`/`486`), remove the
   roadmap source file, and emit a receipt.

State threads between stages through three durable artifacts (investigation §1.1):
`workflow-state.md`, `workflow-plan.md` (the hash-covered `## Meta` + `## Nodes` plus the
mutable `## Node Ledger`), and the `.cache/` evidence + baseline files. **There is no script
today that owns *crossing* the stage boundaries — that is the orchestrator's job, invoked
command by command.** The autopilot closes that loop.

The grounding's recurring finding (investigation §0) is that every primitive #420 needs
already exists: the scout already emits a `confidence` gating signal
(`agents/issue-scout.md:112`), every claim/handoff/node refusal is already typed (#406 emit
envelope), and `computePlanHash` already hashes the WHOLE `## Meta` body
(`plan-validator.js:682-685`). The missing work for this cluster is a *driver* (Part 1) and
*one additive Meta field* (Part 3) — no new core mechanics.

Parts 1+3 cover the "driver + termination" cluster: Part 1 is the loop that crosses stage
boundaries; Part 3 is the optional goal line that gives the loop a clean terminal condition.

## Decision — Part 1: Autopilot Loop

**Decision (P1):** An **orchestrator-level** autopilot iterates the full pipeline
(scout → claim → plan → run → finalize) by SHELLING the existing user-facing commands and
branching STRUCTURALLY on each stage's typed result. It introduces NO new core mechanics: it
is the generalization of the bounded repair loop the orchestrator already drives on
`plan_invalid` (per CLAUDE.md adaptive-handoff bullet) to the whole pipeline, plus a *digest*
surface that accumulates the per-stage typed results so the user reviews one rolling report
instead of re-typing `/workflow-next` each cycle (investigation §1.4).

**Part 1 invariants:**

- **[INV-1] (selection-aloud, the #44 invariant):** The autopilot MUST state the selected
  issue (or bundle) aloud — auditably, the same selection that lands in the claim comment via
  `postAdvisoryClaim` (`claim.js:577-588`) — BEFORE calling `cmdStartup --target-issue N`.
  Selection is an agent decision, never a hidden script decision (CLAUDE.md "Agent Owns
  Reasoning; Scripts Own Atomicity"). The autopilot never silently picks: `cmdStartup` itself
  refuses an auto-pick with a typed `no_target` whose reasoning states "the workflow never
  auto-picks an issue (#44)" (`claim.js:1225-1226`), so the autopilot must supply the target
  it has already announced.

- **[INV-2] (confidence-threshold gate):** The scout's `confidence: "high"`
  (`agents/issue-scout.md:112`) is the ONLY value that lets the autopilot proceed to claim
  without a user confirmation. `medium` / `low` parks the loop with a typed `confidence_halt`
  payload for user review — never a silent proceed, never a silent skip. This preserves the
  scout's own rule that when confidence is not high it must recommend single-issue mode and
  defer the proceed/adjust/fallback decision to the orchestrator (`agents/issue-scout.md:84`,
  `118`). The proceed predicate is `confidence == "high"` AND no active-state conflict AND no
  ambiguity (investigation §1.4).

- **[INV-3] (typed stop conditions, structural not string-matched):** Every loop exit is one
  of a CLOSED set of typed payloads, branched on STRUCTURALLY per the #406 emit-envelope
  contract — the autopilot never string-matches a message. The stop set:
  - `goal_satisfied` — the frozen `## Meta` goal line (Part 3) is judged met at the sink gate
    (the primary terminal; see [INV-7]).
  - `backlog_empty` — the scout returns no candidate bundle.
  - `confidence_halt` — scout confidence below `high`, or a `target_ambiguity`
    (`claim.js:1201`) / active-state conflict, needs user review ([INV-2]).
  - `error_halt` — any stage emitted a typed refusal the autopilot is not authorized to repair
    autonomously (`no_target`, `user_target_red`/`user_target_blocked`/`target_unavailable`/
    `target_unverified` from `claimExplicitTarget` `claim.js:811-828`; `plan_invalid` after the
    bounded repair budget is exhausted, `adaptive-handoff.js`; a verdict gate emitting
    `verdict: fail`).
  - `consent_halt` — a node's `runWriteHalt` (`adaptive-node.js:1454`) recorded a
    `consent` / `security` / `test_thrash` escalation; the halted node STAYS `in_progress`
    (`adaptive-node.js:1542-1543`) and the loop parks for the operator.

- **[INV-4] (never-silent):** EVERY loop exit — including the success exit — produces a typed
  payload (one of [INV-3]'s set) and is recorded in the digest. There is no code path on which
  the autopilot exits with no emission. A silent `process.exit(0)` on backlog-empty or
  goal-satisfied is a violation; the terminal is always a `{ stop: <reason>, ... }` envelope.

- **[INV-5] (subagent-cannot-dispatch-subagent boundary):** The autopilot driver lives at the
  ORCHESTRATOR level (the main session loop), NOT inside any subagent. Subagents cannot
  dispatch subagents (#242 run-lesson), and the pipeline's stages REQUIRE subagent dispatch:
  the scout, the `workflow-planner`, and the per-node role agents are all dispatched by the
  main session. A `kaola-workflow-autopilot.js` aggregator MAY exist to emit typed per-stage
  results and own the digest accumulation, but the actual `Task`/dispatch of scout, planner,
  and role agents MUST stay in the orchestrator — the aggregator computes, the orchestrator
  dispatches. (Whether the autopilot is an aggregator or pure orchestrator prose is an open
  question, OQ-1.)

- **[INV-6] (repair-consent boundary):** The #44 invariant covers *selection*, not *repair*.
  The autopilot's authority to auto-apply a stage repair (e.g. a `write_set_granularity`
  write-set swap + `--freeze`, investigation §2.3) is governed by an EXPLICIT repair-consent
  policy, NOT silently assumed. Absent an explicit operator opt-in, a repair-eligible halt
  parks as `consent_halt` for a yes/no rather than auto-continuing. (The exact policy — auto-
  apply mechanical repairs vs. always park — is OQ-2, deferred to implementation.)

## Decision — Part 3: Goal-Conditioned Bundles

**Decision (P3):** Add an OPTIONAL `goal: <text>` line to the plan's `## Meta` section. It is
hash-covered for free, read (never enforced at freeze) by a new `parseGoal` reader mirroring
`parseLabels`, flows from the user through the scout and planner into the frozen plan, is
checked at the single sink gate as an AC-vs-goal attestation, and serves as the autopilot's
(Part 1) primary termination signal. This is ONE additive Meta field plus ONE reader — no
hash code change, no validator gate, no new freeze spawn (investigation §3.2, §3.3).

**Part 3 invariants:**

- **[INV-7] (goal-satisfied beats backlog-empty):** When a frozen `goal:` line is present, the
  autopilot's primary terminal is `goal_satisfied` ([INV-3]), upgraded from `backlog_empty`.
  At the sink gate, the closed bundle's acceptance criteria are compared against the frozen
  `goal:` line; a met goal terminates the loop cleanly even if the backlog still holds
  unrelated issues. Backlog-empty remains the terminal only when no goal line was set
  (investigation §1.4, §3.3).

- **[INV-8] (hash-coverage for free):** Because `computePlanHash` normalizes and hashes the
  ENTIRE `## Meta` section body (`plan-validator.js:682-685`: `body = norm('Meta') +
  '\n---NODES---\n' + norm(NODES_HEADING)`), a `goal:` line inside `## Meta` is automatically
  covered with NO hash code change. Tampering the goal after freeze trips `plan_hash_mismatch`
  on `--resume-check` (`plan-validator.js:1234`), exactly as tampering `labels:` does
  (`plan-validator.js:679-681`). The goal inherits the same anti-tamper guarantee as
  `labels:`.

- **[INV-9] (backwards compatibility — reader not gate):** The validator gains a `parseGoal`
  READER mirroring `parseLabels` (same `^goal:[ \t]*(.*)$` shape, same `## Meta`-scoped read
  via `sectionBody(content, 'Meta')`, `plan-validator.js:128-132`, `698-702`); it does NOT
  gain a goal GATE. The goal is metadata like the `model`/`selector_source` columns
  (`plan-validator.js:163-166`). Old frozen plans WITHOUT a `goal:` line stay valid and
  hash-stable: an absent optional line changes nothing in the normalized `## Meta` body.
  No validator gate is added; no existing plan is invalidated.

- **[INV-10] (`## Meta`-scoped read, decoy-immune):** Every consumer (the finalize AC-vs-goal
  check, the autopilot terminal) reads the goal from the SAME `## Meta`-scoped reader, so it
  sees the hash-covered value, never a decoy `goal:` line placed elsewhere in the plan. This
  reuses the #B1 audit scoping that confined `parseLabels` to the `## Meta` region precisely so
  an out-of-region decoy cannot override it (`plan-validator.js:698-701`).

- **[INV-11] (goal flows to scout for bundle selection):** The user-supplied goal reaches the
  scout as CONTEXT for bundle clustering — a new env (e.g. `KAOLA_GOAL`) the scout reads and
  clusters against, paralleling the existing scout-output → `KAOLA_TARGET_ISSUE` env wiring
  (`commands/workflow-next.md:143-147`, #380). The scout clusters by goal alignment, not just
  shared scope, and the `workflow-planner` then transcribes `goal: <text>` into `## Meta` at
  authoring time alongside `labels:`. The goal is frozen as part of the EXISTING
  `--freeze`/`--governance-ack` transaction (`adaptive-handoff.js:328`) — no new freeze spawn.

- **[INV-12] (goal is prose, agent-judged at one seam):** The goal is free prose (mirroring
  `labels:`), NOT a structured AC grammar the validator mechanically parses. The finalize
  AC-vs-goal check is therefore an AGENT-judged attestation surfaced as a typed finalize
  result — it presents the goal + the closed-issue ACs for a verdict, mirroring how the
  verdict gates work (`VERDICT_ROLES`, `adaptive-node.js:543`). It lives at the SINGLE sink seam
  `sink-merge.js` already owns (the close-loop / `deriveMemberSet` at `sink-merge.js:636`),
  keeping the goal audit at one place. (Prose vs. structured ACs is OQ-3, with D-420-02 owning
  the final call per investigation §5.2.)

## Open Questions

- **OQ-1 (where the autopilot lives, [INV-5]):** A new `kaola-workflow-autopilot.js`
  aggregator that emits typed per-stage results and owns the digest, OR pure orchestrator prose
  in a command / SKILL pack? An aggregator can emit typed results and accumulate the digest,
  but the dispatch of subagents (scout, planner, role agents) is irreducibly the orchestrator's
  job (#242). The boundary must be fixed at implementation: which computations move into an
  aggregator, and which dispatch/branch logic stays as orchestrator prose. (Investigation §5.5.)

- **OQ-2 (repair-consent policy, [INV-6]):** Once D-420-02's enriched halt attaches a
  ready-made plan-repair diff, should the autopilot AUTO-APPLY a mechanical repair
  (`write_set_granularity` / lockfile / generated-mirror) and continue, or ALWAYS park for the
  yes/no? The #44 invariant covers selection, not repair, so this is an explicit policy choice,
  not a default. (Investigation §5.1.)

- **OQ-3 (goal form — prose vs. structured, [INV-12]):** Is `goal:` a single prose line
  (agent-judged at the sink gate, as decided here) or a structured AC list the validator can
  mechanically check? The grounding supports prose (it mirrors `labels:`); a structured form
  would need a grammar and a new validator gate, contradicting [INV-9]. D-420-02 should make
  the final call. (Investigation §5.2.)

- **OQ-4 (digest persistence and resume):** Where does the per-stage digest live — a durable
  `.cache/autopilot-digest.md`, or transient session state? If the autopilot session crashes
  mid-loop, can it resume the digest, and how does it re-derive which stage the active project
  is in (from `workflow-state.md`'s `phase`/`workflow_path` + the ledger)? Resume continuity is
  unspecified here.

- **OQ-5 (multi-issue bundle vs. goal scope):** When the scout recommends a bundle (up to
  `KAOLA_BUNDLE_MAX_ISSUES`, default 4, `agents/issue-scout.md:82`) under a goal, is
  `goal_satisfied` evaluated per-bundle at each sink, or across multiple sequential bundles
  until the goal is globally met? The single-sink AC-vs-goal check ([INV-12]) is per-bundle;
  whether the autopilot loops to a SECOND bundle after a partial goal is open.

## Consequences

**What becomes easier:**

- The user reviews ONE rolling digest of typed per-stage results instead of re-typing
  `/workflow-next` each cycle (investigation §1.4). The autopilot crosses the stage boundaries
  no single script owns today.
- `goal_satisfied` ([INV-7]) gives the loop a clean, auditable terminal grounded in the frozen,
  hash-covered `## Meta` region — strictly better than "backlog empty," which never knew when
  the user's actual objective was met.
- The goal line costs nothing to add ([INV-8]): no hash code change, no validator gate, no new
  freeze spawn. Old plans stay valid ([INV-9]).

**What becomes harder / what to watch:**

- The autopilot multiplies the blast radius of any single typed-refusal misclassification: a
  stage refusal mis-branched would either silently skip work or wrongly park the loop. The
  never-silent invariant ([INV-4]) and structural branching ([INV-3]) are the guardrails, but
  the loop's correctness depends on every stage's `reason` field being stable (#406).
- The repair-consent boundary ([INV-6]) is a new place the #44 "agent owns reasoning" rule must
  be defended — auto-applying repairs without consent would erode the human-in-the-loop the
  workflow is built on.
- A `goal:` line is forge-neutral metadata, but the env that carries it to the scout
  (`KAOLA_GOAL`, [INV-11]) and the finalize AC-vs-goal check ([INV-12]) live in surfaces that
  must stay forge-neutral and may need to propagate across the four editions (the cross-edition
  contract, CLAUDE.md Validation Policy) — to be assessed at implementation.

**What is deferred to implementation issues:**

- The autopilot's home (aggregator vs. prose) — OQ-1.
- The repair-consent policy — OQ-2.
- The goal's final form (prose vs. structured) — OQ-3, owned by D-420-02.
- Digest persistence / crash-resume — OQ-4.
- Multi-bundle goal evaluation — OQ-5.

**For downstream issues:** the invariants binding every Part-1 implementation are [INV-1]
through [INV-6] (selection-aloud + confidence gate + typed/never-silent stops + orchestrator-
level boundary + repair-consent); the invariants binding every Part-3 implementation are
[INV-7] through [INV-12] (goal-satisfied terminal + hash-coverage-for-free + reader-not-gate
backwards compat + `## Meta`-scoped decoy-immunity + goal-flows-to-scout + prose/single-seam).
D-420-02 (Parts 2+4) continues the invariant numbering from [INV-13].
