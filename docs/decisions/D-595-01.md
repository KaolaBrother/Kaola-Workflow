# D-595-01. Unlink the caller's own just-created lockfile on a payload-write failure

Date: 2026-07-02
Status: Accepted
Issue: #595
Related: D-585-01 (the scheduler mutual-exclusion lock this decision hardens; its
             "Residual accepted risks" section named this exact orphan window as an
             accepted, non-blocking risk — this decision closes it)

## Context

`acquireProjectLock` (`kaola-workflow-adaptive-schema.js`, the byte-identical-×4 drift
anchor) claims `kaola-workflow/{project}/.cache/scheduler.lock` via `fs.openSync(lockPath,
'wx')` (O_EXCL | O_CREAT), then writes a JSON payload (`{pid, host, ts, subcommand}`) and
`fsync`s it before returning `{ok:true, release}`. The held-lock marker
(`_heldSchedulerLock`) — the flag both the caller's `finally` release and the module-level
`process.on('exit')` backstop consult — is set only AFTER that full write+fsync+close
sequence succeeds.

If the payload write or the fsync throws immediately after the `openSync('wx')` succeeded,
the outer `catch` closed the file descriptor (`fs.closeSync(fd)`) but left the just-created,
now-empty lockfile ON DISK. Because the held-lock marker was never set, neither cleanup path
ever removed it. A follow-up `acquireProjectLock` call in the same process (or a later
invocation) then hit `EEXIST` against that orphaned, empty file. `isStaleLock` classifies a
payload it cannot parse as JSON by falling back to the lockfile's mtime rather than a parsed
holder identity; a just-created empty file reads as fresh, so the follow-up acquire was
refused `scheduler_locked` (live holder) — the wrong flavor, since there is no live holder at
all — until `LANE_STALENESS_MS` (24h) elapsed and the mtime fallback finally reclassified it
`scheduler_lock_stale`. A transient write failure (disk pressure, a killed process mid-write,
a filesystem hiccup) could therefore brick a project's scheduler for up to a day, requiring
either the wait or an operator's manual `rm` — even though the fault was entirely local to
the acquiring process and no other holder was ever involved.

D-585-01 named this exact window explicitly as a residual accepted risk
("Create-then-write-failure orphan window") when the scheduler lock first shipped, on the
grounds that it was low-severity (a wrong-flavor refusal, not a double-acquire) and shared
the same manual `rm` recovery as the other residual risks. This decision closes that
specific window.

## Decision

Wrap the payload write in the existing outer `catch (err)` block of `acquireProjectLock`:
when `fd !== undefined` — which is true if and only if THIS call's own `openSync(lockPath,
'wx')` succeeded a few lines above, and is reset to `undefined` only after the full
write+fsync+close sequence completes — best-effort `fs.closeSync(fd)` then best-effort
`fs.unlinkSync(lockPath)` before rethrowing `err`. Previously this branch only closed the
descriptor and left the file on disk.

**No-takeover invariant preserved.** The new `unlinkSync` call is reachable ONLY inside the
`fd !== undefined` branch, and `fd` can only ever be set to a file THIS call created via its
own `'wx'` open:

- The pre-existing `EEXIST` branch (a foreign holder already owns the lock) is guarded by
  `if (!(err && err.code === 'EEXIST')) throw err;` immediately after the `openSync` call —
  it never sets `fd` and never reaches the new unlink code at all. A caller can only observe
  `EEXIST` when `openSync('wx')` itself failed, so `fd` is never assigned to a foreign
  holder's file.
- There is no code path from a stale-lock CLASSIFICATION (`isStaleLock`) to this unlink —
  `acquireProjectLock` never auto-removes another process's lock (that auto-takeover design
  was rejected in D-585-01 for the same reason: a verdict computed before an unlink can race).
  This decision does not reopen that question; it only ever removes a file the CURRENT call
  provably created moments earlier in the SAME synchronous sequence.

RED-first proof: a new `T-595-orphan` unit test in `scripts/test-adaptive-node.js`
monkey-patches `fs.writeFileSync` to throw once immediately after a real
`openSync(lockPath, 'wx')` succeeds, then asserts (against the pre-fix code) that the
lockfile is orphaned and a follow-up acquire in the same process is wrongly refused. Against
the fixed code the same harness asserts the lockfile is gone after the injected failure and
the follow-up acquire succeeds cleanly (4 new assertions). Every pre-existing T-585 lock test
(acquire/release, live-refuse, stale-refuse, the real-subprocess stale-race harness) stayed
green with no regression — the planted dead-pid holder fixtures those tests use are
byte-untouched, confirming the new unlink cannot reach a foreign lock.

## Consequences

- A transient payload-write failure (the process's own write/fsync erroring) now leaves NO
  orphaned lockfile: the very next `acquireProjectLock` call — in the same process or a
  fresh one — acquires cleanly instead of waiting out `LANE_STALENESS_MS` or requiring an
  operator `rm`.
- The D-585-01 residual-risk list's "Create-then-write-failure orphan window" entry is
  resolved; the remaining residual risks there (PID reuse yielding a wrong-flavor refusal;
  operator-misuse cross-unlink) are unaffected by this change and remain accepted as before.
- The fix is purely additive inside the existing `catch` block — the `EEXIST` branch, the
  `release()` `finally`, the module-level exit hook, and the stale-PID classification probe
  are all untouched. Zero-contention, successful-write behavior (the overwhelming common
  case) is byte-identical.
- Cross-edition: `kaola-workflow-adaptive-schema.js` is the byte-identical-×4 drift anchor
  (canonical name in all four trees, not forge-renamed) — the fix was applied to the
  canonical file and byte-copied to the Codex twin and both forge ports, confirmed via `cmp`.

## Alternatives considered

- **Verify-after-write instead of unlink-on-catch.** Re-read the lockfile after the write
  completes and unlink it if the content does not match what was just written. Rejected as
  strictly more complex for no additional safety: the failure this decision targets is a
  THROW during the write/fsync itself (the write never completed to be verified), and the
  ownership argument (only THIS call's own just-created `fd` is ever eligible) is already
  sufficient — no confirmation read is needed to prove ownership.
- **Retry the write in a loop before giving up.** Rejected: retrying does not change the
  correctness picture (a persistent failure still needs to release the file so a later
  attempt — by this process or another — is not blocked), and it reintroduces the
  possibility of masking a real, non-transient disk/filesystem fault behind a spin loop.
  Failing fast and leaving no orphan is the simpler, sufficient fix.
- **Auto-takeover of an orphaned/empty lockfile by a LATER caller instead of self-cleanup
  by the ORIGINAL caller.** Rejected on the same grounds D-585-01 already rejected general
  stale-lock auto-takeover: any code path that lets a caller unlink a lock it does not
  itself hold reopens a takeover race. Cleaning up the file from INSIDE the call that just
  created it needs no such path at all — it is strictly safer and simpler.
