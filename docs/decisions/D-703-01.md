## Context

Kimi Code is a coding-agent **runtime** (like Codex and opencode), not a git forge. Adding it
as a fourth runtime had to respect the additive boundary D-530-02 locked for opencode, while
adapting to three hard constraints of the Kimi integration surface that neither opencode nor
Codex have in this combination:

1. **No named custom subagents.** Kimi Code's Agent tool supports only three built-in
   subagent types — `coder` (full tools), `explore` (read-only), `plan`. The 16 canonical
   roles cannot ship as named agent definitions (the Claude `agents/*.md`, Codex `.toml`, or
   opencode `.opencode/agent/*.md` shape). Kimi's extension surface is **directory-form
   Skills** (`<name>/SKILL.md`, requiring explicit `name` + `description`), which Kimi scans
   from the kimi-home and project skills dirs and auto-registers as slash commands.
2. **No per-dispatch model or effort override.** Every subagent inherits the session model.
   opencode's `mapTier` two-tier effort-variant machinery has no runtime surface to bind to
   on Kimi — there is no variant knob, no per-call model parameter, nothing a seeded config
   could influence.
3. **Hooks are TOML and global-only.** Kimi reads `[[hooks]]` rules only from the global
   `${KIMI_CODE_HOME:-$HOME/.kimi-code}/config.toml`; there is no project-scoped hooks
   config and no `SessionStart"compact"` event (its semantic counterpart is `PostCompact`).

## Decision

Add the kimi edition as an **additive runtime edition** with its own generator
(`scripts/sync-kimi-edition.js`), installer (`install-kimi.sh`), and test
(`scripts/test-kimi-edition.js`, 545 assertions). It is **NOT** wired into `npm test`,
**NOT** added to `edition-sync.js`, **NOT** inserted into `install.sh`, and **NOT** part of
the SIX routing surfaces. Canonical `agents/` / `commands/` / `hooks/` and the reviewer
adapters are untouched. Four load-bearing choices:

1. **Roles as Skills.** Each canonical role ships as a role-contract Skill
   `kaola-role-<role>`. Every canonical dispatch card (`Agent(subagent_type="<role>",
   model="{...}", …)`) is rewritten at generation time to the built-in type for the role's
   kind — read-only roles (canonical frontmatter `tools:` lacking Write/Edit) → `explore`,
   everything else → `coder`, with the kind map computed from canonical, never hand-listed —
   plus a prompt-prefix instruction: *"First invoke the `kaola-role-<role>` Skill and follow
   its contract for the entire task."* Role-contract fidelity is thereby a **dispatch-prompt
   contract** backed by a generated Skill, not a runtime-enforced agent definition.
2. **Inherit-only model tier.** The Codex inherit precedent, not the opencode `mapTier` one:
   no Reasoning/Standard mapping, no seeded variant config, no `model:` fields in generated
   skills. The canonical `model: opus` markers and `agents/profiles/higher/` are skipped;
   the planner's per-node `reasoning`/`standard` tier survives as **metadata only** (recorded
   in the dispatch packet/ledger; `modelDisplay()` renders `parent session (<tier> tier
   metadata)`), mapping to nothing at runtime.
3. **Hooks event mapping with the difference documented.** Canonical `hooks/hooks.json` is
   re-expressed as a generated `[[hooks]]` TOML fragment (`kimi-hooks.toml`): `PreToolUse`
   Bash / `Write|Edit` carry over directly (Kimi matchers are regexes), `SubagentStart` drops
   its matcher, and `SessionStart"compact"` maps to **`PostCompact`** — accepted with the
   semantic difference (fires after a compaction completes, not at a session start following
   one) deliberately documented in `docs/kimi-edition.md` rather than hidden. The installer
   merges the fragment into the global `config.toml` as an idempotent managed block and
   validates with `kimi doctor config`, rolling back on failure. Smoke testing against
   kimi-code 0.26.0 surfaced a second, deeper incompatibility — **payload field names**:
   Kimi's `PreToolUse` Write|Edit carries `tool_input.path` (Claude: `file_path`) and
   `SubagentStart` carries `agent_name` (Claude: `agent_type`), which would leave the
   byte-copied write-lane/dispatch-log scripts silently fail-open (inert). The generator
   therefore emits **payload-adapted copies** of those two scripts (anchored single-string
   rewrites that also accept the Claude field; hard error on a drifted anchor), while
   pre-commit (`tool_input.command`) and compact-context (`cwd`) stay byte-identical.
4. **Global hooks activation acknowledged, not worked around.** Because Kimi has no
   project-scoped hooks config, a project-level install still merges the managed block into
   the global `config.toml` — the hooks activate for every Kimi session on the machine. The
   scripts stay fail-open and operate on the cwd, so the effect remains
   project-scoped; only the activation surface is global. The difference from
   opencode/Claude is documented, not concealed.

Parity is enforced by **manual invocation** — `node scripts/test-kimi-edition.js` (plus
`sync-kimi-edition.js --check` for drift) on any change touching canonical or the kimi tree
— and by the installer's precheck (`--check`, falling back to `--write`), following the
D-530-02 posture.

## Alternatives considered

- **Ship roles as named agent definitions anyway.** Impossible, not merely rejected: Kimi's
  Agent tool has no named custom subagent surface. The only carriers for role contracts are
  Skills (chosen) or plain files a subagent is told to Read (recorded as the fallback if
  Skill invocation proves unreliable in practice — a dispatch-prompt change only, still zero
  canonical impact).
- **Port the opencode `mapTier` two-tier effort mapping to Kimi.** Rejected: there is no
  per-dispatch model/effort override surface for it to bind to. Seeding a variant-shaped
  config no runtime honors would be speculative machinery that silently de-tiers on day one.
  Inherit is the only honest mapping; the tier-as-metadata recording preserves ledger parity
  with the other editions.
- **Wire `test:kaola-workflow:kimi` into `npm test`** (a fifth chain). Rejected per
  D-530-02's reasoning, which applies verbatim: CLAUDE.md names exactly four forge chains and
  scopes the Cross-edition Validation Policy and the SIX surfaces to forge trees; #501 keeps
  correctness inward; coupling a non-forge edition's checks to forge Finalization is the
  unacceptable failure mode. The kimi suite is run directly and is documented in
  `docs/kimi-edition.md`, `README.md`, and CLAUDE.md's additive rule.
- **Emulate `SessionStart"compact"` or drop the resume-state hook.** Rejected: no emulation
  surface exists, and dropping the hook would lose resume-state injection on a runtime where
  context compaction is routine. `PostCompact` is the documented semantic counterpart and is
  close enough for the compact-context script's purpose.
- **Work around the global-only hooks config** (e.g. per-project shims). Rejected: there is
  no project hooks surface to shim into; any workaround would be a second, non-native config
  channel Kimi never reads. The accepted posture is honest documentation of the global
  activation surface.

## Consequences

- **The additive boundary is preserved for a second runtime.** The forge machinery
  (`edition-sync.js`, `install.sh`, the four-chain `npm test`, the SIX-surfaces T-sets) stays
  forge-scoped; no #307 four-chain obligation attaches to kimi-only diffs. Contributors run
  `node scripts/test-kimi-edition.js` instead — the suite regenerates the tree itself and
  covers structure, residue, byte-parity, zero-leakage, dispatch rewrite, reviewer behavior
  identity, hooks, route reachability, and a hermetic real-installer block.
- **Role fidelity rests on a prompt-level contract.** A Kimi subagent that ignores the
  "invoke the Skill first" instruction runs as a bare `coder`/`explore`. This is the accepted
  cost of a runtime without named subagents; the K5 assertions keep the instruction
  provably present and correctly paired per role, and the Read-mode fallback is recorded if
  smoke experience shows it is needed.
- **Model-tier semantics collapse to one tier on Kimi.** Plans carrying
  `reasoning`/`standard` nodes run entirely at the session model; the tier is ledger/audit
  metadata. Cross-edition ledger comparison stays meaningful because the tier token is still
  recorded.
- **Hooks activate machine-wide on any install scope**, including `--target` project
  installs — a documented, deliberate difference from the opencode and Claude editions.
  `--no-scripts` opts out of the merge.
- **`PostCompact` ≠ `SessionStart"compact"`** is an accepted semantic gap for the
  resume-state hook, recorded here and in the edition doc so a future Kimi event-model
  change re-evaluates the mapping deliberately.
- **No canonical change ships from this decision.** The four existing `npm test` chains are
  untouched and must stay green; the kimi suite (`node scripts/test-kimi-edition.js`, 545
  assertions) is green at the time of writing.
