# D-546-G2. File-disjoint shared-infra write lanes co-open by default under the retained gate+PROTECTED net

Date: 2026-06-21
Status: Accepted
Issue: #546
Supersedes-in-part: docs/decisions/D-542-01.md (the "shared-infra overlap ⟹ consent" stance — i.e.
                    the classification clause that lumped a file-disjoint shared-infra frontier in
                    with the coarse/overlap consent path).
Related: #463 (per-leg worktree isolation — the containment primitive),
         #498 (the co-open ⟹ legs-provisioned invariant: groupForm ⟺ legCoupled ⟺ legs),
         #472 (faithful decomposition — fan out exactly as wide as the work decomposes),
         `docs/decisions/0010-runtime-neutral-per-leg-worktree-isolation.md` (the per-leg worktree mechanism),

         #538/D-542-01 (the same "retire the per-run toggle when the structural net already covers it"

D-542-01 retired the four-way activation conjunction for the *planner-proven-disjoint* (`parallel_safe`)
case, but its classification clause still routed a **shared-infra** overlap to the consent path
("disjoint ⟹ default co-open; coarse/shared-infra overlap ⟹ consent; exact-file overlap ⟹ refuse").
This record supersedes that one clause only — the file-disjoint shared-infra sub-case. D-542-01's
kernel model, per-leg mechanism, the #498 invariant, the genuine-overlap consent gate, and the
exact-file refuse all stand unchanged.

## Context

The validator's `writeOverlapRelaxable` predicate (`scripts/kaola-workflow-plan-validator.js`)
classifies an overlapping write pair into `dj.kind ∈ {exact, coarse, shared-infra}`. The
**`shared-infra`** class is a frontier whose write sets land in the SAME shared area (e.g. two files
under `scripts/`) but are **exact-file-disjoint** — no two legs declare the same concrete file.

Under the D-542-01 stance, that file-disjoint shared-infra frontier still required the operator to opt
in (`write_overlap_policy:coarse` plus the per-run `--write-overlap-consent` carrier) before it could
co-open. But the consent ceremony added nothing the structural net does not already guarantee:

- The write sets are **exact-file-disjoint by construction** — there is no shared concrete file to
  clobber.
- A post-dominating `synthesizer`/`code-reviewer` gate covers the merged union (NET-1), so the
  review/chain net sees every change.
- The per-leg git worktree (#463 / ADR-0010) contains each leg, and the per-leg barrier still catches
  any textual conflict at the merge commit.

So the consent gate was withholding a proven-safe default behind manual ceremony no ordinary run
performs — the same over-caution D-538/D-542 already identified and corrected for the disjoint case.
This conflicts with the design precedence (CLAUDE.md): accuracy is non-negotiable (#1), but **then**
automation & efficiency (#2); a gate the structural net already provides is not buying correctness.

## Decision

**A file-disjoint SHARED-INFRA write frontier (same `SHARED_INFRA` area, exact-file-disjoint) co-opens
BY DEFAULT under the RETAINED net — NO `write_overlap_policy:coarse` and NO `--write-overlap-consent`.**
Consent and the `write_overlap_policy` tier now gate ONLY a genuine NON-shared coarse-area overlap.
Exact-file overlap stays blocking, unchanged.

Concretely, in `writeOverlapRelaxable(dj, setA, setB, policy, consent, gatePresent)`:

- **The retained net gates EVERY class at EVERY tier.** (NET-1) a `synthesizer`/`code-reviewer` gate
  post-dominates the relaxed legs themselves (the caller proves it via leg-scoped `gateUncovered` over
  the `--nodes` set); (NET-2) NEITHER set contains a PROTECTED concrete file (`classifier.isProtected`
  — lockfiles / `CHANGELOG.md` / `ROADMAP.md` / install manifests / archive-roadmap artifacts / the
  `kaola-workflow-adaptive-schema.js` ×4 anchor stay blocking at every tier).
- **`shared-infra` ⟹ relax by default.** Once the retained net holds, `dj.kind === 'shared-infra'`
  returns `true` independent of `policy` and `consent`.
- **`coarse` (non-shared) ⟹ unchanged consent gate.** It relaxes ONLY when the tier authorizes the
  class (`write_overlap_policy ∈ {disjoint, coarse}`) AND the per-run `--write-overlap-consent` carrier
  is present AND the retained net holds. A genuine non-shared overlap is a real relaxable case and
  stays opt-in.
- **`exact` (or any future class) ⟹ never relaxes here.** Real reconciliation is the runtime
  `merge_conflict` barrier's job, deferred.

The runtime co-open consequence is in `kaola-workflow-adaptive-node.js`: leg-provisioning fires under
`groupForm && legCoupled` (`legCoupled = parallelWritesDefaultOn(process.env)`, default TRUE; #542),
and a shared-infra frontier reaches `groupForm` without consent because `tryFormLaneGroup`'s
`--parallel-safe` re-check now relaxes the shared-infra class by default. The `resolveLegIsolation`
toggle and `opts.writeOverlapConsent` are no longer the provisioning gate for this case (the comment
block at the leg-provisioning site is corrected to match).

## The #498 invariant is preserved

Co-open **always** provisions legs: `groupForm ⟺ legCoupled ⟺ legs provisioned`. This change widens
*which classifications reach* `groupForm` (shared-infra now reaches it without consent), but it never
decouples group-formation from leg-provisioning — every shared-infra co-open is leg-contained and
synthesizer-reconciled, or it does not co-open at all. The legless, attribution-blind union barrier is
never taken.

## Alternatives considered

- **Keep shared-infra behind consent (status quo, D-542-01 clause).** REJECTED: the consent ceremony
  bought no correctness the structural net (exact-file-disjoint + post-dominating gate + per-leg
  barrier) does not already provide, and an opt-in no ordinary run sets reproduces the unreachability
  D-542 fixed for the disjoint case.
- **Relax the non-shared coarse case too, by default.** REJECTED: violates precedence #1. A non-shared
  coarse overlap is a genuinely-overlapping case with a real clobber window the per-leg worktrees +
  reconcile do not by themselves resolve; its consent gate is load-bearing.
- **Relax exact-file overlap.** REJECTED: an exact-file overlap is a true conflict; reconciliation is
  the runtime `merge_conflict` barrier's job, deferred.

## Consequences

**Positive:**
- A faithfully-decomposed, exact-file-disjoint shared-infra frontier (the common `scripts/` two-file
  case) now co-opens at width by default instead of degrading to a serial chain — the largest
  remaining makespan lever (CLAUDE.md:37) taken for the most common decomposable shape.
- No operator ceremony for the shared-infra case; the proven-safe behavior runs out of the box.
- Correctness is unchanged: the post-dominating gate, the PROTECTED guard set, the per-leg worktree
  containment, the synthesizer reconcile, the non-shared-coarse consent gate, and the exact-file refuse
  are all retained.

**Negative / note:**
- Default-off byte-identity no longer holds for the shared-infra class (the deliberate #546-G2 delta).
  For `coarse` and `exact`, default off/no-consent/no-gate still returns the prior PREVENT verdict.
- The retained net is load-bearing: if a future change removed the post-dominating-gate or PROTECTED
  guard, a shared-infra frontier would relax with NO remaining structural net. Both NET-1 and NET-2
  must stay enforced ahead of the class switch in `writeOverlapRelaxable`.
