# D-596-01. Graduate write-bearing nodes onto the speculative-open kernel, discard-only on gate failure

Date: 2026-07-03
Status: Accepted
Issue: #596
Related: D-419-02 (Part 4 — the original speculative-read kernel, and the explicit "SHIP
             read-overlap, DEFER write-overlap" recommendation this decision reverses for the
             deferred half); D-437-01 / D-542-01 (the per-member `.kw/legs/<project>/<node-id>`
             leg-isolation and lane-group machinery this decision reuses verbatim, with no new
             merge or containment code); D-593-01 (the most recent precedent for extending an
             existing write-relaxation predicate under its already-retained safety net rather
             than inventing a parallel mechanism)

## Context

D-419-02 Part 4 (issue #439) shipped `speculative_open_policy: consent` — a node whose ONLY
unsatisfied dependency is a still-`in_progress` gate may open speculatively, betting the gate
passes, with a baseline-revert discard if it fails. The design record explicitly scoped v3 to
**read-only** descendants and deferred write-bearing ones: "v3 ships ONLY speculative READ
overlap... Speculative WRITE overlap is DESIGNED-but-DEFERRED to a follow-up issue with its own
adversarial gate, given the double-gating and roll-back complexity." The stated risk was
concrete at the time: a speculative write shared the **PARENT worktree** with every other live
node, so a discard required a `git checkout <baseSha> -- <declared paths>` revert-in-place —
fragile for a brand-new file (the checkout target does not exist at the baseline SHA and the
command hard-fails), and blast-radius containment for a write in flight next to concurrent
work was an open question ("blast-radius guard" INV-24 named `KAOLA_LANE_CONTAINMENT` — at the
time an opt-in, off-by-default hook — as the strongest available containment).

That premise no longer holds. Since #439 shipped, #437/D-437-01 and #542/D-542-01 landed the
per-member LEG machinery as the DEFAULT containment for every co-opened write node: a write
member's in-lane changes land in an isolated `.kw/legs/<project>/<node-id>` worktree, never the
parent tree, and rollback is a leg TEARDOWN (`teardownLeg`: worktree remove + branch delete),
not a parent-tree revert. The double-gating concern (gate bet + write isolation, simultaneously)
is likewise already solved by existing, independently-shipped machinery: the lane-group open/
close/barrier/merge cycle (`tryFormLaneGroup` → per-leg barrier → group barrier → octopus
merge) already handles a co-opened write frontier correctly and is reused here without change.
Continuing to gate write eligibility behind a deferred, from-scratch mechanism was no longer
buying any additional safety — it was leaving a capability the leg machinery had already made
safe artificially blocked.

## Decision

**Graduate write-bearing nodes onto the SAME `speculative_open_policy: consent` bet**, reusing
the existing leg/lane-group machinery with no new merge, containment, or barrier code:

- **Static eligibility (`next-action.js`).** The `speculativePending` set now admits a
  write-bearing node under the SAME structural conditions as a read node (own status
  `pending`; not already a normal ready node; single unsatisfied direct dependency is an
  `in_progress` gate), PLUS three additional write-axis conditions: its declared write set is
  EXACTLY resolvable (`hasUnresolvableEntry` — reused verbatim from the `--parallel-safe`
  coarse-relaxation guard, newly exported from `plan-validator.js` for this purpose, no new
  predicate), it declares no PROTECTED file (`classifier.isProtected`, reused verbatim), and it
  is not the plan's unique sink (`uniqueSink`) — a speculative sink would let a bet-on-a-gate
  write skip the sink's own terminal position. A read-only node (empty declared set) trivially
  satisfies all three and is unaffected.
- **Runtime open (`adaptive-node.js runOpenReady`).** When the speculative frontier contains
  write candidates, a new `selectSpeculativeWriteGroup` RE-VERIFIES exact-path disjointness of
  every candidate against every currently-live writer via the validator's `--parallel-safe`
  predicate — the SAME predicate normal write co-open uses, no new disjointness logic — and
  EXCLUDES an overlapping candidate from this open (its disjoint siblings still open; excluded
  ids surface as `speculativeWriteExcluded: { reason: 'overlaps_live_writer', nodeIds }`).
  Survivors open WITH a provisioned leg unconditionally, even a lone candidate: a size-1
  `lane_group` forms, and `closeGroupMember` already degenerates a 1-member group to "last
  member" at close, so no new merge code was needed. The formed group counts against the WRITE
  fan-out cap (`resolveFanoutCap`, folded with `--max`), never the read-only cap. A host with
  no leg capability (`legCoupled` false — the SAME `parallelWritesDefaultOn` condition normal
  co-open gates on, i.e. `KAOLA_PARALLEL_WRITES=0`) excludes ALL write candidates from that
  call (`speculativeWriteExcluded: { reason: 'no_leg_capability', nodeIds }`); read speculation
  is entirely unaffected either way.
- **Close (`runCloseNode` / `speculativeCloseGuard`).** UNCHANGED. The existing
  `gate_not_complete` close-time hold applies uniformly to a read or write speculative member;
  a write member closes through the EXISTING per-leg barrier → (last member) group barrier →
  octopus-merge path any normal co-opened write frontier uses.
- **Discard is DISCARD-ONLY for a write member (`runDiscardSpeculative`) — the deliberate
  asymmetry with the read half's KEEP-or-discard operator review.** On a gate `verdict: fail`
  the operator still decides per read member whether its evidence is still valid; a write
  member is torn down unconditionally, with no keep path offered at all. The step-by-step
  discard now skips the pre-#596 in-lane revert-to-baseline step entirely for a write member —
  the leg's declared writes never touched the parent tree, so a parent-side `git checkout
  <baseSha> -- <path>` is a pure no-op at best and, for a file the leg was about to CREATE
  (which does not exist at the baseline SHA), a hard failure at worst. Instead: tear down the
  leg (worktree + branch + leg-base ref, via the SAME `teardownLeg` every leg-lifecycle site
  already uses; `sweepOrphanLegs` remains the crash backstop for anything a discard call
  misses), purge the stale evidence file (so a future re-open reseeds cleanly instead of
  wedging the evidence-binding check on the old nonce), and — if this cleared the LAST live
  member of its `lane_group` — clear the group entry and drop its group baseline too (mirroring
  `closeGroupMember`'s clean-completion teardown). The success envelope additively carries
  `legTornDown`, `evidenceDiscarded`, and (only when true) `groupCleared`; a read member's
  envelope is byte-unchanged.
- **Crash repair (`reconcile-running-set`) gains a crashed-speculative-write arm.** A crashed
  write member whose ledger row DID flip to `in_progress` (the general roll-forward candidate)
  now rolls FORWARD only if its gate is CONFIRMED `complete` with `verdict: pass` AT RECONCILE
  TIME — conservative on purpose, since the crash window may have let the gate resolve either
  way (or fail) before this reconcile ran, and a speculative write's leg base can be voided by a
  repaired upstream. Any other gate state (still open, complete-but-failed, unparseable, or
  unreadable evidence) rolls it BACK via the SAME existing drop path (folded into the
  pre-existing `rolledBack` array — no new response field), and every rolled-back or
  capped-out speculative write member has its stale evidence purged identically to the
  discard-speculative path. Read speculative members and every non-speculative member are
  entirely unaffected (every new arm is gated on `kind === 'write'`).
- **`hasUnresolvableEntry` is exported from `plan-validator.js`** purely so `next-action.js`'s
  static check reuses the identical resolvability predicate the runtime relaxation already
  applies — an export-only change, no new logic and no behavior change to the validator itself.

## Consequences

- Default `speculative_open_policy: off` (the permanent fallback) is byte-identical — nothing
  in this decision fires without BOTH the Meta-field opt-in and the per-run
  `--speculative-consent` flag, exactly as before #596.
- A `consent`-policy plan can now bet a write-bearing node on an open gate, contained
  end-to-end by machinery that already independently exists and is already exercised by every
  normal co-opened write frontier: leg isolation, the per-leg barrier, the group barrier, and
  the octopus merge. No new correctness surface was introduced to make this safe — #596 only
  connects an existing eligibility gate to an existing containment mechanism.
- The read half's KEEP-or-discard operator judgment call does NOT extend to writes — this is
  intentional (see Alternatives). A write member ALWAYS discards on a failed gate; there is no
  path to keep a write built on a refuted premise.
- The `speculative_open_policy: auto` legal value remains fully deferred and unsupported at
  freeze — this decision does not touch it. `auto` is reserved for a DISTINCT, still-undesigned
  axis (fully-automatic activation with no per-run consent); #596 stays entirely inside the
  existing `consent` value.
- Hosts without leg capability (`KAOLA_PARALLEL_WRITES=0`) simply never speculatively open a
  write candidate — a clean, typed exclusion (`speculativeWriteExcluded`), never a refuse; read
  speculation on the same host is unaffected.
- Cross-edition (#307): `kaola-workflow-next-action.js`, `kaola-workflow-adaptive-node.js`, and
  the `hasUnresolvableEntry` export from `kaola-workflow-plan-validator.js` (canonical + Codex
  twins + both forge ports) were regenerated together; all four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains must stay green.
  `scripts/test-next-action.js` grew 97 → 103 assertions (the write-axis eligibility
  conditions); `scripts/test-adaptive-node.js` grew 1248 → 1310 assertions, including eleven
  real-git integration cases (`T596-1`..`T596-11`) covering the happy path, the close fence, the
  pass/merge path, the fail/discard path, the runtime overlap re-check, no-leg-capability
  exclusion, write fan-out cap accounting, and three distinct crash-repair windows (leg
  provisioned pre-flip, gate resolved fail post-flip, gate resolved pass post-flip).

## Alternatives considered

- **Keep write speculation deferred (status quo).** Rejected: this is the exact call the issue
  reopens. D-419-02's own stated deferral rationale (parent-worktree revert complexity, and an
  unsettled blast-radius question) is now moot — the leg machinery independently shipped by
  #437/#542 already solves both concerns for every OTHER co-opened write node. Leaving write
  eligibility artificially blocked would not be buying any additional safety, only foregone
  automation.
- **Squat the reserved `speculative_open_policy: auto` value, or add a brand-new legal value,
  instead of extending `consent`.** Rejected: `auto` was reserved for a genuinely different
  design axis — fully-automatic activation with NO per-run human consent — and reusing it here
  would conflate two distinct risk postures under one name. A write-bearing node under #596
  requires the EXACT SAME consent ceremony, the EXACT SAME single-open-gate bet, and the EXACT
  SAME leg containment as a read-only node; there is no behavioral difference that would
  justify a second legal value. Extending `consent` is the minimal, coherent choice.
- **Offer a KEEP option for a failed-gate write member, mirroring the read half's
  KEEP-or-discard review.** Rejected: a write's on-disk changes are the asset actually at risk
  on a refuted premise, in a way a read/analysis result is not — keeping possibly-tainted write
  output produced speculatively against a review that has since failed is a real rework-and-
  audit liability the read case does not share (a stale analysis is comparatively cheap to
  re-derive; a stale write risks landing unreviewed content behind a false sense of "already
  done"). Discard-only removes that entire risk class at negligible cost, since the leg is
  disposable by design and never touched the parent tree.
- **Revert the leg's contents out to the parent tree on discard (mirroring the pre-#596
  read-node baseline-revert step) instead of tearing the leg down.** Rejected: this would
  reproduce, by hand, exactly what `git worktree remove` + `git branch -D` already do
  atomically, and reintroduces the SAME hard-failure mode the leg design was adopted to avoid
  (a `git checkout <baseSha> -- <path>` for a file that does not exist at the baseline SHA).
  Tearing the leg down is strictly simpler and cannot hit that failure mode at all.
