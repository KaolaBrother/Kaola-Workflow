# D-430-01. defense in depth — check bundle coherence at claim, handoff, and orient time

Date: 2026-06-12
Status: Accepted
Issue: #430
Related: `scripts/kaola-workflow-claim.js` (`cmdStartup`, bundle claim path),
         `scripts/kaola-workflow-adaptive-handoff.js` (`runHandoff`),
         `scripts/kaola-workflow-adaptive-node.js` (`orient` subcommand),
         `docs/api.md` § Bundle claim (refusal codes table, `target_set_mismatch`, `bundle_state_incoherent`),
         `docs/workflow-state-contract.md` § Bundle Project State Fields (coherence invariant),
         D-427-01 (probe-before-close idempotency, the related #427 fix)

## Context

A bundle project's `workflow-state.md` carries three co-dependent fields that must be mutually
consistent: `bundle_id` (canonical bundle identifier), `issue_numbers` (comma-separated sorted set),
and `issue_number` (primary issue, first in the sorted set). The invariant is:

```
bundle_id == "bundle-" + sorted(issue_numbers).join("-")
```

Before issue #430, this invariant was enforced only at CLAIM time: `claimExplicitBundle` constructs
`bundle_id` and `issue_numbers` from the `--target-issues` argument and writes them atomically.
There was no check AFTER the initial write. This created two silent corruption paths:

**Path 1 — Hand-edit of `workflow-state.md`:** A developer debugging a stuck run might manually
edit `issue_numbers` (e.g. removing one member to test partial-close behavior), not realizing that
`bundle_id` must also change. Or a merge conflict resolution might collapse the comma-separated
list. The resulting incoherent state would propagate silently into the handoff and plan-run, where
scripts read `bundle_id` and `issue_numbers` independently and can produce contradictory behavior.

**Path 2 — Silent field collapse during repair-state or migrate:** A `repair-state` invocation or
a manual migration that re-wrote `workflow-state.md` without the bundle fields would leave `bundle_id`
present but `issue_numbers` absent (or vice versa). `adaptive-handoff.js` reads `bundle_id` to
detect a bundle context and then reads `issue_numbers` to enumerate members — an absent `issue_numbers`
alongside a present `bundle_id` would leave the handoff operating on an empty member set with no
error.

Two remediation postures were considered:

1. **Single gate at the earliest mutating subcommand:** add a coherence check inside
   `adaptive-handoff.js` only. This catches corruption before the plan is frozen, but it leaves
   `orient` (the per-resume consistency check) without a guard — a corruption introduced AFTER the
   handoff (e.g. during a mid-run manual edit) would be silently accepted by `orient` and all
   subsequent `open-next` operations.
2. **Defense in depth — three guards:** add coherence checks at (1) `cmdStartup` re-claim, (2)
   `adaptive-handoff.js runHandoff`, and (3) `adaptive-node.js orient`. Each guard catches the
   corruption at the earliest possible point in its path, so a corruption introduced at any stage
   is surfaced before any mutation occurs.

## Decision

**Bundle coherence is checked at three independent points: claim, handoff, and orient.**

**Guard 1 — `cmdStartup` bundle re-claim path (`target_set_mismatch`):**

When `cmdStartup` is invoked with `--target-issues A,B,C` and an active folder already exists for
that bundle, it reads the persisted `issue_numbers` from `workflow-state.md` and compares it to the
claimed `--target-issues` set (after sort+dedup). If they differ, `cmdStartup` refuses with
`target_set_mismatch` before any state is written.

This catches: hand-edits of `issue_numbers` discovered on a re-startup; a `--target-issues` flag
passed at re-claim that does not match what was originally claimed.

**Guard 2 — `adaptive-handoff.js runHandoff` (`bundle_state_incoherent`):**

Before any handoff operation, `runHandoff` checks: if `bundle_id` is present in `workflow-state.md`
then `issue_numbers` must ALSO be present AND the derived bundle ID from `issue_numbers` must match
`bundle_id`. Any mismatch refuses with `bundle_state_incoherent` before freezing the plan or
writing any state.

This catches: `issue_numbers` absent when `bundle_id` is present; the two fields inconsistent with
each other (e.g. `bundle_id: bundle-42-47` but `issue_numbers: 42,47,53`).

**Guard 3 — `adaptive-node.js orient` (`bundle_state_incoherent`):**

At every plan-run entry, the `orient` subcommand performs the same coherence check as Guard 2.
This catches: corruptions introduced between the handoff (freeze) and a later plan-run resume (e.g.
a hand-edit during the run, or a merge conflict that collapsed the fields).

**Refusal shape (Guards 2 and 3):**

```json
{
  "result": "refuse",
  "reason": "bundle_state_incoherent",
  "bundle_id": "<value found>",
  "issue_numbers": "<value found or null>",
  "expected_bundle_id": "<derived from issue_numbers, or null>"
}
```

**Why three guards instead of one:**

A single gate at the earliest point (claim) cannot protect against corruptions introduced AFTER
that point. A single gate at the latest point (orient) allows corruptions to survive through the
handoff. The three-guard defense-in-depth pattern follows the principle stated in issue #44: agents
own reasoning, scripts own atomicity. Each guard is the natural owner of coherence for its stage:
- `cmdStartup` owns claim-time consistency.
- `runHandoff` owns pre-freeze consistency (the last chance to catch corruption before the plan
  is stamped with `plan_hash`).
- `orient` owns resume-time consistency (the first thing run on every plan-run entry, before any
  node opens).

None of the three guards performs an auto-repair. Repair of an incoherent bundle state requires
human decision (which set of issues is the intended one?); auto-repair would silently fix the wrong
field and could close the wrong set of issues at finalization.

## Consequences

**For operators:**

- A `workflow-state.md` hand-edit that makes `bundle_id` and `issue_numbers` inconsistent will be
  caught at the next `cmdStartup` (re-claim), `runHandoff`, or `orient` invocation with a typed
  refusal naming the inconsistency. The operator must repair the fields manually before proceeding.
- `cmdStartup --target-issues A,B,C` on an existing bundle folder refuses with `target_set_mismatch`
  if the passed set does not match the persisted set. This prevents accidental re-claim of the wrong
  bundle.
- Single-issue projects (no `bundle_id`) are unaffected by all three guards.

**For the refusal codes table in `docs/api.md`:**

- `target_set_mismatch` added: bundle startup — persisted `issue_numbers` does not match the claimed
  `--target-issues` set.
- `bundle_state_incoherent` added: handoff/orient — `bundle_id` is present but `issue_numbers` is
  absent or inconsistent with `bundle_id`.

**For the bundle state fields in `docs/workflow-state-contract.md`:**

- New coherence invariant note: `bundle_id` and `issue_numbers` must be consistent;
  `bundle_id` must equal `bundle-<sorted(issue_numbers)>`.

**For the test suite:**

- Walkthrough scenarios added for: `target_set_mismatch` (re-claim with a different set), and
  `bundle_state_incoherent` (handoff + orient refusing on an incoherent state).
