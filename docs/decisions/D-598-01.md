# D-598-01. Local Codex runtime dispatch config is a verified, report-only part of the install contract

Date: 2026-07-03
Status: Accepted
Issue: #598
Related: docs/decisions/D-584-01.md (agent-guided feature-enablement audit this decision
             extends with the MODE the audit did not previously check); #571 (the
             `--global` install target the delegation-probe fix in this issue makes the
             probe accept); #332 (the schema-valid preflight gate `dispatch_posture` is
             threaded through, alongside its pre-existing `dispatch_mode`/
             `multi_agent_v2_enabled` tool-exposure fields).

## Context

A full adaptive run completed with zero subagent dispatches: `multi_agent_v2` detection
(#584) reported the spawn *tools* were exposed, but the Codex runtime injects a developer
message — outranking skill prose — that model-refuses every spawn unless the effective
`MultiAgentMode` is `proactive`. That mode is effort-gated (`model_reasoning_effort =
"ultra"` -> `proactive`; anything lower or absent -> `explicitRequestOnly`), and there is
no working config key for the mode itself. Every check the workflow ran (`multi_agent_v2`
detected, profiles installed, hooks trusted) passed, and the install/preflight still
printed `status: ok`, while every gate role — including `adversarial-verifier` and
`code-reviewer` — ran inline and self-issued `verdict: pass` with no user-visible warning.
Two further gaps compounded it: the delegation probe only checked the project-local
`.codex/agents/kaola-workflow/` path, so a `--global` install (#571) read as a false
`local-fallback-tool-unavailable`; and no surface distinguished a degraded forward role
(safe to fall back inline) from a degraded gate role (an inline gate reviewing its own
writer-context is no gate at all).

## Decision

**Local Codex runtime configuration — not just the installed files — is a verified part
of the install contract, always report-only where the setting is user-owned.**

1. **One effort->mode derivation, mirrored byte-identically.** `deriveDispatchPosture`
   is authored once and copied verbatim across the installer (×3 plugin trees,
   `validate-script-sync.js` "codex agent-profile installer copies" group) and the
   preflight/doctor script (×4 trees, "codex-preflight copies" group): `[features]
   multi_agent`/`multi_agent_v2` both absent-or-false -> `dispatch_posture: 'none'`;
   otherwise a root-level `model_reasoning_effort = "ultra"` -> `'proactive'`, any other
   value or absent -> `'explicitRequestOnly'`. **Version-guarded**: this is empirically
   reverse-engineered codex-tui 0.142.5 runtime behavior, not a documented stable
   contract, so every surface that reports it also prints the guard note verbatim
   (`DISPATCH_POSTURE_VERSION_NOTE`) rather than presenting the coupling as permanent.
2. **Design boundary: REPORT, never WRITE.** `model_reasoning_effort` is a user-owned
   cost/latency choice — a higher effort tier spends more tokens and adds latency on
   *every* turn, not only on dispatch — so no installer or preflight script ever writes
   it. The installer re-reads the config it just wrote/refreshed and prints the resulting
   posture plus, when non-`proactive`, the exact remediation (set `model_reasoning_effort
   = "ultra"` in `~/.codex/config.toml`, per-session `codex -c
   model_reasoning_effort=ultra`, or an explicit in-session ask). This closes the "ok"
   install / "ok" preflight while dispatch is model-refused gap without ever silently
   mutating a setting the operator owns.
3. **Attestation-style / non-fatal by construction.** The four additive fields
   (`dispatch_posture`, `model_reasoning_effort`, `multi_agent_enabled`,
   `dispatch_posture_warning`) never change an install's or a preflight's exit code, on
   any scope (global/project/plugin_cache). A `warn:` line (installer stdout, preflight
   plain-text, and per-scope in `--doctor`) is the entire effect; an otherwise-fresh
   install or preflight is never reddened by this report. The workflow-init config-audit
   guidance (command + SKILL, mirrored across all 3 editions) is extended the same way:
   "features enabled" is no longer reported as dispatch-ready on its own — the audit
   reads `dispatch_posture` and only classifies `ok` when it reads `proactive`.
4. **Delegation probe accepts both scopes.** The Codex delegation prose (adapt/next
   SKILLs, mirrored across all 3 editions) now probes project-local
   `.codex/agents/kaola-workflow/` OR global `~/.codex/agents/kaola-workflow/`, and
   records `local-fallback-tool-unavailable` only when BOTH are absent — a `--global`
   install (#571) satisfies delegation exactly as a project-local one does; the
   project-local needle stays green (added to, never removed).
5. **Gate-role degradation routes through consent, never through silence.** The six
   plan-run routing surfaces (3 Claude commands + 3 Codex SKILLs) carry a Gate-Role
   Degradation Notice: when dispatch is unavailable (profiles absent at both paths, OR
   the runtime mode-refuses spawns), the executor posts a prominent run-start notice
   naming every gate role that would self-review. For `adversarial-verifier` and
   `code-reviewer` specifically — where the detached context IS the mechanism, not an
   optimization — the run routes through the consent-halt valve
   (`write-halt --reason consent`) instead of silently recording a self-issued
   `verdict: pass`. Forward roles (`code-explorer`, `doc-updater`, `tdd-guide`, etc.)
   keep the existing, already-safe `local-fallback-tool-unavailable` inline path
   unchanged — only a role whose entire job is independent review is unsafe to run
   against its own writer-context.

## Consequences

- An install or preflight can no longer present a model-refused dispatch posture as an
  unqualified `status: ok` — the posture and, when relevant, the remediation are always
  printed alongside it.
- A future Codex release that changes the effort->mode coupling degrades the accuracy of
  the WARN text, never the fatality of the gate (it stays report-only either way) — the
  version-guard note gives the operator a signal to re-verify rather than trust silently.
- A machine-global (`--global`) Codex install (#571) no longer produces a false-negative
  delegation-probe failure.
- `adversarial-verifier`/`code-reviewer` running mode-refused or profile-absent now
  escalate to a human via consent-halt instead of silently self-certifying their own
  work — a deliberate trade of some automation for the correctness precedence rule
  (accuracy is non-negotiable; an inline gate reviewing its own writer-context is not a
  degraded gate, it is no gate).
- Cross-edition (#307): the installer (×3), preflight/doctor (×4), the delegation and
  gate-degradation prose (6 plan-run/adapt/next routing surfaces), and the workflow-init
  config-audit pair (×3 editions) all changed together; all four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains plus
  `scripts/simulate-workflow-walkthrough.js` must stay green.

## Alternatives considered

- **Have the installer silently write `model_reasoning_effort = "ultra"` itself.**
  Rejected: effort is a user-owned cost/latency tradeoff that raises the price and
  latency of every single turn, not just dispatch calls — an install script opting a
  user's whole session into a more expensive inference tier without being asked crosses
  the "escalate irreversible changes / do not alter a user-owned contract" rule. The
  installer reports the fact and the three remediations; the decision to spend more per
  turn stays with the human.
- **Make the dispatch-posture WARN fatal** (fail the install/preflight when the posture
  is not `proactive`). Rejected: `explicitRequestOnly` is Codex's own effort-based
  default and may be a deliberate operator choice; failing an otherwise-correct, fresh
  install over a value the user may have intentionally kept would be a UX regression in
  the opposite direction from the actual defect — the bug was *silence*, not leniency.
- **Hardcode the effort->mode coupling with no version-guard note.** Rejected: it is
  empirically reverse-engineered CLI behavior, not a published stable contract; an
  unflagged hardcode would silently mis-warn (or silently stop warning) on a future
  Codex release with no signal to the operator that the derivation itself may be stale.
- **Hard-fail the run instead of routing gate-role degradation through consent-halt.**
  Rejected: forward-role local fallback (`code-explorer`, `doc-updater`, etc.) has run
  safely inline for years and a blanket hard-fail would also block that harmless path.
  Only the verification gates — where the detached context is the entire safety
  mechanism, not an efficiency optimization — have the "reviewing its own writer-context
  is no gate" problem this decision targets.
