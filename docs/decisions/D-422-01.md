# D-422-01. Agent-profile md↔toml token-pin parity contract

Date: 2026-06-12
Status: Accepted
Issue: #422
Related: ADR 0008 (agent-set exact-match registration surface, issue #340),
         `docs/conventions.md` § Forge-Neutral Plugin Agent Profiles (issue #341),
         `scripts/test-agent-profile-parity.js`, `scripts/validate-script-sync.js`

## Context

The repo maintains two parallel representations of each agent profile:

- `agents/<name>.md` — the canonical prose spec installed into `~/.claude/agents/` by
  `install.sh` for the Claude edition.
- Three `.toml` twins — `plugins/kaola-workflow/agents/<name>.toml`,
  `plugins/kaola-workflow-gitlab/agents/<name>.toml`, and
  `plugins/kaola-workflow-gitea/agents/<name>.toml` — installed by the respective plugin
  editions into the Codex runtime (via `install-codex-agent-profiles.js`).

A previous gap surfaced with issue #413, which added `write_set_granularity` guidance to
`agents/workflow-planner.md` but initially missed the three `.toml` twins. This was the
"#404 planner gap" class: a feature paragraph present in the `.md` but absent from the
`.toml` triples, meaning Codex/GitLab/Gitea runtimes would dispatch the planner without
the new guidance. Because the `.md` and `.toml` files share no symbol that a grep-based
write-set check would catch, the gap was invisible to the standard symbol-based analysis.

Two related drift surfaces existed without machine guards:

1. **Intra-triple byte identity** — the three `.toml` twins must be byte-identical to
   each other (they are forge-neutral by the §341 contract), but this was enforced only
   by the general `assertNoForbidden` scan, not by a byte-comparison entry.

2. **`config/hooks.json` cross-plugin parity** — the three plugin-tree
   `config/hooks.json` files (codex/gitlab/gitea) differ only in the forge-renamed
   compact-resume script path; no guard existed to catch other divergence.

## Decision

A three-part machine-enforced contract is introduced by #422:

### Part 1 — `.toml` triple byte-identity (`validate-script-sync.js` BYTE_IDENTICAL_GROUPS)

Every agent `.toml` present in the codex plugin tree
(`plugins/kaola-workflow/agents/*.toml`) is programmatically added to
`BYTE_IDENTICAL_GROUPS` at startup via a `readdirSync`-based spread, producing one entry
per file with the three-path group:

```
plugins/kaola-workflow/agents/<name>.toml
plugins/kaola-workflow-gitlab/agents/<name>.toml
plugins/kaola-workflow-gitea/agents/<name>.toml
```

The codex tree is the reference (first file in each group). `validate-script-sync.js`
byte-compares every member against the reference and reports drift or missing files. This
covers 14 base-role profiles and 6 `-max` model variants (20 files, 20 triples) and
auto-expands when a new profile is added to the codex tree.

### Part 2 — Feature-token mirroring (`scripts/test-agent-profile-parity.js`)

A regression guard (`test-agent-profile-parity.js`) enforces that any "feature token"
present in an `agents/<name>.md` MUST also appear in all three `.toml` twins. The guard
maintains a curated `FEATURE_TOKENS` list (currently `write_set_granularity` and
`main-session-gate`). For each `agents/<name>.md` that has a full `.toml` triple (all
three trees present), the guard checks every `FEATURE_TOKENS` entry: if the token appears
in the `.md` it must appear in each `.toml` twin, or the test fails with a named drift
message.

Token lifecycle:

- A token is added to `FEATURE_TOKENS` only after the feature paragraph has been
  mirrored into all three `.toml` twins (i.e. the token is GREEN at HEAD before being
  pinned).
- An unused token (not present in any `.md`) never triggers a false failure.
- A `.md` without a full `.toml` triple is silently skipped (not a parity target).

### Part 3 — Chain pinning (all four `validate-*-contracts.js`)

`test-agent-profile-parity.js` is wired into the claude chain (run by the root
`test-kaola-workflow-scripts.js` suite) and its execution is pinned by all four
`validate-*-contracts.js` contract validators. This means:

- A missing or renamed test file reds all four chains.
- A feature-token drift that passes the four chains means the token was already present
  in all twins — the guard is not a rubber stamp.

## Side note — `config/hooks.json` family (#418.1)

As part of the same contract-hardening sweep, `validate-script-sync.js` gains a
`CONFIG_HOOKS_FAMILY` entry covering the three plugin-tree `config/hooks.json` files:

```
plugins/kaola-workflow/config/hooks.json          (reference)
plugins/kaola-workflow-gitlab/config/hooks.json
plugins/kaola-workflow-gitea/config/hooks.json
```

These files are identical except for the `kaola-workflow-codex-compact-resume` →
`kaola-{forge}-workflow-codex-compact-resume` rename in the compact hook's script path.
The `normalizeConfigHooks()` function applies only that single token substitution (the
generic `renameNormalize()` cannot be used because other `kaola-workflow-*.sh` hook
tokens in the file must NOT be renamed). Any other divergence between the three files
will red the validation run with a named drift message.

## Consequences

**For day-to-day development:**

- Adding a new feature paragraph to an `agents/<name>.md` now requires mirroring the
  feature token(s) into all three `.toml` twins before pinning the token in
  `FEATURE_TOKENS`. The four-chain test gates this requirement.
- New feature tokens must be verified GREEN at HEAD (present in the `.md` AND in all
  three twins) before being added to `FEATURE_TOKENS`. Adding a token that is absent
  from any twin will immediately red the claude chain.
- Editing a `.toml` profile in one plugin tree requires editing the other two to maintain
  byte identity. `validate-script-sync.js` will red the chain on any byte mismatch.

**For adding a new agent profile:**

- A new `agents/<name>.md` with no `.toml` triple is silently skipped by the parity
  guard. The existing §340 registration surface still requires the `.toml` triple to be
  created and registered. Once all three `.toml` twins exist, the profile is auto-covered
  by BYTE_IDENTICAL_GROUPS (Part 1) and becomes a parity target for future feature tokens
  (Part 2).

**For the `config/hooks.json` files:**

- Changes to hooks wiring in one plugin tree must be reflected in the other two. The
  `CONFIG_HOOKS_FAMILY` guard will catch any divergence beyond the expected compact-resume
  path rename.

**Chain impact:**

- All four chains now enforce the agent-profile triple byte-identity (via
  `validate-script-sync.js`), the feature-token parity (via the chain-pinned
  `test-agent-profile-parity.js`), and the `config/hooks.json` family. A diff that adds
  a feature token to a `.md` without updating the `.toml` triples will red the claude
  chain on `test-agent-profile-parity.js` and be caught before the four-chain gate.
