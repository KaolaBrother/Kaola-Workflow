# D-606-01. Report-only Claude agent-teams detection, and a teammate-mode dispatch subsection on the plan-run surfaces

Date: 2026-07-03
Status: Accepted
Issue: #606
Related: docs/decisions/D-598-01.md (the Codex dispatch-posture pattern this decision
             mirrors for the Claude runtime — report-only detection, non-fatal by
             construction, remediation that leads with the always-available path); #601
             (established the lead-with-always-available remediation wording reused here
             verbatim); #604 (the announcement contract the named-teammate guidance is
             layered next to, not replaced by)

## Context

The Claude runtime has two dispatch transports. **Classic subagents** — a synchronous
`Agent(...)` call whose child's final text returns as the tool result — are always
available and are the only path every plan-run routing surface documents today. **Agent
teams** is an experimental, opt-in mechanism: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`
(session environment or a settings `env` block) turns each spawn into a named, full-session
teammate with a shared task list and a bidirectional `SendMessage` mailbox — the spawn
returns immediately, the orchestrator keeps working, and mid-run conversation becomes
possible in both directions.

Verified on this box (Claude Code 2.1.198, 2026-07-03): the toggle is a plain binary
environment variable, readable directly by a child Bash process (the session exports it, no
ceremony, no API round-trip) with a settings-file `env`-block scan as the fallback for a
probe that only has file access. No model tier is gated behind it — teammate models are
freely selectable. Two runtime facts matter for the prose: v2.1.178+ removed the
`TeamCreate` step (spawning is seamless), and v2.1.198+ delivers an idle notification on
teammate turn-end with no documented ordering guarantee relative to the teammate's final
deliverable message — observed twice in production runs here as the idle notification
arriving before the deliverable, which makes a premature second nudge produce a duplicate
delivery.

None of this was detected, reported, or taught anywhere in the workflow. The plan-run
orchestration patterns the six routing surfaces already define — concurrent frontier
dispatch, the bounded K=2 out-of-lane repair nudge, mid-run write-set widening addressed to
the same agent, gate status pings, incident surfacing from a running reviewer — are
conversation-layer patterns that a persistent, addressable teammate serves strictly better
than a one-shot subagent call, yet an operator running with the flag on had no signal the
workflow could take advantage of it, and no defense against the idle-notification race.

## Decision

**Mirror the settled Codex dispatch-posture pattern (D-598-01) for the Claude runtime:
detect, report, never write; teach the better transport as an addition, never a
replacement.**

1. **`claude_dispatch_posture: teams | classic` — one detection, report-only.** Probe
   `$CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` first (a child process inherits the session's
   export directly); fall back to scanning the settings `env` blocks (user + project
   `settings.json`/`settings.local.json`) when the environment variable is unset, since a
   probe running outside the flagged session only has file access. Emitted in two places:
   `install.sh`'s post-install report (Claude edition, alongside the existing posture
   output) and the `workflow-init` config-audit surfaces (the Claude command plus its two
   forge command mirrors; the Codex SKILL packs continue to report the Codex posture and
   are untouched by this arm). Non-fatal in both — a `teams`/`classic` line, never a
   changed exit code.
2. **Design boundary carried over unchanged from D-598-01: report, never write.**
   `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` is user-owned configuration — no installer or
   config-audit step ever sets it, and no script ever reads it to make a decision.
   Remediation follows the #601 precedent of leading with what always works: classic
   subagents run every workflow with zero degradation; the qualified ask — set
   `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` in a settings `env` block and restart the
   session, noting agent teams is experimental as of Claude Code 2.1.x — comes second,
   never first.
3. **A teammate-mode dispatch subsection on the six plan-run routing surfaces**, placed
   next to the Dispatch step's existing announcement contract (#604) and qualified by
   runtime plus flag:
   - flag on: spawn each node's role agent as a named teammate (name = node id, so
     announcement lines and mailbox traffic self-document); expect the spawn to return
     immediately and results to arrive as teammate messages; route the bounded repair
     nudges the surfaces already define (the K=2 out-of-lane nudge, mid-run write-set
     widening addressed to the same agent) through `SendMessage` instead of a fresh
     dispatch; reserve a synchronous spawn for when a blocking result is genuinely
     required before the next decision.
   - required idle-race sentence: an idle notification is not a deliverable and carries no
     ordering guarantee relative to the teammate's final message; send exactly one
     follow-up request on idle-without-deliverable, then wait — a second ask before the
     first answer produces duplicate deliveries.
   - flag off / other runtimes: the existing synchronous dispatch flow, verbatim, stays the
     documented default path.
   - every existing contract — evidence-persistence per role-kind, the announcement
     formats, the close-echo line, gate non-delegability — holds identically in both
     modes; teammate mode changes the transport, never the contract.
4. **Machine guards.** Contract-validator needles (all `validate-*-contracts.js` editions
   plus the root command validator) pin the config-audit posture line, the
   teammate-subsection sentinel, and the one-nudge idle-race sentence, so a partial
   propagation reds the corresponding chain. Prose stays PROVENANCE_BAN-clean. Any
   detection helper that lands as testable code gets env-set / env-unset /
   settings-fallback coverage in whichever surface it is added to.
5. **Non-goals, load-bearing.** No script, gate, barrier, or scheduler decision ever reads
   the teams flag — a classic-only session must run every workflow with identical
   behavior. No auto-write of settings, no auto-enable, no remediation beyond the single
   qualified line. No change to Codex or opencode dispatch mechanics, or to the forge SKILL
   packs' runtime prose beyond the shared six-surface subsection. Teammate mode is a
   transport upgrade for the existing running-set scheduler choreography, not a new lane or
   a new plan grammar.

## Consequences

- An operator running with agent teams enabled gets workflow-aware guidance (named spawns,
  mailbox-routed repair nudges, the idle-race defense) instead of the workflow silently
  ignoring the capability; an operator on classic subagents — the default, and expected to
  remain the common case — sees zero behavior change beyond the added posture line.
- A future Claude release that changes agent-teams ceremony or idle-notification timing (it
  has already changed twice: the `TeamCreate` removal and the idle-notification addition)
  degrades the accuracy of the teammate subsection's prose, never any script's correctness
  — nothing outside the two report surfaces and the plan-run prompts reads the flag.
- Cross-edition (#307): the config-audit posture line touches the six `workflow-init`
  surfaces, the teammate subsection touches the six plan-run surfaces, and `install.sh`
  gains the posture report; all four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains plus
  `scripts/simulate-workflow-walkthrough.js` must stay green.
- Sequenced after the dispatch-visibility bundle (#602/#603/#604/#605): the named-spawn
  guidance references the announcement contract's pre-spawn/on-return formats and the
  close-echo line rather than re-deriving them, so it reads as an extension of the existing
  Dispatch step, not a parallel description of it.

## Alternatives considered

- **Have the installer or config-audit auto-write the settings `env` block to enable agent
  teams.** Rejected: the same reasoning as D-598-01's rejection of auto-writing
  `model_reasoning_effort` applies — an experimental runtime toggle is a user-owned choice
  with its own trade-offs (ceremony, notification behavior, vendor-marked experimental); an
  install script silently opting a session into it crosses the "escalate irreversible
  changes / do not alter a user-owned contract" rule.
- **Gate a scheduler or barrier decision on the detected posture** (for example, prefer
  wider fan-out when teams is on). Rejected: the workflow's correctness and makespan model
  must hold identically with zero external capability present — coupling a scheduling
  decision to an experimental, un-owned surface would violate self-sufficiency and make
  behavior depend on something outside the workflow's control.
- **Treat teammate mode as a new orchestration lane or plan-grammar addition.** Rejected:
  the running-set scheduler's choreography (announcements, evidence-persistence,
  close-echo, gate non-delegability) is already runtime-agnostic; teammate mode changes how
  a spawn is transported and how repair nudges are delivered, not what the DAG or the
  scheduler does — modeling it as a new lane would duplicate a contract that already
  generalizes across runtimes.
- **Make the posture report fatal when the flag is off.** Rejected: classic subagents are
  the fully-supported default path with no degradation; failing a report over a value most
  operators will, correctly, never set would be a regression in the opposite direction from
  the actual gap, which was silence, not leniency.
