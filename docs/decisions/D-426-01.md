# D-426-01. copy-then-verify-then-delete is safer than rename for cross-filesystem archive moves

Date: 2026-06-12
Status: Accepted
Issue: #426
Related: `scripts/kaola-workflow-claim.js` (`archiveProjectDir`, `cmdFinalize`),
         `docs/api.md` § Closure Contract (`cmdFinalize` output, closure receipt schema),
         D-427-01 (bundle issue-close idempotency, the related #427 fix)

## Context

`archiveProjectDir` in `cmdFinalize` moved the active project folder into `kaola-workflow/archive/`
by calling `fs.renameSync(src, dest)`. This had two compounding failure modes:

**Cross-filesystem rename:** When the active folder and the archive destination reside on different
filesystems (e.g. the worktree is on a tmpfs or network mount), `fs.renameSync` throws `EXDEV`
(cross-device link). The archive would fail mid-operation, leaving the source intact but the
destination absent — the operator would see a crash with no actionable message and no partial-archive
to recover from.

**No completion verification:** Even on the same filesystem, `fs.renameSync` does not verify that
the rename produced a complete, readable archive. A rename is atomic for the directory entry but a
subsequent read failure (e.g. a disk error or a network filesystem that reported success but lost
data) would leave an apparently-complete archive that was actually corrupt. `cmdFinalize` had no
check — it proceeded to delete the source immediately after the rename, so data loss was silent and
unrecoverable.

**Worktree live copy not deleted:** For linked worktree runs (`keepWorktree: false`), the source
project folder exists in TWO places: the main root (`kaola-workflow/{project}/`) and the worktree
live copy (typically `.kw/worktrees/{project}/kaola-workflow/{project}/`). The prior code deleted
only the main root. A stale live copy in the worktree checkout could mislead subsequent reads if the
worktree was not immediately removed.

Two remediation postures were considered:

1. **Rename with fallback:** attempt `fs.renameSync`, catch `EXDEV`, and fall back to a copy-then-delete.
   The verify step would still be missing from both the rename path and the fallback path.
2. **Copy-then-verify-then-delete (always):** replace the rename with an unconditional `copyDir` →
   `verifyArchiveComplete` (checks that `workflow-state.md` exists in the destination) → `deleteDir`,
   followed by deletion of the worktree live copy when one was provisioned.

## Decision

**`archiveProjectDir` uses copy-then-verify-then-delete unconditionally.**

The rename path is removed entirely. `archiveProjectDir` now:

1. `copyDir(src, dest)` — recursively copies the active folder to the archive destination.
2. `verifyArchiveComplete(dest)` — confirms that `workflow-state.md` exists in the copy (the
   invariant anchor for a complete archive). If absent, throws before deleting the source.
3. `deleteDir(src)` — removes the source only AFTER verification passes.
4. Worktree live copy — for `keepWorktree: false` runs, deletes BOTH the main root (step 3) and the
   worktree live copy (`<worktreePath>/kaola-workflow/{project}/`), so neither checkout retains an
   active-looking folder.

**Why not the rename-with-fallback path:**

The fallback path still requires a verify step on the copy branch; adding the verify only to the
fallback and not to the rename creates a two-code-path maintenance burden where the "fast path"
(same filesystem) is the one LESS verified. Copy-then-verify-then-delete is a single code path that
is correct for both same-filesystem and cross-filesystem cases. Rename's one advantage — atomicity
of the directory-entry pointer — is irrelevant here: the closure sequence is already crash-recoverable
via the manual-archive backstop in `cmdFinalize` (a re-run detects and heals a partially-archived
state), so rename atomicity buys nothing against the scenarios that actually cause data loss.

**`anchored_root` receipt field:**

The closure receipt gains an `anchored_root` field recording the resolved main root path at finalize
time. This makes the main root unambiguous in the receipt for audit and for the worktree live copy
deletion step — the delete is keyed on the worktree path from `workflow-state.md`, but the resolved
main root is what anchors the archive destination.

## Consequences

**For operators:**

- `archiveProjectDir` is now resilient to cross-filesystem archive destinations (e.g. worktrees on
  a different mount than the main repo). No `EXDEV` crash.
- Data loss from a partially-copied archive is prevented: the source is ONLY deleted AFTER the
  destination contains `workflow-state.md`. A partial copy leaves the source intact and the
  `verifyArchiveComplete` failure is surfaced in the receipt.
- Worktree live copies are cleaned up by `archiveProjectDir` on `keepWorktree: false` runs,
  ensuring neither checkout retains a confusable active folder.

**For the closure receipt schema:**

- `anchored_root` (string) added to the `cmdFinalize` output. Records the resolved main root path
  at finalize time. Absent on single-root (non-worktree) runs where the resolution is trivial.

**For the test suite:**

- The archive path in walkthrough tests is now copy-then-verify; any test hook that patched
  `fs.renameSync` for the archive step needs to instead provide a `copyDir`/`verifyArchiveComplete`
  shim.

**For the `archiveProjectDir` API callers:**

- The function signature is unchanged; the copy-then-verify-then-delete change is internal. Callers
  that checked for a returned receipt field can observe `anchored_root` additively; no existing field
  is removed or renamed.
