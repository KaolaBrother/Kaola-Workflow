# D-661-01. Split release preparation from tag publication

**Date:** 2026-07-12
**Status:** Accepted
**Related:** D-442-01, D-651-01

## Context

The former cut operation edited release files and created a tag in one resumable transaction. A tag
could therefore describe the pre-bump tree even though the worktree contained the new version. It
also collapsed two different trust questions: whether release metadata was prepared correctly, and
whether one immutable, fully tested commit was safe to publish.

## Decision

Preparation and publication are separate trust transitions. `--prepare` may edit only the eight-file
release allowlist and records a non-authorizing, version-scoped receipt. An operator then creates one
release-only commit and stamps the complete offline edition-chain receipt at that exact commit. The
plan-independent `--release-check` must pass before `--tag` is invoked.

`--tag` requires strict equality among candidate HEAD, the prepare baseline and hashed surface, the
chain receipt's candidate SHA, both publication receipt rows, the live tag target, and the raw bytes
read from the tag tree. Root/Claude and Codex versions remain independent SemVer axes. Strict binding
is required because a content-similar or rebased commit is not the commit the receipt authorized,
and a correct worktree does not prove the tag contains that tree.

Tag creation uses an atomic zero-old ref update. Candidate and tag trees are verified byte-for-byte;
post-create verification failure compare-deletes only the newly created, still-matching tag. Completed
publication is idempotent only when the entire authorization and completion state remains coherent.

`--cut` cannot retain success semantics. Compatibility success would preserve an unsafe entry point
or silently choose where the new commit-and-validation boundary occurs. It therefore returns only
`cut_compatibility_refusal` with the replacement prepare → commit → receipt → release-check → tag
sequence. Publication remains forge-neutral and no external pipeline is part of the gate.

## Consequences

Crashes during preparation resume missing allowlisted steps for the same binding. Crashes during tag
publication leave explicit authorization/completion state; reruns either prove exact agreement or
refuse. Any version change, candidate-history change, extra file, stale/partial/waived receipt, probe
failure, tag conflict, or tree mismatch fails closed rather than being treated as authorization.

The extra release-only commit and two explicit checks are intentional: they make the tag's commit and
contents auditable and prevent prepared-but-uncommitted state from crossing the publication boundary.
