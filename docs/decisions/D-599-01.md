# D-599-01. Mirror `tryFormLaneGroup`'s fail-closed posture in `selectSpeculativeWriteGroup`

Date: 2026-07-03
Status: Accepted
Issue: #599
Related: D-596-01 (introduced `selectSpeculativeWriteGroup` and the write-graduation
             machinery this decision hardens; the gap was found by the SAME review gate
             that passed D-596-01's own change); D-419-02 Part 2 (the pre-existing
             `tryFormLaneGroup` fail-closed posture this decision brings the speculative-write
             callsite into line with)

## Context

`selectSpeculativeWriteGroup` (`scripts/kaola-workflow-adaptive-node.js`) re-verifies
exact-path disjointness of every speculative write candidate against every currently-live
writer by shelling the plan-validator's `--parallel-safe` check, then loops over the
subprocess result's `overlapping` array (`ps.overlapping || []`) to decide which candidates
to exclude from this open. On a non-ok result that lacks a well-formed `overlapping` array ‚Äî
a validator subprocess crash, a timeout, or garbled JSON on stdout ‚Äî the `|| []` fallback
makes the loop iterate zero times, so it excludes NOTHING: every speculative write candidate
proceeds to open. This is **fail-open**.

It is also asymmetric with the normal (non-speculative) co-open path. `tryFormLaneGroup`
consults the identical `--parallel-safe` predicate and returns `{ok:false}` on ANY non-ok
result, unconditionally degrading the whole frontier to a single serial write ‚Äî **fail-closed**.
Two callsites of the same underlying predicate disagreed on what "the predicate didn't answer"
means.

The issue-596 review gate flagged this as a non-blocking LOW rather than a blocker, because
three independent nets already bound the practical exposure:

1. The freeze grammar already refuses two concurrent antichain siblings that share an exact
   path, so any in-grammar speculative candidate set is disjoint by construction before this
   check ever runs.
2. Each candidate opens in an isolated per-member leg (the #463/D-542-01 leg machinery), so
   no parent-level write collision can occur even if two candidates were, somehow, not
   disjoint.
3. A real textual overlap still fails closed at the existing merge-conflict refusal on the
   pass path.

Those nets make the defect non-exploitable in practice, but they don't make it correct: the
underlying question ‚Äî "what does an unreadable `--parallel-safe` answer mean for a candidate
still deciding whether to open?" ‚Äî has exactly one right answer, and the codebase was giving
two different ones depending on which callsite asked.

## Decision

Mirror `tryFormLaneGroup`'s posture inside `selectSpeculativeWriteGroup`: treat any non-ok
`--parallel-safe` result that is missing a well-formed `overlapping` array as conservative ‚Äî
exclude every speculative write candidate from that open, not just the pair the (absent)
`overlapping` entry would have named. A well-formed non-ok result that DOES carry an
`overlapping` array ‚Äî including an empty one ‚Äî is unchanged: the existing per-pair exclusion
logic runs exactly as before. In other words, the fallback shifts from "no `overlapping` array
‚áí nothing excluded" to "no `overlapping` array ‚áí everything excluded"; a present-but-empty
array still means "nothing overlaps," which is a genuine, distinct answer from "the check
didn't run."

RED-first proof: `T599-1a`/`T599-1b`/`T599-1c` in `scripts/test-adaptive-node.js` inject a
`--parallel-safe` subprocess failure (non-zero exit, and separately unparsable stdout) at
speculative open-time and assert, against the pre-fix code, that every speculative write
candidate still opens; against the fixed code, that none do. A companion case confirms a
well-formed non-ok result carrying a real (possibly empty) `overlapping` array is unaffected ‚Äî
per-pair exclusion still runs, and a fully-disjoint candidate set still opens normally.
`test-adaptive-node` grew 1310 ‚Üí 1314 assertions.

## Consequences

- A validator subprocess failure at speculative open-time now produces the SAME outcome on
  both callsites of `--parallel-safe`: no speculative write candidate opens this call (surfaced
  as the existing `speculativeWriteExcluded` typed outcome), exactly as a normal co-open
  frontier degrades to a single serial write on the identical failure. The two callsites of one
  predicate now share one error posture.
- The positive (validator-reachable) path is byte-unchanged: a well-formed disjoint result still
  opens every candidate; a well-formed overlap result still excludes only the overlapping
  pair(s), and disjoint siblings still open.
- No change to the three redundant nets named above ‚Äî they remain in place and unaffected; this
  decision closes the LOW gap underneath them rather than replacing them.
- Cross-edition: the fix lands in the `kaola-workflow-adaptive-node.js` GENERATED_AGGREGATOR
  group (canonical + Codex twin + both forge ports); all four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains must stay green.

## Alternatives considered

- **Leave it fail-open, given the three redundant nets.** Rejected: the nets bound the
  exposure but don't resolve the underlying inconsistency ‚Äî two callsites of the exact same
  predicate should agree on what an unreadable answer means. Leaving them disagreeing invites
  the next caller (or the next port of this predicate) to copy the wrong one by example.
- **Exclude only the candidates the malformed result happened to name, rather than every
  candidate.** Rejected: a non-ok result without a well-formed `overlapping` array carries no
  reliable information about WHICH candidates are safe ‚Äî there is nothing to partially trust.
  `tryFormLaneGroup` already established the right shape for "the predicate didn't answer":
  degrade the whole decision, not a subset of it.
