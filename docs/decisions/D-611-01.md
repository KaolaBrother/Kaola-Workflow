# D-611-01. Codex dispatch Join Protocol — wait budgets, positive-confirmation writer kill-safety, typed delegation outcomes

Date: 2026-07-04
Status: Accepted
Issue: #611
Related: #598 (the disjoint authorization-half predecessor — spawns are allowed; this issue starts
         where it ends: what happens after a spawn), #463 / D-542-01 (the per-member leg isolation
         machinery an interruptible writer must run inside instead of an in-place kill), D-585-01
         (the scheduler O_EXCL lock `reconcile-running-set` runs under), #451 / #582 (the Codex
         reasoning-effort dispatch mapping the wait-budget tier derivation reuses)

## Context

A live Codex-edition adaptive run (session `rollout-2026-07-03T18-34-29-019f278b`, gpt-5.5 @ xhigh,
codex-tui 0.142.5, multi-agent v2) showed that authorizing dispatch (#598) was only half the
problem: the Codex edition had no join/lifecycle choreography at all, and the model improvised one
badly. Measured over the run: 24 `spawn_agent` calls, 8 `interrupt_agent` kills at a median ~4
minutes after spawn — every victim's own rollout showed it actively executing a command seconds
before the kill, none genuinely wedged; 47 `wait_agent` calls, 36 of which timed out on short
10–180s waits (reflex busy-polling, the opposite of the embedded tool guidance "prefer longer waits
… avoid busy polling"); zero `close_agent` calls all session; `send_message` "are you still there"
status probes went unanswered and directly preceded kills; every kill was followed by the
orchestrator redoing the node inline; and a killed in-place writer leaked a stray file outside its
declared write set (`setup_daily_quant.sh` + tests) that had to be hand-cleaned — a correctness
hazard, not just wasted work. The Codex SKILL packs specified only the spawn side (dispatch card →
`spawn_agent`); nothing told the model that a substantive role node legitimately runs 10–30 minutes.

A follow-up controlled probe (interactive TUI, codex-tui 0.142.5, 2026-07-03) settled the ambiguous
half of the official `close_agent` doctrine ("completed agents remain open and count toward the
concurrency limit until closed"): the 4th rapid child spawn was refused verbatim
(`collab spawn failed: agent thread limit reached`, effective subagent width 3 under the default
4-thread session budget, which INCLUDES the orchestrator), but with three completed-but-unclosed
agents visible in `list_agents`, two further spawns still succeeded — finished agents do NOT hold a
concurrency seat. `close_agent` was not exposed on the probe session's tool surface at all, and the
harness auto-reaped two completed agents from `list_agents` with no close call — so closure cannot
be a mandatory protocol step; it is best-effort hygiene, never the capacity remedy.

## Decision

**Six design arms, each closing one observed failure mode; no new subcommand, one extended one.**

1. **A — wait budget, derived from the plan, not guessed.** `waitBudgetMinutes(model)` in
   `kaola-workflow-adaptive-schema.js` resolves the node's effort tier to a concrete
   `wait_budget_minutes` (`reasoning`→40, `standard`→20, any absent/blank/unrecognized tier→a
   `role_default` of 20 — never null) and a `wait_budget_source` (`planner_model` or
   `role_default`). `buildDispatch()` spreads it onto every dispatch card the three openers
   (`open-next`, `open-ready`, `close-and-open-next`) emit. **Rule: a `running` agent is never
   interrupted before its wait budget expires** — this replaces the improvised 2–7-minute
   impatience ceiling that sat below the observed 10–30-minute runtime of a substantive role node.
2. **B — long-poll join loop, drain-all, no status probes (prose-only).** One `wait_agent` call per
   iteration with a long timeout (minutes, at or near the host's `max_wait_timeout_ms`), passing
   every outstanding agent id where the tool accepts a multi-id wait. On wake, `list_agents` once,
   then drain EVERY completed member (integrate, `record-evidence`, `close-node`, best-effort
   `close_agent` where the tool surface exposes it) before re-waiting. `send_message` "still there"
   probes are PROHIBITED as a liveness signal — structurally unanswerable by a busy agent, and the
   exact pattern that preceded every kill in the motivating run.
3. **C — escalation ladder, replacing impatience-kill (prose-only).** Strictly gated on the wait
   budget (arm A) having expired, each rung requiring the previous one to have run: (1)
   `followup_task` demanding the bounded deliverable; (2) after a ~5-minute grace window with no
   response, `interrupt_agent` (recoverable, not a kill) plus a further `followup_task` asking the
   still-available agent for partial evidence and its changed-file list; (3) reclaim the node —
   documented as the LAST resort, never the first move. Every delegation records a typed outcome —
   column-0 `delegation_outcome: completed | returned_partial | interrupted_unresponsive |
   interrupted_obsolete` (closed vocabulary via `DELEGATION_OUTCOME_VOCABULARY`, absent ⇒
   `completed`), enforced by `checkEvidenceShape` ahead of every role branch and the universal `n/a`
   carve-out, so it governs every role uniformly. This replaces a free-text "it stalled so I did it
   myself" with a machine-checkable record.
4. **D — writer kill-safety: extend `reconcile-running-set`, positive-confirmation, fail-closed.**
   An in-place writer (sharing the parent worktree) is never interrupted before the wait budget and
   the full escalation ladder — a writer that genuinely needs to be interruptible belongs in an
   isolated `parallel_safe` leg instead (#463/D-542-01: interrupting a leg discards it atomically,
   never a partial keep). After reclaiming ANY in-place writer, `reconcile-running-set` now diffs
   every WRITER member leaving the live set (rolled back / capped out / stale) against its declared
   write set via `--barrier-check` — the SAME baseline+diff the per-node barrier uses — BEFORE the
   existing `--drop-base` loop runs (which would remove the baseline the diff needs), and emits a
   per-writer verdict: `writerReconciliation: [{node_id, verdict: 'adopt'|'halt', reason,
   outOfWriteSet}]`, plus a top-level `writerHalt` when any writer halted. `classifyWriterReconcile`
   truth table (final, post-repair):

   | `bc` shape | verdict | reason |
   |---|---|---|
   | `null` / non-object | halt | `barrier_unavailable` |
   | `refuse` with named out-of-set paths | halt | `write_set_overflow` (or `bc.reason`) |
   | `refuse` with `reason: 'no_barrier_base'` | adopt | `no_baseline` (vacuous — nothing to reconcile) |
   | `refuse`, other | halt | `barrier_refused` (or `bc.reason`) |
   | `result: 'pass'` or `'ok'` | adopt | `in_write_set` |
   | resultless (`{exitCode:N}`) or unrecognized `result` | halt | `barrier_unverifiable` |

   Reconcile is NON-DESTRUCTIVE by design — it never auto-deletes a production file. The issue's own
   AC3 asked for a typed `adopt | revert | halt` output; the shipped classifier emits only
   `adopt|halt` from reconcile itself, with `revert` retained as the vocabulary token the
   ORCHESTRATOR may act on via the pre-existing, separately-consent-adjacent `revert-overflow`
   subcommand — see "Alternatives considered" for why reconcile does not perform the revert inline.
5. **E — `fork_turns:"none"` unconditionally.** Every Codex v2 dispatch call now passes
   `fork_turns: "none"` regardless of tier (the prior wording scoped the mandate to tiered nodes
   only). The dispatch card is self-contained by contract (node id, role, absolute worktree path,
   evidence contract), so no role spawn should ever fork the parent's history — inherited history
   adds compaction bloat and, in the motivating run, caused one child to spend its only turn
   answering the PARENT's pending user message instead of its assigned task.
6. **F — frontier dispatch discipline + slot awareness, plus a preflight/installer bounds report.**
   On a same-turn frontier, issue every `spawn_agent` call back-to-back in ONE turn, then run
   exactly one join loop (arm B) covering the whole frontier. The width formula counts only RUNNING
   members (a completed-but-unclosed agent does not hold a seat, per the controlled probe above);
   capacity is a REACTIVE concern — on an actual spawn refusal, wait for or close one finished
   agent, then retry the same spawn once; proactive early closure "just in case" is not required.
   The Codex preflight/doctor (`kaola-workflow-codex-preflight.js`) and installer
   (`install-codex-agent-profiles.js`), version-guarded the same way as #598's posture report, gain
   six additive report fields alongside the existing `dispatch_posture*` fields —
   `max_concurrent_threads_per_session`, `max_concurrent_threads_per_session_source`
   (`'config'|'observed_default'|'not_applicable'|'n/a'`), `effective_subagent_width`,
   `min_wait_timeout_ms`, `max_wait_timeout_ms`, `default_wait_timeout_ms` — all gated on
   `multi_agent_v2_enabled` (every field `null`/`not_applicable` when v2 is off, mirroring how
   `dispatch_posture` itself collapses to `'none'`). An absent or non-positive/non-integer
   `max_concurrent_threads_per_session` falls back to `OBSERVED_DEFAULT_MAX_CONCURRENT_THREADS_PER_SESSION
   = 4` (`source: 'observed_default'`) — this number comes from the issue's own controlled
   probe ("4 available concurrency slots, including you"), NOT from published Codex documentation,
   so it is labeled observed rather than asserted as guaranteed behavior. The three wait-timeout
   fields have NO fabricated numeric fallback (none could be independently verified against the
   installed binary or published docs) and read `null` when not explicitly configured in
   `[features.multi_agent_v2]`.

All six plan-run routing surfaces (3 Claude commands + 3 Codex SKILL packs, per the #400
six-surface rule) gain the Join Protocol prose (arms A–C, E, F, in runtime-appropriate vocabulary —
`SendMessage` replaces `followup_task`/`interrupt_agent` on the Claude/forge command surfaces). A
new reference card, `docs/plan-run-cards/join-protocol.md`, documents the full mechanics in depth
(wait-budget table, long-poll drain-all loop, escalation ladder + `delegation_outcome` table, writer
kill-safety + the real `reconcile-running-set` JSON shape, frontier discipline, and a quick-reference
diagram) — the skeleton stays resident-prose-thin per D-445-01.

## Consequences

- A Codex (or Claude teammate-mode) dispatch now has a principled, plan-derived answer for how long
  to wait, how to escalate, and how to reclaim safely — no timeout number is left to model
  improvisation, and a reclaimed writer's stray edits are surfaced as a typed, non-destructive halt
  instead of silently hand-cleaned or silently adopted.
- The wait-budget numbers (40 / 20 minutes) sit above the observed 10–30-minute runtime of a
  substantive role node, so they should not themselves become a new impatience ceiling under normal
  operation.
- The preflight/installer bounds report is read-only and non-fatal by construction, mirroring the
  #598 posture report's boundary: it never writes the user's `[features.multi_agent_v2]` config.
- Cross-edition (#307): `kaola-workflow-adaptive-schema.js` + `kaola-workflow-adaptive-node.js`
  (canonical + codex twin + both forge ports), `kaola-workflow-codex-preflight.js` (×4) and
  `install-codex-agent-profiles.js` (×3) byte-groups, the six plan-run routing surfaces, and the
  four contract validators; all four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}`
  chains must stay green.

## Non-goals

- **No proactive heartbeat/progress-file protocol.** Zero genuinely wedged agents were observed in
  the motivating run (every kill hit a healthy, actively-working agent); the budget-plus-ladder
  bounds the cost of a true wedge to one node's wait budget, and a heartbeat protocol would touch
  every role profile across every edition to solve a hypothetical. Add one only if the budget+ladder
  proves insufficient in practice, with evidence.
- **No new adaptive-node subcommand.** The writer kill-safety reconciliation (arm D) extends the
  existing `reconcile-running-set` subcommand rather than adding a parallel one — reuse before
  adding, and the existing crash-repair trigger point (a departing writer at reconcile time) is
  exactly where the new diff needs to run.
- **Not a change to spawn authorization.** This issue assumes dispatch is already authorized (#598's
  scope); the Join Protocol governs what happens strictly after a spawn.
- **The preflight bounds numeric defaults are observed, not guaranteed.** `4` is this issue's
  controlled-probe measurement on one host/plan combination, not a documented Codex constant, and is
  labeled `observed_default` accordingly rather than presented as a portable guarantee.

## Alternatives considered

- **Reconcile emits a destructive `revert` verdict inline, matching the issue's literal
  `adopt | revert | halt` AC3 wording.** Rejected: reconcile's entire design posture (and every
  sibling drop-direction path it already implements — #384/#293/#596) is NON-DESTRUCTIVE; the repo
  already has a purpose-built, separately-invoked destructive primitive (`revert-overflow`) for
  exactly this remediation. Having reconcile call it inline would collapse two independently-useful
  decision points (detect vs. remediate) into one, and would make a crash-repair pass capable of
  silently deleting tracked work with no consent step. `revert` stays in the typed vocabulary as the
  token the orchestrator MAY act on, not something reconcile performs.
- **Fail-open on an unverifiable barrier (treat a crashed/resultless `--barrier-check` as an implicit
  pass).** This was the FIRST implementation, refuted end-to-end by an adversarial-verifier gate: a
  SIGKILL'd/jetsam-killed/crashed/non-JSON/missing-validator subprocess returns a truthy, result-less
  object (`shellNode` never throws), which the original classifier's terminal `return adopt` treated
  as clean — silently keeping unverified writer dirt, reachable via exactly the jetsam kills this
  repo documents on constrained hosts. Repaired to positive-confirmation fail-closed (this decision's
  shipped truth table) and re-verified NOT-REFUTED.
- **A proactive close-agent-as-soon-as-finished discipline to manage capacity.** Rejected per the
  controlled probe: closure is not the capacity remedy (finished agents do not hold seats regardless
  of close state), and requiring it as a mandatory step is unenforceable on a host where the tool
  surface does not even expose `close_agent`. Capacity is handled reactively, on an actual spawn
  refusal, instead.
