# D-544-01. Contract-keyed effort-tier mapping + safe default + documented re-sync

Date: 2026-06-20
Status: Accepted
Issue: #544
Related: #537 (opencode runtime dispatch re-resolves the provider — Surface 2, already resilient),
         #530 (opencode edition is additive; this stays inside that surface), D-538-01 (the
         adaptive-default ADR this edition runs under), #227 (original adaptive path).

## Context

The opencode edition migrates Claude Code's closed two-tier model vocabulary (`opus` / `sonnet`)
to opencode's provider-open world by mapping each tier to an **effort variant** of the inherited
model (`mapTier(tier, provider)`). The mapping lived in `kaola-workflow-adaptive-schema.js`
(byte-identical ×4 across editions). Three independent defects were concentrated in that table:

1. **Name-keyed, not contract-keyed.** `PROVIDER_EFFORT_TABLE` had a dedicated
   `zhipuai-coding-plan` entry that emitted `reasoningEffort: max/high`. But GLM-5.2 via z.ai is
   served under the **Anthropic API contract**, whose effort knob is
   `thinking { type: "enabled", budgetTokens: <N> }`, **not** `reasoningEffort`. The table keyed on
   brand name, not on the API contract the provider actually speaks — so the generated config was
   syntactically/semantically wrong for the very provider it singled out.

2. **Unknown → null → de-tier.** `effortForProvider()` returned `null` for any unrecognized
   provider id; `mapTier()` then returned `null`; the dispatch fell back to `role_default`. Both
   tiers collapsed to identical effort — a **silent de-tier** for any provider the table did not
   enumerate (any new entrant, any self-hosted gateway).

3. **Config pinned to the seeded model id.** `renderAdaptiveConfig()` emitted
   `provider.<seededProviderId>.models.<seededModelId>.variants.*`. Switch the opencode model
   without re-running `--adapt` and the `agent.<role>.variant` references no longer resolve under
   the new model → silent de-tier on the **config** side.

The runtime **dispatch** path (`dispatchEffortOpencode` → `resolveOpencodeProvider` →
`KAOLA_OPENCODE_INHERIT_MODEL`) already re-resolved the provider on every dispatch (#537), so the
dispatch *envelope* was already resilient. The gaps were the **data table** (defects 1+2) and the
**generated config + docs** (defect 3).

## Decision

### (a) Contract-keyed resolver — `CONTRACT_EFFORT_TABLE` + `contractForProvider`

Replace `PROVIDER_EFFORT_TABLE` with a **contract-keyed** table. The effort **knob** is determined
by the provider's **API contract**, not its brand name. Four contracts:

| Contract | Knob | top variant | second variant |
| --- | --- | --- | --- |
| `anthropic` | `thinking` budget | `max` (budget 32000) | `high` (budget 16000) |
| `openai` | `reasoningEffort` | `xhigh` | `high` |
| `google` | `reasoningEffort` | `high` | `low` |
| `default` (unknown) | `reasoningEffort` | `high` | `medium` |

A new helper `contractForProvider(providerId)` maps a provider id to one of the four contracts.
GLM-via-z.ai resolves to **`anthropic`** — the `zhipu|^zai|z-?ai|glm` test runs **first**
(before the generic `anthropic|claude` test), so GLM provider ids always land on the Anthropic
contract and emit the `thinking` budget. The regex order mirrors the old `effortForProvider` alias
order, so no existing alias regresses.

**Variant-name preservation invariant (load-bearing).** The contract-keyed table preserves the
existing variant **names** (`max`/`high` for anthropic incl. GLM, `xhigh`/`high` for openai,
`high`/`low` for google). Only the OPTIONS payload flips (`reasoningEffort` → `thinking` for the
anthropic contract). This is why `test-adaptive-node.js` Cases 4/5/6/7/9 — which assert variant
names only — stay GREEN unchanged, and why the `agent.<role>.variant` references in already-seeded
configs keep resolving.

### (b) Safe default for unknown providers — no de-tier

Rewrite `effortForProvider(providerId)` so that a **real unrecognized** id resolves to
`CONTRACT_EFFORT_TABLE.default` (the `high`/`medium` top/second split) instead of `null`. A
previously-unknown provider now gets a genuine two-tier split rather than collapsing to
`role_default`.

The **falsy-guard is preserved** as the backward-compat bridge: `effortForProvider(null)` and
`effortForProvider('')` still return `null`. This keeps the claude/codex dispatch path (which never
sets an opencode provider) behavior-inert — `mapTier` → `null` → `role_default`, exactly as before.
Only a *real* unrecognized provider id (e.g. `acme-corp`) hits the default contract. The distinction
is "no provider available" (null, backward-compat) vs. "a provider we don't recognize" (default
contract, safe split).

### (c) Documented re-sync for the config side

The runtime dispatch path is the **never-de-tiers** safety net: it re-resolves the provider from
`KAOLA_OPENCODE_INHERIT_MODEL` on **every** dispatch, so the dispatch envelope always carries the
correct variant sourced from `planner_model` (not `role_default`), regardless of what the seeded
`opencode.json` happens to say.

The **config-side** variant definitions are necessarily pinned to the seeded model id (opencode's
variant schema is model-scoped — see Alternatives). The failure mode is therefore converted from
**silent** to **documented**: the generated `opencode.json` carries a prominent header comment
stating the seeded model, the contract, the knob, and the exact re-sync command
(`KAOLA_OPENCODE_INHERIT_MODEL=<new> node scripts/sync-opencode-edition.js --write-config --adapt`);
`docs/opencode-edition.md` gains a "Switching models (resilience)" subsection; and
`install-opencode.sh` echoes the guidance at seed time.

## Alternatives considered

**(a) Runtime re-resolve only (no config-side fix).** *Rejected.* opencode applies
`agent.<role>.variant` by reading `opencode.json`'s `provider.<id>.models.<model>.variants.*` —
there is **no per-call variant override** (the `task` tool has no such argument). The
`opencode_variant` the dispatch envelope carries is a recording of intent for the ledger, not a
runtime override opencode honors. So runtime re-resolve fixes the dispatch envelope but cannot fix
the config-side definitions; a stale config still de-tiers. Both surfaces must be addressed.

**(c) Model-agnostic variant scheme.** *Rejected.* opencode's variant schema is **model-scoped**
(`provider.<id>.models.<modelId>.variants.*`); there is no model-agnostic namespace. A variant
defined under `glm-5.2` does not apply when the active model is `claude-sonnet-4-5`. A
model-agnostic mapping is simply not expressible in opencode's config schema, so the config side
must be re-synced when the model changes.

## Consequences

**Positive:**
- GLM-5.2 via z.ai now emits the Anthropic-contract `thinking` budget (32000/16000) — the config
  is correct for the provider the edition actually targets. The `reasoningEffort` mis-emission is
  gone.
- Unrecognized providers no longer silently de-tier; they get the safe `high`/`medium` default
  contract — a real top/second split. The silent-de-tier failure mode is closed for any future
  provider entrant or self-hosted gateway.
- Model switches are a **documented** operation (header comment + docs subsection + install echo)
  rather than a silent de-tier. The runtime dispatch path never de-tiers regardless, so tier
  *selection* is resilient even before the config is re-synced.

**Negative / note:**
- Switching the opencode model still requires a conscious re-sync of the variant **definitions**
  (the `--write-config --adapt` command) for the config side to match — the model-scoped schema
  makes this unavoidable. This is the cost documented above; the alternative (automatic config
  mutation on every run) was rejected as too invasive and not expressible.
- The S1 + A12-unknown assertions in `test-opencode-edition.js` are flipped to assert the new
  behavior (thinking budget for GLM; safe-default contract for unknown). `test-adaptive-node.js`
  Cases stay GREEN unchanged because variant **names** are preserved.
- The ×4 byte-identical schema copies (`scripts/` + 3 plugin mirrors) all carry the identical
  `CONTRACT_EFFORT_TABLE` / `contractForProvider`; `validate-script-sync.js` enforces byte-identity.
