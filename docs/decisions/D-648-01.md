# D-648-01. Finalize validation stamps last, stale self-host refusals carry diagnostics, and consumer citations remain an agent contract

Date: 2026-07-09
Status: Accepted
Issue: #648
Related: #432/#475 (dual-mode finalize gate); #547/D-547-01 (code-tree receipt freshness);
         #608/D-608-01 (additive chain receipt timeout diagnostics); #643/D-643-01
         (evidence-token contracts as close-gate inputs)

## Context

The finalize gate has two validation modes. Self-host repos use `.cache/chain-receipt.json`, while
consumer repos use agent-recorded `.cache/final-validation.md` with a column-0 `verdict: pass`.
Before this decision, a self-host `chains_stale` refusal told the operator only that the receipt was
stale, not which validation-visible paths made it stale. That was accurate but inefficient during
Finalization because the correct remedy is still a full restamp, and the operator needs quick
culprit context without weakening the fail-closed gate.

The same run also tightened runtime prose around validation sequencing. A chain receipt stamped
before the final code or chain-asserted doc update is structurally stale. A consumer final-validation
file that cites a prior run needs an honest reuse boundary so a later finalize-step docs or changelog
edit is not falsely represented as covered by the prior validation run.

## Decision

1. **Self-host receipt stamp is last before Finalization.** The main session runs
   `kaola-workflow-run-chains.js --project <project>` only after all code and test-consumed prose/doc
   changes for the final candidate have landed. Workflow state and inert, non-test-consumed docs are
   validation-invisible and do not stale the receipt. If a code-relevant or chain-asserted doc changes
   after the stamp, `chains_stale` requires a full gated-runner restamp; the receipt is not patched by
   hand.
2. **`chains_stale` gets additive culprit diagnostics.** `--finalize-check --json` keeps the canonical
   refusal shape (`result: "refuse"`, `reason: "chains_stale"`) and may add `stale_paths`,
   `stale_paths_truncated`, and `stale_kind`. The scan is best effort: it requires a clean stamped
   `headSha` in the receipt and a computable local git diff. The path list is sorted,
   validation-visible, capped at 20 entries, and classified as `code`, `prose-only`, or `mixed`.
3. **Diagnostics never change the decision.** Callers still branch on `reason: "chains_stale"`.
   The diagnostic fields do not alter refusal precedence, `operator_hint`, the chain gate remedy, or
   the later attribution sweep. Absence of diagnostics means "generic stale", not pass.
4. **Consumer citation metadata is an agent-facing contract.** Consumer mode still gates only on a
   column-0 `verdict: pass` in `.cache/final-validation.md`. When an unchanged terminal change-gate
   validation run is cited instead of rerun, workflow prose requires `source: cited:<node-id>`,
   `validated_command`, `validated_at_head`, and `reuse_boundary` so the reuse boundary is explicit.
   `--finalize-check` intentionally does not parse those fields; uncertainty means run the command.

## Consequences

- Self-host stale failures become easier to diagnose without creating a partial-restamp path or a
  new consumer obligation.
- Consumer repos keep the #475 "agent owns verification" model: the script remains a minimal
  fail-closed verifier for `verdict: pass`, while the citation lines are durable evidence discipline
  for humans and future agents.
- Existing consumers that parse only `result` and `reason` remain compatible; all new stale fields
  are optional and additive.

## Non-goals

- Does not change `.cache/chain-receipt.json` schema or `kaola-workflow-run-chains.js` chain
  execution semantics.
- Does not add a consumer-mode parser gate for `source`, `validated_command`, `validated_at_head`, or
  `reuse_boundary`.
- Does not make `stale_paths` exhaustive when the local diff is unavailable, the stamped receipt was
  dirty, or more than 20 validation-visible paths changed.
