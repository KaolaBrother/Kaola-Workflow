# D-530-02. opencode stays additive (runtime-not-forge): NOT wired into `npm test`, NOT in `edition-sync`, NOT in `install.sh`

Date: 2026-06-19
Status: Accepted
Issue: #530
Related: #501 (self-sufficient by default; CI/CD is not a gate — the four npm chains are INTERNAL
             self-contained gates, and an external non-forge edition is not one of them),
          #400 (SIX-surfaces routing policy — forge-scoped: 3 Claude + 3 Codex),
          #307 (four-chain cross-edition parity = the forge propagation surface),
          #475 / D-475-01 (the direction this continues: push validation inward to agent checks,
          retire external chain/CI receipts — opencode's install-time suite is the inward form),
          D-530-01 (the companion decision: route-reachability parity is enforced via the opencode
          twin test, NOT by joining the forge T-sets — enabled by this additive boundary),
          #486 (Case-A read-only audit shape this decision was produced inside).

## Context

The opencode edition was added on `feature/opencode-support` as a **runtime edition**: it has its
own generator (`sync-opencode-edition.js`, self-declared twin `:11-12`), its own installer
(`install-opencode.sh`), and its own test (`test-opencode-edition.js`). **Five independent signals**
establish that it is deliberately **not** part of the forge machinery:

1. `edition-sync.js:43` — `FORGES=['gitlab','gitea']`; grep `opencode` = 0.
2. `install-opencode.sh:4-6` — standalone installer, explicitly "does NOT modify `install.sh`";
   `install.sh` grep `opencode` = 0.
3. `package.json:34-42` — no `test:kaola-workflow:opencode` script.
4. `package.json:35` — `npm test` chains only `claude && codex && gitlab && gitea`; opencode is
   never gated at Finalization's `npm test`.
5. `CLAUDE.md:48, :102, :103` — the four npm chains, the Cross-edition Validation Policy (forge
   trees only), and the SIX-surfaces rule (Claude+Codex) are all **forge-scoped**.

Consequence today: opencode parity (`test-opencode-edition.js` + `--check`) runs **only** via
`install-opencode.sh:62-63` (the full 223-assertion suite on every install) + manual invocation —
never in `npm test`, so not gated at Finalization.

**The fragility that bears on this decision is empirically verified** (`.cache/n3-critique.md`
§2). A11 (`test-opencode-edition.js:286`) runs `node --check` on the ESM plugin
`.opencode/plugins/kaola-workflow-hooks.js`; `.opencode/package.json` has NO `"type":"module"`
(and is gitignored, `.opencode/.gitignore:2-3`).

| Check | Result |
|---|---|
| `node --version` | v24.14.0 |
| `node --check .opencode/plugins/kaola-workflow-hooks.js` | exit 0 (`--experimental-detect-module` default ON) |
| `node --no-experimental-detect-module --check …` (sim Node < 22.12 / 20 LTS) | exit 1 — `SyntaxError: Cannot use import statement outside a module` |
| `.opencode/package.json` `"type":"module"` | ABSENT (gitignored) |
| `engines` / `.nvmrc` / `.node-version` / CI workflows | ALL ABSENT |

The plugin is ESM. Node ≥ 22.12 passes A11; Node 20 LTS / pre-22.12 **hard-fails**. There is no
declared Node floor and no CI. Production runtime (Bun) auto-detects ESM regardless of
`"type":"module"` → unaffected (defect R2 is the portability follow-up, not a runtime defect).

## Decision

**Candidate B — opencode stays additive (runtime-not-forge).** It is **NOT** wired into `npm
test`, **NOT** added to `edition-sync.js`, and **NOT** inserted into `install.sh`. opencode
remains a runtime edition with its own generator / installer / test.

Parity is enforced via:

1. **Install-time suite** — `install-opencode.sh:62-63` runs the full 223-assertion
   `test-opencode-edition.js` on every install (the inward-validation form consistent with #475 /
   D-475-01's direction).
2. **Manual invocation** — `node scripts/test-opencode-edition.js` (+ `--check` for drift) on any
   change touching the opencode tree.
3. **The new content-reachability assertions from D-530-01** (defect X1 follow-up) — closes the
   one substantive parity gap (PIN/literal token survival through `transformCommandBody`).
4. **Docs-discoverability follow-up** (defect P5) — so contributors know the edition exists and how
   to test it, without it being a forge chain.

## Alternatives considered

- **Candidate A — wire `test:kaola-workflow:opencode` into `npm test`** (CI enforcement alongside
  the other 4 chains). **REFUTED.**
  - **R2 (A11 `node --check` ESM fragility) is load-bearing:** a Node-version-specific failure in
    an additive non-forge edition would **block all four forge editions' Finalization**. A
    contributor on Node 20 LTS / pre-22.12 would see `npm test` fail at the opencode leg before
    the four forge chains even run — unacceptable coupling of a non-forge, Node-fragile check to
    forge Finalization.
  - **CLAUDE.md:48 names exactly FOUR npm chains; :102 scopes the Cross-edition Policy to forge
    trees; :103 scopes the SIX surfaces to Claude+Codex.** Wiring opencode in contradicts all
    three.
  - **#501 ("self-sufficient; CI/CD is not a gate") keeps correctness inward:** the four npm chains
    are *internal* self-contained gates owned by this repo; an external non-forge runtime edition
    is not one of them. The same logic that retires external CI as a gate (#475 / D-475-01) retires
    a non-forge edition from the forge gate.
- **Candidate C — add a `test:kaola-workflow:opencode` script name to `package.json` for
  discoverability WITHOUT chaining it into `npm test`** (a half-measure on enforcement). **Optional
  add-on to B**, not a replacement. It improves discoverability (a named script is easier to find
  than `node scripts/test-opencode-edition.js`) without the coupling cost of A. Can be combined
  with B at any time; the P5 docs-discoverability follow-up is the more complete answer (it also
  covers `README.md` and `CLAUDE.md`).

## Consequences

- **The deliberate additive boundary is preserved.** opencode remains a runtime edition; the forge
  machinery (`edition-sync.js`, `install.sh`, the four-chain `npm test`, the SIX-surfaces T-sets)
  stays forge-scoped, exactly as CLAUDE.md specifies. No #307 four-chain obligation attaches.
- **Regression risk is mitigated, not eliminated, by design.** An opencode regression is not
  detected by `npm test` — but it IS detected by the install-time suite (every install runs 223
  assertions), by D-530-01's content-reachability assertions (once implemented), and by the P5
  discoverability follow-up (so contributors know to run the opencode test on relevant changes).
  This is the consciously-accepted tradeoff of the additive posture, recorded here in full.
- **No code change ships from this decision.** It is a recorded posture (the frozen plan's #486
  Case-A discipline: the acceptance criterion is a *recorded decision*, not an implementation).
  Candidate C and the P5 docs fix remain available as future follow-ups without reopening this
  decision.
- **R2 (Node-version fragility of A11) stays a follow-up** independent of this decision — hardening
  A11 (e.g. add `"type":"module"` to `.opencode/package.json`, or migrate the check strategy) is
  worthwhile for portability regardless of whether opencode ever joins `npm test`. It does NOT flip
  this decision: even with A11 hardened, CLAUDE.md's forge-scoped policies and #501's inward-gate
  principle keep a non-forge edition out of the forge gate.
