# D-435-01. Run-gap capture is machine-gated at finalize: swept defect classes must be filed or justified.

Date: 2026-06-13
Status: Accepted
Issue: #435
Related: `scripts/kaola-workflow-gap-sweep.js` (new SUPPORT script — the scanner + gate),
         `docs/conventions.md` § Run-gap capture is gated at finalize (#435),
         `docs/architecture.md` § Chain receipt / provenance log (finalize gate area),
         D-432-01 (chain-receipt machine gate — the structural sibling this mirrors),
         Contractor Step 8c.2 (the wiring point — NO `cmdFinalize` edit),
         The "anti-fabrication" lesson: dispatch/provenance logs carry no result/reason fields;
         the scanner uses only machine-reliable provenance signals

## Context

During a live adaptive run, run-discovered defect signals — nodes that required mid-run repair
(provenance reopens), known-red test chains accepted with a waiver (chain-receipt
`accepted_red:true`) — were captured only in prose: the operator wrote a note, or the signals
were silently dropped. Two structural problems feed the gap:

- **ANTI-FABRICATION reality.** `dispatch-log.jsonl` carries `{ts, agent_type, agent_id, cwd}`.
  `provenance-log.jsonl` carries `{timestamp, event, nodeId, nonce, by}` where `event` is in
  `{open, close}`. `node-timings.jsonl` carries `{node, event, ts}`. NONE of these carry
  `result`, `reason`, or `force` fields. A per-finding "reason-class" scanner that reads
  result/reason from these logs would fabricate fields that do not exist. The three machine-
  reliable defect signals are: (1) provenance reopens — a `nodeId` with more than one `open`
  event in `provenance-log.jsonl` indicates an in-run repair (e.g. issue-442 reopened n2/n3);
  (2) chain-receipt `accepted_red:true` — a chain in `.cache/chain-receipt.json` that was
  waived with `--accept-known-red` is a deferred red chain; (3) operator-seeded
  `.cache/run-gaps-manual.md` — a file the operator may populate with `gap: <class> — <text>`
  lines for signals that cannot be machine-derived.

- **Prose drift.** A human or an agent can write "no defects found" without having checked,
  or after an abbreviated review. There was no machine-checkable artifact binding a "no gaps"
  claim to the run's actual provenance and chain-receipt artifacts.

The fix mirrors D-432-01's chain-receipt approach: make run-gap capture a **machine-verifiable
artifact** written by a scanner, and gate finalization by checking that every swept gap has been
mapped in a structured `## Run gaps` section of the finalization summary.

## Decision

A new SUPPORT script scans the per-project `.cache/` for run-discovered defect signals,
deduplicates them, writes a structured artifact, and gates finalization by verifying that every
swept gap is either filed (linked to an open tracking issue) or justified (marked as noise with
a one-line reason).

### 1. Reason classes (closed enum)

`kaola-workflow-gap-sweep.js` recognizes exactly three reason classes:

- `in_run_repair` — a `nodeId` with more than one `open` event in `provenance-log.jsonl`;
  `sample` = the `nodeId`, `count` = extra opens beyond the first.
- `deferred_red_chain` — a chain in `chain-receipt.json` with `accepted_red: true`;
  `sample` = `<chain-name>:<accepted_red_issue>`, `count` = 1.
- `manual:<kebab-slug>` — a line from `.cache/run-gaps-manual.md` matching
  `gap: <class> — <text>` (em-dash or ASCII ` - ` separator accepted); `slug` is the
  kebab-normalized class token, `sample` is the trailing text.

Items are deduplicated by `(reasonClass, sample)` pair; for duplicates the `count` fields are
summed.

### 2. Scanner mode (`--project P --json`)

The default mode scans only `kaola-workflow/<P>/.cache/` (scope guard — no archive bleed),
collects items from `provenance-log.jsonl`, `chain-receipt.json`, and the optional
`run-gaps-manual.md`, deduplicates them, and writes `.cache/run-gaps.json`:

```json
{ "project": "<P>", "sweptClasses": [{ "reasonClass": "...", "sample": "...", "count": 1 }] }
```

With `--json` the scanner also emits one line to stdout:

```json
{ "result": "swept", "project": "<P>", "sweptClasses": [...], "artifact": "<path>" }
```

`result` is always `"swept"` (the scanner never refuses); exit code is always 0. An empty sweep
emits `sweptClasses: []`.

### 3. `## Run gaps` section grammar

The `finalization-summary.md` must contain a `## Run gaps` section when any swept classes exist.
One line per `(reasonClass, sample)` tuple, in one of two forms:

```
- <reasonClass> (<sample>): filed: #N
- <reasonClass> (<sample>): noise: <one-line justification>
```

`filed: #N` links the gap to an open tracking issue (the issue number is probed for existence
when online). `noise: <text>` justifies the gap as not worth tracking. Both forms satisfy the
gate. Silent omission — not mapping a swept tuple — does not.

### 4. Gate mode (`--check [--json]`)

`--check` reads `.cache/run-gaps.json` and the `## Run gaps` section of `finalization-summary.md`
and refuses or passes:

- **Vacuous pass**: `sweptClasses` is empty — passes immediately even if the `## Run gaps`
  section is absent.
- **`{result:'refuse', reason:'gaps_unswept', unmapped:[{reasonClass,sample}]}`** (exit 1):
  emitted when the `## Run gaps` section is absent while swept classes are non-empty, OR when
  one or more swept `(reasonClass, sample)` tuples are not matched by any section entry, OR when
  a `filed: #N` entry references an issue that does not exist online (online probe only; when
  offline the probe is skipped and `verification: "offline"` is noted on a pass).
- **`{result:'pass', mapped:N, filed:N, noise:N}`** (exit 0): all swept tuples are mapped.

Callers MUST classify the gate result by the typed `reason` field, never by string-matching
the detail message.

### 5. Wiring (contractor Step 8c.2; NO `cmdFinalize` edit)

This mirrors D-432-01's run-chains wiring: the gate is owned by the contractor, not by
`cmdFinalize` (`kaola-workflow-claim.js`). The wiring points are:

- **`contractor.md` Step 8c.2** (after Step 8c / chain-receipt): run scanner (`--json`, writes
  `run-gaps.json`), then run gate (`--check`); capture the real exit code; surface
  `gaps_unswept` and stop if the gate refuses.
- **`contractor.toml` sibling bullet** (after the #432 bullet): matching prose.
- **`commands/kaola-workflow-finalize.md`** (`### Run-Gap Sweep Gate` after `### Chain-Receipt Gate`).
- **`kaola-workflow/issue-*/finalization-summary.md` template** (`## Run gaps` line added after
  `## Chain receipt`).
- **Three finalize SKILLs** (the three Codex finalize SKILL packs): same `### Run-Gap Sweep Gate`
  subsection after `### Chain-Receipt Gate`.

Total: six prose surfaces (3 Claude finalize commands + 3 Codex finalize SKILL packs) — the
#400 six-surface propagation contract. NO edit to `kaola-workflow-claim.js` (`cmdFinalize`).

`kaola-workflow-gap-sweep.js` registers as a SUPPORT script (runtime, run by the contractor at
finalize) on all four editions:

- Gitea port: `plugins/kaola-workflow-gitea/scripts/kaola-gitea-workflow-gap-sweep.js`
  (edition-named, functionally identical)

The script also registers in `validate-script-sync.js` `RENAME_NORMALIZED_FAMILIES` (after the
release family), `kaola-workflow-install-manifest.js` `SUPPORT_SCRIPTS` (in BOTH byte-paired
copies — `scripts/` and `plugins/kaola-workflow/scripts/`), and the gitlab + gitea contract
validators' `scriptFiles` and `installSupportScripts` lists. `install.sh` and `edition-sync.js`
consume these registries automatically; no manual edits to those files are required.

### 7. Goal-prose rule

The operator's goal in finishing an issue includes capturing run-discovered defects. Each gap
must be either filed (a tracking issue opened) or justified (noise). Filing satisfies the goal;
silent deferral violates it and causes `gaps_unswept` to refuse. This rule appears in
`workflow-next.md` and in the three Codex next/adapt SKILL packs (six surfaces per the #400
contract). It does NOT appear in `CLAUDE.md` (out of this node's frozen write set).

### 8. Dogfooding

This decision record was authored as part of the #435 adaptive run. The gap-sweep scanner was
run on the issue-435 project folder at finalize (Step 8c.2), and the `## Run gaps` section of
`finalization-summary.md` was populated before the gate was invoked. The feature gates its own
finalization.

## Consequences

**For operators:**

- After the LAST commit, the contractor runs `kaola-workflow-gap-sweep.js --project <P> --json`
  (writes `run-gaps.json`) then `--check` (gate). If any swept gap is unmapped, `gaps_unswept`
  blocks finalization until the operator adds `filed: #N` or `noise: <reason>` lines to the
  `## Run gaps` section of `finalization-summary.md`.
- A run with no provenance reopens, no accepted-red chains, and no `run-gaps-manual.md` sweeps
  zero classes and passes vacuously — no `## Run gaps` section is required.
- A known-red waiver accepted at chain-receipt time (`--accept-known-red name:issue`) will
  appear as a `deferred_red_chain` entry and must be mapped in `## Run gaps`. The `filed: #N`
  value may be the same issue already recorded as `accepted_red_issue` in the chain receipt.

**For the finalization prose surfaces:**

- Six finalize surfaces (3 Claude commands + 3 Codex SKILL packs) gain a `### Run-Gap Sweep Gate`
  subsection. Existing `### Chain-Receipt Gate` subsections are unchanged.

**For the test suite and SUPPORT-script registration:**

- `package.json` `test:kaola-workflow:claude` adds `node scripts/test-gap-sweep.js` after
  `test-release.js`. The script exercises: scanner empty sweep → vacuous pass; provenance-reopen
  detection; chain-receipt `accepted_red` detection; manual file parsing; `gaps_unswept` refuse
  (absent section, unmatched tuple); `filed:`/`noise:` pass; offline skip.
- All four `npm run test:kaola-workflow:*` chains must be green before finalize (verified by a
  chain-receipt, dogfooding D-432-01).
