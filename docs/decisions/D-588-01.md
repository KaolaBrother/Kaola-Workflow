# D-588-01. Write co-open width/mix coverage + the write-cap `max_concurrent` fix

Date: 2026-07-02
Status: Accepted
Issue: #588
Related: #591 (the companion per-member `leg_path`/`leg_branch` dispatch threading,
             `docs/decisions/D-591-01.md`, landed in the same bundle-588-591 audit)

## Context

The write co-open machinery (lane groups, per-member `.kw` legs, the commit-union
synthesizer) is the most heavily tested subsystem in the repo, but a 2026-07-02 coverage
audit found the coverage was narrow in width and mix even where it was deep in mechanism:

1. **Width-2-only.** Every lane-group / leg / synthesizer test (`D437-*`, `LEG-*`, `SYNTH-*`,
   `#552-*`) used exactly two legs (`A`,`B`). No test exercised a ≥3-leg octopus merge, and no
   test pinned the write-frontier `FANOUT_CAP` interaction (does a 5-wide disjoint write
   antichain co-open 4 and queue the 5th, or open all?). Read-only caps were already tested;
   the write-cap path was not.
2. **Mixed read+write ready frontier was unpinned.** Existing tests covered "a live write
   blocks new opens" and pure-read frontiers, but not a frontier containing both read
   verifiers and a `parallel_safe` write group — which subset opens, and in which order, was
   asserted nowhere.
3. **Task-mirror fail-open was tested for the (now-retired) batch path but not the
   running-set scheduler path** (`open-ready` / `close-node`).

## Decision

Extend `scripts/test-adaptive-node.js` with four real-git cases, each asserting durable
artifacts (ledger rows, `running-set.json`, worktree/branch existence, `rev-list --parents`
octopus counts, `diff-tree` vs the declared write union) — never stdout sentinels — matching
the existing `LEG-*`/`SYNTH-*` quality bar. Per the issue's acceptance criteria, any behavior
found undefined at width ≥3 or mixed composition was to be fixed (or explicitly, typedly
refused) rather than silently pinned as-is.

- **(a) `#588-3LEG-OCTOPUS-END-TO-END`** — a 3-leg disjoint write co-open (A/B/C): 3 legs
  provisioned (worktree + branch + manifest), per-leg barriers (A, B deferred; C last), an
  octopus merge with EXACTLY 4 parents (feature + 3 legs), a passing commit-union barrier
  (diff base→merge == the declared union), all 3 members complete, legs torn down. A width-3
  extension of `SYNTH-DISJOINT`. **PINNED CORRECT — no defect.**
- **(b) `#588-WIDE-CAP` + `#588-WIDE-DRAIN`** — a 5-wide disjoint write antichain co-opens
  EXACTLY 4 (the write `FANOUT_CAP`, default 4) and queues the 5th, correctly. **DEFECT
  FOUND:** the running set recorded `max_concurrent` as the READ cap (`resolveFanoutCapReadonly`,
  default 8) instead of the WRITE cap (`resolveFanoutCap`, default 4). `reconcile-running-set`
  uses `max_concurrent` as its crash-resume roll-forward ceiling, so a write group could be
  reconciled forward to more members than co-open ever legally opens — RED was the two
  `#588-WIDE-CAP` assertions (`got 8`, expected `4`).
  **FIX (smallest correct change, all 4 editions):** hoist the write-group's actual ceiling
  into a new `laneGroupCeiling` local — `resolveFanoutCap` folded with `--max`, assigned
  atomically with `groupForm` inside the ≥2-member write branch (so it is always ≥2 whenever
  `groupForm` is set, and never read when it isn't) — and record
  `const maxConcurrent = groupForm ? laneGroupCeiling : <pre-existing read-cap logic>`.
  Read/serial paths are byte-unchanged.
  **PINNED CORRECT (post-fix):** open 4 / queue 5th (pending, no leg); `running-set.json`
  `max_concurrent==4`; the legs manifest covers exactly the 4 opened members; reconciling a
  crashed mid-open (all 4 rows `opening:true`) rolls all 4 forward (ceiling==width==4) and
  preserves `max_concurrent==4` with all 4 legs retained; top-up drain — after the width-4
  group closes (an octopus with EXACTLY 5 parents: feature + 4 legs) — opens the queued 5th
  serially (a single write, no group, no leg).
- **(c) `#588-MIXED-FRONTIER`** — a frontier with 2 read members (`code-explorer`,
  `knowledge-lookup`) plus a disjoint write pair (A,B): `open-ready` opens ONLY the reads; A,B
  stay pending; the running set reflects EXACTLY the two read members; no `lane_group`, no
  legs. A second `open-ready` call while the reads are live returns
  `reason:'write_awaits_drain'`, `opened:[]`. **PINNED CORRECT (deliberate, not a defect):**
  this is the existing "a write node runs strictly alone" invariant — co-open-by-default is
  scoped to write‖write frontiers only (a legged write is synthesized into the feature `HEAD`
  at close, so it is never co-scheduled with a read). Read‖write co-scheduling would be a NEW
  capability, out of scope for a coverage-only issue (accuracy over efficiency), so the
  behavior was pinned as-is rather than changed.
- **(d) `#588-TASKMIRROR-FAILOPEN`** — forced the durable task-mirror write to fail
  (`workflow-tasks.json` made a directory ⇒ `EISDIR` ⇒ non-zero task-mirror exit) on BOTH
  scheduler subcommands. `open-ready` still returns `result:'ok'` with
  `taskMirror.status=='failed'` and the ledger co-open still advances (A,B `in_progress`,
  `running-set.json` + legs written); `close-node` still returns `result:'ok'` with
  `taskMirror.status=='failed'` and the ledger still advances to `complete`. **PINNED
  CORRECT:** the fail-open contract, previously proven only for the retired batch path, now
  holds on the running-set scheduler path.

`kaola-workflow-next-action.js` was audited and confirmed unchanged — no #588-discovered
defect required a fix there.

## Consequences

- `reconcile-running-set`'s crash-resume roll-forward ceiling for a write lane group is now
  always the WRITE cap the group actually co-opened under, never the (larger) read-only cap —
  closing the over-roll-forward hazard case (b) found.
- **Stale pre-fix manifests are provably harmless.** A `running-set.json` written by the
  pre-fix code could carry `max_concurrent:8` for a write group whose real membership never
  exceeded 4 (co-open itself was never wrong — only the recorded ceiling was). Reconcile's
  roll-forward is bounded by the members actually PRESENT in the manifest (`keptAll`), which
  can never exceed 4 for any write group regardless of what `max_concurrent` says, so a
  stale-8 manifest reconciles identically to a correct-4 one — no double-check or migration
  was required.
- Read-only and serial `max_concurrent` recording is byte-unchanged (`groupForm` is falsy on
  both paths) — no regression to the pre-existing read-cap behavior.
- Suite assertion count grew from 1127 to 1219 (+92 total; +79 from this issue's four cases,
  +13 from the concurrently-landed #591 leg-dispatch threading in the same commit).
  `scripts/test-adaptive-node.js`'s `makeLaneRepo` gained a backward-compatible
  `extraMembers` parameter (byte-identical fixture output when the parameter is omitted) to
  support case (a)'s 3-member group without disturbing the existing width-2 fixtures.
- Cross-edition: `scripts/kaola-workflow-adaptive-node.js` (canonical + codex twin
  byte-identical via `sync:editions`) and both forge ports (GitLab, Gitea) carry the
  `laneGroupCeiling` fix identically; all four `npm run test:kaola-workflow:{claude,codex,
  gitlab,gitea}` chains are required green before release per the standard #307 cross-edition
  obligation.

## Alternatives considered

- **Leave `max_concurrent` recording the read cap and special-case `reconcile-running-set` to
  re-derive the write ceiling for a write group at reconcile time.** Rejected: this duplicates
  the ceiling-resolution logic at a second call site (reconcile would need its own
  `resolveFanoutCap`/`--max` fold, independently kept in sync with open time) for no benefit
  over recording the correct value once, at the point it is already computed.
- **Record two ceiling fields (`max_concurrent` for reads, a new `write_max_concurrent` for
  write groups) and let reconcile branch on which is present.** Rejected: `max_concurrent` is
  already documented as "the ceiling this open opened under" — for a write group that ceiling
  IS the write cap; adding a second field to express the same fact is unnecessary schema
  growth for a single-purpose value.
