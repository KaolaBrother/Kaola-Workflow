# D-431-01. generated_port_split is a freeze wall (aggregator-coupling rule)

Date: 2026-06-12
Status: Accepted
Issue: #431
Related: `scripts/kaola-workflow-plan-validator.js` (`validatePlan`, write-set shape wall),
         `scripts/edition-sync.js` (`GENERATED_AGGREGATORS`, `forgeRel`),
         `docs/decisions/0009-freeze-wall-absolute-path-and-finalize-probe-classification.md`,
         `docs/api.md` § Adaptive Plan Validation (freeze-only write-set shape refusals, issue #381),
         Issue #291 (the forge-port ordering gap — the class of defect this wall prevents)

## Context

Several canonical scripts in `scripts/` are **generated aggregators** — files whose forge-specific
ports are REGENERATED from the canonical source by `sync:editions` /
`edition-sync.js --check`. The set is enumerated in `scripts/edition-sync.js` as
`GENERATED_AGGREGATORS` (e.g. `kaola-workflow-plan-validator.js`,
`kaola-workflow-adaptive-node.js`, and their siblings). For each such canonical file the
four-edition surface is:

1. `scripts/<base>` — the canonical source (read/written by plan nodes).
2. `plugins/kaola-workflow/scripts/<base>` — the Codex byte-twin.
3. `plugins/kaola-workflow-gitlab/scripts/kaola-gitlab-workflow-<base>` — the GitLab
   edition-named port.
4. `plugins/kaola-workflow-gitea/scripts/kaola-gitea-workflow-<base>` — the Gitea
   edition-named port.

The aggregator-coupling rule is that all four files MUST be owned by the same node. Splitting them
— declaring the canonical file in one node's write set and the ports in another — is structurally
unsafe because:

- The port content is derived from the canonical content. The `--barrier-check` (per-node,
  `--node-id`) measures writes against the node's OWN declared write set. A node that writes only
  the ports has no canonical in its set; a node that writes only the canonical has no ports in its
  set.
- If the canonical edit lands in node A and the ports are regenerated by `sync:editions` in node
  B, the per-node barrier for node A checks the canonical against node A's write set (pass) and
  ignores the ports (they are not declared). Node B's barrier then sees the port regeneration as an
  attributed write — but the canonical change that CAUSED the regeneration was committed by node A.
  The whole-plan barrier at finalization eventually catches this as `write_set_overflow` (the ports
  are a production write outside the declared set of the node that actually wrote them) or
  `unattributed_write`, but the error arrives at the worst possible moment: mid-run or at the
  finalization merge gate, after days of elapsed work.

Issue #291 surfaced exactly this class of defect: edition-port nodes in a plan were ordered before
their root-writing sibling nodes, causing `write_set_overflow` at barrier time. The forge-port
ORDERING gap (the `--freeze` refusal added in #340) prevents the ordering problem but does not
prevent a split between the canonical and its ports into different nodes. A plan where the canonical
and its ports are in DIFFERENT (correctly ordered) nodes still violates the aggregator coupling
rule and will produce a `write_set_overflow` at the per-node barrier when `sync:editions` runs in
the port node and attributtes the regenerated files to that node's write set.

Two remediation postures were considered:

1. **Barrier-time detection:** detect the split at the per-node `--barrier-check` by comparing
   actual writes to declared write sets and emitting a specialized error when a generated port is
   written without its canonical in the same node's set.
2. **Freeze-time prevention:** detect the split at `--freeze` (and default `--json` validate) by
   inspecting each node's declared write set for a canonical aggregator path without its full
   four-file group in the SAME node.

## Decision

**`generated_port_split` is a freeze wall** — detected at `--freeze` (and default `--json`
validate) by `validatePlan()`, refused with a typed error naming the node and the missing peers.
Barrier-time detection is NOT used for this class.

**Why freeze-time, not barrier-time:**

**1. Planning-time enforcement is more actionable.**

A freeze-time refusal arrives before ANY work has started. The author is still in the plan-authoring
loop, the fix is trivially a write-set merge (move the port declarations into the same node as the
canonical), and no executor state has been committed. A barrier-time failure arrives after the
canonical node has committed its changes and possibly after other nodes have run. Restoring a clean
repair at that point requires a ledger-reset, a new baseline snapshot, and potentially a plan-repair
via `--freeze`. The actionability gap between freeze-time and barrier-time is large.

**2. The #291 defect showed that barrier-time is too late.**

Issue #291's post-mortem (the defect was discovered mid-run, not at authoring) is the concrete
evidence that the class of error is hard to recover from at barrier time and produces a confusing
`write_set_overflow` that does not name the aggregator-coupling rule. The freeze wall catches the
plan before it enters execution and names the exact rule violation.

**3. `write_set_overflow` is a less actionable error than `generated_port_split`.**

A `write_set_overflow` barrier refusal names the files that were written outside the declared set
but does not tell the author WHY those files were out-of-set. An operator seeing `write_set_overflow`
on a port file must trace back through the plan, understand that the ports are generated from the
canonical, and figure out that a node-split is the root cause. The typed `generated_port_split`
freeze refusal names the canonical, the missing peers, and the node — the operator knows
immediately what to fix.

**4. The implementation is anchor-gated and inert on non-canonical installs.**

The `GENERATED_AGGREGATORS` constant and the `forgeRel` mapping function are imported from
`scripts/edition-sync.js`. The import is wrapped in a `try/require` anchor gate: when
`edition-sync.js` is absent (forge/codex/user installs that do not ship `edition-sync.js`), the
guard is a no-op and no plans are refused. The wall only fires in the Kaola-Workflow development
repo itself, where aggregator coupling is enforced by the `sync:editions` pipeline.

**5. Consistency with the existing freeze-wall pattern (issue #381).**

The existing freeze-wall for directory-shaped write-set entries (`src/` → refused at freeze,
reported as a mid-run `write_set_granularity` on in-flight plans) established the pattern: new
write-set shape violations are freeze-only, `--resume-check` is unaffected, in-flight plans resume
without interruption. `generated_port_split` follows the same pattern.

## Consequences

**For plan authors:**

- A node that declares `scripts/kaola-workflow-plan-validator.js` (or any other
  `GENERATED_AGGREGATORS` member) without ALSO declaring the three peer paths
  (`plugins/kaola-workflow/scripts/<base>`, `plugins/kaola-workflow-gitlab/scripts/kaola-gitlab-*`,
  `plugins/kaola-workflow-gitea/scripts/kaola-gitea-*`) in the SAME node is a typed freeze-time
  refusal. Move the port declarations into the same node as the canonical.
- The four-file group may be split across multiple nodes ONLY when each piece is a read-only
  reference (no write set declared for the canonical); the check applies only when the canonical
  appears in a node's declared write set.
- Plans frozen by a pre-#431 validator with a split are NOT retroactively invalidated —
  `--resume-check` does not apply the new freeze wall. The mid-run `write_set_overflow` or
  `unattributed_write` barrier is still the backstop for those plans.

**For the test suite:**

- Walkthrough scenarios added for `generated_port_split` (split-shape plan refuses at freeze) and
  the bundled-plan green path (all four files in one node passes freeze).

**For the plan-validator's freeze-wall section in `docs/api.md`:**

- `generated_port_split` is documented under the "Write-set shape refusals (issue #381, freeze-only;
  round-2 shapes #388)" paragraph as an additional freeze-only refusal.

**For downstream aggregator additions:**

- Any new canonical script added to `GENERATED_AGGREGATORS` in `edition-sync.js` is automatically
  covered by the freeze wall without any change to the plan-validator. The mapping is imported
  dynamically at validate time.
