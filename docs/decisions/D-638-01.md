# D-638-01. Widen `edition-sync.js --check` to cover the COMMON_SCRIPTS + BYTE_IDENTICAL_GROUPS mirrors, matching `--write`'s universe

Date: 2026-07-08
Status: Accepted
Issue: #638 (primary)
Related: #629/D-629-01 (the `runWrite` create-on-missing hardening that widened what `--write`
         regenerates — this decision gives `--check` parity with that wider universe); #365 (the
         original `edition-sync.js` design and the #347 hand-edit-drift class `--check` exists to
         catch)

## Context

`scripts/edition-sync.js` has two modes: `--write` (`npm run sync:editions`, regenerate) and
`--check` (wired into the gitlab/gitea chains, verify-only). `--write`'s `runWrite()` does three
things: (a) regenerate the `GENERATED_AGGREGATORS` forge ports from canonical via the declared
rename map, (b) copy each `COMMON_SCRIPTS` canonical file to its codex mirror, and (c) copy each
`BYTE_IDENTICAL_GROUPS` member's reference file to its sibling copies — steps (b)/(c) gained
create-on-missing behavior under #629/D-629-01.

`--check`'s `runCheck()`, however, only ever verified (a) — it recomputed each generated forge port
and byte-compared it against the committed file, but never re-checked the COMMON_SCRIPTS or
byte-identical-group mirrors. A missing or drifted mirror in either of those two families stayed
green under `edition-sync.js --check` alone. This was not a live hole end-to-end:
`validate-script-sync.js` runs its own byte-identity check over the same two families, in-chain, so
the enrollment loop was fail-closed overall — but it was a cosmetic `--check`/`--write` asymmetry: a
maintainer running `edition-sync.js --check` standalone, expecting it to verify what `--write`
writes, would get a false sense of a narrower guarantee than the tool's own `--write` half actually
maintains.

## Decision

Add `checkMirrors(rootDir, commonScripts = COMMON_SCRIPTS, byteGroups = BYTE_IDENTICAL_GROUPS)` to
`edition-sync.js`, exported alongside the module's existing helpers. It reuses
`checkByteIdenticalGroup` (imported from `validate-script-sync.js`) for **both** families:

- Each `COMMON_SCRIPTS` base name is checked as a degenerate 2-file group
  `{ label: base, files: [canonRel(base), codexRel(base)] }` — i.e. canonical vs. its codex mirror.
- Each `BYTE_IDENTICAL_GROUPS` entry is checked as-is (its own `{label, files}` shape).

This is the same primitive `validate-script-sync.js`'s own checker uses, so the two stay logically
identical by construction rather than by convention. The parameterized signature (defaulting to the
real constants) mirrors `checkByteIdenticalGroup`'s own exported convention, letting tests run
`checkMirrors` against a synthetic fixture tree instead of the real repo.

`runCheck()` now calls `checkMirrors(REPO)` and folds its `missing`/`drift` results into the same
`mismatches` list the `GENERATED_AGGREGATORS` loop already populates, so a single `--check` invocation
now covers exactly the three families `runWrite` actually writes: (a) forge aggregator ports, (b)
`COMMON_SCRIPTS` mirrors, (c) byte-identical groups. The console summary line was widened accordingly
(reports counts for all three families instead of only the forge-port count).

`RENAME_NORMALIZED_FAMILIES` — the hand-ported, forge-vocabulary-carrying data-layer scripts (claim,
sink-merge, roadmap, etc.) — stay deliberately outside **both** `--check` and `--write`: they are not
a pure rename-map reproduction of a canonical file (`edition-sync.js`'s stated #365 scope), so no
generation function exists for them here. `validate-script-sync.js` guards them separately, in-chain,
via its own normalized-diff check (`checkNormalizedFamily`).

## Consequences

- `edition-sync.js --check` now reds a missing or hand-drifted `COMMON_SCRIPTS` codex mirror, or a
  missing/drifted `BYTE_IDENTICAL_GROUPS` copy, directly — not only via `validate-script-sync.js`
  running downstream in the same chain.
- No change to `runWrite`'s behavior, its create-on-missing semantics, or its message format.
  `checkMirrors` is a new read-only verification path; nothing in `--write` was touched.
- Tests: `scripts/test-edition-sync.js` gained T9 (a synthetic fixture tree — green when in sync, red
  on a planted missing codex mirror, red on a planted drifted byte-group copy) and T10
  (`checkMirrors(REPO)` with no override args has zero missing/drift on the real committed tree,
  proving the widened check has no false positive on the actual repo).
- All four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains and
  `simulate-workflow-walkthrough.js` verified green after the change.

## Non-goals

- Does not change `runWrite`'s create-on-missing behavior (#629) — that hardening is unaffected.
- Does not extend coverage to `RENAME_NORMALIZED_FAMILIES` — out of scope for this tool; guarded
  elsewhere by `validate-script-sync.js`.
- Does not change the missing/drift message format `validate-script-sync.js` itself emits when it
  runs its own check on the same two families.

## Alternatives considered

- **Leave `--check` as-is and rely solely on `validate-script-sync.js`'s in-chain guard for these two
  families.** Rejected: it silently under-reports `edition-sync.js`'s own tool contract — a
  maintainer invoking `edition-sync.js --check` standalone (outside the full four-chain run) would see
  a false green on a drifted mirror that only `validate-script-sync.js` would have caught, had it been
  separately invoked.
