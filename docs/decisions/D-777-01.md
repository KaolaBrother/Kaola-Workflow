# D-777-01. Ledger tamper-evidence is deferred, not forgotten: `plan_hash` covers plan identity, and the execution record stays structurally checked

Date: 2026-07-24
Status: Accepted
Issue: #777 (deferred out of v7.0.0 by this decision; the issue remains open and inherits the design below)
Related: #771 (amend-surface atomic fold — the fix whose residual this decision bounds),
         #773 (`verifyGateExecution` widened to the execution node view),
         #778 (Oracle Kernel + the Oracle Test — the audit that produced #777, and the
         work this deferral unblocks), #765 / D-765-01 (spine-always cutover — why the
         executed graph now lives outside the hash), #372 and #725 / D-725-01 (retirement
         of the `PreToolUse`/`PostToolUse` interception hooks)

## Context

`computePlanHash` covers `## Meta` + `## Nodes` + optional `## Node Briefs`. The `## Node Ledger`
— the record of which nodes completed and which gates passed — is **outside** it, as is
`## Expansion Records`. This asymmetry is deliberate and correct as far as it goes: ledger statuses
mutate on every legitimate transition, so a hash over them would refuse every honest run.

The consequence is that `--resume-check` proves *the plan authored at freeze is unchanged* and
proves nothing about *the record of what was executed*. Nothing intercepts a direct `Edit` of the
ledger either: the interception hooks were retired in #372/#725, and
`plugins/kaola-workflow/scripts/simulate-kaola-workflow-walkthrough.js:247-250` **asserts** they
must never return. Every lock, gate, and digest in this system is entered voluntarily by an agent
obeying SKILL prose.

Two facts sharpened the question for the v7.0.0 cut:

- **#771** closed a reachable hole where an amend block could outlive a failed re-open. But
  `parseSurfaceAmendments` admits an amended surface on exactly `parseLedger(content).get(point) ===
  'complete'` — so attribution keys on a ledger status, i.e. on the unhashed half.
- **#765** made `plan_form: spine` the only authored form, so the executed graph — unit `role`,
  `writeSetRaw`, `mode`, `dependsOnNames` — now lives in `## Expansion Records`, also unhashed.

## Decision

**Defer the general ledger tamper-evidence mechanism out of v7.0.0.** Ship this ADR as the decision
of record, state the residual plainly in the release notes, and keep the structural checks that
actually stand over the execution record today. Four parts:

1. **The deferral is explicit, not silence.** The threat #777's acceptance criterion names — *"an
   out-of-band edit is detected and refused"* — is the agent-might-cheat threat that this
   architecture has **already decided not to defend**, by removing interception in #372/#725 and
   asserting it stays removed. Re-admitting that threat model is a values call (axiom 4: humans
   decide values), and belongs in a deliberate decision, not smuggled in as a bug fix — least of all
   in the same release as #778's precondition work, which rests on the opposite premise.

2. **`verifyGateExecution` is RETAINED as defense-in-depth** — this answers #777's open checkbox.
   It is not subsumed by a hash chain, on grounds that are independent:
   - It answers a **different question**. A chain asks *"did this state arise from a legal sequence
     of transitions?"*; `verifyGateExecution` asks *"does this state discharge the plan's
     post-dominance obligations?"* A perfect chain **passes** a run that legally crashed mid-frontier
     leaving an unsatisfied gate, and passes `foldSelectorArms` legally writing `n/a`.
   - The relabel is **not** a per-shape patch (#777's text mischaracterizes it). Relabeling rather
     than removing a non-`complete` gate keeps it as a pass-through edge, so `gateUncovered` asks
     *"is there a sink path from a complete target crossing no COMPLETE gate?"* — the correct
     execution-aware question for **any** reason a gate is non-complete: crash, halt, abandonment,
     operator skip, or tamper. Filtering instead would disconnect the target and mask the leak.
   - **#773 makes it more load-bearing, not less.** Widened to the execution view, it is the only
     structural check standing over `## Expansion Records` on the only shape the planner now authors.
   - It is a **blocking gate others compose with**: `--gate-verify` exits 1, `commit-node` folds it
     into `overallOk`, and `barrierCheck`'s `unattributed_write` floor documents the composition —
     forcing a producer to `complete` puts it back into G1/G2's target set. Retiring it would
     silently weaken `unattributed_write` too.

3. **The design of record is written down now**, so the deferred issue is decision-grade on day one
   rather than re-derived later. Summarized in *Design of record* below.

4. **v7.0.0's release notes state the residual** and phrase #771 as transactional integrity, never as
   a re-review guarantee.

## The residual, stated plainly

> Anyone who can write `workflow-plan.md` can set any `## Node Ledger` row to any status — including
> flipping a never-opened writer to `complete` or a gate to `n/a` — and no integrity check in any
> edition will notice. The remaining teeth are `verifyGateExecution`'s post-dominance relabel,
> `verifyCurrentEpochAuthority`'s row/progress floors, and `ledger-compare`'s regression guard.

Concretely for #771: it makes the amend block and the point's re-open **one transaction**, so an
amendment can no longer outlive a failed or crashed re-open. It does **not** make attribution
conditional on a re-review. Any route producing a `complete` row without a genuine close — a direct
`Edit`, a folder restored from a snapshot, a `cp -R` in the direction `ledger-compare` does not
refuse — re-opens attribution-without-re-review by a different door.

**Permitted claim:** "an out-of-surface write can no longer be attributed by an orphaned amendment."
**Forbidden claim:** "an amended file is always re-reviewed before the sink."

## Why not ship a mechanism now

- **Extending `plan_hash` is under-scoped by exactly one section, on the only shape the planner
  authors.** A chain over ledger *statuses* proves nothing about the unit role or write set a status
  attaches to — those live in the equally-unhashed `## Expansion Records`. Large cost, partial
  coverage of the surface it names.
- **Its dominant risk runs the other way.** The transition relation must be enumerated over **27
  `spliceLedgerNode` call sites** plus `appendLedgerRows` plus `reconcileLedger`'s `absent → pending`,
  including four paths that flip **many** rows inside **one** atomic write (`runReExpandOpen._reactivate`,
  `runRepairNodeCore` step 5, `runOpenReady` Phase 2, `foldSelectorArms`) — so #777's sketched
  per-row entry is not even expressible. Missing one legal transition **wedges honest runs on every
  frontier**, which is a worse failure than an undetected tamper, not a smaller one.
- **The cheap interim option is unsound as specified.** Binding the ledger to the barrier baseline
  rests on an unproven *`complete` ⇒ a baseline exists* invariant, and is structurally blind to the
  shape that matters: it cannot distinguish a legitimate `in_progress → complete` from a hand-typed
  one, because that transition is shape-legal. There is also no total order over baselines. Shipping
  it while #777's text promises "detected and refused" would put a false claim in the CHANGELOG.
- **Budget.** The same hours bought the `state_ledger_authority_invalid` blocker fix (#779) — a
  proven, honest-user failure in this release's headline machinery.

## Design of record (inherited by the deferred issue)

- **Shape:** append-only back-linked chain reusing `verifyConsentLedger`'s existing idiom — do not
  invent a second one.
- **Granularity: per durable WRITE, not per row** (the correction to #777's sketch), carrying
  `epoch_lineage_id`, `plan_hash`, `subcommand`, a `deltas` array, `post_ledger_digest` (reuse
  `ledgerDigest`), a **required** `expansion_records_digest`, `previous_entry_digest`, and
  `entry_digest` over `sha256Canonical`/`canonicalJson`.
- **Head placement:** an HTML-comment sibling of `<!-- plan_hash: … -->` after the first H1 — outside
  every `## ` section, therefore outside every `sectionBody` the hash reads. It must ride the **same
  atomic plan write** as the row it describes.
- **Explicitly NOT a ledger column** — `ledger-compare` is positional, so a column before `status`
  silently disables `would_regress_complete_ledger` with no error. **NOT a ledger row** —
  `verifyCurrentEpochAuthority` requires row-count equality.
- **Scope over `parseLedger`'s `id → status` map, never section bytes** — `--freeze --repair`'s
  `normalizeLedgerHeader` and `reconcileLedger` legitimately rewrite them.
- **Write ordering:** journal append + fsync **first**, plan write **second** (journal-ahead rolls
  forward; the reverse is unrecoverable). Append **only when bytes actually changed** —
  `spliceLedgerNode`'s `alreadyAtTarget` path skips the write, so a crash-resume must produce no
  entry or every resume grows a phantom link.
- **Verification slot — the critical integration constraint.** `mutationGuardPrologue` Layer 1
  recomputes `computePlanHash` in-process and, on a match, **skips the `--resume-check` subprocess
  entirely**. Because the ledger is outside `plan_hash` by construction, every ledger tamper leaves
  the hash matching, so the skip is **always** taken — a chain check placed only in
  `revalidateForResume` would pass its unit tests and be **dead in production**. Either the fast path
  verifies in-process, or `cfg.integrity` stops being satisfiable by a hash match alone. Keep
  `revalidateForResume` content-pure: thread an injected `readCache` seam (the pattern
  `verifyGateExecution` already uses).
- **Migration: tolerate-and-adopt, keyed on head absence, one-way.** No head ⇒ not in force ⇒ PASS
  (the `verifyConsentLedger(null)` arm, and the shape #765 chose for `dag` on resume). The first
  mutating transition writes a **genesis snapshot**, not a forged history. Head present + journal
  missing/unparseable ⇒ **REFUSE**, or deleting the journal is a universal bypass. Genesis is
  per-epoch, bound to `epoch_lineage_id`.
- **Blast radius, measured:** ~293 hand-authored `## Node Ledger` fixtures across 27 files. Several
  harnesses use in-memory `readFile` seams that throw `ENOENT` for any path other than
  `workflow-plan.md`, so the journal read must be fail-soft or they break en masse.
- **Preconditions:** #776 (a `.cache`-resident journal inherits the exact defect #776 fixes) and,
  if it lands, #778's kernel — do not author new 4×-duplicated crypto immediately before a
  de-duplication refactor.

## Consequences

- v7.0.0 ships with ledger tamper-evidence **absent and documented**, not absent and unmentioned.
  The release notes carry the residual above.
- #777 stays open, re-scoped by this ADR: its per-row entry sketch is superseded by per-write
  granularity, its `verifyGateExecution` checkbox is answered **RETAINED**, and its verification slot
  is pinned to the `mutationGuardPrologue` fast path.
- **#778 is unblocked.** Its precondition was that the ledger question be resolved *or explicitly
  deferred with a decision*; this is that decision. #778 still lands alone, after v7.0.0.
- Per `CLAUDE.md` § First Principles: axiom 1 governs (a mechanism that wedges honest runs on every
  frontier is not "correct first"), axiom 4 routes the re-admission of a declared-out-of-scope threat
  model to a human values call, and the tighten-only boundary is respected — nothing here loosens an
  existing gate; `verifyGateExecution` is retained at #773's widened scope.
