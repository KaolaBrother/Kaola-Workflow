# D-589-01. Adversarial-verifier majority-refute ties break toward refuted

Date: 2026-07-02
Status: Accepted
Issue: #589

## Context

`verifyVerdictBlock`'s fan-out arm for the `adversarial-verifier` gate role collects each fan-out
sibling's per-instance `.cache/adversarial-verifier-*.md` verdict and computes `majorityRefute` to
decide whether the gate as a whole refutes. The comparison was a strict majority:
`refutes * 2 > verdicts.length`. For an odd-width fan-out this always resolves (no tie is
possible), but for an even-width fan-out (most commonly a 2-wide fan-out) a 1-1 split is a genuine
tie, and the strict-`>` comparison resolved every tie toward PASS. That inverts the validator's
own governing posture elsewhere: risk classification is over-approximated and fail-closed
(uncertain ⇒ risky — see the Governance section of `docs/api.md`), so a coin-flip 1-1 skeptic
split should not silently pass; it should force the same reopen-and-review path a clear refute
does.

## Decision

- `verifyVerdictBlock`'s `majorityRefute` computation changes from `refutes * 2 > verdicts.length`
  to `refutes * 2 >= verdicts.length` — a required-strict-majority-**to-pass** rule instead of a
  required-strict-majority-to-refute rule.
- Odd-width fan-outs are unaffected (e.g. 2 of 3 refuting still refutes at `4 >= 3`; 1 of 3
  refuting still passes at `2 < 3`).
- Even-width fan-outs now resolve a tie toward refuted (e.g. a 2-wide fan-out with 1 refute / 1
  pass now refutes at `2 >= 2`, where it previously passed because `2 > 2` was false).
- The emitted `reason` string (`fanout majority-refute: <refutes>/<verdicts.length> skeptics
  refuted`) is unchanged — callers that classify structurally on `verdict: 'fail'` / the `reason`
  field need no update.
- Mirrored byte-identically across the `scripts/`, Codex (`plugins/kaola-workflow`), GitLab, and
  Gitea plan-validator copies.

## Consequences

- A tied adversarial-verifier fan-out now blocks the gate (`verdict: 'fail'`) instead of passing,
  matching the fail-closed governance posture applied everywhere else in the validator.
- No planner-authored plan changes shape or hash; this is a pure runtime interpretation change in
  `--verdict-check` / `verifyVerdictBlock`, applied uniformly to every fan-out width.
- A previously-tied 2-wide adversarial-verifier fan-out that had passed under the old rule now
  requires a `reopen-node` and a fresh review round to clear the gate.
