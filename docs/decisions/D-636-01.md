# D-636-01. Relocate cross-runtime dispatch-pin assertions so the #627-descoped dead prose can be fenced off the plan-run routing surfaces

Date: 2026-07-08
Status: Accepted
Issue: #636 (primary — fix#2 of #627, descoped there as "not achievable as a prose-only change")
Related: #627/D-627-01 (the debloat bundle that shipped fix#1/#3/#4/#5 and descoped fix#2 to this
         issue); #611 (the fork-turns unconditional-mandate family whose assertions this decision
         relocates); #606 (the teammate-mode family whose assertions this decision relocates);
         #630 (the routing-surface generation seam this decision's two PIN markers set up slot
         boundaries for — still open);
         `docs/investigations/2026-07-08-630-636-routing-generation-seam.md` (the read-only
         shaping run, "Build Run 1", that produced this decision's relocation map)

## Context

#627 found the six plan-run routing surfaces (3 Claude commands + 3 Codex SKILLs, across the
GitHub/GitLab/Gitea editions) carrying two prose blocks that are dead weight on their non-native
runtime: the **Codex-dispatch effort/v2/v1 block** (the `turn_context.effort` proof gate,
`fork_turns: "none"`, the v1/v2 tiered-dispatch split) is meaningless on the 3 Claude commands,
and the **Teammate-Mode block** (spawning a role agent as a named teammate) is meaningless on the
3 Codex SKILLs. #627 could not fence either block out of its non-native surface as a prose-only
edit: `scripts/test-route-reachability.js`'s T5b array and the `#611`-fork-turns family pinned the
Codex-dispatch tokens on the commands too, and its T14 array and the `#606` teammate family pinned
the teammate tokens on the SKILLs too — mirrored across all four `validate-*-contracts.js`. Fencing
either block out of its non-native runtime without relocating those assertions would turn the
route-reachability test and all four contract validators red on their next run. #627 therefore
descoped this exact relocation to a follow-up issue (#636).

A prior read-only shaping run
(`docs/investigations/2026-07-08-630-636-routing-generation-seam.md`, "Build Run 1") produced the
settled, adversarially-validated relocation map this decision executes, including a specific
correction it surfaced: an earlier draft of the map would have left the `#611`-fork-turns family
asserting its tokens on the command surface in one of the four validators, which — once the
Codex-dispatch block is fenced off the commands — would assert tokens on a surface that no longer
contains them and red all four chains.

## Decision

**Single-source each cross-runtime block to the surface that actually executes it, and relocate
every assertion that pinned it cross-runtime so it now tracks only the native surface.**

1. **Fence the Codex-dispatch block out of the 3 commands**, preserving the always-live
   `Pass dispatch.nonce (evidence-binding token). Instruct the role to:` tail. The two forge
   commands (`plugins/kaola-workflow-{gitlab,gitea}/commands/kaola-workflow-plan-run.md`) had that
   sentence fused into the start of the Codex-dispatch block rather than living separately (as it
   does on the GitHub command), so their fence required a start-splice (replacing the fused tail
   with the preserved sentence) in addition to the end-splice the GitHub command needed — handled
   per file so the always-live base-dispatch sentence survives on all three.
2. **Fence the Teammate-Mode block out of the 3 Codex SKILLs**, preserving the always-live
   delegate-to-base-role-profile tail that follows it.
3. **Add two PROVENANCE_BAN-safe semantic markers** delimiting the fenced-in blocks:
   `<!-- PIN: teammate-mode -->` immediately above `#### Teammate-Mode Dispatch` on the 3 commands,
   and `<!-- PIN: codex-dispatch -->` immediately under the `## Dispatch` heading on the 3 SKILLs.
4. **Relocate every assertion that pinned the fenced tokens cross-runtime:**
   - `scripts/test-route-reachability.js`: the T5b `planRunSurfaces` array (Codex effort tokens)
     narrows to SKILL-only; the T14 array (teammate tokens) narrows to command-only.
   - `scripts/validate-workflow-contracts.js` and its byte mirror
     `plugins/kaola-workflow/scripts/validate-workflow-contracts.js`: the `#582` command-side
     Codex-effort assertion block is deleted outright (it now has no home — the SKILL side already
     had its own equivalent); the `#606` teammate array narrows to command-only; the `#611`-fork
     array narrows to SKILL-only.
   - `scripts/validate-kaola-workflow-contracts.js` (the GitHub Codex validator): the SKILL-side
     `#606` teammate block is deleted (command surface already asserted separately); the
     `#611`-fork loop drops its command-surface entry, leaving it SKILL-only.
   - The GitLab and Gitea contract validators each had a single shared `[command, SKILL]` loop
     covering several pin families at once. That loop is split three ways: symmetric families
     (`#602`, `#604`, `#605`, `#607`, the `#611`-join family) stay in the shared loop untouched;
     the `#606` teammate family is pulled out and relocated to a command-only block; the
     `#611`-fork family is pulled out and relocated to a SKILL-only block.
5. **The load-bearing correction:** the `#611`-fork SKILL-only shrink had to land in **all four**
   validators (plus the byte mirror), not just the ones an initial pass might target. The shaping
   run's adversary empirically proved that omitting it from even one — the shape an earlier draft
   of the map had — reds all four chains, because the fork-turns unconditional-mandate tokens
   (`on EVERY dispatch, tiered or not` / `the unconditional mandate applies identically to this
   dispatch mode`) would still be asserted on a command surface that, after step 1, no longer
   contains them. This is recorded here as the decision's key risk, and it was designed out by
   applying the SKILL-only relocation identically across `validate-workflow-contracts.js`, its
   byte mirror, `validate-kaola-workflow-contracts.js`, and both forge validators.

The deliberate GitLab/Gitea finalize-sink `mr|pr)` contract pins were explicitly left untouched —
they are an unrelated contract family outside this decision's scope.

## Consequences

- The six plan-run routing surfaces are smaller and each now carries only the prose block for its
  own native runtime: the 3 commands carry Teammate-Mode only, the 3 SKILLs carry Codex-dispatch
  only.
- The contract chains stay green: verified via `scripts/test-route-reachability.js` (239
  assertions), all four `validate-*-contracts.js`, `validate-script-sync.js` (byte-mirror parity),
  and the full four-chain `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` suite, all
  passing with zero orphaned assertions found across two independent sweeps (`code-reviewer` and
  `adversarial-verifier`).
- The two new semantic markers (`<!-- PIN: teammate-mode -->`, `<!-- PIN: codex-dispatch -->`) are
  currently inert (no validator asserts their presence or absence) but establish the slot
  boundaries and orphan-sentinel that #630's still-open routing-surface generation seam is
  expected to build on.
- `CHANGELOG.md` is intentionally out of this diff — that entry belongs to the finalize step of
  this run, not to this decision record.

## Non-goals

- **#636 does not generate the six surfaces.** Producing them from a single source (rather than
  hand-maintaining six near-duplicate files with fenced-off dead sections) is #630's two-layer
  manifest-plus-generation seam, deferred to its own build.
- **No change to the `#602`, `#604`, `#605`, `#607`, or `#611`-join symmetric assertion families.**
  They remain in their existing shared-loop or already-correctly-scoped form.
- **No change to the GitLab/Gitea `mr|pr)` finalize-sink contract pins.** Left untouched by design.
- **No change to card content** in `docs/plan-run-cards/` — this decision only relocates dead
  prose and its assertions, not the wait-budget/escalation/speculative-open card material #627
  already restubbed.

## Alternatives considered

- **Leave the dead prose resident (the #627-partial status quo).** Rejected: it blocks #630's
  fenced generation, since a generator cannot single-source a block that is unconditionally
  asserted on both its native and non-native surfaces. #636's relocation is a prerequisite for
  #630, not an optional cleanup.
- **Fence the blocks without relocating the assertions.** Rejected outright — this is exactly what
  #627 tried and had to descope; it reds the route-reachability test and all four contract
  validators on the very next run.
- **Relocate only the assertions that an initial (narrower) pass identified, skipping the
  `#611`-fork shrink in one of the four validators.** Rejected after the shaping run's adversary
  demonstrated it reds all four chains; this decision requires the shrink to land identically in
  all four validators plus the byte mirror, with no exception.
