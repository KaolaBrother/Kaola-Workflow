# D-519-01. Replace process-exit-code with stderr-error-class as the `gh`-fetch error partition axis

**Date:** 2026-06-17
**Status:** Accepted
**Issue:** #519 (axis fix, foundational); #510 (forge `_st`-guard parity); #511 (forge test-pin)
**Related:** #507 (boundary-2 classifier hardening — the bounded-retry layer this fix extends),
            #495 (boundary-1 `claim.js → classifier` subprocess hardening — the escalate envelope),
            `scripts/kaola-workflow-claim.js` (`classifySubprocessError`, `probeIssueState`),
            `scripts/classifier.js` ×4 editions,
            `scripts/active-folders.js` (`getRepoOwnerName`),
            `docs/decisions/D-495-01.md` (boundary-1/boundary-2 taxonomy),
            `docs/api.md` § Classifier verdict envelope.

## Context

The #507 boundary-2 hardening introduced bounded retry for transient classifier-subprocess
faults. It classified `spawn_fault` / `killed` / `clean_nonzero` but treated `clean_nonzero` as
a determinate refusal — mapping every non-zero exit to `target_unavailable` / `result: refuse`.

The premise was wrong. Process exit code is not a reliable discriminant between a transient
infra fault and a genuine negative:

- A TLS handshake timeout, a DNS "could not resolve host" failure, a rate-limit 429, an
  ETIMEDOUT, or a connection reset all produce a non-zero exit from `gh`/`glab`/`tea` — the
  same exit code a genuine 404 or a confirmed-closed issue produces.
- Classifying them identically collapses a transient blip into a hard `result: refuse`, aborting
  a claim on a genuinely-claimable issue. This was observed live: a kaolaGIT session where a
  transient TLS fault during the classifier `gh issue view` call produced `target_unavailable`
  and aborted the claim, when the issue was open and claimable.

The correct discriminant is the **stderr error-class** emitted by the CLI. The CLI writes
recognizable patterns for transient infra failures, and a genuine 404 / "Could not resolve to
an Issue" / closed response produces a distinct, recognizable pattern as well.

The same misclassification existed at four sites:

1. `getRepoOwnerName` / `gh repo view` (`active-folders.js`)
2. Classifier `gh issue view` (`classifier.js` ×4 editions)
3. `claim.js classifySubprocessError` (the boundary-1/boundary-2 seam)
4. `probeIssueState` (the `sink-merge` / `watch-pr` state probe reused at claim time)

Plus the claim-comment detection site.

## Decision

### Axis replacement: partition by stderr error-class, not by process exit code

**Transient-infra class** — any of:
- TLS handshake timeout
- Rate-limit (429 / "API rate limit exceeded")
- DNS "could not resolve host"
- Connection reset / ECONNRESET
- ETIMEDOUT
- 5xx server error

Routes to: `target_(set_)indeterminate` / `result: escalate` (the existing #495 escalate valve).

**Genuine-negative class** — any of:
- 404 / HTTP 404
- "Could not resolve to an Issue" (the gh CLI's standard pattern for a non-existent issue)
- Confirmed-closed issue (state check returns closed)

Routes to: `target_(set_)unavailable` / `result: refuse` (unchanged, fail-closed).

**Unrecognized clean-nonzero** — defaults to **refuse**. Escalation requires positive infra
evidence (a recognized transient pattern). An unknown stderr that could be either a transient
fault or a genuine negative is treated as determinate to avoid escalating on a genuine negative.
This is a conscious fail-closed call: the deny-list can be widened additively as new patterns
are observed (see Known limitations / F1).

### Non-breaking `probeIssueState` discriminant

`probeIssueState` is used by `closure-audit.js` and probe-memo readers in addition to the
claim-acquisition callers. Changing its return shape would break those consumers.

The fix is additive:

- `.state` is **unchanged**: `'unavailable'` for a genuine/unknown negative; `'open'`/`'closed'`
  for a resolved state. No consumer needs to change.
- A new `transient: true` field is **added only** when the failure is classified as transient.
  Only the claim-acquisition callers (which already route on the `indeterminate` verdict) inspect
  this field and route to `escalate`. `closure-audit.js` and probe-memo readers ignore unknown
  fields and are unaffected.

This preserves the boundary described in D-495-01 (escalate values not facts; the claim-time
front-door consent analog) while making the transient path reachable without a breaking change.

### Applied sites

All four sites are updated uniformly across all four editions:

1. `getRepoOwnerName` / `gh repo view` — `active-folders.js`
2. Classifier `gh issue view` — `classifier.js` (canonical + Codex byte-twin + gitlab + gitea ports)
3. `classifySubprocessError` — `claim.js` ×4 editions
4. `probeIssueState` — `claim.js` ×4 editions
5. Claim-comment detection site — `claim.js` ×4 editions

### Forge `_st`-guard parity (#510)

The forge classifiers' `_st` guard already returned `indeterminate` on exit-non-zero. However,
when `parseJson` silently swallowed a malformed or empty exit-0 body (returning `null`/`undefined`),
the guard fell through to a determinate `target_unavailable` verdict — diverging from root's
behavior where a `JSON.parse(raw)` `SyntaxError` inside the fetch `try` block is correctly
classified as transient.

Fix: when the forge classifier receives an exit-0 response whose body `parseJson` cannot parse,
the `_st` guard now returns `indeterminate` instead of a determinate verdict. Root needed no
change (its `JSON.parse` is already in the transient-routed `try` block).

### Forge determinate-refuse test-pin (#511)

Forge end-to-end claim-flow tests that previously used a generic "gh exits 1" stub to cover
the refuse path no longer unambiguously pin it after the axis fix: an unrecognized `clean_nonzero`
still defaults to refuse, but the stub could be misread as covering the transient path.

Fix: the gitlab and gitea claim-flow tests now stub a GENUINE-negative stderr pattern — the real
"Could not resolve to an Issue" or 404 text — so the `clean_nonzero` → `result: refuse` route
is pinned against regression. Tests in `scripts/test-kaola-workflow-gitlab-claim.js` and
`scripts/test-kaola-workflow-gitea-claim.js`.

## Implementation

`scripts/kaola-workflow-claim.js` ×4 editions: `classifySubprocessError` gains a
`classifyStderr(stderr)` helper that pattern-matches against the transient and genuine-negative
classes; `probeIssueState` gains the additive `transient: true` discriminant.

`scripts/classifier.js` ×4 editions: the inner fetch `catch` now calls `classifyStderr` instead
of collapsing all failures to `target_unavailable`.

`scripts/active-folders.js`: `getRepoOwnerName`'s error handler routes on the classified
stderr class.

`scripts/test-kaola-workflow-gitlab-claim.js` + `scripts/test-kaola-workflow-gitea-claim.js`:
stub payloads updated to genuine-negative stderr patterns.

All four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains green.

## Consequences

- A transient infra fault during classifier / claim-time fetch now escalates instead of
  aborting the claim. The operator is asked to retry rather than receiving a silent hard refuse.
- A genuine negative (404, closed, "Could not resolve to an Issue") continues to refuse
  determinately — the fail-closed guarantee is preserved.
- `probeIssueState` callers that do not inspect `transient` are unaffected: the `.state`
  field is byte-identical for all existing return paths.
- The axis fix applies uniformly across all four sites and all four editions; there are no
  edition-specific divergences for the core classification logic.

## Known limitations / conscious calls

### F1 — Unrecognized transient patterns default to refuse

An unrecognized transient stderr (e.g. `context deadline exceeded`, `unexpected EOF`,
`broken pipe`) currently produces a `clean_nonzero` that defaults to refuse. This is a
deliberate fail-closed choice: absent a recognized infra-failure signature, the error could be
a genuine negative, and escalating on an unknown pattern risks an unresolvable loop.

The deny-list (the set of recognized transient patterns) can be widened additively: adding a
new pattern entry requires only a positive identification of the stderr string and a test pin;
no caller or schema change is needed. This is a deferral, not a design gap.

### F2 — Stale test comment in the gitlab #510 test

The gitlab-edition `test-kaola-workflow-gitlab-claim.js` test file contains a comment
referencing the prior generic-stub behavior (`// exits 1 → refuse`) that was not updated when
the stub was replaced with the genuine-negative pattern. The comment is cosmetically stale;
the assertion itself is correct. A follow-up cleanup is noted here but not tracked as a
numbered issue (cosmetic).

### F3 — Forge `probeIssueState` malformed-NON-empty exit-0 body stays `unverified` → refuse

The forge `probeIssueState` implementation (gitlab/gitea) handles a malformed non-empty exit-0
body by returning `{state: 'unverified'}`, which the caller routes to refuse. Root's equivalent
path escalates (the `JSON.parse` `SyntaxError` is transient-routed). This is a narrow #307
cross-edition divergence.

It is NOT fixed here because:
1. A malformed non-empty exit-0 body from the forge CLI is not a real forge-CLI failure mode
   (the CLI either returns valid JSON or exits non-zero on error).
2. Fixing it would reintroduce a test stub-seam breakage in the forge classifier tests (the
   stub mechanism cannot reliably distinguish malformed-non-empty from malformed-empty without
   a shim).
3. Both editions fail-closed (neither wrongly acquires a claim on a genuine infra fault) — the
   correctness guarantee holds even with the divergence.

This is recorded as a conscious decision, not a defect. A future cleanup can unify the paths
if the forge CLI behavior changes.
