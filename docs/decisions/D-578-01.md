# D-578-01. Enforce set-equality between `templates/opencode/plugins/*.js` and `PLUGIN_SCRIPTS` in `sync-opencode-edition.js --check`

Date: 2026-06-30
Status: Accepted
Issue: #578
Related: D-577-01 (tracked canonical source for the hook-adapter plugin â€” this hardens the gap
             that review raised at that decision's close),
         D-530-02 (opencode stays additive â€” runtime-not-forge; no #307 four-chain obligation)

## Context

D-577-01 gave the opencode hook-adapter plugin a tracked canonical source at
`templates/opencode/plugins/kaola-workflow-hooks.js` and threaded it through the sync/check/test
pipeline. The code reviewer flagged a low-severity follow-up (R2, filed as #578): two consumers of
`templates/opencode/plugins/` diverge in how they enumerate plugins, and that divergence is latent
but not guarded:

- **`install-opencode.sh`** deploys `templates/opencode/plugins/*.js` via a **glob** â€” any new
  `.js` file added to that directory is picked up automatically.
- **`scripts/sync-opencode-edition.js`** regenerates `.opencode/plugins/` from an **explicit
  `PLUGIN_SCRIPTS` allowlist** (currently `['kaola-workflow-hooks.js']`).

With a single plugin these are equivalent. The moment a second `.js` file is added to
`templates/opencode/plugins/` without updating `PLUGIN_SCRIPTS`, the installer deploys it (glob)
while `sync-opencode-edition.js --check` and `test-opencode-edition.js` remain silent â€” the
on-disk plugin and the generated plugin differ, but no assertion catches it. There was no guard
covering the reverse direction (an on-disk file absent from the allowlist).

## Decision

Add a **set-equality guard** to `sync-opencode-edition.js` `runCheck()`. After the existing
per-script parity loop (which catches a registered-but-missing plugin), scan
`templates/opencode/plugins/` for all `*.js` files and assert each is present in `PLUGIN_SCRIPTS`.
Any unregistered file produces a mismatch entry and exits `--check` non-zero:

```
unregistered plugin '<file>' present in templates/opencode/plugins/ but absent from PLUGIN_SCRIPTS â€” add it to the allowlist
```

`writePlugin()` (`--write`) is unchanged â€” it already copies only `PLUGIN_SCRIPTS` members, which
is the correct behavior; the guard enforces that `PLUGIN_SCRIPTS` is complete with respect to what
is on disk.

A RED-first `A11-allowlist` assertion in `test-opencode-edition.js` first asserts the clean-state
set-equality holds (the real tree, with only `kaola-workflow-hooks.js`), then injects a synthetic
unregistered file into `templates/opencode/plugins/`, runs `--check`, and asserts a non-zero exit
naming the offending file and `PLUGIN_SCRIPTS`; the probe is removed in a `finally` block so the tree
is restored even if an assertion throws â€” proving the guard fires and cannot regress silently.

## Scope

This is an opencode-edition-only change (additive, builds on D-577-01 and remains not wired into
`npm test`, `edition-sync.js`, `install.sh`, or the SIX routing surfaces per D-530-02). No #307
four-chain obligation applies. Validation runs via `node scripts/test-opencode-edition.js`.

## Consequences

- Adding a second plugin `.js` to `templates/opencode/plugins/` without registering it in
  `PLUGIN_SCRIPTS` now fails `sync-opencode-edition.js --check` (and thus
  `test-opencode-edition.js`) loudly with an explicit message. There is no path to silent drift.
- The installer glob and the sync allowlist are provably equivalent at all times:
  `templates/opencode/plugins/*.js` â‰ˇ `PLUGIN_SCRIPTS` is machine-enforced on every `--check` run.
- `writePlugin()` (`--write`) behavior is unchanged â€” it still copies only the `PLUGIN_SCRIPTS`
  members. The guard changes what fails, not what gets deployed.
- `A11-allowlist` is RED-first: if the guard is ever removed or regressed, the test itself fails.

## Alternatives considered

- **Align the installer to an explicit allowlist** â€” remove the `*.js` glob from
  `install-opencode.sh` and enumerate plugins explicitly (matching `PLUGIN_SCRIPTS`). Rejected: the
  glob is the simpler, safer deploy surface and is correct as-is (it naturally picks up every plugin
  without a second maintenance point). The inconsistency is a parity-checking gap, not an install
  defect. Adding the guard to `runCheck()` is cheaper and more targeted than changing the installer.
- **Leave it as-is** â€” with one plugin the two consumers are currently equivalent. Rejected: the gap
  is only latent with one plugin; once a second exists the silent-drift risk is real. Closing it at
  one-plugin time is lowest cost.
