# D-424-01. Barrier attribution model: narrow .md allowband, drop-base window-lock, finalize attribution sweep, root-pinning, provenance log

## Context

`kaola-workflow-plan-validator.js`'s `--barrier-check` measures the actual writes a node produced
(via the recorded baseline diff) against that node's declared write set. Any file written but not
declared is a `write_set_overflow`. To avoid forcing every node to declare incidental documentation
writes, the barrier carried a **blanket `.md` exemption**: ANY path ending in `.md` passed the
write-set check without being declared.

This blanket exemption is a hole. In a live run, `agents/workflow-planner.md` — a *production* agent
profile, not documentation — was modified during a node that did not declare it, and the barrier
passed silently. Because `agents/*.md` is production behavior (the planner profile drives the
adaptive front-end), an undeclared, unattributed change to it is exactly the kind of drift the
barrier exists to catch. The `.md` suffix is a poor proxy for "barrier-invisible": the genuinely
invisible files are a small, well-known set of documentation and workflow-state paths, while the
codebase ships large amounts of *behavioral* `.md`/`.toml` (agent profiles, command bodies, SKILL
packs, codex agent manifests) that MUST be attributed.

Four further attribution gaps compound the blanket-exemption hole:

- **`--drop-base` laundering.** `--drop-base` clears a node's recorded baseline, which zeroes the
  measured diff. Invoked mid-node, it launders any write made so far — the next `--barrier-check`
  sees an empty diff and passes vacuously. There is no window restriction: the flag is honored at
  any ledger status.
- **Finalize blind spot.** The per-node barrier only ever sees one node's writes against one node's
  set. A file changed on the branch but covered by NO `complete` node's write set (e.g. left behind
  by a crashed node, or written outside any node window) is never re-checked at finalization.
- **Root ambiguity.** Write-set paths resolve relative to the git repo root, but the barrier reads
  `process.cwd()`-relative state. In a worktree whose CWD is not the repo toplevel, the two can
  diverge, so a path could be measured against the wrong root and silently mis-attributed.
- **No provenance.** Baseline mutations (`record-base`, `drop-base`) and node transitions leave no
  durable audit trail, so a post-hoc "who dropped the baseline and when" investigation has nothing
  to read.

## Decision

The barrier moves from a *suffix-based* exemption to an **explicit attribution model** with five
coupled parts. The unifying rule: a write is barrier-invisible ONLY if its path is in a precise
allowband; everything else MUST be attributed to a node's declared write set or flagged.

### 1. Narrow `.md` allowband (replaces the blanket exemption)

A path is barrier-invisible without declaration if and only if it matches one of:

- `docs/**` — documentation tree.
- `CHANGELOG.md` — release notes (repo-root only).
- `README.md` — project readme (repo-root only).
- `kaola-workflow/{project}/**` — workflow state and `.cache/` evidence for the active project.

Matching is path-shape, not suffix: the allowband is a set of glob predicates evaluated against the
repo-root-relative path. `CHANGELOG.md` and `README.md` match only at the repo root (a nested
`README.md` under `plugins/` is NOT in the band). The `kaola-workflow/{project}/**` clause preserves
the existing `.cache/{node-id}.md` evidence carve-out (issue #271) — evidence files are written by
role agents without declaration, and they live under the project state tree, so the band covers them.

Every other `.md`/`.toml`/`.txt` path — including `agents/*.md`, `commands/*.md`,
`plugins/*/skills/**/*.md`, `plugins/*/agents/*.toml`, and any nested non-root `README.md`/`*.md` —
is OUTSIDE the band and MUST appear in the writing node's declared write set, or the barrier refuses
with the existing `write_set_overflow` reason. The `agents/workflow-planner.md` live escape is closed
because `agents/**` is not in the band.

The allowband predicate is exported as a named helper (e.g. `isBarrierInvisible(repoRelPath)`) so the
finalize sweep (part 3) and any future caller share one source of truth — no second copy of the band.

### 2. `--drop-base` window-lock (`drop_base_window_open` typed refusal)

`--drop-base` is honored ONLY when the target node's ledger status is `pending` (before the node
opens). The pre-open window is the only legitimate use: clearing a stale baseline from a prior
attempt before a fresh open.

Once a node is `in_progress`, `--drop-base` is refused with the typed reason `drop_base_window_open`.
This removes the mid-node laundering vector: a baseline can no longer be cleared after writes have
begun. The refusal is part of the barrier emit envelope (precedence-ordered with the other typed
reasons) so callers classify it structurally, never by string-match.

### 3. Finalize-time attribution sweep (`unattributed_change`)

At finalization, after the per-node barriers have all passed, a whole-branch sweep runs:

1. Enumerate every file changed since the branch diverged from main:
   `git diff main...HEAD --name-only`.
2. For each changed path, it is **attributed** if EITHER:
   - it is in the `.md` allowband (`isBarrierInvisible`), OR
   - it is covered by the declared write set of some node whose ledger status is `complete`.
3. Any path that is neither is collected and reported as `unattributed_change`, naming the offending
   paths. This is a finalize-blocking refusal.

This closes the gap where a node completes but the branch carries writes that no completed node owns
(crash residue, out-of-window edits). The sweep reuses the shared allowband helper and the same
write-set glob-matching the per-node barrier uses, so attribution is consistent end to end.

### 4. Root-pinning (`root_mismatch` typed refusal)

At `--barrier-check` time, the validator verifies `process.cwd()` equals
`git rev-parse --show-toplevel`. If they differ, it refuses with `root_mismatch` rather than
resolving write-set paths against a mismatched root. This guarantees the baseline diff and the
declared write set are measured against the same root, eliminating the worktree CWD-vs-toplevel
mis-attribution. The check fails closed: a non-zero or empty `rev-parse` result is treated as a
mismatch. `root_mismatch` joins the barrier emit envelope.

### 5. Lifecycle provenance log (`.cache/provenance-log.jsonl`)

When `adaptive-node.js` runs any of `record-base`, `drop-base`, `open-next`, or
`close-and-open-next`, it appends ONE structured line to `.cache/provenance-log.jsonl` (project-local
`.cache/`, the same root where the barrier resolves evidence):

```json
{"timestamp":"<ISO>","event":"record-base|drop-base|open-next|close-and-open-next","nodeId":"<id>","nonce":"<node nonce>","by":"adaptive-node"}
```

The log is append-only JSONL (one event per line), so a crash mid-write loses at most the trailing
line and never corrupts prior entries. It enables a post-hoc audit of who dropped or re-recorded a
baseline and when — the missing forensic trail behind the live escape. The provenance log lives under
the project `.cache/`, so it is inside the allowband (part 1) and never itself trips the barrier.

## Consequences

**For plan authors:**

- Behavioral `.md`/`.toml` (agent profiles, command bodies, SKILL packs, codex agent manifests) MUST
  now be declared in the writing node's write set. A node that edits `agents/*.md` or
  `plugins/*/skills/**/*.md` without declaring it gets `write_set_overflow` — the same shape as any
  other production-file overflow. This is the intended tightening.
- `docs/**`, repo-root `CHANGELOG.md`/`README.md`, and `kaola-workflow/{project}/**` remain
  declaration-free.

**For the orchestrator / adaptive-node:**

- `--drop-base` is only valid pre-open. A laundering attempt mid-node returns `drop_base_window_open`;
  the correct recovery for a stale baseline is the documented ledger-reset → `pending` → drop → fresh
  open sequence, which keeps the drop inside the legal window.
- `record-base`/`drop-base`/`open-next`/`close-and-open-next` now emit provenance lines. Callers must
  tolerate the additional `.cache/provenance-log.jsonl` artifact (it is allowband-covered and is kept
  out of the sink commit alongside the rest of repo-root/project `.cache/`).

**For finalize:**

- `cmdFinalize` runs the attribution sweep before closing. An `unattributed_change` blocks closure and
  names the paths, so crash residue or out-of-window edits surface at the gate instead of shipping.

**For the barrier emit envelope (`docs/api.md`):**

- Three new typed reasons — `drop_base_window_open`, `root_mismatch`, `unattributed_change` — are
  documented in precedence order alongside the existing `write_set_overflow` / `unattributed_write`
  family. Callers continue to classify structurally on `reason`, never by string-match.

**For cross-edition parity:**

- The allowband helper, the window-lock, the root-pin, and the sweep live in the generated
  `plan-validator.js` aggregator and its three edition ports; the provenance emission lives in the
  `adaptive-node.js` aggregator and its three ports. Both are `GENERATED_AGGREGATORS` members, so the
  four-file group is owned together (D-431-01 aggregator-coupling rule) and all four chains must be
  green before finalize.
