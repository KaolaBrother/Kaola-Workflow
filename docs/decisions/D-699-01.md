# D-699-01 — Frozen adaptive plans advance through planner-owned, claim-preserving epochs

- **Status:** accepted contract; implementation certification pending
- **Date:** 2026-07-16
- **Context:** issue #699; depends on the durable review journal, typed gate mode, and inherited-frontier/G4 contracts

## Context

The adaptive repair transaction can prove and replay a direct repair only when the frozen DAG has
one admissible writer. Its conservative `repair_requires_replan` refusal is correct, but the former
recovery choices were not: editing/re-freezing the frozen plan breaks planner authority, while
discarding/restarting can lose the live claim shape, reset plan-keyed budgets, obscure prior evidence,
or turn already-present unapproved candidate changes into a new baseline.

The system needs a genuinely different planner-authored DAG without changing the issue claim,
feature branch, worktree, product candidate, claim-root authority, inherited review obligations, or
bounded liveness history. It must also recover from a crash after any durable write and preserve the
complete parent proof for final archive verification.

## Decision

### 1. Authority is claim-scoped; the frozen parent is immutable

Fresh claims persist a canonical claim identity and immutable claim-root base. Their digests derive
`epoch_lineage_id`; the derivation deliberately excludes plan hashes, gate names, node ids, candidate
digests, and epoch ordinals. A replacement DAG therefore cannot reset claim-level accounting.

From transaction prepare through parent archive, the current `workflow-plan.md`, its Node Ledger, and
its parent-state authority remain byte-immutable and current. The issue claim/label, feature branch,
worktree, and product candidate persist in place. They are never moved into an epoch snapshot or
discarded to simplify accounting.

Only `workflow-planner` may author the semantic child DAG. The main orchestrator supplies repository
and project identity, typed transition reason, source evidence, authority bindings, and the exact
seeded `workflow-plan.next.md` path. It may not prescribe nodes, roles, dependencies, write sets,
cardinality, shape, model, or build order. Dispatch-log and planner-attestation evidence bind the
packet, transaction, profile, worktree, nonce, and child digest before the re-plan handoff can freeze
the child.

Fresh schema-2 claims recognize exactly two epoch-1 active-authority forms. A `planless` claim has no
plan or snapshots and carries `none` in both plan-hash fields and both first-node fields. A `planned`
claim has one frozen plan whose exact/stored hash equals both state hashes and whose first node equals
the state tuple. Initial handoff publishes the complete planned tuple in one state replacement; child
activation does the same and verifies the task mirror. Partial or mixed tuples fail closed.

Verifying that active tuple is not a byte comparison against a staged copy. One shared
`verifyCurrentEpochAuthority` function — reused by prepare, resume, archive, finalize, and watch —
separates the immutable authored surface (`Meta`/`Nodes`/`Node Briefs`, hash-verified) from runtime
surfaces that legally progress after commitment (`Node Ledger`, `Required Agent Compliance`, and the
task mirror, each parsed and consistency-checked). This closes the self-host defect where a legally
advanced Ledger or Compliance table was mistaken for authoring tamper. The claim-scoped
`epoch_schema_version`/`epoch_lineage_id` envelope is a third, separately verified tier: fully absent
reads as pre-epoch legacy state, while a partial or recomputed-mismatched envelope refuses.

Offline mode overrides either native-worktree setting: it creates neither a worktree nor an in-place
feature branch. A repository with history binds exact commit/tree objects. An initialized repository
without history binds the object-width all-zero commit plus its locally recomputed canonical
empty-tree object. Arbitrary synthetic roots and missing Git authority refuse; target-branch identity
remains a separate claim-identity field.

### 2. Review-driven source authority is mechanical

The real direct-repair transaction writes `.cache/replan-source.json` before it returns
`repair_requires_replan`. The schema-2 `repair_outcome` envelope binds the typed result, attempt,
reason, sorted producer slice, parent plan/claim/root/lineage, review journal/attempt, evidence, and
effective candidate under a canonical `outcome_digest`. Exact retry is byte-stable and preserves the
original timestamp; a competing semantic outcome for the same attempt refuses
`replan_source_conflict`.

`prepare` re-verifies the outcome against the still-failed, lifecycle-settled, unconsumed review
attempt. A hand-written source file, stale journal, substituted evidence, changed candidate, or
schema-1 file cannot start a new schema-2 transaction. A crash after source persistence may prevent
the first CLI response, but it leaves the authoritative outcome ready for retry.

### 3. Re-plan is one resumable transaction

`.cache/replan-transaction.json` is the sole transition authority under the existing project
`scheduler.lock`. Its phases are:

```text
prepared → planner_pending → child_frozen → parent_archived → committed
```

All normal scheduler, node mutation, handoff, mirror refresh, archive, and Finalization paths are
fenced while the transaction is incomplete. `orient` is read-only and reports the phase and hashes.
`kaola-workflow-replan.js resume --project <project>` is the only legal mutation path.

Activation is not represented as one filesystem-atomic operation. It is an ordered, digest-bound
multi-file journal:

```text
child_plan_promoted
→ child_state_promoted_fenced
→ task_mirror_promoted
→ active_cache_cleaned
→ transaction_committed
→ state_unfenced
```

Before promotion the frozen parent is still authority. After promotion begins, recovery never rolls
back to the parent; it verifies every completed prefix and rolls forward under the fence. A repeated
resume converges to one child epoch and an already-committed result.

### 4. Four CAS seams bind the live candidate

The engine recomputes and compares the same tuple at prepare, immediately before child freeze,
before parent snapshot, and immediately before activation:

```text
(candidate_digest, claim_root_base_digest, inherited_frontier_digest)
```

A pre-promotion mismatch records `replan_candidate_changed` with the expected and observed tuple,
keeps the parent authoritative and fenced, invalidates the stale child attempt, and requires fresh
planner authoring. It advances neither `plan_epoch` nor `automatic_review_replans`. The child is
never patched onto a changed candidate.

The conformance matrix independently mutates candidate, claim-root, and inherited-frontier values at
all four seams (12 cells). Each cell must preserve parent plan/state/task authority and have no
epoch, budget, dispatch, snapshot, Case-B, or activation effect beyond the typed mismatch receipt.

Crash replay is backed by one exact ordered inventory of 41 durable-write label families. Dynamic
snapshot-file, cleanup-intent, and cache-unlink labels add sorted ordinal/path digests; dynamic
candidate-change transaction/state labels add the CAS seam. The implementation fires immediately
after the corresponding durable mutation. Tests lock the exact inventory and dynamic grammar,
execute every discovered main-path crash prefix, and require one-resume convergence plus a
byte/cardinality-stable second resume. The separately owned source-outcome prefix is
`after_replan_source_outcome` and is not counted among the 41 re-plan families.

### 5. Inherited work remains subject to G4 and final-digest certification

The candidate and its code/security frontier are derived from the persisted claim root, not trusted
from parent or child prose. The child binds the parent, lineage, claim root, frontier, transition
source, planner dispatch, and designated code/security certifiers. A nonempty inherited frontier
creates the existing validator-owned virtual producer obligation. Real role-specific certifier gates
must remain reachable and their receipts bind the full relevant final digest.

Consequently, a child with no new writer cannot relabel inherited unapproved code or sensitive work
as harmless baseline, and later relevant mutation stales prior certification.

### 6. Projection-bound children and full snapshots form a two-way seal

Before planner dispatch, the transaction computes a canonical `snapshot_authority_projection` from
stable parent plan/task/ledger/state authority, typed source digests, and entry CAS. The planner
packet carries the projection and digest. Despite its historical name, a schema-2 child's
`parent_snapshot_manifest_digest` must equal the projection digest; it does not claim to know the
later full manifest-file digest.

Before activation, the transaction creates and re-verifies
`.cache/epochs/<parent-plan-epoch>/manifest.json` and `files/**`. The schema-2 manifest embeds and
re-derives the projection, then separately seals the exact child and planner attestation, every
sorted path/mode/size/digest row, its `manifest_self_digest`, and its exact manifest bytes. The
snapshot also covers the exact parent plan/Ledger/state, claim/root and entry/exit CAS views,
candidate/frontier, authoritative review journal, findings, every node evidence file, contexts,
receipts, certifiers, validation vectors, task mirror, dispatch provenance, and every attempt's
complete `rebind` ledger (including an explicit empty ledger).

Only manifest-listed, digest-unchanged epoch-local active caches may be removed after plan/state/mirror
promotion. Epoch directories, the transaction, consent ledger, claim/lineage anchors, and dispatch
log are never cleanup targets. Closure recursively verifies and preserves every epoch snapshot and
the complete lineage receipt in the final archive.

### 7. Automatic liveness is bounded at claim scope

`REVIEW_REPLAN_LIMIT = 2`: two automatic review-driven replacement transitions may commit for one
epoch lineage. Failed dispatch, child validation, CAS, snapshot, or crash retry costs zero. A further
automatic transition records a durable consent halt before planner dispatch.

One explicit user action may append one hash-chained consent entry and increase the verified ceiling
by exactly one. The entry binds its prior/new ceiling, user-turn reference, reason, timestamp, and
previous digest. An agent cannot self-clear the halt, reuse the same user-turn reference for more
capacity, or create authority by editing the cached state value.

A diagnosis-to-build transition is a separate no-review source, not a label that can override a
failed review. Neither `review-attempts.json` nor `replan-source.json` may exist. The completed
schema-2 parent binds exact `diagnosis_root_cause`, `falsified_alternatives`,
`acceptance_contract`, and `recommendation` artifacts; all must be regular, contained,
single-linked terminal `diagnosis_complete` records, and every writer must be limited to those paths.
The child cites the exact proof and recommendation digests. Its first verified use costs zero and
consumes the one-shot exemption; repeated, untyped, writer-bearing, citation-missing, or
review-driven variants cost one or refuse.

### 8. Legacy v1 is explicit and fail-closed

A verified frozen v1 parent may complete under its existing v1 recovery contract. It is not silently
rewritten with schema-2 fields. Its only path to a v2 child proves a corroborated immutable legacy
claim root, snapshots the exact v1 plan/state/journal/evidence/receipt/rebind authority, derives the
candidate/frontier against that root, and dispatches a fresh planner. Ambiguous or missing root proof
refuses `legacy_claim_root_unprovable`. A newly authored child without schema 2 is invalid and cannot
claim legacy treatment.

A historical committed schema-1 child may retain `parent_snapshot_manifest_digest: pending` only
when the immutable manifest self/exact digests, copied child row, committed transaction child,
planner attestation, promoted plan, and descendant state all agree. That read-only compatibility
result is `legacy_external_binding`; any missing seal refuses `legacy_snapshot_binding_unsealed`.
New schema-2 work never receives this exception, and the committed validator delegates to the same
recursive verifier instead of constructing a second authority model.

### 9. Archive and Planning Evidence callers fail closed

Initial handoff and child activation replace the complete Planning Evidence tuple—plan hash,
decision, risk, first-node id, and first-node role—along with the active hash/task/snapshot state.
The verifier refuses stale first-node, active-hash, and task-mirror combinations. It diagnoses a
historical mismatch without mutating frozen state.

All archive callers share `archiveSucceeded(result)`, true only for `archived:true` or idempotent
`skipped:"source-missing"`. Finalize, release, merged watch, and closed/abandoned watch stop before
roadmap, issue/label, worktree/branch, closure receipt, or success side effects on every other result.
Lineage preservation is derived only after that boundary and recursive destination verification.

### 10. Completion is owned locally

Hosted CI/CD is not a gate for the transaction or Finalization. Candidate-bound local transaction
tests, all relevant edition contract validators and packaged walkthroughs, role reviews, and frozen
falsification nodes own the verdict. An external pipeline may exist, but it cannot substitute for or
waive those gates.

## Verification status at acceptance

This record accepts the machine contract; it does not certify the current implementation as
terminal-green. The versioned-authority engine this decision governs — transaction rotation, the
four CAS seams, snapshot/manifest sealing, and the current-epoch authority split (Decision 1) — is
implemented, with its focused re-plan suite, resume, edition/script sync, and forge forbidden-token
checks passing.

Integrated PASS remains withheld pending two separate, still-in-progress write surfaces this record
does not itself certify:

- lifecycle/publication callers (archive, finalize, release, and watch cleanup composing
  `verifyArchiveEpochAuthority`, plus the walkthrough's nested subprocess-timeout handling); and
- packaged-edition fixture parity (Codex, GitLab, and Gitea snapshot-authority fixtures).

Terminal certification additionally requires the named code and security certifiers and the three
read-only falsification nodes (budget exhaustion, history/archive integrity, and publication/edition
parity) to each record a pass. Treat the project's Node Ledger and per-node gate evidence as the
current source of truth rather than a fixed list of blockers in this record, since both write
surfaces above are open work as of this correction.

## Consequences

- Replacement topology remains a planner judgment; scripts own identity, atomic writes, fences,
  compare-and-swap, snapshots, cleanup, and bounded accounting.
- The live candidate and all prior evidence survive a DAG change without becoming a fresh baseline.
- Crash recovery has one mutation command and converges by durable prefix instead of rollback.
- Review budgets cannot reset with a new plan hash or gate name.
- Archive size grows with immutable epoch evidence; this is intentional provenance, not disposable
  cache.
- A transition can stop on consent, candidate drift, missing planner attestation, legacy ambiguity,
  or incomplete snapshot rather than guessing a way forward.

## Alternatives considered

- **Edit and re-freeze the parent in place.** Rejected because it changes the evidence authority the
  current Ledger, review journal, and barriers were bound to, and lets main author topology.
- **Discard/restart the adaptive run.** Rejected because it can lose claim shape and evidence,
  reset plan-keyed accounting, and launder inherited candidate changes into a new baseline.
- **Let the harness choose the replacement DAG or repair owner.** Rejected because semantic
  ownership/topology is agent reasoning; scripts prove admissibility and durability.
- **Treat the activation as one atomic filesystem operation.** Rejected because plan, state, task
  mirror, cleanup, and transaction journal are separate durable writes. Explicit roll-forward
  prefixes describe the real crash model.
- **Require hosted CI before commit.** Rejected because the workflow must own its verdict and work
  correctly when no hosted pipeline exists.
