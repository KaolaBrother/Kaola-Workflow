# D-432-01. Machine-verifiable chain receipts replace prose attestation of "all four chains green"

Date: 2026-06-13
Status: Accepted
Issue: #432
Related: `scripts/kaola-workflow-run-chains.js` (new SUPPORT script — the receipt writer),
         `scripts/kaola-workflow-plan-validator.js` (finalize/verdict path — receipt gate),
         `scripts/kaola-workflow-claim.js` (`cmdFinalize` — calls the receipt gate),
         `package.json` (`test:kaola-workflow:{claude,codex,gitlab,gitea}` chain scripts),
         `docs/api.md` § Finalization Contract (chain-receipt schema + typed refusals),
         D-424-01 (barrier attribution — the sibling finalize-trust fix),
         The "verify real exit code" lesson (gate on `$?` directly, never on a piped tail's exit)

## Context

The cross-edition validation policy requires all four chains
(`npm run test:kaola-workflow:{claude,codex,gitlab,gitea}`) green before finalization. In a live run
this was attested **in prose** — the operator wrote "all 4 chains green" into the run notes — over a
run where the claude chain was in fact RED. Finalization trusted the prose, not real exit codes, and
the red chain shipped.

The root cause is structural: there was no machine-checkable artifact tying a green claim to a
specific commit and to real subprocess exit codes. Two well-known failure modes feed the gap:

- **Prose drift.** A human (or an agent) can write "green" without having run the chains, or after
  running an older revision, or after a chain that actually failed.
- **Pipe-masked exit codes.** `npm test` chains the four with `&&`, so it short-circuits on the first
  failure; and any `cmd | tail` form reports the tail's exit (0), masking a red chain entirely. (See
  the "verify real exit code" lesson.) Even a well-intentioned operator can mis-read a masked exit.

The fix is to make "four chains green" a **machine-verifiable receipt** bound to the HEAD sha, written
by a script that captures real `spawnSync` exit codes, and gated at finalize.

## Decision

A new SUPPORT script writes a structured chain receipt; the finalize/verdict path refuses unless a
fresh, all-green (or explicitly-waived) receipt exists for the current HEAD.

### 1. Receipt schema (`.cache/chain-receipt.json`)

```json
{
  "headSha": "<git HEAD sha at the moment chains ran>",
  "workTreeHash": "<hash of `git status --porcelain` output>",
  "startedAt": "<ISO timestamp>",
  "completedAt": "<ISO timestamp>",
  "chains": [
    {
      "name": "claude",
      "exitCode": 0,
      "command": "npm run test:kaola-workflow:claude",
      "duration_ms": 12345,
      "accepted_red": false,
      "accepted_red_issue": null
    }
  ]
}
```

`headSha` binds the receipt to a commit. `workTreeHash` (a hash of `git status --porcelain`) records
the working-tree state so a receipt taken over a dirty tree is distinguishable from one over a clean
tree. Each `chains[]` entry records the real `exitCode`, the exact `command` run, the wall-clock
`duration_ms`, and the waiver fields (`accepted_red`, `accepted_red_issue`).

### 2. `kaola-workflow-run-chains.js` (the receipt writer)

A new Node.js SUPPORT script that:

- Runs each requested chain with `child_process.spawnSync` (NOT a shell pipe), capturing the **real**
  exit code per chain. Each chain runs to completion independently — there is no `&&` short-circuit,
  so a red claude chain does not hide a red codex chain.
- Accepts `--chains claude,codex,gitlab,gitea` (default: all four).
- Accepts `--accept-known-red <name>:<open-issue>` (repeatable; e.g. `--accept-known-red codex:234`)
  to waive a known-failing chain (see part 4).
- Writes `.cache/chain-receipt.json` with `headSha`, `workTreeHash`, timestamps, and the per-chain
  results.
- Exits 0 if every non-waived chain passed (`exitCode === 0`); exits non-zero otherwise. The
  receipt is written in BOTH cases (a red receipt is still a record).

The script captures `headSha` from `git rev-parse HEAD` and `workTreeHash` from a hash of
`git status --porcelain` at the moment the chains run, so the receipt's binding reflects the tree the
chains actually exercised.

### 3. Finalize-gate typed refusals (in the plan-validator)

The finalize/verdict path checks for the receipt and refuses, in precedence order:

- `chains_unverified` — no `.cache/chain-receipt.json` exists. The operator never ran the gated
  chains; prose alone can no longer pass.
- `chains_stale` — the receipt's `headSha` does not equal the current `git rev-parse HEAD`. The tree
  has advanced since the chains ran (a commit landed, a rebase happened); the receipt no longer
  describes HEAD and must be regenerated.
- `chains_red` — at least one chain has `exitCode !== 0` AND `accepted_red === false`. A real red
  chain that has not been explicitly waived blocks finalize.

These join the existing finalize/verdict emit envelope as typed reasons, classified structurally.

### 4. Waiver semantics (`--accept-known-red <name>:<open-issue>`)

A waiver for a chain `<name>` against a tracked `<open-issue>`:

- Sets `accepted_red: true` and `accepted_red_issue: "<open-issue>"` for that chain in the receipt.
- Is recorded in the receipt for audit (it is durable, not a runtime-only flag).
- Does NOT suppress the OTHER chains — every non-waived chain is still run and still gates. A waiver
  narrows the red-blocking to exactly the named chain.
- Requires a well-formed `<name>:<open-issue>` pair. `<name>` must be one of the known chain names;
  `<open-issue>` must be a non-empty issue token (e.g. `234`). Malformed input (missing colon, empty
  issue, unknown chain name) causes `run-chains.js` to refuse with a helpful message and a non-zero
  exit BEFORE running any chain — a typo cannot silently waive nothing or waive the wrong chain.

Because the waiver is per-chain and recorded, the finalize gate's `chains_red` check honors it: a
chain with `accepted_red: true` does not trip `chains_red`, but its waiver and the open-issue are
visible in the receipt for review.

### 5. Cross-edition parameterization

`run-chains.js` knows the per-edition chain names. For all three forges the canonical chain set is the
SAME: `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` (github/canonical, gitlab, and gitea
editions all run the same four `package.json` scripts). The script is a SUPPORT script:

- Canonical source: `scripts/kaola-workflow-run-chains.js`.
- Codex byte-twin: `plugins/kaola-workflow/scripts/kaola-workflow-run-chains.js` (byte-identical to
  canonical).
- GitLab port: `plugins/kaola-workflow-gitlab/scripts/kaola-gitlab-workflow-run-chains.js`
  (edition-named, functionally identical).
- Gitea port: `plugins/kaola-workflow-gitea/scripts/kaola-gitea-workflow-run-chains.js`
  (edition-named, functionally identical).

Because the chain set is identical across editions, the only per-edition difference is the file name;
the chain-name table and logic are shared. The script registers in `COMMON_SCRIPTS` and in all three
`install.sh` `SUPPORT_SCRIPT_NAMES` blocks (the standard SUPPORT-script registration surface).

## Consequences

**For finalize:**

- "All four chains green" can no longer be asserted in prose. `cmdFinalize` invokes the receipt gate;
  absent or stale or red receipts block with the typed reason. The live false-green this issue was
  filed over is structurally prevented.
- The normal pre-finalize flow gains one step: run `kaola-workflow-run-chains.js` after the LAST
  commit so `headSha` matches HEAD. Running chains before a subsequent commit yields `chains_stale`.

**For operators with a known-red chain:**

- A genuinely-known-failing chain (tracked by an open issue) is waived explicitly via
  `--accept-known-red <name>:<issue>`, which is durable and auditable, rather than by quietly
  omitting it. The other three chains still gate.

**For the finalize/verdict emit envelope (`docs/api.md`):**

- `chains_unverified`, `chains_stale`, `chains_red` are documented in precedence order. The
  chain-receipt schema is documented under the Finalization Contract.

**For the test suite and SUPPORT-script registration:**

- A walkthrough scenario exercises `run-chains.js` writing a green receipt and the finalize gate
  passing; a red-receipt scenario asserts `chains_red`; a stale-receipt scenario asserts
  `chains_stale`; a missing-receipt scenario asserts `chains_unverified`; a waiver scenario asserts
  `accepted_red` honors the named chain while still gating the others.
- The new SUPPORT script lands in all four trees and in the three `SUPPORT_SCRIPT_NAMES` blocks; the
  four contract validators' SUPPORT-script counts bump in lockstep. This is a cross-edition diff, so
  all four `npm run test:kaola-workflow:*` chains must be green before finalize — verified by a
  `run-chains.js` receipt, which is the dogfooding of this very feature.
