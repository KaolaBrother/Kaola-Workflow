This record covers **Part 2** (the enriched consent-halt triage payload) and **Part 4** (the
`kaola-workflow-release.js` aggregator) together, because both convert an error-prone, manually-
diagnosed step into a SINGLE typed transaction over data the codebase ALREADY computes: Part 2 threads
the `barrierCheck` diagnosis (which the orchestrator already had in hand) forward into the halt return;
Part 4 composes the release checks that today are scattered across `validate-workflow-contracts.js`
into one verify/cut envelope. Parts 1 (autopilot loop) and 3 (goal line) are recorded in D-420-01,
which numbers invariants [INV-1]..[INV-12]; this record continues at [INV-13].

## Context

Two pipeline steps today are diagnosed-then-discarded or scattered-and-manual:

1. **The halt payload is a bare marker, not a diagnosis (Surface B, investigation §2).** `runWriteHalt`
   (`kaola-workflow-adaptive-node.js:1454`) accepts exactly three reasons —
   `['consent', 'security', 'test_thrash']` (line 1457) — and writes two durable markers: the state
   `escalated_to_full: <reason>` line(s) (a `consent` halt writes BOTH `consent` and `security` in
   lockstep, lines 1470-1500) and the ledger `consent_halt: pending` line (plan-first, lines 1505-1526).
   But the ENTIRE return payload (lines 1544-1550) is `result: 'ok'`, `halt: 'written'`, the literal
   `markers` strings, a `taskTransitions` note, and a refreshed `taskMirror`. It tells the operator a
   halt was recorded and which markers were written, but NOT *why mechanically* or *what to do about it*
   (investigation §2.2). The diagnosis already exists upstream: a `barrier_failed` close
   (`runCloseAndOpenNext`, returning `reason: 'barrier_failed'` with the `barrierOut` envelope,
   `adaptive-node.js:1230-1237`) carries the offending paths and the typed `reason`. `barrierCheck`
   (`plan-validator.js:578-672`) already returns `outOfAllow` (line 623), `sensitiveHits` (line 618),
   `foreignArchiveHits` (line 612), `unattributed` (line 645), and the #406 typed `reason` envelope
   (lines 650-671) — including the #404 `write_set_granularity` subtype (`plan-validator.js:557-567`,
   `659-668`). The gap is purely that the halt does not carry this forward; the operator (or the
   autopilot in D-420-01 Part 1) re-diagnoses by hand.

2. **The release flow is a scattered manual phase (Surface D, investigation §4).** The checks exist —
   tag-before-test (`validate-workflow-contracts.js:556-568`), tag-ancestry orphan guard (lines 580-590
   via `tagAncestry`), the two manifest lockstep families (claude at lines 469-475, codex at lines
   478-496), the README version assertions (lines 464-467, 485), the Codex surface-drift check (lines
   592-599 via `detectCodexReleaseSurfaceDrift`), and the CHANGELOG version-heading PRESENCE check
   (line 552, #156). But they only run RED inside `npm test` AFTER the human has already stamped the
   version, hand-edited the CHANGELOG, and created the tag — and the only CHANGELOG check is
   presence-of-heading, NOT completeness. #417 found the v5.15.0 cut shipped CHANGELOG gaps (commit
   91d9e5e carrying #385/#388/#389/#390 had no entry; #401 Part 2 was missing), backfilled by hand
   (investigation §4.2). The publish step (`<forge> release create --latest`) is run by hand and CANNOT
   be scripted: the editions contract forbids naming a forge-specific binary in a shipped script (the
   same forge-neutral constraint that binds the Surface-B classifier).

The recurring finding (investigation §0) holds for both parts: every piece already exists as a
primitive. Part 2 is *payload enrichment* over data `barrierCheck` already emits; Part 4 is *one
composition* over checks `validate-workflow-contracts.js` already enforces. Neither requires new core
mechanics. Both convert an error-prone step into a typed transaction, mirroring how D-419-02 consolidated
existing primitives behind a typed envelope rather than inventing machinery.

## Decision — Part 2: Enriched Halt Payload

**Decision (P2):** `runWriteHalt` attaches a CLASSIFIED `triage` payload (a forge-neutral JSON object)
to its existing return, in ADDITION to — never instead of — the durable markers it already writes. The
triage object threads the offending-paths arrays from `barrierCheck`, the matched mechanical class (the
#406 typed `reason` generalized with three new mechanical subtypes), and a ready-made plan-repair diff
the orchestrator (or the autopilot, D-420-01 [INV-6]) can apply or show to the user. `test_thrash` gets
the parallel enrichment: the failing-test delta. The marker-writing transaction (state-then-ledger
crash ordering, the `consent`→`consent`+`security` coupling) is UNCHANGED — enrichment is purely
additive to the return shape.

**Part 2 invariants:**

- **[INV-13] (diagnosis is THREADED, never re-derived):** The triage payload carries the offending-paths
  arrays the validator ALREADY computed — `outOfAllow`, `sensitiveHits`, `foreignArchiveHits`,
  `unattributed` (`plan-validator.js:618`, `623`, `612`, `645`, surfaced at line 671). `runWriteHalt`
  does NOT re-run the barrier; it accepts these arrays as an input parameter (passed by the caller that
  already holds the `barrierOut` envelope from the `barrier_failed` close, `adaptive-node.js:1230-1237`).
  This preserves single-source-of-truth: `barrierCheck` is the only place offending paths are computed
  (mirroring D-419-02 [OQ-P2-b]'s "no second implementation" discipline). A halt raised WITHOUT a prior
  barrier diagnosis (e.g. a bare `consent` escalation with no overflow) carries an empty/absent
  offending-paths set, not a fabricated one.

- **[INV-14] (mechanical class is the #406 reason, GENERALIZED — not forked):** The triage `class` field
  is the validator's typed `reason` (`plan-validator.js:656-670`: `foreign_archive` >
  `sensitive_write_unreviewed` > `write_set_overflow` > `write_set_granularity` (#404 subtype) >
  `unattributed_write`), EXTENDED with three additional mechanical subtypes of `write_set_overflow` that
  recur in the run-lesson history: `lockfile_write` (an out-of-allow path is a dependency lockfile, e.g.
  `package-lock.json`), `mirror_write` (an out-of-allow path is a generated edition-sync / mirror
  target the operator forgot to declare), and `count_bump` (an out-of-allow path is a
  `validate-*-contracts.js` / `test-*.js` count-surface file — the "forgotten count-bump file" class the
  memory cites repeatedly, e.g. the #250/#340 count bumps). Like `write_set_granularity`, these three are
  STRUCTURAL subtypes detected by a pure literal-pattern test over `outOfAllow` (no fs, no mutation, no
  re-freeze), and they NARROW `write_set_overflow` — a path that matches none of them stays plain
  `write_set_overflow`. The precedence family is NOT forked: the new subtypes sit UNDER `write_set_overflow`
  in the same precedence-ordered envelope, exactly as #404 placed `write_set_granularity` under it
  (`plan-validator.js:659-668`). A halt never invents a class outside this closed taxonomy.

- **[INV-15] (the repair diff is a SUGGESTION, not an applied mutation):** For an overflow-family class
  (`write_set_overflow` / `write_set_granularity` / `lockfile_write` / `mirror_write` / `count_bump`),
  the triage payload attaches a ready-made plan-repair diff: the exact proposed edit to the offending
  node's `declared write set` cell — the offending paths added to the node's write set — following the
  EXISTING `--freeze` plan-repair pattern (a write-set swap + re-freeze, which preserves the ledger
  because `plan_hash` covers only `## Meta` + `## Nodes`, `plan-validator.js:682-687`; the `reopen-node`
  path at `adaptive-node.js:1632`+ already uses this). `runWriteHalt` COMPUTES and ATTACHES the diff; it
  does NOT apply it (the script never re-freezes from inside a halt). Applying the diff is the
  orchestrator's decision under D-420-01 [INV-6]'s repair-consent boundary — the #44 "agent owns
  reasoning" rule covers repair as well as selection. The diff collapses the operator's decision to a
  yes/no. For a `sensitive_write_unreviewed` or `foreign_archive` class, NO auto-repair diff is offered
  (these require a security-reviewer node or are a stray foreign-archive sweep — not a write-set widen);
  the payload surfaces the offending paths and the required structural fix in prose, not a one-click diff.

- **[INV-16] (`test_thrash` delta is symmetric enrichment, same return shape):** A `test_thrash` halt
  attaches a `testDelta` object — the failing-test identity (test name) and the assertion that failed,
  captured across attempts so the operator sees WHICH test thrashed and HOW. This is NEW capture (no
  structured failing-test record exists today; `test_thrash` is accepted at `adaptive-node.js:1457` but
  carries no payload, investigation §2.3). The delta lives in the SAME `runWriteHalt` return shape as
  the overflow triage — one `triage` object whose populated fields depend on the class. The capture
  SOURCE is deferred to implementation (OQ-P2-c): either a new
  `.cache/test-thrash-<node-id>.md` the dispatching role agent writes, or a field threaded from the
  role-agent dispatch. The invariant is the SHAPE (a `testDelta` field in `triage`), not the source.

- **[INV-17] (forge-neutral JSON — no CLI name in the classifier):** The triage payload is a JSON
  object, never prose naming a specific forge CLI. No `gh` / `glab` / `tea` token appears in the
  classifier or the repair-diff text (the same constraint binding the release publish step, [INV-21]).
  The halt machinery lives in `kaola-workflow-adaptive-node.js`, which ships byte-identical
  (canonical→codex) and is hand-ported to the gitlab/gitea trees; a forge token in the payload would
  break the byte-identity contract. The repair diff describes a `workflow-plan.md` write-set edit — a
  forge-neutral artifact — and the re-freeze step as `--freeze`, never a forge command.

- **[INV-18] (×4 edition propagation):** `runWriteHalt` lives in `kaola-workflow-adaptive-node.js`,
  which ships in all four editions (CLAUDE.md: "Ships in 4 editions; registered in COMMON_SCRIPTS and all
  three install.sh SUPPORT_SCRIPT_NAMES blocks"). The payload enrichment is therefore a CROSS-EDITION
  diff: it MUST reach the canonical tree, the codex byte-twin, and the gitlab/gitea hand-ports, and ALL
  FOUR `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains MUST be green — a green claude
  chain alone is insufficient evidence (CLAUDE.md Validation Policy #307; the `&&` short-circuit hides a
  red forge chain). The three new mechanical subtypes ([INV-14]) and the triage shape must be
  identical across the four; the new classifier helpers belong in the byte-identical surface where the
  edition-sync discipline keeps them aligned. If the operator-facing halt PROSE changes in the plan-run
  surfaces, the six-surface propagation rule (#400: 3 commands + 3 Codex SKILL packs) also applies.

## Decision — Part 4: Release Aggregator

**Decision (P4):** A new `scripts/kaola-workflow-release.js` aggregator with two subcommands —
`--verify` (read-only) and `--cut` (mutating) — COMPOSES the release checks that today live scattered
inside `validate-workflow-contracts.js` (and run only RED after the fact, inside `npm test`) into ONE
typed transaction, ADDING the CHANGELOG-completeness check #417 found missing. `--verify` runs every
check without mutating anything and emits one typed verdict; `--cut` runs only after `--verify` passes,
performs the version bump across all surfaces in one atomic transaction, and PRINTS the publish command
for the operator to run — it NEVER executes the forge CLI itself.

**`--verify` contract (read-only, one typed verdict).** `--verify` runs, as one transaction:
tag-before-test (the tag `kaola-workflow--v<version>` exists, `validate-workflow-contracts.js:556-568`),
tag-ancestry (`tagAncestry`, `release-surface-drift.js:82-102` — INERT on absent/indeterminate, RED only
on a definitive `tag_not_ancestor_of_head`, `release-surface-drift.js:57`), the 3-manifest lockstep
families (claude `plugin.json` × {gitlab,gitea} == `package.json`, lines 469-475; the three
`.codex-plugin/plugin.json` sharing ONE codex baseline, lines 478-496), the README version match (the
`assertIncludes` assertions at lines 464-467, 485), the Codex surface-drift check
(`detectCodexReleaseSurfaceDrift`, `release-surface-drift.js:107-118`), AND the NEW
CHANGELOG-completeness check ([INV-20]). It emits one envelope `{ result: 'pass' | 'refuse', reason,
... }` so each failure is a TYPED refusal, not an English-substring audit finding — an un-changelogged
issue becomes `reason: 'changelog_incomplete'` with the offending issue numbers, never a buried assert.

**`--cut` contract (mutating, gated on `--verify`).** `--cut` runs ONLY if `--verify` passes (it
re-verifies as its first step; a stale `--verify` cannot authorize a `--cut`). On a pass it executes the
release in one atomic transaction: bump the version in `package.json` + the two claude forge manifests +
the three codex manifests + the README version assertions + a CHANGELOG `## [<version>]` heading rename,
create the git tag, then PRINT the publish command as operator guidance ([INV-21]) — it never runs the
forge CLI. The version bump is out-of-allowlist for any barrier-scoped adaptive branch, so `--cut` runs
on a SEPARATE main commit (the run-lesson "release CANNOT ride the barrier-scoped adaptive branch",
investigation §4.1, §4.3); the aggregator is invoked OUTSIDE a plan-run, never as a node write.

**Part 4 invariants:**

- **[INV-19] (`--verify` is read-only; `--cut` re-verifies and gates on the pass):** `--verify` MUTATES
  NOTHING — no version bump, no tag, no file write — it only reads and emits a verdict. `--cut` is the
  ONLY mutating subcommand, and it RE-RUNS the full `--verify` as its first step; a `--cut` proceeds only
  on a fresh in-process pass (never on a remembered prior `--verify` result — the working tree may have
  changed between the two calls). A `--cut` invoked when `--verify` would refuse aborts with the SAME
  typed refusal `--verify` would emit, before any mutation. This mirrors the adaptive-handoff SPAWN-1
  (validate, no write) → SPAWN-2 (re-validate, then write) discipline (`adaptive-handoff.js:268`, `328`),
  where SPAWN-2 re-validates rather than trusting SPAWN-1's verdict.

- **[INV-20] (CHANGELOG completeness — the #417 gap closed as a TYPED refusal):** `--verify` adds a
  completeness check on top of the EXISTING #156 presence check (`validate-workflow-contracts.js:552`,
  which asserts only that a `## [<version>]` heading exists). Completeness detects every issue closed in
  the current batch (since the last tag) that has NO `#N` mention in the `## [Unreleased]` /
  current-version CHANGELOG section, and reports them BY ISSUE NUMBER in the typed verdict (`reason:
  'changelog_incomplete', missing: [<n>, ...]`). This converts the #417 class — a shipped issue with no
  entry, found only by a post-hoc human audit — into a pre-cut typed refusal. The check is the teeth the
  #156 presence check lacks: presence proves the heading exists; completeness proves every shipped issue
  is accounted for.

- **[INV-21] (operator runs the forge CLI — the aggregator only emits the command):** The PUBLISH step
  (`<forge> release create --latest`) is NEVER executed by `kaola-workflow-release.js`. The editions
  contract forbids naming a forge-specific binary in a shipped script (investigation §4.3 — the same
  reason the Surface-B classifier stays forge-neutral, [INV-17]). `--cut` VERIFIES the release is
  publishable (tag pushed, ancestry clean, manifests in lockstep) and PRINTS the publish step as
  operator guidance described forge-neutrally — "run the forge release-create command with `--latest`"
  — NOT `gh release create`. The actual publish stays a human step (or a forge-specific, non-shipped
  wrapper). The aggregator's authority ends at "everything is ready and the tag is pushed; here is the
  command to run"; it never crosses into executing it.

- **[INV-22] (the aggregator REUSES the existing checks — no reimplementation):** `--verify` SHELLS or
  `require()`s the existing check functions — `tagAncestry` and `detectCodexReleaseSurfaceDrift`
  (`release-surface-drift.js:82`, `107`), and the lockstep/README assertions whose logic lives in
  `validate-workflow-contracts.js:469-496` — rather than re-deriving the manifest list or the
  version-comparison logic. The 3-manifest lockstep is ALREADY machine-enforced (investigation §4.3);
  the aggregator is a COMPOSITION over those primitives, not a second implementation of them. Where a
  check is currently inline in `validate-workflow-contracts.js` (the lockstep loops), the implementation
  may extract a shared helper both call, keeping ONE source of truth. This is the same
  composition-not-duplication discipline as the adaptive aggregators (`next-action.js` /
  `commit-node.js` shelled by `adaptive-node.js`, never imported-and-mutated).

- **[INV-23] (×4 edition propagation):** A new shared script touching the release surface must register
  across all four editions: COMMON_SCRIPTS + the three install.sh SUPPORT_SCRIPT_NAMES blocks (via the
  #407 `kaola-workflow-install-manifest.js` single-source manifest), and — because it composes
  `validate-workflow-contracts.js` logic that the gitlab/gitea contract validators mirror — it is a
  CROSS-EDITION diff requiring all four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains
  green, run sequentially (CLAUDE.md Validation Policy #307). The forge editions carry independently-
  numbered codex manifests (`release-surface-drift.js:7-11`), so the aggregator's manifest list and the
  completeness/lockstep checks must be forge-port-aware exactly as the contract validators are. A green
  claude chain alone is insufficient evidence.

## Open Questions

- **OQ-P2-a (the three new mechanical subtypes — pattern source):** `lockfile_write` / `mirror_write` /
  `count_bump` ([INV-14]) need a pattern definition. Is the lockfile/mirror/count-surface pattern a
  hardcoded literal set (e.g. `package-lock.json`; `validate-*-contracts.js` / `test-*.js`; the
  edition-sync target list), a config-driven list, or derived from existing constants (e.g. the
  edition-sync `GENERATED_AGGREGATORS` / install-manifest `SUPPORT_SCRIPTS` for the mirror/count
  classes)? A hardcoded set is simplest but drifts; deriving from existing constants keeps one source of
  truth. Implementation must pick one and pin where the pattern lives so it stays forge-neutral
  ([INV-17]).

- **OQ-P2-b (repair-diff format — for human vs. for autopilot):** The plan-repair diff ([INV-15]) must be
  consumable both by a human (readable) and by the autopilot (D-420-01 [INV-6], machine-applicable). Is
  it a unified-diff text, a structured `{ node, addPaths: [...] }` object the orchestrator renders AND
  applies, or both? The structured form is cleaner for the autopilot's auto-apply path but needs the
  orchestrator to materialize the actual `--freeze` edit; the diff text is human-readable but not
  directly applicable. Pin the shape against D-420-01 [INV-6]'s repair-consent flow.

- **OQ-P2-c (`test_thrash` capture source):** Where is the per-attempt failing-test delta recorded so
  `runWriteHalt --reason test_thrash` can attach it ([INV-16])? A new `.cache/test-thrash-<node-id>.md`
  the dispatching role agent writes (project-local, where the barrier resolves `.cache`, per the adaptive-
  node evidence-path discipline), or a field threaded from the role-agent dispatch into the halt call?
  The `.cache` file is crash-durable and resume-readable; the threaded field is simpler but lost on a
  crash. No structured capture exists today.

- **OQ-P4-a (CHANGELOG-completeness data source — forge-neutrality, investigation §5.3):** Detecting
  "every issue closed since the last tag has an entry" ([INV-20]) needs the closed-issue set since the
  tag — which requires a forge query (`<forge> issue list --state closed`), a FORGE-SPECIFIC read. How
  does `--verify` stay forge-neutral while reading the closed-issue set? Options: read the set the sink
  receipts already recorded (the `sink-merge.js` receipts name the closed issues — a forge-neutral,
  durable source), accept the issue list as an INJECTED parameter (forge query stays the operator's /
  orchestrator's job, the aggregator stays neutral), or derive from CHANGELOG-vs-git-log commit-message
  `#N` mentions. The injected-parameter form best preserves [INV-21]'s forge-neutrality boundary.

- **OQ-P4-b (`--verify` extraction vs. duplication, [INV-22]):** Should `--verify` and
  `validate-workflow-contracts.js` share ONE extracted check module, or does the aggregator call the
  validator's checks in place? Extraction is cleaner long-term (one source of truth) but is a larger
  refactor of a load-bearing validator that runs in every claude chain; calling-in-place risks the two
  drifting. Implementation must decide whether to extract now or thread carefully, ensuring the four
  chains stay green either way.

- **OQ-P4-c (`--cut` atomicity and crash-resume):** `--cut` mutates many surfaces (3 manifests + README
  + CHANGELOG + tag) "in one atomic transaction" ([INV-19]). A crash mid-bump leaves a partial release.
  Is `--cut` re-runnable (idempotent — re-bumping an already-bumped manifest is a no-op), or does it need
  a two-phase marker like the parallel-batch manifest? Given the release rides its own main commit
  (investigation §4.1), the simplest answer may be "re-running `--verify` after a partial `--cut` detects
  the inconsistency and the operator finishes by hand" — but the resume story should be stated.

## Consequences

**What becomes easier:**

- A halted node no longer strands the operator (or the D-420-01 autopilot) with a bare marker: the
  triage payload ([INV-13]/[INV-14]) carries the offending paths and the mechanical class forward, and
  for the overflow family attaches a ready-made repair diff ([INV-15]) that collapses the decision to a
  yes/no. The autopilot's repair-consent flow (D-420-01 [INV-6]) gets the structured input it needs.
- The release becomes one verify-then-cut transaction ([INV-19]) instead of a hand-stamped multi-step
  phase that only goes RED after the fact inside `npm test`. The #417 CHANGELOG gap becomes a pre-cut
  typed refusal ([INV-20]) instead of a post-hoc human audit finding.
- `--verify` is a safe, read-only pre-flight an operator (or the autopilot) can run any time to know
  whether a release is publishable, without touching anything.

**What becomes harder / what to watch:**

- The three new mechanical subtypes ([INV-14]) add classifier surface that MUST stay forge-neutral
  ([INV-17]) and byte-aligned across four editions ([INV-18]); a pattern that drifts (OQ-P2-a) or names a
  forge artifact would break the cross-edition contract. The subtypes NARROW `write_set_overflow`; they
  must never be a fifth precedence family that forks the #406 envelope (the D-419-02 [INV-13] lesson: do
  not fork the emit taxonomy).
- The release aggregator composes a load-bearing validator's logic ([INV-22]); extracting shared helpers
  (OQ-P4-b) risks destabilizing `validate-workflow-contracts.js`, which runs in every claude chain. The
  ×4 propagation ([INV-23]) means a release-aggregator change is a cross-edition diff — all four chains,
  sequentially, every time.
- `--cut`'s atomicity (OQ-P4-c) is genuinely hard: a partial bump is a corrupt release surface. The
  resume story must be explicit before `--cut` ships.

**What is explicitly deferred to implementation:**

- The mechanical-subtype pattern source — OQ-P2-a.
- The repair-diff format (text vs. structured vs. both) — OQ-P2-b.
- The `test_thrash` capture source — OQ-P2-c.
- The CHANGELOG-completeness forge-neutral data source — OQ-P4-a.
- The `--verify` extraction-vs-call-in-place decision — OQ-P4-b.
- `--cut` crash-resume / idempotence — OQ-P4-c.

**For downstream issues:** the invariants binding every Part-2 implementation are [INV-13] through
[INV-18] (threaded-not-rederived diagnosis + #406-reason generalization without forking the taxonomy +
suggestion-not-applied repair diff + symmetric `test_thrash` delta + forge-neutral JSON + ×4
propagation); the invariants binding every Part-4 implementation are [INV-19] through [INV-23]
(read-only `--verify` / re-verifying `--cut` + CHANGELOG completeness as a typed refusal + operator-runs-
the-forge-CLI + reuse-not-reimplement + ×4 propagation). The cross-cutting invariants shared with
D-420-01 are the #44 agent-owns-reasoning boundary (the repair diff is a SUGGESTION the orchestrator
consents to applying, [INV-15] ↔ D-420-01 [INV-6]) and the forge-neutrality contract that binds both the
halt classifier ([INV-17]) and the release publish step ([INV-21]).
