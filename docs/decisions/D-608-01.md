# D-608-01. Persist a chain's timeout verdict in the receipt; recalibrate the default kill ceiling

Date: 2026-07-03
Status: Accepted
Issue: #608
Related: #512 (the original 15-min default this decision raises again)

## Context

`kaola-workflow-run-chains.js` runs each edition's test chain under a `spawnSync`/`spawn` kill
ceiling (`KAOLA_RUN_CHAINS_TIMEOUT_MS`), computing an internal `_timedOut` marker per chain when
the process is killed for exceeding it. That marker was stripped before persisting the receipt
(`.cache/chain-receipt.json`), alongside the raw `_output` field — so a killed chain's receipt
entry looked identical in shape to a genuinely failing chain: same `exitCode: 1`, no field
distinguishing "the tests are red" from "the process was still running when the clock ran out."
An operator (or the plan-validator's `--finalize-check` gate) reading only the receipt had no way
to tell the two apart without re-running the chain or digging through raw stderr.

The default ceiling itself was also under-calibrated for at least one real host: it had already
been raised once (#512, hardcoded 600000ms → 900000ms after a live run measured the claude chain
at ~574s standalone), and a subsequent live run on this same reference box exceeded even the raised
900000ms (15 min) bound, producing a `chains_red` failure that was, per the receipt shape above,
indistinguishable from a real regression.

## Decision

**Observability first: persist the fact of a timeout, not just its exit code. Recalibrate the
default ceiling to the box that outgrew it.**

1. **Promote `_timedOut` into the receipt as `timed_out: true|false`.** No new computation — the
   marker was already computed by both `runChainSync` and `runChainAsync` on every attempt; it was
   simply discarded before persisting. It is now carried into each per-chain receipt entry
   alongside the existing `attempts`/`retried_transient`/`accepted_red` fields. Absent on a receipt
   written before this change (backward-compatible): a reader that does not know the field treats
   its absence as `false`, matching the pre-#608 behavior exactly — no reader needs updating.
2. **Label a timed-out chain inline in the plain-text failure summary.** `run-chains: N chain(s)
   failed: claude (TIMEOUT at <Ns> — raise KAOLA_RUN_CHAINS_TIMEOUT_MS or investigate a hang)` —
   an operator scanning stderr (the common case; the JSON receipt is a second-look artifact) sees
   the remedy without opening the receipt. Non-timed-out failures in the same run are printed
   unlabeled, unchanged from before.
3. **Thread the same distinction into the `chains_red` operator hint.** The plan-validator's
   `--finalize-check` operator-hint function now accepts the set of chains that timed out and, when
   non-empty, names the timeout-vs-red distinction and the `KAOLA_RUN_CHAINS_TIMEOUT_MS` remedy in
   its hint text; when no red chain timed out it falls back to the pre-existing generic hint. This
   is hint text only — the refuse/pass decision (`redChains.length` check) is byte-for-byte
   unchanged, so no existing barrier-logic test is affected.
4. **Raise the default kill ceiling `900000 → 1800000` (15 min → 30 min).** `resolveTimeoutMs`'s
   fallback is raised directly; the doc comments (env-var table, receipt-schema comment, function
   header) are updated in the same commit so the code and its own documentation cannot drift
   relative to each other. Explicit overrides via `KAOLA_RUN_CHAINS_TIMEOUT_MS` are unaffected —
   this changes only what happens when the variable is unset, non-numeric, zero, or negative.

## Consequences

- A receipt now answers "did this chain time out?" on its own, without re-running anything — the
  `timed_out` field is the durable record of a fact that was previously computed and thrown away.
- The failure summary and the `--finalize-check` operator hint both point at the SAME remedy
  (`KAOLA_RUN_CHAINS_TIMEOUT_MS`), so an operator gets a consistent answer whether they read stderr
  first or the JSON receipt first.
- Raising the default buys headroom on the box that outgrew 900s, at the cost of a slower
  discovery of a genuine infinite-hang on a host where the chain would otherwise complete quickly —
  accepted, because a real hang still eventually times out and is now correctly LABELED as a
  timeout rather than silently conflated with a red test run; the field this decision adds is what
  makes that acceptable.
- Cross-edition (#307): `scripts/kaola-workflow-run-chains.js` (canonical + codex byte-twin + both
  forge ports — the forge run-chains ports are hand-rename-normalized, not `edition-sync`-generated,
  per the existing `validate-script-sync.js` classification) and
  `scripts/kaola-workflow-plan-validator.js` (canonical + codex byte-twin + both forge ports); all
  four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains must stay green.

## Non-goals

- **No auto-scaling or adaptive budgets.** The ceiling is a fixed default plus a single env
  override, exactly as before — this decision does not add historical-runtime tracking, per-chain
  learned budgets, or any mechanism that changes the ceiling based on past run data. A future
  timeout regression is a manual recalibration (raise the default again, or set the env var), the
  same mechanism used twice now (#512, this decision).
- **No change to WHICH chains run, their order, or the retry-on-transient-fault policy (#550).**
  Only the kill ceiling and what the receipt records about it are in scope.
- **No change to the receipt's greenness semantics.** A timed-out chain still counts as a failed
  chain for every existing purpose (`chains_red`, `--accept-known-red` waivers, `overallExitCode`)
  — `timed_out` is an additional diagnostic field, not a new pass/fail category.

## Alternatives considered

- **Auto-retry a timed-out chain once, treating it as a transient fault like the existing #550
  retry mechanism.** Rejected: a timeout is not obviously transient (it may be a genuine hang) the
  way the #550 transient-signature class is (a known flaky-infra pattern); silently retrying could
  mask a real regression that happens to also be slow. Surfacing the timeout clearly and letting
  the operator decide (raise the ceiling, or investigate the hang) is the more honest default.
- **Compute a per-host or per-chain adaptive ceiling from historical run durations.** Rejected as
  disproportionate: the repo's test chains are not so numerous or variable that a fixed default
  plus one manual env override has stopped being sufficient — it has needed recalibrating exactly
  twice in the project's history. Adaptive budgeting is real engineering for a problem two
  raise-the-constant fixes have adequately solved so far.
