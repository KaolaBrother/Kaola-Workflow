# D-725-01. Adaptive-only consolidation, Phase A: the `fast`/`full` paths and the install-time opt-in axis are retired

Date: 2026-07-19
Status: Accepted
Issue: #725 (epic; Phase A of 5)
Supersedes: docs/decisions/D-538-01.md (switch-axis flip: adaptive unconditional default,
             `fast`/`full` become install-time opt-ins), docs/decisions/D-543-01.md (the
             `--with-fast`/`--with-full` opt-in partition ported to Codex + the opencode
             standalone fix folded into the same change)
Related: #227 (original adaptive path), #400 (six-surface routing-prose propagation),
         #307 (four-chain cross-edition parity), D-530-02 (opencode is additive, not a forge),
         D-703-01 (kimi is additive, not a forge)

## Context

D-538-01 flipped adaptive from a switch-gated default to the unconditional default, and made
`fast`/`full` install-time opt-ins (`--with-fast` / `--with-full`) recorded in a shared
`installed_paths` config field. D-543-01 ported that same opt-in partition to Codex (and, as a
folded discovery, fixed an opencode path-leak). Both decisions kept `fast` and `full` as real,
reachable, install-time-selectable paths — a second install-time axis alongside forge and
runtime.

In practice `fast`/`full` had converged to near-zero live usage relative to adaptive, while their
presence still cost real, ongoing tax: two extra install flags, a second `installed_paths` config
axis, 56 tracked files (canonical scripts + forge ports + dead tests + commands + Codex skills +
historical investigation docs) that every review, every cross-edition parity guard, and every
route-reachability check had to keep in sync, plus router/finalize prose that had to describe THREE
paths instead of one. Per the project's own axiom order (`CLAUDE.md` § First Principles — axiom 3,
"then spend as little as possible"; axiom 1, "correct first" — an unused surface is an unrewarded
correctness liability), retiring the axis outright removes more tax than the opt-in already saved.

This is Phase A of the #725 epic (adaptive-only consolidation): retire `fast`/`full` and the
opt-in axis entirely, converging the workflow on the single adaptive path.

## Decision

### (a) `WORKFLOW_PATHS` collapses to `['adaptive']`; the opt-in machinery is removed

`kaola-workflow-adaptive-schema.js` (byte-identical ×4): `WORKFLOW_PATHS` changes from
`Object.freeze(['fast','full','adaptive'])` to `Object.freeze(['adaptive'])`. The
`INSTALLED_PATHS_FIELD` constant and the `resolveInstalledPaths(config)` function are removed
entirely, along with every caller. `isLegalWorkflowPath(value, installedPaths)` is kept verbatim
(still a two-arg function) — with no caller ever supplying a non-empty `installedPaths` array
anymore, adaptive is the only value that can pass. A stale `installed_paths` field from a
pre-retirement install is **tolerated on read** (never dereferenced for legality; the classifier's
defensive parse simply never fires) and **never written** by any installer going forward.

### (b) 56 files deleted; every dynamic coupling to them removed

4 canonical scripts (`kaola-workflow-{fast-advance,fast-audit,full-advance,phase4-advance}.js`),
their 9 forge-port mirrors, 4 dead tests, 18 retired commands (`kaola-workflow-{fast,phase1..5}.md`
× 3 editions), 18 retired Codex skill packs (`kaola-workflow-{fast,research,ideation,plan,execute,
review}/SKILL.md` × 3 editions), and 3 fast/full-dedicated historical investigation docs. The two
surviving dynamic couplings to the deleted scripts — `repair-state.js` constructing a
`*-full-advance.js` path by basename-replace, and `claim.js` shelling `kaola-*-full-advance.js` at
Finalization — are both removed at the source.

### (c) `claimProject`'s scaffold defaults to adaptive; `cmdFinalize`'s plan-absent branch collapses
    to a typed `adaptive_plan_missing` refusal

`kaola-workflow-claim.js` (×4): the `writeState` scaffold's `workflow_path` default flips from
`|| 'full'` to `|| adaptiveSchema.ADAPTIVE_PATH` — an absent field can no longer misroute to the
now-nonexistent full path. `cmdFinalize`'s plan-absent branch — which used to shell
`*-full-advance.js phase5-verify` as a verifier-N/A pass — is replaced with an **unconditional**
typed refusal, placed before any archive/close side effect:

```json
{
  "result": "refuse",
  "reason": "finalize_gate_unverified",
  "gate": "workflow_path",
  "inner_reason": "adaptive_plan_missing",
  "workflow_path": "<stale value, reported for diagnostics only>",
  "operator_hint": "Restore the frozen workflow-plan.md before Finalization. No archive or closure side effect was made.",
  "errors": ["adaptive_plan_missing"]
}
```

There is no retired fast/full verifier to shell and no N/A pass — Finalization with no frozen
`workflow-plan.md` present is now always a refusal. The `path_not_installed` refusal (from
D-538-01's legality gate) is unchanged in name and shape; only its resolution source moved from
`resolveInstalledPaths` to the now-empty, adaptive-only `WORKFLOW_PATHS`.

### (d) `install.sh` drops `--with-fast`/`--with-full`; install never writes `installed_paths`

The `WITH_FAST`/`WITH_FULL` variables, their arg-parse cases, the `EFFECTIVE_*` union block, and
the fast/full command-install gating are all removed from `install.sh`. `--with-fast` and
`--with-full` now fall through to the generic unknown-argument branch (exit 2, "Unknown
argument"). The config-write step seeds `parallel_mode` (setdefault `'auto'`) and strips any stale
`installed_paths` key (`config.pop("installed_paths", None)`) alongside the pre-existing
`enable_adaptive` migration — mirroring how D-538-01 itself retired `enable_adaptive`.
`install-codex-agent-profiles.js` (×3, byte-identical) drops the same `WITH_FAST`/`WITH_FULL`
argv parsing and the `installed_paths` UNION writer from `seedKaolaConfig`, which now takes a
single `homeDir` argument. `--enable-adaptive` keeps its existing warn-and-ignore behavior on both
installers.

### (e) Routing, finalize commands, validators, and forge tests converge on the retired tree

`templates/routing/next.skeleton.md`'s Path Intent step collapses to unconditional
`export KAOLA_PATH=adaptive`; the 6 generated `next` surfaces are regenerated (the 6 `plan-run`
surfaces stay byte-identical — they never depended on this content). The 6 finalize commands
(3 Claude + mirrored forge prose) drop the `workflow_path: fast`/`full (or absent)` branches and
describe the actual `adaptive_plan_missing` refusal from (c). `validate-*-contracts.js`,
`validate-script-sync.js`, the forge test scripts, `package.json`'s test chains, and all six
walkthrough simulators are updated to the retired surface (dropping fast/full command-basename
pins, `installed_paths`/`resolveInstalledPaths` assertions, and dead-test entries; walkthrough
finalize fixtures that used to short-circuit through a `workflow_path: fast` marker are migrated
to seed a real, gate-passing frozen adaptive plan instead). The additive `opencode`/`kimi` runtime
editions are re-synced from the retired canonical tree and their own test suites are narrowed to
the surviving surface (opencode: 396 assertions, was 525; kimi: 440 assertions, was 577).

### (f) Trap-1 vocabulary is preserved, not swept

`escalated_to_full` (the adaptive review-escalation token), "full diff"/"full envelope"
barrier-scope vocabulary, git plumbing (`--symbolic-full-name`, "fast-forward"), and the
legacy-compatibility review attestation literal `review_attestation: full_review_completed`
(emitted by `code-reviewer`/`security-reviewer` for the schema-1 compatibility block, unrelated to
the retired full path) are all untouched — none of these name the retired `fast`/`full` **paths**,
only unrelated concepts that happen to share the words "fast"/"full".

## Known residual (not closed by Phase A)

Two write surfaces still carry pre-retirement `fast`/`full` wiring and were **not** touched by this
phase (tracked as open follow-up work, not claimed fixed here):

- The 3 Codex `kaola-workflow-finalize/SKILL.md` packs (one per edition) still shell
  `kaola-*-workflow-full-advance.js` at their phase5-verify point-of-use block — a script this
  phase deletes. A Codex-runtime finalize hitting that branch would fail to resolve the script.
  `commands/workflow-init.md` (× 3 editions) and the matching `kaola-workflow-init/SKILL.md`
  (× 3 editions) still document `--with-fast`/`--with-full` install commands.
- `install-opencode.sh` and `install-kimi.sh` still parse `--with-fast`/`--with-full` and still
  reference the (now-deleted-from-canonical) `kaola-workflow-fast.md` / `kaola-workflow-phase[1-5].md`
  source names in their own opt-in partition logic; `.env.example` still documents the retired
  `install.sh --with-fast`/`--with-full` flags.

Neither of these is claimed retired anywhere in this decision's own propagation surfaces
(agent-facing prompts carry no provenance either way); they remain accurate, tracked gaps for a
follow-up phase or issue to close.

## Alternatives considered

- **Keep `fast`/`full` as install-time opt-ins (status quo, D-538-01/D-543-01).** *Rejected.*
  Near-zero live usage did not justify the ongoing tax of two extra install flags, a second config
  axis, 56 files across every edition, and three-way (not one-way) router/finalize prose — axiom 3
  ("spend as little as possible") tips outright removal over continued opt-in maintenance once
  usage no longer offsets the tax.
- **Deprecate with a warning, remove later.** *Rejected.* A silent-deprecation window adds a
  temporary fourth state (installed-but-deprecated) with no correctness benefit; the project's
  fail-closed posture (axiom 1) favors a clean, typed `path_not_installed` refusal today over a
  warning nobody is guaranteed to read before the eventual removal anyway.

## Consequences

**Positive:**
- One workflow path, one router branch, one finalize branch, one set of tests to keep green — the
  ongoing multi-edition maintenance tax the opt-in axis imposed is gone.
- `installed_paths` becomes pure historical debt: tolerated on read, never written, with no live
  writer left to keep in sync with the field's own schema.
- `cmdFinalize`'s plan-absent case is simpler and stricter — a typed `adaptive_plan_missing`
  refusal instead of a shell-out to a verifier script that could itself go stale.

**Negative / note:**
- A pre-retirement install that still has `installed_paths: ["fast"]` (or `["full"]"`) on disk, or
  an in-flight project whose `workflow_state.md` still carries `workflow_path: fast`/`full`, is
  refused by the new legality gate rather than silently continuing on the old path — this is
  intentional (axiom 1: never trade correctness for compatibility with a retired feature) but is a
  real, user-visible behavior change for anyone still on the old opt-in paths at upgrade time.
- The two residual surfaces named above (Codex finalize/init SKILL packs, opencode/kimi
  installers, `.env.example`) are real, not-yet-closed gaps this phase deliberately did not touch;
  see the epic body (#725) for which later phase closes them.
- `D-538-01`/`D-543-01` are superseded, not deleted — their bodies remain the historical record of
  why the opt-in axis existed and how it was ported to Codex/opencode; their status lines now read
  "Superseded by D-725-01".
