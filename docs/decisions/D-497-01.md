# D-497-01. Wire `closure-audit.js` into all six finalize-route surfaces as defense-in-depth complement to the #497 inline emit fix (issues #496/#497)

**Date:** 2026-06-16
**Status:** Accepted
**Issue:** #497 (the inline emit fix); #496 (the `assertWorktreeClean` fail-closed fix)
**Related:** #165 (closure-audit.js origin — closure drift detection and safe repair),
            #497 (inline `push_main`/`closure` failure emit fix — the transactional complement),
            #400 (the finalize-route ×6 prose surface rule; established that routing prose
                  propagates to 3 Claude commands + 3 Codex SKILLs),
            `scripts/test-route-reachability.js` T6 (the machine-enforcement assertion),
            D-476-01 (prior destructive-script safety record, same broad risk class),
            D-426-01 (copy-then-verify-then-delete archive fix, same integrity posture).

## Context

`kaola-workflow-closure-audit.js` (shipped in #165) detects and repairs **closure drift** —
completed work that still shows as active across roadmap sources, the generated `ROADMAP.md`,
active folders, remote issue state, and the `workflow:in-progress` label. It is a general-purpose
reconciliation sweep, not scoped to any one failure scenario.

Before #496/#497, the script was invoked by zero live prose surfaces: it was documented in
`docs/api.md` and tested by the walkthrough, but no finalize command or Codex SKILL mentioned it
as a post-sink step an operator should run. This meant:

1. **Inline emit gap (#497):** when `push_main` or `closure` failed, `sink-merge` swallowed
   the error, recorded the step as done, and emitted `{result:"ok", status:"sinked"}` — a false
   success. The #497 fix closes this at transaction time (fail the emit immediately on hard
   failure, preserve branch for retry).

2. **Broad drift gap (pre-existing):** even with the #497 inline fix in place, historical runs
   that completed before the fix may carry residual drift (stale labels, orphan sources, unarchived
   folders), and future transient faults that recover and succeed at the script level still benefit
   from a periodic sweep. `closure-audit.js` is the correct tool for this — but with zero prose
   references, operators had no reason to run it after a sink.

Two remediation postures were considered:

**Option A — REMOVE the reference gap by removing `closure-audit.js` from the finalize surface.**
Treat the inline #497 fix as sufficient. The transactional catch covers new failures; historical
drift is a separate concern already handled by standalone `closure-audit.js` invocation.

**Option B (WIRE) — Add `closure-audit.js` to all six finalize-route surfaces as a
defense-in-depth reconciliation step, complementary to the #497 inline fix, and machine-pin it.**
The inline fix and the sweep address orthogonal failure modes: the inline fix is immediate and
transactional (it refuses at the moment of failure); the sweep is periodic and broad (it catches
drift across ALL archived runs, including historical ones and scenarios where transient faults
recovered). Both mechanisms together cover the full space.

## Decision

**WIRE (Option B).**

`closure-audit.js` is wired into all six finalize-route surfaces as a **post-sink
reconciliation sweep**. The six surfaces are:

1. `commands/kaola-workflow-finalize.md` (canonical Claude GitHub command)
2. `plugins/kaola-workflow/skills/kaola-workflow-finalize/SKILL.md` (Codex GitHub SKILL)
3. `plugins/kaola-workflow-gitlab/commands/kaola-workflow-finalize.md` (Claude GitLab command)
4. `plugins/kaola-workflow-gitlab/skills/kaola-workflow-finalize/SKILL.md` (Codex GitLab SKILL)
5. `plugins/kaola-workflow-gitea/commands/kaola-workflow-finalize.md` (Claude Gitea command)
6. `plugins/kaola-workflow-gitea/skills/kaola-workflow-finalize/SKILL.md` (Codex Gitea SKILL)

Each surface carries:
- The `<!-- PIN: closure-audit -->` HTML comment as a machine-readable anchor.
- A "Sink result handling and closure-audit reconciliation sweep (#496/#497)" section explaining
  the two complementary mechanisms: the inline `sink_incomplete` refuse (immediate catch) and the
  `closure-audit.js` sweep (periodic broad drift detection).

The pin is machine-enforced by a **fail-closed T6 assertion** in
`scripts/test-route-reachability.js`: every surface is checked unconditionally (not a warn-gate)
for both `<!-- PIN: closure-audit -->` and the `closure-audit` literal. This runs as part of
`npm run test:kaola-workflow:claude` (and all four edition chains via `npm test`).

**Why not Option A (remove):**

The inline fix (#497) is transactional — it only catches failures it observes at runtime. It
does not retroactively repair historical drift, does not catch drift introduced by other paths
(e.g. a manual `gh issue close` that missed the label removal), and does not surface the
`active_folder_for_closed_issue` / `unarchived_pr_folders` classes that `closure-audit` reports.
The sweep is genuinely complementary, not redundant. Removing the reference gap while the tool
exists and is already tested would leave operators without guidance on when to use it and why.

**Why machine-pin at T6:**

`closure-audit.js` was already documented in `docs/api.md` but carried zero enforced
cross-edition prose references — the exact pattern that #400 identified as the finalize-route
"dead zone" (the two forge-codex SKILLs that #400 proved unguarded). A WIRE without a machine
pin would reproduce the same gap in a new location. T6 is fail-closed (unconditional assert, not
the `anyHasPin` advisory pattern that T5 uses) because the six surfaces are enumerated and their
count is stable.

## Consequences

**For operators:**

- After a successful or retried sink, operators are directed in every finalize surface to run
  `closure-audit.js` (dry-run by default, `--execute` to repair safe local drift). This is
  presented as a defense-in-depth step, not a required gate on success.
- A `sink_incomplete` refuse from `sink-merge` (the #497 inline fix) is now described alongside
  the `closure-audit` sweep in the same "Sink result handling" section — the two mechanisms are
  visible together so operators understand both tools and when each fires.

**For the test suite:**

- `scripts/test-route-reachability.js` T6 is a new fail-closed assertion (lines ~173-194).
  It runs as part of `test-route-reachability.js`, which is invoked by all four edition chains.
  A surface that loses the `<!-- PIN: closure-audit -->` comment fails the chain.
- The T6 assertion uses an unconditional `assert()` loop (not a soft warn-gate), so a
  single missing surface is an immediate chain failure, not a non-blocking warning.

**For the `closure-audit.js` API:**

- No change. The CLI surface (`--execute` flag; JSON always; no `--project`/`--json`) and output
  schema (dry-run: `drift` + `counts`; execute: `repaired` + `reported_not_repaired`) are
  unchanged by this wiring. See `docs/api.md` § "Closure audit and repair (issue #165)" for the
  full schema.

**For drift between the inline fix and the sweep:**

- The inline `sink_incomplete` refuse covers new hard failures at transaction time.
- `closure-audit.js` covers the broad space: all archived runs, historical drift, and scenarios
  where the forge fault recovered after the fact. The two mechanisms are orthogonal and additive;
  neither replaces the other.
