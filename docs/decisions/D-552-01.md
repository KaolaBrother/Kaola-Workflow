# D-552-01. Ledger-derived last-member detection + reconcile self-heal + sink-side lane_group backstop

Date: 2026-06-21
Status: Accepted
Issue: #552
Related: #463/D-463-01 (the per-leg worktree isolation + synthesizer this hardens), #542/D-542-01
         (write co-open default-on — the path this bug lives on), #272 (next-action = pure-ledger, the
         separation this decision preserves), #424 (the barrier-base window-lock the two-write order
         comes from), #484/#496 (the prior fail-closed sink-side backstops this mirrors in shape).

## Context

On the DEFAULT write-parallel path (#542 co-open), `adaptive-node` `closeGroupMember` closes a
lane-group member in TWO non-atomic writes: (1) flip the member's ledger row to `complete` (+ a
compliance row); (2) record the member in `lane_group.closed_members` and drop it from the running set.
The genuine **last member** was detected by reading `closed_members` (`otherIds.every(id =>
closedBefore.has(id))`).

A crash (or an unlocked near-simultaneous RMW) BETWEEN those two writes leaves a member ledger-`complete`
but ABSENT from `closed_members` (and still in `running.nodes`). On resume the real last member then
mis-computes `isLast = false`, takes the DEFER branch, and the synthesizer + group barrier NEVER run —
the other legs' committed work is never merged into the feature branch. Worse, `reconcile-running-set`'s
surviving-group teardown tore down that close-direction member's leg worktree + branch (and dropped it
from the legs manifest), so even a later synthesizer could not recover it. Meanwhile `next-action`'s
`allDone` is pure-ledger (all rows terminal → done) and `sink-merge` was lane_group-blind, so the run
went GREEN and main advanced **with committed code missing** — a fail-OPEN durable-state corruption on
the default path (violates precedence #1).

## Decision

Three coordinated fixes (runtime correctness + a fail-closed backstop), keeping the #272 separation
intact (`next-action` stays pure-ledger; the teeth live at the irreversible main-advance point).

1. **Ledger-derived `isLast` (`closeGroupMember`).** Derive "last member" from the AUTHORITATIVE
   ledger — every OTHER member id has a TERMINAL ledger status (`readLedgerStatuses` + the
   `{complete,n/a,n.a,na}` set), with `closed_members` kept as a union FALLBACK so a transient ledger
   read miss never regresses below today. A member that wrote its ledger row `complete` BEFORE a crash
   is correctly counted even if `closed_members` is stale → the last member still synthesizes.

2. **Reconcile self-heal + leg retention (`runReconcileRunningSet`).** On a SURVIVING lane group, a
   close-direction terminal member (in `closed`) is EXCLUDED from the leg-teardown set (only
   open-direction `dropped`/`cappedOut`/`stale` are torn down) so its committed leg work survives for
   the last-member synthesizer; and its id is appended (deduped) back into `closed_members` so the
   optimization field re-converges with the ledger. A whole-group drop still tears down all legs.

3. **Sink-side `lingering_lane_group` backstop (`sinkPreflight`, ×4 editions).** A clean group
   completion DELETES the `lane_group` key; a residual key at sink time means the group never
   synthesized + merged its legs. `sinkPreflight` (a pure read, zero mutation, BEFORE the dirty-tree
   scan) reads `running-set.json` from BOTH the live `kaola-workflow/<project>/.cache/` and the
   post-finalize `kaola-workflow/archive/<project>/.cache/` locations and refuses
   (`lingering_lane_group`, exit 1, main not advanced) if either carries a non-empty `lane_group`.

## What this deliberately does NOT do

- **It does not make `next-action.js` `allDone` lane_group-aware.** That would couple the pure-ledger
  aggregator (#272) to `running-set.json`. The fail-closed teeth belong at the ONE irreversible point
  (the sink), not in the readiness computer. The runtime fixes (1)+(2) prevent the desync; (3) is the
  backstop.
- **It does not add the backstop to `cmdFinalize` as a SECOND gate.** `sink-merge` is the authoritative
  irreversible main-advance; a second gate risks divergent reason codes / double-refusal for no extra
  accuracy. Surgical, single source of truth (precedence #3).
- **It adds NO declared-write presence floor to the group barrier.** The commit-based union barrier
  already enforces per-leg-head ancestor inclusion (`leg_omitted_from_merge`, `plan-validator`), which
  IS the presence guard for the default leg-live path; a redundant floor would false-refuse legitimately
  `no_op` members (the exact reason the whole-plan floor is skipped in group mode).

## Consequences / known bounds (precedence #1 honored)

- **Defense in depth.** Either runtime fix alone prevents silent loss; the sink backstop catches any
  OTHER path that could leave a lingering lane_group. The fail direction is closed: a lingering group
  REFUSES the sink rather than silently merging with code missing.
- **Scoped to a NON-EMPTY `lane_group` key.** A cleanly-completed run deletes the key (not empties it),
  so a leftover `running-set.json` with no lane_group never false-trips the backstop (proven by the
  walkthrough GREEN companion).
- **Idempotent recovery.** `synthesizeLevel` is idempotent and a merge that committed M before a barrier
  refuse is recoverable, so the remediation (reconcile → resume → re-sink) is safe to repeat.
