## Context

D-420-02 pinned the typed envelope (`{ result, reason }`, `changelog_incomplete` with `missing: [n]`) and
the high-level `--verify` / `--cut` contract for the release aggregator but left the following design
questions open:

- **Data source for closed-issue set** — how `--verify` discovers which issues are closed since the last
  tag without naming a forge CLI (OQ-P4-a in D-420-02).
- **Architecture** — whether the aggregator calls the existing checks in place or extracts shared helpers
  (OQ-P4-b in D-420-02).
- **Version input** — whether version is inferred automatically or required explicitly (not pinned in
  D-420-02).
- **Push ownership** — which step pushes the tag and triggers the forge release, and under what flag.
- **Crash-resume** — how a partial `--cut` is detected and recovered (OQ-P4-c in D-420-02).
- **Registration surface** — whether `kaola-workflow-release.js` lands in the install-manifest
  `SUPPORT_SCRIPTS` block (the issue body implied it did; this record corrects that).

A census of forge plugin manifests was conducted during investigation. The BINDING count is: exactly
**two Claude forge plugin manifests** (`plugins/kaola-workflow-gitlab/.claude-plugin/plugin.json` and
`plugins/kaola-workflow-gitea/.claude-plugin/plugin.json`) and **three Codex manifests**
(`plugins/kaola-workflow{,-gitlab,-gitea}/.codex-plugin/plugin.json`). An earlier investigation document
recorded "four claude manifests"; that figure is wrong and must not drive the bump surface.

Issue #442 settles all of the above. The decisions below are binding for the implementation.

## Decision

### 1. Architecture — composition over extraction

`scripts/kaola-workflow-release.js` is a **new aggregator** that **calls the existing validators in
place** — it does not extract logic from `validate-workflow-contracts.js` or `test-release-surface-drift.js`
into shared modules. This is the house composition pattern: adaptive-node shells `commit-node.js` and
`next-action.js` without importing and mutating them; this aggregator follows the same discipline.
`--verify` shells or `require()`s the existing check functions (`tagAncestry`,
`detectCodexReleaseSurfaceDrift`, the lockstep/README assertions in `validate-workflow-contracts.js`)
rather than re-deriving their logic.

### 2. `--verify` — closed-issue set derivation and greenness signal

`--verify` is **read-only** (no mutation). It derives the closed-issue set by combining two forge-neutral
sources:

- **git-log since the last `kaola-workflow--v*` tag** — commit messages carrying `#N` refs.
- **CHANGELOG `[Unreleased]` section** — every `#N` mention in the current unreleased block.

These two sets are cross-checked against forge issue state **when online** (to confirm the referenced
issues are actually closed). When offline, the check degrades to changelog-only with `verification:
"offline"` in the receipt — a silent pass is **never** permitted. Unaccounted issues produce a typed
refusal: `{ result: "refuse", reason: "changelog_incomplete", missing: [<n>, ...] }`.

For the **greenness signal**, `--verify` consumes the `#432` chain receipt
(`.cache/chain-receipt.json`), reading the real per-chain `exitCode` values recorded by
`kaola-workflow-run-chains.js`. It never reads greenness from prose or from a piped `tail` exit code.

### 3. `--cut --version X.Y.Z` — explicit version, monotonic guard, atomic bump

Version input is **required and explicit** in v1 (no auto-semver inference). `--cut` without
`--version X.Y.Z` is a typed refusal. Non-monotonic versions (where `X.Y.Z` is not strictly greater
than the current `package.json` version) are refused before any mutation.

On a verified pass, `--cut` executes the release in one atomic transaction:

- Renames `## [Unreleased]` to `## [X.Y.Z] - <ISO date>` in `CHANGELOG.md`.
- Bumps the version in `package.json`.
- Bumps the **three `.codex-plugin/plugin.json`** manifests in lockstep (canonical + gitlab + gitea
  codex manifests). This is the #245-class contract.
- Updates the README version assertion(s).
- Creates the git tag `kaola-workflow--vX.Y.Z` **locally** (tag-before-test is preserved — `npm test`
  asserts the tag exists).

`--cut` re-runs the full `--verify` as its first step (an in-process re-verification, not a remembered
prior result); it aborts with the same typed refusal if the working tree has changed since `--verify`
last passed.

### 4. `--push` — explicit gate for remote mutation

Pushing the tag and forge release publish (`--latest`) happens **only under explicit `--push`**. A `--cut`
without `--push` makes no remote mutations — the tag is created locally and no forge CLI is invoked.

The aggregator is **forge-neutral**: `kaola-workflow-release.js` names **no forge CLI binary** (`gh`,
`glab`, `tea`) anywhere in its source. Under `--push`, the publish step is emitted as forge-neutral
operator guidance — "run the forge release-create command with `--latest`" — and the actual invocation
remains a human (or forge-specific, non-shipped) step. This is the same forge-neutrality constraint as
the Surface-B classifier (D-420-02 [INV-17], [INV-21]).

### 5. Crash-resume — step-receipt JSONL

`--cut` writes a **step-receipt JSONL** file (the `#429` pattern) recording each completed mutation step
as it executes. A re-run reads the receipt and skips already-completed steps idempotently. This makes
a partial `--cut` (from a crash mid-bump) detectable and resumable: the re-run re-verifies the working
tree, replays only the remaining steps, and records their completion.

### 6. Registration surface — `COMMON_SCRIPTS` only, NOT `SUPPORT_SCRIPT_NAMES`

`kaola-workflow-release.js` is a **maintainer release tool** run outside a plan-run on a separate main
commit — the same operational profile as `release-surface-drift.js`, which the install manifest
explicitly excludes from `SUPPORT_SCRIPT_NAMES` as a dev/CI tool.

**Verdict:** Register `kaola-workflow-release.js` in `COMMON_SCRIPTS` (so the canonical-to-codex
byte-mirror is enforced by `validate-script-sync.js`) and in the rename-normalized forge-ports family
(the run-chains precedent), but **NOT** in the install-manifest `SUPPORT_SCRIPT_NAMES` block.

If a chain goes red demanding manifest registration after this decision, **stop and surface it** rather
than silently widening the SUPPORT_SCRIPTS surface.

### 7. Forge-neutrality and cross-edition propagation

The aggregator is **byte-mirrored canonical to codex**:

- Canonical: `scripts/kaola-workflow-release.js`
- Codex byte-twin: `plugins/kaola-workflow/scripts/kaola-workflow-release.js` (byte-identical)
- GitLab hand-port: `plugins/kaola-workflow-gitlab/scripts/kaola-gitlab-workflow-release.js`
  (rename-normalized; functionally identical since the script names no forge CLI)
- Gitea hand-port: `plugins/kaola-workflow-gitea/scripts/kaola-gitea-workflow-release.js`
  (rename-normalized; functionally identical)

All four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains **must be green, run
sequentially**, before finalization (#307). A green claude chain alone is insufficient evidence.

## Consequences

**A single typed transaction** fuses tag-before-test, the three-codex-manifest lockstep bump, the README
assertion update, the CHANGELOG rename, the forge release publish guidance (under `--push`), and
changelog completeness into one verify-then-cut envelope. The `#417` class of gaps — a shipped issue
with no CHANGELOG entry, discovered only by a post-hoc human audit — becomes a pre-cut typed refusal
(`changelog_incomplete` with `missing: [n]`) before any mutation occurs.

**Test fixtures** the implementation must cover (binding for the implementer):

- `changelog_incomplete` with `missing: [n]` when a closed issue has no CHANGELOG mention.
- Lockstep-violation refusal when the three Codex manifests are not at the same version.
- Non-monotonic version refusal (e.g. bumping from `5.16.0` to `5.15.0`).
- Offline-verify receipt: `verification: "offline"` is present; silent pass is absent.
- Cut-without-push leaves no remote mutation (no tag push, no forge release created).

**What is no longer open** from D-420-02: OQ-P4-a (closed-issue data source resolved to git-log + CHANGELOG
cross-check with optional online forge confirmation); OQ-P4-b (call-in-place architecture, no extraction);
OQ-P4-c (step-receipt JSONL for crash-resume).

**For downstream implementation:** the census of manifests is binding — two Claude forge manifests, three
Codex manifests. Any implementation touching a different count is wrong and must stop to reconcile before
proceeding.

## Amendment (#455)

**Supersedes in part:** §3 "Bumps the three `.codex-plugin` manifests in lockstep" and the `Consequences`
sentence "the three-codex-manifest lockstep bump".

**Date:** 2026-06-14
**Issue:** #455

### What changed

**Codex manifests now bump in their own independent series (not forced to the root version).**
`--cut --version X.Y.Z` no longer sets the three `.codex-plugin` manifests to the root version X.Y.Z.
Instead it resolves a Codex target version in the Codex series (root `5.x` ↔ Codex `3.x`, per #193):

- **Default (derive-by-kind):** applies the same bump-KIND as root to the current Codex baseline
  (e.g. root minor `5.16.0→5.17.0` ⇒ Codex minor `3.16.0→3.17.0`; root major ⇒ Codex major).
- **Override:** `--codex-version A.B.C` sets the Codex version explicitly.

**Two new pre-mutation typed refusals (fire before any content is written):**

- `codex_version_underivable` — no last root tag exists and no `--codex-version` override was provided;
  cannot determine the bump kind.
- `non_monotonic_codex_version` — the resolved Codex version is not strictly greater than the Codex
  baseline. Envelope fields: `requested_codex_version`, `codex_baseline`.

**Crash-resume idempotency via `codex_resolution` step receipt.** The resolved Codex version is
persisted to the step-receipt JSONL (step `"codex_resolution"`) after all guards pass and before any
content mutation. A resume after the Codex manifests are already bumped reuses the persisted value and
skips re-derive and the monotonic re-guard (avoiding two failure faces: an incorrect re-derivation
against the already-bumped baseline, and a monotonic guard that permanently refuses the explicit target
once the baseline has advanced).

**Success envelope additions:** `codex_version` (the resolved Codex version string) and
`codex_version_source` (`"derived"` or `"explicit"`).

**`.claude-plugin` manifests and README Claude-install lines are now bumped (previously missing).**
The 2 `.claude-plugin` manifests (`plugins/kaola-workflow-{gitlab,gitea}/.claude-plugin/plugin.json`)
are bumped to the ROOT version (not the Codex version). The README "Claude Code command install,
`<edition>` edition" lines are also updated to the root version. The README Codex manifest version
lines continue to be updated to the Codex version. Both corrections were absent from the original §3
implementation, leaving those surfaces stale on every release.

### What is unchanged

The census (two Claude forge manifests, three Codex manifests) remains correct. The `lockstep_violation`
precondition (all three Codex manifests must agree with each other before cutting) is unchanged — #455
only changes the bump *target*, not the coherence requirement. All other §3 decisions remain in force.
