## Context

The worktree sink choreography — merging a worktree branch back to main and closing the associated
issues — required an operator to execute approximately eight discrete steps in order:

1. Stash staged `.roadmap/` files.
2. Push the worktree branch upstream.
3. Merge the branch into main with `--no-ff`.
4. Sync and remove the worktree.
5. Run `cmdFinalize` internals (archive commit, ledger close).
6. Close the GitHub issues.
7. Pop the roadmap stash.
8. Commit the archive result and push main.

A crash between any two steps left the operator with no documented, safe resume path. Worse, certain
steps are NOT idempotent in their raw form: a second `git push` on an already-pushed ref, or a second
`gh issue close` on an already-closed issue, silently succeeds or noisily errors depending on forge
state, but in neither case does the operator know which prior steps completed. The only documented
recovery strategy was "start over" — which risks double-applying mutations that already landed.

The root cause is that atomicity and resumability were entirely the operator's responsibility, with no
durable record of progress. The #296 run lessons established the per-step receipt pattern as the
correct generalization for this class of multi-step transaction. This decision applies that pattern to
the worktree sink.

## Decision

### 1. `--sink` mode in `kaola-workflow-sink-merge.js`

A new `--sink` flag drives the full sink transaction from a single invocation. The script accepts:

- `--project <name>` — the project slug (used to locate state and `.roadmap/` files).
- `--branch <name>` — the worktree branch to sink.
- `--issues <N,M,...>` — comma-separated issue numbers belonging to this sink.
- `--worktree <path>` — absolute path to the worktree directory.
- `--receipt <path>` — (optional) explicit path for the receipt file; defaults to
  `kaola-workflow/<project>/.cache/sink-receipt.json`.

### 2. `sink_blocked` preflight

Before any mutation, `--sink` runs a preflight scan:

- Reads `git status --porcelain` in the main repo.
- Any dirty entry that is NOT in the auto-stashable set (see part 3) causes an immediate
  `sink_blocked` refusal. The refusal message lists the exact dirty paths so the operator can resolve
  them without guessing.
- The preflight runs BEFORE the receipt is created or any step is attempted. No partial state is left
  if the preflight fires.

This prevents foreign-project WIP (another issue's uncommitted edits, staged files from a concurrent
worktree, etc.) from contaminating the merge commit.

### 3. Auto-stash of `.roadmap/` issue files

The preflight recognises one category of "expected dirty" entries: staged or unstaged files matching
`kaola-workflow/.roadmap/issue-N.md` where `N` is in the `--issues` set. These are the per-issue
roadmap source files that `cmdFinalize` normally closes. They are auto-stashed before the merge and
auto-restored after the archive commit. No other files are auto-stashed; all others trigger
`sink_blocked`.

### 4. `sink-receipt.json` — crash-safe per-step record

```json
{
  "schemaVersion": 1,
  "project": "<project>",
  "branch": "<branch>",
  "issues": [429, 434],
  "startedAt": "<ISO timestamp>",
  "steps": {
    "stash_roadmap":      { "state": "done", "completedAt": "..." },
    "push_upstream":      { "state": "done", "completedAt": "..." },
    "merge_no_ff":        { "state": "done", "completedAt": "..." },
    "worktree_sync":      { "state": "done", "completedAt": "..." },
    "finalize_internals": { "state": "done", "completedAt": "..." },
    "close_issues":       { "state": "done", "completedAt": "..." },
    "stash_restore":      { "state": "done", "completedAt": "..." },
    "archive_commit":     { "state": "done", "completedAt": "..." },
    "push_main":          { "state": "done", "completedAt": "..." }
  }
}
```

The receipt is written with an atomic temp-file + rename: the script writes to
`<receipt>.tmp`, then `fs.renameSync` — so the file is never partially written. Each step's entry
moves from absent (not started) to `{ state: 'done', completedAt }` only when the step completes
successfully. A crash mid-step leaves the prior steps' `done` entries intact and the crashing step's
entry absent.

On re-invocation, `--sink` reads the receipt and **skips** any step whose entry is already
`state: 'done'`. This makes the transaction idempotent: re-running after a crash resumes from the
first incomplete step without re-applying completed mutations.

### 5. Nine sequenced steps and their re-entrancy contracts

| Step | Action | Re-entrancy |
|---|---|---|
| `stash_roadmap` | `git stash push -- kaola-workflow/.roadmap/issue-{N}.md ...` for issues in set | Skip if done; stash is pop'd later |
| `push_upstream` | `git push origin <branch>` on the worktree branch | Tolerate already-pushed: if remote ref equals local HEAD, treat as done |
| `merge_no_ff` | `git merge --no-ff <branch>` into main | Skip if merge commit already in log |
| `worktree_sync` | `git worktree remove --force <path>` + prune | Skip if worktree already absent |
| `finalize_internals` | Shell `cmdFinalize` internals: archive commit, ledger close | Idempotent: archive is a no-op if already archived; ledger close skips if already closed |
| `close_issues` | For each issue N, call `probeIssueClosed(N)` first; close only if open | Idempotent per-issue via probe |
| `stash_restore` | `git stash pop` the roadmap stash | Skip if stash absent (was never pushed, or already popped) |
| `archive_commit` | `git commit -m "chore: archive bundle-<issues>"` for the roadmap closure files | Skip if HEAD commit message matches sentinel |
| `push_main` | `git push origin main` | Tolerate already-pushed: remote HEAD equals local HEAD → treat as done |

`close_issues` calls `probeIssueClosed(N)` (a lightweight `gh issue view --json state` call) before
each close, so a forge-level close that happened via an earlier run does not become an error.

`push_upstream` and `push_main` use `git rev-parse origin/<ref>` to probe before pushing. If the
remote ref equals the local HEAD, the push is a no-op and the step is marked done without a network
write.

### 6. Typed refusals from the `--sink` mode

Refusals join the existing emit envelope with `result: 'refuse'` and a `reason` field:

- `sink_blocked` — preflight found dirty paths outside the auto-stash set; `paths` lists them.
- `sink_receipt_corrupt` — the receipt file exists but is not valid JSON or is missing required
  fields; the operator must remove it manually to restart from scratch.
- `step_failed` — a step returned a non-zero exit or threw; `step` names which one; `detail` carries
  the stderr/message. The receipt remains in its partially-done state for crash-resume.

### 7. `cmdFinalize` routes to `--sink`

`cmdFinalize` (in `kaola-workflow-claim.js`) detects the worktree case (the `--worktree` flag or the
presence of a worktree path in workflow-state) and delegates the full sink transaction to
`kaola-workflow-sink-merge.js --sink`. The operator-facing command (`/kaola-workflow-finalize`) and
the finalize command documentation reference `--sink` as the canonical worktree completion mechanism.

## Consequences

**For operators:**

- The hand-described 8-step choreography is replaced by a single `--sink` invocation. There is no
  manual step sequencing, no need to remember which git commands tolerate re-runs, and no ambiguity
  about which steps completed after a crash.
- A crash mid-sink is recoverable: re-invoke `--sink` with the same arguments; the receipt drives
  resume from the first incomplete step.

**For foreign-WIP safety:**

- The `sink_blocked` preflight with exact-path listing makes contamination-by-concurrent-WIP
  explicit and actionable. The operator sees exactly which files are blocking before any mutation
  occurs.

**For the finalize command docs and `docs/api.md`:**

- The sink-receipt schema, the nine-step sequence, the re-entrancy contracts for
  `push_upstream`/`push_main`/`close_issues`, and the typed refusals are documented under
  Sink Transaction in `docs/api.md`.
- The finalize command body references `--sink` as the worktree completion path and notes the
  receipt location.

**For the test suite:**

- Walkthrough scenarios cover: clean-sink happy path (receipt ends all `done`); crash-resume
  (receipt with two steps done, re-run completes); `sink_blocked` preflight fires on foreign-dirty
  files; `push_upstream` and `push_main` tolerate already-pushed; `close_issues` tolerate
  already-closed via `probeIssueClosed`.
