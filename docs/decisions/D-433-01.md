# D-433-01. Open-time evidence contract: single-source role-token registry + open-next seeding + nonce rotation

Date: 2026-06-13
Status: Accepted
Issue: #433
Related: `scripts/kaola-workflow-plan-validator.js` (`ROLE_TOKEN_REGISTRY` export, evidence-shape check),
         `scripts/kaola-workflow-adaptive-node.js` (`open-next`, `open-ready`, `reopen-node` — evidence seeding + nonce rotation),
         `commands/kaola-workflow-plan-run.md` + the 3 Codex SKILL packs (dispatch-contract prose — the SIX-surface set, #400),
         `docs/api.md` § Adaptive Plan Validation (role-token registry, opened payload shape),
         D-424-01 (barrier attribution — sibling fix; the `.cache/{node-id}.md` evidence path the seed writes into),
         Issue #271 / the adaptive node evidence-path lesson (project-local `.cache/{node-id}.md`, not repo-root)

## Context

After `open-next`, the orchestrator had to **hand-craft** `.cache/{node-id}.md` evidence files from
memory and edit them by trial until the gate passed. The schema of "which tokens does each role
require" lived only in prose (scattered across the role agent profiles and the plan-run command), and
the validator's evidence-shape checker had its own ad-hoc copy of those requirements. Two sources of
truth that can drift, plus a manual authoring loop, produce three recurring failure modes:

- **Drift.** The checker's required-token list and the prose contract diverge; an agent fills the
  tokens the prose names, but the checker wants a different set (e.g. a renamed `verdict` token), so
  the gate rejects correct work.
- **Manual toil and typos.** Hand-crafting the binding header (`evidence-binding: {node-id} {nonce}`)
  is error-prone — a mistyped nonce or node-id silently breaks the binding the barrier relies on.
- **Stale-evidence rebinding.** On a `reopen-node` (reset to `pending`, reopen with a NEW nonce), a
  pre-existing `.cache/{node-id}.md` from the prior attempt still carries the OLD `evidence-binding`
  line. If nothing rewrites it, stale evidence from the abandoned attempt can bind to the fresh open
  and pass the gate vacuously.

The fix makes the role-token requirements a single exported source, seeds the evidence skeleton at
open time, surfaces the expected tokens in the `opened` payload, and rotates the nonce in the seeded
file on reopen.

## Decision

### 1. Single-source role-token registry (exported from the plan-validator)

The role-token requirements move out of ad-hoc checker code into ONE exported map in
`plan-validator.js`:

```js
const ROLE_TOKEN_REGISTRY = {
  'tdd-guide':            ['evidence-binding', 'RED', 'GREEN'],
  'implementer':         ['evidence-binding', 'non_tdd_reason', 'regression-green|build-green|smoke-integration'],
  'code-reviewer':       ['evidence-binding', 'verdict', 'findings_blocking'],
  'security-reviewer':   ['evidence-binding', 'verdict', 'findings_blocking'],
  'adversarial-verifier':['evidence-binding', 'verdict'],
  'doc-updater':         ['evidence-binding'],
  'main-session-gate':   ['evidence-binding', 'verdict', 'findings_blocking'],
};
```

Each role maps to its required **token classes**. A class containing `|` is an alternation: the
implementer's `regression-green|build-green|smoke-integration` is satisfied by ANY one of the three
verification-tier tokens (the #359 tier vocabulary). The validator's evidence-shape checker reads
THIS map — there is no second copy. The map is `module.exports`-ed so `adaptive-node.js` imports the
same object for seeding (part 2). One source drives both the writer (seed) and the reader (gate),
eliminating drift.

### 2. Open-time evidence seeding (in `adaptive-node.js`)

When `adaptive-node.js` runs `open-next` (or `open-ready`), immediately after the ledger flips the
node to `in_progress` it creates the seeded `.cache/{node-id}.md` (project-local `.cache/`, the path
the barrier resolves — issue #271, NOT repo-root):

- **Line 1** is the binding header: `evidence-binding: {node-id} {nonce}` — exactly the node id and
  the nonce just minted/reused for this open.
- **Subsequent lines** are stub placeholders, one block per required token class from
  `ROLE_TOKEN_REGISTRY[role]`, each with an HTML-comment hint and an empty token line to fill.

Example — a `tdd-guide` node:

```
evidence-binding: n2-validator ab68f963be9d
<!-- RED: paste failing test output here -->
RED: 
<!-- GREEN: paste passing test output here -->
GREEN: 
```

Example — an `implementer` node:

```
evidence-binding: n1-design ab68f963be9d
non_tdd_reason: <reason this node has no natural failing unit test>
<!-- build-green|regression-green|smoke-integration -->
build-green: 
```

For an alternation class, the seed emits ONE representative stub (e.g. `build-green:`) plus the
comment naming all alternatives, so the agent knows it may substitute `regression-green` or
`smoke-integration`. The framework writes line 1; the agent never does (part 5).

### 3. `opened` payload carries evidence metadata

`open-next` and `open-ready` return payloads gain:

- `evidence_file`: `.cache/{node-id}.md` — the seeded path (so the dispatcher hands the exact path to
  the role agent).
- `required_tokens`: the `ROLE_TOKEN_REGISTRY[role]` list for this node's role (so the dispatcher and
  the agent both know the expected token classes without re-deriving them).

These are additive fields; existing `opened` consumers are unaffected.

### 4. Re-nonce propagation (nonce rotation on reopen)

Two open paths, two nonce behaviors:

- **Idempotent re-open** (`open-next --node-id <id>` for an ALREADY-`in_progress` node, e.g.
  crash-resume): the nonce is REUSED. The node was never reset, so its binding is still valid; the
  seeded file is left as-is (the agent's in-progress evidence is preserved). Reported as
  `nonce_rotated: false`.
- **Reopen** (`reopen-node`: reset the node to `pending`, then reopen): a NEW nonce is generated, AND
  the seeded evidence file's FIRST LINE is rewritten to `evidence-binding: {node-id} {new-nonce}`.
  This is **nonce rotation**. It invalidates any stale evidence from the prior attempt — the old
  binding line no longer matches the new open's nonce, so leftover token content from the abandoned
  attempt cannot bind to the fresh open and pass vacuously. The rewrite touches ONLY line 1; any
  scaffold below is left intact (the agent re-fills it). Reported as `nonce_rotated: true`.

Nonce rotation pairs with the D-424-01 provenance log: the reopen's `open-next` event records the new
nonce, so the rotation is auditable.

### 5. Dispatch-contract prose (the SIX-surface set, #400)

The plan-run command body and the three Codex SKILL packs (plus the two forge-codex SKILL packs — the
full six-surface routing set, #400) gain the open-time dispatch contract. After `open-next`, the
evidence skeleton already exists, so the dispatched role agent:

- Reads the seeded `.cache/{node-id}.md` to learn the expected tokens (and reads `required_tokens`
  from the `opened` payload).
- Fills the token stubs with real evidence from its work.
- NEVER modifies the `evidence-binding:` header line — it is framework-owned; editing it breaks the
  barrier binding.
- Appends any additional findings/notes AFTER the required tokens (the gate checks for token
  PRESENCE; trailing prose is allowed).

This prose is part of the adaptive/routing surface, so it propagates to all six surfaces and is
machine-enforced by the route-reachability contract (#400) — a four-of-six propagation gap fails the
contract.

## Consequences

**For the orchestrator:**

- No more hand-crafting `.cache/{node-id}.md` from memory. `open-next`/`open-ready` seed the file with
  a correct binding header and per-role stubs; the dispatcher forwards `evidence_file` +
  `required_tokens` to the role agent. The manual edit-until-it-passes loop is gone.

**For role agents (the SIX surfaces):**

- A seeded skeleton with the exact required token classes exists at dispatch. Agents fill stubs rather
  than guess the schema, and they must not touch line 1.

**For the validator:**

- `ROLE_TOKEN_REGISTRY` is the single exported source for both the evidence-shape gate and the
  open-time seed. Adding/renaming a role's required tokens is a one-place edit; the seed and the gate
  stay in lockstep by construction.

**For crash-resume vs. reopen:**

- Idempotent re-open preserves in-progress evidence (`nonce_rotated: false`). `reopen-node` rotates
  the nonce and rewrites line 1 (`nonce_rotated: true`), structurally preventing stale-attempt
  evidence from binding to the fresh open. The `nonce_rotated` flag is returned for the orchestrator
  to log/branch on.

**For `docs/api.md`:**

- The `ROLE_TOKEN_REGISTRY` shape, the alternation-class semantics, the seeded evidence-file layout,
  the additive `opened` payload fields (`evidence_file`, `required_tokens`), and the `nonce_rotated`
  flag are documented under Adaptive Plan Validation.

**For cross-edition parity:**

- The registry + evidence-shape check live in the generated `plan-validator.js` aggregator; the
  seeding + nonce rotation live in the generated `adaptive-node.js` aggregator. Both are
  `GENERATED_AGGREGATORS` members (D-431-01 coupling), so each four-file group is owned together and
  all four chains must be green before finalize. The dispatch prose reaches all six routing surfaces
  (#400).
