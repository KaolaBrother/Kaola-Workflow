# D-607-01. Three-layer write fence around a main-session-gate acceptance window

Date: 2026-07-03
Status: Accepted
Issue: #607
Related: D-419-01 (Part 1 вЂ” narrows [INV-2], the hard byte-identity invariant that `open-next`
             "MUST NOT begin writing a `running-set.json`"; see the narrowing note below); #596
             (the kind-consumer audit here reuses the same running-set-consumer verification
             discipline that decision applied when graduating write nodes onto the speculative
             kernel); #400 (the six-surface routing-prose propagation rule the planner-authoring
             and plan-run guidance follow)

## Context

A `main-session-gate` is a non-delegable acceptance check вЂ” the main session itself performs a
confirmation an automated reviewer cannot (a GPU/visual check, a human sign-off) and records
`verdict: pass|fail` into its `.cache` evidence. It is read-only by grammar: no declared write
set, never a subagent dispatch, excluded from fan-out. Nothing, however, previously stopped the
gate itself вЂ” or a stray tool call issued during its open window вЂ” from authoring or deleting a
product file in the parent worktree. The close-time per-node barrier (`barrierCheck`, a
baseline-to-now tree diff) is blind to a write-then-delete that nets to zero inside that window:
a gate that wrote a probe file, ran it, and cleaned up leaves no diff for the barrier to see, so
the authorship is invisible to every downstream mechanism that depends on the write-set contract
(attribution sweep, review scope, the disjointness proof over subsequent nodes).

## Decision

**Close the gap with three independent layers, so no single layer has to be perfect.**

1. **Layer 1 вЂ” upstream provisioning (prose-only, no runtime code).** A gate never authors or
   deletes files. Any instrumentation a gate's acceptance check needs вЂ” a probe script, build
   wiring to make a probe runnable вЂ” is authored by an UPSTREAM WRITER node, inside that node's
   own declared write set, before the gate ever opens. The plan states the durability decision for
   that instrumentation: durable (committed, kept), env-gated (kept but inert without a flag), or
   ephemeral (a downstream node owns its deletion) вЂ” never left implicit. Out-of-repo scratch
   (a temp dir, the scratchpad) stays legal for the gate's own transient use and is not
   "instrumentation" in this sense. This is taught on the seven planner-authoring surfaces (the
   `workflow-planner` profile Г—3 editions plus the `kaola-workflow-adapt` SKILL Г—3 forges) and
   referenced, not re-derived, on the six plan-run routing surfaces (#400) next to the existing
   gate-non-delegability guidance.

2. **Layer 2 вЂ” runtime write fence.** `open-next` and the `close-and-open-next` fused advance now
   record an opened `main-session-gate` into `running-set.json` as a minimal entry вЂ”
   `{ id, role, kind: 'gate', declared_write_set, model, baseline: 'recorded', openedAt? }` вЂ”
   via a new `recordGateInRunningSet` helper, called AFTER the ledger flip to `in_progress` (a
   crash before the flip leaves no phantom; a crash after leaves an `in_progress` gate that
   `reconcile-running-set` treats as a live, non-`opening` member and no-ops on вЂ” see the sticky-
   fence consequence below). The write-lane hook (`hooks/kaola-workflow-write-lane.sh`) gains a
   new rule (c), evaluated FIRST (before the existing lane-containment rules a/b, so it fires even
   when `KAOLA_LANE_CONTAINMENT` is off): while any `kind:'gate'` node is open, a `Write`/`Edit`
   landing INSIDE the active worktree, OUTSIDE the workflow bands (`kaola-workflow/`, any
   `.cache/`), is DENIED (exit 2) unless it is under the `.kw/` band (member worktrees / legs /
   co-open speculative work), inside a member worktree (governed by rule (a) instead), or under a
   live co-open writer's own declared lane (so a #596 speculative write co-opened behind the same
   gate is unaffected). The switch is DEFAULT-ON with an explicit `KAOLA_GATE_WINDOW_FENCE=0`
   opt-out вЂ” the inverse polarity of the pre-existing `KAOLA_LANE_CONTAINMENT` arm (fail-closed
   default OFF), because a gate write is a narrower, higher-confidence signal (there is no
   legitimate reason for an out-of-band write during a read-only acceptance check) than the
   general lane-containment heuristic.

3. **Layer 3 вЂ” close-time evidence token.** A `main-session-gate`'s `.cache` evidence must carry a
   column-0 `instrumentation: none | <node-id>` token, checked by `checkEvidenceShape` alongside
   the existing `verdict: pass|fail` shape check вЂ” its absence is refused the same way a missing
   verdict is. `none` attests the gate ran no in-worktree instrumentation; `<node-id>` names the
   Layer-1 upstream writer that authored whatever probe the gate ran, so the probe's own
   correctness is covered by that writer's post-dominating `code-reviewer` gate rather than
   self-reviewed by the verdict-bearing gate itself.

### The kept state channel вЂ” kind-consumer audit conclusion

Reusing `running-set.json` (rather than inventing a parallel gate-window marker file) was
verified safe by auditing every existing consumer of the manifest before shipping: the
excl-scheduler guard's `serialLive`/`runningSetLive` derivation, `open-ready`'s read-slot math,
`reconcile-running-set`'s roll-forward budget, `liveHasWrite`, `selectSpeculativeWriteGroup`, and
the group-close/removal paths. Every write-oriented count (`liveHasWrite`,
`selectSpeculativeWriteGroup`, the `open-ready` slot base, and the `reconcile-running-set`
`liveStable` budget) now explicitly excludes `kind:'gate'`, so a live gate is invisible to write
concurrency accounting; a lone gate is classified a valid running set at `orient` (not an orphan);
and `close-and-open-next`'s own close of the gate is unaffected because that guard prologue does
not include the excl-scheduler arm. The one behavioral delta found (an independent non-speculative
writer that is ready alongside a live gate now returns `write_awaits_drain` instead of co-opening,
because the read-slot-math `liveNodes.length === 0` shortcut is no longer vacuously true) is a
pure makespan effect in a narrow, currently-unreachable DAG shape вЂ” the orchestrator drives a gate
to completion serially in every reachable plan today вЂ” not a correctness change.

### Narrowed Layer 3 interpretation

The `instrumentation: <node-id>` token carries only a node id, not the instrumentation's own file
paths, so "the named node's declared write set covers the instrumentation" cannot be mechanically
resolved to specific paths from the token alone. The enforced interpretation is narrower but
mechanical: the named node must EXIST in the ledger AND be a WRITER (a non-empty declared write
set, via the existing `isReadOnlyNode` check) вЂ” not a deeper "the specific probe path is inside
that node's declared set" proof. This check runs only when the caller supplies `opts.ledgerNodes`
(all three real close paths вЂ” `close-and-open-next`, `close-node`, and `record-evidence --verify`
вЂ” do); the token-presence gate fires unconditionally regardless.

### Honest residual holes

Disclosed rather than silently accepted:

- **Bash-mediated writes are not covered.** The hook intercepts `Write`/`Edit` tool calls only вЂ”
  a Bash-issued `echo > file` or `sed -i` during the gate window is not fenced by Layer 2. The
  pre-existing close-time barrier accounting (per-node `--barrier-check` own-lane allowlist, seal
  vacuity) remains the backstop for this class, unchanged by this decision вЂ” it was already the
  documented boundary for the lane-containment hook, and this decision does not widen it.
- **The `.cache/`-anywhere carve-out is pre-existing, not a regression.** `inWorkflowBand` allows
  any path containing `/.cache/` (the #376 band semantics), so a write into an arbitrary
  `.cache/`-named directory escapes both the lane hook and the gate fence. This predates #607 and
  is out of this decision's scope to widen.
- **A crash mid-gate leaves the fence STICKY, intentionally.** `reconcile-running-set` hits the
  `not_opening` early-return on a lone, non-`opening` gate entry and preserves it rather than
  clearing it вЂ” every in-worktree product write stays fenced repo-wide until the run resumes and
  closes the gate (or the operator sets `KAOLA_GATE_WINDOW_FENCE=0`, or manually removes the
  manifest entry). This is a deliberate fail-closed tripwire matching the issue's own crash-residue
  motivation, not an oversight; it is opt-out-able for exactly this reason.

### Narrow exception to [INV-2] (D-419-01)

D-419-01's Part 1 states a HARD invariant: "with `KAOLA_LANE_CONTAINMENT` off + no running-set +
вЂ¦ every guard layer is vacuously-pass and the serial path is byte-identical to pre-#383 behaviorвЂ¦
`open-next` MUST NOT begin writing a `running-set.json`." Layer 2 above makes `open-next` write a
`running-set.json` precisely when вЂ” and only when вЂ” the node it opens is a `main-session-gate`.
This decision explicitly and narrowly amends [INV-2]'s scope rather than silently violating it:
the invariant's PURPOSE вЂ” write-frontier concurrency accounting stays byte-identical when nothing
is co-opening вЂ” is preserved (a gate carries no write set and is excluded from every write count
audited above); only the invariant's literal text, which did not anticipate a non-write state
channel riding the same file, is narrowed. `docs/architecture.md` records this narrowing next to
its restatement of [INV-2]. Every other write-node code path is untouched, so serial write-node
behavior stays exactly as [INV-2] describes.

## Consequences

- A gate window is now a runtime-enforced read-only zone for in-worktree out-of-band writes by
  default, closing the write-then-delete blind spot the close-time barrier could not see.
- Operators who need to disable the fence (a legitimate workflow that writes during a gate window
  for a reason the fence does not anticipate) have a single, named opt-out
  (`KAOLA_GATE_WINDOW_FENCE=0`), not a silent bypass.
- A crash mid-gate degrades toward MORE restriction (the sticky fence), never less вЂ” consistent
  with this repo's fail-closed posture on crash-repair surfaces.
- Cross-edition (#307): the running-set state channel, the hook rule, and the evidence-token check
  touch `scripts/kaola-workflow-adaptive-node.js` and `hooks/kaola-workflow-write-lane.sh` in all
  four editions; the planner-authoring and plan-run prose touch 13 surfaces (7 planner-authoring +
  6 plan-run, per #400); all four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains plus
  `scripts/simulate-workflow-walkthrough.js` must stay green.

## Alternatives considered

- **A dedicated gate-window marker file instead of reusing `running-set.json`.** Rejected: it
  would require every running-set consumer (the excl-scheduler guard, the slot math, reconcile,
  the write-oriented counts) to ALSO probe a second file, doubling the surface for a
  kind-consumer-style audit to miss a case; reusing the existing manifest with a discriminated
  `kind` field keeps exactly one state channel for every "what is currently open" question.
- **Make the gate-window fence advisory (warn, not deny), matching `KAOLA_LANE_CONTAINMENT`'s
  fail-open posture.** Rejected: the risk profile is different вЂ” there is no legitimate in-worktree
  out-of-band write during a read-only acceptance window, so a deny-by-default costs nothing
  correct while a warn-only posture would have caught nothing in the motivating incident.
- **Resolve the write-set-coverage half of Layer 3 by parsing the instrumentation's own paths out
  of the gate's evidence.** Rejected for v1: the token format carries only a node id; requiring a
  path list would burden every gate author with declaring paths the upstream writer already
  declares in its own write set, duplicating information the ledger already has. The narrower
  "named node is a writer" check is accepted as sufficient given the disclosed limitation above.
- **Amend D-419-01's [INV-2] text directly instead of narrowing it from this decision record.**
  Rejected: D-419-01 documents a decision already shipped and tested; rewriting its historical
  invariant text would erase the record of what was originally guaranteed. Narrowing it from here,
  with an explicit cross-reference, preserves both records' honesty.
