# D-517-01. Post-push keep-open reopen: probe and reopen after `push_main` to guarantee a keep-open issue stays open against commit-keyword auto-close

**Date:** 2026-06-17
**Status:** Accepted
**Issue:** #517
**Related:** #336 (`issue_action: comment_keep_open` keep-open terminal; the keep-open contract),
            #497 (hard `push_main` / `closure` failure emit fix — the inline error-path complement),
            `scripts/kaola-workflow-sink-merge.js` (the `push_main` step and the new verification),
            `docs/workflow-state-contract.md` § Workflow State Fields (`issue_action`),
            `docs/api.md` § Closure Contract (`remote_issue_closed` enum).

## Context

A keep-open run (`issue_action: comment_keep_open`) intentionally leaves the issue open: the
`sink-merge` closure step is bypassed, and instead a comment is posted to the issue. However,
GitHub's commit-keyword auto-close operates **server-side**, after the push: if any commit or
PR message in the merged set contains a close/fix/resolve keyword followed by the issue number,
GitHub automatically closes the issue regardless of the client-side workflow action.

This is a trigger outside workflow control — the workflow correctly bypasses its own close step,
but the forge's own machinery can re-close the issue after the push anyway. Before #517 there
was no detection of or recovery from this race: after `push_main` succeeded the workflow
recorded `remote_issue_closed: kept_open` (the bypassed-close sentinel) and exited, never
probing whether the issue had actually been auto-closed by the forge.

The defect is single-issue scoped: a keep-open run operates on one issue, and the auto-close
trigger is per-issue commit-keyword matching. No bundle-path analog is required.

## Decision

**Add a keep-open verification step immediately after `push_main` in `sink-merge`.** The
verification:

1. **Probes live issue state** using the same `probeIssueState` helper already used on the
   `watch-pr`/`watch-mr` merged lane.
2. **If `keepIssueOpen` is true and the probe finds the issue closed**, reopens it via the
   forge API and records `remote_issue_closed: reopened_after_autoclose` in the receipt.
3. **If the probe confirms the issue is still open**, records `remote_issue_closed: kept_open`
   (unchanged behavior for the nominal case).

**Why post-push probe, not pre-push guard:** The auto-close trigger is a server-side reaction
to the pushed commit payload. There is no pre-push way to predict or prevent it without
parsing every commit message in the push set against the forge's keyword rules — an
over-engineering for a case that is uncommon and, when it occurs, is deterministically
correctable. A post-push probe is the minimal, reliable fix: the issue state after push
reflects the auto-close if it happened, so the probe result is authoritative.

**Why reopen, not suppress:** A keep-open run signals an explicit owner intent to keep the
issue active (a partial close, a multi-slice run, or a deliberate hold). Auto-close defeating
that intent silently is a correctness violation. Reopening deterministically restores the
stated intent.

**The `kept_open` sentinel is preserved** for the nominal path where no auto-close occurred:
`remote_issue_closed: kept_open` continues to mean the workflow bypassed closure and confirmed
the issue remained open. `reopened_after_autoclose` is additive, distinguishing the corrected
case from the nominal one so receipt auditing can identify runs where auto-close interference
was detected and reversed.

## Implementation

`scripts/kaola-workflow-sink-merge.js` (canonical + the Codex byte-twin + the two forge ports)
gains a post-`push_main` verification block on the keep-open lane. The block is conditional
on `keepIssueOpen` to leave the close lane byte-identical. The `remote_issue_closed:
reopened_after_autoclose` value is added to the receipt schema and the enum in `docs/api.md`.
Forge sink tests are extended to cover the auto-close detection and reopen path.

## Consequences

- A keep-open run is now durably resilient to commit-keyword auto-close: even if the forge
  auto-closes the issue post-push, the workflow detects and reverses it before reporting success.
- The `remote_issue_closed` field gains a new observable value `reopened_after_autoclose` that
  receipt auditing can use to identify corrected auto-close events.
- The nominal keep-open path (`kept_open`) is byte-identical: the probe only reopens when
  the issue is found closed, so a run with no auto-close interference is unchanged.
- Probe availability: if `probeIssueState` is unavailable (offline or API fault), the receipt
  records the probe result honestly; the reopen step is guarded to run only when the probe
  succeeds and returns a closed state.
