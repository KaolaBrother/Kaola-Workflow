# D-582-01. Codex per-node effort tiers are explicit per-spawn overrides or fail closed

**Date:** 2026-07-01
**Status:** Accepted
**Issue:** #582
**Related:** #451, #581

## Context

#451 retired the `<role>-max` profile matrix and moved Codex effort selection onto the adaptive
dispatch descriptor. At that point, only `model: opus` emitted an explicit
`codex_reasoning_effort`; `model: sonnet` and absent model tiers both inherited the parent/session
effort. #581 added direct per-spawn `reasoning_effort` and v2 task-name identity, but the `sonnet`
branch still did not produce an effective tier.

That left a contract gap: a frozen plan that explicitly declares `model: sonnet` could silently run at
the orchestrator's current effort instead of the plan's intended second-tier effort.

## Decision

- `dispatchEffort('opus')` emits `codex_reasoning_effort: "xhigh"` with source `planner_model`.
- `dispatchEffort('sonnet')` emits `codex_reasoning_effort: "high"` with source `planner_model`.
- Absent, blank, or unknown model values emit `codex_reasoning_effort: null` with source
  `role_default`; only this branch intentionally inherits the base profile/session default.
- Codex v2 task-name dispatch passes tiered nodes with `task_name`, `agent_type`,
  `fork_turns: "none"`, and direct `reasoning_effort`.
- Codex v1 tiered dispatch is not silently trusted. It must be used only after the installed runtime
  has been proved to honor per-spawn effort; otherwise the executor refuses before dispatch with
  `codex_effort_override_unavailable`.
- Base role TOMLs stay unpinned: they still omit `model_reasoning_effort` so the per-spawn override
  is the active tiering mechanism.

## Consequences

- Explicit `opus` and `sonnet` tiers are now both enforceable descriptor values.
- Silent inheritance is limited to absent/blank model tiers.
- Plan-run prose and Codex SKILL copies describe the same fail-closed runtime contract.
- Regression coverage pins both the descriptor mapping and the six plan-run dispatch surfaces.

## Live validation recipe

Descriptor tests prove the workflow emits the right request, but they do not prove the installed
Codex runtime honored it. To prove a local Codex install before allowing v1 fallback, run a read-only
probe from a Codex session and inspect runtime metadata:

1. Enable role-aware v2 task-name dispatch with the current Codex 0.144.1 compatibility posture:
   `multi_agent_v2 = { enabled = true, tool_namespace = "agents", hide_spawn_agent_metadata = false, non_code_mode_only = true }` (or the equivalent dotted table). The default `collaboration` namespace is server-reserved and cannot carry the visible role schema.
2. Spawn a tiny read-only child with `agent_type` set to any installed Kaola role, `task_name` set to
   a probe-safe name, `fork_turns: "none"`, and `reasoning_effort: "high"`.
3. Locate the spawned child session JSONL under `~/.codex/sessions/` and read its first
   `turn_context` record.
4. Treat the probe as pass only when that child `turn_context.effort` is exactly `high`. Repeat with
   `reasoning_effort: "xhigh"` when validating the `opus` tier.
5. Do not accept a child self-report or the parent-side descriptor alone as proof; if metadata is
   missing or records the parent effort instead, v1 tiered dispatch remains unavailable and must
   refuse with `codex_effort_override_unavailable`.
