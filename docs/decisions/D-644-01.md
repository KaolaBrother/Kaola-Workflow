# D-644-01. Close the two fenced residuals of the #641 read‚à•write co-open relaxation, plus a third seam surfaced in-run

Date: 2026-07-08
Status: Accepted
Issue: #644
Related: #641/D-641-01 (the relaxation this decision closes the deferred residuals of ‚Äî both A1
         and A2 were named there as known-fenced, non-blocking follow-ups); #622/D-622-01 (the
         `merge_awaits_read_drain` fence A1 extends); #642/D-642-01 (the sibling channel decision

## Context

#641 relaxed the scheduler's read‚à•write co-open serialization (G1/G2) in two slices ‚Äî a
leg-contained writer co-opening behind live reads, and an `observes: scratch` consent tier ‚Äî while
keeping the invariant those guards protect (closed-work observation) intact through the G4
`merge_awaits_read_drain` fence and several fail-closed preconditions. That run's own
`code-reviewer` and `adversarial-verifier` gates surfaced two residuals, both confirmed real but
fenced by other invariants, and both deliberately shipped non-blocking rather than fixed in that
run:

- **A1 (medium).** `liveReadsAtMerge` (the G4 merge fence) and the R1/R2b co-open preconditions
  counted `kind==='read'` only ‚Äî a live `kind:'gate'` (a `main-session-gate`) was invisible to both,
  so a `{main-session-gate ‚à• writer}` antichain could, in principle, R1-co-open or merge behind a
  live gate. Fenced at the time by single-orchestrator discipline (the gate's observation window is
  the orchestrator's own work window; no documented `orient`/resume path issues `open-ready` inside
  it), by the write-set-aware G3 (a production-set writer cannot antichain a gate at freeze, so only
  docs-band bytes could ever land this way), and by atomic close.
- **A2 (low).** The plan's `validation_test_consumes` widening (`testConsumedExtra`) was not
  threaded into `tryR2bLeglessCoopen`'s `scratchObservableWriteSet` call, so a self-hosting fork's
  extra test-consumed prose was not honored by the runtime R2b band check ‚Äî fork-only, bounded to a
  spurious refute by the scratch contract (no post-domination, so no laundering channel).

This bundle's plan named both as the fixes to make; its own `adversarial-verifier` gate, running
live-CLI reproductions of each claimed seam (the same lens that originally surfaced A1/A2 from
#641), additionally found a THIRD, previously-undocumented instance of the exact same class at an
adjacent seam that #644's original scope did not name: the fused advance inside
`close-and-open-next` opens the next `## Nodes` row with no live-gate check of its own at all.
Front-door reproduction: freezing `{writer, gate}` and closing the writer first opened the gate
correctly (closed-only); freezing the SAME antichain with the writer's `## Nodes` row preceding the
gate's row and closing a different upstream member instead opened the writer in_progress while the
gate was still live ‚Äî the outcome depended on table order, not on whether a gate was actually live.
Not a genuine invariant violation (the same fences that bounded A1 ‚Äî single-orchestrator discipline,
write-set-aware G3, atomic close ‚Äî bound this seam too), but real, and structurally the same defect
A1 already named at the other two seams.

## Decision

**A1 ‚Äî count `kind:'gate'` as a live observer at all three seams it governs, including the one
surfaced in-run:**

1. `liveReadsAtMerge` now filters `n.kind === 'read' || n.kind === 'gate'` ‚Äî the last-member merge
   holds while a gate is live, keeping the typed reason `merge_awaits_read_drain` stable (a live
   gate is treated as a live observer for the merge fence, the same semantics a live read already
   had).
2. The relaxed `write_awaits_drain` else-branch holds at its very first check ‚Äî
   `if ((liveNodes||[]).some(n => n.kind === 'gate')) return holdDrain('gate_live')` ‚Äî before any of
   the existing `legCoupled`/`lane_group_live`/`parent_dirty` checks, so a live gate always wins the
   race regardless of which other precondition would otherwise have passed.
3. `tryR2bLeglessCoopen`'s co-open precondition gets the same defensive top guard, independently
   returning `null` (hold) when any live member is a gate.
4. **In-run addition (the third seam):** the fused advance in `close-and-open-next` now checks for a
   live `kind:'gate'` member before opening the next `## Nodes` row, exactly mirroring the other
   three seams ‚Äî order-independent: whether the writer's or the gate's row comes first in
   `## Nodes` no longer changes the outcome. Fixed in the same run the adversary found it, per the
   run's zero-follow-up mandate; the gate re-ran after the fix landed.

Serial-fallback byte-identity is preserved throughout: with no running set (or `KAOLA_PARALLEL_WRITES=0`),
none of the four `gate_live` branches are ever reached, and no `serialDegradeReason` field is
emitted ‚Äî a lone write still opens exactly as it did before this decision.

**A2 ‚Äî thread `testConsumedExtra` into `tryR2bLeglessCoopen`'s scratch-observable check.**
`parseValidationTestConsumes(planContent)` (already exported from the validator) is read once
inside `tryR2bLeglessCoopen` and passed as `opts.testConsumedExtra` into the
`scratchObservableWriteSet` call ‚Äî a one-argument thread-through; `scratchObservableWriteSet`
already honored `opts.testConsumedExtra` and needed no change. This closes the fork-widening gap:
a plan declaring `validation_test_consumes: docs/fork-guide.md` now correctly widens the
R2b-legal band for that path, matching what the freeze-time `--parallel-safe` check already
honored.

## Consequences

- A `{main-session-gate ‚à• writer}` antichain now HOLDS at co-open, at the merge fence, and at the
  fused advance while the gate is live ‚Äî the same fail-closed hold, order-independent, at all four
  places a live-set member's `kind` is inspected for this purpose.
- A fork declaring `validation_test_consumes` for a path now gets the correct, widened R2b co-open
  legality for that path instead of a spurious `parent_dirty` hold.
- `kaola-workflow-adaptive-node.js` is a GENERATED aggregator; the canonical edits were regenerated
  to the codex byte mirror and both forge rename ports via `npm run sync:editions`
  (byte-/rename-identical ‚Äî no hand-porting).
- Tests (`test-adaptive-node.js`): the A1 co-open-hold (`{main-session-gate ‚à• doc-updater
  docs/api.md}`, `open-ready` returns `write_awaits_drain` + `serialDegradeReason: 'gate_live'`,
  writer not opened, plan byte-identical), the A1 merge-fence-hold (a live lane-group writer + a
  live gate ‚áí `close-node` refuses `merge_awaits_read_drain` while the gate is `in_progress`), the
  serial-fallback byte-identity control, the A2 fork-widening RED (a `docs/fork-guide.md` writer
  over a dirty parent held without the Meta line, co-opened legless with it) ‚Äî plus, added in-run,
  the fused-advance gate-hold pinned order-independently (both the writer-row-first and
  gate-row-first table orderings). All four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains verified green (run
  sequentially).

## Non-goals

- Does not relax or re-open the read‚à•write co-open decision itself (#641/D-641-01) ‚Äî this decision
  only closes gaps in that decision's own fencing.
- Does not add a new running-set `kind` value or change what `kind:'gate'` means anywhere else in
  the scheduler ‚Äî it only extends where an existing live-gate value is checked.
- Does not change `main-session-gate`'s non-delegable freeze-time contract (G3) ‚Äî the fixes here are
  entirely runtime scheduler checks.
