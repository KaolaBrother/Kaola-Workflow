**Part 4** (optional consent-gated speculative gate overlap) together, because both are behind a flag

in D-419-01. Parts 1 and 3 are recorded in D-419-01.

## Context

Two capabilities have been DESIGNED-but-dormant for some time:

1. **Disjoint write parallelism.** ADR 0008 excised the write-role batch member-worktree isolation
   machinery because the harness cannot force a dispatched subagent's CWD, so member subagents wrote
   to the PARENT worktree and `seal` refused `empty_member` after the work was done. ADR 0008 recorded
   a reintroduction condition: bring write isolation back only behind #376 (a `PreToolUse` `Write|Edit`
   containment hook, `KAOLA_LANE_CONTAINMENT`, fail-closed default false) plus #377 (the per-node
   running-set scheduler). Both #376 and #377 shipped. The write-lane hook
   (`hooks/kaola-workflow-write-lane.sh`) exists and self-exempts the open write node's own lane (#386
   architecture ii); the running-set scheduler (`open-ready` / `close-node`) exists. But the scheduler
   still hard-gates ALL write nodes to `write_node_exclusive`, and `KAOLA_LANE_CONTAINMENT` defaults
   FALSE and is read only by the hook script, never by the Node aggregators. The capability is wired
   but inert.

2. **The validator already computes disjointness — and throws it away.** At freeze time the validator
   classifies every antichain write pair (`plan-validator.js` L1028-1056): EXACT-file overlap ⟹ RED
   refuse; coarse / shared-infra overlap with a shared ancestor ⟹ ASK
   (`concurrentAmbiguousOverlap`); truly disjoint ⟹ silently pass. The third outcome (disjoint) is
   discarded after deciding pass/refuse. A survey grep confirms NO `parallel_safe` and no freeze-time
   `write_node_exclusive` annotation exist anywhere in the codebase today.

A separate efficiency gap is structural: a gate node (e.g. a `code-reviewer`) and its post-dominated
descendant always run strictly in sequence, even when the descendant could begin speculatively while
the gate runs. There is no mechanism to overlap a gate with discardable downstream work.

The OVERARCHING principle for both parts: the repo already CONTAINS every mechanism v3 needs; what is
missing is wiring + a stamp, not new machinery. The validator already computes disjointness; the
scheduler already has crash-safe two-phase opens, baselines, and longest-path priority; the guard
prologue already proves the three surfaces are mutually exclusive; the write-lane hook already enforces
containment. v3 is a CONSOLIDATION + ACTIVATION design that keeps the permanent serial fallback
byte-identical and lets every new capability degrade to today's behavior under its OFF condition.

## Decision

### Part 2 — Lane-attributed disjoint write parallelism (the #376 graduation)

**Decision (P2):** Graduate #376 by having the validator STAMP the disjointness verdict it already
computes onto write nodes at freeze time, and having `open-ready` lift `write_node_exclusive` ONLY for
a candidate whose declared write set is disjoint from EVERY currently-open write node per that stamp.
The entire capability is behind `KAOLA_LANE_CONTAINMENT`; with the flag OFF (the permanent default) it
is fully dormant and the serial-write fallback ([INV-6], D-419-01) holds.

**Freeze-time contract (the stamp).** For each WRITE-bearing node N,
`parallel_safe(N) = true` iff for EVERY other write-bearing node M that is in an ANTICHAIN with N
(neither reaches the other — the existing antichain test), `disjointWriteSets([N.writeSet,
M.writeSet]).verdict === 'green'`. Otherwise `parallel_safe(N) = false`. The check is PAIRWISE, not
global: a node is parallel_safe only relative to its concurrent peers; a node whose only overlaps are
with its own ancestors/descendants (ordered, never co-scheduled) is parallel_safe. Coarse-overlap
ASK-class pairs stamp `parallel_safe: false` (acking the plan authorizes the PLAN; it does not grant
write concurrency to a coarse-overlap pair). The stamp lives in `running-set.json`, recomputed at open
time from the frozen, hash-covered write sets — it does NOT enter the frozen plan body and does NOT
change `plan_hash` (a derived projection of already-hashed data must not re-enter the hash; identical
reasoning to D-419-01's `max_concurrent`).

**Runtime contract (`open-ready` use).** Replace the unconditional
`liveHasWrite ⟹ write_node_exclusive` gate with a lane-checked variant, ACTIVE ONLY when
`resolveLaneContainment(env) === true`. Flag OFF (default): behavior is IDENTICAL to today — any live
write node ⟹ `write_node_exclusive`, write nodes open alone, [INV-6] preserved verbatim. Flag ON: a
candidate write node W co-opens with currently-open write nodes iff
`disjointWriteSets([W.writeSet, ...each open write node's writeSet]).verdict === 'green'` against
EVERY open write node (recomputed, per [INV-9]). On overlap ⟹ the candidate is held with the EXISTING
`write_awaits_drain` reason — no new reason code for the hold.

**Part 2 invariants:**

- **[INV-8]** The stamp is DERIVED, never authored. The planner does NOT write `parallel_safe`; the
  validator computes it from declared write sets at freeze and stamps it. An author-supplied
  `parallel_safe` is ignored/overwritten (fail-closed: a hand-stamped `true` on overlapping sets must
  never grant concurrency). The existing RED-refuse on exact overlap is intact — a RED pair never
  reaches the stamp because freeze refuses first.
- **[INV-9]** `parallel_safe` is recomputed deterministically from the hash-covered write sets on
  EVERY read; a tampered annotation cannot grant concurrency (the scheduler recomputes, never trusts a
  persisted bool blindly). It does NOT enter the frozen plan body / `plan_hash`.
- **[INV-10]** The check is against ALL currently-open write nodes, transitively pairwise — open a
  write node only if it is green against the live write SET, not merely against one member. This is the
  same all-pairs guarantee the freeze-time stamp provides.
- **[INV-11] (the #283 non-empty-in-lane guard MUST survive the lane-attribution shift):** Under v3,
  write nodes still run in the PARENT worktree (per ADR 0008: the harness cannot force a subagent CWD;
  member-worktree isolation was EXCISED). So the #283 vacuity guard takes its post-#364 form, NOT the
  worktree form: at close (`close-node` step (b), the per-node `--barrier-check`), the barrier's
  own-lane allowlist IS the ground truth — a write node that produced no in-lane writes against a
  non-empty declared write set is caught by the barrier. The seal-time `git status --porcelain
  non-empty AND in-lane` check from the batch path is SUBSUMED by the per-node barrier (close-node runs
  the barrier per node); there is no member worktree to porcelain-check. The guard SURVIVES as: barrier
  allowlist (no out-of-lane write lands) + the write-lane hook (no out-of-lane write at write time,
  when flag ON). Two layers, both parent-side, both already present.
- **[INV-12]** The barrier remains the GROUND TRUTH (verbatim from ADR 0008 + the write-lane hook
  header "Hook = fast-fail containment; barrier = ground truth"). The hook is a fast-fail convenience;
  if the hook is somehow bypassed, the barrier still refuses the close.
- **[INV-13]** A Bash-mediated out-of-lane write CAN bypass the hook (the hook intercepts `Write|Edit`
  only) but CANNOT bypass the barrier (the barrier tree-diffs ACTUAL writes against the declared lane
  at close). The barrier is the SECURITY boundary; the hook is the ERGONOMICS boundary. This asymmetry
  is stated explicitly so no reader assumes the hook is a security control.
- **[INV-14]** Part 2 requires the running-set scheduler (the `max=N` kernel arm) to be active; it
  CANNOT work on the serial `open-next` surface (one row only). Concurrent disjoint writes are
  dispatched via `open-ready` (which can hold a set in the running set), never via `open-next`. State
  the dependency as: **P2 depends on the P1 kernel MODEL + the `max_concurrent` field, NOT on the P1.3
  wrapper collapse.**

**Out-of-lane writes — two LAYERED failure modes (defense in depth):**

1. **Write-time (hook, flag ON only):** the #376 `PreToolUse(Write|Edit)` hook DENIES (exit 2) an
   out-of-lane Write/Edit before it lands. Self-exempt for the open write node's own lane (#386 arch
   ii). The FAST-FAIL layer. Bash-mediated writes (`echo >`, `sed -i`, redirections, `git apply`)
   bypass it — INTENTIONAL layering, not a hole.
2. **Close-time (barrier, ALWAYS — flag-independent):** the per-node `--barrier-check` allowlist
   refuses the close if ANY change (including a Bash-mediated one that slipped past the hook) is out of
   the node's declared write set. Reason: the EXISTING `barrier_failed` (with the underlying validator
   reason in the emit envelope: `write_set_overflow` / `write_set_granularity` /
   `sensitive_write_unreviewed` / `foreign_archive`).

There is NO new `unattributed_write` reason code. An out-of-lane write surfaces as either a hook
`exit 2` (write-time) or the existing `barrier_failed` / `write_set_overflow` family (close-time).
Adding a fifth reason would fork the emit envelope; reuse the existing taxonomy.

**Part 2 open questions (recorded, NOT resolved here):**

- **OQ-P2-a:** Where does `parallel_safe` live physically — recomputed-only vs persisted-in-
  running-set.json? Recommendation: recompute-on-open + persist as a witness for reconcile continuity,
  mirroring `max_concurrent`. Pin: "recompute is authoritative; the persisted value is a witness."
- **OQ-P2-b:** The classifier `disjointWriteSets` is the SAME engine freeze and the scheduler use;
  confirm the doc cites the single-source-of-truth reuse (no second disjointness implementation).

### Part 4 — Optional consent-gated speculative gate overlap

**Decision (P4):** Allow a NON-gate descendant node to open SPECULATIVELY (`in_progress`) while its
post-dominating gate node is still `in_progress`, with the speculative work DISCARDABLE if the gate
fails. This is the highest-risk part: it is OPT-IN, OFF by default, and bounded by an explicit
invariant that post-dominance over the COMPLETE state is never weakened.

**Consent model.** A per-PLAN `speculative_open_policy: off | consent | auto` field, default `off`:

- `off` (default): no speculative open — today's behavior, the permanent fallback. A descendant cannot
  open until its gate is `complete`.
- `consent`: the planner MAY mark specific edges as speculative-eligible; the scheduler opens a
  speculative descendant ONLY after the USER grants consent for THIS run (at the existing
  `decision:ask` checkpoint — which here becomes the consent capture point). Consent is per-run, not
  persisted in the frozen plan (it does not enter the hash).
- `auto`: reserved/unsupported in v3 — named but inactive so the schema is forward-compatible.
  REJECTED at freeze with a typed refusal until a future issue defines its risk envelope.

`speculative_open_policy` IS a Meta field (hash-covered) for ELIGIBILITY, because it changes the legal
EXECUTION SHAPE and must be tamper-evident. This is the ONE new field that SHOULD enter the hash —
deliberately ASYMMETRIC vs. the runtime-cap fields (`max_concurrent` in D-419-01 P1.2 and
`parallel_safe` in P2.1) that stay OUT of the hash because they are pure resource limits / derived
projections. Per-run consent is non-persisted and gates ACTIVATION.

**Part 4 invariants:**

- **[INV-19]** Consent is captured per-run by the USER (the human at `decision:ask`), never
  auto-granted by an agent. The policy field gates ELIGIBILITY (plan-level); the per-run consent gates
  ACTIVATION. Both must be true to speculatively open.
- **[INV-20]** The discard is a roll-BACK to the speculative node's recorded baseline, never a
  roll-forward. No partial speculative write survives a gate failure. The baseline-snapshot revert is
  the existing barrier-base mechanism; v3 adds the revert-on-gate-fail wiring, not a new snapshot
  machine.
- **[INV-21] (the load-bearing invariant of Part 4):** "Speculative open" means the descendant is
  `in_progress`, NEVER `complete`, while the gate is `in_progress`. The gate's close (to `complete`
  with `verdict: pass`) remains the STRICT prerequisite for the descendant's OWN
  `close-and-open-next` / `close-node`. A new close-time refusal `gate_not_complete` is returned if a
  node N tries to close while its post-dominating gate G is not `complete` with a passing verdict.
- **[INV-22]** The existing `--gate-verify` post-dominance EXECUTION check (verify gate execution over
  the runtime Node Ledger) is the enforcement point. Speculative open does not touch it; it adds a
  sibling close-time guard that the gate is `complete` before the descendant's close. No descendant
  reaches `complete` ahead of its gate ⟹ post-dominance over the complete-set is preserved exactly.
- **[INV-23]** Part 4 REQUIRES the P1 kernel `max>1` arm (the running-set scheduler), because
  speculative overlap means TWO nodes (gate + descendant) are `in_progress` simultaneously — exactly
  the running-set fan-out the serial `open-next` surface cannot represent. State the dependency: **P4
  depends on P1 (the scheduler / `max>1` arm) AND is INDEPENDENT of P2** (the gate may be a read-only
  review node and the speculative descendant may be read-only — no write-lane attribution needed for
  the read case). Where the speculative descendant WRITES, P4 additionally inherits P2's lane
  discipline.
- **[INV-24] (the blast-radius guard):** a speculative descendant that WRITES is opened ONLY when (a)
  its write set is disjoint from the gate's review surface AND from every concurrent write node
  ([INV-10] / P2 lane discipline), AND (b) the discard path ([INV-20] baseline roll-back) is wired and
  tested. The strongest containment: a speculative WRITE descendant is gated behind BOTH
  `speculative_open_policy: consent` AND `KAOLA_LANE_CONTAINMENT=true` (so the write-lane hook +
  barrier bound its writes). A speculative READ-ONLY descendant has near-zero blast radius (no writes
  to revert; evidence discard only) and is the RECOMMENDED default scope for v3.
- **[INV-25]** If the gate fails, ALL speculative descendants opened under it roll back ([INV-20]), not
  just the conflicting one — the gate failure invalidates the ENTIRE speculative frontier opened behind
  it. Conservative-correct: a gate failure means the premise the speculation built on is void.

**Discard path (in order, on gate `verdict != pass`):**

1. **Ledger:** reset the speculative node's row `in_progress → pending` (NOT a new state — [INV-4]
   from D-419-01 holds; reuse the existing reopen/reset transition). It becomes re-runnable.
2. **Baseline:** drop its per-node baseline (`commit-node --drop-base`, the same mechanism
   `reconcile-running-set` uses to drop a dropped member's baseline).
3. **Running set:** remove the speculative node from `running-set.json` (the existing close-direction
   removal, mirroring `close-node`'s removal step).
4. **Evidence:** discard its `.cache/{node-id}.md` evidence file (it is bound to the now-dropped
   baseline nonce; discard explicitly to avoid an `evidence_stale` confusion on re-run).
5. **On-disk writes (if any):** `git`-level revert of the speculative node's declared write set back to
   the baseline snapshot that `commit-node --start` recorded. The revert TARGET is the baseline, which
   is why the baseline drop (step 2) is ordered conceptually around the revert.

**Recommendation — SHIP read-overlap, DEFER write-overlap.** v3 ships ONLY speculative READ overlap
(`off` / `consent` for read descendants — the common case: pre-fetching the next analysis while a
review runs). Speculative WRITE overlap is DESIGNED-but-DEFERRED to a follow-up issue with its own
adversarial gate, given the double-gating and roll-back complexity. The schema records `auto` and the
write-overlap path as forward-compatible without shipping the risk.

**Part 4 open questions (recorded, NOT resolved here):**

- **OQ-P4-a:** `speculative_open_policy` as a Meta field (hash-covered, frozen, resume-stable) vs a
  runtime flag. Recommendation: Meta field for ELIGIBILITY + per-run consent for ACTIVATION — the
  eligibility SHOULD be in the hash (it changes the legal execution shape and must be tamper-evident),
  unlike concurrency caps which are pure resource limits. This is the deliberate asymmetry vs.
  P1.2 / P2.1.
- **OQ-P4-b:** The new `gate_not_complete` close-refusal reason must be added to the emit-envelope
  precedence family; confirm its precedence rank with the validator's reason ordering.
- **OQ-P4-c:** v3 ships read-overlap; defers write-overlap (the DESIGNED-but-DEFERRED scope above).

## Consequences

**What ships in v3 (this record's scope):**

- **P2:** the validator derives + stamps `parallel_safe` (recompute-on-open, witness-persisted in
  `running-set.json`, out of `plan_hash`); `open-ready` co-opens disjoint write nodes ONLY when
  `KAOLA_LANE_CONTAINMENT=true`. With the flag OFF (default) behavior is byte-identical to today.
- **P4:** the `speculative_open_policy` Meta field (`off` default, `consent` supported, `auto`
  refused), the `gate_not_complete` close-refusal, and the baseline roll-back discard path — SHIPPING
  speculative READ overlap only.

**What is explicitly DESIGNED-but-DEFERRED:**

- Speculative WRITE overlap (P4) — deferred to a follow-up with its own adversarial gate (OQ-P4-c).
- `speculative_open_policy: auto` — named but refused at freeze until a future issue defines its risk
  envelope.

**For day-to-day development:**

- Turning on write parallelism requires setting `KAOLA_LANE_CONTAINMENT=true` AND the write nodes
  being validator-stamped `parallel_safe` (disjoint declared write sets in an antichain). Overlapping
  write sets stay serial regardless of the flag ([INV-10]).
- The barrier remains the security boundary ([INV-12]/[INV-13]); the write-lane hook is an ergonomics
  accelerant, not a security control. A Bash-mediated out-of-lane write surfaces at the barrier, never
  silently.
- Using speculative overlap requires both the plan-level `speculative_open_policy: consent` AND a
  per-run human grant at `decision:ask` ([INV-19]). No agent can auto-grant it.
- A speculative descendant can never reach `complete` ahead of its gate ([INV-21]); a failing gate
  rolls back its entire speculative frontier ([INV-25]).

**For downstream issues:** the invariants that bind every Part-2/Part-4 implementation are [INV-8]
through [INV-14] (derived stamp + barrier ground-truth + scheduler-arm precondition) and [INV-19]
through [INV-25] (consent model + roll-back discard + speculative = in_progress never complete +
blast-radius guard). The cross-cutting invariants that bind EVERY downstream issue are: [INV-2] serial
byte-identity; [INV-4] four ledger states only; [INV-5] running-set `opening` crash state + reconcile;
[INV-6] write-alone fallback (unless P2 flag on + disjoint); [INV-9]/[INV-12] barrier is ground truth;
[INV-16] four-chain cross-edition green; [INV-21] speculative = in_progress never complete.
