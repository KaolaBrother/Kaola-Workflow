# D-576-01. Machine-enforce the provenance-ban convention with a PROVENANCE_BAN guard

Date: 2026-06-29
Status: Accepted
Issue: #576
Related: D-575-01 (convention established, enforcement deferred — this decision supersedes that deferred note)

## Context

D-575-01 (issue #575) established the rule that design-rationale provenance must not appear in
agent-facing prompt surfaces, and recorded it as a durable convention in `CLAUDE.md` and
`docs/conventions.md`. The decision explicitly deferred machine enforcement:

> **Lint guard / contract pin.** A durable machine-enforced ban — a carefully allowlisted regex
> scanning the prompt-surface file set for banned token patterns — would prevent regression without
> relying on review discipline alone. The allowlist requires care: runtime forms
> (`KAOLA_TARGET_ISSUE=N`, `"issue N"`) and illustrative user examples (`"work on #42"`) must not
> be banned. A follow-up issue should scope and implement this guard.

Without a machine check, the ban was a manual discipline enforced only by code review and the
convention text. A single prompt-surface edit in any subsequent issue could silently reintroduce
provenance tokens.

## Decision

1. **Implement `PROVENANCE_BAN` as a hard gate in all five contract validators and the opencode
   test suite.** The guard scans agent-facing prompt surfaces with the following regex:

   ```
   /#\d{1,4}|D-\d{3}-\d{2}|\bINV-\d+|ADR[ -]\d{2,4}|\b(?:PR|MR|AC)#\d+/
   ```

   A match is a hard chain failure. The error message names the offending `file:line` and token,
   and points back at `docs/conventions.md` § Enforcement.

2. **Banlist taxonomy.** The regex covers five classes of external-pointer provenance:

   | Class | Regex arm | Examples |
   |---|---|---|
   | Issue refs | `#\d{1,4}` | `#472`, `(#307)` |
   | Decision IDs | `D-\d{3}-\d{2}` | `D-542-01`, `D-430-01` |
   | Invariant tags | `\bINV-\d+` | `INV-01`, `[INV-04]` |
   | ADR citations | `ADR[ -]\d{2,4}` | `ADR-0008`, `ADR 0004` |
   | Forge request refs | `\b(?:PR|MR|AC)#\d+` | `PR#123`, `MR#456`, `AC#789` |

   The forge-request-refs arm (`PR#/MR#/AC#`) was not in the original #575 banned-token table; it
   is added here as a fifth class and reflected in `docs/conventions.md`.

3. **Allowlist — forms the guard does not flag:**

   - **Numeric placeholders**: `#N`, `#<issue>`, `#<n>` — contain no digit-only sequence after
     `#` and therefore do not match `#\d{1,4}`.
   - **Runtime target-issue variables**: `KAOLA_TARGET_ISSUE=N`, `--target-issue <N>`,
     `Closes #<issue>` — placeholder forms with no digit sequence.
   - **Audit/gate short-labels**: `G1`, `G3`, `H5`, `AC7`, `M4` — letter-digit labels that match
     no banned arm.
   - **Illustrative examples with actual numbers** (e.g. `"work on #42"`) match the `#\d{1,4}`
     arm and are therefore not allowed in prompt surfaces; use placeholder form instead
     (e.g. `"work on #<N>"`).

4. **Per-edition surface placement.** Each edition's validator scans its own prompt-surface tree:

   | Validator | Surfaces scanned |
   |---|---|
   | `validate-kaola-workflow-contracts.js` (claude) | `agents/*.md`, `commands/*.md`; byte-mirrored to the codex copy |
   | `validate-kaola-workflow-contracts.js` (codex) | `plugins/kaola-workflow/agents/*.toml`, `plugins/kaola-workflow/skills/` |
   | `validate-kaola-workflow-gitlab-contracts.js` | `plugins/kaola-workflow-gitlab/` agents, commands, skills |
   | `validate-kaola-workflow-gitea-contracts.js` | `plugins/kaola-workflow-gitea/` agents, commands, skills |
   | `scripts/test-opencode-edition.js` (opencode, assertion A25) | Regenerated `.opencode/` agent and command mirrors |

   The guard runs in all four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains and
   in the additive opencode suite. A violation is a hard failure in whichever chain covers the
   edited surface; no cross-edition propagation is required for an opencode-only diff (D-530-02).

5. **Supersedes the deferred note in D-575-01.** The `## Deferred` section of D-575-01 described
   this guard as a future follow-up. That note is resolved: the guard is implemented as of #576.

## Consequences

- Provenance cannot silently re-enter a prompt surface via an ordinary edit: the chain covering
  that surface will fail closed with a `file:line` diagnostic.
- The allowlist is structural, encoded in the regex arms and their non-match behavior — no
  separate allowlist config file is required.
- The opencode edition (additive, not wired into `npm test`) enforces the ban via its own suite;
  D-530-02's "additive, not a forge" classification is preserved.
- `docs/conventions.md` § Enforcement is updated to reflect the machine-enforced state, the full
  banlist taxonomy (including the new forge-request-refs class), and the corrected allowlist
  (illustrative examples must use placeholder form).

## Alternatives considered

- **Enforce only in the claude chain.** Rejected: prompt surfaces span all four editions plus
  opencode. An edition-scoped guard leaves three edition trees and the opencode mirrors unprotected.
- **Separate allowlist config file.** Rejected: the regex arms already handle placeholders
  structurally; a data-driven allowlist adds complexity with no benefit.
- **Fail-warn instead of fail-hard.** Rejected: a warning degrades to a manual discipline — the
  same state as before the guard. A soft check adds no value over code review alone.
- **Context-exempt illustrative examples with actual numbers.** Rejected: context-aware exemption
  requires AST-level understanding of the prose, not a line-pattern scan. Preferring placeholder
  form in examples is the simpler, unambiguous rule.
