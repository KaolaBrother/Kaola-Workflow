# D-687-01. Codex role profiles inherit the parent session runtime pair

Date: 2026-07-15
Status: Accepted
Issue: #687
Related: #451 (base-profile effort inheritance); #582 (runtime verification boundary);
         #598 (user-owned root dispatch posture); #610 (neutral tier vocabulary)

## Context

Kaola's Codex role profiles had converged on two standalone runtime pairs: Sol/medium for the
standard class and Sol/xhigh for the reasoning class. Those pins overrode the effective parent
session. A workflow started on another model or effort therefore did not reliably carry that
runtime choice into its role children, even though the plan's standard/reasoning vocabulary was
intended to describe work class and orchestration budget rather than silently replace the user's
session choice.

Profile text and spawn arguments are not sufficient runtime proof. The contract needs both an
inheritance representation that Codex actually honors and a bounded, thread-bound observation of
the model and effort persisted for the parent and child turns.

## Decision

1. **Omission is the inheritance form.** All 48 generated Codex role-profile TOMLs omit the
   top-level `model` and `model_reasoning_effort` keys. There is no sentinel value and no
   per-spawn model/effort override. The named child profile therefore inherits the effective pair
   from the parent session.
2. **Legacy pairs migrate; partial or foreign pins fail schema validation.** The exact managed
   Sol/medium and Sol/xhigh pairs are classified as stale and the installer/preflight migrates
   them to omission. A profile containing only one runtime key, or any other explicit pair, is
   malformed. A second install/preflight is idempotent. The root user-owned
   `model_reasoning_effort` setting is separate and is never rewritten by this migration.
3. **Tier vocabulary remains declarative.** `standard` and `reasoning`, plus the permanent legacy
   aliases `sonnet` and `opus`, remain frozen-plan metadata. They continue to select the default
   wait budget (20 and 40 minutes respectively), displays, and non-Codex edition mappings. They no
   longer select a Codex child runtime pair.
4. **Dispatch cards state inheritance explicitly.** Known Codex roles emit
   `codex_profile_mode:'inherit'`, their declarative `codex_profile_tier`, and
   `codex_profile_compatible:true`. Model and effort sources read `parent_session`; the summary
   renders `effort=inherit`. A genuinely unresolved plan tier remains an independent
   `codex_tier_unresolved` refusal.
5. **Runtime equality is proven from session JSONL.** Before dispatch, plan-run loads a fresh
   parent proof bound to the current thread. After dispatch it verifies the requested installed
   profile bound to the child and requires the child's persisted `turn_context.model` and
   `turn_context.effort` to equal the freshly re-read parent/card pair. Missing or stale proof,
   ambiguous binding, profile drift, and pair mismatch fail closed with typed refusals.
6. **Reasoning-floor roles inspect the inherited parent, not a profile pin.** A reasoning-floor
   dispatch requires fresh current-session proof classified at `gpt-5.6-sol`/`xhigh` or an
   accepted stronger effort (`max` or `ultra`). Missing, stale, below-floor, and unknown-model
   observations refuse as `reasoning_floor_proof_missing`, `reasoning_floor_proof_stale`, or
   `reasoning_floor_violation`.

## Live proof and fail-closed boundary

A real Codex 0.144.3 parent/child probe used the shipped `code-explorer` profile and a spawn call
that omitted model and reasoning-effort arguments. The persisted parent and child turn contexts
both reported `gpt-5.6-sol` with `xhigh`; the child was linked to exactly one parent and selected
the requested role profile. One equality proof is sufficient for the shared omission mechanism:
the standard/reasoning classes no longer encode different Codex runtime values, so duplicating the
same inheritance probe per metadata tier would not exercise another runtime branch.

The proof loader is bounded and fail-closed. Candidate JSONL files must bind uniquely to the
current thread, and every file is reclassified after open. If a directory entry classified as a
regular JSONL changes to a directory, symlink, or other non-regular object before inspection, the
scan is incomplete and no proof is accepted. This closes the regular-file classification race
without widening traversal or accepting a partial scan.

## Consequences

- A Codex role child follows the model and reasoning effort the operator actually selected for the
  parent session.
- Role classes still communicate expected reasoning depth and preserve existing wait budgets,
  Claude aliases, opencode variants, and frozen-plan compatibility.
- Existing managed static pairs repair automatically, while partial or unexpected explicit pins
  produce a precise malformed-profile refusal instead of being silently normalized.
- Runtime claims depend on durable parent/child evidence. Configuration prose alone cannot certify
  inheritance, and an unavailable or ambiguous proof stops dispatch.

## Alternatives considered

- **Keep static role pairs and override only selected spawns.** Rejected because profile reload and
  override precedence make the effective child pair fragile, and a static role pair still defeats
  the parent-session contract.
- **Encode inheritance with a string sentinel.** Rejected because omission is the runtime-supported
  representation; a sentinel would be another explicit value with version-dependent parsing.
- **Remove standard/reasoning tiers on Codex.** Rejected because the vocabulary remains useful,
  hash-stable orchestration metadata and is shared with the other editions.
