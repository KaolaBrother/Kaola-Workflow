# D-522-01. Gate `cmdFinalize` on `--finalize-check` before any irreversible side effect

**Date:** 2026-06-18
**Status:** Accepted
**Issue:** #522
**Related:** `scripts/kaola-workflow-claim.js` (`cmdFinalize`),
            `scripts/kaola-workflow-plan-validator.js` (`--finalize-check`, unchanged),
            `scripts/kaola-workflow-run-chains.js` (unchanged),
            `agents/contractor.md` (Step 8c demoted — n2),
            `plugins/kaola-workflow/agents/contractor.toml` ×3 forge editions (n2),
            `docs/decisions/D-512-01.md` (run-chains timeout context).

## Context

### The fail-open gap

`cmdFinalize`'s archive commit was unconditional. The adaptive-path finalization gate (`kaola-workflow-plan-validator.js --finalize-check`) enforces dual-mode validation:

- **Self-host (npm) mode** — `chain-receipt.json` bound to HEAD must exist, be fresh, and be all-green.
- **Consumer (non-npm) mode** — `.cache/final-validation.md` with a column-0 `verdict: pass` must exist.
- **Both modes** — attribution sweep (B).

This gate was authored as a `--finalize-check` flag in `plan-validator.js` and was intended to be run by the contractor subagent (Step 8c) before the archive commit. But the mechanism broke silently:

1. Receipt **generation** requires running `run-chains`, which takes approximately 15 minutes for the `claude` chain.
2. The contractor is dispatched from a Bash tool call. The Claude Code Bash tool auto-backgrounds commands that exceed its 600s ceiling; a subagent cannot await a backgrounded process across turns.
3. Therefore the contractor could not wait for the receipt to be produced, and `cmdFinalize` proceeded without a verified receipt — a fail-OPEN of a fail-CLOSED gate.

This failure mode was observed live during the #520 run: the archive commit landed, the issue was closed, and there was no receipt verifying the chains had passed.

### Root cause

The gate existed and was correct in `plan-validator.js`. The failure was at the **caller**: `cmdFinalize` (in `claim.js`) never shelled the gate itself. The contractor (a subagent) was expected to do so, but had no mechanism to await a 15-minute subprocess within a single Bash turn. The gate was therefore structurally unreachable at the moment it was needed.

## Decision

### Gate before the commit, inside `cmdFinalize`

`cmdFinalize` (all four `claim.js` editions) now shells `kaola-workflow-plan-validator.js --finalize-check --json` immediately before calling `archiveProjectDir`. Placement is before `archiveProjectDir` so that no irreversible side effect — archive rename, worktree removal, issue close — has occurred on a refusal.

The gate fires unconditionally for both the `--keep-worktree` worktree path and the in-place path, **when a `workflow-plan.md` is present** (adaptive run). Non-adaptive runs (no plan file) skip the gate and proceed as before.

On any non-`pass` result, `cmdFinalize` emits:

```json
{
  "result": "refuse",
  "reason": "finalize_gate_unverified",
  "inner_reason": "<inner reason from validator>",
  "operator_hint": "Resolve the inner refusal, then re-run finalize. No archive commit was made.",
  "errors": ["<inner reason>"]
}
```

with exit 1. No archive commit is made.

### Procedure change: main session owns receipt generation; contractor verifies

Receipt **generation** (running `run-chains` and waiting for the `chain-receipt.json`) moves to the main session/orchestrator, which can await the long backgrounded run across multiple turns without the 600s Bash ceiling affecting it. The contractor's Step 8c is **demoted from generate to verify-or-fail-closed**: the contractor confirms the receipt exists and is bound to HEAD before the finalize call, but it does not launch `run-chains` itself.

This change propagates to `contractor.md` + 3 `contractor.toml` forge editions + the 6 `#400` finalize surfaces (n2 nodes). `plan-validator.js` and `run-chains.js` are **unchanged** — only the caller (`cmdFinalize`) and the procedure prose change.

### No bypass flag

There is no `--skip-finalize-check` or equivalent. Adding a bypass would reintroduce the fail-open: any bypass path is a path where the archive commit is unconditional again. The operator_hint already tells the operator what to do when the gate refuses.

## Alternatives considered

### Move `run-chains` to a resumable / chunked mode

`run-chains` could in principle be restructured so that individual chains can be launched, persisted, and polled across multiple Bash turns. This would allow the contractor to own the full generate-and-verify loop without the 600s ceiling being a blocker.

Rejected for this issue: the effort is large and uncertain (requires rearchitecting `run-chains.js` and the receipt-writing flow), and a pre-existing workaround (#512 raised the `spawnSync` kill ceiling to 900000ms) already handles the common case of a slow-but-finishing chain. The cheapest-sufficient fix (gate the caller, move generation to the orchestrator) achieves the correctness objective without the larger effort (#3: cheapest sufficient mechanism, per the design precedence).

### Add a bypass flag (`--skip-finalize-check`)

A bypass would allow a finalize to succeed without the gate, intended for situations where the receipt cannot be produced (e.g., no network access to run the chain).

Rejected unconditionally: a bypass reintroduces the fail-open and violates precedence #1 (accuracy is non-negotiable). If the gate cannot be satisfied, finalization must not proceed — the operator_hint already tells the operator how to resolve the inner refusal.

## Implementation

`scripts/kaola-workflow-claim.js` (`cmdFinalize`): a block is inserted immediately before the `archiveProjectDir` call. It resolves `livePlanPath` via `adaptiveSchema.PLAN_FILE`; when the file exists, it shells `execFileSync(process.execPath, [validatorScript, livePlanPath, '--finalize-check', '--json'], { cwd: root, encoding: 'utf8', timeout: 120000 })`. A non-zero exit from the validator is caught; stdout is parsed for a JSON envelope. Any non-`pass` `result` triggers the typed refusal above. The `timeout: 120000` covers the validator's own sync work (not `run-chains`; the receipt file is expected to already exist by the time `cmdFinalize` is invoked).

Applied byte-identically to the codex twin (`plugins/kaola-workflow/scripts/kaola-workflow-claim.js`) and rename-normalized to the GitLab and Gitea ports. All four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains green (#307).

## Consequences

- The archive commit, worktree removal, and issue close are now gated on a verified finalization receipt for every adaptive run. The fail-open gap that let `cmdFinalize` succeed with no verified receipt is closed.
- Non-adaptive (fast/full) runs are unaffected: the gate fires only when `workflow-plan.md` is present.
- The procedure change is self-reinforcing: the finalization of this very run (#522) is itself gated by the code this issue delivered — a live dogfood proof.
- The contractor's Step 8c role is narrowed from generate+verify to verify-only, removing the structural impossibility of awaiting a 15-minute subprocess within a single Bash turn.
- `plan-validator.js` and `run-chains.js` are unchanged. All existing call sites of `--finalize-check` are unaffected.
