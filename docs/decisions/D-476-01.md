# D-476-01. `--help` and unrecognized flags are safe no-ops / typed refusals on the destructive lifecycle scripts (issue #476)

**Date:** 2026-06-15
**Status:** Accepted
**Issue:** #476 (sibling of #475; both surfaced by the KaolaTerminal issue-85 run)
**Related:** #346 (zero-mutation refuse on a dirty worktree), #427 (idempotent close), #466
            (worktree-authority-split guard) — the existing safety net that an unrecognized flag
            **walked straight past**; #307 (the four-edition cross-edition contract); #395.5 (the
            `--strict` closure-invariant gate).

## Problem

`--help` (and any unrecognized flag) was **not a safe no-op** on `kaola-workflow-claim.js` and
`kaola-workflow-sink-merge.js`. Their `parseArgs` was a permissive collector: known booleans had
explicit branches, a generic `--flag value` branch captured the rest, and a `--flag` that matched
neither (a bare `--help`, a typo) was **silently dropped** — so execution fell through into the
subcommand body. For `finalize` + the sink transaction that meant a full **close issue + merge +
delete branch + remove worktree**, fired by what the operator believed was a help probe. This was the
root trigger of the KaolaTerminal issue-85 orphan, and it **bypassed #346/#427/#466** (all of which
assume the *intended* path). `--help`/`-h` was implemented nowhere, and no `unknown_flag` refusal
existed anywhere in the script set — so this is a new contract, not a tweak.

## Decision

**D1 — `--help`/`-h` is a guaranteed safe no-op.** On both scripts, `--help`/`-h` anywhere in argv
prints usage and exits 0 with **zero side effects**, checked BEFORE any subcommand dispatch /
destructive operation. (claim.js checks the whole argv so the flag is honored even in the subcommand
slot, e.g. `claim.js --help`.)

**D2 — Unrecognized flags refuse with a typed `unknown_flag` + zero mutation.** A `-`-prefixed token
that is not a recognized flag is recorded during parse and turned into a typed refusal
(`{result:'refuse', reason:'unknown_flag', unknownFlags:[…], operator_hint}` under `--json`, else a
stderr line) with a non-zero exit, BEFORE any side effect. A help probe or a typo can no longer fall
through to a destructive body.

**D3 — A closed per-script flag allowlist, not a global registry.** The two scripts have different
flag sets, so each carries its own allowlist. claim.js keeps its explicit boolean branches and gates
the generic value branch on a `KNOWN_VALUE_FLAGS` set (a `--flag value` whose name is not in the set
is unknown); sink-merge.js uses a `KNOWN_FLAGS` set that INCLUDES the `--sink` transaction-mode flag
(read by `main()` via `rawArgv.includes`, so it must be recognized by the parser too, else the guard
would false-reject the sink transaction). A **known flag missing its value** (a bare `--branch`) is
NOT mis-flagged as unknown — it is in the allowlist and fails its own validation later. This avoids the
false-positive that a naive "any unmatched `--flag` is unknown" rule would produce.

**D4 — NOT extended to status/probe subcommands' permissiveness beyond the guard.** The guard is in
the single `main()` chokepoint, so it applies uniformly; status/probe subcommands are read-only and
unaffected except that they too reject genuinely-unknown flags (simpler + safer than per-subcommand
carve-outs). The `closure_invariants.ok===false`-should-fail-closed question (#395.5) is explicitly
left to its own discussion — out of scope here.

## Editions

claim.js + sink-merge.js are `COMMON_SCRIPTS` (canonical ↔ Codex byte-identical, synced by
`edition-sync`). Their gitlab/gitea forge ports are **divergent hand-maintained** forge-specific
scripts (~180 fewer lines; forge CLI), in no rename-normalized family — so the guard is
**hand-propagated** to all four forge ports, with the per-port `USAGE` referencing the port's own
script name. The four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains gate the result.

## Consequences

- A help probe on a destructive script is now provably side-effect-free; the issue-85 class of orphan
  cannot recur via a dropped flag.
- The existing safety net (#346/#427/#466) is no longer reachable-around by an unrecognized flag.
- Regression coverage drives the **real subprocess CLI** (`test-claim-hardening.js`): `finalize --help`
  → usage + no archive; `finalize --typo --json` → `unknown_flag` + no archive; `sink-merge --help` →
  usage + branch not merged/deleted; `sink-merge --bogus` → `unknown_flag`; and a valid flag
  (`status --json`) is not false-rejected.
