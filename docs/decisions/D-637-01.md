# D-637-01. Add a distinctive interior content token so a marker-preserving gut of the fn-closure-audit block reds the manifest checker

Date: 2026-07-08
Status: Accepted
Issue: #637 (primary — the fn-closure-audit vacuous-guard follow-up filed by #630's own change-gate
       adversary)
Related: #630/D-630-01 (the Layer-1 required-block manifest and derived-universe presence checker
         this decision hardens; the gap was surfaced there and recorded as an accepted residual
         under "Non-goals / accepted residual"); #624 (the whole-finalize-gate-block-loss class the
         Layer-1 manifest exists to close by construction — this decision closes a narrower,
         same-family gap on one specific block)

## Context

The #630 Layer-1 manifest (`templates/routing/required-blocks.js`) declares each routing-prose block
once with a topic, tags, and a set of distinctive `content_tokens`; a derived-universe presence
checker (`checkManifest` in `scripts/test-route-reachability.js`) asserts every one of a block's
obligated surfaces contains every declared token. The `fn-closure-audit` block — covering the six
finalize surfaces (3 commands + 3 SKILLs) — carried
`content_tokens: ['<!-- PIN: closure-audit -->', 'closure-audit']`. The second token is a bare
SUBSTRING of the first (the marker itself), so a mutation that guts the block's interior prose while
surgically preserving the `<!-- PIN: closure-audit -->` marker trivially satisfies both tokens and
stays vacuous-green — the checker would not notice the interior was gutted. This gap was surfaced by
#630's OWN change-gate adversary and recorded as an accepted residual in D-630-01 rather than fixed in
that same diff.

## Decision

Add `sink_incomplete` as a third, DISTINCTIVE `content_token` on the `fn-closure-audit` block —
verified present (3 raw occurrences each) on all six finalize surfaces at HEAD, anchoring the
interior "transactional catch (n1's `sink_incomplete` emit)" prose that already ships under the
closure-audit section, and verified NOT a substring of the `<!-- PIN: closure-audit -->` marker (so
it cannot reproduce the same bug). No token substitution was needed — the chosen token checks out on
all six surfaces unmodified.

A RED-first proof was added to `scripts/test-route-reachability.js`, modeled on the file's existing
6-case RED-PROOF battery: it imports the LIVE `fn-closure-audit` block from `REQUIRED_BLOCKS` (not a
synthetic stand-in), derives its real obligated 6-surface set via `deriveObligated`, builds an
in-memory fixture where every obligated surface is gutted to just the bare marker, runs
`checkManifest` against it, and asserts `failures.length > 0`.

- **Confirmed RED before the fix**: with the test added but `required-blocks.js` still at the buggy
  2-token definition, the new assertion itself failed (`checkManifest` returned 0 failures on the
  gutted fixture — vacuous-green, proving the bug is real) — 1 failure / 282 passed, exit 1.
- **Confirmed GREEN after the fix**: with `sink_incomplete` added as the third token, the identical
  gut-plant fixture correctly reds inside `checkManifest` (missing-token: `sink_incomplete`), which is
  exactly what makes the outer assertion — and the whole suite — pass: 283/283 assertions, exit 0.

## Consequences

- A marker-preserving interior gut of the `fn-closure-audit` block on any of the six finalize surfaces
  now reds the derived-universe presence checker (`test-route-reachability.js`), closing the specific
  vacuous-guard gap D-630-01 recorded as an accepted residual.
- No finalize surface content changed — `sink_incomplete` was already present in all six; only the
  manifest's `content_tokens` array and the new RED-PROOF test case were touched. The diff (filtered
  of the token addition and its comment) shows nothing else in `required-blocks.js` was modified, and
  no finalize surface file was touched.
- All fast-oracle chains re-verified green after the change: `test-route-reachability.js` (283
  assertions), all four `validate-*-contracts.js`, `generate-routing-surfaces.js --check` (12
  surfaces), and `validate-script-sync.js` (unaffected — confirms zero incidental drift).

## Non-goals

- **Does not change `checkManifest`'s whole-file token scoping.** The checker still asserts a token's
  presence anywhere in a surface's full content, not scoped to the block's own interior region — a
  pre-existing design property, not something this decision revisits. Equivalence to
  block-interior-only checking holds today only because every `sink_incomplete` occurrence on all six
  surfaces happens to sit inside the closure-audit region (whose `<!-- PIN: -->` is the last marker
  per surface) — a design note about the current state of the surfaces, not a defect introduced or
  fixed by this decision.
- **Does not touch the six finalize surfaces themselves.** The chosen token was already present on
  all six; this is a manifest-only hardening.

## Alternatives considered

- **Scope `checkManifest` to check tokens only within a block's own delimited interior region (marker
  to next marker).** Rejected as out of scope for this narrow follow-up: it would change the checker's
  general scoping behavior for every block, not just close this one block's vacuous-substring gap, and
  carries a materially larger verification burden than a single well-chosen distinctive token.
- **Pick a different existing token already resident on the surfaces.** Rejected in favor of
  `sink_incomplete` specifically because it was independently verified (not taken on faith) to be
  edition-neutral (no CLI-binary/forge-brand token), present on all six surfaces, and structurally
  unable to reproduce the marker-substring bug.
