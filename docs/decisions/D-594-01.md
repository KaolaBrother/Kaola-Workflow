# D-594-01. Remove the dead `batch_active` mutual-exclusion guard

Date: 2026-07-02
Status: Accepted
Issue: #594
Related: D-586-01 (the parallel-batch retirement that deliberately KEPT this guard as
             backward-compat crash detection, explicitly flagging its future removal "once
             its diagnostic value is judged to no longer outweigh its maintenance cost" —
             this decision is that future call)

## Context

D-586-01 retired the standalone `parallel-batch.js` aggregator — the only writer of
`kaola-workflow/{project}/.cache/active-batch.json` — but deliberately KEPT the
`batch_active` mutual-exclusion guard in `kaola-workflow-adaptive-node.js`'s layered guard
prologue (`probeCoordination`/`coordinationRefusal`), reasoning that a residual
`active-batch.json` on disk (a leftover from a pre-retirement checkout, or a hand-crafted
test fixture) should still refuse a live-coordination conflict rather than silently
proceeding. That decision recorded the guard as "vestigial-but-safe" and named its own
removal as a separate, future, deliberate change out of its scope.

A repo-wide grep for any write/create path to `active-batch.json` — across
`scripts/` and `plugins/kaola-workflow/scripts/` — confirmed **zero writers**. The only
non-comment references outside the mutual-exclusion guard itself were: a
`validate-workflow-contracts.js` comment citing it as a historical example of a durable
file the `writeFileAtomicReplace` helper protects (an unrelated atomic-replace concern, out
of this decision's scope), and a historical design-lineage comment in
`kaola-workflow-adaptive-schema.js`. Nothing in the live product can produce
`active-batch.json` for this guard to legitimately fire against.

Separately, `runReopenNode` and `runRepairNode` (the plan-repair primitives) each carried a
sibling guard — `active_batch_exists` — that refused over the SAME manifest read directly
(not via the shared `probeCoordination` path). This is the identical dead-manifest-guard
family: it can only fire against a file nothing produces.

## Decision

**Remove the `batch_active` mutual-exclusion guard in full.** Specifically, from
`kaola-workflow-adaptive-node.js`:

1. `probeCoordination`'s `active-batch.json` read (the `manifestPath` read, its
   `safeJsonParse`, and the `manifest` result field).
2. `readCoordinationState`'s `batchOpening` / `batchActiveState` / `batchLive` derived
   fields, and their use in the `serialLive` / `collisions` computations.
3. `coordinationRefusal`'s `batch_active` arm (the `want.has('batch')` branch).
4. The `'batch'` entry from every per-command `excl` array: `open-next`'s
   `['scheduler', 'batch']` → `['scheduler']`; `close-and-open-next`'s `['batch']` → no
   `excl`; `open-ready`'s `['serial', 'batch']` → `['serial']`.
5. The `batch_active` `OPERATOR_HINT_REGISTRY` entry.

**Expand scope to the sibling `active_batch_exists` plan-repair-reopen arm.** The plan
that authorized this change permitted scope expansion beyond the issue's headline
`batch_active` guard "only if provably dead" for the candidate surface. `active_batch_exists`
met that bar — same manifest, same zero-writer proof — so it is removed too:

6. The `active_batch_exists` refusal arm in `runReopenNode` (its direct
   `active-batch.json` read + members/state check).
7. The identical arm in `runRepairNode`.
8. The `active_batch_exists` `OPERATOR_HINT_REGISTRY` entry.

The parallel **running-set** exclusion arm (`scheduler_active`, guarding against a live
`.cache/running-set.json` fan-out) is a DIFFERENT, LIVE coordination surface and is
untouched by this removal — both `runReopenNode` and `runRepairNode` keep their
`scheduler_active` guard over a live/opening running set.

## Scope boundary — deliberately kept

`runOrient`'s read-only `active-batch.json` legality reconstruction (~L1338 in
`kaola-workflow-adaptive-node.js`: the `batch_topup_incomplete` reason, the AC#5 batch
member-set arm, and the `manifest`/`batch` fields on `orient`'s return payload) is **NOT**
removed by this decision. Three reasons:

1. **It is not the guard this decision targets.** `batch_active` is a MUTUAL-EXCLUSION
   refusal that blocks a mutating subcommand from running at all; `orient`'s reconstruction
   is a READ-ONLY diagnostic that reports legality state back to the caller. They consume
   the same manifest but serve structurally different purposes, and the authorizing plan's
   wording scoped this removal to "reads that ONLY the dead guard consumes" — `orient`'s
   read is not exclusively consumed by the guard.
2. **It is contract-bearing.** Existing fixtures (`T20a` et al.) assert `orient` returns a
   `batch` field (always `null` in producible state) and specific legality shapes woven
   together with the LIVE #377/#384/#305/#293 running-set legality reconstruction it shares
   code with. Removing it would be a materially larger change to `orient`'s contract than
   this issue's declared scope, breaking roughly six fixtures the plan-repair guard removal
   does not touch.
3. **It is byte-identical for every producible state either way.** Since nothing writes
   `active-batch.json`, the manifest is always `null`, so `orient`'s reconstruction always
   takes the same "no batch" branch regardless of whether the dead-manifest read stays or
   goes — keeping it costs nothing operationally; removing it would cost real fixture churn
   for zero behavioral change.

An opus adversarial-verifier gate independently confirmed this boundary is sound (WITHSTOOD,
non-blocking LOW finding documenting the same rationale) and confirmed the removal is
behavior-neutral for every producible state across all four editions.

## Consequences

- The guard refusal taxonomy shrinks from three arms (`serial_node_live`,
  `scheduler_active`, `batch_active`) to two (`serial_node_live`, `scheduler_active`).
- No natural failing unit test exists for pure dead-code removal (there is no producible
  input that could have exercised the removed branches either before or after). The proof
  of correctness is: (a) grep-absence of `batch_active`/`active_batch_exists` in live
  scheduler code post-removal (both survive only in explanatory comments); (b) the full
  `test-adaptive-node.js` suite and `simulate-workflow-walkthrough.js` staying green with no
  regression; (c) the ~5 pre-existing fixtures that directly exercised the removed guard
  converted to their surviving `scheduler_active` sibling (where that preserves equivalent
  coverage) or retired as no longer producible.
- A stray `active-batch.json` left over from a pre-retirement checkout is now silently
  inert to the mutual-exclusion guard (previously it would have refused). This is an
  accepted consequence: nothing in the current product can create such a file, so the only
  way one exists is a manual artifact from before D-586-01 — a one-time, self-resolving
  condition (the file simply sits unused until an operator removes it), not a live hazard.
- Cross-edition (#307): `kaola-workflow-adaptive-node.js` is a GENERATED_AGGREGATOR
  (canonical + Codex byte-twin + GitLab/Gitea rename-generated forge ports) — regenerated
  via the edition-sync mechanism; all four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}`
  chains must stay green.

## Alternatives considered

- **Leave the guard in place indefinitely.** This was D-586-01's original chosen posture
  ("vestigial-but-safe"). Rejected now because the guard has proven to carry ongoing
  maintenance cost with zero diagnostic payoff for any producible state — every fixture
  that exercises it must hand-construct `active-batch.json` directly (there is no live
  producer to drive it naturally), which is pure test-authoring overhead for a condition
  that can never occur in practice.
- **Remove only `batch_active`, leave `active_batch_exists` for a further follow-up.**
  Rejected: both arms guard the exact same dead manifest via the exact same zero-writer
  proof: there is no meaningful risk-profile difference between them that would justify
  splitting the removal into two separate changes.
- **Remove `orient`'s read-only reconstruction too, for full consistency.** Rejected — see
  "Scope boundary" above: it is a different kind of consumer (diagnostic, not a mutation
  guard), it is contract-bearing on fixtures well outside this issue's scope, and it costs
  nothing to keep since the manifest is always `null`.
