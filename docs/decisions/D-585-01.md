# D-585-01. Scheduler mutual-exclusion lock

Date: 2026-07-02
Status: Accepted
Issue: #585
Related: #590 (the companion baseline-first `open-next` reorder, `docs/decisions/D-590-01.md`,
             found by the same 2026-07-02 parallel-execution audit)

## Context

The running-set scheduler's mutual-exclusion guard was **advisory-only**: Layer-3
coordination (`probeCoordination` / `coordinationRefusal`, `adaptive-node.js`) is a pure
*read* of state files followed by an in-memory refusal decision — no lockfile, `flock`,
O_EXCL claim, or PID check exists anywhere in the mutation path. `writeFileAtomicReplace`
(`adaptive-schema.js`) is atomic PER FILE (write-to-temp, fsync, rename) but performs a
blind `renameSync` with no compare-and-swap against the on-disk version, so every durable-
state mutation is an unlocked whole-file read-modify-write. All concurrency safety rested
on the unenforced assumption that exactly one orchestrator serializes scheduler invocations
per project.

Two concrete lost-update races followed directly, both reproduced deterministically by a
genuine two-process interleave harness (`raceN`: a `node -e` orchestrator `Promise.all`-
spawning real `adaptive-node.js` CLI subprocesses against ONE project) before this fix:

1. **Concurrent `open-ready`×2.** `runOpenReady` deliberately omits `'scheduler'` from its
   exclusion set, so two concurrent calls both pass the guard, both read `running-set.json`
   as `null`, both compute the same frontier, and both dispatch it — a double-open
   (5/5 trials).
2. **Concurrent `close-node`×2.** `runCloseNode` reads the whole plan, splices one ledger
   row + a compliance row, then rewrites the entire file. Two closes for different
   completed members of a co-open frontier each read the same base plan; the second rename
   clobbers the first member's `complete` flip back to `in_progress` — a lost update
   (5/5 trials).

This is the same hazard class as the #579 concurrent same-repo-session hardening, one
level down: two sessions (or a future parallelized orchestrator loop) invoking the
scheduler CLI on one project race with last-write-wins.

## Decision

Add a project-scoped **O_EXCL lockfile** (`kaola-workflow/{project}/.cache/scheduler.lock`),
acquired at the `adaptive-node.js` `main()` CLI boundary — after the worktree-authority-
split guard, before the layered guard prologue and the entire mutating subcommand body —
for exactly the `SPLIT_GUARDED_SUBCOMMANDS` set (`open-next`, `open-ready`, `close-node`,
`close-and-open-next`, `reconcile-running-set`, `write-halt`, `clear-halt`, `reopen-node`,
`revert-overflow`, `repair-node`, `route-findings`, `discard-speculative`), released in a
`finally` around the dispatch. Read-only subcommands (`orient`, `mirror-project`,
`record-evidence --verify`) stay lock-free, matching the existing worktree-authority-split
exemption set.

- **Claim primitive:** `fs.openSync(lockPath, 'wx')` (O_EXCL | O_CREAT) — the dependency-free
  Node-core primitive that fails `EEXIST` if a holder exists. On success the payload
  `{pid, host, ts, subcommand}` is written and `fsync`ed before close.
- **Contention is a typed, NON-blocking refusal** — one serial orchestrator is the designed
  model, so there is no spin-wait or queue. On `EEXIST` the holder is CLASSIFIED
  (`isStaleLock`: a dead same-host PID via `process.kill(pid,0)` ESRCH is stale; a live PID
  or `EPERM` is live; a cross-host or corrupt/pidless payload falls back to
  `LANE_STALENESS_MS` age) and refused as `scheduler_locked` (live) or `scheduler_lock_stale`
  (dead/aged) — never a fabricated pass.
- **Fail-closed, no auto-takeover.** `acquireProjectLock` NEVER unlinks another process's
  lock, even for a lock classified stale. Recovery from a genuinely dead holder is ONE
  explicit operator `rm "<lockPath>"` (named literally in the `scheduler_lock_stale`
  `operator_hint`, along with the holder's subcommand/pid/host/since), then a re-run.
- **Release:** the returned `release()` unlinks OUR OWN lock (idempotent, swallows `ENOENT`)
  and clears the held-lock marker; a module-level `process.on('exit')` hook is a
  belt-and-suspenders backstop for a crash that skips the caller's `finally`.

### Rejected: auto-takeover of a stale lock

The FIRST implementation included an auto-takeover tail on the stale branch: re-read the
lock to confirm the stale decision still holds, `unlinkSync` it, then retry `wx` ONCE. This
was empirically REFUTED by the in-run adversarial gate using the SAME race harness that
proved the two lost-update races above: the unlink executes a stale VERDICT made BEFORE the
unlink, so two concurrent takers who both observed the same stale holder both unlink-and-
reclaim — 6/500 double-acquisitions at 2 concurrent takers, 111/500 at 6, on the real
function. A double-acquire silently reopens the exact lost-update races this lock exists to
close. The issue's acceptance criteria explicitly permit "stale takeover OR typed operator
hint," so the fail-closed disjunct was available and chosen: auto-takeover was removed in
full (the unlink call site deleted, not merely guarded), replaced by the pure classify-and-
refuse contract described above. Re-verification against the same harness after the repair
showed 0/500 double-acquisitions at both N=2 and N=6.

### Rejected: verify-after-claim (CAS-lite)

Claim with `wx`, then re-read the just-written file to verify it is still our own payload
before proceeding. This does not close the takeover race — it only shrinks the window
between two racing UNLINKS, it does not remove the unlink itself. Any code path that can
unlink a lock it does not hold is unsafe under concurrency; the fix has to remove the
unlink-of-another's-lock capability entirely, not narrow its timing.

### Rejected: inode-verified rename

Verify the lockfile's inode number is unchanged immediately before an unlink-based
takeover (detecting a fresh claim that raced in under the same name). Rejected as
unnecessary complexity once auto-takeover itself was removed — there is no longer any
unlink-of-another's-lock code path to protect. It would also not be portable across every
target filesystem class without extra caveats (see the shared-filesystem note below).

### Rejected: a dedicated takeover sub-lock

A second, shorter-lived lock guarding the takeover decision-then-unlink sequence
(effectively a lock-for-the-lock). Rejected: it only moves the identical race down one
level — two takers can race for the sub-lock exactly as they raced for the primary lock —
and it doubles the lock-file surface (two files, two cleanup paths, two staleness
classifiers) for no closed hazard.

### Why not `flock(2)` / `fcntl(2)` advisory locking

Node core has no `flock`/`fcntl` binding; using either would require a native addon or a
third-party dependency, which the workflow scripts' dependency-free Node-core-only posture
does not carry. `O_EXCL` on `open()` is the one atomic claim primitive available from Node
core alone (`fs.openSync(path, 'wx')`), and it is sufficient for this contract: the failure
mode this lock exists to prevent is two PROCESSES racing to become the sole scheduler
holder, not byte-range locking within one already-held session.

## Consequences

- Recovery from a crashed holder is always ONE explicit, informed operator action
  (`rm "<lockPath>"` named in the hint) from a single session — never automatic, never a
  silent takeover.
- **Residual accepted risks** (all resolve via the same manual `rm`, none risk a
  double-acquire):
  - **PID reuse.** If the OS recycles the dead holder's PID before the operator notices, the
    live-PID check reports `alive` and the refusal flavor is `scheduler_locked` instead of
    `scheduler_lock_stale` until the recycled-pid process itself exits — a wrong HINT, not a
    wrong acquire.
  - **Create-then-write-failure orphan window.** A lock created by `openSync('wx')` but
    never fully written (e.g. the process is killed between open and write) is classified by
    the lockfile's mtime rather than a parsed payload; a just-created empty file reads
    `stale:false` (protecting a holder genuinely mid-write) for up to `LANE_STALENESS_MS`
    before the age fallback flips it stale — again a wrong-flavor refusal window, not a
    double-acquire.
  - **Operator-misuse cross-unlink.** Nothing prevents an operator from `rm`-ing a lock that
    is NOT actually dead (ignoring the hint's verification instruction); this is inherent to
    any manually-recoverable lockfile design and is mitigated only by the hint's explicit
    "verify no other orchestrator session is recovering this project ... from ONE session
    only" wording, not by machinery.
  - **Shared-filesystem `O_EXCL` caveat.** `O_EXCL` is not guaranteed atomic over NFSv2 (a
    single-host model is assumed here, consistent with the rest of the adaptive-path
    scripts).
- Single-orchestrator serial invocation — the designed model — is behaviorally unchanged:
  zero contention means a clean acquire every time, so all pre-existing assertions and the
  walkthrough stay green.
- `.cache/scheduler.lock` is barrier-exempt (the `kaola-workflow/` prefix allowband) and is
  never itself durable state — see `docs/workflow-state-contract.md`.
