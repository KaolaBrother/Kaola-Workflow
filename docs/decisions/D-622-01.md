# D-622-01. Relax write-lane exclusivity to a leg-contained invariant, fence the last-member merge against live reads, exclude speculative writes while a group is live, and seed tracked evidence stubs at group formation

Date: 2026-07-07
Status: Accepted
Issue: #622 (primary — leg-contained write exclusivity relaxation + last-member merge fence);
       #633 (companion — tracked-evidence-seeding collision fix, same bundle)
Related: R4 (in-run adversarial-verifier finding against the #622 fix, repaired within the same
         bundle before shipping — the `lane_group_live` speculative-write exclusion below); #621
         (baseline-first ordering extended to `close-and-open-next`'s fused advance and
         `runReopenNode`, same bundle, independent fix — see `docs/decisions/D-590-01.md`, the
         original `open-next` instance of the same invariant); #596 (the pre-existing speculative-
         write fallback this decision's R4 arm patches); #542/D-542-01 (default-on disjoint write
         lane-group co-open, the mechanism this decision relaxes); #437/#419 (lane-group foundation:
         group baseline, group-scoped close barrier, `running-set.json` schema)

## Context

Before this decision, `runOpenReady`'s `write_node_exclusive` check refused **any** co-open —
including a pure read frontier — whenever **any** write node was live in the running set,
regardless of whether that write was a legless single serial writer or a member of an already
leg-isolated lane group (D-437-01/D-542-01: each lane-group member runs against its own
`.kw/legs/<project>/<node-id>` worktree, never the shared parent tree). That blanket exclusion was
correct for the legless case (a serial writer commits straight to the parent worktree, so nothing
else may run) but unnecessarily conservative for a leg-contained write: a read cannot observe or
interfere with a leg member's isolated edits, so there was no correctness reason to make it wait.

Relaxing that exclusivity check surfaces two obligations that the strict pre-#622 behavior had
made moot:

1. **The eventual group merge must not race a still-live read.** A lane group's last member
   triggers an octopus-merge of the disjoint legs into the parent branch. If a read is allowed to
   open alongside a live group, that read could still be running (reading the pre-merge parent
   tree) at the moment the last member tries to merge.
2. **The pre-existing #596 speculative-write fallback becomes reachable in a new state.** That
   fallback fires when the normal frontier is empty and a write candidate's only unsatisfied
   dependency is an open gate; before #622, `write_node_exclusive` made it structurally impossible
   to reach this fallback while a write lane group was still live (a read gate could never co-open
   alongside a live write in the first place). Once reads can co-open alongside a live group, a
   dependent write can become `speculativePending` while that group is still live. An
   adversarial-verifier pass on the #622 diff (finding **R4**, blocking/high-severity) caught that
   the speculative-write branch's group-formation path did not account for this: it would form the
   speculative candidate its own size-1 lane group and unconditionally overwrite
   `running-set.json`'s single `lane_group` key with the new descriptor — discarding the live
   group's tracking of its still-open member(s) entirely. That member's later `close-node` call
   then no longer recognized itself as a lane-group member, fell through to the serial close path,
   and its committed in-lane leg work never merged to `HEAD` (silently lost), leaking its leg
   worktree. This was caught and fixed within the same bundle, before shipping — never released in
   the vulnerable shape.

Independently, in the same bundle, issue **#633** found a companion correctness gap in how a lane
group's evidence files cross the leg boundary. A write-role member self-writes its own
`.cache/{node-id}.md` evidence **inside its own leg** (the absolute `dispatch.leg_path` its
working_dir names), never syncing it back to the parent. Before this fix, the parent's copy of
that same relative path was seeded as a plain, untracked filesystem write. The eventual
last-member octopus-merge could then fail with git's "untracked working tree file … would be
overwritten by merge" collision if the parent's copy was still untracked at merge time.

## Decision

**Fix all three at their structural root inside `scripts/kaola-workflow-adaptive-node.js`, rather
than adding parallel checks alongside the existing lane-group machinery.**

1. **Relax `write_node_exclusive` to a leg-contained invariant (#622).** `runOpenReady` (function
   starting ~L4186) now computes `liveGroupId` (the live `lane_group.group_id`, if any) and
   `isLegContainedWrite(n)` (true iff `n.group_id === liveGroupId`), then derives
   `liveHasLeglessWrite = liveNodes.some(n => n.kind === 'write' && !isLegContainedWrite(n))`
   (~L4238). Only a live **legless** write (no lane group — the plain single-writer serial path)
   still refuses `write_node_exclusive` (~L4241); a live **leg-contained** lane-group member no
   longer excludes a read (or another dependent) from opening alongside it. The pre-existing
   `!liveHasWrite` gate the #596 speculative fallback already relied on is renamed to
   `!liveHasLeglessWrite` (same semantics, same call site) so it, too, only excludes on a legless
   live write.
2. **Fence the last-member merge against live reads (#622).** `closeGroupMember` (~L5081) adds a
   **LAST-MEMBER MERGE FENCE** (~L5199): immediately before the group-scoped merge/barrier, it
   refuses — typed, zero-mutation — with reason `merge_awaits_read_drain` while any live
   running-set member is `kind: 'read'`. The caller retries `close-node` once the live read(s)
   close. A legless or read-free group (the pre-#622 shape) never has a live read at this point, so
   the fence is a no-op there — byte-identical.
3. **R4 repair: exclude speculative writes while a group is already live.** `running-set.json`
   carries at most **one** `lane_group` descriptor at a time. `runOpenReady`'s speculative-write
   branch (~L4356) adds an `else if (liveGroupId)` arm — sibling to the pre-existing
   `no_leg_capability` and `parent_dirty` exclusions — that excludes **all** speculative write
   candidates with a typed reason `lane_group_live` (surfaced as
   `speculativeWriteExcluded: { reason: 'lane_group_live', nodeIds: [...] }`, top-level
   `reason: 'speculative_write_excluded'`) whenever a write lane group is already live. This closes
   the R4 hole: a new speculative group can never form — and therefore can never overwrite the live
   descriptor — while one is already open. The excluded write simply waits; once the live group
   fully drains and clears `lane_group` entirely, it opens normally (non-speculatively) on a later
   tick, at no cost to reachable parallelism.
4. **Seed and commit tracked evidence stubs at group formation (#633).** A new `legMirrorPath(legPath,
   absPath)` helper (~L3811) resolves a leg's own copy of a repo-root-relative path. At lane-group
   formation, `runOpenReady`'s **TRACKED-EVIDENCE-SEEDING** step (~L4535) seeds every `toOpen`
   member's evidence stub and commits it (`git add` + `kw-stub: <group_id>` commit) **on the parent
   branch, BEFORE `baseRev` is captured and legs branch off** — so every leg inherits the stub as an
   ordinary tracked file, and the leg member's real self-written content later merges in as a
   routine three-way change rather than colliding with an untracked parent file. A commit failure
   here refuses the open (`stub_commit_failed`) rather than silently leaving the pre-#633 collision
   risk in place. Correspondingly, `runCloseNode`'s evidence read (~L4897) now resolves the node's
   current `lane_group` membership **first** (a pure hoist of the existing `readRunningSet` call)
   and prefers a live member's **own leg copy** of `.cache/{node-id}.md` (via `legMirrorPath`) when
   present, falling back to the parent's copy otherwise — byte-identical for every non-lane-group
   case (a legless write, a read node, or a harness that seeds evidence parent-side by convention).

## Consequences

- A read (or another dependent gate) can now open **concurrently** with a live leg-contained write
  lane-group member; only a legless single serial writer still runs strictly alone. This recovers
  parallelism the pre-#622 blanket `write_node_exclusive` check needlessly forfeited whenever a
  disjoint-write lane group happened to still be open.
- `running-set.json` still carries **at most one** `lane_group` descriptor at any time — the R4
  exclusion is what makes that invariant hold once reads can co-open alongside a live group; a
  speculative write can never silently orphan a still-open member's tracking.
- The group's last-member merge can no longer race a live read against the pre-merge parent tree;
  it holds (zero-mutation, retryable) until the read(s) drain.
- A lane-group member's self-written leg evidence can no longer trigger an untracked-file merge
  collision at the last-member merge; the parent's stub stays a clean, tracked, never-touched-again
  file through to that merge.
- Validated end-to-end with a real-git-repo RED→GREEN regression test reproducing the exact R4
  interleaving (`test-adaptive-node.js`, `R4-REGRESSION` block: co-open → defer → read co-opens
  alongside the live member → the regression tick), a from-scratch adversarial falsification suite
  (84 assertions, could not break the fix), and the full `test-adaptive-node.js` suite (1478/1478),
  `simulate-workflow-walkthrough.js`, and cross-edition byte-parity (`edition-sync.js --check`).

## Non-goals

- **No change to legless write-node exclusivity.** A single serial writer (no lane group formed —
  a genuine exact-path overlap, a non-exactly-resolvable coarse pair, the `KAOLA_PARALLEL_WRITES=0`
  opt-out, or a host without worktree support) still runs strictly alone; nothing may co-open
  alongside it. `write_awaits_drain` (a write waiting for live reads to drain before it can run
  alone) is unaffected.
- **No change to genuinely-overlapping (`exact`) write serialization**, fan-out caps, or the
  `parent_dirty` / `no_leg_capability` speculative-write exclusions this decision's `lane_group_live`
  arm sits beside.
- **No new `running-set.json` schema field.** `lane_group` and its `legs` sub-schema are unchanged;
  this decision only changes *when* `write_node_exclusive` fires and *which* candidates the
  speculative-write branch admits.
- **#621 (baseline-first ordering) is a separate, independently-landed fix in the same bundle**,
  extending the existing `open-next` invariant (D-590-01) to `close-and-open-next`'s fused advance
  and `runReopenNode`. It touches the same file but a disjoint code path and is out of this
  decision's scope.

## Alternatives considered

- **Guard the running-set write-descriptor *assignment* itself** (make the `groupForm ? {...} :
  existingLaneGroup` ternary refuse to overwrite an already-live descriptor) instead of excluding
  the speculative candidate at selection time. Rejected: exclusion-at-selection mirrors the
  established sibling `no_leg_capability`/`parent_dirty` pattern exactly, and short-circuits
  *before* any leg provisioning or baseline recording happens for the excluded candidate — avoiding
  a provision-then-discard unwind that guarding the assignment site alone would still require after
  the fact.
- **Leave `write_node_exclusive` fully strict and seek parallelism elsewhere.** Rejected: leg
  containment already guarantees a read cannot interfere with an isolated write lane-group member,
  so the strict pre-#622 exclusion forfeited genuinely-safe concurrency for no correctness benefit;
  the merge fence (item 2 above) supplies the one safety obligation the relaxation actually needed.
- **For #633, sync leg-written evidence back to the parent immediately after each member's dispatch**
  rather than seeding+committing a tracked stub up front. Rejected: nothing else in the running-set
  scheduler observes a leg's contents before its member closes, so this would require a new
  poll/sync mechanism; seeding a tracked stub before the legs branch off is a one-time, one-line
  fix at the single point of leg creation, and the close-side leg-preferred read is symmetric with
  it.
