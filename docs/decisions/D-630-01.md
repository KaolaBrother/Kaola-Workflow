# D-630-01. Generate the plan-run/next routing surfaces from one canonical skeleton; guard finalize with a single-source presence manifest instead

Date: 2026-07-08
Status: Accepted
Issue: #630 (primary — the routing-surface generation seam)
Related: #400 (the six-surface propagation rule this decision automates for 12 of the 18 surfaces);
         #369/#380 (prior 4-of-6 propagation-by-copy drift); #624 (a whole finalize gate block lost
         on the forge-codex SKILLs while all four chains stayed green — the class this decision
         closes); #636/D-636-01 (the dead-prose-fencing prerequisite that shipped first and set up
         the `<!-- PIN: teammate-mode -->` / `<!-- PIN: codex-dispatch -->` slot boundaries this
         decision's generator relies on); #627/D-627-01 (the debloat bundle that restored the
         skeleton target and descoped the fencing work to #636);
         `docs/investigations/2026-07-08-630-636-routing-generation-seam.md` (the read-only shaping
         run that settled this decision's two-layer design)

## Context

The six #400 routing surfaces per topic (3 Claude commands + 3 Codex SKILL packs, across the
GitHub/GitLab/Gitea editions) were kept in sync by hand-copy propagation. This is a documented
drift class: #369 (`--issue-numbers`) and #380 (auto-bundle restructure) each shipped reaching the
commands plus the github-codex SKILL but not the two forge-codex SKILLs (the "×4" symptom), and
#624 lost an entire finalize gate block on the forge-codex SKILLs while all four
`npm run test:kaola-workflow:*` chains stayed green — the token-pin regime asserted presence of
individual tokens but never derived the full set of surfaces a block was obligated to appear on,
so a block could be silently dropped from a subset without any pin noticing. #630 asks to generate
the routing surfaces from one canonical source, killing propagation-by-copy drift at its root.

The design was settled by a read-only shaping run
(`docs/investigations/2026-07-08-630-636-routing-generation-seam.md`), which also identified that
the plan-run and next command/SKILL forms are close enough in structure to generate byte-for-byte
from a shared skeleton, while the finalize command/SKILL forms diverge roughly 2:1 — making
finalize a poor byte-generation target. The #636 prerequisite (relocating the cross-runtime
dispatch-pin assertions so dead prose could be fenced off the plan-run surfaces, D-636-01) shipped
first, since a generator cannot single-source a block that is unconditionally asserted on both its
native and non-native surfaces.

## Decision

**A two-layer design**, scoped differently per topic:

- **Layer 1 — presence, all 18 surfaces including finalize.** `templates/routing/required-blocks.js`
  is a single-source required-block manifest: each routing-prose block is declared once, tagged
  with its topic, a `runtime_tag` (`claude-live` / `codex-live` / `both`), a `surface_type_tag`
  (`command` / `skill` / `both`), and its distinctive content tokens. A derived-universe presence
  checker added to `scripts/test-route-reachability.js` computes each block's obligated file set
  from those tags crossed with the existing per-edition directory tables (never a hand-typed file
  list), then asserts every obligated file contains every token. Because the obligated set is
  *computed*, a block cannot structurally obligate a subset of its true surface set — silently
  dropping a routing block from one or more of its required surfaces is impossible by construction.
  A bidirectional orphan-sentinel also scans every in-scope surface for a `<!-- PIN: -->` /
  `<!-- CARD: -->` marker with no matching manifest block (self-disarm protection) and flags any
  manifest block whose tags are internally inconsistent.
- **Layer 2 — byte, the 12 template-shaped plan-run/next surfaces only.** A prose generator
  (`scripts/generate-routing-surfaces.js`) renders each of the 12 surfaces from one canonical
  skeleton per topic (`templates/routing/plan-run.skeleton.md`, `templates/routing/next.skeleton.md`)
  plus `templates/routing/slots.js` (frontmatter, H1, the setup-resolver bash block, and
  runtime-conditional whole-section regions), sub-sentence splices for mid-paragraph divergences,
  and `templates/routing/rename-table.js` (forge-noun renames). A `--check` mode renders in-memory
  and byte-compares against the committed surface (exit 1 with a diff on mismatch); `--write`
  renders and writes. `--check` is wired into all four `npm run test:kaola-workflow:*` chains.
  **Finalize stays Layer-1-only** — hand-authored, guarded only by the manifest — because its
  command/SKILL forms diverge enough (~2:1) that forcing byte-generation would itself be a
  precedence-#1 accuracy risk (a rushed, error-prone rewrite of surfaces that already work).

**Shipped as a behavior-preserving no-op capture.** The two skeletons, their slots, and the rename
table were reverse-engineered from the current committed surfaces so that `--write` reproduces all
12 generated surfaces byte-for-byte — the diff for this issue changes no routing-surface content;
the 12 surfaces are byte-unchanged. Both gates live-proved the guarantee each layer adds: a planted
whole-block drop (deleting a manifest-tagged block including its marker) reds the Layer-1 manifest;
a planted unpinned-prose mutation on a generated surface reds `--check` even though the pre-existing
token-pin regime stayed green, proving Layer 2 catches a drift class Layer 1 alone cannot.

## Consequences

- Future edits to the 12 plan-run/next surfaces flow through one canonical skeleton (plus slots and
  the rename table) and `--write`; hand-editing a generated surface directly now reds its own
  `npm run test:kaola-workflow:*` chain via `--check`, rather than silently drifting.
- All 18 routing surfaces (including finalize) are covered by the Layer-1 manifest's derived-universe
  presence checker, closing the #624 whole-block-drop class by construction rather than by an
  ever-growing set of hand-maintained pins.
- The pre-existing token pins in `scripts/test-route-reachability.js` and all four
  `validate-*-contracts.js` (including the deliberate GitLab/Gitea `mr|pr)` finalize-sink pins) are
  untouched; the manifest is an additive superset over them, not a replacement.
- Cross-edition #307: this diff touches the edition trees and all four contract validators; all four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains were run and are green.

## Non-goals / accepted residual

- **Finalize is not byte-generated.** A present-but-wrong-prose defect on a finalize surface,
  outside its manifest-pinned tokens, remains uncaught by this decision — an accepted residual,
  equal in strength to the pre-#630 status quo (not a regression). It is tracked as a follow-up to
  add a distinctive interior content token to the `fn-closure-audit` manifest block, since that
  block's current second token is a substring of its own marker and so does not by itself detect an
  interior gut of the block that leaves the marker in place.
- **The deliberate GitLab/Gitea `mr|pr)` finalize-sink contract pins are untouched** — an unrelated,
  intentional contract family outside this decision's scope.

## Alternatives considered

- **Byte-generate all 18 surfaces, including finalize.** Rejected: finalize's command/SKILL forms
  diverge enough (~2:1) that forcing a shared skeleton would require a substantial rewrite of
  surfaces that already work correctly — a precedence-#1 accuracy risk for no drift-prevention gain
  proportional to the risk, given Layer 1 already closes the whole-block-drop class on finalize.
- **Bare-marker presence sentinels (assert a marker exists, not its content tokens).** Rejected: this
  re-admits exactly the hollowed-block-yet-green state the manifest is meant to close — a marker can
  survive while the content behind it is gutted.
- **Leave routing-surface propagation by hand-copy.** Rejected: this is the #624 root cause and the
  #369/#380 "×4" drift pattern this decision exists to end.
