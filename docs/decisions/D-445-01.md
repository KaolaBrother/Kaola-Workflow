## Problem

Two distinct operator-ergonomics gaps in the adaptive plan-run surface, settled together because the
fix for both lands in the same ×6 surface pass.

**Gap 1 — bare typed envelopes.** Every typed refusal/halt envelope emitted by the four aggregators
(`adaptive-node.js`, `commit-node.js`, `plan-validator.js`, `parallel-batch.js`) carries a `result`
and a `reason`, but NO actionable next step. When an operator hits `result: refuse, reason:
write_set_overflow`, the envelope tells them WHAT failed but not WHAT TO DO. They must re-read the
~812-line plan-run prose to find the matching recovery branch. The structured `triage` /
`proposed_repair` payload (D-440-01) helps a machine consumer, but a human operator wants one
sentence and the exact next command.

**Gap 2 — residential prose bloat.** `plan-run.md` is ~812 lines. Most of those lines are
per-situation recovery branches (resume, governance, repair-routing, reopen-complete-node,
frontier-batch) that fire RARELY. The ×6 surfaces (3 Claude commands + 3 Codex SKILL packs, per the
#400 six-surface rule) each carry ALL this prose residentially — every routine plan-run pays the
full token cost of branches that almost never execute. The common path (open node → dispatch →
close → advance) is buried under rare-branch reference material.

Both gaps are settled here. The `operator_hint` field gives the human the one-sentence pointer at
emit time; the skeleton/card split moves the rare-branch prose OUT of the resident ×6 surfaces and
INTO `docs/plan-run-cards/`, leaving a ~150-line loop skeleton that drills into the relevant card
only when a refusal points to it.

## Decision

### 1. `OPERATOR_HINT_REGISTRY` — a per-aggregator `const` map, hints generated at emit time

Each of the four aggregators (`adaptive-node.js`, `commit-node.js`, `plan-validator.js`,
`parallel-batch.js`) gains a module-level `const OPERATOR_HINT_REGISTRY` — a map of
`reason → templateFn(ctx)`:

```js
const OPERATOR_HINT_REGISTRY = {
  write_set_overflow: (ctx) =>
    `Node ${ctx.nodeId} wrote outside its declared set. Run: node scripts/kaola-workflow-adaptive-node.js revert-overflow --node-id ${ctx.nodeId}`,
  barrier_failed: (ctx) =>
    `Barrier rejected node ${ctx.nodeId}. Review the offending paths, then revert-overflow or repair-node the writer.`,
  // ... one entry per typed reason the aggregator can emit
};
```

`getOperatorHint(reason, ctx)` is the single emit-time accessor: it looks up `reason` in the
registry, calls the template with the emit context, and returns the one-sentence string (or a safe
generic fallback when a reason has no registered template). Hints are **generated at emit time, not
stored** — the registry is the single source, and the string is materialized only when the envelope
is built. The registry lives INSIDE each aggregator (NOT a shared module): each script owns the
hints for the reasons IT emits, so the registry stays co-located with the emit sites and no new
cross-script import is introduced.

**Rationale.** A per-script `const` registry keyed by the typed `reason` makes the hint a pure
function of the envelope's own `reason` + context — the same structural-classification discipline the
emit envelope already uses (#406). Generating at emit time (not storing) means there is no second
durable artifact to keep in sync; the hint is always consistent with the `reason` it accompanies. Co-
location (vs. a shared module) avoids a new import edge into four scripts and keeps each registry's
entries auditable against that script's actual emit set.

### 2. Hint shape — `operator_hint: string`, top-level, on ALL typed outcomes

`operator_hint` is a `string` field added to the typed envelope at the SAME level as `result` /
`reason` — never nested inside `triage`, `proposed_repair`, or any sub-object:

```json
{
  "result": "refuse",
  "reason": "write_set_overflow",
  "operator_hint": "Node n4 wrote outside its declared set. Run: node scripts/kaola-workflow-adaptive-node.js revert-overflow --node-id n4",
  "nodeId": "n4"
}
```

It is present on ALL typed outcomes that carry a `reason`: `result: refuse`, `result: halt`, and
`result: warn`. A success envelope (`result: ok` / `ready_to_run` with no `reason`) carries no
`operator_hint` (there is nothing to act on). The field is **additive** — existing consumers that
read `result` / `reason` are unaffected; only consumers that WANT the hint read it.

**Rationale.** Top-level placement (next to `result`/`reason`) means a consumer reads the hint with
the same flat access it already uses for the classification — no traversal into a sub-object, no
dependence on whether a `triage` payload happens to be present. Presence on refuse/halt/warn (the
three actionable outcomes) and absence on success keeps the field's presence itself a signal:
`operator_hint` present ⟺ there is a next step.

### 3. Hint vocabulary contract — #424/#434 canonical primitives, NO drop-base, NO forge tokens

The hint strings are bound by the #424/#434 repair-primitive vocabulary:

- The `write_set_overflow` family (including its `write_set_granularity` / `lockfile_write` /
  `mirror_write` / `count_bump` subtypes, D-440-01) MUST reference `revert-overflow` — NEVER
  `drop-base`. `--drop-base` is the laundering anti-pattern locked by D-424-01; a hint that told an
  operator to drop the baseline would launder the writer's accumulated work. The canonical overflow
  recovery is `revert-overflow` (D-434-01 §1).
- A crash-repair / reopen-writer situation MUST reference `repair-node` (D-434-01 §2) — the anti-
  laundering primitive that keeps the original baseline (`baselineReused: true`).
- NO forge CLI token (`gh` / `glab` / `tea`) appears in ANY hint. The hints name `node scripts/...`
  workflow commands only; they are forge-neutral, consistent with the four aggregators being
  forge-neutral and shipping in all four editions.

**Rationale.** The hint is the operator-facing rendering of the recovery, so it MUST name the same
sanctioned primitive the structured `proposed_repair` (D-440-01 §2) names — one vocabulary across the
human channel (`operator_hint`) and the machine channel (`proposed_repair.kind`). Banning `drop-base`
from the hint vocabulary at the design level closes the laundering vector D-424-01 closed at the
script level. Forge-neutrality is non-negotiable because the registries ship byte-aligned across four
editions.

### 4. Skeleton content contract — ~150-line loop, verbatim pins + `--summary` + card markers

The ×6 plan-run surfaces are reduced to a ~150-line LOOP SKELETON: the common path (orient → open →
dispatch → record evidence → close-and-advance) plus the refusal-drill entry. The skeleton:

- **Retains the `frontier unit` literal verbatim** (the parallel-frontier vocabulary the route-
  reachability contract asserts) and the route-reachability pin (`validate-workflow-contracts.js`
  812-family). These survive the rewrite unchanged — the contract validators assert them.
- **Introduces a `--summary` mode reference** (D-446-01): the interactive loop runs the aggregators
  with `--summary`, reading the one-line summary, and drills into the cached
  `.cache/<op>-envelope.json` ONLY on `result: refuse`.
- **Introduces card-pointer markers** `<!-- CARD: <name> -->` immediately before the brief stub for
  each rare branch. A refusal whose `operator_hint` points at a recovery routes the operator to the
  named card under `docs/plan-run-cards/`.

**Rationale.** The skeleton carries the hot path resident; the cold paths become pointers. Keeping
the `frontier unit` literal + route-reachability pin verbatim is mandatory — the machine contract
(`scripts/test-route-reachability.js` + the four `validate-*-contracts.js`, #400) asserts these
tokens are present on every surface; dropping one in the slim-down would be a propagation regression.

### 5. Card scope — exactly 5 cards under `docs/plan-run-cards/`, ~100–200 lines each

The rare-branch prose moves to five reference cards:

| Card file | Covers |
|---|---|
| `resume.md` | Crash/interrupt resume — `--resume-check`, reconcile-running-set, baseline re-open |
| `governance.md` | Planner freeze/governance-ack handshake, `governance_ack_stale`, risk-assessment |
| `repair-routing.md` | `route-findings` consumption (D-446-01), `revert-overflow` / `repair-node` choice, plan-repair via `--freeze` |
| `reopen-complete-node.md` | Reopening a `complete` writer (`repair-node` vs `reopen-node`), baseline-reuse rules, the reopen-needs-allDone trap |
| `frontier-batch.md` | Parallel `frontier unit` fan-out — `open-batch` / `top-up` / `seal-member` / `seal` / `reconcile`, serial-degrade |

Each card is ~100–200 lines: enough to be self-contained for its situation, scoped to ONE recovery
family. The cards are reference material, NOT part of the ×6 resident surfaces — they live ONCE under
`docs/plan-run-cards/` and are pointed at by the skeleton's `<!-- CARD: <name> -->` markers.

**Rationale.** Five families map cleanly to the five rare-branch clusters in the current ~812-line
prose. One card per family (vs. one giant reference doc) lets the skeleton point at exactly the card
the situation needs. Living ONCE (not ×6) is the whole point of the diet — the cards are not subject
to the six-surface propagation rule because they are not part of the plan-run command surface; only
the skeleton is.

### 6. Route-reachability pin — `<!-- PIN: frontier unit -->` before the literal

The skeleton emits the comment marker `<!-- PIN: frontier unit -->` IMMEDIATELY before the `frontier
unit` literal, so the contract validators can assert the pin anchors the literal positionally. This
makes the route-reachability assertion robust against the slim-down: the validator finds the pin,
then asserts the literal directly follows, on every one of the ×6 surfaces.

**Rationale.** The slim-down moves prose around; without a positional anchor, a validator that
greps for `frontier unit` could be satisfied by an incidental mention. The `<!-- PIN: -->` marker
makes the assertion structural — the literal is where the pin says it is — so the route-reachability
contract survives the rewrite intact.

## Consequences

**Every typed refusal/halt/warn now strands no operator.** The `operator_hint` field (decisions 1–2)
puts one actionable sentence + the exact next command at the top level of the envelope, generated at
emit time from the per-aggregator `OPERATOR_HINT_REGISTRY`. The human channel (`operator_hint`) and
the machine channel (`proposed_repair`, D-440-01) name the SAME #424/#434 primitives (decision 3) —
`revert-overflow` for overflow, `repair-node` for crash-repair, never `drop-base`, never a forge
token.

**The ×6 plan-run surfaces lose ~660 lines of resident rare-branch prose.** The ~812-line prose
becomes a ~150-line skeleton (decision 4) plus five cards (decision 5) that live ONCE. The skeleton
keeps the `frontier unit` literal + route-reachability pin verbatim (decisions 4, 6) so the #400 /
route-reachability contract still passes on every surface. The skeleton/card rewrite rides the SAME
combined ×6 node as #438 (D-419 P3) — one node, the same six files.

**This is additive and compat-safe.** `operator_hint` is a new optional top-level field; existing
consumers reading `result`/`reason` are unaffected. The skeleton's interactive loop adopts
`--summary` (D-446-01), but the default FULL-JSON output of the aggregators is unchanged, so
orchestration scripts/tests that parse full JSON are unaffected.

**Test fixtures the implementation must cover (binding for the implementer):**

- Each of the four aggregators defines `OPERATOR_HINT_REGISTRY` and `getOperatorHint(reason, ctx)`;
  every typed `reason` the aggregator can emit has a registered template (or the documented generic
  fallback fires, never an empty string).
- A `write_set_overflow` (and each subtype: `write_set_granularity` / `lockfile_write` /
  `mirror_write` / `count_bump`) envelope carries an `operator_hint` referencing `revert-overflow`;
  NONE references `drop-base`.
- A crash-repair / reopen-writer envelope's `operator_hint` references `repair-node`.
- NO `operator_hint` string in any of the four registries contains `gh` / `glab` / `tea` (forge-
  neutrality), asserted across all four editions.
- `operator_hint` is top-level (sibling of `result`/`reason`), never nested; present on
  `refuse`/`halt`/`warn`, absent on a `reason`-less success envelope.
- The ~150-line skeleton on all ×6 surfaces contains `<!-- PIN: frontier unit -->` immediately
  followed by the `frontier unit` literal; `scripts/test-route-reachability.js` + the four
  `validate-*-contracts.js` pass on every surface.
- Exactly five cards exist under `docs/plan-run-cards/` (`resume.md`, `governance.md`,
  `repair-routing.md`, `reopen-complete-node.md`, `frontier-batch.md`), each ~100–200 lines; the
  skeleton carries a `<!-- CARD: <name> -->` marker for each.
- The skeleton references `--summary` mode (D-446-01) for the interactive loop and the drill-into-
  cached-envelope-on-refuse step.

**For downstream implementation:** the load-bearing constraints are decision 3 (the vocabulary
contract — `revert-overflow` / `repair-node`, never `drop-base`, never a forge token) and decisions
4/6 (the verbatim `frontier unit` + route-reachability pin survival). The skeleton/card split is a
prose move bound by the #400 six-surface rule and rides the #438 combined node; the cards themselves
live ONCE and are NOT six-surface-replicated.
