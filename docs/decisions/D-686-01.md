# D-686-01 — Barrier-ref reap at archive + a keep-set legacy sweep

Status: accepted
Date: 2026-07-15
Issue: #686

## Context

Every writer/reviewer node anchors its barrier baseline under a git ref so `git gc` cannot prune the
baseline commit while the node's window is open: `--record-base` writes
`refs/kaola-workflow/barrier/<sanitize(project)>/<node>` (plan-validator's `barrierRef`), and `--drop-base`
removes it — but `--drop-base` is window-locked to a `pending` node (D-424-01, the anti-laundering model),
so it only fires on the narrow drop-on-pass path. Archive (finalize-closed, discard-abandoned, or the
active-folders backstop) preserved the baseline `.cache` FILES but never reaped the refs, and no sweep
enumerated them. A 2026-07-14 audit found **239** stranded `refs/kaola-workflow/barrier/*` refs across ~42
archived projects. (The group baselines under `refs/kaola-workflow/barrier-base/*`, which drop when the
group barrier passes, sat at zero stranded — confirming the per-node refs were an omission, not a design
choice.)

## Decision

Two additions to `scripts/kaola-workflow-claim.js` (propagated to all four editions):

1. **Archive-time reap** — inside `archiveProjectDir` (the single convergence point for every archive path:
   finalize, discard, and the active-folders backstop), AFTER the live copy is deleted and the archive is
   verified, delete every `refs/kaola-workflow/barrier/<sanitize(project)>/*` ref (enumerate with
   `git for-each-ref`, delete each with `git update-ref -d`). **Fail-soft is correctness-critical:** the
   whole reap is wrapped so a ref-delete failure can never throw, block, or roll back the archive (the
   evidence is already archived) — a lost reap only leaves a dangling ref for the legacy sweep to collect.

2. **A one-shot legacy sweep** — a new `barrier-ref-sweep` CLI subcommand for the pre-fix stranded refs.
   It enumerates all `refs/kaola-workflow/barrier/<tag>/*`, groups by `<tag>`, builds a KEEP set, and deletes
   only refs whose tag is NOT kept. The KEEP discipline is deliberately conservative and mirrors the #680
   orphan-baseline sweep:

   - **Keep universe = every worktree.** The keep set is built across the union of ALL worktree roots from
     `git worktree list --porcelain -z` (main + every linked worktree + every `.kw/legs/` leg) unioned with
     the invoking root and the resolved main root. This is required because custom refs live in the shared
     git common dir while a live claim's project folder lives under whichever worktree root ran the claim —
     scoping the keep scan to only the invoking cwd (or only {root, mainRoot}) over-reaps a live sibling
     project claimed from another worktree.
   - **Byte-exact worktree parse.** `--porcelain -z` (NUL-separated) is used so a worktree path containing
     any character (embedded newline, trailing space) round-trips exactly; the path is taken as the bytes
     after `worktree ` with no trimming.
   - **Three keep signals**, case-folded: (a) an active-folder tag from `readActiveFolders` (network-free —
     the sweep never makes a `gh` call, so a forge outage can never flip a keep into a delete); (b) a project
     with a live `.cache/running-set.json`; (c) a project folder whose `workflow-state.md` EXISTS but cannot
     be read (unprovable-dead ⇒ keep — the file-granularity analog of the directory-granularity abort below).
   - **ADD-only collision discipline.** A sanitizer collision only ever ADDs to KEEP (fail-safe under-reap,
     never over-reap); any tag that cannot be probed is kept.
   - **Fail-closed on an unknowable keep universe.** If `git worktree list` itself throws, the sweep aborts
     and deletes nothing (`worktree_enum_failed`) — an unenumerable worktree set means no tag can be proven
     dead.
   - **Scope = `barrier/` only.** The reap and sweep touch exactly `refs/kaola-workflow/barrier/<tag>/*`.

## Ref namespaces and what this path does NOT touch

`refs/kaola-workflow/` carries three ref namespaces with three lifecycles:

- `refs/kaola-workflow/barrier/<tag>/<node>` — **per-node/writer baselines.** Anchored at open by
  `--record-base`; dropped on pass by the window-locked `--drop-base`; and now **reaped at archive + by the
  legacy sweep** (this decision).
- `refs/kaola-workflow/barrier-base/<group_id>` — **group (lane-group) baselines.** Dropped when the last
  member passes the group barrier, or on reconcile rollback. NOT part of the #686 reap.
- `refs/kaola-workflow/leg-base/<project>/<node>` — **leg baselines.** Torn down with the leg. NOT part of
  the #686 reap.

(`barrier-base-<sanitized>` without a `refs/` prefix is a `.cache/` FILE name, not a ref — do not confuse it
with the `refs/kaola-workflow/barrier-base/` group-ref namespace.)

## Safety campaign (adversarial verification)

The legacy sweep is a delete-refs operation, so its dangerous direction is deleting a LIVE project's
gc-anchor (which lets `git gc` prune the baseline commit, making the `barrier_base_mismatch` ref-restore
repair impossible). Adversarial verification drove four over-reap classes to closure, each reproduced
end-to-end and fixed RED-first:

- keep-universe too small (cwd-only, then {root,mainRoot}) → all-worktrees union;
- worktree-path misparse (embedded newline / trailing space via plain `--porcelain`) → `--porcelain -z`;
- a present-but-unreadable `workflow-state.md` dropping a live project → keep signal (c).

A final real-flow reachability pass (real claim / record-base / bundle / keep-open / repair-cycle / a
212-sweep concurrency hammer) found **no reachable over-reap**: every shipped flow writes `status: active`
before any ref can exist, and the write-state → ref-create ordering closes the sweep's TOCTOU window.

**Documented residual limitations** (out-of-band or hand-crafted; no shipped flow produces them):

- A live project whose project DIRECTORY is `chmod 000` (its state file unreachable through the unsearchable
  parent, so `existsSync` returns false) is still reaped. Filed #691 with the `statSync`-ENOENT-vs-fault fix.
- Two simultaneously-claimed hand-crafted aliasing `--project` names (e.g. `issue.500` vs `issue_500`) that
  sanitize to the same tag: the archive-reap of one deletes the other's anchor. The pre-existing ref keying
  already cross-breaks such a run upstream; no shipped name generator produces aliasing names. Candidate
  follow-up: refuse `claim --project` names that sanitize-collide with a live folder.
- A worktree moved by plain `mv` without `git worktree repair` reports a stale path in `git worktree list`;
  git's own worktree bookkeeping is stale identically, so this is inherited git degradation.

## Consequences

- Archived projects no longer strand their per-node barrier refs; the one-shot `barrier-ref-sweep` collects
  the ~239 pre-fix stranded refs safely (keep-everything-live discipline).
- `kaola-workflow-claim.js` is a COMMON_SCRIPTS rename-normalized port; the change ships in all four editions
  and passed all four `test:kaola-workflow:{claude,codex,gitlab,gitea}` chains.
