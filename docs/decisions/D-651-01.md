# D-651-01. Mechanical pre-tag release gate (`--release-check`), strictly sha-pinned and unwaived

Date: 2026-07-10
Status: Accepted
Issue: #651
Related: #432/#475 (dual-mode finalize gate — the shape this gate mirrors); #547/D-547-01
         (codeTreeHash freshness relaxation — deliberately NOT reused here); #618 (chains_empty
         precedence tier this gate reuses); #635/#648/D-648-01 (chain-receipt format and
         hint-only `stale_paths`/`stale_kind` culprit diagnostics, reused as-is); #632/D-632-01
         (existing decision that `--cut` stays informational-only on chain greenness — not
         reversed by this record); #602/#630 (Self-Sufficient by Default — no CI/CD coupling)

## Context

Tag `kaola-workflow--v6.21.3` was cut at a commit that, empirically re-run in a scratch clone
during a 2026-07-10 post-ship audit, had a RED Claude chain (`test-adaptive-node.js` 46 FAILs,
`simulate-workflow-walkthrough.js` failing at #611 AC5) and zero recorded validation receipt. The
release procedure that was supposed to prevent this was prose-only — `CLAUDE.md` Validation
Policy and `docs/conventions.md` release sequencing described the required chain-receipt
discipline, but nothing mechanical refused tagging a commit with no fresh, unwaived, four-chain
receipt at that exact sha. The existing receipt machinery (`kaola-workflow-run-chains.js`
receipts, the plan-validator's `--finalize-check`, the #648 `chains_stale` culprit diagnostics)
guards only the adaptive finalize path; a hand-rolled release commit bypasses it entirely.

A second, independent process gap in the same tag: the 100-file release commit bundled an
unrelated BREAKING evidence-shape hardening under the subject "pin standalone subagent
profiles" — the half that broke the gate was invisible in the commit subject and buried at the
tail of the CHANGELOG entry.

## Decision

1. **New check-only gate, same family as `--finalize-check`.** `releaseCheck(args)` in
   `scripts/kaola-workflow-plan-validator.js`, invoked via `--release-check`, is a check-only,
   PLAN-INDEPENDENT twin of the `--finalize-check` chain-receipt arm — intercepted in `main()`
   BEFORE the plan path is read, since at release time the adaptive run is already archived and
   no `workflow-plan.md` exists. It reuses the receipt reader, the `getOperatorHint` registry
   (new `chains_waived` entry), and the #648 `attachChainsStaleDiagnostics` hint-only culprit
   scan verbatim rather than re-implementing any of them.
2. **Typed precedence, structurally classified (extended by the R2 addendum below):**
   `chains_unverified` (missing/unparseable receipt) > `chains_stale` (headSha
   unbound/`'unknown'`/missing, headSha != candidate, or the receipt stamped over a dirty
   worktree) > `chains_empty` (zero chains recorded — same tier #618 established for
   `--finalize-check`) > `repo_kind_undetermined` (expected chain set unresolvable) >
   `chains_incomplete` (receipt does not cover every declared chain) > `chains_red` (an unwaived
   red chain) > `chains_waived` (**new** reason — see below). The two coverage tiers
   (`repo_kind_undetermined`'s new call site and `chains_incomplete`) were added in a
   post-review repair window after an adversarial pass found a subset receipt passing the base
   gate — see the Addendum at the end of this record. Every consumer (the walkthrough's 13
   cases) classifies on the `reason` field, never on refusal prose.
3. **STRICT sha equality, no freshness relaxation.** The candidate defaults to `HEAD` and accepts
   an explicit `--candidate <sha-ish>`, normalized via `git rev-parse --verify <arg>^{commit}`;
   an unresolvable candidate fails CLOSED into the stale arm. The receipt's `headSha` must equal
   the candidate EXACTLY. The #547 `codeTreeHash` content-address relaxation — legitimate at
   adaptive finalize, where an inert doc-only commit since the stamp should not force a re-run —
   is deliberately NOT applied: a release tag names one exact commit, so only that commit's
   receipt counts.
4. **`headSha: 'unknown'` refuses; `kaola-workflow-release.js`'s own leniency is not copied.**
   `chainReceiptGreenness` (`kaola-workflow-release.js:254`) treats an unbound `headSha` as
   green for its informational `--verify` warning. This gate treats the same value as a hard
   refusal — an unbound receipt proves nothing about the release candidate, and a release tag is
   a stricter contract than an informational warning.
5. **A dirty-stamped receipt refuses**, beyond the four literal precedence tiers above: when
   `workTreeHash !== 'clean'`, the chains validated the stamped commit PLUS uncommitted edits,
   not the tree the tag would name, so the receipt cannot stand in for the candidate. `run-chains`
   always writes this field (`'clean'` or a diff hash), so a modern receipt is never falsely
   refused; a legacy or hand-edited receipt missing the field fails closed under the same reason.
6. **A waived chain is a hard refusal — the base window's one new reason code.**
   `chains_waived` fires on ANY `accepted_red: true` chain in the receipt. A waiver
   (`--accept-known-red <name>:<issue>`) is legal at adaptive finalize, where an open tracking
   issue documents the accepted risk for that run; it is never legal for a release tag, which
   represents the whole public release surface as green. This was the only new `reason` value
   the base window introduced (every other base-window tier reuses an existing precedence name);
   a second new reason, `chains_incomplete`, was added in the R2 repair window — see the
   Addendum.
7. **Self-owned, per Self-Sufficient by Default.** The gate reads only the receipt file and
   local `git`; it makes no forge or network call and mutates nothing (stdout + exit code only).
8. **Wired into the documented sequence, not into `--cut`'s own code — non-reversal of
   D-632-01.** `docs/conventions.md` documents the pre-tag sequence: bump commit → stamp a
   receipt at the candidate → `--release-check` passes → tag → online `npm test` → push. This is
   a documentation-level wiring: `kaola-workflow-release.js` itself is untouched by this issue.
   D-632-01's decision that `--cut` stays informational-only on chain greenness is **not
   reversed** — the same structural reason applies (the receipt that would gate `--cut` predates
   the tag `--cut` creates internally as its own step). `--release-check` is a separate,
   script-independent step an operator (or future automation) runs in front of `--cut`, not a
   change to what `--cut` reads.
9. **Release-commit hygiene rule recorded alongside the gate (AC5).** A release/tag commit is
   version bump plus release docs only (`package.json`, the Codex/Claude-plugin manifests,
   `CHANGELOG.md`, README release-version lines) — never unrelated behavior-changing code.
   Bundling more, as the v6.21.3 commit did, invalidates whatever receipt was checked (the
   receipt no longer describes the tree the tag actually names) and re-runs the whole sequence.
   This is a documentation rule (`docs/conventions.md`), not a mechanically enforced check — no
   commit-content classifier is added by this issue.

Validated with 10 new `simulate-workflow-walkthrough.js` cases in the base window (pass; missing
receipt; unparseable receipt; older-sha stale with `stale_paths`/`stale_kind` hints; unbound
`headSha` refuses; `chains_waived` naming the waived chain; `chains_red` precedence over a
waived sibling; `chains_empty`; explicit `--candidate` pass-then-stale-at-advanced-HEAD;
dirty-stamp refuses), written RED-first against the pre-implementation validator — 3 more cases
(11)-(13) were added in the R2 repair window, bringing #651 walkthrough coverage to 13; see the
Addendum. Ported to all three generated editions via `npm run sync:editions` (byte-identical).
All four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains green.

## Consequences

- A release tag can no longer be cut past a missing, stale, red, or waived receipt without an
  explicit, typed refusal naming the exact remedy.
- The gate is additive: it introduces one new CLI mode and two new `reason` values
  (`chains_waived` in the base window, `chains_incomplete` in the R2 repair — see Addendum); no
  existing `--finalize-check`, `--barrier-check`, or other plan-validator mode's behavior
  changes.
- `kaola-workflow-release.js` (`--verify`/`--cut`/`--push`) is unchanged; its own informational
  `chain_warning` at `--verify` time and D-632-01's non-gating `--cut` behavior stand as before.
- The gate is advisory to run, not self-invoking — nothing currently calls `--release-check`
  automatically before a manual `git tag` or before `--cut`; it is a documented step an operator
  (or future release tooling) must invoke.

## Non-goals

- Does not make `--cut` read chain greenness or call `--release-check` internally — see point 8.
- Does not mechanically enforce the release-commit hygiene rule (point 9) via a diff classifier;
  it is a documented, human-followed rule.
- Does not change the `.cache/chain-receipt.json` schema, `kaola-workflow-run-chains.js`'s chain
  execution semantics, or the `--finalize-check` precedence family it mirrors.
- Does not add CI/CD coupling of any kind (Self-Sufficient by Default is unaffected).

## Alternatives considered

- **Gate `--cut` itself on chain greenness.** Rejected for the same structural reason D-632-01
  already recorded: at the moment `--cut` runs, the receipt that would gate it predates the tag
  `--cut` is about to create, so gating `--cut` internally would break the documented flow rather
  than merely add a check. A separate pre-`--cut` step composes without that conflict.
- **Reuse the #547 `codeTreeHash` freshness relaxation instead of a strict sha pin.** Rejected: a
  release tag names one exact commit as the release surface; a content-address match that
  tolerates commit drift (legitimate for the adaptive finalize's "no test-consumed change since
  the stamp" case) would let a receipt stamped at a *different* commit with identical
  code-relevant content pass for a release tag it was never run against.

## Addendum (R2 repair window, 2026-07-10): coverage-before-greenness arm

An adversarial-verifier pass after the base window above found that a subset receipt — a
legitimate `kaola-workflow-run-chains.js --chains claude` producer output — PASSED
`--release-check`: the base gate checked only the chains present in `chains[]`, never that the
receipt covered every declared chain. This was a genuine gap: a release tag could be cut off a
receipt that only ever ran one of the four edition chains.

**Fix — a coverage arm placed between `chains_empty` and `chains_red`.**

- `releaseCheck` resolves the release candidate's expected chain set from
  `<git-toplevel>/package.json`: `['claude','codex','gitlab','gitea'].filter(n => typeof
  scripts['test:kaola-workflow:'+n] === 'string')` — the identical predicate
  `kaola-workflow-run-chains.js`'s own chain resolution and the `--finalize-check` repo-kind
  discriminator already use, so producer and gate can never disagree about what "the full chain
  set" means.
- **`chains_incomplete`** (new reason): the receipt's `chains[]` names are a strict subset of the
  resolved expected set. The refuse envelope carries structural `missingChains: string[]` and
  `expectedChains: string[]` (plus a registry `operator_hint` naming the missing chains) — e.g.
  `{"missingChains":["codex","gitlab","gitea"],"expectedChains":["claude","codex","gitlab","gitea"]}`
  for a claude-only receipt.
- **`repo_kind_undetermined`** (reused reason, new call site): when the expected chain set itself
  cannot be resolved — `package.json` missing, unreadable, unparseable, or declaring zero
  `test:kaola-workflow:*` scripts — the gate fails CLOSED rather than treat an unresolvable set
  as vacuously covered (a fail-open: an empty expected set would let any receipt "cover" it).
  This is deliberately STRICTER than `--finalize-check`'s ENOENT→consumer-mode downgrade for the
  same missing-manifest case, because a release is self-host-by-definition — there is no
  legitimate "consumer repo" release-tag case to degrade into.
- **Coverage sits BEFORE greenness in the precedence order**, extending the family established
  above: `chains_unverified > chains_stale > chains_empty > repo_kind_undetermined >
  chains_incomplete > chains_red > chains_waived`. An incomplete receipt is refused on its
  incompleteness even when every chain it does carry is green — completeness is checked before
  the red/waived arms are reached at all.

Validated RED-first: three new walkthrough cases (11)-(13) reproduced the gap exactly before the
fix (case (11): a claude-only receipt PASSED with `"result":"pass"`) and lock the fix after it —
case (11) asserts the `chains_incomplete` refusal names `missingChains:
["codex","gitlab","gitea"]`; case (12) locks `chains_incomplete` ABOVE `chains_red` (a
claude-only receipt that is itself red still refuses `chains_incomplete`, not `chains_red`);
case (13) locks the fail-closed `repo_kind_undetermined` probe (an otherwise all-green
four-chain receipt in a repo with no `package.json` still refuses). Pre-existing case (8), which
locks `chains_empty` ABOVE the new coverage arm (an empty `chains[]` in a four-chain-declaring
fixture still refuses `chains_empty`, not `chains_incomplete`), was re-verified unchanged.
Total walkthrough coverage for #651 is now 13 cases (10 base + 3 repair-window). All three
generated editions were re-synced (`npm run sync:editions`, byte-identical) and all four
`npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains re-ran green.

A companion, unrelated test-harness hermeticity fix landed in the same repair window (both
forge-helpers test suites now unset an ambient `KAOLA_WORKFLOW_OFFLINE` before any require that
captures it at require-time) — a test-reliability fix, not a `--release-check` design change, so
it is not detailed further in this record.
