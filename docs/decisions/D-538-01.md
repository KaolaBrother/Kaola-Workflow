# D-538-01. Adaptive is the unconditional default; fast/full are install-time opt-ins; the on/off switch + auto-fallbacks retire

Date: 2026-06-20
Status: Accepted
Issue: #538
Supersedes: docs/decisions/0007-adaptive-default-under-switch-on.md
Related: #227 (original adaptive path), #254 (adaptive-default-under-switch, the ADR now
         superseded), #44 (agent-owns-reasoning / typed refusals — no silent path substitution),
         #501 (self-sufficient by default — same design direction).

## Context

ADR 0007 (`docs/decisions/0007-adaptive-default-under-switch-on.md`) established adaptive as the
default route **under an ON switch**, with a boolean `enable_adaptive` config field (and its
`KAOLA_ENABLE_ADAPTIVE` env mirror) acting as a per-session kill-switch. That model had two
remaining costs:

1. **The switch is deliberation work.** Every run must resolve the switch — env > config > OFF —
   before routing. Even after #254 made ON the install-time default, the agent still carried the
   mental overhead of a switch that could be OFF, a Branch A / Branch B fork in the router prose,
   and a typed `authoring_refused` path for adaptive-under-OFF. This deliberation adds tokens and
   wall-clock for no accuracy benefit.

2. **`fast` and `full` had no install-time gate.** The switch gated only `adaptive`; `fast` and
   `full` were always available regardless of whether their command and skill files were present.
   A `KAOLA_PATH=fast` request on a system where fast was never installed would route to a missing
   command rather than a typed, actionable refusal.

Issue #538 resolves both by flipping the switch axis: instead of an adaptive on/off toggle, the
only option is which extra paths are installed.

## Decision

### (a) Switch-axis flip

Retire the boolean `enable_adaptive` config field and its `KAOLA_ENABLE_ADAPTIVE` env mirror. They
are replaced by a list-valued `installed_paths` field (default `[]`) in the same shared config file
(`~/.config/kaola-workflow/config.json`). Adaptive is implicit-always and **never** appears in the
array. The only "switch" is now which extra paths (`fast`, `full`) have been installed — an
install-time fact, not a per-run toggle. No per-session env override exists for `installed_paths`
("installed" follows on-disk state; an env claiming a path is installed would route to absent
command files).

Schema exports that change in `kaola-workflow-adaptive-schema.js` (byte-identical ×4):
- REMOVED: `WORKFLOW_PATHS_NO_ADAPTIVE`, `ENABLE_ADAPTIVE_FIELD`, `ENABLE_ADAPTIVE_ENV`,
  `resolveEnableAdaptive`.
- ADDED: `INSTALLED_PATHS_FIELD` (`"installed_paths"`), `resolveInstalledPaths(config)` (returns a
  frozen, de-duplicated subset of `["fast","full"]`; unknown tokens dropped).
- CHANGED: `isLegalWorkflowPath(value, installedPaths)` — adaptive is legal unconditionally
  (`value === ADAPTIVE_PATH` short-circuits); `fast`/`full` require membership in `installedPaths`.

### (b) Legality = {adaptive} ∪ installed_paths

`claimProject` now calls `resolveInstalledPaths(readAdaptiveConfig())` and passes the result to
`isLegalWorkflowPath`. A `KAOLA_PATH` (or `--workflow-path` flag) naming a path not in
`installed_paths` is a **typed `path_not_installed` refusal** (`result: refuse`) — not a silent
adaptive substitution, not a crash. The old `workflow_path_refused` reason string is retired;
`path_not_installed` is its successor. The old `path_requires_explicit_opt_in` guard (the #515
reciprocal guard) is removed: with adaptive as the unconditional default, a defaulted claim is
always `adaptive` (always legal), so the guard's condition can never fire. The bundle lane keeps its
constraint (bundle is adaptive-only) but renames the refusal: `target_set_not_adaptive` →
`bundle_requires_adaptive`.

The router (`workflow-next.md` Step 0a-1) collapses to a single unconditional flow: a path-name
keyword (`"fast path"` / `"full review"` / etc.) or an explicit `KAOLA_PATH` simply exports the
named path and hands it to the claim. The claim's `path_not_installed` refusal is the single
authority — the router does not read `installed_paths` or perform a soft fall-through.

### (c) Re-install unions `installed_paths`; reset = uninstall → reinstall

The installer (`install.sh`) exposes two new opt-in flags: `--with-fast` and `--with-full`.
`--enable-adaptive` is retired — the flag is accepted but warns and ignores. The installer reads
the pre-existing config, computes the effective set (union of what is already recorded and what was
requested), and writes the union back — it never removes an already-installed path. A bare
`./install.sh` re-installs the already-installed set and restores any stale-cleaned files for those
paths. `uninstall.sh` removes `~/.config/kaola-workflow/config.json`; reset to adaptive-only =
uninstall → reinstall.

### (d) No automatic fallback between paths

Adaptive never silently downgrades to `fast` or `full`. The failure floor is inside adaptive:
bounded planner repair (~2x) → discard+restart a fresh adaptive run (`claim.js discard` then a
fresh start) → stop+ask with a concrete blocker. Retired: the `kaola-workflow-adapt.md` "downgrade
to full path" clause in the exhaustion tail, the `workflow-next.md` "Adaptive fallback => full"
sub-bullet, and all `adaptive-under-OFF` typed refusals (`authoring_refused`,
`path_requires_explicit_opt_in`).

### (e) Dead code removed

`WORKFLOW_PATHS_NO_ADAPTIVE`, `ENABLE_ADAPTIVE_FIELD`, `ENABLE_ADAPTIVE_ENV`,
`resolveEnableAdaptive`, `workflow_path_refused`, `target_set_not_adaptive`,
`path_requires_explicit_opt_in`, `authoring_refused`, and the Branch A (switch-OFF) prose block
across all six routing surfaces are removed. `0007` is left in place as a superseded historical
record (the ADR-supersession convention).

## Consequences

**Positive:**
- No deliberation step on every run. Adaptive just runs; the agent has nothing to resolve.
- `fast`/`full` are now gated by install state — a `path_not_installed` refusal surfaces an
  actionable "re-run install with `--with-fast`" message rather than a missing-command failure.
- Router prose shrinks: Branch A (switch-OFF) is deleted across all six surfaces (#400).
- The config field (`installed_paths: []`) is self-documenting: an empty array means adaptive-only.

**Negative / note:**
- Existing installs with `enable_adaptive: false` in their config lose the kill-switch on the next
  install (the `config.pop('enable_adaptive')` migration in the installer removes the stale field).
  Operators who actively wanted adaptive off must either not reinstall or use `KAOLA_PATH=full`
  explicitly. This is intentional.
- `0007` is superseded, not deleted. Its body (the ADR that established ON-as-default) is historical
  context; the status line reads "Superseded by D-538-01".
