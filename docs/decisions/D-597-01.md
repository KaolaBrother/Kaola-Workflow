# D-597-01. Define the `speculative_open_policy: auto` tier and flip the default `off` -> `auto`

Date: 2026-07-03
Status: Accepted
Issue: #597
Supersedes-in-part: docs/decisions/D-419-02.md (Part 4, [INV-19] — "consent is captured per-run
             by the USER... never auto-granted by an agent." The ceremony this invariant
             protected is retired; every OTHER Part-4 invariant — [INV-20] through [INV-25]:
             the roll-back discard, the `gate_not_complete` close fence, post-dominance over the
             complete-set, and the blast-radius conditions — is untouched. See "The D-419-02
             supersession" below.)
Related: docs/decisions/D-596-01.md (graduated write-bearing nodes onto the `consent` bet with
             leg-contained discard-only rollback — the safety net this decision reuses verbatim
             for the `auto` tier, unchanged); docs/decisions/D-542-01.md (precedent rung 1:
             default-on disjoint-write parallelism, planner-proven antichains co-open with no
             operator toggle); docs/decisions/D-546-01.md (titled D-546-G2; precedent rung 2:
             file-disjoint shared-infra write lanes co-open by default, dropping operator
             consent once the containing net was already structural)

## Context

Issue #439 (D-419-02 Part 4) shipped `speculative_open_policy: off | consent | auto`, but only
`off` and `consent` were live — `auto` was named-but-refused at freeze
(`speculative_policy_unsupported`) "until a future issue defines its risk envelope." Issue #596
(D-596-01) then graduated the `consent` tier to admit write-bearing speculative nodes, reusing
the existing per-member leg/lane-group machinery for containment and making a write bet
discard-only on a failed gate. Every correctness property of speculation — read or write — is
now structural, not consent-derived:

- A speculative node can never reach `complete` before its gate (`speculativeCloseGuard` ->
  `gate_not_complete`); post-dominance over the complete-set is exact and untouched by this
  decision.
- A failing gate discards the speculative work unconditionally for a write (leg teardown, parent
  untouched) and under an evidence-validity review for a read.
- A speculative writer's own output is still reviewed — G1 post-dominance is freeze grammar,
  untouched.

What the per-run `--speculative-consent` ceremony still gated, post-#596, was purely
**economics**: tokens spent on a speculative node that turns out to lose its bet. That cost is
bounded (the existing fan-out caps apply to a speculative frontier identically), the failure
rate is low in practice, and the decision of whether that bounded cost is worth paying is an
efficiency judgment, not a value, standing, or irreversibility call — so it does not belong at
the `consent`/`decision:ask` valve. Requiring a human grant per run in order to permit *saving*
wall-clock had become inverted friction: the very thing the ceremony existed to gate (a
correctness risk) had already been made structural by #596.

**This is the same call this repo has made twice before**, each time a consent ceremony
protecting a capability was later found to add nothing once the containment became structural:

1. **D-542-01** — disjoint-write co-open went from flag-gated to default-ON once per-leg
   isolation plus the barrier became the containment net.
2. **D-546-01** — shared-infra co-open dropped operator consent with the recorded rationale that
   "the operator-consent ceremony added nothing the structural net does not already guarantee."

Speculation post-#596 has the identical shape: a structural net (the close fence, the leg/group
containment, the discard path) already carries the correctness weight the ceremony was invented
to backstop.

**Live-exercise precondition (satisfied before this issue started).** #596's leg-speculation
machinery was exercised end-to-end in project `issue-599` (the adaptive run that shipped the
#599 fail-closed fix): while that run's code-review gate was `in_progress`, `open-ready
--speculative-consent` opened a downstream docs writer speculatively in its own isolated leg; a
deliberate `close-node` attempt while the gate was still open correctly refused
`gate_not_complete`; the gate then closed `verdict: pass` and the speculative member closed
through the per-leg barrier -> octopus merge -> commit-based union barrier cleanly. That
exercise also surfaced the two operational gotchas recorded in
`docs/plan-run-cards/speculative-open.md` §8 (a gate opened via the fused serial advance cannot
admit speculation; a parent-branch commit landing while a speculative leg is open voids the
leg's anchored base) — both are pre-existing mechanical behavior, folded into the card as
documentation, not new code in this decision.

## Decision

**Define the `auto` tier and flip the freeze-time default `off` -> `auto`.**

1. **Schema (`kaola-workflow-adaptive-schema.js`, the byte-identical ×4 drift anchor).**
   `SPECULATIVE_OPEN_POLICY_LEGAL` gains `'auto'` — `['off', 'consent']` -> `['off', 'consent',
   'auto']` — which is the constant the validator's freeze gate actually consults
   (`!LEGAL.includes(specPolicy)` refuses `speculative_policy_unsupported`); `auto` was
   previously excluded from `LEGAL` itself, not merely listed as refused. The separate
   `SPECULATIVE_OPEN_POLICY_REFUSED_AT_FREEZE` (an informational/forward-compat constant, not
   directly consulted by the freeze check) empties `['auto']` -> `[]` to stay in sync.
   `SPECULATIVE_OPEN_POLICY_DEFAULT` flips `'off'` -> `'auto'`. The `speculative_policy_unsupported`
   operator hint text is updated to drop the stale "auto is deferred" language and instead name
   `auto` as the freeze-time default.

2. **Tier semantics.**
   - `auto` (new default): `speculativePending` is emitted by `next-action.js` AND `open-ready`
     acts on it without `--speculative-consent` and without a `decision:ask` capture. Covers
     read speculation and (per #596) leg-contained write speculation identically; every #596
     eligibility/safety condition — exact-path disjointness against live writers, no PROTECTED
     file, exact resolvability, not the plan's unique sink, leg capability, fan-out caps —
     applies exactly as it does at `consent`. `auto` removes the ceremony, never a safety
     condition.
   - `consent`: unchanged from D-419-02/D-596-01 — emission plus the per-run
     `--speculative-consent` flag, captured at `decision:ask`. Remains fully authorable for an
     operator who wants the explicit ask.
   - `off`: no speculation; emission omitted — the existing byte-identical shape.
   - `--speculative-consent` at `auto` is accepted as a documented no-op (back-compat: a caller
     that always passes the flag needs no tier-branching logic).

3. **Freeze-time materialization — the resume-safety rule.** The default flip applies at
   FREEZE, never retroactively:
   - `parseSpeculativePolicy` keeps returning `off` for an ABSENT `## Meta` field — the absence
     fallback is a fixed literal, DECOUPLED from `SPECULATIVE_OPEN_POLICY_DEFAULT` (which is now
     `'auto'`). A naive `return DEFAULT` would have retroactively flipped every in-flight plan
     frozen before this change.
   - `kaola-workflow-adaptive-handoff.js`'s freeze transaction (Step 1.75) calls
     `schema.materializeSpeculativePolicy(content, SPECULATIVE_OPEN_POLICY_DEFAULT)` when the
     author omitted the field on a FRESH (not-yet-frozen) plan, injecting an explicit
     `speculative_open_policy: auto` line into `## Meta` before computing the freeze hash — so
     the field is hash-covered and the frozen plan is self-describing. `materializeSpeculative-
     Policy` is a no-op when the field is already present (an author's explicit `off` or
     `consent` choice is never overwritten) and when the plan has no `## Meta` section at all.
     An already-frozen (legacy) plan with an absent field is left untouched — `parseSpeculative-
     Policy` keeps resolving it to `off` on resume.

4. **Discard telemetry — no silent cost.** `runDiscardSpeculative` in `adaptive-node.js` now
   passes `{ role, gate }` through `appendProvenanceLog`'s existing `extra` parameter on every
   speculative discard (read or write), recording the discarded node's id, role, and the gate it
   bet on into the run's `.cache/provenance-log.jsonl`. This makes the economics of `auto` — how
   often a bet does not pay off, and what it cost — observable per run instead of anecdotal, so
   the posture can be re-evaluated on evidence if the cost profile changes.

5. **Telemetry-accuracy fold-in (O1).** In `selectSpeculativeWriteGroup`, the fail-closed branch
   that excludes ALL write candidates on a crash/garbled non-open running-set read previously
   mislabeled `speculativeWriteExcluded.reason` as `overlaps_live_writer` (implying a genuine
   per-pair disjointness conflict). It is now distinguished as `parallel_safe_indeterminate` (the
   disjointness verdict could not be computed, not a known overlap); a genuine per-pair overlap
   still reports `overlaps_live_writer`. This distinction matters more now that `auto` fires
   speculative writes routinely rather than only when an operator opted in.

### The D-419-02 supersession

D-419-02 Part 4 recorded **[INV-19]**: *"Consent is captured per-run by the USER (the human at
`decision:ask`), never auto-granted by an agent. The policy field gates ELIGIBILITY (plan-level);
the per-run consent gates ACTIVATION. Both must be true to speculatively open."*

**This decision deliberately supersedes [INV-19] for the `auto` tier** (operator-directed,
2026-07-02 — recorded in the issue rationale, not independently re-derived by an agent). At
`auto`, activation no longer requires a per-run human grant: eligibility (the policy field, now
resolvable to `auto`) is sufficient on its own. The supersession is scoped precisely:

- **Superseded:** the per-run consent CEREMONY as a universal precondition for activation.
  `[INV-19]`'s "both must be true" clause no longer holds when the plan's policy is `auto` — the
  policy field alone gates activation there.
- **NOT superseded — every mechanical invariant survives verbatim, for both tiers:**
  - `[INV-20]` (discard is roll-back to the recorded baseline / leg teardown, never roll-forward).
  - `[INV-21]` (speculative = `in_progress`, never `complete`, ahead of its gate — the
    `gate_not_complete` close fence).
  - `[INV-22]` (`--gate-verify` post-dominance is the enforcement point; speculative open never
    touches it).
  - `[INV-23]` (Part 4 requires the running-set scheduler's `max>1` arm).
  - `[INV-24]` (the blast-radius guard: a speculative write is gated on disjointness, resolvability,
    non-PROTECTED, non-sink — the #596 conditions this decision leaves untouched).
  - `[INV-25]` (a failing gate rolls back the ENTIRE speculative frontier that bet on it, not just
    a conflicting member).
  - `speculative_open_policy` remains a hash-covered `## Meta` field (eligibility, tamper-evident)
    — the deliberate asymmetry vs. runtime-cap fields like `max_concurrent` is unchanged.
  - The `consent` tier itself is not removed: an operator who wants the per-run ask still gets it
    by authoring `speculative_open_policy: consent`. `[INV-19]` still fully holds for that tier.

The supersession is narrow by design: it retires a ceremony, not a safety property. Every
correctness guarantee `[INV-19]` was originally paired with — the discard path, the close fence,
post-dominance, the blast-radius conditions — remains exactly as strict as it was under
`consent`-only.

## Consequences

- **A fresh freeze now speculates by default.** Any plan with an eligible gate-descendant
  topology gets automatic speculative overlap unless the planner explicitly authors `off` or
  `consent`. This is a real behavior change for every NEW plan (not a flag an operator has to
  discover and opt into).
- **No plan frozen before this change changes behavior.** The absence-fallback stays `off`;
  `auto` is materialized only into a FRESH freeze. Upgrading the installed scripts mid-run and
  resuming an in-flight plan cannot silently start speculating.
- **The token cost of a lost bet is now observable, not silent** — the discard-telemetry record
  (node id, role, gate) lands in the durable provenance log on every discard, at either tier, so
  the `auto` posture's real-world cost profile can be reviewed on evidence.
- **The `consent` tier remains fully live** for an operator or planner who wants a per-run human
  checkpoint on a specific speculative bet — this decision adds a tier, it does not remove one.
- **`off` remains the permanent, byte-identical fallback** for a plan or host that wants no
  speculation at all (e.g. `KAOLA_PARALLEL_WRITES=0` hosts still exclude every write candidate
  via `speculativeWriteExcluded: { reason: 'no_leg_capability' }`, unaffected by tier).
- Cross-edition (#307): `kaola-workflow-adaptive-schema.js` (canonical + codex twin + both forge
  ports, byte-identical), `kaola-workflow-plan-validator.js`, `kaola-workflow-adaptive-handoff.js`,
  `kaola-workflow-next-action.js`, and `kaola-workflow-adaptive-node.js` (each canonical + codex
  twin + both forge ports) were regenerated together; all four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains must stay green, together with
  `scripts/simulate-workflow-walkthrough.js` and the route-reachability contract over the six
  plan-run/planner prose surfaces this issue also updated.

## Alternatives considered

- **Keep `speculative_open_policy: auto` permanently refused (status quo).** Rejected: this is
  the exact call the issue reopens. The premise for gating activation behind a human grant — an
  unproven containment story — is now moot; #596 already made write speculation as safe as read
  speculation under the SAME structural net every other default-relax in this repo's history
  (D-542-01, D-546-01) has relied on to retire its own consent ceremony.
- **Retire the `consent` tier entirely now that `auto` exists, collapsing to a two-value
  `off | auto` schema.** Rejected: an operator or a planner may still want a specific bet
  reviewed before it fires (e.g. an unusually large or novel speculative frontier); removing
  `consent` would take away a real, still-useful escalation path for no correctness gain, and
  would be a needless breaking change to any plan already authoring `consent` explicitly.
- **Introduce a fourth policy value (e.g. `auto-read-only`) instead of making full `auto`
  symmetric with `consent`'s read+write scope.** Rejected: #596 already established that a
  write-bearing speculative node is exactly as safe as a read-only one under the leg/lane-group
  net; splitting `auto`'s scope by node kind would reintroduce the asymmetry #596 deliberately
  eliminated, for no additional safety.
- **Apply the default flip retroactively to already-frozen plans on resume (treat an absent
  field as `auto` unconditionally).** Rejected outright — this would silently change the
  execution shape of a plan an operator already reviewed and froze under the OLD default,
  violating the tamper-evidence property `speculative_open_policy` is hash-covered specifically
  to protect. The resume-safety rule (a fixed `off` absence-fallback, decoupled from the freeze-
  time default) is the only choice consistent with "frozen plans are self-describing; posture
  can never drift across an upgrade or a resume."
