# D-617-01. Derive merge-lane close deferral from durable state, wire remote-closed-after-publish, reorder closure-last, add a verify-sink audit, and fail chain-receipt greenness closed on signal death

Date: 2026-07-07
Status: Accepted
Issue: #617 (primary — close-before-publish); #618 (companion fix in the same bundle —
       chain-receipt greenness fail-open)
Related: #164 (`remote-closed-after-publish` first declared as closure invariant 5, left
         deferred/unevaluated), #429 (`SINK_STEPS` ordered step contract), #396/#396.4/#396.5
         (the `close_pending`/`close_disposition` merge-lane deferral this decision hardens),
         #592 (bundle-close gate + `closed_issues` receipt recording, unchanged by this decision),
         #608 (the `timed_out`/receipt-schema precedent `signal` extends the same way)

## Context

A live incident (2026-07-06) closed a GitHub issue on the forge while its ~3000-line
implementation never actually reached the default branch: the merge sink transaction ran its
`closure` step (issue close) three steps before its final `push_main` step, so a crash or partial
failure between `closure` and `push_main` left a closed issue with no merged code behind it. The
`remote-closed-after-publish` invariant that exists precisely to catch this (`checkClosureInvariants`,
declared since #164) had never been wired to any evaluation — it was a description-only entry in
`CLOSURE_INVARIANTS`, never assigned a violation condition. Independently, in the same review pass,
the chain-receipt greenness gate (`--finalize-check`, self-host mode) was found to fail open in two
related ways: a child process terminated by an OS signal (e.g. an external OOM-kill or operator
`SIGKILL`, not necessarily our own per-chain timeout) could still map to `exitCode: 0` in both the
sync and async run-chains dispatch paths, and a fresh, HEAD-bound receipt whose `chains[]` array was
empty (zero chains verified) passed the gate vacuously, since a red-chain filter over an empty array
is trivially empty.

## Decision

**Close the gap at its structural root — ordering, an ancestor check, and fail-closed exit-code
mapping — rather than adding a second parallel check on top of the existing one.**

1. **Derive merge-lane close deferral from durable state, not solely `--keep-worktree`.**
   `cmdFinalize` now computes `mergeLaneDeferred` by reading the persisted `sink` field from
   `workflow-state.md` (default `merge`) whenever `--keep-worktree` was not passed, and gates the
   finalize-side issue-close on it instead of on the raw flag alone. An unreadable or absent field
   fails TOWARD deferral (never toward a premature close) — a caller that forgot
   `--keep-worktree` on a merge-lane run can no longer close the issue before the merge sink runs,
   which is exactly what happened in the incident.
2. **Wire the previously-declared `remote-closed-after-publish` invariant.**
   `checkClosureInvariants(root, receipt, archiveDest, opts)` gained an optional fourth parameter:
   when a caller supplies `opts.implRef` and `opts.sinkTarget`, it runs
   `git merge-base --is-ancestor <implRef> <sinkTarget>` and records the verdict on
   `receipt.remote_closed_after_publish` (`'verified'` / `'failed'`), pushing a violation on
   failure. A caller that omits `opts` (e.g. `cmdFinalize`'s merge-lane, which defers its own close
   and so has no genuine close to verify at this seam) leaves the check a pure no-op — byte-identical
   to before this decision. `sink-merge`'s `postMergeCleanup` now supplies both refs, capturing the
   feature branch's SHA before the branch ref is deleted later in teardown.
3. **Reorder `SINK_STEPS` so `closure` is the last step.** Previously
   `['preflight', 'push_upstream', 'merge', 'worktree_sync', 'finalize', 'closure', 'stash_restore',
   'archive_commit', 'push_main']` ran the issue close three steps before the push that actually
   publishes the merge. It is now
   `['preflight', 'push_upstream', 'merge', 'worktree_sync', 'finalize', 'stash_restore',
   'archive_commit', 'push_main', 'closure']`. The `closure` step body additionally re-resolves the
   feature branch's current tip and re-runs the ancestor check as a hard gate immediately before
   closing any issue — trusting the ordering is not enough on its own, because a resumed/stale
   receipt or a future reordering regression must still be caught; a failed check refuses loudly
   (non-zero exit, `remote_closed_after_publish_unverified`, no issue closed, step left not-done for
   retry) rather than silently proceeding.
4. **Add a standalone `verify-sink` audit subcommand.** `kaola-workflow-claim.js verify-sink
   --project P` is a pure read-only check an operator (or the next session) can run independently
   of any live merge sink, to catch exactly the incident shape after the fact: it resolves the
   recorded implementation commit (from a live-or-archived `sink-receipt.json`, falling back to a
   still-lingering feature branch), verifies it is an ancestor of the resolved default branch, and
   checks that no worktree/branch/active-folder residue lingers. It exits non-zero with a typed
   `reasons` array on any failing leg and never mutates state.
5. **Make chain-receipt greenness fail closed on signal death and on an empty `chains[]`.**
   In `kaola-workflow-run-chains.js`, both the sync (`spawnSync`) and async (`spawn`) dispatch
   paths now map a signal-terminated child (`status === null` with a signal or spawn error present)
   to `exitCode: 1` unconditionally — never falling through to the prior `(code != null) ? code : 0`
   false-green branch — and record the terminating signal name (`signal: "SIGKILL"` etc., `null` on
   a normal exit) in the receipt for diagnosability, independent of the pre-existing `timed_out`
   field (a signal death is not necessarily our own timeout kill). In
   `kaola-workflow-plan-validator.js --finalize-check`, an empty `chains[]` array now refuses with a
   typed `chains_empty` reason, precedence-ordered between `chains_stale` and `chains_red`, mirroring
   the producer's own `no_chains` refusal (which already declines to *write* an empty-chains
   receipt) with a matching refusal on the *consumer* side. `test-run-chains.js` is wired into the
   `test:kaola-workflow:claude` npm chain so these paths run under `npm test`, not only the bare
   walkthrough.

## Consequences

- An issue can no longer be closed by either lane (`cmdFinalize` merge-lane, or `sink-merge --sink`)
  before its implementation commit is a verified ancestor of the actual sink target — the ordering
  fix and the hard ancestor gate are independent, redundant defenses against the same failure mode.
- `remote-closed-after-publish` is a live, evaluated invariant for the first time since being
  declared in #164; `docs/api.md`'s existing prose describing it as "intentionally deferred to
  issue #165" is now stale and needs a follow-up documentation pass — out of scope for this
  decision's write set.
- `verify-sink` gives an operator (or a later audit) a way to detect an already-landed instance of
  this failure shape without re-deriving the check by hand.
- A signal-killed test chain (external OOM-kill, operator `SIGKILL`, or any non-normal-exit
  termination) can never again read as a false green in `.cache/chain-receipt.json`, and an
  empty-chains receipt can never again pass `--finalize-check` vacuously.
- Validated with hermetic RED→GREEN tests (`scripts/test-bundle-finalize.js` for the closure-gate
  and `verify-sink` arms; `scripts/test-run-chains.js` T26–T29 for the signal-death and
  `chains_empty` arms) plus all four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}`
  chains and `node scripts/simulate-workflow-walkthrough.js`.

## Non-goals

- **No change to the `remote-members-closed` bundle all-or-nothing invariant, `close_pending`
  qualifier, or keep-open lane semantics.** Those existing #396/#369 mechanisms are untouched; this
  decision only adds the previously-missing publish-before-close check alongside them.
- **No retry, auto-scaling, or adaptive-timeout mechanism for chain execution.** The signal-death
  fix only corrects exit-code mapping; the existing kill-ceiling and retry-on-transient-fault
  policies (#608, #550) are unchanged.
- **No new closure-receipt enum values.** `remote_closed_after_publish` is additive
  (`'verified'|'failed'`, absent on a caller that supplies no verification refs); no existing enum
  in `CLOSURE_RECEIPT_FIELDS` changed shape.

## Alternatives considered

- **Add a post-hoc reconciliation sweep that scans closed issues for unmerged branches on a
  schedule**, rather than gating the close itself. Rejected: detection-after-the-fact does not
  prevent the incident, only shortens time-to-discovery, and still requires an external trigger
  (schedule, CI, or manual run) this project does not assume exists (`CI/CD is not a gate`).
  Gating the close at the source — ordering plus a hard ancestor check — prevents the bad state
  from ever being written, which is strictly stronger and no more expensive to run.
- **Leave `SINK_STEPS` ordering unchanged and rely solely on the new ancestor check inside the
  existing early `closure` step position.** Rejected: the ancestor check alone would correctly
  refuse most of the time, but a partially-published state (branch merged locally but not yet
  pushed to the remote at the time `closure` ran) could still pass the ancestor check against a
  local ref while `push_main` had not yet happened — the same class of gap the incident exploited,
  just narrower. Moving `closure` after `push_main` removes the ordering hazard entirely; the
  ancestor check is then a belt-and-suspenders assertion, not the sole line of defense.
