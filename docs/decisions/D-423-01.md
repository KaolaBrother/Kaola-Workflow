# D-423-01. test-bash-block-guards.js: fixture augmentation + no-plan scenario D

Date: 2026-06-12
Status: Accepted
Issue: #423
Related: `scripts/test-bash-block-guards.js`,
         `agents/contractor.md` (Step 8a ledger-compare guard),
         `scripts/kaola-workflow-plan-validator.js` (`--barrier-check` / ledger-compare),
         D-425-01 (ledger_header_invalid freeze wall, the related validator fix)

## Context

Issue #423 was filed when the claude chain turned red at commit 58ffd81. The commit landed a
production Step-8a ledger-compare guard in the contractor profile — the guard shells the
plan-validator's `--barrier-check` with a `--source` plan path to verify that a node's declared
write set matches its ledger entry. The guard was present in all four edition trees (the three
byte-mirror contractor toml twins included).

`scripts/test-bash-block-guards.js` exercises the contractor Step-8a bash block via a parse-and-run
harness. Its existing scenario A tested the nominal path: the block completes without error when
the expected artifacts are present. After 58ffd81, scenario A began failing because the test
fixture wrote `workflow-state.md` but did NOT write a minimal `## Node Ledger` section to
`workflow-plan.md`. The guard's `--source` path therefore pointed at a plan with no ledger, and the
plan-validator exited non-zero, which the test harness surfaced as a failure.

Three remediation shapes were considered:

1. **Skip the ledger-compare entirely in the test** — stub out or comment out the `--source` guard
   so it never runs in the harness.
2. **Mock the plan-validator subprocess** — inject a fake validator that always exits 0.
3. **Fixture augmentation: write a minimal `## Node Ledger` to the test plan** — give the guard a
   valid ledger to compare against, matching the in-production precondition (a frozen plan always
   has a ledger when the guard runs).

Additionally, the guard shipped with no test coverage of the negative path: a full/fast-path
project that has no `workflow-plan.md` at all. The guard must no-op (logged skip) in that case,
but no scenario asserted this.

## Decision

**Fixture augmentation (option 3) is the correct fix**, paired with a new negative scenario D for
the no-plan path.

**Why not skip or mock:**

- Skipping (option 1) removes coverage of the guard entirely. The guard is production code that
  protects against write-set drift; a test that does not exercise it provides no regression
  value. A future change to the guard could break production silently.
- Mocking (option 2) is a false green: a mock that always exits 0 asserts only that the harness
  can dispatch a subprocess, not that the guard's arguments, exit-code interpretation, or
  plan-path resolution are correct. The harness exists precisely to run the real bash block with
  real scripts.

**Why fixture augmentation:**

- The guard's precondition in production is that `workflow-plan.md` already exists and has been
  frozen (it is written by the `workflow-planner` before `plan-run` begins, and it always gains a
  `## Node Ledger` section as nodes open). Writing the minimal ledger to the test fixture replicates
  the real precondition rather than sidestepping it.
- The fix is local to the fixture setup, touches no production code, and leaves the guard running
  at full strength in the test.
- A minimal `## Node Ledger` with a single `| id | status |` header row plus one `| n1 | complete |`
  row is sufficient: it satisfies the guard's schema check without expanding the test's surface area
  beyond what scenario A already covers.

**Negative scenario D (no-plan path):**

The guard must detect that no `workflow-plan.md` exists (full/fast-path project) and exit 0 with a
logged skip token, never blocking a non-adaptive finalize. This is a meaningful behavioral assertion
that was absent from the original test suite. Scenario D verifies:

- The block exits 0 when `workflow-plan.md` is absent.
- The block emits the skip token to stdout (so the orchestrator can observe the no-op).
- Any rename operations in the block still complete normally (the rest of Step 8a is unaffected).

Scenario D uses the same harness infrastructure as scenarios A–C — no mocking, real bash block
execution — which is consistent with the test's design principle.

## Consequences

- Scenario A is green again: the fixture now supplies a valid ledger, so the guard exits 0 and the
  test passes.
- Scenario D is green: the no-plan fast-path is covered and the skip behavior is asserted.
- The production guard is unchanged; no behavioral regression is possible from this fix.
- Future changes to the Step-8a guard (new ledger columns, changed exit codes) will surface
  regressions in scenario A or D before reaching the four-chain gate.
- The fix does not require touching the contractor profile or any edition-port file: the issue was
  entirely in the test fixture, not in production code.
