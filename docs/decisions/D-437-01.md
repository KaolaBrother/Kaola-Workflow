# D-437-01. Lane-group co-open and group-scoped close barrier (D-419 Part 2 implementation)

Date: 2026-06-13
Status: Accepted
Issue: #437
Related: D-419-01 (Parts 1 + 3: the coordination kernel this activates),
         D-419-02 (Part 2 design: the pre-implementation ADR this implements),
         ADR 0008 (`docs/decisions/0008-excise-write-role-batch-isolation.md`, the excise whose
           reintroduction condition this satisfies),
         #376 (the `KAOLA_LANE_CONTAINMENT` write-lane hook),
         #377 (the per-node running-set scheduler),
         #283 (the seal-vacuity / non-empty-in-lane guard whose lane-group form this carries forward),
         #424 (the `.md` allowband narrowing and finalize attribution sweep).

This record covers the **as-built** implementation of D-419 Part 2: four settlements that wire
lane-attributed disjoint write co-open and a group-scoped close barrier behind
`KAOLA_LANE_CONTAINMENT`, with a permanent flag-OFF byte-identical serial fallback.

## Problem statement

The adaptive path's `open-ready` scheduler hard-gated every write node to
`write_node_exclusive`: if any write was live, no further opens occurred. This was correct for
the original design (writes share a parent worktree and can clobber each other), but it ruled
out safe parallelism even when two write nodes had provably disjoint declared write sets.

The ADR D-419-02 Part 2 recorded a design for lifting the restriction — but the design had a
broken close-side story: the original sketch diffed each member against the worktree at the
moment of its close, which meant a non-last member's close would see the still-open SIBLING's
in-lane writes as "unattributed strays" and refuse. Four settlements replace that broken story:

1. A lane-group co-open records ONE shared baseline for the whole group.
2. A non-last member close runs only evidence-shape and per-member in-lane vacuity — NO diff
   barrier.
3. The diff barrier runs ONCE at the last member's close, over the UNION of all members'
   declared write sets.
4. The `--parallel-safe` validator flag exposes the existing disjointness pair-loop as a
   read-only CLI check so `open-ready` can gate co-open on a proven pairwise-disjoint frontier.

All four changes are gated on `KAOLA_LANE_CONTAINMENT`. With the flag OFF (the permanent
default), every changed code path is dead and the serial behavior is byte-identical to
pre-#437.

## The four settlements

### Settlement 1 — `lane_group` extension to `running-set.json`

One optional top-level key `lane_group` is added to the existing `running-set.json` (which
lives at `kaola-workflow/{project}/.cache/running-set.json`). At most one write lane group is
live at a time — a group runs alone, exactly as a single write node does today. Shape:

```json
{
  "state": "open",
  "max_concurrent": 8,
  "lane_group": {
    "group_id": "lg-n2a-n2b",
    "members": ["n2a", "n2b"],
    "closed_members": [],
    "baseline": "<commit-sha>",
    "write_union": ["scripts/x.js", "scripts/y.js"],
    "openedAt": "2026-06-13T10:12:00.000Z"
  },
  "nodes": [
    { "id": "n2a", "group_id": "lg-n2a-n2b", "role": "implementer", "kind": "write",
      "declared_write_set": "scripts/x.js", "model": "opus", "baseline": "recorded" },
    { "id": "n2b", "group_id": "lg-n2a-n2b", "role": "implementer", "kind": "write",
      "declared_write_set": "scripts/y.js", "model": "opus", "baseline": "recorded" }
  ]
}
```

- `group_id` — deterministic `'lg-' + sortedMemberIds.join('-')`, sanitized for
  file/ref keys by the same `[^A-Za-z0-9_-]→_` rule `cacheBaseFile`/`barrierRef` already apply.
- `members` — full member id array, sorted, STABLE (never shrunk during the group lifetime;
  `closed_members` tracks per-member close state so `members` stays the union-allowlist source).
- `closed_members` — accumulates closed member ids so the last-member detector does not need to
  read the ledger.
- `baseline` — ONE shared baseline commit SHA recorded at group-open time via
  `--record-base --node-id <group_id>` (reuses the per-node baseline machinery keyed by
  `group_id` instead of a node id; anchored in a gc-protected ref, guarded by the #368
  mismatch check).
- `write_union` — union of each member's parsed `declared_write_set` (convenience snapshot;
  the group barrier re-reads the plan for authoritative attribution).
- Each node entry gains `group_id` (string) so `close-node` knows it is a group member.

The group co-open is orchestrated by `runOpenReady` (`adaptive-node.js` L2550) and the new
internal helper `tryFormLaneGroup` (`adaptive-node.js` L2522). The formation is two-phase
crash-safe: `running-set.json` is written with `state:'opening'` BEFORE any ledger row flips,
then promoted to `state:'open'` after all baselines and ledger flips succeed.

### Settlement 2 — member close: deferred barrier + per-member vacuity

`runCloseNode` (`adaptive-node.js` L2838) detects group membership by reading `running-set.json`
and finding `lane_group` with this node's id in `members`. Detection is gated on both
`resolveLaneContainment(process.env)` AND `lane_group` being present — so a serial run (no
`lane_group` key) is unconditionally unaffected. When a group member closes:

- **Evidence-shape + binding check** — identical to the serial path (steps (a.1–a.4) in
  `runCloseNode`). Refused on failure.
- **Per-member in-lane vacuity** (`memberInLaneChanges`, `adaptive-node.js` ~L2826) — scopes
  `git status --porcelain` to the member's declared set only; non-empty ⟹ pass; empty ⟹
  check evidence for a `no_op: <reason>` line; absent `no_op:` ⟹ refuse
  `{ reason: 'member_vacuity' }`.
- **Non-last member** — the diff barrier is DEFERRED. The member's ledger row is closed, a
  compliance row carries the literal `deferred_to_group` marker in the Evidence cell, and
  `running-set.json`'s `closed_members` is updated. The close returns
  `{ barrier: 'deferred_to_group' }`.
- **Last member** — the group barrier (Settlement 4) runs BEFORE closing or removing this member
  (so `lane_group.members` still holds the full set the validator reads). Barrier refuse ⟹
  typed refusal, NO ledger advance, `lane_group` untouched. Barrier pass ⟹ close the last
  member, compliance row with `group_passed`, clear `lane_group`, drop the group baseline via
  `--drop-base --node-id <group_id>`. The close returns `{ barrier: 'group_passed' }`.

The actual member-close logic lives in `closeGroupMember` (`adaptive-node.js` L2996).

### Settlement 3 — `--parallel-safe` validator CLI flag

New read-only CLI handler in `kaola-workflow-plan-validator.js` (~L1627). Exposes the EXISTING
antichain pair-loop predicates (exact-file rule + `classifier.disjointWriteSets`) as a 2+-node
subset check. No file writes, no baseline, no git diff.

Invocation:
```
node kaola-workflow-plan-validator.js <planPath> --parallel-safe --nodes A,B[,C] --json
```

Success output shape:
```json
{ "result": "ok", "nodes": ["A", "B"], "overlapping": [] }
```

Failure output shape:
```json
{
  "result": "refuse",
  "reason": "overlapping_write_sets",
  "nodes": ["A", "B"],
  "overlapping": [{ "a": "A", "b": "B", "kind": "exact", "path": "scripts/x.js" }]
}
```

`kind` is `'exact'` (exact-file clash) or the classifier's verdict string (e.g. `'yellow'`)
for coarse/shared-infra overlap. The flag is toggle-agnostic — `open-ready` gates the call.

`tryFormLaneGroup` shells this flag over the write frontier; `result:'refuse'` ⟹ `open-ready`
DEGRADES to a single serial write open. Overlap is therefore structurally prevented from
co-opening.

### Settlement 4 — the group barrier

New CLI handlers `--group-barrier --group-id <id>` in `kaola-workflow-plan-validator.js`
(~L1914). Invoked by `closeGroupMember` at the last member's close (via `shell(validatorPath, [planPath, '--group-barrier', '--group-id', lg.group_id, '--json'])`).

The barrier diffs the group baseline → a now-snapshot (`snapshotWorktree(root, groupId+'-now')`)
and passes all paths to `barrierCheck(content, actualPaths, { groupMembers: members, root, project })`.

`barrierCheck` (`plan-validator.js` L658) gains one new opts arm placed BEFORE the existing
per-node and whole-plan arms:

```javascript
if (opts.groupMembers && opts.groupMembers.length) {
  for (const id of opts.groupMembers) {
    const n = nodes.find(x => x.id === id);
    if (n) for (const p of n.writeSet) declared.add(p);
  }
}
```

This sets the allowlist to the UNION of the named members' write sets. Every path in the diff
that is NOT in the union lands in the EXISTING rank-4 `outOfAllow` arm (`unattributed_write` /
`write_set_overflow` reason). No new reason code — exactly the issue's settlement. Because the
group is formed from PAIRWISE-DISJOINT write sets (guaranteed by Settlement 3), a path in the
union belongs to EXACTLY ONE member — attribution is unambiguous by construction.

Success output: `{ "result": "pass", "errors": [] }` (the `barrierCheck` envelope).

Typed refusal reasons (from existing `barrierCheck` arms):
- `write_set_overflow` / `unattributed_write` — a diff path belongs to no member's set.
- `running_set_unreadable` — cannot parse `running-set.json`.
- `group_not_found` — the named `group_id` is absent or mismatched in `running-set.json`.
- `no_group_base` — no baseline recorded for this `group_id`.
- `barrier_base_mismatch` — `.cache` file SHA ≠ gc-anchored ref SHA (laundering guard, #368).
- `root_mismatch` — barrier called from outside the repo toplevel.

## Flag-OFF invariant (INV-6)

With `KAOLA_LANE_CONTAINMENT` unset, `0`, `false`, or `no`:

- `resolveLaneContainment(process.env)` returns `false` (adaptive-schema.js L352).
- `runOpenReady`: the `if (containment && writeNodes.length >= 2)` guard is false; the existing
  `else { toOpen = [writeNodes[0]]; openKind = 'write'; }` runs verbatim. `tryFormLaneGroup`
  is never called, `groupForm` is `undefined`, `lane_group` is never written, the running-set
  shape is byte-identical to pre-#437.
- `runCloseNode`: `lg` is `null` (guarded by `containment &&`); `closeGroupMember` is never
  called; the existing serial close (evidence → `commit-node --node-id` barrier → ledger
  complete → compliance → selector → remove → recompute) runs verbatim.
- `barrierCheck`: `opts.groupMembers` is never passed by any existing caller; the new arm is
  skipped; allowlist construction is byte-identical.
- The ×4 walkthroughs (which never set `KAOLA_LANE_CONTAINMENT`) exercise the unchanged serial
  paths and are the flag-OFF assertion.

## Group baseline mechanism

The group baseline reuses the per-node baseline machinery entirely, keyed by `group_id` instead
of a node id:

- **Recorded at**: `runOpenReady` Phase 1, via `shell(commitNodePath, [planPath, '--node-id', group_id, '--start', '--json'])` BEFORE any ledger flip.
- **Storage**: `.cache/barrier-base-<sanitized-group_id>` (written by `--record-base`) and the
  gc-protected ref `refs/kaola-workflow/barrier-base/<sanitized-group_id>` (the anchor).
- **Crash-safety**: the group baseline is recorded before the `running-set.json` write (state
  `'opening'`), so the manifest never references an unanchored baseline.
- **Drop**: `--drop-base --node-id <group_id>` after barrier pass; a `group_id` has no ledger
  row, so the `drop_base_window_open` guard (#424) never fires for it.
- **Mismatch guard**: the `--group-barrier` handler cross-checks the `.cache` file SHA against
  the gc-anchored ref SHA (same #368 guard as per-node barriers).

## Per-path attribution and cross-lane protection posture

- **Within the union allowlist**: because the group is formed from pairwise-disjoint sets, a
  path in the union belongs to exactly one member. Attribution is unambiguous — the barrier
  does not need to identify which member, only whether the path is in the union.
- **Out-of-union paths**: land in the existing `outOfAllow` rank-4 arm → `unattributed_write` /
  `write_set_overflow` refusal. No new reason code.
- **Cross-lane runtime protection** — ADVISORY only. While A and B run concurrently in the
  shared parent worktree, nothing PREVENTS A from writing into B's lane in real time. The
  `KAOLA_LANE_CONTAINMENT` `PreToolUse` Write/Edit hook (#376) is advisory (fail-open); it
  enforces out-of-lane warnings but does not block the write. Enforcement is retrospective:
  (a) the GROUP BARRIER at the last close refuses an out-of-union stray, and (b) the
  `--finalize-check` attribution sweep (#424) verifies every branch change is attributed to a
  complete node's declared set. This is a deliberate scope boundary, not a gap.

## Advisory finding R1 (n5-review)

The code-reviewer (n5) identified one advisory issue:

> R1 (advisory): `Math.max(2, groupCeiling)` floors the lane group to 2 even when `--max 1` was
> passed under containment, opening 2 disjoint writes instead of honoring the explicit ceiling.
> Not a safety defect (writes are parallel-safe-verified and the group barrier still applies),
> but it overrides an explicit concurrency limit. Cleaner fix: suppress grouping when `max === 1`
> and degrade to serial. Filed as a follow-up; non-blocking for #437.

The affected line is in `runOpenReady` (~L2643): `groupCeiling = Math.max(2, groupCeiling)`. The
guard is the deliberate floor that prevents a 1-member "group" from being formed; the oversight
is that `--max 1` should suppress the group path entirely rather than floor to 2.

## Decision summary

1. `lane_group` is ONE additive key inside the existing `running-set.json` — no new file, no new
   schema constant, outside `plan_hash`. Minimal surface; reconcile-compatible; flag-OFF invisible.
2. The group baseline reuses the per-node baseline machinery (`cacheBaseFile` / `barrierRef` /
   `anchorBase` / `snapshotWorktree` / `--record-base` / `--drop-base`) keyed by `group_id`.
   Zero new baseline code; gc-anchored + #368 mismatch-guarded for free.
3. Member close = evidence + per-member in-lane vacuity ONLY; the diff barrier is DEFERRED to
   the LAST member. The structural fix — never diff a member against a still-open sibling's
   writes.
4. The group barrier REUSES `barrierCheck` with a union-of-members allowlist. The EXISTING rank-4
   `unattributed_write` arm fires on cross-lane strays. No new reason code.
5. `--parallel-safe` EXPOSES the existing antichain pair-loop predicates as a read-only CLI
   check. `open-ready` degrades to serial on overlap, so overlap never co-opens.
6. Cross-lane runtime protection is ADVISORY; the group barrier + `--finalize-check` attribution
   sweep are the enforcement teeth.
7. ALL new behavior is gated on `resolveLaneContainment`; flag OFF ⟹ byte-identical ×4 (INV-6).
