## Context

`#512` raised the per-chain `run-chains` kill ceiling (600s → 900s, `KAOLA_RUN_CHAINS_TIMEOUT_MS`,
D-512-01) to stop false-killing a passing-but-slow `test:kaola-workflow:claude` chain, but it
**explicitly deferred diagnosing *why* the chain takes ~574s** ("genuine suite growth vs.
environment — the larger, uncertain effort"). This decision records that deferred diagnosis.

The chain is 37 `&&`-chained commands: 35 `node scripts/…` validators, a `bash -n install.sh`
syntax check, and a `package.json` parse, ending in `scripts/simulate-workflow-walkthrough.js`
(~14,377 LOC, 246 scenarios, ~505 static subprocess-spawn call-sites).

This was run as a `#486` **Case-B read-only shaping investigation** (probe → assume → adversarially
falsify → converge): map the cost surface, measure it, hypothesize, try to refute, then decide the
shape of any remaining work. The adversarial falsification **overturned the issue's seed premise**.

## AC#1 — Profile breakdown (measured)

Total measured ~358s in this session; the issue's ~574s reference is the **same structure under
heavier system load** (the walkthrough alone swung 140s → 180s, ~28%, between two consecutive
same-machine runs). The durable, load-invariant finding is the **structural ranking**, not any
absolute second-count.

- **#16 `scripts/test-adaptive-node.js` — ~122s (~34% of the chain).** Instrumented: 780 top-level
  spawns; time-inside-spawns = 99.4% of the run. The dominant slice is **112 real
  `kaola-workflow-adaptive-node.js` CLI calls = ~104.7s at ~935ms/call**, of which bare process
  startup is only ~30ms (~3%); the other ~97% is genuine work (plan-hash validation, ledger splice,
  barrier diff, task-mirror regeneration across nested real validator / commit-node / next-action
  processes).
- **#37 `scripts/simulate-workflow-walkthrough.js` — ~160s (~44% of the chain).** Cost is
  **distributed**, not point-concentrated: testAdaptive\* (45 scenarios, ~33s), testSink\* (28, ~29s),
  testFinalize\*/shared-tmp (~18s), testClosure\* (~11s). Each E2E scenario runs a full real git
  worktree + finalize + sink-merge transaction.
- **#16 + #37 together ≈ 75% of the chain.** The remaining ~25% is the 34 other validators,
  mostly cheap (most <1s; a long tail at #9/#19 ~14s, #20 ~11s, #6 ~11s, #11 ~9s, #14 ~8s, #25 ~6s —
  all genuine subprocess setup, none pathological).

## AC#2 — Decision: genuine suite growth, no safe behavior-preserving reduction

The three hypotheses were tested empirically by an independent skeptic (n4):

- **H1 REFUTED — spawn-count is not the lever.** The per-spawn startup tax is ~0.02–0.05s, not the
  assumed 150–250ms. `(t × effective_spawn_count) / 131s` is ~7% (≤17% on a deliberately inflated
  upper bound) — below the 15% refute threshold. The time is inside per-invocation **logic / real
  work**, so reducing spawn *count* cannot help without removing the work itself.
- **H2 REFUTED — no avoidable redundancy.** Clone-from-template `s_clone` (158ms) ≈ `s_init` (159ms)
  (`cp -r` is *slower*, 214ms) → `N × Δ ≈ 0`; and git is only ~5% of #16's cost. The 6×
  `--freeze --repair` plan builds in #16 are **6 structurally distinct plan DAGs** (linear / fan-out
  / solo / diamond / split-guard / speculative), each a different `plan_hash` over a different Nodes
  table for a different failure mode — deduplicating them *deletes coverage*, it does not share
  redundant work.
- **H3 CONFIRMED — irreducible coverage.** #16 is already optimally partitioned: 327 pure-seam
  (no-subprocess) assertions via injected seams, and only ~112 deliberate real-CLI spawns that ARE
  the **#292 anti-false-green discipline** — the cross-process on-disk state handoff (one process
  writes ledger + baseline and exits; the next re-reads it) IS the property under test. Collapsing
  the real spawns into one process, stubbing the integrity seams, or batching the sequential
  lifecycle re-introduces exactly the in-process false-green class #292 exists to prevent — a
  coverage regression.

**The one remaining candidate, cross-command / cross-scenario parallelism, is rejected for #523.**
It reduces *makespan* but not *CPU work*, and under precedence #1 (accuracy is non-negotiable; a
flaky correctness gate is the most expensive outcome) its downside dominates:

1. it is the exact CPU/IO contention n2 deliberately avoided to keep measurement accurate;
2. it risks races/flakiness across the real `git worktree` / `merge --no-ff` / `update-ref` / ledger
   transactions the suite performs;
3. the 15-member shared-tmp group is one indivisible serial isolation unit by construction
   (`simulate-workflow-walkthrough.js:12731-12747`);
4. it breaks the `A && B && C` deterministic first-failure attribution and clean exit code;
5. it is a #307 cross-edition change touching the shared `&&`-chain runner (`package.json`) and the
   six-copy walkthrough family.

Per CLAUDE.md this is **"genuine suite growth — no safe reduction."** The ~574s is the price of the
project's non-negotiable internal correctness gates doing exactly the work they exist to do.

## Recommended next step

**Close #523 with this documented-growth decision. Do NOT author a D-523-02 build run.** Shipping a
parallelism change would be net-negative under precedence #1. If makespan reduction is later judged
worth a dedicated, risk-bounded effort, it belongs in a **separate design track**, where the
per-leg isolation and race-safety machinery to make concurrent real-git tests safe would be designed
properly — not bolted onto a passing correctness gate as a speed fix.

**Amendment (2026-06-18, post-closure, owner direction):** that dedicated design track is now filed
as **#526** (race-safe test-suite parallelism for chain makespan), carrying the five hard constraints
above as its acceptance gate. This supersedes the earlier in-record pointer to #463: #463 is **closed**
and was about per-leg write isolation in the adaptive *run* path (a runtime concern), **not** test-suite
*validation* parallelism. The original "no new issue is filed" recommendation reflected the at-closure
judgment that the lever was not worth pursuing as a quick fix; #526 tracks it explicitly as a *future
dedicated design effort* (still gated on the same accuracy-first constraints), not a quick fix.

## Consequences

- The #512-deferred "why ~574s" question (D-512-01) is resolved: genuine end-to-end transaction work
  in two correctness gates, not timer padding, not a regression, not avoidable scaffolding.
- No code change ships for #523; the deliverable is this decision record (docs-only, zero code blast
  radius — no #307 obligation attaches to closing #523).
- The `KAOLA_RUN_CHAINS_TIMEOUT_MS` ceiling (D-512-01, default 900s) remains the correct operational
  guard for the genuinely-slow-but-passing chain.
- Future readers proposing a "speed up the test suite" change should start here: spawn-reduction and
  fixture-sharing are already refuted; only race-safe parallelism (a design-track effort) remains.
