# D-682-01 — Review attempts are authoritative transactions; repair ownership stays agent-selected

- **Status:** accepted
- **Date:** 2026-07-14
- **Context:** issue #682

## Decision

Persist every adaptive gate close in a plan-bound, schema-validated
`.cache/review-attempts.json` journal. A review attempt is identified by the frozen plan hash, a
canonical logical-gate identity, a gate-local positive ordinal, the reviewed candidate digest, and
the exact evidence generations. Canonical gate identity is the gate kind plus sorted origin/member
sets; display ids and reusable fan-out labels are metadata, not authority. Physical JSON array order
is never chronological authority.

All close paths use one effective-verdict predicate: `verdict: pass`, zero blocking findings, and no
unresolved canonical in-scope `action=fix` finding. Sequence gates settle one exact receipt. Fan-out
gates remain provisional until every exact member has voted, then settle by strict majority. A
failure folds the gate members to `pending`, removes them from the running set, and records lifecycle
settlement last. The pending rows remain held by the settled unconsumed attempt: orient, ordinary
opening, and generic reopen fail closed until direct repair consumes it.

Repair ownership remains an agent judgment over the frozen DAG. The agent supplies an attempt id and
writer id; the harness proves that writer is the unique maximal executed producer common to the gate,
then verifies the unchanged candidate digest, immutable original barrier identity, and original
barrier result before its first mutation. The harness never chooses between multiple writers and
never rewrites the DAG: zero or multiple valid owners require replan.

The repair transaction records the selection, reopens the writer while retaining its original
baseline, removes stale downstream artifacts, records repair settlement, and consumes the attempt in
separate durable steps. Retrying after any step resumes the same attempt. When sibling cleanup sees
multiple matching attempts, it selects the greatest validated ordinal independently for each logical
gate member, so a physically reordered journal cannot let an older pass erase a newer unresolved
failure. Each canonical logical gate permits five settled-and-consumed repairs; the sixth refuses
without mutation. Canonical findings routing is stored with the attempt, while
`findings-route.json` remains a regenerable projection only.

## Why

Gate evidence, ledger state, running-set state, and repair routing previously could be interpreted as
separate snapshots. That made crash boundaries and repeated gate generations vulnerable to divergent
classification, and made physical serialization order look like chronology. One authoritative
transaction binds the receipt, candidate, gate identity, routing, settlement, and repair lineage while
leaving the frozen plan and Node Ledger as the scheduler's source of truth.

Keeping ownership agent-selected preserves the workflow boundary: agents decide semantic repair
responsibility; scripts prove identity, atomicity, and bounded retry. Unique-maximal-producer proof
allows safe direct repair when the frozen graph has one defensible owner and produces a typed replan
when it does not.

## Consequences

- Malformed, plan-mismatched, duplicate, non-contiguous, receipt-substituted, or route-divergent
  journals fail closed.
- Both `close-node` and `close-and-open-next` share the same verdict and settlement behavior.
- Fan-out cannot certify early, substitute a receipt after outcome, or replay a settled failure as a
  new vote.
- Crash retries converge after attempt/outcome write, plan fold, running-set removal, writer reopen,
  downstream cleanup, repair settlement, and consumption.
- The journal is durable evidence, not a scheduler, repair-owner selector, or second state machine.
- R17 is explicitly deferred and pre-existing: the shared atomic replace helper fsyncs the temporary
  file and atomically renames it, but does not fsync the parent directory. Issue #682 neither changes
  nor worsens that helper; parent-directory fsync remains a separate follow-up.
