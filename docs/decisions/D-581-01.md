# D-581-01. Codex dispatch descriptors carry task-name identity, direct effort overrides, and profile metadata

Date: 2026-07-01
Status: Accepted
Issue: #581

## Context

Codex role profiles are installed globally by default, but the standalone TOML files only carried
`name` and `developer_instructions`. The richer managed config block already had each role's
`description` and `nickname_candidates`, so custom subagent discovery could lose useful identity
metadata when reading the standalone profile files.

The adaptive dispatch guidance also still described an older effort workaround where an Opus-planned
node required changing the parent session effort before spawning. Current Codex supports per-spawn
`reasoning_effort`, and the v2 multi-agent surface accepts caller-provided task names. That means the
workflow can preserve stable node identity when v2 is enabled while still acknowledging that the
stable v1 fallback may show thread IDs in wait/close rows.

## Decision

- Standalone Codex role TOMLs carry `name`, `description`, `nickname_candidates`, and
  `developer_instructions`.
- The installer and preflight schema checks derive expected descriptions and nicknames from
  `config/agents.toml` and reject stale/malformed profile metadata.
- The adaptive dispatch descriptor carries `codex_dispatch_mode` and `codex_task_name` in addition to
  the existing role, evidence, nonce, and effort fields.
- `codex_task_name` is derived from workflow node id plus role, lowercased, restricted to
  `[a-z0-9_]`, collapsed, trimmed, and falls back to `node` if empty.
- `codex_reasoning_effort` is passed directly as per-spawn `reasoning_effort` when present.
- v2 task-name dispatch is used only when explicitly enabled; v1 remains the stable default and does
  not claim that runtime wait rows are renamed.

## Consequences

- A single global Codex install exposes complete role identity metadata for every repo.
- Plan-run can map a visible v2 task name back to the frozen node ledger without relying on timing of
  nickname metadata.
- v1 runs remain honest: prompts and evidence carry node identity, but runtime status rows may still
  show thread IDs.
- The four edition chains remain the release gate because installer, preflight, generated aggregator,
  profile TOML, and plan-run surfaces all move together.
