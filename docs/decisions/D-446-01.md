## Problem

Two structural friction points in the adaptive plan-run, both rooted in adaptive-node.js, settled
together because both are adaptive-node-only Ă—4-port changes.

**Gap 1 â€” unstructured gate findings.** A gate node (`code-reviewer`, `security-reviewer`,
`adversarial-verifier`) records its findings as free-prose `finding:` lines in its evidence file
`.cache/{node-id}.md`. To route a repair, an orchestrator must RE-READ the full evidence prose and
mentally map each finding to (a) the file it concerns, (b) which plan node owns that file, and (c)
which role should fix it. There is no machine-consumable routing artifact â€” the routing is redone by
hand every gate close.

**Gap 2 â€” full-JSON envelope flooding.** Every aggregator subcommand prints the FULL JSON envelope on
EVERY invocation, success or failure. In an interactive plan-run the operator sees a wall of JSON on
every routine `close-and-open-next`, even when `result: ok` and there is nothing to inspect. The full
envelope is only useful on `result: refuse` (when the operator must read the offending paths / triage
/ hint). The signal (a refusal worth reading) drowns in the noise (dozens of successful full
envelopes).

Both are settled here. `route-findings` turns the gate-evidence prose into a structured routing JSON;
`--summary` collapses the routine envelope to one line and caches the full JSON for drill-in only on
refusal.

## Decision

### 1. `route-findings` is a SUBCOMMAND on adaptive-node.js â€” not a new script

`route-findings` is a new subcommand on `scripts/kaola-workflow-adaptive-node.js`, NOT a standalone
script. It reads a gate node's evidence file `.cache/{node-id}.md`, parses its `finding:` lines, and
writes `.cache/findings-route.json`.

Because `adaptive-node.js` is ALREADY registered in COMMON_SCRIPTS (the install manifest) and in all
three `install.sh` SUPPORT_SCRIPT_NAMES blocks, adding a subcommand requires **NO new install
manifest entry and NO new install.sh line** â€” the install surface is unchanged. This is the deciding
reason for "subcommand not new script": a new script would force a four-edition install-manifest +
SUPPORT_SCRIPT_NAMES count-bump (the recurring count-bump landmine); a subcommand inherits the host's
existing registration.

**Rationale.** The adaptive-node aggregator already owns the `.cache/{node-id}.md` evidence lifecycle
(`record-evidence`, the evidence-check in `close-and-open-next`). Routing the findings it already
reads is a natural extension of that ownership, and it keeps the install manifest stable â€” exactly the
constraint that makes a subcommand (not a script) the correct shape.

### 2. `findings-route.json` schema â€” array of `{ finding_id, file, owning_node, fix_role, status }`

`route-findings` writes `.cache/findings-route.json`: an ARRAY of objects, one per parsed `finding:`
line:

```json
[
  {
    "finding_id": "F1",
    "file": "scripts/kaola-workflow-adaptive-node.js",
    "owning_node": "n4",
    "fix_role": "implementer",
    "status": "open"
  }
]
```

Field contract:

- **`finding_id`** (`string`) â€” the finding's identifier as written in the evidence (`F1`, `F2`, â€¦).
- **`file`** (`string`) â€” the file the finding concerns, parsed from the `finding:` line.
- **`owning_node`** (`string | null`) â€” the frozen-plan node whose declared write set contains `file`,
  resolved by a write-set lookup over the frozen plan's nodes. An UNOWNED file (no node declares it)
  yields **`owning_node: null` â€” the plan-repair signal**: the finding concerns a file no node owns, so
  the orchestrator must widen a write set (or add a node) before a writer can fix it.
- **`fix_role`** (`string`) â€” inferred, in precedence:
  1. `security` appears in the finding text â†’ `security-reviewer`.
  2. else the last code-producing node in the dependency path that DECLARED `file` â†’ `implementer`.
  3. else (no producing node found) â†’ `code-reviewer`.
- **`status`** (`"open" | "n/a"`) â€” `open` for an actionable finding; `n/a` for an explicitly
  dismissed / non-blocking finding line.

**Rationale.** The array shape is the routing TABLE an orchestrator dispatches from: each row says
"finding X, in file Y, owned by node Z, fixed by role R, currently S." `owning_node: null` being the
plan-repair signal makes the most expensive case (a finding about an unowned file) machine-detectable
rather than discovered at barrier-fail time. The `fix_role` precedence encodes the real dispatch rule:
security findings go to the security reviewer; a file with a known producing writer goes back to that
writer (`implementer`); an orphan goes to the reviewer to re-scope.

### 3. `close-and-open-next` auto-invokes `route-findings` on a `VERDICT_ROLES` close

`close-and-open-next` AUTOMATICALLY invokes `route-findings` when the node it is closing is a gate
node â€” i.e. its role is in:

```js
const VERDICT_ROLES = ["code-reviewer", "security-reviewer", "adversarial-verifier"];
```

The auto-invoke is **silent and non-blocking**: any error from `route-findings` is LOGGED, never
raised â€” it MUST NOT block the node advance. A gate node that closes clean still produces a
`findings-route.json` (possibly an empty array); a gate node with findings produces the routed table
as a side effect of closing, with no extra operator step.

**Rationale.** The routing artifact is most valuable EXACTLY when a gate node closes â€” that is when
findings exist and a repair must be routed. Auto-invoking at that close means the orchestrator always
has `findings-route.json` available when it needs it, with no separate manual call. Making it silent /
non-blocking is mandatory: routing is a convenience artifact, never a gate; a `route-findings` failure
(e.g. malformed evidence) must degrade to "no routing table" and let the advance proceed, never strand
the node.

### 4. `--summary` one-line format + cached full envelope

`--summary` is a NEW flag on `adaptive-node.js`. When present, the subcommand prints ONE line:

```
summary: <result> [| reason: <reason>] [| hint: <operator_hint>]
```

â€” `result` always; `| reason: <reason>` only when a `reason` is present; `| hint: <operator_hint>`
only when an `operator_hint` (D-445-01) is present. The FULL envelope JSON is written to
`.cache/<op>-envelope.json`, where `<op>` is the subcommand name (e.g.
`.cache/close-and-open-next-envelope.json`, `.cache/open-next-envelope.json`). On `result: refuse`
the interactive loop (the D-445-01 skeleton) reads `.cache/<op>-envelope.json` for the full detail;
on success it needs nothing more than the one line.

**Rationale.** The one-line format carries exactly the three things an operator scans for â€” outcome,
why (if not ok), and what-to-do (if actionable) â€” and nothing else. Caching the full envelope to a
deterministic `<op>`-named path means the drill-in target is computable from the subcommand name
alone (no parsing the summary to find a file). The bracketed-optional segments keep a success line
minimal (`summary: ok`) while a refusal line stays self-describing
(`summary: refuse | reason: write_set_overflow | hint: ... revert-overflow ...`).

### 5. Compat contract â€” `--summary` is purely additive; default output unchanged

Default output (no `--summary` flag) is **unchanged FULL JSON**. `--summary` is opt-in. Every
orchestration script and test that parses the full-JSON stdout is unaffected â€” they simply do not pass
`--summary`. Only the D-445-01 interactive plan-run skeleton opts into `--summary`.

**Rationale.** The full-JSON default is the orchestration contract: scripts and the walkthrough
simulation parse it. Changing the default would be a breaking change across the test suite and the
aggregator-direct loop. `--summary` therefore MUST be additive â€” a new flag that changes output ONLY
when explicitly passed â€” so the diet helps the human interactive path without touching the machine
path.

### 6. Ă—4 edition propagation â€” adaptive-node-only, ONE node

`route-findings` and `--summary` are adaptive-node-ONLY: they live entirely in
`kaola-workflow-adaptive-node.js`, which ships in all four editions (canonical, codex byte-twin,
gitlab/gitea hand-ports). Per the #340/#431 Ă—4-port rules, BOTH features propagate to all four trees
in ONE node (n5 of this bundle's plan), and ALL FOUR
`npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains MUST be green, run sequentially â€” a
green claude chain alone is insufficient (the `&&` short-circuit hides a red forge chain, #307).
`VERDICT_ROLES`, the `findings-route.json` schema, and the `--summary` format must be functionally
identical across the four ports.

**Rationale.** A subcommand + flag on a four-edition script is a CROSS-EDITION diff by definition: a
feature present only in the canonical tree would diverge the editions. Landing both in one node
(rather than scattering the four ports across nodes) keeps the four-chain green gate a single
verifiable checkpoint, the #340/#431 discipline.

## Consequences

**Gate findings become a routing table, not re-read prose.** `route-findings` (decisions 1â€“2) parses
`finding:` lines into `.cache/findings-route.json` â€” each row carrying `{ finding_id, file,
owning_node, fix_role, status }`. `owning_node: null` is the machine-detectable plan-repair signal;
`fix_role` encodes the dispatch rule (security â†’ `security-reviewer`, known producer â†’ `implementer`,
orphan â†’ `code-reviewer`). `close-and-open-next` produces this table automatically on every
`VERDICT_ROLES` close (decision 3), silently and non-blockingly.

**The interactive envelope stops flooding.** `--summary` (decision 4) collapses a routine outcome to
one line and caches the full envelope at `.cache/<op>-envelope.json` for drill-in on `result:
refuse`. The one-line format surfaces `operator_hint` (D-445-01), so the human path is: scan one line,
act on the hint, drill into the cached envelope only when refused.

**Both are additive and install-stable.** `route-findings` is a subcommand (no install-manifest entry,
decision 1); `--summary` is an opt-in flag with unchanged default output (decision 5). The install
surface and the full-JSON orchestration contract are both untouched.

**Test fixtures the implementation must cover (binding for the implementer):**

- `route-findings` parses N `finding:` lines from `.cache/{node-id}.md` into an N-element
  `.cache/findings-route.json` array with the exact `{ finding_id, file, owning_node, fix_role,
  status }` keys.
- A finding whose `file` is declared by a frozen-plan node resolves `owning_node` to that node id; a
  finding whose `file` is owned by NO node resolves `owning_node: null` (the plan-repair signal).
- `fix_role` inference: a finding containing `security` â†’ `security-reviewer`; a file with a known
  producing/declaring node â†’ `implementer`; otherwise â†’ `code-reviewer`.
- `close-and-open-next` on a `code-reviewer` / `security-reviewer` / `adversarial-verifier` node
  writes `findings-route.json` as a side effect; a `route-findings` failure during that close LOGS and
  does NOT block the advance (the node still closes/advances).
- `close-and-open-next` on a NON-`VERDICT_ROLES` node does NOT invoke `route-findings`.
- `--summary` prints exactly `summary: <result>` (+ optional `| reason:` + `| hint:` segments) and
  writes the full envelope to `.cache/<op>-envelope.json` named by the subcommand; default (no
  `--summary`) output is byte-unchanged FULL JSON.
- `--summary` surfaces `operator_hint` (D-445-01) in the `| hint:` segment when present.
- `route-findings`, `--summary`, `VERDICT_ROLES`, and the `findings-route.json` schema are present and
  functionally identical in all four editions; all four chains green, run sequentially (#307).

**For downstream implementation:** the load-bearing constraints are decision 1 (subcommand, NOT a new
script â€” keeps the install manifest stable), decision 2's `owning_node: null` plan-repair signal +
`fix_role` precedence (the dispatch rule), decision 3's silent-non-blocking auto-invoke (routing never
gates an advance), and decision 5's additive-compat default (full-JSON default unchanged). Both
features land in ONE Ă—4 node (decision 6) under the four-chain green gate.
