# D-570-01. Gate the `run-chains.js` invocation on self-host detection at the plan-run→finalize seam (consumer repos route to `validation_command` + `final-validation.md`)

Date: 2026-06-26
Status: Accepted
Issue: #570
Related: #475 (consumer `final-validation` gate — the branch this pushes up to the point of use),
         #432 (chain-receipt — the self-host greenness witness), #546 (run-chains `--project`),
         #501 (self-sufficient; CI/CD is not a gate), #464 (push validation inward to agent checks),
         #307 (four-chain cross-edition parity), #400 (six-surface routing-prose propagation).

## Context

The self-host vs consumer validation split (#475) was documented in the finalize *prerequisite*
section and enforced at the `--finalize-check` gate, but the two **operational** steps that invoke
`kaola-workflow-run-chains.js` — plan-run §5 "All done" and the finalize pre-contractor step — were
stated **unconditionally**.

`run-chains.js` only runs the maintainer edition chains (`npm run test:kaola-workflow:*`) and is
meaningful solely in the Kaola-Workflow self-host. A consumer (non-npm) product repo declares no
`test:kaola-workflow:*` scripts, so the script can only return `chains_config_missing` there.

Runtime evidence (a real downstream consumer run): the agent followed the prose, launched run-chains,
received `chains_config_missing`, and only then pivoted to the consumer path — having narrated that
run-chains would execute the project's own `validation_command`. It does not; it runs the edition
chains. The guard held, but the prose wasted a round-trip and invited a wrong mental model. Latent
risk: consumer mode is detected by the *absence* of the scripts, so a repo that vendored or forked
Kaola-Workflow (and therefore declares such scripts) would have run-chains actually fire the
maintainer chains — meaningless in that repo.

## Decision

Push the existing #475 branch up to the point of use. Gate both operational steps on self-host
detection (presence of `test:kaola-workflow:*` in the repo's `package.json`):

- **Self-host (npm):** run `run-chains.js` → `.cache/chain-receipt.json`; the contractor verifies the
  receipt; `cmdFinalize` gates fail-closed. **Unchanged.**
- **Consumer (non-npm):** do **NOT** invoke `run-chains.js`. Run the plan's `## Meta`
  `validation_command` and record `.cache/final-validation.md` with a column-0 `verdict: pass`.
  `--finalize-check` auto-detects consumer mode (absence of the scripts) and gates on that file (#475).

This is a **prose / routing change only** — no script logic changes. `run-chains.js` and the
`--finalize-check` consumer auto-detection already behave correctly; the `chains_config_missing`
refusal remains as the defense-in-depth backstop.

## Consequences

- Consumer-repo agents go straight to the correct validation path — no wasted run-chains attempt, no
  conflation of the maintainer edition chains with the project's own `validation_command`.
- Self-host behavior is byte-unchanged.
- Cross-edition (#307): the branch is applied at both points of use across all six routing surfaces
  (#400) — `commands/kaola-workflow-plan-run.md` + 3 Codex plan-run SKILLs + gitlab/gitea command
  ports, and `commands/kaola-workflow-finalize.md` + 3 Codex finalize SKILLs + gitlab/gitea command
  ports. All four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains green.
- The route-reachability suite + the four `validate-*-contracts.js` continue to machine-enforce
  surface presence (PIN comments) and the `final-validation.md` contract needle (#475).

## Alternatives considered

- **Make `run-chains.js` self-detect and route prose-side** — rejected: the script already refuses
  cleanly (`chains_config_missing`); the gap is that the *prose* relied on that runtime refusal as the
  branch signal instead of pre-branching. Fixing the prose is the minimal, correct change (precedence
  #3: cheapest sufficient mechanism).
- **Stop installing `run-chains.js` into consumer repos** — rejected: it is a shared support script
  and harmless when unused; removing it would be a larger, lower-value change and does not address the
  finalize-seam prose.
