# D-495-01. Front-door three-bucket split — typed transient retry + `result: escalate` for indeterminate classifier faults at claim time (issue #495)

**Date:** 2026-06-16
**Status:** Accepted
**Issue:** #495 (adaptive starting contract hardens opaque classifier-subprocess fault)
**Related:** #44 (Agent Owns Reasoning; Scripts Own Atomicity), #287 (Escalate values, not facts),
            D-476-01 (unknown-flag typed refusal — prior claim.js hardening),
            plan-validator.js `barrierCheck` typed `reason` field (the reused typed-envelope pattern)

## Problem

`classifyIssue` in `kaola-workflow-claim.js` (the startup entry for `cmdStartup` / `cmdPickNext`)
spawned the classifier subprocess via `execFileSync`. When that spawn faulted transiently — ENOENT,
EAGAIN, EMFILE, SIGKILL, or a timeout — the error was **swallowed at catch** (lines 690–692 in the
pre-#495 code): the function returned `{ verdict: 'target_unavailable' }`, the same verdict as a
determinate forge-unreachable failure.

This collapsed two fundamentally different situations into one opaque signal:
- **Determinate:** the classifier ran and concluded the forge was unreachable → refuse, human must
  fix the network.
- **Indeterminate:** the classifier never ran because the Node.js subprocess layer itself faulted →
  unknown whether the target is claimable; retrying later may succeed.

An agent receiving `target_unavailable` for a spawn-fault had no way to distinguish the two and was
forced to hard-stop unnecessarily on a transient fault.

## Decision

### D1 — Scripts Own Atomicity: typed error classification + bounded transient retry in `classifyIssue`

A new `classifySubprocessError(e)` helper in `claim.js` partitions subprocess errors into three
classes, applied **before** any retry decision:

- `spawn_fault` — spawn never started (e.g. ENOENT, EAGAIN, EMFILE; `e.status` is null and no
  signal)
- `killed` — subprocess was signalled or timed out (`e.signal` present, or timeout property)
- `clean_nonzero` — subprocess ran and exited non-zero (`e.status` is a non-null number ≠ 2)

`status === 2` is handled separately before error classification: it means an active folder already
exists and is returned immediately as `{ verdict: 'owned' }` — never reaches retry.

Only `spawn_fault` and `killed` are **transient-eligible**. `clean_nonzero` is determinate —  the
subprocess ran and made a decision — so it breaks the retry loop immediately and maps to
`target_unavailable` (the pre-existing graceful path).

**Retry policy:** `MAX_ATTEMPTS = 3` (1 original + up to 2 retries) with a small configurable
backoff (`KAOLA_CLASSIFIER_BACKOFF_MS`, default 50 ms; overridable to 0 in tests). When all
attempts exhaust on a transient class, `classifyIssue` returns a new verdict:

```js
{ verdict: 'indeterminate', reasoning_class: 'classifier_error', reasoning: '...' }
```

### D2 — New three-bucket `result` field on the startup refusal envelope

A **`result`** field is added to the startup/claim refusal envelope to give callers a machine-readable
routing classification without string-matching `status`:

| `result` value | meaning | `status` tokens |
|---|---|---|
| `'refuse'` | Determinate failure — hard stop; do not retry | `target_unavailable`, `target_set_unavailable`, `target_set_red`, `target_set_conflicts_active_work`, `target_set_has_closed_issue` |
| `'escalate'` | Indeterminate fault — pause; ask the user | `target_indeterminate`, `target_set_indeterminate` |
| *(absent)* | Pre-#495 path or non-classify verdict | `target_unverified`, `target_set_unverified`, early validation gates |

The field is additive: pre-#495 consumers that do not read `result` are unaffected. The `result`
field is at the top level of the returned object, at the same position as `status` and `claim`.

### D3 — Agent Owns Reasoning: route on `result` at the front door (#44 / #287)

On `result: 'refuse'` — hard stop. The underlying condition is determinate; the agent should
diagnose and remediate before retrying.

On `result: 'escalate'` — **pause and ask the user** whether to retry or abort. This is the
**front-door analog of the consent valve** (per #44 "Escalate values, not facts" / #287). At claim
time there is no plan or ledger yet, so there is no adaptive-node `write-halt` ledger marker to
write. The escalation is a plain consent-halt: the script surfaces the indeterminate verdict and the
agent routes the decision to a human rather than silently failing or silently proceeding.

No approval gate is bolted onto the planner, and no agent is permitted to override a determinate
`result: 'refuse'` red verdict — D3 reuses the existing pattern, not a new mechanism.

### D4 — Cross-edition boundary analysis (load-bearing; why the fix is claim.js-only)

The issue named "claim.js / classifier.js (all 4 editions)." The fix is **claim.js-only** because
of two distinct spawn boundaries and their edition-specific applicability:

**Boundary-1 (root/Codex editions only):** `claim.js → classifier subprocess`

This is the #495 bug. `claim.js` spawns `kaola-workflow-classifier.js` via `execFileSync`. The
fault that is swallowed is the failure of THIS spawn boundary. Root and Codex run `classifier.js`
as a separate Node.js subprocess — they have boundary-1. The retry/indeterminate logic binds here.

**Boundary-2 (all four editions; in-scope for `classifier.js` or forge in-process path):**
`classifier → gh/glab/forge-CLI` internal fetch

Inside the classifier, forge API calls (`gh`/`glab`/`tea`) can fail with their own transient
network faults. These are currently swallowed gracefully: a forge CLI failure → `target_unavailable`
with no retry. This is out of scope for #495 and is the same defect class one layer down (see
Known Follow-up below).

**Forge editions (GitLab / Gitea):** The forge ports run the classifier **in-process** (no
`execFileSync` spawn). They have **only boundary-2** (the in-process forge CLI call, equivalent to
root's graceful boundary-2). There is **no boundary-1** for the retry/indeterminate logic to bind
to. Therefore, the forge-edition change is **envelope-only parity**: the `result` field is added
and a forward-compatible `indeterminate` handler is wired — treating the forge editions identically
to root/Codex on the routing contract. The forge editions are not shortchanged; boundary-1 simply
does not exist in their architecture.

This rationale makes the deviation legible without treating it as a special case: the fix is
mechanically correct for boundary-1 on the editions that have it, and the parity change on the
forge editions leaves them ready for the follow-up (boundary-2 hardening) to activate their
forward-compat handler.

## Known Related Follow-up (not this issue)

**Boundary-2 internal CLI-fetch swallow is edition-uniform across all four classifiers.** In the
root/Codex classifier, a `gh` fork fault (network drop, auth error, rate limit) is caught and
surfaced as `target_unavailable` — no retry, no `result: escalate`. The forge editions have the
same pattern for `glab` / forge-CLI. This is the same defect class as #495 one layer down: a
transient fetch fault masquerades as a determinate forge-unreachable verdict.

Hardening boundary-2 (adding retry + `indeterminate` escalation inside the classifier for forge CLI
spawns) would activate the forge forward-compat `result: escalate` handlers added in this issue.
This is a scoped follow-up and is explicitly out of scope for #495.

## Consequences

- A transient classifier-subprocess fault no longer silently hard-stops as `target_unavailable`.
  After bounded retry (3 total attempts), it surfaces as `target_indeterminate` / `result: escalate`
  and routes to a human consent-halt instead of a dead end.
- Determinate verdicts remain fail-closed — `result: 'refuse'` is a hard stop; no agent can
  override it.
- The `result` field is additive; zero consumers are broken by its addition.
- The retry loop adds at most 2 extra classifier invocations on a transient fault (50 ms backoff
  each by default), with zero overhead on the common success path.
- Reuses the existing typed-envelope pattern (`plan-validator.js` `reason` family; D-476-01
  `unknown_flag` refusal) rather than introducing a new mechanism.
