# D-572-01. Re-ground the workflow-init-injected `## Kaola-Workflow` block on the adaptive DAG-of-roles model and lock it with a phase-ban + cross-forge/opencode parity

Date: 2026-06-27
Status: Accepted
Issue: #572
Related: #538 (adaptive is the unconditional default — the contradiction this resolves),
         #309 (one semantic prose change spanning N editions must be one node, mirrored verbatim),
         #341 (forge-neutral plugin prose), #274 (COMMON_SCRIPTS byte-mirror pair),
         #307 (four-chain cross-edition parity), #400 (init-surface propagation),
         D-530-02 (opencode is an additive runtime edition, not a forge).

## Context

`workflow-init` injects a `## Kaola-Workflow` block (the `<!-- KW-CLAUDE-TEMPLATE-START -->` …
`<!-- KW-CLAUDE-TEMPLATE-END -->` region) into a consumer's `CLAUDE.md`. That block still taught the
**retired 6-phase-as-default** model: numbered `Phase 1/2/3/4` boundaries, "phase files", a "current
phase file", and `fast-summary.md`/bare `.cache/` as the default durable artifacts. #538 made the
adaptive DAG-of-roles the **unconditional default** (`fast`/`full` are install-time opt-ins), so the
injected template actively contradicted the runtime every consumer now gets — onboarding new repos
with stale vocabulary.

Parity was also only half-enforced. The 6 template surfaces (3 `commands/workflow-init.md` + 3 init
`SKILL.md`) carry the region byte-identically modulo a single forge-noun line, but no contract
**banned** the phase vocabulary or proved **cross-forge** content equivalence; and the opencode
edition (generated from canonical, `.opencode/` fully gitignored) had no phase-ban/parity check of
its own.

The contract validators additionally PIN durable-state needles on those surfaces via `assertConcept`,
so the durable-state bullet could not be rewritten without moving the validators in the same commit —
making the content rewrite and the needle update inseparable (one node).

## Decision

1. **Re-ground all 6 injected templates on the adaptive model, phase-free** (mirrored verbatim modulo
   the single forge-noun line, #309). The block now: states the model in one line — a task-shaped DAG
   of role nodes; the `planner` authors and freezes `workflow-plan.md`; the executor runs it
   node-by-node via the running-set scheduler — names nodes by **function** (read/research →
   `code-explorer`/`knowledge-lookup`; strategy/blueprint → `planner`/`code-architect`; execution →
   `tdd-guide` test-first or `implementer` for refactors/scaffolding/config with no natural failing
   test; gates → `code-reviewer`/`adversarial-verifier`; docs → `doc-updater`), points durable-state
   bullets at `workflow-plan.md` (its `## Node Ledger`) + `.cache/{node-id}.md` instead of "phase
   files", and demotes `fast`/`full` to an explicit install-time opt-in aside. The exact
   `KW-CLAUDE-TEMPLATE-START/END` markers and the non-template prose (`MANDATORY` sentinel, redirect
   blocks, the Codex-note outside the markers) are preserved.

2. **Lock it with enforcement.** The phase-number ban (`/Phase\s+\d/`) + phase-file/artifact ban +
   GitHub≡GitLab≡Gitea cross-forge content parity (modulo the forge-noun line) live in
   `scripts/validate-kaola-workflow-contracts.js` (the one validator that already reads all 6
   `initFiles` and all three forge trees). The opencode phase-ban + parity (modulo the
   `Claude Code agents → subagents` runtime-noun transform) lives in `scripts/test-opencode-edition.js`
   — opencode's own additive suite (D-530-02).

3. **Move the pinned needles, not the unrelated ones.** Every init durable-state `assertConcept`
   (claude `scripts/`, its #274 plugin byte-mirror, codex, gitlab, gitea) drops `fast-summary.md` +
   bare `.cache/` and adds `workflow-plan.md` + `## Node Ledger` + `.cache/{node-id}.md`. The
   unrelated fast-path `fast-summary.md` needles on the classifier / `workflow-next` surfaces are left
   untouched — the fast path still exists as an opt-in.

This is a genuine tdd-guide cycle: the `Phase <n>` ban is a meaningful failing test (RED against the
phase-laden template, GREEN only after the rewrite). Contrast #570 (pure prose, no writable failing
test → `implementer`).

## Consequences

- A freshly `workflow-init`'d consumer repo gets a `CLAUDE.md` block that matches the adaptive default
  it actually runs — no retired phase vocabulary.
- The phase vocabulary cannot regress into the injected block: the codex chain fails closed on any
  `Phase <n>` / "phase file" token in any forge's extracted template, and on any cross-forge content
  drift; the opencode suite enforces the same on the generated edition.
- The `.opencode/` enforcement stays in opencode's own suite — the four-chain contract validators
  never read the gitignored, generated edition (D-530-02). opencode triggers no #307 obligation.
- Cross-edition obligation (#307): the diff touches the edition trees, so all four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains + `simulate-workflow-walkthrough.js`
  must be green, plus `node scripts/test-opencode-edition.js` for the regenerated opencode edition.
- The injected template's Non-Negotiable Rules block remains hand-ported byte-identical across the 6
  markdown surfaces (not edition-sync'd); this change does not alter that.

## Alternatives considered

- **Split content rewrite from the needle update (two nodes).** Rejected: the validators PIN the
  durable-state needles on the surfaces, so rewriting the bullet without moving the validators in the
  same commit leaves the tree red mid-run — they cannot move independently.
- **Per-surface fan-out (one node per forge).** Rejected: a single semantic prose change spanning N
  editions must be one node with a shared canonical spec (#309), or parallel implementers diverge the
  prose (#254).
- **Enforce opencode parity in the four-chain contract validators.** Rejected: `.opencode/` is fully
  gitignored and generated — absent from a fresh checkout — so a contract validator that read it would
  be non-deterministic. `test-opencode-edition.js` regenerates from canonical and is the correct home
  (D-530-02).
- **Drop `fast`/`full` from the template entirely.** Rejected: they remain valid install-time
  opt-ins; demoting them to an explicit aside is accurate without erasing a real capability.
