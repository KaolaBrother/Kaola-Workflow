# D-653-01. Strengthen the finalize evidence contract: attestation propagation,
sink-journal disposal, candidate-hash binding, selection/gap durability

Date: 2026-07-11
Status: Accepted
Issue: #653
Related: #277/M2 (WARN-FIRST dispatch attestation this issue makes durable); #338 (contractor
         self-attest back-fill, unchanged); #424/#432/#475 (dual-mode `--finalize-check`, the gate
         this issue extends); #547/D-547-01 (`codeTreeHash` content-address hashing, reused as-is
         by the candidate-hash producer); #648/D-648-01 (citation fields `source: cited:<node-id>`
         / `validated_command` / `validated_at_head` / `reuse_boundary`, untouched — prose-only,
         no parser); #435/D-435-01 (run-gap sweep gate, whose vacuous-pass hole this issue closes);
         #328 (bundle lane; the auto-bundle `issue-scout` recommendation this issue docks);
         #602/#630 (Self-Sufficient by Default — no CI/CD coupling, unaffected)

## Context

A post-ship audit of the finalize/closure surface found four independent gaps where a
completed adaptive cycle's evidence was either lossy, ungated, or silently vacuous:

1. **Attestation warnings were receipt-only.** `checkDispatchAttestations` (#277 M2) recorded
   `claim_planner_attested`/`finalize_contractor_attested` plus any `ATTESTATION WARNING` text in
   the `cmdFinalize` JSON stdout and the mutable `.cache/dispatch-log.jsonl` — neither of which
   survives past the finalize CLI invocation as durable, human-readable archive evidence. An
   inline-bypass warning that fired during the run could go unnoticed in a later audit of the
   archived project.
2. **Sink transaction journals never expired.** `sink-receipt.json`/`sink-fallback.json` are
   crash-resume artifacts for `sink-merge.js`'s `--sink` transaction; they had no terminal cleanup
   and accumulated indefinitely in both the live and archived `.cache/` directories after a
   successful sink.
3. **Consumer final-validation had no tree binding.** The #475 consumer-mode `--finalize-check`
   gate accepted a bare column-0 `verdict: pass` in `.cache/final-validation.md` with no proof the
   verdict was ever computed against the candidate tree being finalized — a stale evidence file
   from an earlier candidate would pass silently.
4. **Gap-sweep's vacuous pass was one-directional.** `kaola-workflow-gap-sweep.js --check`
   verified every machine-swept gap was mapped in `finalization-summary.md`'s `## Run gaps`
   section (the forward direction), but never verified the reverse: a hand-typed `## Run gaps`
   entry for a gap the scanner never observed passed vacuously whenever `sweptClasses` was empty,
   bypassing machine verification entirely. Separately, the auto-bundle `issue-scout`
   recommendation that drives a bundle claim had no durable record in the archived project.

## Decision

### A. Planner-attestation warning persistence (issue #277 M2 hardening)

1. **`--attest-planner-spawn` is mandated on every planner startup call** (single-issue AND
   bundle) across all three TOML profiles, the three `kaola-workflow-adapt` SKILL packs (both the
   single-issue call and the bundle startup block), and the pre-existing
   `agents/workflow-planner.md` reference — a genuinely-dispatched `workflow-planner` back-fills
   its own otherwise-unloggable dispatch marker; an inline main-session run must not pass it.
2. **`persistAttestationToSummary(destDir, receipt)`** (new, `kaola-workflow-claim.js` +
   byte-identical Codex copy) durably appends a script-owned, presence-guarded
   (`/^## Attestation$/m`, create-if-absent) `## Attestation` section to the archived
   `finalization-summary.md`, always writing both column-0 status fields — even when both are
   `attested`, a clean result is a positive statement, not an absence — plus every
   `receipt.warnings` entry beginning `ATTESTATION WARNING` or `attestation:`, verbatim. Called in
   `cmdFinalize` immediately after `checkDispatchAttestations`, before `computeGoalCheck`.
3. **`appendClosureBlock`'s field set gains the two attestation fields** so the archived
   `workflow-state.md`'s `## Closure` block is independently self-contained — a second,
   mutually-reinforcing durable copy alongside the `## Attestation` section.
4. **Contract-validator needles** pin `--attest-planner-spawn` on every planner TOML/SKILL surface
   across all four editions, plus a persistence lock (`assertIncludes('scripts/kaola-workflow-claim.js',
   '## Attestation')` on the claude/codex validators) so removing the append silently is a
   red chain, not a silent regression.
5. **Known residual, deliberately not closed by this issue:** a contractor-authored
   `finalization-summary.md` that pre-seeds a column-0 `## Attestation` heading before finalize
   suppresses the script append (the presence guard exists for crash-resume idempotence, not
   tamper-resistance) — fenced by the script-owned `## Closure` block + stdout receipt (which
   still carry the true fields in the same run) and by finalize prose forbidding removing or
   summarizing the section. The bundle watch-pr merged-closure lane also records only the two
   status fields into each member's `## Closure` block, not verbatim warning text, into a
   per-member summary — status-field durability still prevents an unqualified clean report; full
   warning transcription on that lane is out of scope (design-scoped to the `cmdFinalize` call
   site). Both residuals were adversarially constructed and confirmed non-blocking during review.

### B. Sink-journal terminal disposal

1. **`disposeSinkJournals(mainRoot, project)`** (new, `kaola-workflow-sink-merge.js` +
   byte-identical Codex copy, hand-mirrored into the divergent gitlab/gitea sink-merge ports)
   unlinks all four candidate paths — live and archived `.cache/`, `sink-receipt.json` and
   `sink-fallback.json` — per-file try/catch, `ENOENT` treated as already-disposed (not a
   failure), any other error a non-fatal stderr warning that never fails an otherwise-successful
   sink.
2. **Call-site ordering is the correctness argument, not a test.** The call sits immediately after
   `finalReceipt` is parsed from disk into memory, strictly AFTER every `SINK_STEPS` entry, the
   #484 ancestry freshness guard, and worktree/branch teardown have all completed — so ANY earlier
   crash or refusal path returns before ever reaching the dispose call and the journal survives
   for resume, byte-unchanged from pre-#653 crash-resume behavior.
3. **The success emit gains `journal_disposed: true|false`.** Finalize/plan-run prose across all
   six surfaces now names the journals as sink-owned transaction artifacts that are never part of
   the deliverable: a stray one found on a later "clean and synced" check (an older cycle's
   residue) must be deleted, never committed.

### C. Consumer final-validation candidate-hash binding

1. **Producer: `kaola-workflow-plan-validator.js --candidate-hash [--json]`** (new mode, read-only,
   executes no tests) emits the deterministic `computeCodeTreeHash` snapshot of the CURRENT
   candidate (committed + working landable tree, minus validation-invisible paths) over the SAME
   `validation_test_consumes` band the self-host chain-receipt producer uses — the frozen plan's
   `## Meta` block is the shared band source, so producer and gate can never disagree about what
   counts as code-relevant. A git failure refuses the typed `candidate_hash_unavailable`.
2. **Parser: `parseValidatedCandidateHash(text)`** in `kaola-workflow-adaptive-schema.js` (×4
   byte-identical), following the exact `parseNodeVerdict` discipline — pure, fence-blind,
   column-0-anchored, last-well-formed-match-wins, lowercased — so a malformed 64-hex line trips
   the same fail-closed refusal as an absent one, never a silent partial bind.
3. **Gate: the consumer arm of `--finalize-check`** parses `.cache/final-validation.md`'s column-0
   `validated_candidate_hash:` line after the existing verdict check and refuses
   **`final_validation_unbound`** (no well-formed hash line — fail-closed: an omitted field must
   never bypass the binding) or **`final_validation_stale`** (the recorded hash no longer equals a
   fresh recompute — refusal payload carries `recorded_candidate_hash` + `current_candidate_hash`).
   Extended precedence: `final_validation_unverified > final_validation_failed >
   final_validation_unbound > final_validation_stale`. A matching hash's pass payload gains
   `validated_candidate_hash`.
4. **Bind LAST, never re-run.** Workflow prose across all consumer-mode finalize/plan-run surfaces
   requires computing the hash AFTER every file the validation covered has landed, and re-running
   the recorded `validation_command` (never hand-patching the hash) on a stale refusal. The gate
   compares two hashes and executes no tests anywhere — #475's "the agent owns verification"
   boundary is unchanged, and the self-host chain-receipt arm and #648's citation fields
   (`source: cited:<node-id>` / `validated_command` / `validated_at_head` / `reuse_boundary`) are
   untouched in decision terms; a citation additionally requires a FRESH
   `validated_candidate_hash` computed at citation time, since the binding is what proves the
   cited run still covers the candidate.

### D. Selection evidence + run-gap reverse containment

1. **Gap-sweep reverse containment.** `kaola-workflow-gap-sweep.js --check` (all four editions —
   root, Codex byte-copy, and the two rename-normalized forge ports) now parses the
   `## Run gaps` section FIRST, unconditionally, even when `sweptClasses` is empty. Every
   strict-grammar entry (`- <reasonClass> (<sample>): filed:|noise: ...`) must exist in the swept
   set as an exact `(reasonClass, sample)` tuple, or the gate refuses the new
   **`observed_gap_unseeded`** (payload: `unseeded: [{reasonClass, sample}]` + a remedy naming
   `.cache/run-gaps-manual.md`). A vacuous pass now requires BOTH sides empty — no swept classes
   AND no strict-grammar entries. Free-text lines that don't match the strict entry regex (e.g.
   `- none`) are ignored by design, preserving back-compat with existing summaries. The
   pre-existing forward `gaps_unswept` check is byte-unchanged below this point.
2. **Selection-evidence docking (prose-only mechanism).** On the no-issue-named auto-bundle
   branch, `workflow-next.md`'s router persists the `issue-scout`'s entire JSON recommendation
   verbatim (fenced, with a one-line `selection_mode: auto-bundle|single-issue` header) to
   `kaola-workflow/{project}/.cache/selection-evidence.md` before dispatching the executor — it
   archives with the cycle automatically. `agents/issue-scout.md` and its TOML trio each gain one
   sentence distinguishing this pre-claim persistence contract from the existing plan-node
   self-write contract.
3. **Closure receipt field.** `CLOSURE_RECEIPT_FIELDS` (×4 byte-identical) gains
   `selection_evidence: ['present', 'absent']` (plus the `goal_check`-style `null` default in
   `emptyReceipt()`). `probeSelectionEvidence(cacheDirCandidates)` (new, `kaola-workflow-claim.js`
   + byte-identical Codex copy) mirrors the attestation probe's archive-then-live candidate order.
   Advisory only — no invariant, no warning on absence, since a user-named claim legitimately has
   none (the scout never runs on that branch).

## Consequences

- A missing/inline-bypassed planner or contractor dispatch, or an out-of-band `## Attestation`
  pre-seed, can no longer produce an archived summary that reads unqualified-clean without at
  least one durable, independently-verifiable signal (the `## Closure` block's status fields)
  surviving alongside it.
- `.cache/sink-receipt.json` and `.cache/sink-fallback.json` no longer accumulate past a
  successful sink; a "clean and synced" post-sink check on a fresh terminal success finds neither
  file.
- A consumer-repo finalize can no longer pass on a stale `final-validation.md` recorded against a
  tree that has since changed — the binding is checked, not re-derived; no new test execution is
  introduced anywhere on the consumer path.
- A hand-typed `## Run gaps` entry can no longer bypass machine gap-tracking by relying on an
  empty sweep; every observed gap must be either machine-swept or manually seeded through
  `.cache/run-gaps-manual.md` before it can be mapped.
- An auto-bundle claim now leaves a durable, archived record of the `issue-scout` recommendation
  that produced it (`selection_evidence: present` in the closure receipt, the docked file itself
  under `kaola-workflow/archive/{project}/.cache/`).
- Additive throughout: four new typed refusal names (`final_validation_unbound`,
  `final_validation_stale`, `candidate_hash_unavailable`, `observed_gap_unseeded`), five new
  receipt/payload fields (`validated_candidate_hash`, `recorded_candidate_hash`/
  `current_candidate_hash` on the stale refusal, `journal_disposed`, `selection_evidence`), one
  new plan-validator CLI mode (`--candidate-hash`), and two new durable artifacts (the archived
  `## Attestation` section, `.cache/selection-evidence.md`). No existing refusal, field, or
  artifact schema changed shape or was removed.

## Non-goals

- Does not close the R1 pre-seed residual (a contractor-authored `## Attestation` heading
  suppressing the script append) — accepted as fenced by the `## Closure` block + stdout receipt,
  per the review disposition above.
- Does not extend verbatim warning transcription to the bundle watch-pr merged-closure lane —
  design-scoped to the `cmdFinalize` call site (R2, non-blocking).
- Does not add a commit-content classifier, CI/CD coupling, or any new test execution anywhere in
  the candidate-hash binding gate — it compares two hashes only.
- Does not change the `.cache/chain-receipt.json` schema, `kaola-workflow-run-chains.js`'s
  self-host chain execution semantics, or the `--release-check` gate (#651/D-651-01) — a disjoint,
  unrelated finalize-adjacent surface.
- Does not change gap-sweep's forward direction (`gaps_unswept`) or its back-compat tolerance for
  free-text `## Run gaps` lines that don't match the strict entry grammar.

## Alternatives considered

- **Make the `## Attestation` presence guard tamper-resistant** (e.g. always overwrite rather than
  presence-guard). Rejected: the guard exists for crash-resume idempotence (a re-run of
  `cmdFinalize` must not duplicate the section); overwriting on every run would also erase a
  legitimately-earlier warning from a first attempt if a later attempt runs clean, which is a
  worse silent-loss failure mode than the fenced pre-seed residual this issue accepts.
- **Re-run the validation command inside the candidate-hash gate instead of comparing hashes.**
  Rejected: it would silently widen the consumer `--finalize-check` gate into a second full test
  execution, reversing #475's "the agent owns verification" boundary and duplicating whatever the
  agent already ran.
- **Drop the vacuous-pass carve-out entirely (require a `## Run gaps` section always) instead of
  the reverse-containment fix.** Rejected: it would break every pre-#653 `finalization-summary.md`
  with no run-gap activity, a needless back-compat break for a check whose actual defect was
  directional, not presence-based.
