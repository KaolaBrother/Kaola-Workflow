# D-619-01. Fail-closed close/push verification across all issue-close sites, additive `published_head` cycle tracking, and merge-base-proven stale-branch deletion

Date: 2026-07-07
Status: Accepted
Issue: #619 (primary — fail-closed receipt integrity for issue-close and push-upstream
       verification); #620 (companion — stale-worktree-cleanup data-safety guard, same bundle);
       #631 (companion — additive `published_head` sink-receipt field, same bundle)
Related: #497/D-497-01 (the `--sink` transaction's original `sink_incomplete` closure refusal —
         the pattern #619 extends to the legacy direct-merge path); #518/D-518-01 (`branch_head`
         cycle-identity binding — left untouched by #631; `published_head` is a sibling field, not
         a replacement); #617/D-617-01 (`remote-closed-after-publish` gate + `SINK_STEPS`
         reordering on the same file — a prior fix to the same close-integrity surface, and the
         origin of the 2026-07-06 closed-issue-without-merged-code incident that #620's fix
         prevents from destroying the evidence of); #429/D-429-01 (`sink-receipt.json` crash-safe
         per-step schema, extended here); #396.5 (the pre-existing already-closed-issue-exits-1
         probe-to-disambiguate pattern this decision's new success-path probe sits beside, not
         replaces)

## Context

A live adversarial/reliability audit found three related gaps in the receipt-integrity surface of
`kaola-workflow-sink-merge.js` and `kaola-workflow-claim.js`:

1. **Fail-open close verification (#619).** `gh issue close` (or the forge-equivalent call)
   exiting `0` was trusted unconditionally on the SUCCESS path at four independent close sites —
   `postMergeCleanup`'s legacy single-issue close, its bundle-member loop, the `--sink`
   transaction's `closeOne`, and `claim.js`'s `closeIssueIdempotent` — while only the FAILURE
   (catch/throw) path re-probed the live state. A rare forge/API race (webhook lag, a flaky
   `--comment` post) can leave an issue open despite a `0` exit, and every one of these sites would
   have reported a false success. Independently, the legacy (non-`--sink`) direct-merge path in
   `postMergeCleanup` had **no refusal at all** for a genuinely failed close — it emitted
   `status:'merged'` exit `0` with only a stderr warning (issue #168's original fail-open
   baseline), unlike the `--sink` transaction's own `sink_incomplete` refusal for the identical
   failure (#497/D-497-01). And the `--sink` transaction's `push_upstream` step ran `stepDone`
   unconditionally regardless of whether the push actually reached the remote, silently attesting
   `push_upstream:'done'` on a push that failed.
2. **Unconditional stale-branch force-delete (#620).** `cmdStaleWorktreeCleanup` treated any
   closed-issue branch as stale and force-deleted it (`git branch -D`) regardless of merge status.
   `worktreeDirtyState` only inspects *uncommitted* changes (`git status --porcelain`), so a branch
   carrying committed-but-unmerged work read `'clean'` and was destroyed anyway — reproducing, in
   the very tool built to remedy it, the exact 2026-07-06 data-loss shape #617 fixed at the
   sink-merge close gate.
3. **Stale `branch_head` false-alarming a genuinely published sink (#631).** `branch_head`
   (#518/D-518-01) is stamped once at receipt init, BEFORE `doRebase` runs. A mid-flight rebase
   (a concurrent `origin/main` advance racing a `--sink` run) rewrites the feature branch's
   commits, orphaning the pre-rebase SHA from the sink target's ancestry even though the rebased
   content genuinely landed. `cmdVerifySink`, which resolved `implRef` from `branch_head` alone,
   then false-alarmed `impl_commit_not_ancestor` on a cleanly rebased, genuinely published sink.

## Decision

**Close all three gaps at their structural root — a live post-close probe on the success path, a
merge-base ancestry proof before any force-delete, and an additive fresh-tip receipt field —
rather than adding a parallel check on top of each existing mechanism.**

1. **Post-probe the live state on the close SUCCESS path too (#619).** Every close site now calls
   a probe after an exit-0 close and treats "still open" as a failure, not a success:
   - `postMergeCleanup`'s legacy single-issue close and bundle-member loop, and the `--sink`
     transaction's `closeOne`, call the sink-merge module's existing `probeIssueClosed` on the
     success path (previously it only ran in the catch branch).
   - `claim.js`'s `closeIssueIdempotent` gained a NEW, un-memoized `probeIssueClosedLive` helper,
     called on BOTH the success and catch paths. This was load-bearing, not additive-only:
     `probeIssueState` (the existing memoized probe) is called once, pre-close, at the top of
     `closeIssueIdempotent`, priming its per-process memo with the pre-close `'open'` verdict.
     Reusing that memo for a post-close re-check would always replay the STALE pre-close state —
     breaking every genuine successful close (not merely leaving a gap uncovered) and leaving the
     pre-existing catch-branch "probe" a permanent no-op. `probeIssueClosedLive` fixes both
     branches correctly on all four outcomes: success+still-open → `'failed'`; success+confirmed →
     `'closed'`; throw+confirmed → `'already_closed'`; throw+still-open → `'failed'`.
2. **Fail the legacy direct-merge path CLOSED on a genuine close failure (#619).**
   `postMergeCleanup` gains a `closeFailed` gate mirroring the `--sink` transaction's existing
   `sink_incomplete` pattern (#497): when a close was attempted and genuinely failed (single-issue
   `remote_issue_closed === 'failed'`, or any bundle member in `failed_issue_closures`), it emits
   `{result:'refuse', reason:'sink_incomplete', step:'closure', remote_issue_closed, closure_receipt,
   closure_invariants}` and exit code `1` — never `status:'merged'` exit `0`. The merge into the
   default branch has already happened by this point (irreversible); this is purely truthful
   reporting. `closeWasAttempted` excludes `OFFLINE`, keep-open, and no-issue-passed runs, so a sink
   with nothing to close is never false-flagged. `main()` is generalized from `cleanupResult.exitCode
   === 3` to any returned `exitCode`, so this new `1` propagates alongside the pre-existing `3`
   (merge-impossible) case.
3. **Verify `push_upstream` parity instead of trusting the push exit code (#619).** The `--sink`
   transaction's `push_upstream` step now checks `branch@{u}` ahead-count (`git rev-list --count
   <upstream>..<branch>` must be `0`) after the push attempt, instead of treating any push outcome
   as done. On non-parity it records `receipt.push_upstream = 'failed'`, emits a typed
   `sink_incomplete`/`step:'push_upstream'` refusal, and leaves the step NOT done so a re-run
   retries it. A `KAOLA_WORKFLOW_FORCE_PUSH_UPSTREAM_FAIL` test-only hook (mirrors the pre-existing
   `FORCE_PUSH_MAIN_FAIL`) forces a deterministic RED case.
4. **Remove the dead `worktree_sync` step; fold its copy into `merge` (#619).** The standalone
   `worktree_sync` step always ran AFTER `merge`'s worktree removal, so its own `git worktree list`
   scan could never find a match — a permanent no-op `stepDone` every run. `SINK_STEPS` drops the
   entry; the `merge` step now **stages** the worktree's project folder to a `mkdtemp` temp
   directory BEFORE the worktree is removed, then **lands** the staged copy into `mainRoot` only
   AFTER checkout/rebase/FF-merge resolve, and only if the branch does not already track
   `kaola-workflow/<project>/` there (mirroring the original guard,
   `!fs.existsSync(mainProjDir)`, evaluated post-checkout so branch-tracked content always wins).
   An earlier draft landed the copy pre-checkout and regressed a pre-existing end-to-end test: when
   the branch itself commits `kaola-workflow/<project>/`, an untracked pre-checkout copy at that
   exact path collides with `git checkout` ("untracked working tree files would be overwritten").
   Staging first and landing only when the path is still absent post-checkout fixed it while still
   preserving genuinely worktree-only (untracked) content, e.g. a `.cache/` crash-resume journal.
5. **Stamp an additive `published_head` at the closure gate, never touching `branch_head` (#631).**
   Once the live feature-branch tip resolves as genuinely published (`git merge-base --is-ancestor`
   into the sink target verified, immediately before the `closure` step closes any issue), the
   receipt gains a NEW field: `receipt.published_head = implRef` (the fresh, post-rebase tip).
   `branch_head` is never mutated — it is written only at its two pre-existing receipt-init sites,
   unchanged, preserving its #518 cycle-identity role exactly. `cmdVerifySink` now resolves the
   implementation ref via `r.published_head || r.branch_head`, preferring the fresh field and
   falling back to `branch_head` only for legacy receipts written before this field existed.
6. **Prove ancestry before any force-delete in `stale-worktree-cleanup` (#620).** A new, distinct
   helper, `removeBranchIfMerged(root, branch, defBranch)`, proves `git merge-base --is-ancestor
   <branch> <defBranch>` before any `-D`; unproven, it falls back to the SAFE `git branch -d` (git
   itself refuses to delete a genuinely unmerged branch); on that refusal it returns
   `{removed:false, mode:'skipped_unmerged', tip:<sha>}` — never destroying anything.
   `cmdStaleWorktreeCleanup`'s `buckets` gains a `skipped_unmerged: []` array so a survived branch
   is reported loudly, not silently dropped. `defBranch` is resolved once per `--execute` run (a
   local, offline-safe `git` read) and reused for every candidate. `removeBranch` (the pre-existing
   unconditional `-D` helper) is left untouched and remains reachable ONLY from `cmdRelease` — a
   user-consented discard/abandon that legitimately still needs unconditional force-delete. This is
   a deliberate, minimal-blast-radius design choice: the new helper is additive, not a rewrite of
   the existing one, and nothing else in any of the four editions calls the unconditional path.

   **R4 doc-clarity note (adversarial-verifier finding, non-blocking).** A branch that is *pushed
   and at parity with its own upstream* is deleted by the SAFE `-d` leg, not preserved as
   `skipped_unmerged` — git's own semantics consider a branch "merged" once it is an ancestor of
   its configured upstream, regardless of whether it is also an ancestor of the LOCAL `defBranch`.
   Such a branch lands in `deleted_branch`. This is **not data loss**: the tip stays reachable via
   `refs/remotes/origin/<branch>` and on the remote itself (verified live in the adversarial pass —
   `git branch -d` succeeded, and the tip remained resolvable from both the remote-tracking ref and
   the remote). It is worth stating explicitly, though, because an operator reading only the
   `deleted_branch`/`skipped_unmerged` buckets could otherwise assume every `deleted_branch` entry
   was ancestor-proven into `defBranch` specifically — `-d`'s guarantee is narrower ("no unpushed,
   unmerged-anywhere work is destroyed"), not "every deletion was proven merged into `defBranch`".

All six fixes are reimplemented in each of the four editions (canonical `scripts/`, the Codex
byte-twin under `plugins/kaola-workflow/scripts/`, and hand-ports for GitLab/Gitea in their
`glab`/`tea` forge-CLI bodies). The pure-`git` helpers (`removeBranchIfMerged`, the merge-step
stage-then-land sequence, the `push_upstream` parity check) need no forge-specific body and are
carried verbatim into both hand-ports; only the forge-call bodies (`ghExec`/`forge.closeIssue`
etc.) differ by edition.

## Consequences

- A close can no longer be reported successful (`'closed'`/`'already_closed'`) when the forge still
  shows the issue open, at any of the four close sites across all four editions.
- A genuinely failed or unverified close on the legacy direct-merge path now fails the whole sink
  CLOSED (exit `1`, typed `sink_incomplete`/`step:'closure'`) instead of silently reporting
  `status:'merged'` exit `0` with only a stderr warning — closing the #168 fail-open gap that #497
  had already closed for the `--sink` transaction but not for the legacy path.
- A `push_upstream` failure in the `--sink` transaction can no longer be silently swallowed; the
  step is left not-done so a re-run retries it, and the branch is never falsely attested as backed
  up on the remote.
- The dead `worktree_sync` step is gone; worktree-only untracked content (e.g. a `.cache/`
  crash-resume journal) still survives into `mainRoot` when the branch does not already track the
  same path, now via a mechanism that actually runs (the prior step never matched).
- A cleanly rebased, genuinely published sink no longer false-alarms `impl_commit_not_ancestor` in
  `cmdVerifySink`; `branch_head`'s #518 cycle-identity role is unchanged and untouched by
  `published_head`.
- `stale-worktree-cleanup --execute` can never again force-delete committed-but-unmerged work; the
  only remaining unconditional-force-delete path in any of the four editions is `cmdRelease`, a
  user-consented discard.
- Validated with hermetic RED→GREEN tests reproducing every fixed defect live (not asserted from
  memory) in `scripts/test-bundle-finalize.js`, `scripts/test-claim-hardening.js`,
  `scripts/simulate-workflow-walkthrough.js`, and both forge-side suites
  (`test-gitlab-sinks.js`/`test-gitea-sinks.js`), plus a from-scratch adversarial falsification
  pass (5 independent reproduction drivers, 68/68 checks, could not find an input that destroys
  unmerged work, false-reports a failed close, or false-alarms a genuinely published rebased sink).
  The pre-existing `test-{run,gitlab-run,gitea-run}-chains.js` signal/receipt-timing flake (#635) is
  unrelated (zero references to either changed file; reproduces identically on the untouched tree)
  and is carried as a waiver, not treated as a regression from this bundle.

## Non-goals

- **No change to `removeBranch`'s unconditional `-D` semantics or its one remaining call site
  (`cmdRelease`).** A user-consented discard/abandon still force-deletes; only the
  automatically-triggered `stale-worktree-cleanup` path gained the ancestry proof.
- **No change to the #518 `branch_head` cycle-identity binding or its stale-receipt detection
  logic.** `published_head` is purely additive; no existing enum value or field semantics changed.
- **No retry/backoff policy for a failed `push_upstream` or a failed close beyond the existing
  manual re-run-after-fix path.** This decision only corrects false-success reporting; it does not
  add automatic retry.
- **No change to the `remote-closed-after-publish` gate or `SINK_STEPS`'s `closure`-last ordering
  from #617/D-617-01.** This decision extends the SAME fail-closed direction to the legacy path and
  to `push_upstream`/close-verification, but does not revisit #617's own fix.

## Alternatives considered

- **For #619, reuse the existing memoized `probeIssueState` for the post-close re-check in
  `claim.js` instead of adding `probeIssueClosedLive`.** Rejected: the memo is primed by the
  pre-close probe at the top of the same function with the `'open'` verdict; reusing it
  post-close would always replay that stale state, turning every genuine successful close into a
  false `'failed'` — a regression, not a fix.
- **For #620, guard the branch-deletion call site itself (make `removeBranch` conditionally refuse)
  rather than add a new, distinct `removeBranchIfMerged` helper.** Rejected: `removeBranch`'s
  unconditional force-delete is a legitimate, deliberate behavior for `cmdRelease`'s
  user-consented discard; changing its semantics in place would either weaken that legitimate path
  or require a flag threaded through every call site. A new, narrowly-scoped helper used only by
  `cmdStaleWorktreeCleanup` is the minimal-blast-radius fix and mirrors `sink-merge.js`'s own
  post-merge branch-teardown ancestry gate.
- **For #631, overwrite `branch_head` at the closure gate instead of adding a sibling field.**
  Rejected: `branch_head`'s entire purpose (#518/D-518-01) is to bind receipt identity to the
  branch tip AT INIT TIME, before any rebase; mutating it later would silently break the stale-cycle
  detection the field exists for. An additive, differently-named field with its own, narrower
  purpose (resolving the true implementation commit for `verify-sink`) avoids that collision
  entirely.
