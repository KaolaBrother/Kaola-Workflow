# D-509-01. Scope --verdict-check for the adversarial-verifier by post-dominance

Date: 2026-06-17
Status: **Accepted** — shipped in the bundle-501-509 build run.
Issue: #509
Related: #486 (investigation shape design: probe→assume→adversarial-critique→converge; recommended
         majority-refute fanout), #400 (six-surface prose contract), #341 (forge-neutral prose),
         #227 (adaptive path; adversarial-verifier role), #231 (script-enforced barrier),
         `scripts/kaola-workflow-plan-validator.js` (`verifyVerdictBlock`, `advVerifierIsChangeGate`),
         `agents/adversarial-verifier.md`, `commands/kaola-workflow-finalize.md`,
         `plugins/kaola-workflow/skills/kaola-workflow-finalize/SKILL.md`.

## Context

The adaptive path ships two distinct uses of the `adversarial-verifier` role:

1. **Change-gate** — post-dominates a code-producing (`tdd-guide`, `implementer`,
   `build-error-resolver`) or sensitive node. The gate purpose is to catch a false GREEN
   before the plan closes. A refuted verdict here means the implementation claim is
   unsound; the plan must not finalize.

2. **Investigation** — post-dominates NO code-producing or sensitive node. The role is a
   read-only skeptic: it probes a hypothesis, refutes a planner conclusion, or runs the
   #486-recommended majority-refute fan-out (`assume→{crit1,crit2,crit3}→converge`) as
   part of an investigation DAG. Refutation is analytical OUTPUT — it routes the claim
   into a bounded repair loop or surfaces a RISKY escalation — but the investigation
   produces no new code or sensitive writes to gate.

The pre-#509 `--verdict-check` did not distinguish these shapes. An investigation
adversarial-verifier emitting `verdict: refuted` (or `findings_blocking: 1`) would false-block
the finalize gate even though it dominated no code/sensitive node. This contradicted the
"never a gate" prose in `agents/adversarial-verifier.md`, line 74 (pre-fix) and created a
practical obstacle to using the #486-recommended investigation pattern in real plans — the
recommended shape is the read-only majority-refute FANOUT, and all three fanout instances
would have required `verdict: pass` to finalize.

## Options considered

**Option A — Scope the gate by post-dominance (chosen).**
Add a pure graph helper (`advVerifierIsChangeGate`) that returns `true` iff the
adversarial-verifier node lies on a path from some code-producing or sensitive node to the
unique sink. In `verifyVerdictBlock`'s `checkOne` loop, when `node.role ===
'adversarial-verifier'` AND `!advVerifierIsChangeGate(...)`, return `{ ok: true, exempt:
'investigation_adversarial_verifier' }` regardless of shape. The gate stays strong for
change-gate adversarial-verifiers. Prose is updated on 4 of the 6 `#400` surfaces (the 2
that carry the `--verdict-check` description plus the agent `.md` and its 3 toml mirrors).

Cost: one graph helper (~15 lines), two exemption paths in `checkOne`, tests pinning both
directions (a: investigation exempt; b: change-gate still blocks), and prose on 4 surfaces.

**Option B — Introduce a new `verdict: complete` vocabulary for investigation nodes.**
Add a third machine-readable verdict token that `--verdict-check` accepts unconditionally
for adversarial-verifier nodes, letting them signal "investigation done" without triggering
the pass/fail gate. Cost: new vocab propagated across 6 prose surfaces (#400 full set) +
schema (`parseNodeVerdict`) + validator changes across 4 editions. Heavier than needed.

**Option C — Document-only; accept the contradiction.**
Leave the code unchanged and add a prose note that investigation adversarial-verifiers
should emit `verdict: pass` regardless of their prose verdict. Rejected: this does NOT
remove the false finalize-block for a genuine REFUTED investigation — the gate reads
machine state, not prose. A plan using the recommended majority-refute investigation fanout
with all three instances emitting `verdict: refuted` (the correct signal) would still block.
Documentation without a code change is insufficient.

## Decision

**Option A** — scope by post-dominance. It is the cheapest-sufficient mechanism: one helper
function, tests in both directions (the (b) test proves the fix did NOT weaken the gate for
genuine change-gate adversarial-verifiers), and surgical prose on 4 surfaces.

## Key properties of the chosen design

**Post-dominance is the sole exemption key.** The exemption fires when the adversarial-verifier
node does NOT lie on a path from any code-producing or sensitive node to the unique sink.
This covers BOTH sequence and fanout investigation shapes — deliberately generalizing the
issue's "non-fanout" wording. The #486-recommended investigation shape is the read-only
majority-refute FANOUT (`plan-validator.js:688-707`); a "non-fanout-only" exemption would
have been a half-fix, leaving the recommended shape false-blocking.

**The gate stays strong for change-gate adversarial-verifiers.** An adversarial-verifier
that post-dominates a code-producing or sensitive node keeps both the fanout majority-refute
branch and the sequence pass/fail branch of `verifyVerdictBlock` active and fail-closed. The
`code-reviewer` (G1) and `security-reviewer` (G2) post-dominance requirements are unchanged.

**The "never a gate" assertion is now consistent, not contradictory.** An investigation
adversarial-verifier genuinely is never a finalize gate — its REFUTED verdict routes work
via the repair loop, not via `--verdict-check` blocking. The prose update is reconciliation,
not correction of the code's behavior.

**Prose propagation.** The `kaola-workflow-adapt` command and its 3 SKILL pack mirrors (the
other 3 of the 6 `#400` surfaces) describe the investigation shape and say "never a gate" —
Option A makes that assertion TRUE, so those surfaces are already consistent and need no
change. The 2 finalize surfaces that carry a `--verdict-check` description (one Claude
command + one Codex SKILL; the gitlab/gitea finalize SKILLs carry no adaptive section due
to pre-existing #294 drift) receive an explicit exemption note.

## Consequences

- Investigation adversarial-verifiers in both sequence and fanout shapes may emit
  `verdict: refuted` or `findings_blocking: 1` without blocking finalization.
- Change-gate adversarial-verifiers are unchanged: a refuted verdict still blocks.
- The #486 majority-refute fanout investigation pattern is now correctly exempt.
- Prose is consistent across all surfaces: "never a gate" holds for investigation
  adversarial-verifiers; "keeps full verdict-check coverage" holds for change-gate ones.
- All four walkthrough chains confirmed green with both exemption and gate-stays-strong
  tests pinned (n1-fix evidence: exit 0 / "investigation_adversarial_verifier" token,
  plus test (b) still blocks on change-gate refute).
