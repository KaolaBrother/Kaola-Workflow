# D-475-01. Consumer (non-npm) repos gate finalize on the agent's `final-validation.md`, not a re-executed chain receipt (issue #475)

**Date:** 2026-06-15
**Status:** Accepted
**Issue:** #475 (split from #464)
**Related:** #464 (the v6.2.0 `kaola-workflow/chains.json` consumer escape hatch this record RETIRES as
            the consumer contract), #432 (the chain-receipt machine gate this record scopes to the
            self-host), #44 ("Agent Owns Reasoning; Scripts Own Atomicity" — the founding principle),
            #424 (the attribution sweep, reused as the allowband-aware freshness check), #307 (the
            four-chain cross-edition gate), #400 (the finalize/contractor prose ×6-surface rule).

## Problem

#464 made the finalize chain-receipt gate work in a non-npm product repo (e.g. a Swift/Xcode app,
the KaolaTerminal issue-85 case) by adding a repo-local `kaola-workflow/chains.json` resolver to
`kaola-workflow-run-chains.js`: the consumer declared its build/test commands, the producer ran them,
and the receipt gated finalize like the self-host's npm chains. But that still required a **script to
re-execute** the validation suite the `final-validation` node had already run — redundant work that
contradicts the project's founding principle (#44: the agent owns verification; scripts own atomicity).
The over-engineering surfaced during the painful KaolaTerminal issue-85 finalization.

## Decision — Pure option A (owner-decided)

**D1 — The consumer finalize gate IS the agent's recorded evidence.** In a repo whose `package.json`
declares no `test:kaola-workflow:*` script, `plan-validator --finalize-check` gates on
`.cache/final-validation.md`: present + a column-0 `verdict: pass` (reusing the #251 `parseNodeVerdict`
vocabulary). Fail-closed, zero mutation: `final_validation_unverified` (absent/empty) >
`final_validation_failed` (no `verdict: pass`). No script re-runs the suite.

**D2 — The repo kind is auto-detected; the self-host gate is byte-unchanged.** `--finalize-check`
reads `package.json` and classifies the repo self-host iff it declares an edition chain script
`test:kaola-workflow:<claude|codex|gitlab|gitea>` ⇒ the existing #432 chain-receipt gate
(`chains_unverified` > `chains_stale` > `chains_red`), unchanged. Otherwise ⇒ consumer ⇒ the D1 gate.
The discriminator uses the **EXACT** predicate `run-chains.js resolveChains` uses (the four KNOWN_CHAINS,
not a loose `test:kaola-workflow:` prefix) so the producer and the gate can never disagree about the
repo kind (a divergent predicate could deadlock a repo with a non-standard `test:kaola-workflow:<x>`
script). It reads `package.json` at the **git top-level** (`git rev-parse --show-toplevel`), not
`findRepoRoot`'s `root` — an intermediate `agents/` directory can make `findRepoRoot` stop early (e.g.
at `kaola-workflow/`) and misclassify a real self-host as a consumer, which would skip the chain-receipt
gate (a fail-OPEN). The git top-level is unambiguous (with a `root` fallback if git cannot resolve it).

**D3 — No `validated_head` equality probe; the attribution sweep is a PARTIAL freshness check, and the
rest is the explicit #44 trust tradeoff.** A naive "agent-validated HEAD == current HEAD" check would
FALSE-REFUSE every correct finalize: per #324 AC3 the finalize node records `final-validation.md`
*before* it commits its own docs/CHANGELOG (allowband) edits, so HEADs legitimately differ at gate time.
The existing attribution sweep (part B of `--finalize-check`, run for BOTH modes) catches the
**unattributed** subset of post-validation churn: a change with no attributing `complete` node surfaces
as `unattributed_change`. It does **NOT** catch **in-write-set churn** — a re-edit of a `complete`
node's OWN declared file after `final-validation.md` was recorded passes the sweep (it is attributed)
and the consumer gate (the verdict still reads `pass`), so a regression inside an already-attributed
file can reach a green consumer finalize. The self-host #432 gate refuses the identical case via
`chains_stale` (headSha mismatch); the consumer gate deliberately does not, because adding a staleness
probe would re-introduce the script-owned re-verification this issue removes. This residual is the
**#44 trust tradeoff** (the agent owns verification), recorded in full below — the sweep is a partial
backstop, not a per-mode staleness guarantee.

**D4 — Presence + the agent's own verdict, NOT script re-verification.** Per #44 the gate trusts the
agent's recorded `verdict` (the honest-failure guard) and its presence; it does not re-execute or
re-judge. This is the deliberate, recorded tradeoff (below).

**D5 — `chains.json` is retired as the consumer contract (no middle-ground).** The `resolveChains`
chains.json branch in `run-chains.js` is deleted; the producer is now self-host-only (npm-default →
`chains_config_missing`). A consumer never runs `run-chains.js`. The contractor Step 8c prose is taught
that the `chains_config_missing` refusal is EXPECTED in a consumer repo (skip the receipt, confirm
`final-validation.md`).

## Known tradeoff (deliberate, recorded)

Pure A is, on its face, **less mechanically strict** than the #432 receipt:
- It trusts agent-recorded evidence (presence + the agent's own `verdict`) — exactly what #432's
  machine receipts exist to backstop (an agent once attested "4 chains green" over a red chain). A
  `verdict: pass` token only guards the *honest-recorded-failure* case, not mis-attestation.
- `final-validation` may run a **focused** test set, not the full suite an `xcodebuild` chain would —
  so a regression outside the focused set ships green.
- **In-write-set churn after validation is not detected (D3).** If a `complete` node's own declared file
  is re-edited *after* `final-validation.md` was recorded, the attribution sweep still passes (the change
  is attributed) and the verdict still reads `pass`, so the regression reaches a green consumer finalize.
  The self-host #432 gate refuses this via `chains_stale`; the consumer gate does not, because a staleness
  probe would re-introduce script-owned re-verification. The mitigation is procedural: record
  `final-validation.md` LAST (after the final code commit), as the finalize flow already sequences it.

This is accepted as the #44 trade: in a consumer product repo the agent owns verification. The
self-host (4-edition) keeps its machine-verifiable receipt unchanged, where the cost of running all four
chains is borne by the project that authored them.

## Consequences

- The KaolaTerminal-class consumer repo finalizes on the evidence its agent already produced; it no
  longer needs to author `chains.json` or run a second mechanical gate.
- The change is **prose-primary**: the load-bearing surfaces are the finalize/contractor prose (the
  canonical command, the three Codex finalize SKILLs, `contractor.md` + three `contractor.toml`) plus
  the `--finalize-check` discriminator. The dual-mode prose is pinned by every contract validator that
  pins the finalize gate (canonical + codex + gitlab + gitea), closing the prior zero-pins drift gap.
- The four-chain (#307) gate stays green: the plan-validator + run-chains changes propagate ×4
  (codex byte-twin + rename-normalized forge ports), and a RED-first walkthrough proves the consumer
  fail-closed path (absent → `final_validation_unverified`; no-pass → `final_validation_failed`;
  present+pass+clean → pass; orphan code change → `unattributed_change`).
