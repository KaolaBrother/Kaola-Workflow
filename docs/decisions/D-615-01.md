# D-615-01. Gate lane-group formation on parent cleanliness to avoid an unsatisfiable last-member close

Date: 2026-07-05
Status: Accepted
Issue: #615
Related: #437 / D-419-01 Part 2 (the lane-group co-open + group-scoped close barrier this decision's
         formation gate sits inside), #542 / D-542-01 (default-on disjoint-write co-open — the
         mechanism whose last-member close hits the deadlock), #596 / D-596-01 (the speculative-write
         co-open path — the second formation site this decision also gates)

## Context

A mixed-shape plan — one or more SERIAL write nodes followed by a later frontier of ≥2 disjoint
write nodes eligible for lane-group co-open — could reach a state where the group's last-member
close was structurally unsatisfiable.

Serial write nodes never commit: under the finalize-owned-commit contract, a serial node's
production writes land in the parent (integration) worktree and stay uncommitted until
Finalization. When a later write frontier forms a lane group over that same parent worktree, its
members open in isolated legs, but the group's last-member close still has to reconcile against the
parent: it runs the parent-clean fence (`--parent-clean-check`) before the commit-based group
barrier that diffs a shared group baseline against the union of all members' declared write sets.

That produced two structurally contradictory requirements on the same uncommitted serial file:

- **Horn A.** The parent-clean fence refuses the close outright (`parent_dirty`) as long as the
  serial node's file sits uncommitted in the parent worktree.
- **Horn B.** Committing that file to satisfy the fence puts it into the merge commit the
  commit-based group barrier diffs against — but that file was never in the lane group's own
  declared write-set union, so the barrier now refuses `write_set_overflow`.

Clearing one horn creates the other. Nothing in the existing scheduling logic recognized this
combination of shapes (serial writes upstream, lane-group co-open downstream, over the same shared
parent worktree) as unsafe before the group ever opened — the deadlock was only visible at the
group's last-member close, by which point the group could not be closed cleanly at all.

## Decision

**Prevent the mixture at scheduling time; do not touch either fence, and do not relax the serial
finalize-owned-commit contract.**

A new helper, `parentCarriesProductionDirt(planPath, project, shell)` in
`kaola-workflow-adaptive-node.js`, shells the exact same `--parent-clean-check` invocation the
last-member close already runs. Reusing the identical fence (rather than authoring a second,
parallel dirt check) means the producer/consumer classification of "is the parent clean" can never
drift between formation time and close time. The helper fails closed: a `parent_dirty` result, an
unrelated refuse (e.g. `root_mismatch`), or a crashed/non-JSON subprocess are all treated as dirt —
none of them is trusted to mean "safe to co-open."

This precondition now gates lane-group formation at both sites in `runOpenReady` that can form one:

1. **The normal co-open path** (`liveNodes.length === 0 && writeNodes.length > 0`). On dirt, this
   degrades to opening a single serial write instead of forming a group — reusing the existing
   serial-degrade fallback that already exists for a genuine write-set overlap, rather than adding a
   new code path. The serial per-node barrier tolerates prior dirt by construction (its baseline is
   a full-worktree snapshot taken at open, so accumulated dirt sits on both sides of the diff and is
   invisible to it), so degrading to serial is always safe here.
2. **The speculative-write path** (the `openingSpeculative` branch). A speculative write always
   opens with a provisioned leg — even a lone candidate forms a size-1 lane group — so it runs the
   identical commit-based group barrier and parent-clean fence at close and is exposed to the same
   two-horned deadlock. This site was initially missed by the fix and caught in a subsequent
   code-review pass; on dirt, it now excludes all write candidates from the speculative open via
   `speculativeWriteExcluded: { reason: 'parent_dirty', nodeIds }`, mirroring the existing
   `no_leg_capability` exclusion shape. The write is not lost — it simply waits for its gate to
   resolve normally, exactly like the non-speculative serial-degrade.

The fence subprocess is only spawned when a group could actually form (legs are capable and a
qualifying write frontier exists) — evaluated last in each site's boolean guard so a frontier that
was never going to form a group never pays the extra subprocess cost.

This direction was selected over two alternatives (see below) because it is accuracy-first (fails
closed on any uncertain fence result rather than guessing), preserves both existing contracts
unmodified (the parent-clean fence and the "serial nodes never commit; commits are finalize-owned"
rule), and is the cheapest sufficient mechanism — it reuses the fence that already exists and the
serial-degrade / exclusion shapes that already exist, adding one new precondition function and no
new subcommand, schema field, or state.

## Consequences

- Both known lane-group formation sites are gated; a mixed serial-then-lane-group plan shape now
  completes correctly via a serial degrade or a deferred speculative write instead of reaching an
  unsatisfiable last-member close.
- A pure-parallel or group-first plan carries no prior production dirt at the point a group would
  form, so the fence passes and the group forms exactly as before — no currently-safe parallelism is
  lost; only the genuinely-mixed shape is affected.
- Neither `--parent-clean-check` nor the commit-based group barrier changed — both fences, and the
  finalize-owned-commit contract for serial nodes, are exactly as they were before this decision.
- `kaola-workflow-plan-validator.js` was not modified; the fix is entirely scheduling-side
  (`kaola-workflow-adaptive-node.js`).
- Validated with hermetic RED→GREEN tests added to `scripts/test-adaptive-node.js` driving real
  `open-ready` subprocesses over real git repositories: one case for the normal co-open
  serial-degrade site, one for the speculative-write exclusion site. Regenerated into all forge
  ports via `edition-sync.js --write` and verified byte-consistent; all four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains, plus
  `node scripts/simulate-workflow-walkthrough.js` and `node scripts/test-adaptive-node.js`, are
  green. An adversarial-verifier pass independently reproduced the original deadlock and could not
  find a bypass, a third formation site, a TOCTOU race between the formation-time fence check and
  the close-time fence check, or a regression (confidence: high, not refuted).

## Non-goals

- **No change to `--parent-clean-check` or the commit-based group barrier.** Both fences keep their
  existing semantics; this decision only decides, earlier, whether it is safe to attempt a group
  formation that would later hit them.
- **No relaxation of the serial finalize-owned-commit contract.** A serial write node still never
  commits; this decision works around the resulting uncommitted dirt rather than forcing an
  out-of-band commit to clear it.
- **No new adaptive-node subcommand.** The fix is a new precondition function called from the
  existing `open-ready` formation sites, not a new lifecycle step.

## Alternatives considered

- **Teach both fences to treat already-`complete` nodes' declared write sets as attributed
  allowband**, so a serial sibling's uncommitted file would not read as unattributed dirt at the
  parent-clean fence or as overflow at the group barrier. Rejected: doing this without forcing a
  commit would require relaxing the parent-clean fence itself, reopening a floated-slip hole the
  fence exists to close, and the synthesizer's merge behavior against a dirty-but-disjoint tree is
  version/case-dependent and should not be assumed safe as a foundation for a correctness fence.
- **Re-anchor the group barrier's baseline at the parent's HEAD at group open**, on the theory that a
  later-arriving baseline would exclude the serial dirt from the diff. Rejected as a misdiagnosis:
  the baseline is already the parent HEAD at group open — the overflow does not come from a stale
  baseline, it comes from the merge commit carrying the serial node's committed work between that
  baseline and the merge commit. Re-anchoring the baseline cannot exclude that commit without
  breaking the leg-base ancestor invariants the group-barrier diff depends on.
