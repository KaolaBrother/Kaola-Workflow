# D-428-01. both roots must be cleaned to prevent stale roadmap source files

Date: 2026-06-12
Status: Accepted
Issue: #428
Related: `scripts/kaola-workflow-claim.js` (`reconcileRoadmapForClosure`, `archiveProjectDir`),
         `scripts/kaola-workflow-roadmap.js` (`regenerateRoadmap`),
         `docs/workflow-state-contract.md` § Generated Mirrors,
         `docs/api.md` § Closure Contract (closure receipt schema, `roadmap_removed_by_root`),
         D-426-01 (copy-then-verify-then-delete, the related #426 fix)

## Context

`reconcileRoadmapForClosure` is responsible for removing `.roadmap/issue-N.md` source files and
regenerating `ROADMAP.md` during finalization. Before issue #428, it operated only on the MAIN root
(`git rev-parse --show-toplevel`). For linked worktree runs, the active project folder is also
present in the WORKTREE root (the `.kw/worktrees/{project}/` checkout). The roadmap cleanup step
ran in the worktree (because `cmdFinalize` runs from the worktree cwd), but it only cleaned the
main root via the explicit `--show-toplevel` resolution.

**The consequence:** after finalization of a linked worktree run, `kaola-workflow/.roadmap/issue-N.md`
would be absent from the main root (correctly cleaned) but would REMAIN in the worktree checkout
(`<worktree>/.kw/worktrees/{project}/kaola-workflow/.roadmap/issue-N.md`). The worktree is not
immediately discarded after `cmdFinalize` — it persists until `sink-merge` removes it. A
`git add -A` in the worktree before the sink-merge commit would sweep the stale source file into
the implementation commit, causing it to reappear in the main branch even though `cmdFinalize`
correctly cleaned it from the main root.

This was a latent defect that only manifested when:
1. The run used a linked worktree (`run_posture: worktree`), AND
2. A `git add -A` or similar broad staging was used in the worktree between `cmdFinalize` and
   `sink-merge`.

The defect was not caught by the closure invariant checker because invariants 1 and 2 check the
MAIN root state post-merge, by which point the stale source would already be in the committed tree.

Two remediation postures were considered:

1. **Guard at sink-merge:** before staging, explicitly remove any stale `.roadmap/issue-N.md` from
   the worktree. This is a band-aid that does not fix the root cause and adds cleanup logic to
   sink-merge's already-complex flow.
2. **Dual-root cleanup in `reconcileRoadmapForClosure`:** extend the cleanup to iterate over BOTH
   roots (main + worktree when provisioned), removing and regenerating in each.

## Decision

**`reconcileRoadmapForClosure` iterates over BOTH roots.**

When `cmdFinalize` is called in a linked worktree context (i.e. a `worktree_path` is recorded in
the project's `workflow-state.md`), `reconcileRoadmapForClosure` resolves BOTH the main root
(via `git rev-parse --git-common-dir`, then `dirname`) and the worktree root (`worktree_path`), and
performs the `.roadmap/issue-N.md` removal + `regenerateRoadmap` call in each.

**Return value extended:**

`reconcileRoadmapForClosure` now returns:

```json
{
  "roadmap_removed_by_root": {
    "/path/to/main/root": ["issue-42.md", "issue-47.md"],
    "/path/to/worktree/root": ["issue-42.md", "issue-47.md"]
  },
  "roadmap_residue": ["/path/to/worktree/root/kaola-workflow/.roadmap/issue-53.md"]
}
```

- `roadmap_removed_by_root` — per-root list of `.roadmap/issue-*.md` files successfully removed.
- `roadmap_residue` — full paths to any `.roadmap/issue-*.md` sources that could not be removed
  (e.g. file did not exist, permission error). Should be empty on a clean close; non-empty signals
  a residue that the closure invariant checker will catch.

**New closure contract invariant: `roadmap-residue-clean`:**

A new closure invariant checks that `roadmap_residue` is empty after `reconcileRoadmapForClosure`
completes. A non-empty residue means a roadmap source survived finalization and is a violation
(same severity level as `roadmap-source-absent`). This catches both the worktree-residue case and
any future scenario where a root is inaccessible at cleanup time.

**Why dual-root beats guard-at-sink-merge:**

The guard-at-sink-merge approach adds complexity to a path that should be purely mechanical
(push, merge, close, clean branch) and is not the natural owner of roadmap state. Roadmap cleanup
is the domain of `archiveProjectDir`/`reconcileRoadmapForClosure`. Moving the fix there means the
worktree is already clean by the time sink-merge runs, regardless of how sink-merge stages its
commit. Defense in depth: the new `roadmap-residue-clean` invariant catches any failure in the
dual-root cleanup before it reaches the merge.

## Consequences

**For operators:**

- Finalization of a linked worktree run now cleans `.roadmap/issue-N.md` from BOTH the main root
  and the worktree root. The stale-source-in-worktree defect is fixed; `git add -A` in the worktree
  between `cmdFinalize` and `sink-merge` will not re-add the source to the commit.
- Single-root (in-place) runs are unchanged; `reconcileRoadmapForClosure` iterates over one root
  (the main root) as before.

**For the closure receipt schema:**

- `roadmap_removed` (per-root list) and `roadmap_residue` added to the `cmdFinalize` output.
  `roadmap_removed` supersedes the existing flat `roadmap_sources_removed` array (still present for
  backward compatibility on bundle receipts); `roadmap_residue` is new.

**For the closure invariants:**

- `roadmap-residue-clean` added as a new hard-gating invariant (violations set
  `closure_invariants.ok = false`).

**For the test suite:**

- Walkthrough scenarios added for the dual-root cleanup path: linked worktree run where both roots
  are cleaned, and for the `roadmap-residue-clean` invariant firing when a source survives.

**For the Generated Mirrors contract (`docs/workflow-state-contract.md`):**

- The single-owner finalize invariant note is updated to reflect that both roots are cleaned; the
  ordering (remove source → regenerate) applies to each root independently.
