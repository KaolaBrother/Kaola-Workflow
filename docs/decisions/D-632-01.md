# D-632-01. Close the second fail-open `chainReceiptGreenness` consumer with a `chains_empty` guard, and resolve `--cut` to stay informational-only

Date: 2026-07-07
Status: Accepted
Issue: #632 (primary — `chainReceiptGreenness` fail-open on empty/missing chains, plus the
       stale `--cut` comment); #635 (companion — `test-run-chains.js` signal-death flake,
       same bundle)
Related: #618/`plan-validator.js` `chains_empty` refusal (the original fix this decision
         mirrors — closed the plan-validator's `--finalize-check` gate only, leaving this
         second consumer open); #501 (CI/CD is not a gate — orthogonal; this decision concerns
         the workflow's own chain-receipt greenness reporting inside `release.js`, not an
         external pipeline)

## Context

`chainReceiptGreenness` (`scripts/kaola-workflow-release.js:249`, plus its byte-identical
codex twin and the gitlab/gitea forge ports) computes whether the recorded four-chain test
receipt (`.cache/chain-receipt.json`) is genuinely all-green. Before this fix it fell
**fail-open** whenever `receipt.chains` was an empty array or a missing/non-array key: the
red-chain loop body simply never ran, so "zero chains were verified" read identically to
"every chain is green" (`{green: true}`). This was the **second** consumer of this exact
fail-open shape — `#618` had already fixed the analogous gap in the plan-validator's
`--finalize-check` gate (`plan-validator.js:~2818`, `chains_unverified > chains_stale >
chains_empty > chains_red`), but `chainReceiptGreenness` was never brought into line with
that precedent.

Separately, the `runVerify` comment near `release.js:~364` (pre-fix) asserted that chain
greenness "blocks `--cut`" — factually wrong: `runCut` (`release.js:~390`) never reads
`greenness` at all (verified: zero `green` references in its body). The underlying issue
asked for an explicit call on whether `--cut` *should* gate on greenness, since the stale
comment implied it once did (or was meant to).

## Decision

**Close the fail-open with a `chains_empty` guard mirroring the #618 precedence exactly, and
resolve the `--cut` question to keep it informational-only — not escalate it.**

1. **`chains_empty` guard, same precedence slot as #618.** `chainReceiptGreenness` now returns
   `{ green: false, reason: 'chains_empty' }` when `receipt.chains` is not a non-empty array,
   inserted after the `chains_stale` HEAD-bound check and before the red-chain loop
   (`release.js:265-266`). The precedence is now identical across both consumers:
   `chains_unverified > chains_stale > chains_empty > chains_red`. The guard covers both an
   explicit empty array (`chains: []`) and a missing/non-array `chains` key; the old
   `if (Array.isArray(receipt.chains))` wrapper around the red-chain loop was removed since
   the new guard already guarantees a non-empty array by that point.
2. **`--cut` stays informational-only — resolved, not escalated.** `runCut` never reads chain
   greenness and this decision does not add such a read. The reason is structural, not a
   preference: the documented release sequencing runs the **offline** pre-cut check *before*
   the **online** `npm test` run that produces the green receipt, so at the moment `--cut`
   executes, the receipt that would gate it does not yet exist. Gating `--cut` on greenness
   would break the documented flow, not merely add a check. Greenness continues to surface as
   an informational `chain_warning` field only at `--verify` time (`release.js:373-378`),
   where a fresh receipt is expected to already exist. This changes no user-owned contract
   (behavior is unchanged; only the fail-open bug and the stale comment are fixed), so no
   consent-halt was warranted for this call. The stale line-364 comment was corrected to state
   this reality and cite this record, so a future reader does not re-litigate "why doesn't
   `--cut` block on a red chain?"
3. **Ported atomically across all four editions.** The guard and comment fix landed together in
   the canonical `scripts/kaola-workflow-release.js`, its byte-identical codex twin
   (`plugins/kaola-workflow/scripts/kaola-workflow-release.js`), and the rename-normalized
   gitlab/gitea forge ports (`plugins/kaola-workflow-gitlab/scripts/kaola-gitlab-workflow-release.js`,
   `plugins/kaola-workflow-gitea/scripts/kaola-gitea-workflow-release.js`) — all four carried
   the identical bug at the same lines, and the fix carries no CLI-brand tokens, so it is
   forge-neutral in all four.

Validated with a hermetic RED-first test in `scripts/test-release.js` (T14a/T14b) that
reproduces the pre-fix fail-open exactly (`{chains: []}` and a receipt with no `chains` key
both returned `{green: true}` before the fix; `{green: false, reason: 'chains_empty'}` after),
plus precedence/regression guards (T14c: stale+empty still reports `chains_stale`, not
`chains_empty`; T14d: a genuine all-green non-empty receipt is unaffected). Reviewed
(`code-reviewer`) and adversarially probed across 19 zero-verified-chains shapes — empty
array, missing key, wrong-typed `chains` (string/object/null/number), stale+empty,
stale+missing — every one refuses `chains_empty` (or the higher-precedence `chains_stale`);
no shape was found that reports a false `green: true`. All four `npm run
test:kaola-workflow:{claude,codex,gitlab,gitea}` chains ran green, unwaived.

## Consequences

- A chain receipt with zero verified chains (empty array or missing key, in any of the four
  editions) can no longer be reported as `green: true`; it now reports
  `{green: false, reason: 'chains_empty'}`, surfaced as a `chain_warning` at `--verify`.
- `--cut` remains informational-only with respect to chain greenness, matching its actual
  (and only sequencing-compatible) behavior; the comment now states this truthfully instead of
  the prior factually-wrong "blocks `--cut`" claim.
- Future readers of `release.js` have a durable citation (this record) for why `--cut` does
  not gate on chain greenness, avoiding re-litigating the question the underlying issue raised.

## Non-goals

- **No change to `--cut`'s actual behavior.** This decision documents and corrects a stale
  comment; it does not add a new gate, flag, or read of `greenness` inside `runCut`.
- **No change to the `chains_red` or `chains_stale` precedence tiers.** Only the missing
  `chains_empty` tier was added, in the same slot #618 established for the plan-validator.
- **No change to `kaola-workflow-run-chains.js`'s own receipt-writing behavior** (it already
  refuses to write an empty `chains[]` receipt — see the plan-validator's `no_chains`
  operator hint); this decision only closes the read-side consumer that had not caught up to
  that producer-side guarantee.

## Alternatives considered

- **Escalate the `--cut` gating question via a consent-halt.** Rejected: the sequencing
  constraint (offline pre-cut check runs before the receipt-producing online test) makes
  gating `--cut` on greenness structurally impossible without reordering the documented
  release flow itself — not a genuine value judgment between two workable options, so no
  consent-halt was warranted. Resolving it directly and recording the reasoning here was
  sufficient.
- **Add a `chains_empty`-specific message distinct from #618's wording.** Rejected: reusing
  the exact precedence order and intent of the established #618 precedent keeps the two
  consumers (plan-validator and `release.js`) in lockstep, so a future reader who understands
  one understands the other.

## Companion fix in the same bundle: #635 — deterministic signal-death delivery in `test-run-chains.js`

Not a design decision in its own right (a test-harness reliability fix, not a behavior change),
but recorded here briefly since this is the bundle's only ADR.

`scripts/test-run-chains.js`'s T26/T27/T28 signal-death assertions wrote a self-SIGKILL mock
and raced a real `process.kill(pid, 'SIGKILL')` against the runner's own per-chain timer; under
system load the child was sometimes reaped as a timeout/SIGTERM first, flipping which
`signal === 'SIGKILL'` / `timed_out === false` assertions failed on a given run. This forced an
`--accept-known-red …:635` waiver on every bundle finalized in this session up to and including
the previous one. The fix replaces the race with a deterministic in-process seam: a reserved
sentinel command (`__kaola_test_signal_death__:<signal>`) is intercepted by a patched
`spawnSync`/`spawn` and answered with a canned signal-death result synchronously (or on the next
tick, strictly before the runner's timer can be due) — no subprocess, no race, for T26/T27/T28.
One real end-to-end self-kill subprocess case (`T26b`) is kept, asserting only the
`exitCode === 1` class rather than the exact signal name, since the untouched
`kaola-workflow-run-chains.js` signal-to-exit-code mapping (the correct #618 fix) guarantees
that class outcome regardless of which real signal wins. The scope guard held throughout:
`kaola-workflow-run-chains.js` itself was not modified. Adversarially verified load-insensitive
across 22 runs (19 under heavy induced CPU load that reliably flaked the pre-fix harness 4/6),
all reporting the identical assertion count with zero failures. This bundle's own finalize
four-chain receipt is expected to go genuinely green with no `:635` waiver as a direct
consequence.
