# D-643-01. Complete the per-role evidence-recording contract: registry rows, agent-file evidence sections, and a future-agent wall

Date: 2026-07-08
Status: Accepted
Issue: #643
Related: #433/D-433-01 (`ROLE_TOKEN_REGISTRY`'s original introduction — this decision extends its
         vocabulary, never its shape); #392 (`checkEvidenceShape`, the close gate this decision
         generalizes); #642/D-642-01 (the sibling channel decision landed in the same run; its
         consumed-proof carve-out is deliberately kept consistent with this decision's shape gate);
         #634/D-634-01 (the `metric-optimizer` registry row and evidence contract this decision's
         wall now also protects going forward); #625 (a prior manual fix of six agent profiles'
         evidence-persistence direction that this decision's future-agent wall now makes structural

## Context

The evidence-persistence contract was advertised as "one contract — no per-agent guesswork," but a
roster audit (comparing `agents/*.md` against `CANONICAL_ROLES`/`ROLE_TOKEN_REGISTRY` in
`plan-validator.js`, and against the plan-run routing surfaces' role-kind enumerations) found
roughly half of the fourteen in-plan node roles had no registry row, or a row that degenerated to
`['evidence-binding']` (presence-only). Concretely: `build-error-resolver` (a WRITE/IMPLEMENT role)
had no row and was missing from the routing surfaces' WRITE-role list; `code-architect` and
`code-explorer` (read-producer roles) had no row and their vendored profiles were never adapted to
carry any evidence-recording instruction at all. For every uncovered role, `checkEvidenceShape`
(the close gate) fell through to a bare non-empty check on whatever the role happened to write, so
a lossy one-line summary of a rich deliverable (a full architectural blueprint reduced to "designed
it") passed the close gate exactly as well as the real thing — the gate could not tell the
difference because it had nothing role-specific to check.

A second, structural gap sat behind the first: even after filling in the missing rows and prose for
today's roster, nothing would stop the *next* new node-role agent from shipping the same way — no
row, no evidence-recording section, silently degenerating to presence-only. The registry and the
agent-file prose are two independently-editable surfaces with no shared derivation; a maintainer
adding role N+1 has to remember both, by hand, forever.

## Decision

**Eight new `ROLE_TOKEN_REGISTRY` rows** (`plan-validator.js`), naming the content-bearing token(s)
each role's evidence must carry beyond `evidence-binding`:

```
'code-architect':       ['evidence-binding', 'files_to_create|files_to_modify', 'build_sequence'],
'code-explorer':        ['evidence-binding', 'findings'],
'knowledge-lookup':     ['evidence-binding', 'findings', 'sources'],
'planner':              ['evidence-binding', 'recommendation'],
'issue-scout':          ['evidence-binding', 'recommendation'],
'build-error-resolver': ['evidence-binding', 'build-green'],
'synthesizer':          ['evidence-binding', 'merge_outcome'],
'doc-updater':          ['evidence-binding', 'docs_updated'],
```

`files_to_create|files_to_modify` is an alternation (`|`): satisfied by any one alternative present,
mirroring the implementer's existing `regression-green|build-green|smoke-integration` vocabulary and
the seed's existing alternation handling (the seed writes the first alternative as the stub key).
Every one of the eight roles reaches at least two tokens — the future-agent wall's floor — with no
`PRESENCE_ONLY_RATIONALE` entry needed.

**`checkEvidenceShape` generalization.** The prior hardcoded per-role branches
(`main-session-gate`, `tdd-guide`, `implementer`, `metric-optimizer`) are untouched. The bare
"other roles: non-empty" else-arm is replaced by a registry-driven branch: for each token class in
`ROLE_TOKEN_REGISTRY[role]` beyond `evidence-binding` whose KEY is present in the evidence (see the
back-compat rule below), a single token requires a non-empty column-0 `<token>:` value; an
alternation requires any one alternative present. A role with no registry row, or with no keys
present at all, falls back to the pre-existing bare non-empty check. This closes the lossy-
paraphrase hole: a truncated `code-architect` evidence (e.g. `files_to_create:` present but empty)
now genuinely refuses `evidence_shape_failed` instead of passing on bare key presence.

**Back-compat (shared with D-642-01's DD-5).** Both new/generalized close gates key enforcement off
whether the relevant `<token>:` line is PRESENT in the evidence, not off any nonce or timestamp.
New-code opens guarantee presence — `seedEvidenceFile` writes the registry stubs at open time, and
`runRecordEvidence` re-injects any the agent dropped (as an empty line) for the generic-branch
registry tokens and for `upstream_read: <up-id>` keys on `IMPLEMENT_ROLES` consumers — deliberately
excluding `tdd-guide`/`implementer`/`metric-optimizer`/the verdict-gate roles from re-injection
(re-injecting an empty `RED:`/`GREEN:`/`verdict:` would falsely satisfy those roles' own presence
regexes). Old in-flight nodes recorded before this feature shipped have no such keys at all and
record-evidence will not re-run over them, so they stay exempt — no in-flight breakage, and no
fabrication hole (a truncated NEW-code evidence still gets an empty re-injected key, which still
fails the non-empty check).

**Evidence-contract prose, rendered per role-kind (read producers vs. write roles), into the eight
newly-covered role profiles.** Read-producer roles (`code-architect`, `code-explorer`,
`knowledge-lookup`, `planner`, `issue-scout`) get: RETURN the full structured deliverable as the
final message; the orchestrator persists it verbatim via `record-evidence --stdin`, which
re-injects the node's `evidence-binding:` header — never add, alter, or strip that header;
include every content-bearing token the role produces with a non-empty value (a lossy paraphrase is
refused at close). Write roles (`build-error-resolver` → `build-green`; `doc-updater` →
`docs_updated`) get: SELF-WRITE directly into the seeded `.cache/{node-id}.md`, reading and
preserving the seeded `evidence-binding:` header verbatim, appending content below it; same
non-empty-token requirement. `synthesizer` already carried the self-write needle and required no
edit. Rendered into all four agent-profile surfaces per role (root `.md` + three plugin `.toml`
editions, byte-identical ×3, forge-neutral — no forge-CLI or brand nouns).

**Future-agent wall (`validate-vendored-agents.js`'s `checkFutureAgentWall`).** Runs over every
node-role agent — the managed roster minus the two orchestration roles (`contractor` drives
claim/finalize bookkeeping; `workflow-planner` authors the plan; neither is a node-role evidence
producer) — derived from the `agents/` directory listing itself, never a hand-list, so a future
agent addition is automatically covered:

1. **Registry floor** — a `ROLE_TOKEN_REGISTRY` row with at least two tokens, or an explicit
   `PRESENCE_ONLY_RATIONALE[role]` one-line reason (the allowlist ships empty after this decision;
   every current node role reaches the floor without one).
2. **Role-kind evidence needle, kind DERIVED from the tool manifest.** The agent's own front-matter
   `tools:` line is parsed; `Write` or `Edit` present ⇒ write-kind (must carry `SELF-WRITE` +
   `evidence-binding`); absent ⇒ read-kind (must carry `RETURN` + `record-evidence`) — never a
   hand-maintained kind list, so the wall cannot silently drift from what the agent is actually
   capable of. A **fail-closed correction**: an agent file with no parsable `tools:` line at all
   previously defaulted to read-kind (the weaker needle) — a fail-open gap for a hypothetical
   future agent that omits its manifest entirely. It now refuses
   `agent_contract_manifest_missing` instead, surfaced as a non-blocking nit by this run's own
   `code-reviewer` and fixed in the same run (no live agent is affected — all sixteen current
   agents declare an explicit `tools:` line).

Mirrored `.toml` needle checks land in the codex/gitlab/gitea contract validators (the same
per-file `--forbidden-only` pattern those validators already use for forge-neutrality).

**×6 role-kind enumeration re-derivation (routing surfaces).** The two stale hardcoded lists
(READ-ONLY: `code-explorer, knowledge-lookup, adversarial-verifier, planner`; WRITE:
`implementer, tdd-guide, metric-optimizer` — both missing several actual roles) are replaced by the
same manifest-derived sentence the wall itself uses: "any node role without `Write` in its tool
manifest RETURNS its deliverable for orchestrator persistence; Write-manifest roles SELF-WRITE,"
listing the current roster only as EXAMPLES. `test-route-reachability.js` and the four
`validate-*-contracts.js` grep-assert no surface still carries either stale enumeration.

## Consequences

- A truncated or lossy-paraphrased evidence file from any of the fourteen node roles now genuinely
  refuses at close (`evidence_shape_failed`) instead of passing on bare `evidence-binding` presence.
- Every node-role agent profile now states its evidence-recording obligation explicitly and
  correctly for its own kind; the next agent added to the roster cannot ship without either the
  registry row or the evidence-contract section — the wall throws a typed refusal at the first
  missing half.
- `plan-validator.js` is a GENERATED aggregator; the canonical edit was regenerated to the codex
  byte mirror and both forge rename ports via `npm run sync:editions`. The agent `.toml` profiles are
  NOT edition-synced (hand-mirrored, byte-identity checked by `test-agent-profile-parity.js`); all
  three plugin trees were edited by hand and verified byte-identical.
- Tests: `test-adaptive-node.js` pins the new registry rows' seed/`required_tokens` behavior and the
  close-refusal on a truncated content-bearing token, for both the alternation and single-token
  forms, plus a legacy-exempt (old in-flight, no re-injected key) control; `validate-vendored-agents.js`
  itself is the future-agent wall's runtime enforcement, exercised by a scratch-dir fixture harness
  (never the live repo root) covering a missing-row refusal, a missing-section refusal on an
  otherwise-compliant agent, and the manifest-missing refusal. All four
  `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains verified green (run
  sequentially).

## Non-goals

- Does not add, remove, or rename any node role — this is a contract-completion pass over the
  existing roster (the 22-path agent-registration wall does not fire; no `CANONICAL_ROLES` /
  `config/agents.toml` / `install.sh` delta).
- Does not change `checkEvidenceShape`'s hardcoded branches for `main-session-gate`, `tdd-guide`,
  `implementer`, or `metric-optimizer` — only the else-arm serving every other role.
- Does not retroactively re-validate old in-flight nodes recorded before this feature shipped; the
  back-compat exemption is deliberate, not a residual gap.
