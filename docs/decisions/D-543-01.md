# D-543-01. Install-time `--with-fast`/`--with-full` partition ported to Codex (+ opencode); opencode `.claude/` path-leak closed

Date: 2026-06-20
Status: Accepted
Issue: #543
Related: #538 (the adaptive-default ADR whose `install.sh` partition this ports), #544 (the opencode
         edition this folds a discovered leak fix into), #530 (opencode edition is additive),
         D-538-01 (adaptive unconditional default; the `installed_paths` config field this writes),
         #227 (original adaptive path), #307 (four-chain cross-edition parity),
         #400 (six-surface routing-prose propagation).

## Context

D-538-01 made adaptive the unconditional default and converted `fast` / `full` into install-time
opt-ins (`--with-fast` / `--with-full`) recorded by UNION into the shared
`~/.config/kaola-workflow/config.json` `installed_paths` field. That partition landed on the Claude
Code `install.sh` only. The **runtime** legality gate — `kaola-workflow-claim.js:849-863` returning a
typed `path_not_installed` refusal, reading `installed_paths` via
`kaola-workflow-adaptive-schema.js` (byte-identical ×4) — was already shipped and enforced across
**all** editions. So on Codex (and opencode) a `KAOLA_PATH=fast` request on a default install
already refused correctly; the **only** gap was the install-time *writer* that records the opt-ins.
No installer on those editions ever wrote `installed_paths`, so the opt-in paths were permanently
unreachable from a default install on those runtimes.

Issue #543 closes that gap by porting the partition to Codex, and — as a folded #544-run discovery —
fixes an opencode edition path-leak that was surfaced while wiring the opencode side of the same
partition.

## Decision

### Part A — Codex install-time opt-in partition (Option A: installer config-writer)

The runtime legality gate already enforces across all editions; the only missing piece was the
install-time writer. The cheapest sufficient mechanism is therefore an **installer config-writer**:
add `--with-fast` / `--with-full` arg parsing plus a pure-JS `seedKaolaConfig` UNION writer to the
byte-identical `install-codex-agent-profiles.js` triplet (`plugins/kaola-workflow/`,
`-gitlab/`, `-gitea/`), byte-semantically mirroring `install.sh:712-734` step D4.

- **Arg parsing** (module top): `WITH_FAST` / `WITH_FULL` via position-robust
  `process.argv.includes(...)`; `--enable-adaptive` is accepted but warns + ignores (retired by
  #538); unknown args are ignored (never hard-fail — preflight + tests invoke positionally).
- **`seedKaolaConfig(homeDir, withFast, withFull)`** is a pure-JS port (no `python3` shell-out —
  node is guaranteed present on a Codex install, so this is strictly more robust than the bash
  mirrors). It reads `~/.config/kaola-workflow/config.json` (WARN-first on corrupt JSON / non-object,
  leaving the file untouched and never throwing), computes the effective set as a UNION of the
  existing `installed_paths` and the requested flags, and writes back in canonical order
  `['fast','full'].filter(p => paths.has(p))` — add-only, never removes, unknown tokens dropped.
  `parallel_mode` is setdefault-only (`'auto'`, never overwriting a user value); the retired
  `enable_adaptive` field is migrated away. The write is crash-safe (write-temp-then-rename,
  mirroring the `copyAgentProfiles` pattern).
- **Integration**: called in `main()` after `updateHooks()` and before `pruneStaleProfiles(...)`,
  mirroring `install-opencode.sh`'s `seed_kaola_config` ordering; `seedKaolaConfig` is exported for
  unit-testability.
- **NOT changed**: no `plugin.json` edit (the `"skills":"./skills/"` assertion is validator-pinned
  ×3); no skill/profile conditional-removal (the `kaola-workflow-fast` SKILL.md is
  validator-mandatory; the 15 agent profiles are role-based, not path-based, so there is no
  fast/full subset to conditionally deploy); no version bump (lands under the current 4.7.0 Codex
  manifest series — a version bump is a release-surface event).

### Part B — opencode standalone fix (folded #544-run discovery)

While wiring the opencode side of the same partition, two defects in the opencode edition were
surfaced:

1. **Claude Code filesystem path-leak.** The `kaola_script()` resolver shipped `$CLAUDE_PLUGIN_ROOT`
   + `$HOME/.claude/kaola-workflow/scripts` verbatim into every generated command and agent profile
   (146 matches across 12 command files + 2 agents), and `install-opencode.sh` deployed its support
   scripts into `~/.claude/`. A machine with no Claude Code installed got a phantom `~/.claude/`
   namespace and commands that could not resolve their scripts.

2. **No install-time partition.** opencode had only the generator-time "strip the path-selection
   step" behavior from #530; it had no `--with-fast` / `--with-full` parity with Claude/Codex.

Both are fixed:

- **Generator transform** (`scripts/sync-opencode-edition.js`): a new pure function
  `rewriteClaudeScriptPaths(text)` rewrites the `kaola_script()` resolver (and the agent-profile
  "Re-derive your own script paths" prose) to opencode-native paths —
  `${OPENCODE_CONFIG_DIR:-$HOME/.config/opencode}/kaola-workflow/scripts` plus `./scripts`,
  honoring `$OPENCODE_CONFIG_DIR`. It is wired into `transformCommandBody` (commands) as a final
  pass and into `renderAgent` (agents). The committed `.opencode/` tree now carries **zero**
  `$CLAUDE_PLUGIN_ROOT` / `~/.claude/kaola-workflow` tokens. Canonical `commands/*.md` /
  `agents/*.md` are untouched (additive D-530-02).
- **Installer partition** (`install-opencode.sh`): gains the same `--with-fast` / `--with-full`
  arg parsing + R1 effective-opt-in (UNION of shared-config `installed_paths` and this-run flags,
  computed before `copy_tree`) + D2 per-file conditional deploy (`kaola-workflow-fast.md` iff
  effective-fast; `kaola-workflow-phase[1-5].md` iff effective-full; adaptive-core always) + D4
  `seed_kaola_config` UNION writer (byte-identical to `install.sh:712`). `--enable-adaptive` is
  accepted-but-ignored with a warning.
- **Standalone support-script deploy**: `install_support_scripts` now deploys to
  `${OPENCODE_CONFIG_DIR:-$HOME/.config/opencode}/kaola-workflow/scripts` (the path the rewritten
  resolver searches), not `~/.claude/`.
- **Generator output set unchanged**: `writeCommands` still emits all 12 command files into the
  committed `.opencode/command/` (the single source the installer copies from). The partition is
  install-time COPY selection, not generator-time trimming — so the existing A9/A20/A21
  committed-tree assertions stay green.

## Alternatives considered

- **Plugin split (separate fast/full marketplace entries).** *Rejected.* Massively invasive — a new
  `marketplace.json`, a release/check-lockstep rewrite, and a new version-coordinate axis. The
  runtime gate already enforces legality across all editions, so the invasive split buys nothing the
  config-writer does not already provide. Disproportionate to the gap (which was a missing writer,
  not a missing enforcement point).

- **Manifest skills-subset (conditionally list only installed skills in `plugin.json`).** *Rejected.*
  Breaks the `"skills":"./skills/"` assertion that `validate-kaola-workflow-contracts.js` (×3 forge
  validators) and `validate-workflow-contracts.js` all pin. The validator-mandatory
  `kaola-workflow-fast/SKILL.md` must remain on-disk; only its runtime *reachability* is gated.

- **Installer conditional-deploy D2 on Codex (mirror Claude's per-file conditional copy).**
  *Rejected* for the Codex surface. Codex's installer deploys agent profiles (role-based, not
  path-based) and skills (validator-mandatory); there is no natural file target whose
  presence/absence maps to the fast/full partition. The config-writer (R1 + D4) alone is sufficient
  because the runtime gate is the enforcement point.

- **Docs-only config-gate (tell users to hand-edit `config.json`).** *Rejected.* User-hostile — a
  Codex-only or opencode-only user would be forced to hand-edit JSON to reach the opt-in paths,
  while Claude users get a flag. The partition must be reachable by a flag on every edition for
  behavioral parity.

## Consequences

**Positive:**
- **Behavioral parity across all three editions.** A default install on Claude Code, Codex, or
  opencode now ships adaptive-only; `fast` and `full` are reachable by the same `--with-fast` /
  `--with-full` flags on every edition, recorded by UNION into the same edition-agnostic shared
  config (`~/.config/kaola-workflow/config.json`, canonical order `["fast","full"]`). The config is
  written byte-identically by all three installers, so no edition clobbers another's opt-ins.
- **opencode is self-contained.** It resolves its support scripts under opencode-native paths and
  deploys there; it runs standalone on a machine with no Claude Code and creates no phantom
  `~/.claude/` namespace. Zero `CLAUDE_PLUGIN_ROOT` / `.claude/kaola-workflow` tokens remain in the
  committed `.opencode/` tree.
- **Defense-in-depth restored.** The install-time partition (what is copied / what is recorded) and
  the runtime gate (`path_not_installed`) together enforce the #538 posture on every edition; the
  gate was already present, the partition now matches it.
- **Pure node port on Codex.** The Codex writer has no `python3` dependency, making it strictly more
  robust than the bash mirrors (which python3-guard the UNION write).

**Negative / note:**
- The Codex `kaola-workflow-fast/SKILL.md` and the 15 agent profiles remain on-disk after a default
  install even though fast is not reachable — this is intentional (validator-mandatory +
  role-based), and the runtime gate is what makes the partition effective. There is no file-level
  "is fast installed?" signal on Codex; `installed_paths` in the shared config is the single
  authority.
- A re-install without flags preserves prior opt-ins (UNION semantics); the reset path is
  uninstall → reinstall (which clears the shared config), the same as Claude.
