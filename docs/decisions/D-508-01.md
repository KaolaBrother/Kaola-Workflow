## Context

`cmdFinalize` archives the project and stages the `chore: archive` commit before any remote
push occurs. For a **single-issue** project, this pre-push window is correctly reflected in
the closure receipt as `remote_issue_closed: close_pending` â€” the issue cannot be closed
before the commit is pushed, so closure is deferred to sink-merge and the receipt says so
honestly.

For a **bundle** project on the merge/keep-worktree lane, the bundle close-loop in
`cmdFinalize` iterated over each member and, if none could yet be closed (the expected state
â€” the commit hasn't been pushed), reported `remote_issue_closed: partial`. `partial` carries
the semantics of a **mixed** outcome: some members closed and some not. In the pre-push
window ALL members are open, so the outcome is not mixed â€” it is uniformly deferred, exactly
mirroring the single-issue `close_pending` case.

The genuine `partial` case â€” where some members close successfully and others genuinely fail
after a push attempt â€” is a real failure state that #497 surfaced. Using `partial` for the
pre-push all-open case collapsed two distinct states into one token, making it impossible to
distinguish a correct deferral from a real partial failure in receipt auditing.

## Decision

**Align the bundle merge/keep-worktree lane with the single-issue `close_pending` semantics.**

`cmdFinalize`'s bundle close-loop now reports `remote_issue_closed: close_pending` when all
bundle members are still open on the merge/keep-worktree lane â€” the pre-push state where no
member has had an opportunity to close yet. Remote member closure remains deferred to
sink-merge, which closes all members atomically as part of the #369 `all_or_nothing` policy.

The `partial` value is preserved for the genuinely-mixed case: when some members closed
successfully after a push attempt and others did not. The `failed_issue_closures` list
accompanies `partial` in that case (per the #497 contract).

**The close-before-push invariant (#497) holds for bundles without change.** No member is
closed before the branch is pushed and merged. `cmdFinalize` only stages the archive commit
and records the `close_pending` deferral; `sink-merge` performs the actual closures once the
merge is confirmed.

## Implementation

In `scripts/kaola-workflow-claim.js` (canonical + the Codex byte-twin + the two forge ports):
the bundle close-loop in `cmdFinalize` detects the pre-push all-open case (no push has
occurred yet in the finalize flow) and emits `remote_issue_closed: close_pending` instead of
`partial`. The `test-bundle-finalize.js` test suite gains coverage asserting that the
merge/keep-worktree lane produces `close_pending` rather than `partial` when all members are
still open, and that the genuinely-mixed case (some closed, some not) still produces `partial`.

## Consequences

- Bundle finalize receipts on the merge/keep-worktree lane now correctly read
  `remote_issue_closed: close_pending`, matching the single-issue deferral semantics.
- A receipt reader can distinguish a correct pre-push deferral (`close_pending`) from a real
  post-push partial failure (`partial`) without ambiguity.
- The `partial` + `failed_issue_closures` contract from #497 is preserved for the genuinely
  mixed post-push failure case.
- The close-before-push invariant and the #369 `all_or_nothing` policy are unchanged.
