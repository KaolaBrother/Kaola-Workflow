# D-696-01 — Generate reviewer profiles from one versioned behavior contract

- **Status:** accepted
- **Date:** 2026-07-15
- **Context:** issue #696

## Context

`code-reviewer` and `adversarial-verifier` existed as separately maintained Claude Markdown and
Codex TOML prompts across runtime/forge editions. Token presence could prove that selected phrases
were mirrored, but not that the complete behavioral policy was equivalent. Runtime-specific tools,
model policy, and evidence transport also risked leaking into the behavioral core, while manual
copies could silently diverge.

The proof boundary also needed to be honest. Exact tracked or installed prompt files are observable;
the natural-language output of different stochastic models and the private prompt-loading internals
of proprietary runtimes are not byte-equivalence surfaces.

## Decision

Make `templates/reviewers/behavior-contracts.json` the strict, versioned canonical source for the two
runtime-neutral reviewer roles. It owns description, nickname candidates, stable behavior sections,
receipt vocabulary, and finding schema. Reject unknown roles/keys/section ids, duplicate ids,
non-string policy lines, contradictory description/body statements, and prompt provenance.

Keep runtime differences in the closed `templates/reviewers/runtime-adapters.json` schema. An adapter
contains exactly `tools`, `model_policy_ref`, and `evidence_transport`; it cannot inject arbitrary
prompt prose. Codex uses inherit-by-omission, so generated TOMLs contain neither `model` nor
`model_reasoning_effort`.

`scripts/generate-reviewer-profiles.js` is the sole writer for:

- Claude base `code-reviewer` and `adversarial-verifier`;
- Claude higher-profile `code-reviewer`; and
- the six GitHub/GitLab/Gitea Codex TOMLs.

It emits deterministic LF/final-newline bytes and supports `--write`, `--check`, and
`--manifest-json`. Every render carries:

- `behavior_contract_hash`, the SHA-256 of canonical JSON for the runtime-neutral role behavior,
  excluding adapter/output/runtime syntax; and
- `resolved_profile_hash`, the SHA-256 of the complete rendered profile after replacing its one
  unique self-hash value with exactly 64 ASCII zeroes.

The same delimited normalized behavior core is emitted into Markdown and decoded TOML instructions.
OpenCode derives from the generated Claude root and must preserve that core, role, behavior version,
and behavior hash.

Installer and preflight checks compare selected source bytes, installed bytes, behavior/resolved
hashes, managed manifests, and plugin-cache copies at their respective scopes. Claude's documented
`model: inherit` install rewrite recomputes the resolved self-hash over the transformed installed
bytes. These are filesystem-byte proofs only; they do not attest proprietary runtime prompt loading.

## Consequences

- Reviewer behavior changes begin in canonical JSON and regenerate all outputs atomically; generated
  profiles are never hand-edited.
- Runtime adapters cannot silently alter judgment policy.
- Exact normalized behavior equivalence is machine-checkable across Claude, Codex, GitLab, Gitea,
  and OpenCode.
- Identical behavior contracts do not require identical stochastic findings, explanations, or
  domain outcomes. Conformance compares identities and invariant classes, not prose equality.
- Repository, global/user, project, and plugin-cache drift remain separate observable scopes with
  scope-specific repair instructions.

## Alternatives considered

- **Continue token-pin mirroring.** Rejected because sparse tokens cannot prove complete behavioral
  equivalence or detect arbitrary extra prose.
- **Generate from one runtime's final prompt without structured source.** Rejected because runtime
  adapter data and judgment policy would remain entangled.
- **Claim runtime prompt-load proof from installed files.** Rejected because filesystem equality
  does not reveal private loader behavior.
