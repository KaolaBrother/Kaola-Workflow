# D-515-01. Reciprocal switch-ON guard: refuse defaulted fast/full under an ON adaptive switch

Date: 2026-06-17
Status: **Accepted** — shipped in the #515 build run.
Issue: #515 (gap filed as memory entry `feedback_default_to_adaptive_path.md`)
Related: #254 (adaptive-as-default under ON; fast/full as explicit escapes), #235 (authoring-allowed
         guard, the sibling this record mirrors), #44 (Agent Owns Reasoning — guard validates,
         never selects), #287 (planner-first; no approval gate), #400 (six-surface prose contract),
         `scripts/kaola-workflow-claim.js` (`claimProject`, guard site),
         `scripts/test-route-reachability.js` (T11 prose-floor block),
         `docs/conventions.md` § Switch-ON path guard (#515).

## Context

#254 established the contract: under an ON adaptive switch, adaptive is the **contract-determined
default**; `fast`/`full` are explicit user escapes only. The router (`workflow-next.md` Branch B)
exports `KAOLA_PATH` on EVERY switch-ON route, so a legitimate fast/full escape always carries an
explicit `KAOLA_PATH` value to the sub-process.

The gap: `authoring-allowed` (#235) hard-guarded *adaptive authoring* — it refuses adaptive-phase
writes when the switch is OFF — but there was **no reciprocal guard** for the reverse direction:
*entering* `fast`/`full` under an ON switch without an explicit opt-in. Any agent that reached
`cmdStartup` / `cmdClaim` directly (not via `/workflow-next`) and omitted `--workflow-path` /
`KAOLA_PATH` would silently collapse to the `full` path via the `|| 'full'` default, bypassing the
#254 contract with zero error signal.

The bypass window is narrow — the router enforces the guard procedurally on the normal path — but
it is a real structural gap: the procedural guard existed in prose but not in code.

## Decision

Two complementary levers. Both are the **cheapest-sufficient** mechanism (#3); neither over-engineers:

### Lever 1 — Script guard in `claimProject` (reciprocal `authoring-allowed` analog)

Insert one guard block in `claimProject` across all four `claim.js` editions, immediately after the
existing `workflow_path_refused` legality check. It fires only in the previously-silent bypass
window and is strictly additive:

```js
const pathWasDefaulted = !args.workflowPath && !process.env.KAOLA_PATH;
if (adaptiveEnabled && pathWasDefaulted) {
  return {
    status: 'path_requires_explicit_opt_in',
    claim: 'none',
    issue: issueNumber,
    project,
    reasoning: 'adaptive switch is ON, so adaptive is the default path; "' + requestedPath +
      '" was reached by default (no --workflow-path / KAOLA_PATH). fast/full are explicit ' +
      'escapes only — pass --workflow-path fast|full (or export KAOLA_PATH) to opt in. ' +
      'Refusing to silently downgrade to a non-adaptive path under an ON switch (#254/#44).'
  };
}
```

**Predicate refinement.** The `requestedPath = args.workflowPath || process.env.KAOLA_PATH || 'full'`
collapse means a *defaulted* path (both inputs falsy) ALWAYS resolves to `'full'`. `'fast'` can only
arrive from an explicit truthy input — "defaulted fast" is the **empty set** (unconstructable). So
`adaptiveEnabled && pathWasDefaulted` is the complete predicate; no `requestedPath` membership test
is needed, and a `requestedPath === 'fast'` disjunct would be unreachable dead code.

The guard reuses `adaptiveEnabled` already computed at line 789 (no new config surface) and mirrors
the `workflow_path_refused` family shape. The reasoning string is forge-neutral and byte-identical
across all four editions, mirroring the `cmdAuthoringAllowed` body discipline (#341).

### Lever 2 — Prose floor on the 12 fast/full-entry surfaces

A `<!-- PIN: adaptive-default-contract -->` comment plus the `path_requires_explicit_opt_in` literal
are added to the 12 prose surfaces where fast/full entry is described (6 per path: 3 Claude commands
+ 3 Codex SKILL packs for each of fast and full, per the #400 six-surface rule). A new fail-closed
**T11 block** in `scripts/test-route-reachability.js` asserts both the PIN comment and the literal
are present on all 12 surfaces; the block is `assert` not `warn` so it red-closes on any missing
surface.

### Harness hermeticity fix

The new guard fires on every bare `startup` under an ON switch, which revealed that the six walkthrough
test harnesses were non-hermetic: they could inherit the developer's HOME-level `enable_adaptive=true`
config. All six walkthroughs received an unconditional module-top `KAOLA_ENABLE_ADAPTIVE='0'` default
(with the `runNode` re-add placed BEFORE `...(extraEnv||{})` so explicit `'1'` sub-tests still win).
This is test-hermeticity, not a product behavior change — the product guard is exercised by test
case (a') in `test-claim-hardening.js`.

## Boundaries preserved (accuracy non-negotiable, #1)

| Boundary | Verdict | Mechanism |
|---|---|---|
| B1: switch-OFF default→full intact | SAFE | Guard gated on `adaptiveEnabled`; OFF → `adaptiveEnabled===false` → guard skipped; `\|\| 'full'` survives. |
| B2: explicit `--workflow-path fast\|full` under ON allowed | SAFE | `args.workflowPath` truthy → `pathWasDefaulted===false` → guard skipped. Router exports `KAOLA_PATH` on every Branch B route, so legitimate escapes always carry an explicit value. |
| B2: explicit `KAOLA_PATH=fast\|full` under ON allowed | SAFE | `process.env.KAOLA_PATH` truthy → `pathWasDefaulted===false` → guard skipped. |
| B3: adaptive + bundle claims under ON allowed | SAFE | Adaptive front end always passes `--workflow-path adaptive` (explicit → guard skipped). Bundle adaptive-only lane passes `--workflow-path adaptive` explicitly. |

## Why this shape (not the alternatives)

- **Inline in `claimProject`, not in a schema helper.** The predicate is two truthiness checks and
  the refusal object is already shaped like the `workflow_path_refused` family. A schema helper would
  add four new-file edits for zero reuse. INLINE keeps the edition surface minimal.
- **`pathWasDefaulted` not `requestedPath==='full'`.** The collapse identity `defaulted ⇒ full` makes
  a membership test redundant AND misleading — it would read as if explicit `full` were the trigger,
  when the distinction is defaulted-vs-explicit. The predicate names the actual condition.
- **T11 as fail-closed `assert`, not warn.** A warn-mode pin gives false confidence; the #400
  route-reachability design is fail-closed and this guard follows the same discipline.
- **No planner gate.** The front-door claim check is the right enforcement point (#287: planner-first,
  no approval gate bolted onto the planner). The planner authors plans; the router decides the path;
  the claim script validates and refuses if bypassed.

## Consequences

- A bare `startup`/`bootstrap` invocation under an ON switch without `--workflow-path`/`KAOLA_PATH`
  now returns `path_requires_explicit_opt_in` (claim: none) instead of silently downgrading to full.
- Fast/full remains fully accessible via `--workflow-path fast|full` or `export KAOLA_PATH=fast|full`.
- The prose floor is machine-pinned — a surface missing the PIN or the typed token reds the chain.
- The six walkthroughs are now hermetic and environment-independent.
- Cross-edition: all four `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains + the
  walkthrough simulation run clean (verified by n4-review gate, verdict:pass).
