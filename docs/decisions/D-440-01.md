## Context

D-420-02 pinned the Part 2 enriched-halt design: `runWriteHalt` attaches a classified `triage` payload —
carrying the offending-paths arrays `barrierCheck` already computes, a matched mechanical `class`, and a
ready-made plan-repair diff — to its existing return, additively, without re-running the barrier and without
touching the durable marker-writing transaction. D-420-02 introduced three new mechanical subtypes of
`write_set_overflow` (`lockfile_write` / `mirror_write` / `count_bump`) under [INV-14], bound the payload to
forge-neutral JSON under [INV-17], and required ×4 edition propagation under [INV-18]. But it left three
questions open:

- **OQ-P2-a** — where the subtype classification pattern physically lives (hardcoded literal set vs.
  config-driven vs. derived from existing constants), and how it stays forge-neutral.
- **OQ-P2-b** — the repair-diff format: unified-diff text, a structured object the orchestrator renders AND
  applies, or both.
- **OQ-P2-c** — where the per-attempt `test_thrash` failing-test delta is recorded so the halt can attach it.

A fourth concern was implied but not pinned: the threading path — how the `barrierOut` envelope the
`barrier_failed` close already returns (`adaptive-node.js:1586-1591`, returning `reason: 'barrier_failed'`
with the full `barrierOut` object) reaches `runWriteHalt` so the halt carries the diagnosis forward rather
than re-deriving it. And a fifth: the `barrier_failed` refusal envelope and the `write-halt` payload should
carry ONE triage shape, not two divergent ones, so a consumer (operator or the D-420-01 autopilot) classifies
the same structure on both channels.

Issue #440 settles all five. The decisions below are binding for the implementation. The design is transcribed,
not redesigned — #440 is an implementation-settlement record over the D-420-02 invariants, exactly as D-442-01
is for D-420 Part 4.

## Decision

### 1. Pattern source (OQ-P2-a) — the classification table lives in `adaptive-schema.js`

The subtype classification table — the literal-pattern definitions that narrow `write_set_overflow` into
`lockfile_write`, `mirror_write`, and `count_bump` — lives in `scripts/kaola-workflow-adaptive-schema.js`, the
byte-identical forge-neutral constants home and the cross-edition drift anchor. It is **never** a forge token
and **never** names a forge CLI binary.

This placement is binding because `adaptive-schema.js` is the ONE surface the edition-sync discipline keeps
byte-identical across all four editions: a classification table that drifts between the canonical tree and a
forge port would let the same overflow path classify differently per edition, breaking the cross-edition
contract (D-420-02 [INV-18]). Putting the table in the schema home — rather than inline in `adaptive-node.js`
or duplicated per forge — makes the byte-identity machine-enforced rather than hoped-for. The classifier is a
pure literal-pattern test over the `outOfAllow` array (no fs, no mutation, no re-freeze), consistent with how
`write_set_granularity` (#404) is detected.

### 2. Repair-diff format (OQ-P2-b) — a STRUCTURED `proposed_repair` object, not a literal diff

`proposed_repair` is a **structured object**, not a unified-diff text blob:

```json
{
  "kind": "write_set_swap | add_to_write_set | revert_overflow | repair_node",
  "node": "<node-id>",
  "paths": ["<path>", "..."]
}
```

The `kind` vocabulary reuses the **#434 sanctioned-repair-primitives vocabulary** (D-434-01) so the proposed
repair names a primitive the orchestrator already knows how to apply:

- `write_set_swap` / `add_to_write_set` — plan-edit primitives: the `--freeze` write-set repair pattern that
  widens (or swaps) a node's declared write set to admit the offending `paths`, preserving the ledger because
  `plan_hash` covers only `## Meta` + `## Nodes`. The structured form gives the autopilot (D-420-01 [INV-6])
  exactly the `{ node, paths }` it needs to MATERIALIZE the `--freeze` edit; a unified-diff text would be
  human-readable but not directly applicable.
- `revert_overflow` / `repair_node` — the #434 subcommand primitives (`adaptive-node.js revert-overflow` /
  `repair-node`): when the correct recovery is to surgically undo the overflow (`revert_overflow`) or to
  re-open a `complete` writer for re-dispatch without laundering its baseline (`repair_node`), `proposed_repair`
  names that primitive directly.

A structured object — not a literal diff — is the settlement because it is consumable BOTH by a human
(readable, named primitive + node + paths) AND by the autopilot's auto-apply path (machine-dispatchable to the
named primitive). It collapses the operator's decision to a yes/no (D-420-02 [INV-15]). `runWriteHalt` COMPUTES
and ATTACHES `proposed_repair`; it does NOT apply it (the script never re-freezes or runs a primitive from
inside a halt). Applying it is the orchestrator's decision under D-420-01 [INV-6]'s repair-consent boundary —
the #44 "agent owns reasoning" rule covers repair as well as selection. For a `sensitive_write_unreviewed` or
`foreign_archive` class, no `proposed_repair` is offered (these require a security-reviewer node or are a stray
foreign-archive sweep — not a write-set widen); the triage surfaces the offending paths and the required
structural fix in prose.

### 3. `test_thrash` delta source (OQ-P2-c) — #432 chain receipt when present, else node-evidence RED/GREEN

A `test_thrash` halt attaches the failing-test delta sourced, in precedence:

1. **From the #432 chain receipt** (`.cache/chain-receipt.json`, D-432-01) when present — it records the real
   per-chain `exitCode` and is the crash-durable, machine-written record of what failed.
2. **Else from the node-evidence RED/GREEN lines** — the role agent's `.cache/{node-id}.md` evidence file
   carries the per-attempt RED→GREEN test transitions; when no chain receipt exists, the delta is read from
   those lines.

This is the settlement of D-420-02's [INV-16] (the `testDelta` field) and OQ-P2-c (the capture source). The
delta lives in the SAME `triage` return shape as the overflow case — one `triage` object whose populated
fields depend on the class. No new structured capture machine is added; both sources already exist (the #432
receipt and the node-evidence file).

### 4. Threading (settlement 4) — `write-halt --triage-json <path|->` consumes `barrierOut`

`write-halt` gains a `--triage-json <path|->` input that consumes the `barrierOut` envelope already returned by
the `barrier_failed` close. `close-and-open-next` ALREADY returns that envelope at `adaptive-node.js:1586-1591`
(`{ result: 'refuse', reason: 'barrier_failed', nodeId, barrierOut }`), so the diagnosis the validator computed
flows forward into the halt rather than being re-derived (D-420-02 [INV-13]: diagnosis is THREADED, never
re-run). The orchestrator that holds the `barrier_failed` envelope passes it to `write-halt` via `--triage-json`
(a file path, or `-` for stdin); `runWriteHalt` reads the offending-paths arrays (`outOfAllow`, `sensitiveHits`,
`foreignArchiveHits`, `unattributed`) and the typed `reason` from that envelope, classifies the subtype per the
`adaptive-schema.js` table (decision 1), and attaches the `triage` payload. `runWriteHalt` does NOT re-run
`barrierCheck`; `barrierCheck` (`plan-validator.js`) remains the single source of the offending-paths arrays. A
halt raised WITHOUT a prior barrier diagnosis (a bare `consent` escalation with no overflow) carries an
empty/absent offending-paths set and no `proposed_repair`, not a fabricated one.

### 5. `barrier_failed` triage (settlement 5) — ONE shape on BOTH channels

The overflow `barrier_failed` refusal envelope carries the **SAME `triage` shape** as the `write-halt` payload.
A consumer classifies one structure regardless of which channel surfaced it:

- the `barrier_failed` close (`adaptive-node.js:1586-1591`) — the close-time channel, and
- the `write-halt` return — the escalation channel.

Both attach `{ class, offending paths, proposed_repair?, testDelta? }`. There is no second triage schema for the
`barrier_failed` channel: the `barrier_failed` envelope's `triage` is built by the same classifier (decision 1)
over the same `barrierOut` arrays, so the autopilot and the operator branch STRUCTURALLY on one shape. This is
the single-shape settlement that makes the threading (decision 4) coherent — the envelope the close emits is the
same envelope `write-halt` enriches, not a re-serialization into a different schema.

### 6. The three subtypes NARROW `write_set_overflow` — do NOT fork the taxonomy

The three mechanical subtypes — `lockfile_write`, `mirror_write`, `count_bump` — each NARROW
`write_set_overflow` within the precedence-ordered `barrierCheck` `reason` envelope (the emit envelope). They are
**not** a fifth precedence family. A path that matches none of the three stays plain `write_set_overflow`; a path
that matches one is reported with the narrower subtype as its `class`, but it sits UNDER `write_set_overflow` in
the same precedence-ordered family, exactly as #404 placed `write_set_granularity` under it.

This is binding per the **D-419-02 [INV-13] do-not-fork-the-taxonomy lesson**: the emit envelope is a CLOSED,
precedence-ordered set of failure families so callers classify structurally and never by string-match; adding a
fifth precedence family for a class that is really a narrowing of an existing one forks the envelope and breaks
the structural-classification contract. The classification table (decision 1) reports a `class` that is a
subtype LABEL within `write_set_overflow`, never a new top-level `reason`. The precedence order
(`foreign_archive` > `sensitive_write_unreviewed` > `write_set_overflow` { + its `write_set_granularity` /
`lockfile_write` / `mirror_write` / `count_bump` subtypes } > `unattributed_write`) is unchanged.

### 7. Forge-neutral JSON + ×4 edition propagation

The `triage` payload (including `class`, `proposed_repair`, and `testDelta`) is forge-neutral JSON. No
`gh` / `glab` / `tea` token appears in the classification table (which lives in `adaptive-schema.js`, the
byte-identical home — decision 1), in the `proposed_repair` object (which describes a `workflow-plan.md`
write-set edit and the named #434 primitives, never a forge command), or in the `testDelta`. `runWriteHalt`
lives in `kaola-workflow-adaptive-node.js`, which ships in all four editions (canonical, codex byte-twin,
gitlab/gitea hand-ports). The enrichment is therefore a CROSS-EDITION diff: it MUST reach all four trees, and
ALL FOUR `npm run test:kaola-workflow:{claude,codex,gitlab,gitea}` chains MUST be green, run sequentially — a
green claude chain alone is insufficient evidence (the `&&` short-circuit hides a red forge chain; CLAUDE.md
Validation Policy #307). The subtype table and the triage shape must be byte-identical across the four in
`adaptive-schema.js`; the consuming logic in `adaptive-node.js` must be functionally identical per edition. If
the operator-facing halt prose changes in the plan-run surfaces, the six-surface propagation rule (#400) also
applies.

## Consequences

**A halted (or barrier-failed) node no longer strands the operator** with a bare marker. The `triage` payload
carries the offending paths and the mechanical `class` forward on BOTH channels (decision 5), and for the
overflow family attaches a structured `proposed_repair` (decision 2) that names a #434 primitive the
orchestrator — or the D-420-01 autopilot — can apply or render. The decision collapses to a yes/no; the
autopilot's repair-consent flow (D-420-01 [INV-6]) gets the structured input it was designed to consume.

**The diagnosis is threaded, never re-derived** (decision 4): the `barrierOut` envelope the close already
returns flows into `write-halt` via `--triage-json`, preserving `barrierCheck` as the single source of the
offending-paths arrays.

**The taxonomy is not forked** (decision 6): the three subtypes narrow `write_set_overflow` exactly as #404's
`write_set_granularity` does, honoring the D-419-02 [INV-13] do-not-fork lesson. A caller still classifies the
same closed, precedence-ordered emit envelope structurally.

**What is no longer open from D-420-02:**

- OQ-P2-a — the classification pattern lives in `adaptive-schema.js`, the byte-identical forge-neutral home,
  never a forge token (decision 1).
- OQ-P2-b — `proposed_repair` is a structured `{ kind, node, paths }` object using the #434 primitives
  vocabulary, not a literal diff (decision 2).
- OQ-P2-c — the `test_thrash` delta is sourced from the #432 chain receipt when present, else the node-evidence
  RED/GREEN lines (decision 3).

**Test fixtures the implementation must cover (binding for the implementer):**

- An overflow `barrier_failed` whose `outOfAllow` matches `package-lock.json` classifies `class:
  "lockfile_write"` (a subtype of `write_set_overflow`, never a new top-level `reason`).
- A `mirror_write` / `count_bump` path classifies to its subtype; a path matching none stays plain
  `write_set_overflow`.
- `write-halt --triage-json <barrierOut>` attaches the SAME `triage` shape the `barrier_failed` close carries
  (one shape, both channels — decision 5).
- An overflow-family triage attaches a structured `proposed_repair` with a `kind` from the #434 vocabulary; a
  `sensitive_write_unreviewed` / `foreign_archive` triage attaches NONE.
- A `test_thrash` halt attaches `testDelta` from the chain receipt when present, else from the node-evidence
  RED/GREEN lines.
- No forge CLI token appears in the classification table, `proposed_repair`, or `testDelta` (forge-neutrality).
- The classification table is byte-identical in `adaptive-schema.js` across all four editions; all four chains
  green, run sequentially (#307).

**For downstream implementation:** the binding invariants are D-420-02 [INV-13] through [INV-18] (threaded-
not-rederived diagnosis + #406-reason generalization without forking the taxonomy + suggestion-not-applied
repair + symmetric `test_thrash` delta + forge-neutral JSON + ×4 propagation), now closed at the implementation
level by the five settlements above. The do-not-fork-the-taxonomy constraint (decision 6, D-419-02 [INV-13]) is
the single most load-bearing: the three subtypes are LABELS under `write_set_overflow`, never a fifth precedence
family.
