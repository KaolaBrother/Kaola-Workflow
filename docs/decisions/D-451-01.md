## Problem

#405 implemented per-node reasoning-effort tiering on Codex by shipping a parallel matrix of
`<role>-max` profiles: for each of six "opus-eligible" roles, a committed `agents/<role>-max.toml`
carrying `model_reasoning_effort = "xhigh"` plus an `[agents.<role>-max]` registration block, generated
deterministically by `variantProfileText(base)` and pinned by a derivation guard in every contract
validator and walkthrough. When a plan node resolved to `model: opus`, the Codex dispatch selected the
`<role>-max` variant; otherwise it used the base `<role>` profile.

This matrix is a standing maintenance and correctness cost: 18 generated profile files + 18 config
blocks across three editions, a generator subcommand, and bijection/derivation guards in five
contract surfaces — all to express a single bit per role ("run this spawn at xhigh"). It also could
not express any effort other than the base-tuned value or `xhigh`, and it bound effort to the
profile name rather than to the node the planner actually reasoned about.

## Feasibility (AC6 — settled docs-based, no live observation required)

Codex 0.139 has **no per-spawn `reasoning_effort` override field** (mechanism #1 does not exist). The
unblocking mechanism is #2: `model_reasoning_effort` is **optional** in an `[agents.<name>]` profile;
when a profile **omits** it, a spawned agent **inherits the parent session's reasoning effort**, and
PR #14807 (shipped in 0.139) makes agent-config **win over** the project profile. Per-node effort is
therefore **session-scoped**: the executor sets the session effort per the dispatch descriptor before
each spawn.

This feasibility rests on authoritative, version-pinned primary sources (the Codex 0.139 changelog +
PR #14807), which is sufficient for AC6 — **live Codex observation is not required**. The runtime
behavior (omitting the line actually yields session inheritance in 0.139) is the one claim that cannot
be proven from this repository's code; the G1 review confirmed the descriptor/SKILL **wiring** is
correct and self-consistent, and bounded the runtime claim to its version-pinned source.

## Decision

**D1 — Omit `model_reasoning_effort` from all 14 base profiles (×3 editions = 42 files).**
A *pinned* base effort would WIN over the session (PR #14807) and defeat per-node control, making the
feature inert. So the base profiles carry no effort line; the session default IS the role default.
**This is a deliberate deviation from AC4's literal "preserve base tuned effort"** — there is no tuned
base effort anymore. The installer + preflight `validateProfileText` were relaxed to treat
`model_reasoning_effort` as **optional** (present ⇒ must be a legal value; absent ⇒ valid), so the
stripped base profiles still pass the schema wall.

**D2 — The dispatch descriptor carries the effort signal.** As originally accepted,
`dispatchEffort(model)` (in the ×4 byte-identical `kaola-workflow-adaptive-schema.js`) resolved
`opus → { codex_reasoning_effort: "xhigh", codex_reasoning_effort_source: "planner_model" }` and
`sonnet`/absent `→ { codex_reasoning_effort: null, codex_reasoning_effort_source: "role_default" }`.
D-582-01 supersedes the `sonnet` branch: `sonnet` now resolves to `high` with `planner_model`, while
absent/blank tiers still resolve to `null`/`role_default`. `buildDispatch` spreads it onto the descriptor
alongside `agent_type` (always the base role); a single builder flows to every dispatch caller
(`open-next`, the fused advance, and the running-set scheduler).

**D3 — `agent_type` is always the base role; `OPUS_ELIGIBLE_ROLES` is removed.** There are no eligible
vs. ineligible roles anymore — any node may carry any tier, and the effort signal (not a profile
suffix) carries it. Non-dispatchable roles (`finalize` sink, `main-session-gate`) carry `model: null`
→ effort `null`.

**D4 — The Codex plan-run SKILL applies the signal.** Before dispatching a node, the executor delegates
to the base `dispatch.agent_type` profile and, when `dispatch.codex_reasoning_effort` is non-null,
passes it directly as per-spawn `reasoning_effort`; when null it leaves the standing session effort
untouched.

**D5 — The matrix is deleted, not deprecated.** The 18 `<role>-max.toml` files + 18
`[agents.<role>-max]` config blocks are removed; the generator (`generateMaxVariants` +
`variantConfigBlock` + `--generate-variants`) is deleted; the five contract/walkthrough derivation
guards are replaced by a FORBID (0 `*-max.toml`, 0 `[agents.*-max]`) and a profile-count flip
(20 → 14). On upgrade, the six `<role>-max.toml` names are added to `RETIRED_PROFILE_FILES` so the
installer prunes a machine that installed #405 — **user-owned `*-max` profiles are never
blanket-globbed** (only the six Kaola-generated names are retired; unknown `*-max` files are preserved
via `extra_unmanaged`).

## Historical tradeoff — session-effort persistence (the "don't raise" semantic)

D-582-01 supersedes this tradeoff for explicit `sonnet` tiers by mapping them to per-spawn `high`
effort. The inheritance tradeoff below remains only for absent/blank model tiers.

The former `sonnet`/`role_default` branch was **"don't raise the session effort"** (D2): it left the
standing session effort untouched rather than resetting it to a baseline. Consequence: once any
`opus` node had raised the session to `xhigh`, a subsequent `sonnet` node inherited `xhigh` — the
executor did not lower it back. This was accepted for #451 because the alternative required defining
and threading a session baseline. After D-582-01, that caveat applies only to absent/blank model tiers
that intentionally inherit.

## Consequences

- One bit per node ("run at xhigh") is now expressed by the node's tier + the session, not by an
  18-file generated matrix — the dispatch path, installer, validators, and walkthroughs all shrink.
- The four-chain (#307) gate stays green: the schema/dispatch/installer/validator/walkthrough/SKILL
  changes propagate to all four editions in lockstep, and `validate-script-sync` confirms the ×4
  schema, ×4 preflight, ×3 installer, and 14 base-toml triples remain byte-identical.
- Effort is now bound to the node the planner reasoned about (the tier), not to a profile name —
  matching the #382 per-node tier model end to end.
