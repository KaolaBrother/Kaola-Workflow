## Context

`test-route-reachability.js` is the machine that enforces CLAUDE.md's **SIX-surfaces** routing
contract (#400): adaptive / routing / finalize-wiring PINs and literal tokens must survive in 3
Claude commands **+** 3 Codex SKILL packs (the historic dead zone). Its T4–T11 T-sets assert
**content-reachability** — that specific wiring tokens (`<!-- PIN: frontier unit -->`,
`<!-- PIN: closure-audit -->`, `result: escalate`, `--write-overlap-consent`,
`fast_compliance_unresolved`, `path_requires_explicit_opt_in`, `<!-- PIN: adaptive-default-contract -->`,
…) are **present** in 6 or 12 generated surfaces. T1–T3 cover the Claude+Codex receipt/contract
surface.

The opencode edition (`.opencode/command/*.md`, 12 commands generated from canonical by
`sync-opencode-edition.js`) is covered only by **A9** in `test-opencode-edition.js:245-262`, which
asserts **bare target resolution** — that the 5 receipt-emitted command files **exist**. A9 is the
twin of T2 (existence), **not** a content-reachability check like T4–T11.

**The gap is real and empirically verified** (`.cache/n3-critique.md` §1). The adversarial verifier
injected a throwaway `text = text.replace(/<!--\s*PIN:.*?-->/g, '')` into
`transformCommandBody` (`sync-opencode-edition.js:199`), regenerated via `--write`, and ran
`test-opencode-edition.js`:

| State | Assertions | RED |
|---|---|---|
| Baseline | 223 pass | — |
| PIN-comment strip only | 223 pass | none |
| PIN strip + 7 literal tokens (`closure-audit`, `result: escalate`, `--write-overlap-consent`, `--speculative-consent`, `fast_compliance_unresolved`, `path_requires_explicit_opt_in`, `frontier unit`) | 223 pass | none |

No existing assertion catches it:

- **A6** (`test-opencode-edition.js:136-140, :153-157`) compares
  `read('.opencode/command/X') === renderCommand(...)` — but both sides call the same edited
  `transformCommandBody`, so they **mutate together**.
- **A14** (`:120-130`) checks model-prose **absence** patterns, never PIN **presence**.
- **A9** (`:245-262`) is file-existence only.

Tokens were stripped from all 7 commands with all 223 assertions green. (Tokens currently survive
*incidentally* — verified present in `finalize.md:923`, `adapt.md:113`, `next.md:356`,
`auto.md:98`, `plan-run.md:128`.) Tree restored via `git checkout`; baseline re-confirmed (223
pass, PINs back). This is the exact class of regression T4–T11 prevents for claude/codex.

## Decision

**Candidate C — extend `test-opencode-edition.js` with content-reachability assertions for the
PIN/literal wiring tokens on the 6 generated opencode commands.**

Concretely: assert the **presence** of the load-bearing wiring tokens (the same token set T4–T11
checks for claude/codex — `closure-audit`, `result: escalate`, `--write-overlap-consent`,
`--speculative-consent`, `fast_compliance_unresolved`, `path_requires_explicit_opt_in`,
`frontier unit`) in the generated `.opencode/command/*.md` files. This closes the gap surgically
**within the declared opencode twin test** (`test-opencode-edition.js:10-12` self-declares as the
opencode twin of the forge test), where `.opencode/command/*.md` is already in scope — no new
file, no policy change. Estimated cost: ~15–20 new assertions. **Implementation is a follow-up**
(defect X1) — this record is the *decision*, not the in-this-issue implementation.

## Alternatives considered

- **Candidate A — add `.opencode/command/` to T4–T11 in `test-route-reachability.js`** (join the
  formal forge T-set surface). **REJECTED.** CLAUDE.md:103 defines the SIX surfaces as "3 Claude
  commands + 3 Codex SKILL packs" — a **forge-scoped** policy. opencode is non-forge (see
  D-530-02); admitting it into `test-route-reachability.js` is a heavyweight policy change that
  introduces a non-forge surface into a forge-specific file, and would force every T-set's
  surface-count assumptions (6 or 12) to be revisited. The gain (content-reachability for 6
  opencode commands) is achievable without that policy change — which is exactly what Candidate C
  does.
- **Candidate B — status quo (A9 only).** **FALSIFIED** by the strip counterexample above: a
  `transformCommandBody` edit stripping `<!-- PIN:...-->` (and the 7 literal tokens) passes all
  223 assertions. The gap is not theoretical.

## Consequences

- **The gap is closed surgically**, within the opencode twin test, with no forge-policy change and
  no new propagation surface (no #307 four-chain obligation — `test-opencode-edition.js` is not a
  forge chain).
- **~15–20 new assertions** added to `test-opencode-edition.js`; they run wherever the opencode
  twin test runs today (install-time via `install-opencode.sh:62-63`, and manually). Combined with
  D-530-02's additive boundary, this is the parity enforcement that replaces "join the forge
  T-sets."
- **Implementation is a follow-up** (defect X1, filed at finalize). The acceptance criterion for
  issue #530 is a *recorded decision* (this document), not the implementation — per the frozen
  plan's no-speculative-fix-nodes discipline (#486 Case A).
- **Token set is shared conceptually with T4–T11** but asserted independently — if a future
  routing change adds a new PIN, both the forge T-sets and the opencode assertions need a parallel
  update. That is the correct cost shape for an additive edition (mirrors how
  `sync-opencode-edition.js` already mirrors the canonical command source).

- `.cache/n3-critique.md` §1 — **empirical validation**: the PIN-strip + 7-literal-strip

  existence-only. Verdict: gap conclusively real; Candidate B refuted; Candidate C correct.
