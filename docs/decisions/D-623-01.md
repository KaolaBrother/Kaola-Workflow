# D-623-01. Scope the rolling top-up promise to read frontiers; a write frontier wider than the cap runs in fixed group waves, not live-group admission

Date: 2026-07-07
Status: Accepted
Issue: #623 (primary — rolling top-up into a live write lane group was promised on three routing
       surfaces but never implemented)
Related: #622/D-622-01 (the `write_node_exclusive` leg-contained relaxation and the group-identity
         invariants — baseline, `write_union`, membership fixed at formation — that a future
         live-group admission mechanism would have to extend); #627 (same bundle, same
         `commands/kaola-workflow-plan-run.md` lane, n1); #628 (same bundle, same
         `docs/plan-run-cards/frontier-batch.md` file, n4); #472/#463/D-542-01 (planner-proven-disjoint
         write frontiers co-opening in isolated legs — the mechanism this decision's promise was
         describing)

## Context

Three routing surfaces — `commands/kaola-workflow-plan-run.md` (mirrored across all 6 plan-run
command/SKILL surfaces), `agents/workflow-planner.md`, and `docs/plan-run-cards/frontier-batch.md`
— carried a false promise: that a fan-out wider than `FANOUT_CAP` "drains via a rolling top-up as
members close," i.e. that `open-ready`'s re-run admits a new member into an already-live group as a
slot frees.

That promise is true for a **read** frontier (`code-explorer` / `knowledge-lookup` /
`adversarial-verifier` reads have no group-identity to protect, so admitting a new one mid-flight is
safe). It is **false** for a **write** frontier: a lane group's membership, its `write_union`, and its
baseline are all fixed at group formation (D-437-01/D-542-01), and `write_node_exclusive` refuses any
new open while any group member is still live (relaxed only for reads by #622/D-622-01, not for new
write members). A write frontier wider than `FANOUT_CAP` therefore cannot top up into the live
group — the prose was describing a mechanism that does not exist.

## Decision

**Chose prose honesty (Option 1) over implementing live-group write admission (Option 2).**

The three surfaces are corrected to state: rolling slot-level top-up (`open-ready` admitting a new
member as a slot frees) is a **read-frontier** behavior only. A **write** frontier wider than
`FANOUT_CAP` runs in **fixed group waves** instead — the first `≤cap` members form a group and run to
completion (each wave paying its own synthesizer-merge and group barrier), then the next wave forms as
a **new** group. Makespan is the sum of the per-wave maxima, not a rolling drain. The `FANOUT_CAP` /
`KAOLA_FANOUT_CAP_READONLY` tokens and all other machine-pinned literals on the three surfaces are
kept; only the incorrect drain clause is rewritten to this shared canonical wording (kept
byte-for-byte consistent in meaning across all three surfaces, since they are edited in two disjoint
file lanes — n1 for the command/SKILL surfaces, n4 for the planner + card — and must not diverge).

Option 2 (build member-admission into a live write group) was considered and rejected for now: it
would let a new member join a group's `write_union`/baseline after formation, mid-flight, while other
members are still writing.

## Consequences

- The three surfaces no longer promise a mechanism that does not exist; a wide write frontier's actual
  behavior (fixed group waves, each wave its own merge + barrier) is now documented accurately.
- No change to `write_node_exclusive`, group formation, `write_union`, or baseline semantics — this is
  a documentation-only fix, verified behavior-preserving by the bundle's adversarial-verifier pass.
- A genuinely wide (`>FANOUT_CAP`) planner-proven-disjoint write fan-out still completes correctly
  today, just in sequential waves rather than a single rolling-admission group; makespan for that case
  is worse than the (unimplemented) rolling-admission ideal, but correctness was never at risk since
  the prior prose described a capability the code never had.
- The wave-vs-admission gap remains open for a future decision: it should only be revisited if wide
  (`>cap`) write fan-outs become common in practice, or if a future read/write co-open relaxation makes
  the admission bookkeeping fall out cheaply as a side effect of other work — neither condition holds
  today.

## Non-goals

- **No change to `FANOUT_CAP` or `KAOLA_FANOUT_CAP_READONLY` semantics.** This decision only corrects
  the prose describing what happens when a frontier exceeds the cap; the cap value and its read-only
  override are untouched.
- **No change to `write_node_exclusive`, lane-group formation, or the running-set scheduler.** This is
  scoped to routing-surface prose on three files; no script logic changed.

## Alternatives considered

- **Option 2 — implement member-admission into a live write group.** Rejected for now: admitting a new
  member post-formation touches the group-identity invariants (`write_union`, baseline, membership)
  that the group barrier, `reconcile-running-set`, and the synthesizer merge all key on today. The
  blast radius (four editions' worth of running-set scheduler logic) is high for a gain that is
  efficiency-only (shorter makespan on a fan-out shape that is not yet common) and does not affect
  correctness either way — the fixed-group-wave behavior is already correct, just not maximally
  concurrent. Deferred rather than built speculatively, per the project's precedence-ordered
  design theory (accuracy and simplicity first; parallelism width is sized to genuine, not
  hypothetical, scope).
