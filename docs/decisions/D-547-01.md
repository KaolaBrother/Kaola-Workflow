# D-547-01. Content-addressed code-tree-hash freshness key for the chain receipt (Stage 2)

Date: 2026-06-21
Status: Accepted
Issue: #547
Related: #432 (the chain receipt this extends), #424/D-424-01 (the `isBarrierInvisible` narrow
         allowband reused as the freshness band's base), #475/D-475-01 (the consumer final-validation
         gate — left UNTOUCHED), #551 (the ~30-min self-host repeat this eliminates),
         the #547 Stage-1 commit `60877d75` (the prose "cite-don't-rerun" discipline this builds on).

## Context

#547 Stage 1 (commit `60877d75`, prose-only) added a "cite-don't-rerun" discipline to the adaptive
path but DEFERRED the mechanical receipt and silently omitted the `validation_command` Meta field its
own text promised. Stage 2 (this record, the user's "Right-sized" decision) finishes the issue.

The self-host Finalization gate (`plan-validator --finalize-check`, self-host branch) keyed chain-receipt
freshness on **commit-SHA equality** (`receipt.headSha !== HEAD` → `chains_stale`). So ANY new commit
after the chains ran — including a docs-only / `CHANGELOG`-narrative / workflow-state-only commit whose
**code tree is byte-identical** — forced a full four-chain re-run (~30 min, #551). The producer already
wrote a `workTreeHash`, but it hashed only `git diff HEAD` (the uncommitted delta) and the gate ignored
it — half a content-addressed receipt, unused.

A naive fix ("exclude all `*.md` / `docs/**` from the freshness hash") is a CONFIRMED correctness bug
for this repo: the four chains' contract validators read and assert on `docs/api.md`,
`docs/workflow-state-contract.md`, `docs/agents-source.md`, root `README.md`, and root `CHANGELOG.md`.
Excluding those would let a real regression to them be cited-as-unchanged and ship green — exactly the
`*.md`-blanket bug #424 already paid to remove once.

## Decision

**Re-key the chain receipt's `chains_stale` check from the headSha pin to a content-addressed
code-tree hash, over the narrow allowband MINUS the test-consumed prose.** No new subsystem — this
extends the existing receipt and reuses the existing allowband predicate.

1. **`computeCodeTreeHash(root, project, testConsumedExtra)`** (`plan-validator.js`, exported) =
   `sha256` of the filtered, sorted `git ls-tree -r <snapshotWorktree(root)>` listing (`mode sha path`,
   so content changes flip it). `snapshotWorktree` captures the committed+working **landable** set —
   the SAME set the per-node barrier and merge gate scope to, so a hash match proves the chains
   validated this exact code. Returns `null` on any git failure → caller fails CLOSED.

2. **The freshness band — `isValidationInvisible(p, project, extra)`** — a path is excluded from the
   hash iff a change to it cannot flip a chain verdict: the #424 `isBarrierInvisible` allowband
   (`docs/**`, root `README`/`CHANGELOG`, the active project tree) PLUS the whole `kaola-workflow/`
   workflow-state tree (folded in **project-independently** so the producer and gate agree regardless
   of `--project`), **MINUS `testConsumes(p)`** — the prose the chains read + assert on, which stays
   CODE. `testConsumes` checks first, so a verdict-affecting doc is never excluded.

3. **`SELF_HOST_TEST_CONSUMED`** = `{README.md, CHANGELOG.md, docs/api.md,
   docs/workflow-state-contract.md, docs/agents-source.md}` — the exact set this repo's four chains
   read, verified by a machine guard (`test-validation-allowband.js`) that scans the chain validators
   for allowband-member prose literals and refuses if any is not kept-as-code. A self-hosting fork
   extends it at runtime via the `validation_test_consumes` `## Meta` field (no predicate fork).

4. **Producer/gate agreement.** `run-chains.js` records `codeTreeHash` + the resolved
   `validationTestConsumes` in the receipt via the SAME exported helper the gate recomputes; the gate
   replays the recorded band. A legacy receipt lacking `codeTreeHash` falls back to the headSha pin.

5. **`validation_command` `## Meta` field** (`parseValidationCommand`) — the planner records the
   consumer's validation command ONCE at freeze; nodes/Finalization reuse it instead of re-deriving
   (closes the Stage-1 omission). Reader-only (no freeze gate).

## What this deliberately does NOT do

- **It does not touch the #475 consumer gate.** A consumer (non-npm) repo never runs `run-chains`;
  its Finalization still gates on the agent-recorded `.cache/final-validation.md` (`verdict: pass`).
  D-475-01's deliberate no-staleness-probe decision stands. The accuracy floor is unchanged; Stage-2
  de-dup is a makespan optimization layered ON TOP of, never replacing, the agent's verdict gate.
- **It builds no durable consumer-side `validation-receipt.json` / `validate-or-cite` helper.** The
  issue's "primary design" (the "Full receipt" option) was considered and not chosen — the consumer
  eliminable surface is small (the implementer regression-green baseline + finalize Step 1), and the
  hash mechanism is self-host-only by nature. Right-sized per precedence #3 (cheapest sufficient).

## Consequences / known bounds (precedence #1 honored)

- **Accuracy preserved structurally.** A change to code or any chain-asserted doc still flips the hash
  → `chains_stale`. The fail direction is closed: an unlisted inert doc is only a missed de-dup, never
  a missed regression; the guard test stops the keep-as-code list from drifting behind the tests.
- **CHANGELOG stays code.** A `CHANGELOG`-only commit still re-runs (its version-heading assertion is
  verdict-affecting). The eliminated waste is the inter-node state-churn re-runs + narrative-doc /
  ADR / investigations commits — the bulk of #551.
- **Worktree/dual-location finalize.** If the producer (worktree) and gate (main) hash different
  checkouts, the hashes differ → re-run (fail-closed, correct, just not optimal). Documented bound.
- **Pure function of the tree.** A flaky/time/network-dependent suite violates "same tree → same
  verdict"; a cited pass could mask a now-failing flaky test (not worse than today, now explicit).

## Alternatives considered

- **`git rev-parse HEAD^{tree}` (committed tree SHA).** Rejected: doesn't exclude docs (so it does
  not fix the bug) and ignores uncommitted code the chains validated.
- **Gate-only derivation from `receipt.headSha`.** Rejected: the validated working-tree state is gone
  by gate time if the chains ran with uncommitted code — the producer must record the hash.
- **Auto-scan tests for consumed docs at receipt time.** Rejected as the primary (fail-open if a read
  is missed); adopted instead as a *test-time* guard over an explicit fail-closed list.
