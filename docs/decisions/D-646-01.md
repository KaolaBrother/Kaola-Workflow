# D-646-01. Reverse the earlier ungoverned-issue-scout decision now that the full two-node placeholder/case render contract exists

Date: 2026-07-09
Status: Accepted
Issue: #646
Related: `kaola-workflow/archive/issue-328/.cache/design.md` (the decision this reverses —
         "ISSUE_SCOUT_MODEL placeholder NOT NEEDED"); `kaola-workflow/archive/issue-443/.cache/n5-commands.md`
         (the regression this closes for good — a partial land of the placeholder without the
         matching `install.sh` case broke install-render, and the fix at the time was to remove the
         placeholder rather than complete the wiring); analogous prior art: `code-reviewer`'s
         higher-profile wiring (`install.sh`'s profile-swap check plus its
         `model_for_placeholder`/`placeholders` entries)

## Context

An earlier decision concluded `issue-scout` should stay dispatched via prose only, with no
`model="{ISSUE_SCOUT_MODEL}"` lever, reasoning that it is a pre-claim, read-only, selection-only
role like `adversarial-verifier`/`code-explorer` and therefore does not need one. A later, separate
incident then reinforced that posture the hard way: someone introduced the placeholder text into
command prose without adding the matching `model_for_placeholder` case in `install.sh`, which broke
`test-install-model-rendering.js`'s unrendered-token guard; the fix at the time removed the
placeholder rather than complete the wiring. #646 reopens the original decision, judging the
resulting state — issue-scout's effective dispatch tier is ungoverned, with no lever at all — worth
fixing now that the full two-node contract (the placeholder text AND its install.sh resolution
case, landed atomically) is actually in scope for this bundle.

## Decision

1. Add `agents/profiles/higher/issue-scout.md` (`model: opus`), mirroring `code-reviewer`'s
   higher-profile shape exactly — full verbatim body copy of the base agent, one line changed. The
   base `agents/issue-scout.md` stays `model: sonnet`.
2. Wire `ISSUE_SCOUT_MODEL` into BOTH of `install.sh`'s placeholder lists in the same edit pass —
   the `model_for_placeholder()` case AND `render_command_file()`'s `placeholders` array. This is the
   direct lesson from the earlier regression: a partial land of either list alone reproduces it.
3. The placeholder lands on the three `workflow-next` COMMAND surfaces only, never on any Codex
   SKILL — `install.sh` renders `{X_MODEL}` tokens in commands only, and the Codex skills keep
   prose-only scout dispatch, matching the model-less `issue-scout.toml` twins across all three
   plugin trees. A placeholder in the shared routing-skeleton body (rather than the command-only
   region) would leak `{ISSUE_SCOUT_MODEL}` into the Codex skills and fail the render/route checks.
4. Resolver defaults are unchanged: `DEFAULT_AGENT_MODELS['issue-scout']` stays `sonnet`;
   `REASONING_FLOOR_ROLES` is untouched. This is explicitly a quality-tier lever available under
   `--profile=higher`, not a change to the reasoning floor every profile guarantees.
5. New render-completeness pins target the regression shape directly rather than merely avoiding it:
   `test-install-model-rendering.js` gains manifest-tier assertions (`issue-scout` resolves to
   `opus` under `--profile=higher`, `sonnet` under common), and `validate-workflow-contracts.js`
   gains an explicit `assertIncludes` pin for the `model="{ISSUE_SCOUT_MODEL}"` dispatch line on
   `commands/workflow-next.md` — a surface outside the generic `phaseCommands` dispatch scanner,
   which would not otherwise cover it.
6. No edit lands in any of the three edition contract validators. Reasons: agent counts stay
   unchanged (issue-scout is already a registered base agent; only its higher-profile *file* is
   new, and that is a Claude-only install artifact — no `plugins/*/profiles/higher` tree exists);
   the placeholder is Claude-command-only by design, so pinning it in an edition validator would
   forbid the correct state; the axiom-adjacent six-surface propagation is already centrally
   machine-enforced by `required-blocks.js`; and each edition validator's generic per-agent
   dispatch-table check already auto-covers issue-scout.

**In-run repair.** This run's own `code-reviewer` and `adversarial-verifier` gates found that the
generic `{X_MODEL}` placeholder strip in `scripts/sync-opencode-edition.js` was mangling the
scout-dispatch paragraph on the generated opencode surface — it left an empty inline-code span plus
two sentences false for opencode (an install.sh render step and a placeholder that opencode does not
have). Fixed in the same run with a scoped paragraph-level rewrite, placed before the generic strip,
substituting opencode-true wording ("its effort variant resolves centrally from `opencode.json`"); a
new `test-opencode-edition.js` assertion was proven load-bearing by independently reconstructing the
old mangled output and confirming it reds the new negative checks.

## Consequences

- `issue-scout` now has a governed higher-profile lever exactly like the `code-reviewer`/
  `security-reviewer` family — an operator running `--profile=higher` gets `opus`-tier scout
  dispatch without any further wiring.
- The historical unrendered-placeholder regression shape is closed by a render-completeness pin
  (both the manifest-tier assertion and the explicit dispatch-line pin) rather than by removing the
  feature, as the earlier fix did.
- The opencode-generated scout-dispatch paragraph is now accurate to the opencode runtime, and the
  guard proving it (`test-opencode-edition.js`) is proven load-bearing, not merely present.
- The reversal is intentional and narrow: it does not change issue-scout's dispatch timing or its
  read-only nature, only adds a model-tier lever that did not exist before.

## Non-goals

- Does not change issue-scout's dispatch timing, its read-only nature, or its role in bundle
  selection.
- Does not add issue-scout to `REASONING_FLOOR_ROLES` or change any other resolver default.
- Does not add any Codex-skill-side model placeholder — Codex dispatch remains prose-only by
  design across all three plugin editions.
- Does not touch the three edition contract validators — the reasons for leaving them unedited are
  recorded above and were independently verified in-run.
