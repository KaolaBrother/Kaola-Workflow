# D-444-01: dispatch descriptor (`buildDispatch`) + `record-evidence --verify`

**Status**: Accepted
**Date**: 2026-06-13
**Issue**: #444 (D-421 P1+P2)

---

## Problem

Two related gaps were identified in the adaptive-node lifecycle:

### 1. Dispatch descriptor drift (#411 class)

The `opened` payload emitted by `runOpenNext` and the fused advance in
`runCloseAndOpenNext` each assembled their per-node metadata independently.
When a field was added to one opener it could be missing from the other —
the root cause of the #411 class of bugs. There was no structural guarantee
that the two paths produced the same descriptor for the same node.

Additionally, the `opened` payload lacked a sub-object that aggregated all
dispatch-relevant fields under one key, forcing orchestrators to destructure
the flat `opened` object and hope nothing drifted.

### 2. Evidence verify gap

The `record-evidence` subcommand only supported `--stdin` (write path). There
was no READ-ONLY mode to verify that on-disk evidence was well-formed BEFORE
dispatching a close. The `--verify` gap meant that close failures (binding
mismatch, missing token) were only detectable at close time, not proactively.

---

## Decision

### 1. `buildDispatch(nodeInfo, context)` — single shared builder

ONE module-level function assembles the `dispatch` sub-object for ALL three
openers: `runOpenNext`, `runOpenReady`, and the fused advance in
`runCloseAndOpenNext`. This closes the #411 class by construction: there is
exactly one producer, so the three call sites cannot drift.

Field set (stable shape):
```
dispatch: {
  node_id:            string,
  role:               string,
  model:              string|null,
  working_dir:        string|undefined,
  declared_write_set: string,        // RAW write-set cell (byte-fidelity)
  evidence_file:      string,        // '.cache/<node-id>.md'
  nonce:              string|null,
  required_tokens:    string[],
  forge_rider:        string|null,   // null until a concrete rider is supplied
  guards:             string[],      // computed by deriveGuards
  goal_line?:         string,        // optional; key absent when no goal
}
```

The pre-existing sibling fields on `opened` (`id`, `role`, `model`, etc.) are
kept in place for one release (back-compat with existing readers).

### 2. `deriveGuards(nodeInfo)` — script-owned guard derivation

A pure function computing `guards[]` from the node's role and declared write
set. Deterministic order:

- `'read-only'` — GATE_ROLES (`code-reviewer`, `security-reviewer`,
  `adversarial-verifier`, `main-session-gate`).
- `'RED-fixture-in-$TMPDIR'` — `tdd-guide` role (#424: RED fixtures must
  never be written to the worktree).
- `'sync:editions'` — write set contains a GENERATED_AGGREGATORS sibling
  (any of canonical + codex + forge ports). Anchor-gated on `edition-sync.js`
  availability: absent → returns `false` → no guard (no false positives).

### 3. `GATE_ROLES` promoted to module level

Previously defined inline inside `runReopenNode`. Promoted so both
`runReopenNode` and `deriveGuards` share one definition (single source of
truth), consistent with `VERDICT_ROLES` already being module-level.

### 4. `record-evidence --verify` subcommand (READ-ONLY)

New `runVerifyEvidence(opts)` handler for `--verify`. Behavior:
1. Resolves role from `## Nodes` (read-only).
2. Resolves evidence path from `.cache/<node-id>.md`.
3. Returns `{result:'refuse', reason:'evidence_absent'}` if the file is absent.
4. Reads content and nonce (the same `readNonce` the close path uses).
5. Delegates to `checkEvidenceShape` (the same checker the close path uses)
   with `{expectedNonce, expectedNodeId}`. This covers both the anti-replay
   binding check (#392) AND the role-specific token checks in one call.
6. Maps result to typed returns:
   - `{result:'ok', nodeId, role, evidence_file}` on pass.
   - `{result:'refuse', reason:'evidence_stale'|'evidence_unbound'|'evidence_shape_failed',
     missingTokenClass, ...}` on failure — same reason vocabulary as the close gate.

`--stdin` stays for one release (backward compat). `--verify` is mutually
exclusive with `--stdin`. `--verify` writes NOTHING.

### 5. `working_dir` plumbing

The three openers accept `working_dir` from opts and pass it into
`buildDispatch` context. The CLI `main()` currently passes `null` as a
placeholder (the active worktree path resolution is documented in the
orchestrator prose; this issue adds the plumbing channel without a concrete
source, consistent with FLAG-0 in the n1 design contract).

### 6. Boundary decision

`scripts/kaola-workflow-next-action.js` and `scripts/kaola-workflow-commit-node.js`
(and their ports) are NOT edited. The descriptor is assembled in the OPENER
from data these scripts already return. No new field is required from them.

---

## Consequences

- The `opened.dispatch` sub-object is the canonical dispatch descriptor for
  orchestrators. The flat sibling fields remain for one release, then can be
  removed.
- `record-evidence --verify` enables proactive pre-close evidence validation
  without any side effects.
- `deriveGuards` is deterministic and fork-able: adding a new guard class
  means adding one condition here, not touching multiple opener sites.
- `GATE_ROLES` is now module-level; `runReopenNode` no longer redeclares it.
- All four editions (canonical + codex + gitlab + gitea) receive these changes
  via `npm run sync:editions`.
