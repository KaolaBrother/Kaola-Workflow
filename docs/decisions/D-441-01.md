## Context

D-420-01 covered Part 1 (the autopilot loop) and Part 3 (the optional `goal:` line in `## Meta`) together as
one "driver + termination" cluster. Part 3 supplied the autopilot's primary terminal: a met goal beats
backlog-empty. D-420-01 pinned the goal line as an additive, hash-covered, reader-only `## Meta` field
([INV-7] through [INV-12]) — `computePlanHash` already hashes the WHOLE `## Meta` body, so the line is covered
for free with no hash code change; a `parseGoal` READER mirrors `parseLabels`; no validator GATE is added; old
goal-absent plans stay valid and hash-stable.

But D-420-01 left the FORM and the SEAT of the goal-satisfaction check open:

- **OQ-3 (goal form — prose vs. structured)** — is `goal:` a single prose line (agent-judged) or a structured
  AC list the validator mechanically checks? D-420-01 supported prose (it mirrors `labels:`) but explicitly
  deferred the final call: "D-420-02 should make the final call."
- **The attestation seat** — D-420-01 [INV-12] placed the AC-vs-goal attestation at the SINGLE sink seam
  `sink-merge.js` already owns. That placement predated D-427-01's finding that the LIVE finalize flow runs
  `cmdFinalize` in `claim.js` and (per #427) never invokes `sink-merge.js` on the live path. The seat must move
  to where finalize actually runs.
- **Blocking semantics** — does the goal-satisfaction check BLOCK claim or finalize, or is it informational
  metadata in v1?

Issue #441 settles all of these as **Part 3** of the D-420 cluster — the goal-conditioned-bundles
implementation-settlement record. The six settlements below are binding for the implementation. The design is
**transcribed, not redesigned**: #441 is an implementation-settlement record over the D-420-01 Part 3
invariants, exactly as D-440-01 is the Part 2 settlement record and D-442-01 is the Part 4 settlement record.

## Decision

### 1. Goal form (OQ-3 / settlement 1) — a single `## Meta`-scoped prose line read by a decoy-immune `parseGoal`

`goal:` is a single **prose** line in the `## Meta` section, NOT a structured AC grammar. The validator gains a
`parseGoal` READER that reads `^goal:[ \t]*(.*)$` from the `## Meta` section using the **SAME**
`classifier.sectionBody(content, 'Meta')` reader that `parseLabels` uses (`plan-validator.js:507` / `803`). This
is decoy-immune by construction: scoping the read to the hash-covered `## Meta` region means a decoy `goal:` line
placed ELSEWHERE in the plan cannot override the frozen value — the same #B1 audit scoping that confined
`parseLabels` to `## Meta`.

`parseGoal` is a **READER only** — it does NOT add a goal GATE. Freeze accepts goal-absent plans **unchanged**:
an absent optional line changes nothing in the normalized `## Meta` body, so old frozen plans stay valid and
hash-stable. This settles OQ-3 in favor of prose (D-420-01 [INV-9]/[INV-12]); a structured form would have
needed a grammar and a new validator gate, contradicting the reader-not-gate invariant.

The line is **hash-covered for free**: `computePlanHash` (`plan-validator.js:783`) normalizes and hashes the
ENTIRE `## Meta` body (`body = norm('Meta') + '\n---NODES---\n' + norm(NODES_HEADING)`), so a `goal:` line inside
`## Meta` is automatically covered with NO hash code change. Tampering the goal after freeze trips
`plan_hash_mismatch` on `--resume-check`, exactly as tampering `labels:` does. The goal inherits the same
anti-tamper guarantee as `labels:` (D-420-01 [INV-8]).

### 2. Env pinned (settlement 2) — `KAOLA_GOAL` is the operator-side entry; the goal ALSO travels in dispatch prompts

`KAOLA_GOAL` is the **operator-side env entry**: the env the operator/orchestrator sets to supply the
user-stated goal, paralleling the existing scout-output → `KAOLA_TARGET_ISSUE` env wiring (D-420-01 [INV-11]).
The scout reads it as CONTEXT for bundle clustering, and the `workflow-planner` transcribes `goal: <text>` into
`## Meta` at authoring time alongside `labels:`, frozen as part of the EXISTING `--freeze`/`--governance-ack`
transaction — no new freeze spawn.

Because subagent shells do NOT inherit `KAOLA_GOAL` (#430 — the env-inheritance boundary), the goal text ALSO
travels in the scout/planner **dispatch prompts**: when the orchestrator dispatches the `issue-scout` and the
`workflow-planner`, it places the goal text in the dispatch prompt so the subagent receives it even though the
env does not cross the spawn boundary. This is an operator/orchestrator concern, not a script-owned env read
inside the subagent.

### 3. `goal_check` field (settlement 3) — a machine-readable closure-receipt field

A machine-readable `goal_check` field is added to the finalize closure receipt. Its value is the enum
`satisfied | unsatisfied | absent`:

- `satisfied` — a frozen `goal:` line was present and the attestation judged the closed bundle's acceptance
  criteria to meet it.
- `unsatisfied` — a frozen `goal:` line was present and the attestation judged the ACs NOT to meet it.
- `absent` — no `goal:` line was frozen in `## Meta` (the goal-absent backwards-compatible case).

The field is added to `CLOSURE_RECEIPT_FIELDS` in `scripts/kaola-workflow-closure-contract.js`
(`closure-contract.js:20`), the shared closure-contract schema — so the receipt builder, the invariant checker,
and `docs/api.md` § Closure Contract all carry the field from one source.

### 4. Attestation seat (settlement 4) — emitted by `cmdFinalize` in `claim.js`, NOT `sink-merge.js`; ADVISORY in v1

The attestation is emitted by **`cmdFinalize` in `kaola-workflow-claim.js`** — NOT `sink-merge.js`. This
relocates the seat D-420-01 [INV-12] originally placed at the `sink-merge.js` sink seam. The relocation is
binding per **D-427-01 / #427**: the LIVE finalize flow runs `cmdFinalize`, and the live path never invokes
`sink-merge.js`, so an attestation wired only into `sink-merge.js` would never fire on the live flow.

The attestation is **ADVISORY in v1**: `cmdFinalize` writes the typed `goal_check` value into the finalize
receipt and nothing more. It does NOT block. The agent-judged AC-vs-goal verdict (D-420-01 [INV-12] — the goal
is prose, judged at one seam) is recorded as the `goal_check` enum, not enforced as a gate. Flip-to-blocking is
deferred to the **#429 follow-up** (D-429-01's resumable-sink lane is where a blocking goal check would live).

### 5. Scout integration (settlement 5) — goal input + `goal_alignment` note across the md + 3 toml twins; cross-ref #430

The scout gains the goal as an input across **all four scout surfaces**: the `agents/issue-scout.md` source and
its **3 toml twins** (the codex + the two forge-codex SKILL/agent ports). The scout clusters by goal alignment —
not just shared scope — and surfaces a **`goal_alignment` note** in its recommendation so the operator sees WHY
a bundle was recommended under the goal.

This cross-references **#430 (target-set integrity)**: a goal-conditioned bundle must still validate its claimed
set. Goal alignment is a clustering INPUT to the scout's recommendation; it does NOT relax the D-430-01
bundle-coherence invariant. A bundle the scout recommends under a goal is still subject to the
`bundle_id == "bundle-" + sorted(issue_numbers).join("-")` coherence check and the `target_set_mismatch` claim
guard. Goal conditioning narrows WHICH issues cluster together; it never lets an incoherent claimed set through.

### 6. Blocking semantics (settlement 6) — ADVISORY in v1; informational metadata only

`goal_check` is **informational metadata** in the receipt in v1. It does NOT block claim and does NOT block
finalize in this version. The goal-satisfaction verdict is recorded for the operator (and for the D-420-01
autopilot's `goal_satisfied` terminal to read), but no stage refuses on an `unsatisfied` value. Flip-to-blocking
is **deferred to a follow-up** (the #429 lane per settlement 4). This keeps v1 strictly additive: a goal that is
judged `unsatisfied` surfaces in the receipt but never strands a finalize.

## Consequences

**The goal line costs nothing to add and is decoy-immune.** `parseGoal` mirrors `parseLabels` exactly — same
`^goal:[ \t]*(.*)$` shape, same `## Meta`-scoped `sectionBody` read — so the goal is hash-covered for free
(decision 1), inherits the `labels:` anti-tamper guarantee, and cannot be overridden by an out-of-region decoy.
No hash code change, no validator gate, no new freeze spawn. Old goal-absent plans stay valid and hash-stable.

**The attestation fires where finalize actually runs** (decision 4). Moving the seat from the D-420-01 [INV-12]
`sink-merge.js` sink-seam to `cmdFinalize` in `claim.js` aligns the goal check with the LIVE finalize flow that
D-427-01/#427 established. An attestation left on the dead sink-seam would have been a silent no-op on every live
run.

**The goal travels two ways because the env does not cross the spawn boundary** (decision 2). `KAOLA_GOAL` is
the operator entry, but #430's env-inheritance boundary means subagent shells do not see it — so the goal text
also rides the scout/planner dispatch prompts. The orchestrator owns placing it in both.

**A goal-conditioned bundle still validates its claimed set** (decision 5, cross-ref #430). Goal alignment is a
scout clustering input and a `goal_alignment` recommendation note; it never relaxes the D-430-01 coherence /
target-set-integrity guards.

**v1 is advisory; blocking is a named follow-up** (decisions 4 + 6). `goal_check` is `satisfied | unsatisfied |
absent` informational metadata in the closure receipt (decision 3). It does not block claim or finalize.
Flip-to-blocking is deferred to the #429 follow-up — the resumable-sink lane (D-429-01) where a blocking goal
gate would be enforced.

**What is no longer open from D-420-01:**

- **OQ-3 (goal form)** — `goal:` is a single `## Meta`-scoped prose line read by a decoy-immune reader-only
  `parseGoal`, NOT a structured AC grammar (decision 1). The D-420-01-deferred call is discharged.
- **The attestation seat** — moved from the `sink-merge.js` sink-seam ([INV-12]) to `cmdFinalize` in `claim.js`
  per D-427-01/#427 (decision 4).
- **Blocking semantics** — advisory in v1; informational metadata; flip-to-blocking deferred to the #429
  follow-up (decisions 4 + 6).

reaches the plan-run / next surfaces.
