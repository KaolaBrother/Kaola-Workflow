# D-667-01. Fail closed on a structurally-ambiguous claimed Scope

**Date:** 2026-07-12
**Status:** Accepted
**Related:** D-660 (not recorded as a decision; see below)

## Context

The claim-overlap guard reads a claimed fast project's declared write set from its
`fast-summary.md` `## Scope` section so an in-flight file footprint stays visible to overlap
detection. #660 made the section scanner fence-aware: `sectionBody` now distinguishes a
genuinely absent section from one that is structurally ambiguous (an unclosed fence, or a
duplicate heading the scanner cannot resolve to one body), but it still returns the same empty
string for both outcomes.

The overlap-guard consumer read the Scope exclusively through `sectionBody`. An ambiguous Scope
therefore collapsed to `''`, which the consumer's overlap check reads as "no write set" ->
"no overlap" -> a silent green verdict on a candidate that may in fact overlap the claimed
project's real, unreadable write set. A fail-closed claim-overlap defense had quietly become
fail-open on malformed self-authored input, and the trade was never recorded as a decision.

## Decision

The failure direction for a structurally-ambiguous (unclosed-fence or duplicate-heading) claimed
`## Scope` is **fail closed**: the overlap-guard consumer now reads the claimed Scope's parse
state directly (not just its body) and, when that state is ambiguous, classifies the candidate's
write set as indeterminate and returns a conservative `red` — reversing the #660 fail-open trade
and restoring the pre-#660 protective intent.

This is a value call, not a fact the workflow can auto-derive: when the claimed write set cannot
be determined, the safe verdict is red. A fail-closed defense must not silently become fail-open
because its input happened to be malformed rather than absent.

## Scope of the change

Only the overlap-guard consumer's "ambiguous -> no overlap risk" mapping changed. The section
scanner primitives are unchanged and still refuse to manufacture a missing section: a genuinely
absent `## Scope` (no heading present at all) continues to read as no write set and does not
manufacture an overlap. The change is applied identically across the canonical, Codex, GitLab,
and Gitea classifier editions.

## Consequences

An unclosed fence or duplicate heading anywhere in a claimed fast project's `fast-summary.md`
`## Scope` now conservatively reds any candidate the classifier evaluates while that project is
active, rather than silently going green — strictly safer at the cost of a false-positive red
until the claimed project's malformed Scope is corrected. RED-first regression tests pin the new
consumer behavior (an unclosed pre-Scope fence now asserts `red`, not `green`), alongside a
companion green-path test proving a genuinely absent Scope still does not manufacture an overlap.
