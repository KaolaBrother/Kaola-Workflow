# D-579-01. Harden concurrent same-repo sessions: single main-root authority, liveness marker, and clean-check selectivity

Date: 2026-06-30
Status: Accepted
Issue: #579
Related: D-419-01 (running-set scheduler — parallel execution foundation),
         D-542-01 (disjoint write co-open default),
         D-577-01 / D-578-01 (opencode tracked canonical source — separate additive edition)

## Context

When two workflow sessions operate on the same repo simultaneously (an adaptive worktree run
alongside a manual claim on the main checkout, or two worktrees from two different issues), three
independent weaknesses created hazards:

1. **Authority split on main-root derivation.** The body of `getCoordRoot`/`mainRootFromCoord`
   was triplicated across `claim.js`, `adaptive-node.js`, and `sink-merge.js` (with a local
   re-impl named `getMainRoot` in adaptive-node). A subtle divergence in any copy silently
   returned a different path from a linked or detached worktree, causing the executor to look
   in the wrong location for state files.

2. **No liveness information on active-folder lanes.** `cmdStatus` and `cmdResume` could not
   distinguish an own lane (same session), a live co-tenant lane (another active session), or a
   stale leftover. `cmdResume` prompted ambiguously whenever two folders existed. There was no
   machine-readable basis for "leave this alone" vs "resume this" vs "ask the user".

3. **Over-strict clean-worktree gate.** The gate in `sink-merge.js`/`claim.js` considered the
   entire working tree. In a co-tenant checkout, untracked or modified files belonging to ANOTHER
   lane's `kaola-workflow/<other>/` folder blocked the gate — unrelated to the current session's
   merge and impossible to remove safely.

A full liveness/heartbeat scheme was considered and rejected: a periodic heartbeat requires a
refresh path that touches state outside the claim seam, opens a re-stamp race, and adds
complexity for minimal accuracy gain (a run that completes within a day needs only a single
claim-time anchor, not continuous refresh). The scope was resolved to a minimal-seatbelt approach:
claim-time-only stamp, single conservative staleness constant, pure-function classifier.

The retired `## Lease` mechanism (`session_id`, `last_heartbeat`, `expires`, `owner_session_id`,
`claim_comment_id`) was stripped by `removeLegacyStateBlocks` and MUST NOT be reused. The new
fields (`session_marker`, `claim_ts`, `main_root`) were chosen specifically to avoid those names.

## Decision

### Move 1 — Single main-root authority

Canonicalize `getCoordRoot`, `mainRootFromCoord`, and `resolveMainRoot` in
`scripts/kaola-workflow-adaptive-schema.js` — the byte-identical cross-edition drift anchor — and
delete the duplicate bodies from `claim.js` and `adaptive-node.js`. `claim.js` re-exports
`getCoordRoot` and `mainRootFromCoord` (backward-compatible for `sink-merge.js` and the
forge-port export-superset); `adaptive-node.js` delegates `getMainRoot` to `resolveMainRoot`.

`writeState` (`claim.js`) records `main_root: <resolveMainRoot(root)>` in the `## Sink` block at
claim time. `adaptive-node.js` reads `main_root` back from the local `workflow-state.md` path
before falling back to `getMainRoot(repoRoot)` — so a node running in a linked worktree uses the
authority stamped at claim time rather than re-deriving from cwd. Backward-compatible: absent
`main_root` falls back to the re-derive path unchanged.

### Move 2 — Minimal-seatbelt liveness marker

Two new claim-time fields added to the `## Sink` block by `writeState`; never refreshed:

- **`session_marker`** — produced by `resolveSessionMarker(env)` (new export of
  `kaola-workflow-classifier.js`): `KAOLA_SESSION_MARKER` when set (stable identity across
  multiple script invocations in one session), otherwise `s-<pid>-<timestamp-base36>`.
- **`claim_ts`** — `new Date().toISOString()`, the liveness anchor.

A single staleness constant **`LANE_STALENESS_MS = 86400000`** (24 hours) is exported from
`kaola-workflow-adaptive-schema.js`. The value is deliberately conservative: a run completes well
within a day, so any marker newer than 24 hours should be treated as a potential active co-tenant.
A 48-hour alternative was considered; 24 hours was chosen as the less-risk-tolerant option.

`active-folders.js` (`parseStateFile` + `readActiveFolders`) surfaces all three new fields on
each lane item.

**Four-bucket classifier** (`classifyLane` in `kaola-workflow-classifier.js`) — pure function,
injectable `now`/`staleMs` for testing. Precedence ladder:

1. `session_marker` matches own session → `mine`.
2. Explicit resume instruction (`explicitResumeIssues`) names this lane's issue → `stale`
   (adopt; explicit intent beats freshness).
3. `KAOLA_COTENANT=1` blanket co-tenant signal → `live` (leave untouched).
4. `claim_ts` present and age < `LANE_STALENESS_MS` → `ambiguous` (ask); otherwise → `stale`.

**Consumption:** `cmdStatus` annotates each lane with `lane_bucket` + `lane_bucket_reason`.
`cmdResume` excludes `live` from resume candidates; `stale`/`mine` are resumable; `ambiguous` or
multiple candidates trigger the existing `resume_ambiguous` refusal.

### Move 3 — Clean-check selectivity (merge protocol unchanged)

`assertCleanWorktree`/`assertWorktreeClean` (sink-merge.js) and `treeDirty` (claim.js) gain a
parked-lane filter that exempts non-owned co-tenant scratch from the dirty check. The filter
uses two new exports from `kaola-workflow-adaptive-schema.js`:

- **`PARKED_LANE_PREFIXES = ['kaola-workflow/', '.kw/worktrees/', '.kw/legs/']`**
- **`isParkedLanePath(relPath, ownedProjects)`** — returns `true` (exempt) only when the path is
  under a parked prefix, the second segment is project-shaped (not `.roadmap`, not `archive`, not
  a dot-leading name, not `ROADMAP.md`/`config.json`), and that segment is not in `ownedProjects`.

`parsePorcelainPaths(statusText)` (also in `adaptive-schema.js`) parses the `git status
--porcelain` output into POSIX repo-relative paths (handling rename lines, quoted paths).

Ownership: `ownedProjects` = own project + bundle members at sink time; own project alone at
claim time. Own in-progress state is never exempted.

**Fail-closed invariant preserved:** the filter runs AFTER probe-fault / catch-dirty / stderr
handling — unverifiable tree → dirty, unconditionally. The filter narrows which known-clean states
pass; it never relaxes the unverifiable-is-dirty posture.

**Merge protocol unchanged:** `ffMergeLoop` and the true-conflict halt are byte-unchanged.
The clean-check runs BEFORE `ffMergeLoop`, so the non-owned exemption cannot change conflict
handling. Each lane cleans only its own branch/worktree/folder after its own merge lands.

## Consequences

- The resolver (`getCoordRoot`/`mainRootFromCoord`/`resolveMainRoot`) has one canonical location
  (`adaptive-schema.js`) replicated correctly across all four editions via the byte-identical
  group enforcement; divergence is caught by `validate-script-sync.js`.
- `workflow-state.md` `## Sink` block gains three fields: `main_root`, `session_marker`,
  `claim_ts`. Pre-#579 state files are backward-compatible (absent fields fall back to existing
  behavior; `classifyLane` returns `stale` on absent `claim_ts`).
- `cmdStatus` output is enriched with `lane_bucket`/`lane_bucket_reason` per lane; `cmdResume`
  is more selective (excludes `live` lanes).
- Sink-merge and claim clean-checks no longer fail spuriously on co-tenant lane scratch; the
  fail-closed posture on unverifiable trees is preserved.
- A periodic heartbeat/refresh scheme remains out of scope. If sub-24-hour stale detection is
  ever needed, the `LANE_STALENESS_MS` constant is the single point of change.

## Alternatives considered

- **Periodic heartbeat.** A background refresh path that updates `claim_ts` or a separate
  `last_heartbeat` field every N minutes. Rejected: requires a refresh code path outside the
  claim seam, introduces a re-stamp race, and adds complexity for negligible accuracy gain (runs
  complete well within the 24-hour threshold).
- **48-hour staleness constant.** More conservative (fewer false-live classifications). Rejected
  in favor of 24 hours as the stricter, lower-risk choice for the initial implementation; the
  single constant makes the threshold trivially changeable.
- **Two staleness constants (soft-ambiguous / hard-stale).** Rejected as over-engineering: the
  four-bucket classifier already captures the key states; a second threshold would add decision
  surface without a clear use case.
- **Reuse retired field names (`session_id`, `last_heartbeat`).** Rejected outright:
  `removeLegacyStateBlocks` (claim.js) actively erases those names on archive/close.
- **Leave the resolver triplicated.** Rejected: the three-copy drift was the root cause of the
  authority split; consolidation in `adaptive-schema.js` (the byte-identical group anchor) is
  the correct structural fix with zero per-edition maintenance cost.
