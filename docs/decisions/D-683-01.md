# D-683-01 — Simultaneous-failed-gates repair: the candidate re-bind transaction

Status: accepted
Date: 2026-07-15
Issues: #683 (primary), #684, #685 (bundle `bundle-683-684-685`)
Supersedes/relates: extends the #682 review/repair journal; discharges the R17 deferral recorded in
[D-682-01](D-682-01.md) via #685.

## Context

The #682 review/repair journal made a failed review gate a settled, crash-safe transaction: the failing
gate's attempt records `outcome:'fail'`, fences every opener via `reviewJournalBlocker`, and is resolved
only by an agent-selected `repair-node` that consumes the attempt (bounded to five consumed repairs per
logical gate). A post-ship adversarial audit found a **liveness dead-end** when two gates fail on one
candidate at the same time (reachable when two writers co-open as a read frontier and both gates settle
`fail`):

- **W1 — whole-tree candidate digest.** `computeReviewCandidateDigest` hashes the entire worktree (minus
  the project/`.kw`/`.git` prefixes), so a second writer's fix invalidates the first attempt's
  `candidate_digest` even when the two writers touch disjoint paths → `candidate_digest_changed`.
- **W2 — orphan guard.** Reopening writer A leaves it `in_progress`; the sibling repair of gate B then
  refuses `would_orphan_in_progress`.
- **W3 — per-node barrier baseline poisoning (discovered empirically, not named in the issue).** The
  per-node barrier diffs a writer's ref-anchored baseline against a *whole-worktree* snapshot and rejects
  any path outside that writer's declared set. Any path a *sibling* writer touched after the baseline is an
  out-of-allowlist overflow — so `repair-node` on the *first* gate already dies on `original_barrier_failed`
  / `write_set_overflow`, before any sibling repair exists, with the tree byte-identical to what was
  reviewed. **W3 proves that scoping the digest alone cannot fix #683.**

Only a whole-plan discard recovered. A same-root-cause crash-retry variant (crash between
`repair_settled_written` and `repair_consumed_written`, writer edits before retry) wedged identically because
the `consumptionPending` resume path sat *below* the digest check.

## Decision

Ship an explicit, append-only journal **re-bind transaction** (issue direction (c)), not a relaxation of any
existing refusal. When a repair's writer diff is poisoned only by *attributable* sibling movement, the writer's
barrier baseline is re-anchored to a **synthetic tree** that agrees with the current tree everywhere except the
writer's own declared paths, where it keeps the old baseline content. The writer's reviewed diff is provably
byte-identical (`baselineReused` stays honest); the barrier stops being a whole-tree approximation and becomes
exact. `barrierCheck()` itself is untouched. The harness never chooses a repair owner, splits a node, or
rewrites the DAG — it only makes the mechanical proofs precise.

### The admissibility predicate (`proveRebindAdmissible`)

A failed attempt `X` survives an unrelated repair by writer `w` iff, under **P0** (serialization —
`would_orphan_in_progress` is NOT relaxed; repairs stay one-writer-at-a-time), all of:

- **P1 — residue unchanged.** Every path declared by *no* node is byte-exact with the reviewed candidate.
- **P2 — own slice unchanged.** Every path in the attempt's own producer slice is byte-exact.
- **P3 — foreign delta fully attributed.** Every changed *other-declared* path is either (P3a) unchanged
  since the candidate, or (P3b) owned by a writer whose repair on a *different* gate is **currently live** and
  whose declared set is disjoint from the slice. No attributed path escapes review — the owner's own
  post-dominating gate is folded back to `pending` and must re-review it.
- **P4 — rebind cap.** Bounded rebind records per attempt.
- **P5 — re-anchor safety.** The synthetic base cannot alter any path the repairing writer may write
  (`rebind_base_rewrite_unsafe`), asserted before any durable byte.

**No-silent-waiver theorem.** Every tree path is in exactly one of three partitions — own slice (P2), other-
declared (P3), residue (P1). P1 and P2 pin their partitions to byte-exact equality with the reviewed candidate
(the strength the whole-tree digest gave there); the only relaxation is P3, and there a moved path is not
ignored but *attributed to a named owner whose gate must re-review it*. No unreviewed byte reaches the sink.

New refusals (all fail-closed, zero mutation): `candidate_residue_changed` (P1), `candidate_slice_changed`
(P2), `candidate_delta_unattributed` (P3), `rebind_base_rewrite_unsafe` (P5), `rebind_limit_reached` (P4),
`rebind_replay_diverged` (crash replay), plus schema `review_journal_rebind_malformed` /
`review_journal_rebind_chain_invalid`. `candidate_digest_changed` narrows to the fail-closed residual.
The five-consumed-repairs breaker is untouched and sits strictly above every rebind proof and mutation; a
rebind never writes `logical_gate`/`outcome`/`repair.settled`/`consumed_by`, cannot mint or un-consume an
attempt, and leaves `consumed_by:null` so the fence still holds — it cannot launder the breaker.

### Crash-retry

The `consumptionPending` short-circuit is hoisted above the digest/producer/identity/barrier gates (guarded by
`repair.selected_writer===nodeId`, `repair.settled===true`, `consumed_by==null`, writer `in_progress`, and an
`effectiveProducerBinding` identity re-check): once a repair is durably settled the proofs already passed
pre-crash, so the retry consumes instead of re-proving against a tree the reopened writer is legitimately
editing. The three rebind failpoints (`rebind_recorded` → `rebind_base_written` → `rebind_settled`) make an
interrupted rebind inert and idempotently completable; the sole refusing case (a lost/replaced anchored ref)
has a documented non-discard recovery (restore the ref from `.cache/barrier-base-<writer>`).

## The soundness campaign (five adversarial holes, all closed)

The partition proof was adversarially refuted five times, each a distinct axis of "a comparator, keyspace,
examination set, or attribution premise weaker than the digest/barrier." Each was reproduced end-to-end
through the real CLI and fixed RED-first:

1. **R1 — examination set.** P3 measured partition 2 against the *writer's baseline* (`foreign`), not the
   reviewed *candidate*; a sibling's approved file reverted/deleted out-of-band fell in no partition. Fixed:
   `examine = foreign ∪ { p ∈ declaredUnion∖slice : now.declared[p] ≠ cand.declared[p] }`.
2. **R3 — mode axis.** The declared blob map was sha-only while the digest/residue/P5/barrier are mode-aware;
   a mode-only revert of an approved `chmod +x` (and, equivalently, a file↔symlink swap sharing a blob)
   escaped. Fixed: the map records `'<mode> <sha>'` from the same `ls-tree` line, made *unrepresentable*
   weaker by a strict schema format gate.
3. **R4 — keyspace.** A declared path literally named `__proto__` was silently dropped by the plain-object
   map. Fixed: `Object.create(null)`.
4. **R6 — integer keys (fail-closed, deferred follow-up).** Canonical-integer path keys enumerate
   numeric-first, so a correctly-built map can fail its own `isCanonicalBlobMap` check → permanent
   `review_journal_malformed`. Verified fail-closed on every consumer; filed for order-insensitive
   canonicalization.
5. **R7 — temporal axis.** P3b's `repairWriters(X)` admitted any writer *ever* selected for repair
   (`repair.selected_writer` is set once, never cleared), so after an owner's gate re-passed and went
   `complete`, later out-of-band movement of its path was still laundered in. Fixed: admit an owner only when
   its selecting gate is **currently** not-complete —
   `(a.logical_gate.members||[]).some(m => ledgerStatuses[m] !== 'complete')` — so attribution requires a
   *live, pending* re-review, not a discharged one.

**Invariant-dependence (recorded so a future change re-examines it):** the R7 predicate treats every
non-`complete` status as "will re-review." Two dependencies keep this sound today and must be re-checked if
selector-arm pruning or per-member fan-out folds ever change: (a) the `.some` fan-out quantifier is safe only
because a partially-complete fan-out gate leaves an `in_progress` member that the `would_orphan_in_progress`
guard refuses before any durable byte (N2R2); (b) an `n/a` (pruned) member is admitted as "will re-review"
though it never does, and unlike `in_progress` is not orphan-fenced — unreachable today because `n/a` arises
only from selector-arm folding and a gate holding a repair record always sits in a selected, non-pruned arm
(N3R1). A stricter posture (`status ∈ {'pending','in_progress'}`, and a fail-closed default when the ledger is
absent) is filed as a hardening follow-up.

## Process note — repairable adversarial verification

The first run of this bundle authored one bundle-wide adversarial fan-out gate post-dominating three
independent writers (an antichain). When it refuted, `repair-node` returned `repair_requires_replan` because
an antichain has no unique maximal producer, so the refutation could not be routed to any single writer and the
plan could only be discarded. **Lesson:** an adversarial (or any repairable) gate must post-dominate a single
writer or a dependency chain with a unique maximal producer — never a fan-out over independent siblings — so a
refutation is single-writer repairable. The re-plan concentrated all new write work in one writer.

## Bundled decisions

- **#684 — mutation-killing coverage.** Six #682 fail-closed paths (the `open-next`/`open-ready` journal
  fences, `candidate_digest_changed`, the unique-maximal-producer `repair_requires_replan`,
  `writer_identity_changed`, and the missing-journal witness detector) plus the #664 collective fold now each
  have a regression that fails when its single guard is disabled, driven through the real subcommand/CLI. The
  #664 fold is exercised on a real frozen plan with a real journal attempt; both fold implementations have
  independent single-site killers.
- **#685 — parent-directory fsync (discharges D-682-01 R17).** `writeFileAtomicReplace` now fsyncs the parent
  directory after `renameSync`, in the canonical helper (`kaola-workflow-adaptive-schema.js`, byte-identical
  across all four editions) and its genuine sibling (`kaola-workflow-roadmap.js` + edition ports).
  `kaola-workflow-claim.js` was confirmed to *delegate* to the helper (the issue text's "sibling copy" claim
  was incorrect) and needed no change. The directory fsync is platform fail-soft — a directory that cannot be
  opened or fsynced degrades silently and never turns a previously-accepted write into a refusal, and never
  swallows a real rename/`ENOSPC` error. The same gap in the opt-in `fast`/`full` advance scripts'
  `writeFileAtomic` is filed as a separate follow-up.

## Consequences

- The simultaneous-failed-gates dead-end and its crash-retry variant recover in place, with no whole-plan
  discard, under mechanically-precise fail-closed proofs; the seven-step scenario ends both gates settled-pass
  with an unchanged `plan_hash`.
- `kaola-workflow-adaptive-node.js` is a generated aggregator (propagated via `sync:editions`);
  `kaola-workflow-adaptive-schema.js` remains a byte-identical anchor across all four editions. Every change
  in this bundle passed all four `test:kaola-workflow:{claude,codex,gitlab,gitea}` chains run sequentially.
