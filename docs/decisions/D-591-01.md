# D-591-01. Per-member `leg_path`/`leg_branch` threaded into `open-ready` dispatch

Date: 2026-07-02
Status: Accepted
Issue: #591
Related: #588 (the companion write co-open width/mix coverage + `max_concurrent` fix,
             `docs/decisions/D-588-01.md`, landed in the same bundle-588-591 audit)

## Context

Write-leg isolation is **discipline-dependent, not transparent**: when a `parallel_safe`
write frontier co-opens, `provisionLeg` creates a real per-member
`.kw/legs/<project>/<node-id>` git worktree (default-on), and the plan-run prose instructed
the orchestrator to "dispatch each leg with its absolute `legPath`" — but the `open-ready`
JSON's per-member `dispatch` object (`buildDispatch`) was assembled with only the shared
`working_dir`, not the member's own leg. The leg path was recoverable only by cross-
referencing the separate top-level `laneGroup` descriptor against member ids.

The single most safety-critical routing fact for a co-open write dispatch therefore lived
OUTSIDE the per-member dispatch card the orchestrator is told to use. The existing
fail-closed safety net still holds if the orchestrator slips (a relative-path own-lane slip
lands in the parent, invisible to the per-leg barrier, and is caught by the parent-clean
fence before merge → `merge_conflict`/repair — never silent cross-contamination or loss), but
that net fires late, at group synthesis, after every leg has already run. Found by a
2026-07-02 parallel-execution audit; this path is also seldom exercised on live issues
(recent releases were mostly serial chains), so ergonomics defects here are not caught by
ordinary use.

## Decision

Thread each opened member's provisioned leg directly into that member's OWN `dispatch`
object, rather than requiring a cross-reference against `laneGroup`:

- **`buildDispatch()`** conditionally attaches `d.leg_path` / `d.leg_branch` ONLY when the
  caller supplies non-null, non-blank `ctx.leg_path` / `ctx.leg_branch` — mirroring the
  pre-existing `goal_line` conditional-attach pattern. Every other `buildDispatch()` call site
  (`orient`/`open-next`, `close-and-open-next`'s fused advance) passes no leg fields, so those
  paths are unaffected.
- **`runOpenReady()`'s `opened[]` map** reads `legInfo = legs[n.id]` — the Phase-1
  provisioned leg for that member, populated only inside the ≥2-member write co-open branch —
  and passes `legInfo.legPath` / `legInfo.legBranch` into that member's own `buildDispatch()`
  call. `legInfo` is `null` on the serial/read path (no legs are ever provisioned there), so
  `buildDispatch()` omits both keys.
- **Byte-identity guard.** A new test asserts the serial `open-ready` dispatch payload
  contains NO `leg_path`/`leg_branch` key anywhere (a raw-string probe over the JSON), proving
  flag-OFF byte-identity rather than merely asserting the two keys are individually absent.
- **Prose repointing (all six routing surfaces + the frontier card).** The 3 Claude commands,
  the 3 Codex SKILL packs, and `docs/plan-run-cards/frontier-batch.md` were reworded
  identically (byte-diff-confirmed across all six changed regions) to dispatch each leg
  directly from that member's own `dispatch.leg_path` (and `dispatch.leg_branch`) — every
  `Edit`/`Write` an absolute `<dispatch.leg_path>/...` path, every `Bash` a
  `cd "<dispatch.leg_path>" &&` — instead of requiring the laneGroup cross-reference. The
  `laneGroup`/`laneGroup.legs` descriptor is retained (not removed) for group-level
  observability (shared baseline, `write_union`). No provenance (issue refs, decision IDs) was
  added to the six agent-facing prompt surfaces, per the standing convention.

## Consequences

- The orchestrator now routes every co-opened write leg directly from the per-member payload
  it is already told to dispatch from — no separate lookup against `laneGroup` is required for
  routing (though `laneGroup` remains available for observability).
- Serial/read-path `open-ready` output is byte-identical to pre-#591: the existing shape tests
  stay green, reinforced by the new raw-string byte-identity guard.
- The fail-closed containment net (parent-clean fence → `merge_conflict`/repair, bounded K=2
  repair thrash) is unchanged and still the backstop if a leg agent ignores its own
  `dispatch.leg_path` — this issue narrows the WINDOW in which that net is needed, it does not
  replace it.
- RED-first: an extension of the existing `LEG-PROVISION-ON` real-git test asserted each
  opened member's `dispatch.leg_path` == that member's provisioned leg path (and
  `dispatch.leg_branch` == `kw/legs/<project>/<node-id>`) and failed `undefined` against the
  pre-fix code (6 failing assertions); now green.
- Suite assertion count grew alongside the concurrently-landed #588 coverage work in the same
  commit (1127 → 1219 total; +13 attributable to this issue's `LEG-PROVISION-ON` extension and
  byte-identity guard).
- Cross-edition: `scripts/kaola-workflow-adaptive-node.js` (canonical + codex twin
  byte-identical via `sync:editions`) and both forge ports (GitLab, Gitea) carry the
  `buildDispatch`/`runOpenReady` change identically; the six routing-surface edits are
  hand-mirrored across all three commands and all three Codex SKILL packs (the propagation
  contract `scripts/test-route-reachability.js` + all four `validate-*-contracts.js` machine-
  enforce full six-surface reach); all four `npm run test:kaola-workflow:{claude,codex,gitlab,
  gitea}` chains are required green before release per the standard #307 cross-edition
  obligation.

## Alternatives considered

- **Keep `laneGroup` as the sole leg-routing source and only tighten the prose's warning about
  cross-referencing it correctly.** Rejected: this does not remove the structural hazard — the
  routing-critical fact still lives outside the per-member payload it is meant to route — it
  only documents around the hazard rather than closing it.
- **Attach the full lane-group descriptor (or a nested map of every member's leg) to each
  member's `dispatch` object, instead of two scalar fields.** Rejected: unnecessary
  duplication — every member would carry every OTHER member's leg info too, growing the
  payload with data the member never needs. The two scalar fields (`leg_path`, `leg_branch`)
  are exactly what a member needs to route its own dispatch, mirroring the existing scalar
  `goal_line` pattern rather than introducing a new nested shape.
