## Plan Notes

### Settled design contract (canonical spec for n1-finalize-gate — implement EXACTLY this)

This is **Case A** (#486): #520's live repro already established the root cause; shape and
fix are both knowable. No investigation arc. The single genuinely-open mechanism — the
receipt-vs-finalize-commit ordering — is settled below by the planner (do NOT re-derive it).

**ROOT CAUSE (verified):** `cmdFinalize` (claim.js), in its `--keep-worktree` block, commits
`chore: archive {project}` UNCONDITIONALLY — it never checks the chain-receipt. The #432
chain-receipt gate is a SEPARATE `--finalize-check` step the contractor/orchestrator is
*supposed* to run before Step 8, but (a) the contractor is a subagent and run-chains (~15 min,
claude chain alone ~574s) exceeds the Bash 600s ceiling → auto-backgrounded → a subagent cannot
await it across turns, so the receipt is never written; and (b) even a working Step 8c produces a
receipt whose `headSha` is wrong relative to the already-landed commit. Result: the archive
commit lands with no verified receipt — a fail-OPEN of a fail-CLOSED gate.

**ORDERING FACT (traced, do not re-trace):** between the LAST implementation commit (HEAD) and
`cmdFinalize`'s `--keep-worktree` archive-commit block, NOTHING commits — Step 8a (artifact
mirror) only `cp`s `.cache` files, it does not commit. So at the instant `cmdFinalize` is about
to commit `chore: archive`, **HEAD is still the impl commit**, which is exactly the commit the
main-session-generated receipt covers (`receipt.headSha === HEAD`). Therefore the ordering fix is
**gate-BEFORE-the-archive-commit**, NOT "accept the receipt at the impl commit vs literal HEAD".
This means **`kaola-workflow-plan-validator.js` does NOT change** — `cmdFinalize` SHELLS the
EXISTING, UNCHANGED `--finalize-check` gate (which already compares `receipt.headSha` to current
HEAD and fails closed on `chains_unverified` > `chains_stale` > `chains_red`, dual-mode self-host
vs consumer via the existing package.json discriminator). Do NOT touch plan-validator.js or its
forge ports. Do NOT touch run-chains.js (generation logic is unchanged; only its CALLER moves).

**THE FIX (three coupled parts — all in n1-finalize-gate):**

1. **Move receipt GENERATION to the main session/orchestrator.** The orchestrator runs
   `kaola-workflow-run-chains.js` over the last impl commit (it CAN await a backgrounded run via
   task-notifications + re-invocation; a subagent cannot). This is a PROCEDURE change — documented
   in n2-finalize-prose. No script needs to "generate" inside the contractor any more.

2. **`cmdFinalize` runs the `--finalize-check` gate FAIL-CLOSED, before committing.** In claim.js
   `cmdFinalize`, immediately BEFORE the `--keep-worktree` archive-commit (`chore: archive`) is
   staged/committed, shell `kaola-workflow-plan-validator.js --finalize-check --json` against the
   active plan + repo root. On a non-`pass` result (any typed refusal: `chains_unverified`,
   `chains_stale`, `chains_red`, `final_validation_*`, `unattributed_change`), `cmdFinalize` must
   **refuse to commit** — exit non-zero with a typed refusal (e.g. `finalize_gate_unverified`
   carrying the inner reason + operator_hint), leaving NO archive commit. Run this gate
   **unconditionally at finalize-time** (not only on the worktree path): on the worktree path it
   guards the `chore: archive` commit; on an in-place run (where cmdFinalize's commit block is
   skipped and the contractor's Step 8 makes `chore: finalize`) it still runs and fails closed, so
   the contractor's Step 8 is provably gated for BOTH paths. The dual-mode discriminator and the
   consumer (`final-validation.md`) branch are the EXISTING `--finalize-check` behavior — reuse it,
   do not reimplement.

3. **NO BYPASS FLAG.** Do NOT add `--skip-receipt-check` / `--no-verify` / any escape hatch. A
   bypass on a fail-closed gate is exactly how this hole was opened; the AC ("the finalize commit
   cannot land without a verified-green chain receipt") forbids a bypass by construction. The
   breaking tests (below) get REAL fixtures, never an escape hatch.

**Cross-edition mirror:** `kaola-workflow-claim.js` is hand-ported (NOT a GENERATED_AGGREGATOR — it
is NOT regenerated by edition-sync), so it will NOT trip `generated_port_split`, but the
`cmdFinalize` behavior change MUST be mirrored byte-for-byte (modulo forge nouns) into all FOUR
claim editions — root + codex twin + gitlab + gitea forge ports — declared in n1's write set.
Canonical spec for the three mirrors: the FULL accumulated root `cmdFinalize` diff, every hunk
modulo forge nouns. Keep these forge-neutral if any prose strings are added (no `gh`/`glab`).

**Breaking tests get real fixtures (in n1, RED-first):** the new fail-closed gate refuses any
self-host `--keep-worktree` finalize that lacks a chain-receipt. Every test that finalizes in a
self-host scratch repo via the gated path must get a `chain-receipt.json` fixture (self-host) or
`final-validation.md` (consumer), per the existing discriminator — across all four chains:
`scripts/simulate-workflow-walkthrough.js` (claude), its codex twin
`plugins/kaola-workflow/scripts/simulate-kaola-workflow-walkthrough.js`, the two forge
`test-{gitlab,gitea}-workflow-scripts.js`, `scripts/test-bundle-finalize.js`, and the new
fail-closed RED assertion in `scripts/test-claim-hardening.js`. tdd-guide: write the failing
"cmdFinalize refuses to commit without a fresh green receipt" test FIRST, then make it green.
